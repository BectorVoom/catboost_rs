# Phase 07.3: Code Review Report

**Reviewed:** 2026-06-20
**Depth:** standard
**Files Reviewed:** 3
**Status:** issues_found

## Summary

Reviewed the Phase 7.3 `pointwise_hist2` kernel family (non-binary 5/6/7/8-bit,
half-byte 4-bit, binary 1-bit), the host-side dispatch/launch seam in
`gpu_runtime.rs`, and the self-oracle harness in `kernels/pointwise_hist.rs`.

The phase honors most of its own stated invariants well: all three kernels are
`F: Float` generic (no hard-coded float types in kernel bodies), strides are
`CUBE_COUNT * CUBE_DIM` topology-derived with no warp-size literals or `& 31`/`& 63`
masks, the FROZEN binSums index `(feature * n_bins + bin) * 2 + channel` is written
byte-identically across all three families and the host reference, source/test
separation is respected (kernels + seam carry no `#[cfg(test)] mod`, all assertions
live in `pointwise_hist.rs`), and the production seam contains no
`unwrap`/`panic`/`expect`. Host-side length guards correctly surface
`CbError::LengthMismatch` before any launch.

The headline defect is a **device-side out-of-bounds / UB hole**: the explicit
bounds-safety requirement for this phase ("every `bin_sums[cell].fetch_add(...)` and
`cindex[...]`/`indices[...]` access must be guarded against OOB — a malformed bin index
or length mismatch must not fault/UB") is only PARTIALLY met. The host guards validate
buffer *lengths* but never validate the *values* inside `cindex` (the bin) or `indices`
(the object id). A bin value `>= n_bins`, or an `indices[i]` value `>= n`, drives an
unchecked device array index and faults / corrupts memory. The self-oracle never
exercises this because `make_fixture_f64` constructs all bins via `% n_bins` and all
indices via `(0..n)`, so every value is in range by construction — the test cannot fail
on the very case the requirement calls out.

## Critical Issues

### CR-01: Unchecked device array index on `cindex` bin value and `indices` object id — OOB/UB on malformed input

**File:** `crates/cb-backend/src/kernels.rs:644-657` (non-binary), `760-774` (half-byte), `879-893` (binary)
**Issue:**
The phase's stated bounds-safety bar requires that "a malformed bin index or length
mismatch must not fault/UB" and "return a typed error rather than panicking." The
host-side guards in `launch_pointwise_hist2_into` (`gpu_runtime.rs:910-930`) only check
buffer *lengths* (`weight.len()`, `indices.len()`, `cindex.len()`). They never validate
the *values*:

1. **Bin value OOB (non-binary kernel only).** In `pointwise_hist2_nonbinary_kernel`:
   ```rust
   let bin = cindex[feature * n + obj] as usize;          // NOT masked, NOT clamped
   let cell = (feature * n_bins + bin) * 2usize;
   bin_sums[cell].fetch_add(d);                            // OOB if bin >= n_bins
   bin_sums[cell + 1usize].fetch_add(w);
   ```
   If any `cindex` entry for a feature is `>= n_bins` (a malformed quantization, a
   feature with more borders than its dispatched bit-width, or a caller bug), `cell`
   exceeds the `bin_sums` length (`n_features * n_bins * 2`) and the atomic write is an
   unchecked out-of-bounds device store — UB / cross-cell corruption / fault. The
   half-byte and binary kernels happen to mask (`& 15`, `& 1`) so their `bin` is
   structurally bounded, but the non-binary family — the dominant path — does not.

2. **`indices[i]` value OOB (all three kernels).** `obj` is the *value* read from
   `indices`, then used to index `der1[obj]`, `weight[obj]`, and `cindex[feature*n+obj]`:
   ```rust
   let obj = indices[i] as usize;   // i < n is guarded; the VALUE indices[i] is not
   let d = der1[obj];               // OOB if indices[i] >= n
   ```
   The loop guard `i < n` only bounds the *position* `i`; nothing bounds the *value*
   `indices[i]`. A single out-of-range entry in `indices` reads `der1`/`weight`/`cindex`
   out of bounds on the device. The kernel doc comment claims "Every device read is under
   a bounds guard" (`kernels.rs:728`, `847`), which is false for these value-derived reads.

Because device-side indexing in CubeCL is unchecked, neither case surfaces a typed
`CbError` — it faults or silently corrupts the histogram. The self-oracle cannot catch
it: `make_fixture_f64` produces bins via `% n_bins` and `indices` via `(0..n)`, so the
malformed case is never tested. The `length_mismatch_is_typed_error` test validates only
the length guard, not value ranges.

**Fix:** Add a host-side value-range validation in `launch_pointwise_hist2_into`, before
launch, returning a typed error (so the contract "typed error, not UB" holds for all
three families uniformly):

```rust
// After the length guards, before the empty short-circuit:
// Validate object ids fit der1/weight/cindex addressing.
if let Some(&bad) = indices.iter().find(|&&ix| (ix as usize) >= n) {
    return Err(CbError::OutOfRange(format!(
        "indices value {bad} >= n ({n}); object id would read der1/cindex out of bounds"
    )));
}
// Validate every bin fits the dispatched line size (n_bins). The half-byte/binary
// kernels mask, but the non-binary kernel does not — guard uniformly so a malformed
// quantization cannot drive a device OOB store.
if let Some(&bad) = cindex.iter().find(|&&b| (b as usize) >= n_bins) {
    return Err(CbError::OutOfRange(format!(
        "cindex bin value {bad} >= n_bins ({n_bins}); would write bin_sums out of bounds"
    )));
}
```

And add a regression test in `pointwise_hist.rs` asserting that an out-of-range bin and
an out-of-range `indices` entry each surface `CbError::OutOfRange` before launch (mirror
`length_mismatch_is_typed_error`). Also correct the false "every device read is under a
bounds guard" claims in the kernel docs (`kernels.rs:728`, `847`, and the non-binary
doc near `602`) to scope them to position guards only.

## Warnings

### WR-01: Non-binary kernel does not mask the bin, diverging from the half-byte/binary families' defensive masking

**File:** `crates/cb-backend/src/kernels.rs:652` vs `768` / `887`
**Issue:** The half-byte kernel masks `cindex[...] & 15` and the binary kernel masks
`& 1`, both of which keep `bin < n_bins` structurally. The non-binary kernel reads the
bin raw with no mask and no clamp. This is an inconsistency in defensive posture across
the "same frozen seam" family — the family that is documented as byte-identical is in
fact NOT uniform on input hardening. Even after CR-01's host-side guard lands, the
in-kernel asymmetry is a latent trap for the 7.5 consumer and any future caller who
bypasses the seam. Consider documenting explicitly that the non-binary kernel relies on
the caller (host guard) for bin-range validity, since masking an 8-bit value to 8 bits
is a no-op and cannot be the kernel's safety net.
**Fix:** Add a one-line comment at `kernels.rs:652` stating the non-binary bin is NOT
masked and relies on the host-side range guard (CR-01) for safety, so the divergence
from the half-byte/binary masking is deliberate and documented rather than an oversight.

### WR-02: `host_reference_hist2` uses raw bin, masking a kernel/reference divergence for non-power-of-range bins

**File:** `crates/cb-backend/src/kernels/pointwise_hist.rs:123`
**Issue:** The host reference computes `bin = cindex[feature * n + obj]` raw, while the
half-byte/binary kernels apply `& 15` / `& 1`. For the current fixtures these agree
(fixture bins are already `< n_bins`), but the oracle therefore does NOT verify that the
kernel's masking matches the reference's intent for any bin value `>= n_bins`. If a
future fixture or the 7.5 consumer passes bins outside `0..n_bins`, the half-byte/binary
kernels would fold them into a masked bin while the host reference would index
`delta_members[feature * n_bins + bin]` out of range and panic (test-side) — i.e. the
oracle's own reference is not robust to the same malformed input the kernels are claimed
to handle. This is latent (not fixture-reachable today) but couples the oracle's
correctness to the fixture generator's `% n_bins` invariant.
**Fix:** Either mask the host reference identically (`bin & (n_bins - 1)` when `n_bins`
is a power of two) to match the kernels, or assert at the top of `host_reference_hist2`
that all bins are `< n_bins`, documenting that the reference requires in-range bins.

### WR-03: `read_handle_f64` uses `.unwrap()` on the device read-back

**File:** `crates/cb-backend/src/kernels/pointwise_hist.rs:171`
**Issue:** `let bytes = client.read_one(h).unwrap();` — this is test code (exempt from the
no-`unwrap` production rule), but a device read-back failure here aborts the whole test
binary with an opaque panic rather than a localized assertion failure, and it is the one
place in the harness that reads a hand-off handle through a *freshly constructed* client
(`read_handle_f64` builds a new `client` at line 169-170). The production seam comments
elsewhere (`gpu_runtime.rs:368-373`) warn at length that a Handle is bound to its
originating client and reading it through a second client "violates a
`slice::from_raw_parts` precondition in the HIP IO controller." `read_handle_f64`
deliberately does exactly that (new device + new client to read a handle allocated by the
seam's internal client). If that pattern is in fact unsafe per the seam's own
documentation, this test relies on undefined behavior; if it is safe, the production
comment is overstated. The two cannot both be true.
**Fix:** Reconcile the contradiction: if cross-client read-back of a hand-off handle is
unsupported (as the production comment asserts), the `handoff` test must read through the
SAME client the seam used (requires exposing the client or a read helper), not a fresh
one. If it is supported, soften the production comment. Either way, replace `.unwrap()`
with `.expect("hand-off handle read-back failed")` for a clearer failure message.

### WR-04: Integer-overflow risk in `feature * n + obj` and `hist2_binsums_len` for large inputs

**File:** `crates/cb-backend/src/kernels.rs:652`, `768`, `887`; `crates/cb-backend/src/gpu_runtime.rs:820-822`
**Issue:** The kernel index `feature * n + obj` and the host-side
`hist2_binsums_len = n_features * n_bins * HIST_CHANNELS` (`gpu_runtime.rs:821`) are plain
`usize` multiplications with no overflow check. On 64-bit targets these are not reachable
for realistic feature/object counts, but the cindex stride `feature * n` is the product of
two unbounded caller-supplied dimensions, and the kernel performs it in (typically 32-bit
device) index arithmetic. There is no host-side cap on `n_features` or `n` analogous to
the `DepthExceeded` cap elsewhere in `CbError`. A pathological `n_features * n` overflow
would wrap the cindex index and silently read the wrong cell (no fault, wrong histogram).
**Fix:** Add a host-side `checked_mul` guard in `launch_pointwise_hist2_into` for
`n_features.checked_mul(n)` and `hist2_binsums_len`, returning `CbError::OutOfRange` on
overflow, so a degenerate dimension is rejected rather than wrapping silently. This makes
the "no silent wrong result" contract hold at the boundary the seam owns.

## Info

### IN-01: `_ = AtomicFinalizePath::HostSumFallback;` lint-suppression is a code smell

**File:** `crates/cb-backend/src/kernels/pointwise_hist.rs:601`
**Issue:** The `let _ = AtomicFinalizePath::HostSumFallback;` at the end of
`length_mismatch_is_typed_error` exists only to suppress an unused-import warning, as the
comment states. Tying a lint workaround to an unrelated test obscures intent and will
break confusingly if that test is renamed or removed. The import is already used via the
`(_, path)` destructuring in the other tests.
**Fix:** Remove the dead statement and, if the import is genuinely unused in some build
configuration, gate the `AtomicFinalizePath` import with `#[allow(unused_imports)]` at the
`use` site instead of materializing a value in an unrelated test.

### IN-02: `debug_assert!(bits <= HIST_MAX_BITS)` is unreachable and provides no release-mode protection

**File:** `crates/cb-backend/src/gpu_runtime.rs:1089`
**Issue:** `bits` is assigned only from the exhaustive `match n_bins { 32 => 5, 64 => 6,
128 => 7, 256 => 8, _ => return Err(...) }`, so `bits` is provably `<= 8 == HIST_MAX_BITS`
on every reachable path. The `debug_assert!` can never fire and is compiled out in
release. It documents intent but adds no runtime protection; the real guard is the
module-level `const _: () = assert!(...)` at lines 88-92, which is the load-bearing one.
**Fix:** Either drop the `debug_assert!` (the comptime const assert already enforces the
invariant at compile time) or convert the comment to plainly reference the const-assert as
the actual guard, to avoid implying the `debug_assert!` is protecting anything at runtime.
