# Phase 07.3: Code Review Report

**Reviewed:** 2026-06-20T08:59:03Z
**Depth:** standard
**Files Reviewed:** 3

## Summary

Fresh independent review of the Phase 7.3 pointwise-histogram family seam
(`gpu_runtime.rs`), the `#[cube]` kernels (`kernels.rs`), and the self-oracle
(`kernels/pointwise_hist.rs`).

Verification of prior findings:

- **CR-01 (unchecked device array index / host-side value-range guard):** RESOLVED.
  `launch_pointwise_hist2_into` validates every `indices` value `< n` and every
  `cindex` value `< n_bins` host-side (gpu_runtime.rs:959-970) before launch, with
  `out_of_range_value_is_typed_error` exercising both arms. The kernel's value-derived
  reads stay in-bounds given those guards.
- **WR-01 (non-binary bin masking):** RESOLVED in intent — the non-binary kernel
  deliberately reads the bin raw (kernels.rs:664) and relies on the CR-01 host guard;
  the half-byte/binary kernels mask (`& 15`/`& 1`). Consistent with the host reference.
- **WR-02 (host reference raw bin):** RESOLVED. `host_reference_hist2` asserts
  `bin < n_bins` before raw-indexing (pointwise_hist.rs:130-134).
- **WR-03 (`read_handle_f64` unwrap):** Still uses `.expect(...)` (pointwise_hist.rs:189),
  but this is TEST code (`kernels::pointwise_hist` is a `#[cfg(test)]` module), which is
  exempt from the unwrap prohibition. Acceptable.
- **WR-04 (integer overflow):** PARTIALLY resolved — the overflow guards exist but are
  ordered AFTER an earlier unchecked multiply that they are meant to protect. See WR-01
  below (re-numbered for this pass).
- **IN-01 / IN-02:** the redundant `debug_assert!(bits <= HIST_MAX_BITS)` is still present
  (gpu_runtime.rs:1145) and is unreachable-true. See IN-01 below.

No BLOCKER-class defects (no UB, no data loss, no silent wrong result that escapes a
guard) were found in the current source. Two WARNINGs and four INFO items remain.

## Warnings

### WR-01: WR-04 overflow guard is ordered after the multiply it protects

**File:** `crates/cb-backend/src/gpu_runtime.rs:941` (guard at 977)
**Issue:** The cindex shape guard computes the product `n_features * n` with an
**unchecked** `usize` multiply at line 941 (and again at line 944 for the error's
`expected` field). The WR-04 overflow guard that is supposed to reject exactly that
overflowing product — `if n_features.checked_mul(n).is_none()` — does not run until
line 977, i.e. AFTER the unchecked multiply has already happened.

Consequences for a degenerate `n_features * n` that overflows `usize`:
- Debug build: line 941 panics on multiply overflow — violates the file's stated "no
  panic in this production helper" contract (gpu_runtime.rs:890-891) and the project's
  "no panics in the production seam" rule.
- Release build: line 941 wraps silently, so the cindex length comparison is done
  against a wrong (wrapped) length. A caller could pass a `cindex` whose real length
  matches the wrapped product while the kernel later strides by the true (overflowing)
  `n_features * n`. The WR-04 guard at 977 does eventually fire, but only after the
  wrapped comparison has already been trusted — the "reject overflow with a typed error,
  never wrap" contract (gpu_runtime.rs:972-976) is defeated for the cindex-stride product.

**Fix:** Move the two `checked_mul` overflow guards ABOVE the cindex length guard so the
product is validated before it is ever computed unchecked, and reuse the checked value:
```rust
// Overflow guards FIRST (before any unchecked product is formed).
let cindex_stride = n_features.checked_mul(n).ok_or_else(|| {
    CbError::OutOfRange(format!(
        "n_features ({n_features}) * n ({n}) overflows usize (cindex stride)"
    ))
})?;
if hist2_binsums_len_checked(n_bins, n_features).is_none() {
    return Err(CbError::OutOfRange(format!(
        "n_features ({n_features}) * n_bins ({n_bins}) * {HIST_CHANNELS} overflows usize (binSums length)"
    )));
}
// Now the length guard uses the already-checked product, never re-multiplying.
if cindex.len() != cindex_stride {
    return Err(CbError::LengthMismatch {
        column: "cindex".to_owned(),
        expected: cindex_stride,
        actual: cindex.len(),
    });
}
```

### WR-02: Doc claims a histogram "host-sum fallback" that the code never performs

**File:** `crates/cb-backend/src/gpu_runtime.rs:879-882`
**Issue:** The `launch_pointwise_hist2_handle` doc states the cross-thread merge is the
in-kernel atomic "when the device advertises the needed atomic-add support ... else a
documented host-sum fallback." For the histogram path there is NO host-sum fallback:
`launch_pointwise_hist2_into` (non-wgpu arm, lines 1196-1217) and
`launch_pointwise_hist2` always launch the f64 `Atomic<F>::fetch_add` kernel regardless
of `device_supports_f64_atomic_add`, and the reported `AtomicFinalizePath` is explicitly
"informational ... the kernel always ran the in-kernel atomic" (lines 1264-1266,
1287-1289). The doc describes behavior that is true of the 7.1 *reduce* helper
(`launch_block_reduce_atomic_f64`) but not of this histogram seam. A future maintainer
reading the doc could wrongly assume a degraded-but-correct fallback exists on a device
that lacks f64 atomics, when in fact the kernel would run an unsupported atomic op.

**Fix:** Correct the doc to match the code — there is no host-sum fallback on the hist2
merge; the channel float type (f64 vs f32 on wgpu) is the only capability adaptation, and
the reported path is informational:
```text
/// # Atomic merge (D-03 / Pitfall 1)
///
/// The cross-thread merge into `binSums` is ALWAYS the in-kernel `Atomic<F>::fetch_add`
/// (a handle cannot host-sum without a read-back). The only capability adaptation is the
/// channel float type: f64 on rocm/cuda/cpu, f32 on wgpu (WGSL has no f64 atomics —
/// RESEARCH A1). The reported `AtomicFinalizePath` is INFORMATIONAL (the device's
/// advertised f64-atomic capability), NOT a selector — unlike the 7.1 reduce helper there
/// is no host-sum fallback on this path.
```

## Info

### IN-01: Unreachable `debug_assert!` adds no value

**File:** `crates/cb-backend/src/gpu_runtime.rs:1145`
**Issue:** `debug_assert!(bits <= HIST_MAX_BITS)` can never fail: `bits` is assigned only
`5|6|7|8` from the immediately-preceding `match n_bins` (lines 1129-1140) and
`HIST_MAX_BITS == 8`. The invariant is already a compile-time fact via the module-level
`const _: () = assert!(2 * (1 << HIST_MAX_BITS) <= HIST_SHMEM)` and the literal match arms.
**Fix:** Delete the line (the compile-time `const _` assert already guards the shared-mem
sizing), or convert it to a comment documenting the match-arm invariant if retention is
preferred.

### IN-02: Duplicated launch-geometry computation in `launch_pointwise_hist2_into`

**File:** `crates/cb-backend/src/gpu_runtime.rs:994-1003` and `1156-1162`
**Issue:** `num_cubes`/`count`/`dim` are computed once at lines 994-1003 (used by the
binary and half-byte branches) and then recomputed identically at lines 1156-1162 for the
non-binary branch, shadowing the first set. The comment at 996 even claims "one geometry"
(IN-02 intent) but the non-binary arm re-derives it. Harmless but contradicts the stated
single-geometry goal and is dead duplication.
**Fix:** Drop the second computation (1156-1162) and let the non-binary branch use the
`count`/`dim` already in scope from 998-1003.

### IN-03: `max_divergence` indexes `device[i]` over `baseline.len()` without a length precondition

**File:** `crates/cb-backend/src/kernels/pointwise_hist.rs:70-84`
**Issue:** The reporter loops `for i in 0..baseline.len()` and indexes `device[i]`. It is
only safe because every caller asserts `device.len() == baseline.len()` immediately before
calling it. This is test code (exempt from the indexing rules), but the coupling is
implicit; a future test that forgets the length assertion would panic with an opaque
index-out-of-bounds rather than a clear message.
**Fix (optional, test hygiene):** zip the two slices (`device.iter().zip(baseline)`) or
add a `debug_assert_eq!(device.len(), baseline.len())` at the top of `max_divergence`.

### IN-04: Over-strict cindex value guard for the masked (binary/half-byte) families

**File:** `crates/cb-backend/src/gpu_runtime.rs:964-970`
**Issue:** The value guard rejects any `cindex` value `>= n_bins` uniformly, including for
the binary (`n_bins == 2`, kernel masks `& 1`) and half-byte (`n_bins == 16`, kernel masks
`& 15`) families whose kernels would tolerate larger raw values via masking. This is SAFE
(stricter than necessary, never UB) and keeps the host reference — which does not mask —
consistent with the device. It is flagged only because the guard's rationale comment
(lines 964-965) speaks solely to "the non-binary path (which does not mask)", which could
mislead a reader into thinking the masked families are unguarded; in fact they are guarded
(more strictly than they need to be).
**Fix (optional):** Extend the comment to note the guard intentionally also constrains the
masked families to keep the host-reference (unmasked) parity exact, or relax the guard to
mask-aware bounds per family if a caller ever needs raw multi-bit cindex on a masked
feature.
