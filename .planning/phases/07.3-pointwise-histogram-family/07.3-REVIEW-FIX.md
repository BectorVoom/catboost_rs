## Fixed Issues

### WR-01: WR-04 overflow guard is ordered after the multiply it protects

**Files modified:** `crates/cb-backend/src/gpu_runtime.rs`
**Commit:** f5a4bd4
**Applied fix:** Moved both `checked_mul` overflow guards (cindex stride
`n_features * n` and binSums length) ABOVE the cindex length guard, before any
unchecked product is formed. Bound the checked cindex stride to a new
`cindex_stride` local and reused it for the length comparison (`cindex.len() !=
cindex_stride`) and for every `ArrayArg::from_raw_parts(cindex_handle, ...)`
launch site (6 call sites across binary / half-byte / non-binary branches),
eliminating the redundant unchecked `n_features * n` re-multiplications. A
degenerate dimension is now rejected with a typed `CbError::OutOfRange` before
the product is ever computed unchecked — no debug-build panic, no release-build
wrap-then-compare. Verified: `cargo check -p cb-backend` clean.

### WR-02: Doc claims a histogram "host-sum fallback" that the code never performs

**Files modified:** `crates/cb-backend/src/gpu_runtime.rs`
**Commit:** 730f875
**Applied fix:** Rewrote the `# Atomic merge` doc on
`launch_pointwise_hist2_handle` to state the cross-thread merge is ALWAYS the
in-kernel `Atomic<F>::fetch_add`; the only capability adaptation is the channel
float type (f64 on rocm/cuda/cpu, f32 on wgpu). Clarified that the reported
`AtomicFinalizePath` is informational (advertised f64-atomic capability for the
7.6 sign-off), not a selector, and that — unlike the 7.1 reduce helper
`launch_block_reduce_atomic_f64` — there is no host-sum fallback on this path.
Doc-only change; intra-doc link target verified to exist.

## Fixed Issues (Info)

### IN-01: Unreachable `debug_assert!` adds no value

**Files modified:** `crates/cb-backend/src/gpu_runtime.rs`
**Commit:** 730f875
**Applied fix:** Deleted `debug_assert!(bits <= HIST_MAX_BITS)` (always true:
`bits` is one of {5,6,7,8} from the preceding match and `HIST_MAX_BITS == 8`).
Replaced it with a comment documenting that the invariant is a compile-time fact
already enforced by the module-level `const _: () = assert!(...)`.

### IN-02: Duplicated launch-geometry computation

**Files modified:** `crates/cb-backend/src/gpu_runtime.rs`
**Commit:** 730f875
**Applied fix:** Removed the second `num_cubes`/`count`/`dim` computation in the
non-binary branch and let it reuse the single `count`/`dim` already in scope
(also used by the binary/half-byte branches). The earlier set is now used by all
paths — confirmed no unused-variable warning. This realizes the stated
single-geometry (IN-02) intent.

### IN-03: `max_divergence` indexes `device[i]` over `baseline.len()` without precondition

**Files modified:** `crates/cb-backend/src/kernels/pointwise_hist.rs`
**Commit:** 252c33a
**Applied fix:** Rewrote `max_divergence` (test code) to zip the two slices
(`device.iter().zip(baseline)`) instead of indexing `device[i]` over
`baseline.len()`, and added a `debug_assert_eq!(device.len(), baseline.len())`
documenting the precondition. Removes the opaque OOB-panic failure mode if a
future test forgets the length assertion. Verified: `cargo check --tests
-p cb-backend` clean.

### IN-04: Over-strict cindex value guard for the masked families

**Files modified:** `crates/cb-backend/src/gpu_runtime.rs`
**Commit:** 730f875
**Applied fix:** Extended the cindex value-guard comment to document that the
uniform `< n_bins` bound is applied INTENTIONALLY to the masked binary (`& 1`)
and half-byte (`& 15`) families too — keeping the unmasked host reference
bit-exact with the device for every family — and noted when a mask-aware relax
would be warranted. Comment-only; no behavior change.
