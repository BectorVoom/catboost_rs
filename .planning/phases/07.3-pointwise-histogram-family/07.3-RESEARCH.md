# Phase 7.3: Pointwise Histogram Family - Research

**Researched:** 2026-06-20
**Domain:** GPU compute kernels (CubeCL / cubecl-hip on AMD gfx1100) — porting the CatBoost CUDA `pointwise_hist2*` 2-channel histogram fill family at structural parity, device-resident over `R: Runtime`/`F: Float`, with CUDA-style in-kernel atomic accumulation.
**Confidence:** HIGH for the upstream structure and the cubecl 0.10.0 atomic/shared-memory/comptime surface (read from the vendored CUDA source + the pinned crate source in `~/.cargo/registry`); MEDIUM for the exact wave-agnostic shared-memory line-size geometry (a design choice constrained by D-09, validated empirically on gfx1100 in execution); the GPU-06 epsilon is deliberately NOT signed off here (7.6).

## Summary

Phase 7.3 ports the **fill** half of CatBoost's pointwise (single-feature) 2-channel histogram kernels — `ComputeHist2NonBinary<Bits>` (5/6/7/8-bit), `ComputeHist2HalfByte`, and `ComputeHist2Binary` — to run device-resident on the GPU runtimes. `hist2` accumulates, per (feature, bin), two channels: Σ der1 ("target") and Σ weight. Upstream's structure is a **per-block shared-memory histogram** (`TPointHist*` structs hold a `counters[]`/`smem[]` working set sized to the block), filled by an aligned strided object loop (`ComputeHistogram`/`ComputeHistogram2`/`ComputeHistogram4` in `compute_point_hist2_loop.cuh`), reduced in-block (`THist::Reduce()`), then merged into the global `binSums` buffer with `atomicAdd` **only when more than one block contributes to a feature** (`BLOCKS_PER_FEATURE > 1`/`M > 1`); the single-block case is a plain `WriteThrough`. This is the D-03 in-kernel-atomic contract in concrete form.

The three families are genuinely distinct structural kernels (different shared-memory layouts, bin-extraction bit math, and reduce trees) — D-7.3-02's "three families, not one mega-kernel" is correct and matches upstream. The 5/6/7/8-bit specialization within the non-binary family is a **comptime bit-count** that drives the histogram line size, inner/outer histogram bit split, and unroll (upstream carries it as the C++ template parameter `Bits`, `TDeclarePassInnerOuterBitsTrait<Bits-5>`). In CubeCL this maps cleanly to a `#[comptime] bits: u32` kernel argument (the same mechanism as the existing `use_plane`/`inclusive` comptime flags), with the host launch helper selecting it per feature group — pruned at JIT time, no runtime branch, no warp-size literal.

**Primary recommendation:** Build narrowest-first per D-7.3-01: Plan A lands the **device seam + one-byte non-binary path parameterized to 8-bit** (the `ComputeHist2NonBinary<8>` analog) as a `#[cube]` shared-memory-accumulate + atomic-merge kernel reusing the 7.1 `block_reduce_atomic_kernel` atomic-finalize building block and the 7.2 der-handle input seam; self-oracle it against a generalized `reduce_leaf_stats` (leaf→bin) host reference. Then add 5/6/7-bit on the SAME non-binary kernel via the comptime `bits` arg (Plan B), then half-byte (Plan C), then binary (Plan D) as additive plans on the same device seam. Do NOT port `ScanPointwiseHistograms`/`UpdatePointwiseHistograms` (that is 7.5). Express the histogram output as a device-resident `binSums`-shaped handle (`2 * histLineSize * partCount * foldCount` floats, channel-interleaved per bin) so 7.5 plugs in with no host round-trip.

## User Constraints (from CONTEXT.md)

### Locked Decisions (D-7.3-01 … D-7.3-07, plus binding umbrella D-01…D-09, D-7.1-*, D-7.2-*)

- **D-7.3-01 (variant coverage + sequencing):** The FULL `pointwise_hist2` family is in scope — general one-byte non-binary path PLUS 5/6/7/8-bit, half-byte, binary specializations, all running on `rocm` (SC-1). Sequence **narrowest-first across plans**: general one-byte non-binary (parameterized to 8-bit) + device seam first; then 5/6/7-bit on the same template; then half-byte; then binary. Every variant lands in THIS phase — sequencing is plan decomposition, not a scope cut.
- **D-7.3-02 (bit-width specialization mechanism — research-flagged):** Mirror upstream's THREE distinct kernel families (one-byte non-binary, half-byte, binary) — NOT one mega-kernel — because their shared-memory layout and load/unroll differ structurally (D-01). Within the non-binary family, carry the bit-count (5/6/7/8) as a `#[comptime]` parameter controlling histogram line size, shared-mem blocking, and unroll — NOT a runtime branch (mirrors the 7.1 `use_plane` comptime pattern). The exact CubeCL comptime-vs-monomorphization expression is the research-flag item, subject to D-09 (no warp constants) and AGENTS.md generics-float.
- **D-7.3-03 (accumulation strategy):** Mirror `compute_point_hist2_loop.cuh`: per-block shared-memory histogram accumulation over the block's object slice, then an atomic merge of block-local histograms into the global histogram buffer, honoring D-03's in-kernel-atomic determinism + f64 finalization. Reuse the 7.1 `block_reduce_atomic_kernel` atomic-finalize building block and the wave-agnostic plane/shared-mem pattern (D-09 — shared-mem size is a `#[comptime]` const, stride derives from `CUBE_DIM_X`/`PLANE_DIM`, never a literal 32/64). This is the real on-device fill — distinct from the scatter-only `histogram_scatter_kernel` (D-02 baseline). D-03/D-09 are locked by the umbrella, restated not reopened.
- **D-7.3-04 (self-oracle + fixtures):** Self-oracle the device histogram against a host reference 2-channel histogram built with the existing ordered `cb-core::sum_f64` reduction — generalize `cb-compute/src/histogram.rs::reduce_leaf_stats` gather-then-`sum_f64` from leaves to (feature, bin) cells (per bin: Σ der1 = `sum_weighted_delta`, Σ weight = `sum_weight`). The harness REPORTS observed max abs + rel divergence (run-to-run atomic variance absorbed); it does NOT hard-code or sign off the GPU-06 epsilon (7.6's job). Fixtures: random f32 + f64, non-trivial weights, edge cases (empty, single object, size not a multiple of block/cube, large N), across the covered bit-widths/border counts. Tests in dedicated `_test.rs`, run on `rocm` in-env.
- **D-7.3-05 (device-residency I/O seam):** Histogram kernels consume the 7.2 der1/der2 device handles directly (NO host fold — D-7.2-04; per-object weight folded HERE downstream) plus a device-resident quantized-bin input (`cindex`) + indices/partition layout mirroring upstream's `(features, cindex, target, weight, indices, partition, binSums)` signature. Output is a device-resident histogram buffer handle (upstream's `binSums`), consumable by the 7.5 score/split + grow loop with NO host round-trip. The post-fill `Scan`/`Update` integration is 7.5; this phase delivers the FILL and its handle.
- **D-7.3-06 (verification gating):** Per-crate, honoring disk pressure: `rocm` gets build + run + self-oracle in-env on gfx1100; `wgpu` gets build + run on host; `cuda` gets build/compile only. `rocm` GPU tests run locally/manually, never in GitHub Actions (D-06).
- **D-7.3-07 (boundary preservation):** All cubecl coupling stays inside `cb-backend`. Do NOT widen the frozen `cb-compute` `Runtime`/`Float` traits; prefer backend-internal launch helpers. `cb-compute` stays cubecl-free; `cb-core`/`cb-model` byte-unchanged. The CPU `cb-compute/src/histogram.rs` reduction is the parity baseline and is NOT modified for the GPU path.

### Pattern 1: comptime bit-count specialization (the research-flag resolution)
**What:** One non-binary `#[cube]` kernel carries the bit-count as a `#[comptime] bits: u32` arg (the SAME mechanism as the existing `use_plane`/`inclusive` comptime flags in `kernels.rs`). The host launch helper passes `5/6/7/8` based on the feature group's bit-width (mirroring upstream's `DISPATCH_ONE_BYTE(..., FromBit, ToBit)` and the C++ template param `Bits`). Comptime values drive the shared-memory line size, the inner/outer histogram bit split (`TDeclarePassInnerOuterBitsTrait<Bits-5>`), and any unroll — pruned at JIT time, zero device divergence, no warp-size literal.

**When to use:** The one-byte non-binary family (Plans A+B). Half-byte and binary are SEPARATE kernels (their layouts differ structurally — D-7.3-02), not comptime cases of the non-binary kernel.

**Example (the established comptime-flag shape this extends):**
```rust
// Source: crates/cb-backend/src/kernels.rs (block_reduce_kernel, use_plane comptime arg)
#[cube(launch)]
pub fn block_reduce_kernel<F: Float>(
    input: &Array<F>,
    output: &mut Array<F>,
    #[comptime] use_plane: bool,    // <-- resolved ONCE host-side, pruned at JIT time
) { /* ... plane path vs shared-mem fallback, both wave-agnostic ... */ }
```
```rust
// New (sketch): the non-binary histogram kernel carries bits as comptime.
// SharedMemory size is a comptime usize const (Pitfall 3); the histogram line size
// and inner/outer split derive from `bits` at comptime, never a runtime branch.
#[cube(launch)]
pub fn pointwise_hist2_nonbinary_kernel<F: Float>(
    der1: &Array<F>, weight: &Array<F>,   // der1 UNWEIGHTED in, weight folded HERE (D-7.3-05)
    cindex: &Array<u32>, indices: &Array<u32>,
    bin_sums: &Array<Atomic<F>>,          // global histogram, atomic-merged (D-03)
    #[comptime] bits: u32,                // 5/6/7/8 — drives line size + unroll
    #[comptime] use_plane: bool,
) { /* per-block shared-mem hist; AddPoint loop; Reduce(); atomic merge */ }
```

### Pattern 2: shared-memory histogram + atomic merge (D-7.3-03 / D-03)
**What:** Each block owns a `SharedMemory<F>` working histogram (the `TPointHist*::counters/smem` analog), zero-initialized, filled by an `AddPoint`-style loop over the block's object slice (per object: read `cindex[indices[i]]` → bin, add `der1[i]` to the target channel and `weight[i]` to the weight channel), in-block reduced (`Reduce()`), then merged into the global `bin_sums` via `Atomic<F>::fetch_add` **only when more than one block contributes to a feature** (upstream's `BLOCKS_PER_FEATURE > 1`/`M > 1` guard); a single-block feature does a plain write (no atomic).

**When to use:** All three families. The atomic merge is the D-03 contract; the f64-atomic capability gate (Pitfall 4 below) decides between the in-kernel atomic path and the documented host-sum fallback, exactly as 7.1's `launch_block_reduce_atomic_f64` does.

**Example (the atomic-finalize building block to reuse):**
```rust
// Source: crates/cb-backend/src/kernels.rs (block_reduce_atomic_kernel)
if tid == 0u32 {
    acc[0].fetch_add(cube_partial);   // in-kernel atomic merge; order non-deterministic (D-03)
}
// Atomic<F> API confirmed: cubecl-core-0.10.0/src/frontend/element/atomic.rs
//   fetch_add(&self, value) -> Inner   (on Atomic<Inner: CubePrimitive<Scalar: Numeric>>)
```

### Pattern 3: device-resident I/O seam (der handles in → binSums handle out, D-7.3-05)
**What:** The launch helper takes the 7.2 der1/der2 device HANDLES (`launch_der_*_handle`/`const_der_handle` outputs) plus device-resident `cindex`/`indices`/`partition`/`feature_meta`, and returns the histogram as a device `Handle` with NO `read_one` on the hand-off path (the 7.2 `launch_der_binary_handle` precedent). The read-back happens ONCE, only in the `_test.rs` self-oracle.

**When to use:** Every plan. This is the load-bearing seam — define it in Plan A and keep it stable across B/C/D so 7.5 plugs in unchanged.

**Example (the handle-out precedent to clone):**

### Anti-Patterns to Avoid
- **One mega-kernel with a runtime bit-count branch:** violates D-7.3-02/D-01; the three families differ structurally and the runtime branch would diverge device-side. Use three `#[cube]` families + a comptime `bits` arg in the non-binary one.
- **A literal `32`/`64` in any accumulation/reduce STRIDE:** violates D-09. The shared-memory SIZE may be a comptime `usize` const (it is the launch geometry, the existing `BLOCK_REDUCE_SHMEM = 32` precedent); strides derive from `CUBE_DIM_X`/`PLANE_DIM`. Upstream's `threadIdx.x & 31` / `tiled_partition<32>` / `<8>` constructs are wave32-specific and must be re-expressed wave-agnostically (D-09).
- **Reading the histogram handle back on the hand-off path:** violates D-7.3-05 / the 7.2 no-readback seam (and the HIP empty-buffer read-back hazard, Pitfall 5). Read back ONCE in `_test.rs` only.
- **Modifying `cb-compute/src/histogram.rs` to serve the GPU path:** violates D-7.3-07. The host-reference 2-channel histogram lives in the `cb-backend` `_test.rs`; it REUSES the `reduce_leaf_stats` SHAPE but does not alter the frozen baseline.
- **Hard-coding a float type or signing off the GPU-06 epsilon:** violates AGENTS.md generics-float and D-7.3-04. Kernels are `<F: Float>`; the oracle REPORTS divergence, 7.6 signs off.
- **Folding the weight into der upstream:** the 7.2 seam hands UNWEIGHTED der1 (D-7.2-01/02); the weight is the histogram's SECOND CHANNEL, folded HERE.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Cross-block histogram merge | A custom lock/spin or a second host-side reduction kernel | `Atomic<F>::fetch_add` (the 7.1 `block_reduce_atomic_kernel` pattern) + the `atomic_type_usage(Add)` capability gate | D-03 mandates the in-kernel atomic; the gate + host-sum fallback is already solved in `launch_block_reduce_atomic_f64`. |
| Wave-agnostic in-block reduce | A `threadIdx.x & 31` warp reduce (upstream) | The 7.1 `plane_sum`/shared-mem tree-reduce with `CUBE_DIM_X`/`PLANE_DIM` strides | D-09 forbids warp-size literals; the wave-agnostic fold is already authored and oracle-tested in `kernels.rs`. |
| Host↔device transfer | Manual `unsafe` pointer copies | `client.create(Bytes::from_elems(...))` / `bytemuck::cast_slice::<u8, F>` | The uniform 7.1/7.2 transfer path; handles the HIP IO controller's `from_raw_parts` preconditions. |
| Ordered host-reference sum for the oracle | A naive iterator `.sum()` | `cb_core::sum_f64` over gathered-in-object-order vecs (the `reduce_leaf_stats` shape) | D-7.3-04 / D-05: the parity baseline must use the frozen sequential order, not an arbitrary fold. |
| Typed launch errors | `unwrap()`/`panic!` on shape/readback failures | `CbResult` + `CbError::Degenerate`/`LengthMismatch` (the 7.2 `gpu_runtime.rs` precedent) | D-13 / workspace lints ban `unwrap` in production; the 7.2 helpers already model the typed-error read-back (WR-05). |

**Key insight:** Almost every hard sub-problem (wave-agnostic reduce, f64-atomic capability gating + fallback, device-handle hand-off with no read-back, typed errors, ordered host reference) was already solved in 7.1/7.2. The genuinely NEW surface in 7.3 is (a) the per-block **2-channel shared-memory histogram fill** with bin extraction from `cindex`, and (b) the **bin-feature `binSums` layout** the output handle must match for 7.5. Everything else is composition of existing patterns.

## Common Pitfalls

### Pitfall 1: f64 in-kernel atomic-add is not guaranteed on HIP/gfx1100
**What goes wrong:** `Atomic<f64>::fetch_add` may be unsupported (or emulated/slow) on the HIP backend; HIP natively supports f32 atomic-add, f64 is optional. A blind f64-atomic merge could fail to compile/run or silently degrade.
**Why it happens:** AMD RDNA3 atomic-float support differs by type; cubecl exposes this via `client.properties().atomic_type_usage(ty).contains(AtomicUsage::Add)`.
**How to avoid:** Reuse the 7.1 `AtomicFinalizePath` pattern verbatim: query `device_supports_f64_atomic_add` on the host, take the in-kernel atomic path when present, fall back to a documented host-sum (or f32-atomic) finalize otherwise — and REPORT which path ran. Note upstream's `binSums` is `float*` (f32) and the per-block in-block reduce is f32; the project's f64-finalization contract (D-03) applies to the FINAL fold. Resolve in Plan A whether the global `binSums` channel is f32 (upstream-faithful) with f64 finalization in the oracle, or f64 device-side; the f32-atomic path is the most HIP-portable and matches upstream's `atomicAdd(float*)`.
**Warning signs:** A HIP compile error mentioning `atomic` + `f64`; a self-oracle that passes on `cpu`/`wgpu` but errors on `rocm`.

### Pitfall 2: the `binSums` layout must match what 7.5 expects (or the hand-off breaks)
**What goes wrong:** If the device histogram handle is laid out differently from upstream's `binSums`, the 7.5 score/split path cannot consume it without a re-layout (a host round-trip — exactly what D-7.3-05 forbids).
**Why it happens:** Upstream's layout is specific: `binSums` is indexed `(GetHistogramOffset(part,fold) * histLineSize + (FirstFoldIndex + fold)) * 2 + channel`, where `histLineSize = 2 * totalBinFeatures` and the 2 channels (target, weight) are interleaved per bin (`split_properties_helpers.cuh::ShiftPartAndBinSumsPtr`, `gpu_structures.h` layout comment "stat0 ... stat1 ..."). `FirstFoldIndex`/`Folds` come from `TCFeature`.
**How to avoid:** In Plan A, define the output handle as the `binSums`-equivalent flat buffer `[partCount * foldCount * histLineSize]` floats with the 2 channels interleaved per (feature, bin), and document the exact index formula in the launch helper. Keep it byte-stable across Plans B/C/D. The single-tree fill needs only `partCount = 1` (or the full-pass part), `foldCount = 1` initially (D-7.3-05 says the multi-part grow-loop depth is 7.5's; this phase does the FILL with minimal partition handling).
**Warning signs:** The 7.5 planner needing to "re-pack" the histogram; the oracle indexing not matching the kernel's write index.

### Pitfall 3: SharedMemory size must be a comptime `usize` const, not a runtime/topology value
**What goes wrong:** `SharedMemory::<F>::new(SIZE)` requires `SIZE` known at compile time; passing a runtime block/topology value fails to compile.
**Why it happens:** CubeCL allocates shared memory statically (the `Cubecl_shared_memory.md` constraint; the existing `BLOCK_REDUCE_SHMEM` const precedent).
**How to avoid:** The histogram line size per bit-width is a comptime const derived from `bits` (e.g. non-binary 8-bit needs a larger line than 5-bit). Compute it at comptime from the `bits` arg, or use a fixed upper-bound comptime size sufficient for 8-bit (the worst case) and use only the needed prefix — mirroring upstream's `__shared__ float counters[32 * BLOCK_SIZE]` worst-case sizing. The size must cover `2 channels * (1 << bits) folds` per the block's feature group at worst.
**Warning signs:** A "shared memory size must be constant" / `ExpandElementTyped` compile error.

### Pitfall 4: upstream's bin math is wave32-specific — must be re-expressed wave-agnostically (D-09)
**What goes wrong:** Directly porting `threadIdx.x & 31`, `tiled_partition<32>`, `(threadIdx.x / 32)`, `<8>` thread tiles, or the `512 * (threadIdx.x/32)` slice offsets bakes in wave32 and breaks D-09 (and would be wrong on wave64).
**Why it happens:** The CUDA kernels are tuned to a 32-lane warp; gfx1100 is wave32-native but the code must stay wave-size-agnostic (the project validates the no-warp-constant PROPERTY on wave32, D-09).
**How to avoid:** Express thread→(bin slot, channel) mapping via `UNIT_POS`/`CUBE_DIM_X`/`PLANE_DIM`/`UNIT_POS_PLANE` and comptime sizes — never a literal 32/64. The in-block reduce reuses the 7.1 wave-agnostic plane/tree-reduce. The STRUCTURE (per-block shared histogram → reduce → atomic merge) is what D-01 requires, not a lane-for-lane transcription of the `& 31` arithmetic.
**Warning signs:** A `32`/`64`/`& 31`/`& 63`/`tiled_partition<32>` literal in a stride or slice offset; a kernel that only passes when launched at exactly 32 threads.

### Pitfall 5: never `read_one` an empty (or hand-off) device buffer on the production path
**What goes wrong:** `read_one` on a zero-length HIP buffer dereferences a null/zero-length pointer (`slice::from_raw_parts` precondition in the HIP IO controller); and any read-back on the hand-off path violates D-7.3-05.
**Why it happens:** Documented in the 7.2 `gradient_gpu.rs` edge-case test and the 7.2 MEMORY note ("Handle bound to its client, never read 0-len").
**How to avoid:** Empty input (n=0 or no features/bins) short-circuits to a zero-length handle with NO launch and NO read-back (the 7.2 `launch_der_*_into` precedent). The self-oracle reads the handle back ONCE, only for non-empty fixtures, and through the SAME client that allocated it (the 7.2 single-client rule).
**Warning signs:** A HIP panic on the empty/n=0 fixture; a read-back through a freshly-constructed client.

### Pitfall 6: the cubecl-hip-sys 7.2 vs HIP 7.1 version skew
**What goes wrong:** The lock carries `cubecl-hip-sys 7.2.5321100` (HIP 7.2 headers) while the box runs ROCm/HIP 7.1.1 — a potential symbol/ABI mismatch at runtime.
**Why it happens:** Flagged in 7.1/7.2 CONTEXT; not yet observed to break (7.1/7.2 ran on rocm in-env). The sys `build.rs` auto-detects the HIP patch from `hipconfig` (do NOT manually pin `cubecl-hip-sys/hip_xxx` — it panics; `cb-backend/Cargo.toml` note).
**How to avoid:** Proceed on the current pin (7.1/7.2 validated it works). If a runtime HIP symbol error appears, the registry cache HAS `cubecl-hip-sys-7.1.5280200` available as a fallback pin. Document, don't pre-emptively re-pin.
**Warning signs:** A HIP runtime "undefined symbol"/version error that did NOT occur in 7.1/7.2.

## Open Questions

1. **f32 vs f64 global histogram channel + atomic support on gfx1100.**
   - What we know: upstream `binSums` is `float*`; the atomic merge is `atomicAdd(float*)`; cubecl exposes `atomic_type_usage(Add)` to gate. HIP supports f32 atomic-add natively.
   - What's unclear: whether gfx1100 + cubecl-hip 0.10.0 advertises f32 atomic-add (almost certainly) and/or f64 atomic-add (uncertain).
   - Recommendation: Plan A queries the capability on the host (reuse 7.1's `device_supports_*_atomic_add`), takes the atomic path when present, host-sum fallback otherwise, and REPORTS which ran. Make the channel f32 (upstream-faithful) with f64 finalization in the oracle; revisit only if f32 atomics are also limited.

2. **Exact `binSums` index formula the 7.5 seam will consume.**
   - What we know: `histLineSize = 2 * totalBinFeatures`; index = `(GetHistogramOffset(part,fold) * histLineSize + (FirstFoldIndex + fold)) * 2 + channel`.
   - What's unclear: how much of the `(part, fold)` offset math 7.5 needs encoded in the 7.3 handle vs computed by 7.5.
   - Recommendation: Plan A defines and documents the full index formula in the launch helper, fills it for the single-tree case (`part=fold=0`), and freezes it across B/C/D. Flag the `fullPass=false` offset as a 7.5 forward dependency (like 7.1's cross-cube-carry note).

3. **Whether the half-byte/binary families need the `ComputeHistogram2`/`4` vectorized (uint2/uint4 / float2/float4) load paths for parity, or whether the scalar `ComputeHistogram` (OneElement) load is sufficient structural parity for MVP.**
   - What we know: upstream selects 1/2/4-element loads by `ELoadType` (arch + full-pass dependent); the `ComputeHistogram` (scalar) path is the simplest and is the `__CUDA_ARCH__ < 520` fallback.
   - What's unclear: whether D-01 structural parity requires the vectorized loads or accepts the scalar load as the faithful-structure MVP (the vectorization is a perf optimization over the same accumulation structure).
   - Recommendation: MVP per plan = the scalar `ComputeHistogram` (OneElement) accumulation structure (faithful blocking + shared-mem + atomic merge); note the uint2/uint4 vectorized loads as an additive perf follow-up (the umbrella defers perf-parity-beyond-structural to a follow-up — 07-CONTEXT Deferred). CubeCL `Line<F>`/dynamic vectorization (`06_vectorization.md`) is the vehicle if/when added.
