## FROZEN binSums layout (do NOT change in Plans B/C/D or 7.5)

```text
histLineSize = HIST_CHANNELS * totalBinFeatures      (HIST_CHANNELS = 2; totalBinFeatures = n_features * n_bins)
index(part, fold, feature, bin, channel) =
    (GetHistogramOffset(part, fold) * histLineSize + (FirstFoldIndex(feature) + bin)) * HIST_CHANNELS + channel
```

Single-tree collapse delivered by this plan (`partCount = foldCount = 1`, `GetHistogramOffset(0,0)=0`, one feature group with `FirstFoldIndex=0`):

```text
index(feature, bin, channel) = (feature * n_bins + bin) * 2 + channel       channel 0 = Σ der1, channel 1 = Σ weight
buffer length = n_features * n_bins * 2 floats
```

This mirrors upstream `split_properties_helpers.cuh::ShiftPartAndBinSumsPtr` + `pointwise_hist2_one_byte_templ.cuh:132-145` (`... * 2 + w`). The host oracle indexes the buffer cell-for-cell with the kernel write index. The `fullPass=false` multi-part offset (`ShiftPartAndBinSumsPtr` else-branch) is a **7.5 forward dependency** (RESEARCH A2), documented and NOT filled here.

## der-handle-in -> binSums-handle-out seam (frozen for 7.5)

`launch_pointwise_hist2_handle(der1, weight, cindex, indices, n_bins, n_features) -> CbResult<Handle>` consumes the 7.2 der1 (UNWEIGHTED) + weight handles + a device-resident `cindex` (feature-major `cindex[feature*n + obj]`) / `indices` (object visiting order), folds the per-object weight HERE as channel 1 (D-7.3-05), and returns the binSums device handle with **NO `read_one` on the seam**. The 7.5 consumer reads this handle through the same crate; the read-back happens ONCE only in the oracle.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] CubeCL `usize`/`u32` index-type mismatch in the kernel**
- **Found during:** Task 2 (first rocm compile)
- **Issue:** The kernel mixed comptime `n: u32` / `n_features: u32` with `usize`-typed topology intrinsics (`ABSOLUTE_POS`, `CUBE_COUNT` are `constant_usize!`; `CUBE_DIM` is `u32`) and `usize`-returning `Array::len()`, producing `NativeExpand<usize>: From<NativeExpand<u32>>` and add-assign errors at the `#[cube(launch)]` expansion.
- **Fix:** Loaded `cubecl_error_solution_guide/mismatched types.md` (per AGENTS.md, before any fix). Per its point 3 ("avoid `usize`/host-only types in device code" is about VALUES; indexers require `usize`), kept all index arithmetic in `usize` (`n = indices.len()`, `stride = CUBE_COUNT * (CUBE_DIM as usize)`), cast the `u32` array-read VALUES (`obj`, `bin`) to `usize` at the index site, dropped the comptime `n`/`n_features` args (n derived from `indices.len()`, n_features made a runtime `u32` scalar arg), kept `bits` comptime.
- **Files modified:** `crates/cb-backend/src/kernels.rs`, `crates/cb-backend/src/gpu_runtime.rs`
- **Commit:** `4c85303`

**2. [Rule 1 - Bug] wgpu produced an all-zero histogram (f64 atomics absent in WGSL)**
- **Found during:** Task 3 (wgpu host run)
- **Issue:** The f64 `Atomic<f64>::fetch_add` kernel silently produced zeros on wgpu (`rel_div = 1.0`, off by the full value) because WGSL has no f64 atomics at all (unlike HIP, which emulates them). The oracle correctly FAILED.
- **Fix:** Backend-specific gap fix inside cb-backend only (Task 3 explicitly permits this). Per RESEARCH A1, `launch_pointwise_hist2_into` now dispatches an **f32** binSums channel on the wgpu arm (upstream-faithful `atomicAdd(float*)`, natively supported) and keeps the **f64** channel on rocm/cuda/cpu; `read_binsums_f64` upcasts the f32 channel transparently so the 7.5 seam + oracle observe a uniform f64 layout. The oracle bound `HIST_BOUND` and the test handle read are cfg-gated (f32 ~1e-3 on wgpu, f64 ~1e-9 elsewhere).
- **Files modified:** `crates/cb-backend/src/gpu_runtime.rs`, `crates/cb-backend/src/kernels/pointwise_hist.rs`
- **Commit:** `eede2ef`

## Notable observation (for 7.6 epsilon sign-off)

On gfx1100, `device_supports_f64_atomic_add` (the cubecl capability query) returns **false** — HIP does not ADVERTISE f64 atomic-add — yet the in-kernel `Atomic<f64>::fetch_add` kernel **ran correctly** (HIP emulates it). So the reported `AtomicFinalizePath=HostSumFallback` on rocm is the device-advertised capability, NOT a statement that the kernel host-summed (it did not — the f64 device channel ran the in-kernel atomic, producing the ~1e-10 non-deterministic divergence that proves the atomic merge is live). 7.6 should treat the AtomicFinalizePath report as informational about the advertised capability, not as the actual execution path.
