# Phase 21.5: Code Review Report

**Reviewed:** 2026-07-07
**Depth:** standard
**Files Reviewed:** 5
**Status:** issues_found

## Summary

This phase fuses per-feature histogram accumulation + `O(n_bins)` split scoring into
one `rayon` parallel-over-features pass inside `select_level_plain` /
`select_level_perturbed` (the Amdahl fix, Spike 006), consuming new buffer-reusing
primitives in `cb-compute/histogram.rs`. The overriding constraint is byte-for-byte
`f64::to_bits` parity with the pre-change serial path and ≤1e-5 with upstream.

I traced the parity-critical mechanics against that mandate:

- **Summation order** is preserved. Per-feature `build_bucket_histogram` scatters in
  ascending-object order via `scatter_add_f64` (left-to-right `+=`, the scatter form
  of `sum_f64`); the FALSE-side running prefix and `total − prefix` TRUE-side in
  `scan_and_score_borders_into` match `scan_borders_to_leaf_stats` element-for-element.
  Splitting the old whole-partition histogram into one-per-feature histograms is
  cell-wise identical because leaf is the outermost flat index and feature the next, so
  each cell belongs to exactly one feature.
- **Subtraction trick** (`derive_feature_level_hist`) is the same `relocate_sub`
  (shift=0) + `add_relocated` (shift=n_parent) operand sequence the pre-change
  `advance()` used, restricted to one feature — the smaller-sibling choice
  (`n_true <= n_false`) is a GLOBAL count so every feature picks the same branch,
  matching the old whole-partition behavior. No regression versus the old path.
- **map_init scratch reuse** is race-free: each rayon task owns its
  `ScanScoreScratch` / `FusedFeatureScratch` exclusively; `der1`/`weight`/`bins`/
  `leaf_of`/parent histograms are read-only shared; every scratch buffer is
  cleared+resized at entry so reuse cannot leak stale state.
- **Perturbed RNG contract** is intact: the fused pass (0) draws NO RNG and only
  moves the (RNG-free) accumulate+score forward; passes (1)–(3) — `gen_rand`, the
  per-feature `from_seed(rand_seed + task_idx).advance(10)` reseed, one `std_normal`
  per border ascending, and the per-feature `GetInstance` — are byte-for-byte
  unchanged, indexed by `feature` (not `task_idx`), so alignment with the raw scores
  holds even for empty-border features.

No correctness BLOCKER or security issue was found; `unwrap()`/`expect`/`panic` are
absent from the production files, and the `#[cfg(test)] #[path=...] mod` declarations
in `tree.rs` point to separate test files (the project-permitted structure, not
embedded `mod tests`). The findings below are parity-assurance / test-coverage gaps
and a documentation-invariant inaccuracy that matter given the phase's byte-identity
mandate.

## Warnings

### WR-01: Byte-identity guard covers only `dim=1`; multiclass (`approx_dimension > 1`) fused path is unguarded

**File:** `crates/cb-compute/tests/fused_feature_parity_test.rs:75-124`
**Issue:** Both `check_level0` and `fused_subtraction_trick_equals_full_build` hard-code
`let dim = 1usize`. The fused build/scan/score primitives all loop over
`0..approx_dimension` (e.g. `build_bucket_histogram_into` channels, the per-dimension
FALSE prefix in `scan_and_score_borders_into` at `histogram.rs:1077-1106`), and
`select_level_plain` / `select_level_perturbed` are reachable with `approx_dim > 1`
(oblivious multiclass training: `scratch.approx_dim = der1.len() / n_objects`). A
per-dimension parity regression in the fused path (wrong channel offset, per-dim
`total_d` mis-indexing) would pass this always-run guard. Given the phase constraint is
byte-for-byte parity, the highest-risk shape has no `to_bits` net.
**Fix:** Add a `dim >= 2` case to `check_level0` (dimension-major `der1[d*n + i]`,
`n_channels = dim + 1`) asserting `to_bits` equality of fused vs two-pass per
(feature, border), mirroring the existing dim=1 body.

### WR-02: Always-run guard exercises a primitive the production hot path does not use, and never directly locks `derive_feature_level_hist`

**File:** `crates/cb-compute/tests/fused_feature_parity_test.rs:90-116, 206-238`; `crates/cb-train/src/tree.rs:800-963`
**Issue:** `fused_feature_scan_and_score` — the primitive the RED-first guard locks — is
used ONLY in tests/harness (confirmed: no production caller in `crates/`). Production
`select_level_plain`/`select_level_perturbed` instead inline `build_bucket_histogram`
(level 0) or `derive_feature_level_hist` (level > 0) + `scan_and_score_borders_into`.
Level-0 is covered transitively (same underlying functions), but the level > 0
production derivation `derive_feature_level_hist` — the `relocate_sub(shift=0)` +
`add_relocated(shift=n_parent)` composition — is NOT directly asserted. The only
subtraction-trick test, `fused_subtraction_trick_equals_full_build`, uses a DIFFERENT
decomposition (`relocate_sub(shift=2)` building the FALSE sibling, then
`add_relocated(shift=0)`), so it validates that *a* decomposition equals a fresh build,
not the one production runs. A regression confined to `derive_feature_level_hist`
(wrong shift, wrong sibling slotting) would slip past the always-run net and rely
solely on looser ≤1e-5 / externally-gated oracle suites.
**Fix:** Add an always-run test that calls `derive_feature_level_hist` directly
(both `n_true <= n_false` and `n_true > n_false` branches, level ≥ 2) and asserts
cell-by-cell `to_bits` equality against `build_bucket_histogram(col, .., leaf_of,
n_next, 1, ..)` over integer der1/unit weights.

### WR-03: `relocate_sub` documented bit-identity to `relocate().remove()` is false for uncovered `+0.0` cells (signed-zero divergence)

**File:** `crates/cb-compute/src/histogram.rs:341-407`
**Issue:** The docstring states `relocate_sub` is "Bit-identical to
`self.relocate(n_new_leaves, shift).remove(minus)`". For destination cells NOT covered
by the relocated parent (leaves outside `[shift, shift+n_leaves)`), `relocate_sub`
initializes `data[cell] = -minus[cell]` (`minus.data.iter().map(|&b| -b)`, line 376).
When `minus[cell]` is `+0.0`, this yields `-0.0`, whereas `relocate().remove()` computes
`0.0 - 0.0 = +0.0` (line 296 style). `(-0.0).to_bits() != (+0.0).to_bits()`, so the
documented invariant does not hold for that case. In the current production usage
(shift=0 followed by `add_relocated`) the `-0.0` is washed back to `+0.0` by a
subsequent `+0.0` add, and signed zero is additively neutral in `sum_f64`/`calc_average`
so scores are unaffected — but the invariant is relied on by the RED-first cell-level
`to_bits` comparison and a future caller that consumes a `relocate_sub` result WITHOUT
a following add could observe a parity divergence.
**Fix:** Either correct the docstring to scope the bit-identity to parent-covered cells
(and note uncovered cells differ by signed zero), or normalize the initialization so
uncovered cells are `+0.0` (e.g. seed uncovered slots to `0.0` and subtract `minus`
only where covered, matching `relocate().remove()` exactly).

### WR-04: Empty-border features still pay a full `O(n)` per-feature build/derive every level

**File:** `crates/cb-train/src/tree.rs:915-931, 1046-1060`
**Issue:** In both fused passes every feature index `0..n_features` builds (level 0) or
derives (level > 0) its single-feature histogram before the `borders.is_empty()` check,
solely to keep `feature_hists[feature]` positionally aligned with `parents.get(feature)`
at the next level. A feature with no borders is never scored, yet it incurs a full
`scatter_add` object pass (level 0) or a `relocate_sub`+`add_relocated` derivation
(level > 0) per level. This is a correctness-neutral inefficiency, but it partially
undercuts the phase's own serial-fraction-collapse goal for feature sets with many
border-less columns. (Flagged as maintainability/robustness rather than a perf-only
item because the alignment coupling is implicit and easy to break.)
**Fix:** Store `Option<BucketHistogram>` per feature (or a parallel skip-mask keyed by
`borders.is_empty()`) so border-less features skip the build/derive while preserving
positional alignment; guard `derive_feature_level_hist` to short-circuit when the
parent slot is `None`.

## Info

### IN-01: Hard wall-clock speedup assertion is inherently machine/load dependent

**File:** `crates/cb-train/tests/fused_scaling_test.rs:371-376`
**Issue:** `assert!(fused_16 >= 3.0, ...)` gates on a measured wall-clock speedup ratio.
It is `CB_PERF`-gated (skips under a plain `cargo test`), so it will not flake normal CI,
but if ever wired into an automated gate it would be sensitive to core count, thermal
throttling, and background load.
**Fix:** Keep it opt-in (current) and, if promoted to CI, add margin/retry or downgrade
to a warn-on-regression report rather than a hard fail.

### IN-02: `feature_col` conflates arithmetic overflow with a legitimately empty column

**File:** `crates/cb-train/src/tree.rs:882-892` (`feature_col`)
**Issue:** On `checked_mul`/`checked_add` overflow `feature_col` returns `&[]`, the same
value it would return for a genuinely zero-length column. The downstream
`build_bucket_histogram` over an empty slice silently produces an all-zero histogram
rather than surfacing the index-arithmetic fault. Given `MAX_DEPTH=16` and realistic
`n_features * n_objects`, overflow is not reachable in practice, so this is defensive
dead-weight, not a live bug.
**Fix:** Optional — no change needed; if desired, debug-assert the non-overflow path so
a future contract change (larger index types) does not mask a real fault.
