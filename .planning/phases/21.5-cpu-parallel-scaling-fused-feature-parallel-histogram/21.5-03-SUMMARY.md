# Phase 21.5 Plan 03: Fused feature-parallel histogram in the perturbed oblivious split search Summary

**`select_level_perturbed` now FUSES the per-feature histogram build/derive + `O(n_bins)` scan-score into ONE rayon `map_init`-scratch parallel-over-features pass — moving the O(n·nf) accumulation INTO the parallel region (the Amdahl fix, Spike 006) — while the Pitfall-3 random-draw order, count, and per-feature bookkeeping stay byte-for-byte identical, because that fused pass draws NO RNG and only supplies the unchanged RAW scores to the serial, feature-ascending SetBestScore/SelectBestCandidate passes.**

## Accomplishments
- Rewrote `select_level_perturbed` to fuse build(level 0)/derive(level>0) + scan-score per feature in one `into_par_iter().map_init(ScanScoreScratch::new, ...)` pass (identical to the Plan 02 plain path), producing `Vec<(raw_scores, feature_hist)>`, then splitting it into the per-feature RAW border scores (the SOURCE the RNG pass consumes) and the retained per-feature histograms (the next level's parent).
- Held the Pitfall-3 draw contract byte-for-byte: the fused pass draws NO RNG; `perturb.rng.gen_rand()` (one per level) still precedes the serial, feature-ascending SetBestScore loop, which reseeds a fresh `TFastRng64::from_seed(rand_seed + task_idx).advance(10)` per candidate feature (task_idx skip-on-empty-borders unchanged), draws exactly one `std_normal` per border in ascending order via `random_score_instance`, keeps the strict `>` first-wins `feature_best`, and the per-feature main-RNG `GetInstance` SelectBestCandidate pass. Only the SOURCE of each `raw` changed.
- Made `select_level_perturbed` return `(Split, Vec<BucketHistogram>)` like `select_level_plain`; the grow loop now stores the returned histograms and calls the cheap `advance_leaf_only` for BOTH paths.
- Deleted the now-dead full deriving `advance` method (its only caller was the perturbed path), the `build_initial_hists` flag + level-0 build in `GrowScratch::new` (both paths fuse level 0 in their own pass), the now-unused `weight` parameter of `new`, and the now-unread `n_features` field of `GrowScratch`.

## Decisions Made
- **Accumulation runs before `gen_rand`, draws no RNG:** the fused parallel pass touches no RNG state, so ordering it before `perturb.rng.gen_rand()` leaves the main-RNG sequence and every `feat_rng` seed byte-for-byte identical. This is the plan's "keep the accumulation fused per feature while the RNG-bearing SetBestScore loop stays feature-ascending" arm — the strongest byte-identity guarantee.
- **Symmetry with the plain path:** returning the per-feature histograms + using `advance_leaf_only` unifies the two grow paths on one transition and lets `new` skip the redundant level-0 build for the perturbed path too (the exact serial O(n·nf) fraction this phase removes).
- **`scan_and_score_borders_into` + `to_vec`:** consistent with the plain path and byte-identical (Plan 01 guard); the per-feature raw scores are copied out so the reused `ScanScoreScratch` is free for the next feature the task handles (D-07).

**1. [Rule 3 - Blocking] Dead-code cleanup forced by unifying the two grow paths**

- **Fix:** Deleted the full `advance` method, the `build_initial_hists` flag + its level-0 build branch, the `weight` param of `new` (kept `der1` for `approx_dim`), and the `n_features` field, updating the single `new` call site.
