## Fixed Issues

### WR-01: Byte-identity guard covers only `dim=1`; multiclass fused path unguarded

**Files modified:** `crates/cb-compute/tests/fused_feature_parity_test.rs`
**Commit:** 2b2a885
**Applied fix:** Parametrized `make_inputs`/`check_level0` by `dim` (dimension-major
`der1[d*n+i]`, generalizing the existing per-object hash generator); `dim=1` produces
byte-for-byte the same values as before, so the existing
`fused_equals_two_pass_level0` call sites are unaffected. Added
`fused_equals_two_pass_level0_multiclass`, asserting `to_bits` equality of the fused
vs two-pass path at `dim=3` and `dim=2` on the two existing contrasting shapes
(wide-feature nf=20/nbins=128 and low-feature/high-bin nf=8/nbins=254).

### WR-02: Always-run guard never directly locks `derive_feature_level_hist`

**Files modified:** `crates/cb-train/src/tree_test.rs`
**Commit:** 3de96f9
**Applied fix:** Added `derive_feature_level_hist_matches_fresh_build_both_branches`,
calling the ACTUAL production derivation (`crate::tree::derive_feature_level_hist`,
private-but-accessible from this `tree::general` child test module) directly at
`level=2` (`level >= 2` per the finding), covering both the `n_true <= n_false` and
`n_true > n_false` branches via a skewed pass/fail split, and asserting cell-by-cell
`to_bits` equality against a fresh `build_bucket_histogram` over the same next-level
partition.

### WR-03: `relocate_sub` docstring bit-identity claim is false for uncovered `+0.0` cells

**Files modified:** `crates/cb-compute/src/histogram.rs`
**Commit:** 5ecbeb2
**Applied fix:** Rewrote the doc comment to scope the "bit-identical to
`relocate().remove()`" claim to cells COVERED by the relocated parent, and added an
explicit paragraph documenting the uncovered-cell `+0.0` vs `-0.0` divergence, why it
is additively neutral for every downstream consumer (`sum_f64`/`calc_average`), and
why the current production caller (`derive_feature_level_hist`, shift=0 then
`add_relocated` shift=n_parent) washes it back to `+0.0`. Doc-only change; no
numerical computation altered (per the finding's first suggested fix option).

### WR-04: Empty-border features still pay a full `O(n)` build/derive every level

**Files modified:** `crates/cb-train/src/tree.rs`
**Commit:** ec18f46
**Applied fix:** `matrix.feature_borders` is fixed across levels, so
`borders.is_empty()` is a static per-feature property — a border-less feature is
never scored at ANY level. Changed `GrowScratch::feature_hists` from
`Vec<BucketHistogram>` to `Vec<Option<BucketHistogram>>`, and in both
`select_level_plain` and `select_level_perturbed` moved the `borders.is_empty()`
check to the START of each rayon `map_init` closure, short-circuiting to `(Vec::new(),
None)` BEFORE any build/derive work for border-less features. Positional alignment
with `parents.get(feature)` is preserved via the `Option`;
`derive_feature_level_hist`'s existing `Option<&BucketHistogram>` parameter and
shape-drift fallback already handle a `None` parent correctly, so no change was
needed there. Correctness-neutral per the review (borders are score-gating only) —
re-verified via the full `cb-train --lib` suite (241/241), the rayon
byte-identity-across-thread-counts test (2/2), and the fused parity suite (3/3), all
green with zero regressions.
