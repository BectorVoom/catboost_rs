# Phase 21.5 Plan 02: Fused feature-parallel histogram in the plain oblivious split search Summary

**`select_level_plain` now FUSES the per-feature histogram accumulate/derive + `O(n_bins)` scan-score into ONE rayon `map_init`-scratch parallel-over-features pass — moving the O(n·nf) accumulation INTO the parallel region and deleting the serial whole-partition build from the per-level hot path — while preserving the per-feature subtraction trick, the feature×border enumeration order, and the strict `>` first-wins tie-break byte-for-byte.**

## Performance

- **Duration:** ~35 min
- **Started:** 2026-07-07
- **Completed:** 2026-07-07
- **Tasks:** 2
- **Files modified:** 1 (`crates/cb-train/src/tree.rs`)

## Accomplishments
- Restructured `GrowScratch` from a whole-partition `hist: BucketHistogram` to per-FEATURE single-feature `feature_hists: Vec<BucketHistogram>` — byte-identical because bins are feature-major so each cell belongs to exactly one feature (D-03).
- Added `derive_feature_level_hist`: the per-feature subtraction-trick derivation (smaller GLOBAL sibling built in its low slots, larger derived `parent − sibling` via `relocate_sub`, reunited via `add_relocated`), reconstructing the parent partition + split outcome from the forward-bit level-L `leaf_of` (`leaf & (n_parent-1)` / `leaf & n_parent`) with no extra threaded state. Bit-for-bit the prior whole-partition `advance` decomposition restricted to one feature (D-06), plus a defensive per-feature fresh-rebuild on parent-shape drift.
- Added `feature_col` (feature-major contiguous bin-column slice) and split `advance` into `advance_leaf_only` (cheap plain-path leaf/level transition) + full `advance` (perturbed-path leaf transition + per-feature PARALLEL derivation).
- Rewrote `select_level_plain` to fuse build(level 0)/derive(level>0) + scan-score per feature in one `into_par_iter().map_init(ScanScoreScratch::new, ...)` pass, reusing the per-task scratch across features (D-07) and returning the level's per-feature histograms; the grow loop retains them and calls `advance_leaf_only`.
- `GrowScratch::new` gained `build_initial_hists` so the plain path skips the redundant up-front level-0 build (now built inside the fused pass) while the perturbed path still materializes level 0.

## Decisions Made
- **Per-feature store over whole-partition hist:** replacing `GrowScratch.hist` with `feature_hists` is the simplest byte-identity-preserving shape (D-06 discretion note) and is what lets the accumulation move inside the parallel-over-features region.
- **`build_bucket_histogram` (allocating) for the smaller sibling, not `build_bucket_histogram_into`:** the subtraction trick needs `BucketHistogram` objects (for `relocate_sub`/`add_relocated`), which cannot be constructed from a raw `Vec` outside `cb-compute` (private fields). `build_bucket_histogram` (which itself delegates to `build_bucket_histogram_into`) is used per feature; D-07 scratch reuse is realized on the dominant SCAN buffers via `ScanScoreScratch` in `map_init` (see Deviations).
- **Perturbed path left in its Task-1 shape:** it scores from the retained `feature_hists` and its full `advance` derives per-feature in parallel — Plan 03 (depends_on 21.5-02) owns fusing it, reusing `derive_feature_level_hist`.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] Smaller sibling built with `build_bucket_histogram`, not `build_bucket_histogram_into`; per-task scratch is `ScanScoreScratch`, not `FusedFeatureScratch`**
- **Found during:** Task 1 / Task 2
- **Issue:** Task 1 suggests deriving the smaller sibling "using `build_bucket_histogram_into` writing into the map_init scratch (D-07)" and Task 2 truth names `FusedFeatureScratch` as the map_init unit. But the per-feature subtraction trick requires `BucketHistogram` objects to call `relocate_sub`/`add_relocated`, and `BucketHistogram`'s fields are module-private to `cb-compute` — so `tree.rs` cannot wrap a `build_bucket_histogram_into` `Vec` into a histogram. Likewise `FusedFeatureScratch`'s only entry, `fused_feature_scan_and_score`, builds-and-discards the histogram internally and never returns it, so it cannot supply the retained parent the per-feature subtraction chain needs. Plan 02's `files_modified` is `tree.rs` only, so the `cb-compute` API is fixed.
- **Fix:** Use `build_bucket_histogram` (which delegates to `build_bucket_histogram_into`, so the single accumulation code path is still exercised) for the single-feature smaller sibling, and reuse the per-rayon-task **`ScanScoreScratch`** (the same Plan 01 buffer-reusing scan+score type that `FusedFeatureScratch` wraps) via `map_init` for the dominant scan buffers (`per_dim` `Vec<Vec<LeafStats>>`, `num`/`den`, `total_*`, `acc_*`, `scores`). The only per-feature allocations left are the single-feature histogram Vecs inherent to `relocate_sub`/`add_relocated` returning owned histograms — small relative to the scan scratch, and the >8-thread allocator-contention dip Spike 006 flagged is on the scan buffers, which ARE reused (D-07 intent satisfied).
- **Files modified:** `crates/cb-train/src/tree.rs`
- **Verification:** byte-identity held — `fused_feature_parity_test` (both cases), `rayon_determinism_test` (both), and 28 tree lib unit tests all green.
- **Committed in:** `8c47241` (Task 1), `400b799` (Task 2)

**2. [Rule 3 - Blocking] `GrowScratch::new` gained a `build_initial_hists` flag**
- **Found during:** Task 2
- **Issue:** With the plain path building level 0 inside its own fused pass AND `new` unconditionally building level-0 per-feature histograms (needed by the still-unfused perturbed path), the plain path would do a redundant full O(n·nf) level-0 build per tree — exactly the serial-fraction cost this phase removes.
- **Fix:** `new` takes `build_initial_hists`; the grow function passes `perturb.is_some()`, so the plain path leaves `feature_hists` empty (built in the fused pass) and the perturbed path materializes level 0. No behavior change, no redundant build.
- **Files modified:** `crates/cb-train/src/tree.rs`
- **Verification:** determinism + parity guards + tree lib tests green.
- **Committed in:** `400b799` (Task 2)
