# Phase 21.5 — EVIDENCE: Fused Feature-Parallel Histogram Recovers CPU Parallel Scaling

**Requirement:** PERF-04 — "byte-for-byte identical, ≥3× at 16 threads."
**Measured:** 2026-07-07 · box cores = **16** (`std::thread::available_parallelism`)
**Harness:** `crates/cb-train/tests/fused_scaling_test.rs` (`CB_PERF`-gated, `--release`)
**Raw log:** `target/fused_scaling_run.log` · greppable `RSBENCH215` records
**Reproduce:** `CB_PERF=1 cargo test -p cb-train --release --test fused_scaling_test -- --nocapture`

---

## 1. Headline result

- **Parity: byte-for-byte identical** — the fused per-level pass equals the pre-fusion
  two-pass path to `f64::to_bits` on the Spike-002 shape (`RSBENCH215 part=per_level_parity
  byte_identical=true features=20`), the full ≤10⁻⁵ CPU oracle suite stays 100% green with
  **NO fixture re-baseline** (§4), and single- vs multi-thread determinism holds (§4).
- **Scaling recovered ≥3× at 16 threads** — the isolated per-level fused pass (the exact
  accumulate+score work `select_level_plain` now performs) hits **3.85× at 16 threads**
  (peaking **4.59× at 8 threads**), versus the pre-fusion two-pass path's **1.97×** ceiling
  on the same shape. **PERF-04's headline claim is met.**

## 2. PRIMARY — isolated per-level pass (hard gate), Spike-002 shape

Shape: `n=10000, nf=20, nbins=128, n_leaves=32, dim=1`, `EScoreFunction::L2`, `l2=3.0`,
median-of-41 inside sized `rayon::ThreadPool`s (warmup first).

- **FUSED** = production `cb_compute::fused_feature_scan_and_score` + `FusedFeatureScratch`
  driven `(0..nf).into_par_iter().map_init(...)` — the accumulate+score work INSIDE the
  parallel region (what Plans 02/03 fused into `tree.rs`).
- **TWO-PASS** = the pre-fusion `tree.rs` hot path: serial whole-partition
  `build_bucket_histogram` (rayon-free accumulation OUTSIDE the parallel region) then
  parallel-per-feature `scan_and_score_borders` — the Amdahl-bound baseline.

| threads | fused ms | **fused speedup** | two-pass ms | two-pass speedup |
|--------:|---------:|------------------:|------------:|-----------------:|
| 1  | 1.0993 | 1.00× | 1.0772 | 1.00× |
| 2  | 0.5293 | 2.08× | 0.7886 | 1.37× |
| 4  | 0.3970 | 2.77× | 0.6394 | 1.68× |
| 8  | 0.2395 | **4.59×** | 0.5615 | 1.92× |
| 16 | 0.2852 | **3.85×** | 0.5470 | 1.97× |

**Hard gate:** `assert!(fused_16t_speedup >= 3.0)` → **3.85 ≥ 3.0 PASS**
(`RSBENCH215_DONE fused_per_level_speedup_16t=3.85`; the test fails loudly otherwise).

**Before → after:** the two-pass path plateaus at **~1.97×** (the serial O(n·nf)
accumulation left outside the parallel region — the ~1.7× ceiling Spike 005/006 measured).
Fusing accumulation INTO the parallel region collapses that serial fraction and recovers
**3.85–4.59×**, tracking Spike 006's isolated ~5.0× @16t. The fused 16t point (3.85×) sits
below the 8t point (4.59×): at `nf=20` with 16 workers, per-feature granularity leaves some
threads idle plus the >8-thread allocator effect — the exact low-nf / >8-thread regime that
is DEFERRED (§5). It remains comfortably above the ≥3× bar.

## 4. Parity preserved — full oracle + determinism suite, NO re-baseline

The existing ≤10⁻⁵ CPU oracle suite is the regression guard (D-04). Re-run in full after the
fusion, with **zero fixture / expected-output edits** (`git status` shows no oracle asset
modified):

- **cb-compute:** 197 lib tests + `fused_feature_parity_test` (2, D-04 byte-identity guard) +
  `depth6_reference_test` (1) + `score_functions_test` (9) + `custom_objective_test` (5) —
  **all green, 0 failed.**
- **cb-train:** all 64 integration/lib test binaries executed one-at-a-time (disk-pressure
  landmine, §6). **63/64 fully green (0 failed)**, including:
  - `rayon_determinism_test` — **2 passed** (D-08: single- vs multi-thread trained models
    byte-identical, symmetric + leaf-wise).
  - `regularization_oracle_test` — 4 passed / 2 ignored (perturbed `random_strength`
    byte-exact vs upstream catboost 1.2.10; the 2 ignored are pre-existing D-11/TRAIN-04
    tree-1+ RNG residuals, unrelated to this phase).
  - `depth6_reference_test`, `leaf_methods_oracle_test`, all loss/ranking/CTR/ordered/
    multiclass/pairwise/yetirank/stochasticrank oracle families — green.
  - `spike005_parallel_scaling_test`, `spike006_fused_parallel_test` — green (their internal
    `byte_identical=true` parity asserts hold against the current fused code).
- **Ignored counts** (bootstrap 1, overfit 3, regularization 2) are documented pre-existing
  D-11 residuals — NOT introduced here.
- **One out-of-scope pre-existing failure:** `monotone_oracle_test::
  monotone_non_symmetric_and_region_are_typed_errors` asserts `grow_policy=Region` is
  *rejected* ("Region OUT", Phase 6.6). Phase 12 BUILT the CPU Region grower, so Region is

  `select_level_perturbed` path — so the fusion cannot affect it. `region_e2e_test` (2) and
  `device_region_fit_test` (1) pass, confirming Region is a supported CPU path. Logged to
  `deferred-items.md`; **not a fusion parity regression.**

**Conclusion:** every shipped ≤10⁻⁵ CPU oracle fixture stays bit-exact; determinism holds;
no re-baseline.

## 5. Explicitly DEFERRED (out of scope, per CONTEXT)

- **Low-nf (`nf < cores`) and >8-thread WITHIN-feature row-block parallelism.** Feature-outer
  parallelism starves when there are fewer features than cores, and >8-thread scaling needs
  finer (within-feature) granularity — visible here as the fused 16t point (3.85×) dipping
  below 8t (4.59×) at `nf=20`. That regime LOSES byte-identity and requires order-independent
  fixed-point-u64 accumulation with parity RE-VERIFIED against the UPSTREAM oracle (not
  current rs). It is a separate future spike/phase — **not** attempted here.
- Extending the fused-parallel restructuring to the non-symmetric growers
  (Depthwise/Lossguide/Region) and the online-CTR-feature scoring path — deferred until after
  the oblivious path lands (this phase).

## 6. Environment note (disk pressure)

The box has the documented root-disk pressure landmine: linking all ~60 cb-train oracle test
binaries at once (~1.24 GB each in debug, ~71 GB total) overflows the disk and the harness
tmpfs (both on `/dev/nvme0n1p7`). The full oracle sweep (§4) was therefore executed by running
each already-linked test binary DIRECTLY, one at a time, deleting each after capture to keep
peak disk bounded — logs in `target/cbtrain_logs/`. This is an environment constraint on HOW
the suite was run, not a parity result; every binary that ran is reported faithfully in §4.
