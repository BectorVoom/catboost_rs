# Phase 21.5: CPU Parallel-Scaling — Fused Feature-Parallel Histogram - Context

**Gathered:** 2026-07-07
**Status:** Ready for planning
**Source:** Spike findings 005 + 006 (root cause proven + fix prototyped, byte-identical). No discuss-phase needed — the spikes ARE the design contract.

<domain>
## Phase Boundary

**What this phase delivers:** the CPU tree-grow per-level split search stops paying the
Amdahl tax that Phase 21 left behind. Phase 21 built the histogram (`BucketHistogram` +
`build_bucket_histogram` + `scan_and_score_borders`) and parallelized only the *scoring*
pass; the O(n·nf) *accumulation* stayed a serial, rayon-free pass (D-03) OUTSIDE the
parallel region. Spike 005 measured `serial_fraction≈0.41` → Amdahl 16-thread ceiling
≈2.2×, and the sole parallel pass (≈`nf` coarse scoring tasks) regresses past 4–8 threads,
so end-to-end 1→16-thread speedup is only ~1.5–1.9× (vs official CatBoost 2–3×).

This phase **fuses accumulate + score into ONE `rayon` parallel-over-features pass**
(upstream `CalcStatsAndScores` shape, `CATBOOST_CORE_DESIGN.md` §3c) so the expensive
accumulation runs inside the parallel region. Spike 006 prototyped exactly this and
measured **5.0× @16t vs the two-pass 1.66×**, **byte-identical** to the current scores.

**In scope:** the oblivious CPU split search — both `select_level_plain` AND
`select_level_perturbed` in `crates/cb-train/src/tree.rs`, plus the `GrowScratch` per-level
histogram wiring and the `cb-compute` histogram/scoring surface they call.

**Out of scope (do NOT touch / defer):**
- Low-nf (`nf < cores`) and >8-thread scaling via WITHIN-feature row-block parallelism —
  that regime loses byte-identity and needs order-independent fixed-point-u64 accumulation
  + upstream (not self) parity re-verification. Deferred to a future spike/phase.
- The non-symmetric growers (Depthwise/Lossguide/Region) and the online-CTR-feature
  scoring path: keep them correct and unregressed, but the fused-parallel restructuring
  targets the oblivious plain/perturbed hot path first (largest, cleanest win). If the
  fused per-feature helper is written reusably, extending later is cheap — but MVP is the
  oblivious path.
- The ordered-boosting path (`score_candidate_ordered`) — unchanged, per Phase 21 scope.
- GPU/`cb-backend` — never add a `cb-train`→`cb-backend` dependency (feature-unification
  landmine); transcribe inline.
</domain>

<decisions>
## Implementation Decisions (LOCKED — from Spikes 005/006)

### Structure — fuse accumulate+score per feature
- D-01: Restructure `select_level_plain` (`tree.rs:813`) so the existing
  `(0..matrix.n_features()).into_par_iter()` task BOTH builds its feature's bucket
  histogram (the O(n) column scan) AND scores its borders — one fused pass. Remove the
  separate serial whole-partition `build_bucket_histogram` call from the per-level hot path
  (currently built into `GrowScratch.hist` / `tree.rs:826` reads a prebuilt shared hist).
- D-02: Apply the same fusion to `select_level_perturbed` (`tree.rs:898`). Its per-feature
  RNG draw order (`from_seed(rand_seed + task_idx).advance(10)`, one `std_normal` per
  border, per-feature `feature_best` bookkeeping) MUST stay byte-for-byte the same draw
  sequence — parallelize accumulation WITHOUT perturbing the documented Pitfall-3 draw
  contract.
- D-03: The fused per-feature accumulation MUST be **feature-outer / object-inner**: each
  task scans its feature's contiguous column `bins[feature*n_objects .. (feature+1)*n_objects]`
  (bins are feature-major) in ascending object order. This keeps every histogram cell's
  members folded in the exact ascending-object-order `sum_f64` order → byte-for-byte
  identical to the current serial `build_bucket_histogram`.

### Parity — byte-identical, no re-baseline
- D-04: Parity is byte-identity, NOT tolerance. The fused per-border scores MUST be
  bit-identical (`f64::to_bits`) to the pre-change serial-build + parallel-score path.
  Spike 005-C and 006 proved this holds (`byte_identical=true`). NO fixed-point-u64, NO
  oracle re-baseline. The existing ≤10⁻⁵ CPU oracle suite is the regression guard and MUST
  stay 100% green (every shipped fixture bit-exact).
- D-05: A RED-first bit-identity guard test is required BEFORE the refactor: assert the new
  fused path's per-feature scores equal the old two-pass path's, cell-by-cell, on ≥2 shapes
  (wide-feature nf=20/nbins=128 AND low-feature nf=8/nbins=254) — mirroring
  `spike006_fused_parallel_test.rs::assert_parity`.

### Subtraction trick + scratch reuse
- D-06: Preserve the subtraction trick (child = parent − sibling). The current
  whole-partition `build_bucket_histogram` + `relocate_sub` (`tree.rs:745-796`,
  `GrowScratch::advance`) must be reconciled with per-feature fusion: build the smaller
  child's per-feature stats and subtract to get the sibling, INSIDE the per-feature task.
  Parallelization is orthogonal to the sub-trick; both compose.
- D-07: Reuse per-task scratch — use rayon `map_init` (or a per-thread scratch pool sized
  `n_leaves * n_bins * n_channels`) so the per-feature histogram buffer is NOT allocated
  inside the `.map` closure. Spike 006 showed in-closure allocation causes the >8-thread
  allocator-contention dip; `map_init` removes it.

### Determinism
- D-08: The `rayon_determinism_test.rs` guarantee MUST hold: single-threaded and
  multi-threaded runs produce byte-identical trained models. The candidate enumeration
  order (feature ascending × border ascending) and strict `>` first-wins tie-break are
  unchanged — collect per-feature results into a feature-ordered `Vec<Vec<_>>` then flatten,
  exactly as the current code does (`tree.rs:876-879`).

### Claude's Discretion
- Exact helper factoring (a shared `fused_feature_scan_and_score(...)` in `cb-compute` vs
  inline in `tree.rs`), the `map_init` scratch struct shape, and whether the fused helper is
  written generalign enough to later serve the non-symmetric/CTR paths. Keep `cb-compute`
  functions the single source of the scan/score math (reuse `scan_and_score_borders` /
  `multi_dim_split_score_into` — do not fork the scoring arithmetic).
- Whether to keep `GrowScratch.hist` for the level-transition subtraction bookkeeping or
  replace it with per-feature scratch entirely — whichever preserves D-06 parity most
  simply.
</decisions>

<specifics>
## Specific Ideas

- The measured target: recover per-level/per-tree 16-thread speedup from the ~1.7× two-pass
  ceiling to ≥3× (Spike 006 isolated ~5.0×). Document the 1→16-thread curve at depth 2/4/6
  on the Spike-002 baseline shape (n=10000, nf=20, nbins=128).
- Feature-major bin layout is the enabling fact: `bins[feature * n_objects + obj]`
  (`histogram.rs:598`) — one feature's column is a contiguous slice, so per-feature
  accumulation is cache-friendly AND disjoint-cell (no cross-thread float reduction).
- Parity assertion mechanism: `f64::to_bits()` cell-by-cell equality, as in
  `spike006_fused_parallel_test.rs::assert_parity` — cheaper and stronger than the ≤1e-5 bar.
</specifics>

<deferred>
## Deferred Ideas

- **Low-nf / within-feature ROW-BLOCK parallelism + fixed-point-u64** — the only regime
  where feature-parallelism starves (nf < cores) and >8-thread scaling needs finer
  granularity. This path LOSES byte-identity and requires order-independent fixed-point
  accumulation (Phase 10/11 GPU winner) with parity re-verified against the UPSTREAM oracle
  (not current rs). Explicitly a separate future spike/phase.
- Extending the fused-parallel restructuring to the non-symmetric growers and the
  online-CTR-feature scoring path — after the oblivious path lands and the helper shape is
  proven.
</deferred>
