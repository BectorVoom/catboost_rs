# Walking Skeleton — catboost-rs

**Phase:** 1
**Generated:** 2026-06-13

## Capability Proven End-to-End

> The smallest end-to-end slice that proves the whole parity machine works.

The workspace builds and is lint-clean; `cb-oracle` reads ONE committed `.npy` fixture and runs ONE real per-stage oracle comparison that passes at absolute error <= 1e-5; the clippy restriction-lint gate and the `anyhow` ban are live; and a CPU-only GitHub Actions lane runs build + clippy + tests + the anyhow grep green. (Library translation of "walking skeleton end-to-end" — there is no UI/HTTP/DB layer in this project; the parity pipeline IS the stack.)

## Architectural Decisions

| Decision | Choice | Rationale |
|---|---|---|
| Workspace shape | Fine-grained Cargo workspace, all 8 crates stubbed day one: `cb-core`, `cb-data`, `cb-compute`, `cb-backend`, `cb-train`, `cb-model`, `cb-oracle`, `catboost-rs` | D-01 — fix crate boundaries early, never refactor them later |
| Crate naming | internal crates `cb-` prefix; public facade `catboost-rs` | D-04 |
| Backend strategy | exactly ONE feature-gated `cb-backend` crate resolving to a single `cfg`-gated `SelectedRuntime` type alias (`cpu`/`wgpu`/`cuda`/`rocm`); NO per-backend crates; NO runtime dispatch | D-02 |
| Generic seam | `cb-compute` is pure-generic (`R: Runtime`/`F: Float`) and MUST NOT depend on CubeCL; all CubeCL coupling lives in `cb-backend` | D-03 |
| Error strategy | `thiserror`-derived typed errors in all library crates (`CbError`, `OracleError`); `anyhow` only as dev-dependency / at edges; structural ban + grep backstop | D-14, D-15; CLAUDE.md |
| Lint policy | centralized `[workspace.lints.clippy]` deny table (`unwrap_used`/`expect_used`/`panic`/`indexing_slicing`); each lib opts in `[lints] workspace = true`; test code exempted via in-code `#![cfg_attr(test, allow(...))]` (manifest overrides forbidden) | D-13; RESEARCH Pitfall 1 |
| CI lane split | Phase 1 ships ONLY `.github/workflows/ci.yml` (CPU lane): build + `cargo clippy --lib -D warnings` + cargo test + anyhow grep + source/test grep. NO GPU/ROCm job (no AMD runners on GH Actions); ROCm is standalone scripts wired in Phase 7 | D-16 |
| Oracle method | Python-first: pinned `catboost==1.2.10` generator (`get_borders`/`save_model(json)`/`staged_predict`/`predict`), `thread_count=1`, fixed seed; build-time only, NEVER in CI | D-06, D-07, D-12 |
| Fixture format | hybrid `config.json` (seed/version/params) + `.npy` f64 arrays; Rust reads via `ndarray-npy`, Python writes via numpy; bit-exact f64 round-trip | D-09 |
| Fixture layout | `cb-oracle/fixtures/inputs/<dataset>/` (frozen shared INPUT corpus) + `cb-oracle/fixtures/<scenario>/` (per-phase expected OUTPUT) | D-10, D-11 |
| Parity tolerance | absolute error <= 1e-5, single audited comparator primitive (`assert_abs_close` / `compare_stage`) | D-12 |
| PRNG | exact `TFastRng64` port in `cb-core` (two PCG-XSH-RR 32-bit LCGs concatenated to 64 bits), `wrapping_*` arithmetic, validated against vendored `fast_ut.cpp` vectors; no C++ build | INFRA-05; RESEARCH Pitfall 4/5 |
| Source/test separation | dedicated `*_test.rs` files; no inline `#[cfg(test)]` test bodies in production modules; script-enforced | D-17 / INFRA-06 |
| Toolchain | latest stable Rust pinned via `rust-toolchain.toml`; latest crate versions pinned in `[workspace.dependencies]` | CLAUDE.md |

## Out of Scope (Deferred to Later Slices)

> Explicit minimalism — prevents later phases re-litigating Phase 1.

- Any CatBoost algorithm (quantization, training, trees, CTR, losses) — Phases 2–6.
- Real runtime behind `SelectedRuntime` / any CubeCL kernel — `cb-backend`/`cb-compute` are inert stubs this phase (P3 generic seam, P7 GPU).
- C++ instrumentation of `catboost-master` for training-internal oracle values — deferred to the phase that needs it (primarily P5). Phase 1 uses only the Python API (D-08).
- ROCm/GPU CI job and self-hosted AMD runner provisioning — Phase 7 (scripts defined now, automation wired later) (D-16).
- Pool/QuantizedPool ingestion, NumPy zero-copy, Arrow/Polars — Phase 2 (DATA-*).
- `.cbm` serialization, SHAP, Builder API, Python bindings — Phases 4 and 8.
- Facade-as-edge anyhow reclassification — revisit at Phase 4 if a real ergonomics need surfaces (default: `catboost-rs` is a thiserror-only library, D-15).

## Subsequent Slice Plan

Each later phase adds one oracle-passing vertical slice on top of this skeleton WITHOUT altering the architectural decisions above (crate boundaries, lint policy, fixture format/layout, comparator API, PRNG):

- Phase 2: Pool / QuantizedPool + `GreedyLogSum` borders + audited reduction — oracle-locked against the frozen input corpus.
- Phase 3: generic `R: Runtime`/`F: Float` seam realized in `cb-compute` + CPU plain boosting + oblivious trees (per-tree split/leaf oracle).
- Phase 4: `.cbm` serialize/apply + SHAP + Builder API — first full train→serialize→predict oracle lock.
- Phase 5: ordered boosting + ordered CTR + categoricals (high-risk parity slice; per-object intermediate oracle).
- Phase 6: full loss/metric + advanced feature parity, additively.
- Phase 7: GPU backends via CubeCL on the locked generic boundary (`cb-backend` realized), documented GPU tolerance.
- Phase 8: PyO3 dual sklearn + CatBoost-native API + per-backend wheels.
