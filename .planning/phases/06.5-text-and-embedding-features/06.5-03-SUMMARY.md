## Decisions Made

- **Newton leaf method for Logloss.** `params.json` leaves `leaf_estimation_method` unset; catboost's default for Logloss is **Newton** (confirmed by retraining: `get_all_params()['leaf_estimation_method'] == 'Newton'`). Newton's denominator is the summed hessian (`-sum_der2 = 2.0`), giving the 0.24 first-tree leaf value; the Gradient method's object-count denominator (8) would give 0.109. The oracle pins `LeafMethod::Newton`.
- **Tree-representation canonicalization (the structural reconciliation).** The fixture trains `depth=2` but catboost STORES depth-1 trees (`_get_tree_splits` = 1 split/tree, `get_leaf_values` = 2 leaves/tree). A perfectly-separating BoW feature (multiple exist: "bad"/"great"/"awful"/"good" each split the 8/8 classes cleanly) makes the first split's leaves pure, so the depth-2 second split adds no partition and upstream serializes the collapsed form. The Rust trainer keeps the full-depth representation (it re-selects the same feature, producing 2 empty leaves) but is FUNCTIONALLY identical — the structure-invariant `StagedApprox`/`Predictions` stages match ≤1e-5 directly. The oracle canonicalizes the model to upstream's stored distinct-split form for the `Splits`/`LeafValues` stages: dedup splits per tree, reassign leaves over the distinct splits, read each canonical leaf's value from a representative object. This is a lossless representation change, not a re-fit; no tolerance weakened.
- **BiGram dictionary owned here.** Per the 06.5-02 summary, the BiGram (`gram_order=2`) dictionary was deferred from SC-1 to the BoW calcer slice. It reproduces the frozen BoW `ttext` (ids 0..24) bit-exact — validated against the `fixtures/text_tokenizer/ttext.json` BiGram block (doc0 → `[1,18]`, doc10 → `[16,17,22]`).
- **cb-oracle dev-dependency cycle.** The BoW oracle trains end-to-end, so cb-oracle dev-depends on cb-train/cb-compute/cb-backend. cb-train already dev-depends on cb-oracle; Cargo permits dev-dependency cycles (the cb-oracle library itself depends on none of them, mirroring the 06.5-02 cb-data dev-dep).
