## Summary

Phase 6.5 implements the **text & embedding feature-encoding layer**: a tokenizer + frequency-based dictionary (the load-bearing D-01 first gate), three text calcers (BoW, NaiveBayes, BM25), and two embedding calcers (LDA, KNN). All calcer outputs are **numeric float feature columns** that join the existing feature set and flow into the unchanged `cb-data` quantization → `cb-train` tree path. The four target-aware calcers (NaiveBayes, BM25, LDA, KNN) are `IOnlineFeatureEstimator`s computed **per-fold over the `TFold` learn permutation** with the exact read-before-update prefix discipline that Phase-5 CTRs already use — `cb-train/src/ctr/online.rs` is the precise pattern to mirror.

The tokenizer/dictionary algorithm is fully transcribable with HIGH confidence: `ByDelimiter` splits on the delimiter string (default `" "`, `SkipEmpty=true`), optional lowercasing; the dictionary counts tokens, filters by `OccurrenceLowerBound`, sorts by **`(count DESC, token ASC)`** — a fully deterministic tie-break that makes token→id assignment reproducible regardless of hash-map iteration order — truncates to `MaxDictionarySize`, and assigns ids `StartTokenId, StartTokenId+1, …`. `TText` is then a `(tokenId, count)` list **sorted ascending by tokenId** (run-length encoded). Defaults to pin: rotten_tomatoes train = exactly 1000 rows → `OccurrenceLowerBound = 5` (not 1, not the lib default 50), `MaxDictionarySize = 50000`, unknown tokens **dropped** (`EUnknownTokenPolicy::Skip`).

The two genuine parity risks are **embedding-side**: LDA solves a generalized symmetric eigenproblem via LAPACK `ssygst_`/`ssyev_` in **single precision (`float`)** — the upstream LDA fixture itself rounds errors to 2 decimals before canonicalization, signaling LDA is not bit-stable across BLAS/LAPACK implementations. KNN uses an **HNSW approximate nearest-neighbor index** (`library/cpp/online_hnsw`) whose graph construction is order- and tie-break-dependent. Both require careful escalation (instrumented GT) and may need a documented tolerance discussion if exact `f32` LAPACK/HNSW reproduction proves infeasible.

**Primary recommendation:** Slice strictly tokenizer-first (D-01). Land the tokenizer+dictionary as one oracle-gated gate validated against an instrumented token-stream/dictionary-id dump (D-07), then BoW (target-independent, simplest), then NaiveBayes+BM25 (share the online text-estimation seam), then LDA, then KNN (highest risk, last). Mirror `cb-train/src/ctr/online.rs` verbatim for the per-fold prefix loop.

### Pattern 1: Online (ordered) estimation = Compute-then-Update prefix
**What:** For each line in the learn permutation, compute the calcer's output using ONLY the state accumulated from *earlier* lines, THEN update state with this line's label/text. This prefix discipline is the no-leakage property.
**When to use:** NaiveBayes, BM25, LDA, KNN (all four target-aware calcers). NOT BoW.
**Example (verbatim upstream loop, the exact structure to mirror):**
```cpp
// Source: catboost/private/libs/feature_estimator/base_text_feature_estimator.h:59-64
for (ui64 line : learnPermutation) {
    const TText& text = ds.GetText(line);
    Compute(featureCalcer, text, line, samplesCount, learnFeatures); // READ prefix state
    calcerVisitor.Update(target.Classes[line], text, &featureCalcer);  // THEN write
}
// learnFeatures laid out column-major: learnFeatures[f * samplesCount + line]
```
The Rust analog already exists for CTRs — `cb-train/src/ctr/online.rs:298` ("READ the prefix counts BEFORE incrementing"). Reuse that exact ordering and the same `TFold` learn permutation (`fold.cpp:181 ff.GetLearnPermutationArray()`).

### Pattern 2: `TText` = sorted run-length-encoded (tokenId → count)
**What:** A digitized text is a `Vec<(tokenId: u32, count: u32)>` **sorted ascending by tokenId**, built by sorting the raw token-id vector then RLE-collapsing duplicates.
**Why it matters:** BoW's `Compute` walks `TText` and the active feature set *in lockstep assuming both are sorted ascending* — order is load-bearing.
**Example:**
```cpp
// Source: catboost/private/libs/data_types/text.h:155-164
TText(TVector<ui32>&& tokenIds) {
    Sort(tokenIds);
    for (const auto& tokenId : tokenIds) {
        if (TokenToCount.empty() || TokenToCount.back().Token() != tokenId)
            TokenToCount.push_back({TTokenId(tokenId), 1});
        else
            TokenToCount.back().IncreaseCount();
    }
}
```

### Pattern 3: Deterministic dictionary token→id assignment
**What:** Counts collected in a hash map (order-nondeterministic) are made deterministic by sorting indices on `(count DESC, token-string ASC)`, then ids assigned `StartTokenId++` in that order. Hash-map iteration order is therefore irrelevant.
**Why it matters:** This is what makes the dictionary reproducible in Rust despite a different hash map. Replicate the comparator EXACTLY.
```cpp
// Source: library/cpp/text_processing/dictionary/dictionary_builder.cpp:146-162
Sort(indices, [&](ui32 l, ui32 r){
    return counts[l] > counts[r] || (counts[l]==counts[r] && tokens[l] < tokens[r]);
});
// filter first: skip tokens with value < OccurrenceLowerBound (strict <)
// truncate to min(size, MaxDictionarySize); assign globalTokenId = StartTokenId++ in order
```

### Anti-Patterns to Avoid
- **Using a different HNSW / ANN library for KNN:** any non-identical graph → different neighbors → parity fail. Hand-port upstream HNSW or escalate.
- **f64 everywhere in LDA:** upstream LDA is **f32** through BLAS/LAPACK. Blindly using f64 changes results; but blindly using f32 raw `+=` violates D-04. The summation-order tension here needs explicit instrumented validation, not assumption.
- **Recomputing the dictionary per fold:** the dictionary is built once from learn texts (offline, target-independent); only the *online calcer state* (class freq tables) is per-fold/prefix. Don't conflate.
- **Assuming OccurrenceLowerBound = 1 or 50:** it is data-dependent (1 if learnPoolSize<1000 else 5); rotten_tomatoes = 1000 rows → **5**.
- **Inline `#[cfg(test)] mod tests`:** prohibited (INFRA-06). Separate `_test.rs` files.

## Common Pitfalls

### Pitfall 1: LDA single-precision LAPACK eigensolver is not bit-stable (PRIMARY FEAT-02 risk)
**What goes wrong:** LDA's `CalculateProjection` calls `ssygst_` (reduce generalized→standard eigenproblem) then `ssyev_` (symmetric eigenvectors) in **`float`** precision, then takes the trailing rows of the eigenvector matrix as the projection `[VERIFIED: lda.cpp:10-37]`. Eigenvector sign, ordering of degenerate eigenvalues, and low-bit values differ across LAPACK builds and especially between f32 and any f64 reimplementation.
**Why it happens:** Symmetric eigensolvers have sign/ordering freedom; f32 amplifies divergence; `Exp2f(-M_LN2_INV * 0.5 * deg)` likelihood and the incremental scatter (`IncrementalCloud::Update`, f32 GEMM with a Welford-like shift) compound it.
**How to avoid:** (1) Instrument the upstream trainer (D-07) to dump the projection matrix and intermediate scatter matrices, compare directly. (2) Try to reproduce f32 `ssyev` exactly (LAPACK binding or hand-port) BEFORE accepting f64. (3) If exact f32 reproduction is infeasible, ESCALATE for a documented LDA-specific tolerance decision — do NOT silently weaken (note: the upstream LDA *fixture* itself rounds to 2 decimals `[VERIFIED: test.py:10512-10514]`, strong evidence exact ≤1e-5 may be upstream-impossible for the learn-error metric; per-stage encoding parity must be judged on the *projected feature values*, instrumented).
**Warning signs:** projection matrix differs in sign on whole columns; eigenvalue ties reorder; per-document projected feature off by >1e-5 only on some rows.

### Pitfall 2: KNN HNSW approximate index is order/tie-break dependent (PRIMARY FEAT-02 risk)
**What goes wrong:** KNN builds an **online HNSW graph** (`TOnlineHnswDenseVectorIndex<float, TL2SqrDistance>`, build opts `{CloseNum, 300}`) incrementally as each learn vector is added, then queries k-NN (default k=5; fixtures use k=10/20) `[VERIFIED: knn.h:17,94-111; knn.cpp:30-68]`. HNSW is *approximate* and its graph depends on insertion order, distance ties, and internal heuristics.
**Why it happens:** The neighbor set for a query is not guaranteed to be the true k nearest; it depends on graph structure built during the online prefix (insertion order = learn permutation order). A different HNSW implementation returns different neighbors → different class-vote counts → fails ≤1e-5.
**How to avoid:** (1) Insertion order = the per-fold learn permutation (already required by online estimation — good). (2) Hand-port the EXACT upstream `library/cpp/online_hnsw` build+query, or instrument the trainer to dump the per-query neighbor id lists (D-07) and validate against those. (3) Check whether small fixture sizes make HNSW degenerate to exact (if N ≤ index ef/M params, neighbors may be exact) — if so a brute-force exact L2 k-NN might match; verify with instrumented neighbor dumps before relying on it.
**Warning signs:** neighbor id lists differ from instrumented GT for any query; class-vote feature off by an integer/k.

### Pitfall 3: Wrong dictionary defaults break the entire token→id mapping
**What goes wrong:** Using the *library* default `OccurrenceLowerBound = 50` (`options.h`) instead of catboost's data-dependent default produces an empty/wrong dictionary.
**Why it happens:** There are THREE layers of defaults: library (`50`/`-1`), catboost `DEFAULT_DICTIONARY_BUILDER_OPTIONS` (`3`/`50000`), and the data-dependent override `SetDefaultMinTokenOccurrence` (`1` if learnPoolSize<1000 else `5`) `[VERIFIED: text_processing_options.h:46-49; options_helper.cpp:394-401]`. The last one wins for default dictionaries.
**How to avoid:** Pin `MaxDictionarySize = 50000`, `OccurrenceLowerBound = (learnPoolSize < 1000 ? 1 : 5)`. rotten_tomatoes = 1000 rows → **5**. Filter is **strict `<`** (token kept iff `count >= OccurrenceLowerBound`) `[VERIFIED: dictionary_builder.cpp:135]`.

### Pitfall 4: BiGram dictionary + multi-dictionary feature wiring
**What goes wrong:** The default classification text processing applies BoW over **two** dictionaries (BiGram `gram_order=2` + Word unigram) and NaiveBayes over the Word dictionary only `[VERIFIED: text_processing_options.cpp:184-202; test.py:9933-9934]`. Forgetting the bigram path or mis-ordering dictionaries changes feature count and indices.
**Why it happens:** Fixtures parametrize `BoW`, `NaiveBayes`, `BM25` and combos; BoW's `dicts['BoW']=['Bigram','Word']`. Bigram dictionary build uses `TMultigramDictionaryBuilderImpl` (n-gram keys, `CompareNGram` tie-break) `[VERIFIED: dictionary_builder.cpp:229-291]`.
**How to avoid:** Implement unigram first (covers NaiveBayes/BM25 + Word-BoW); add bigram for full BoW fixture coverage. EndOfSentence policy = Skip (default) `[VERIFIED: text_processing_options.h:43]`.

### Pitfall 5: NaiveBayes / BM25 feature dimensionality vs numClasses
**What goes wrong:** NaiveBayes `BaseFeatureCount(numClasses) = numClasses>2 ? numClasses : 1` (binary → **1 feature**, multiclass → numClasses) `[VERIFIED: naive_bayesian.h:38-40]`. BM25 `BaseFeatureCount = numClasses` always `[VERIFIED: bm25.h:38-40]`. Getting the output width wrong shifts every downstream feature index.
**How to avoid:** Replicate `BaseFeatureCount` exactly per calcer; `ForEachActiveFeature` controls which of these are emitted after `TrimFeatures`.

### Pitfall 6: Float vs double in calcer accumulators
**What goes wrong:** NaiveBayes/BM25 accumulate in `double` (LogProb, scores) `[VERIFIED: naive_bayesian.cpp:14-44; bm25.cpp:33-83]` but the calcer OUTPUT is written through a `float` iterator (estimated features are `f32`). BoW outputs 0/1. The dictionary counts are `ui32`/`ui64`.
**How to avoid:** Match the exact type at each step: `double` math internally, store as `f32` feature, `sum_f64` for the reductions (textLen, scores, norms). `Softmax` operates on `double` then is downcast `[VERIFIED: helpers.h:33-47]`.

### BoW — target-independent, binary presence over active token ids
```cpp
// Source: catboost/private/libs/text_features/bow.cpp:7-21
// text is sorted ascending by tokenId; active feature ids ascending; walk in lockstep
void TBagOfWordsCalcer::Compute(const TText& text, TOutputFloatIterator out) const {
    auto it = text.begin();
    for (ui32 activeFeatureId : GetActiveFeatureIndices()) {
        while (it != text.end() && (ui32)it->Token() < activeFeatureId) ++it;
        *out = (it == text.end() || (ui32)it->Token() > activeFeatureId) ? 0 : 1;
        ++out;
    }
}
// FeatureCount = NumTokens (dictionary size); each token id is one binary feature.
```

### NaiveBayes — multinomial log-prob with priors 0.5, then softmax
```cpp
// Source: catboost/private/libs/text_features/naive_bayesian.cpp:14-63
// ClassPrior = TokenPrior = DEFAULT_PRIOR = 0.5; SEEN_TOKENS_PRIOR = 1
double LogProb(freqTable, classSamples, classTokensCount, text):
    value = log(classSamples + ClassPrior)
    classTokensCount += TokenPrior * (NumSeenTokens + SEEN_TOKENS_PRIOR)
    textLen = 0
    for (token,count) in text:
        textLen += count
        num = TokenPrior + (freqTable[token] if present else 0)
        if token not in freqTable: classTokensCount += TokenPrior  // unseen-word prior adjust
        value += count * log(num)
    value -= textLen * log(classTokensCount)
    return value
// Compute: logProbs[c] for each class → Softmax → emit active features
// Online Update: classCounts[token] += count; ClassTotalTokens[class] += count;
//                ClassDocs[class] += 1; NumSeenTokens = |SeenTokens set|
```

### BM25 — class-as-document IDF + saturation (k=1.5, b=0.75, truncate=1e-3)

// Source: catboost/private/libs/text_features/bm25.cpp:12-83
InvClassFreq(numClasses, classesWithTerm, eps) =
    max(log((numClasses - classesWithTerm + 0.5)/(classesWithTerm + 0.5)), eps)
Score(tf,k,b,meanLen,classLen) = tf==0 ? 0 :
    tf*(k+1)/(tf + k*(1 - b + b*meanLen/classLen))
// Compute: meanClassLength = TotalTokens/NumClasses
//   for each (token,count) in text:
