# Phase 06.5: Code Review Report

**Reviewed:** 2026-06-18
**Depth:** standard
**Files Reviewed:** 39
**Status:** issues_found

## Summary

This phase adds the text/embedding feature pipeline: tokenizer → dictionary →
digitizer → `TText` (cb-data), the pure calcer math (BoW / NaiveBayes / BM25 /
LDA / KNN in cb-compute), the LDA hand-rolled f32 linear algebra, and the
cb-train online/offline estimation seams that join estimated columns to the
existing quantizer.

The production code is unusually disciplined on the no-panic / checked-access
front (CLAUDE.md library discipline is honored throughout — no `unwrap`/`expect`
/raw index in production paths, only in `#[cfg(test)]`-gated test files via the
documented in-code lint exemption). Source/test separation is respected (all
tests in sibling `_test.rs` files referenced from `lib.rs`/`mod.rs`).

The defects below are concentrated in (a) public-API numeric robustness gaps
where division-by-zero / NaN can escape silently, (b) a parity-relevant class
cast that diverges from upstream `(ui32)target` semantics on out-of-domain
labels, and (c) brittleness in the offline run-once fixture generator tooling.
One BLOCKER: an unguarded divide-by-`size` in the public LDA scatter assembly
that produces NaN/Inf weights silently and propagates into the eigensolve.

## Narrative Findings (AI reviewer)

## Critical Issues

### CR-01: Unguarded divide-by-`size` in public `total_scatter` / `between_matrix` produces silent NaN/Inf

**File:** `crates/cb-compute/src/embedding_calcers.rs:224` and `:289`
**Issue:** Both functions compute `let weight = cloud.total_size() / size;` with
no guard on `size == 0.0`. `total_scatter` and `between_matrix` are `pub` (and
re-exported from `cb-compute::lib.rs:36-39`), so they are reachable with
`size = 0.0` independently of the `offline_lda_features`/`online_lda_prefix`
callers that happen to error on empty input first. With `size == 0.0` every
`weight` is `inf` or `NaN`, the assembled matrices become non-finite, the
Cholesky `sum <= 0.0` SPD check in `cholesky_lower` does NOT catch `NaN` (`NaN
<= 0.0` is `false`), so the non-SPD guard is bypassed and NaN silently flows
through `reduce_generalized` → `jacobi_symmetric_eig` into the projection. The
result is a silently-NaN feature column rather than a typed error — a data-
integrity failure for any caller using these primitives directly (e.g. a future
multiclass LDA path that fits clouds before establishing `size`).
**Fix:**
```rust
// In total_scatter and between_matrix, before the weight loop:
if !(size > 0.0) {  // catches 0.0, -0.0, and NaN
    return Err(CbError::Degenerate(format!(
        "total_scatter: non-positive size {size}"
    )));
}
```
Additionally, harden `cholesky_lower` so the SPD check rejects non-finite
pivots: `if !(sum > 0.0) { return Err(...) }` instead of `if sum <= 0.0`.

## Warnings

### WR-01: `KnnCalcer::update` class cast diverges from upstream `(ui32)target` on negative / fractional labels

**File:** `crates/cb-compute/src/embedding_calcers.rs:643-647`
**Issue:** The code maps the target to a class via
`if target.is_finite() && target >= 0.0 { target as usize } else { 0 }`. The
upstream comment claims this mirrors `(ui32)target`. It does not: upstream
`(ui32)(-1.0)` and `(ui32)NaN` are implementation-defined / wrap, whereas this
silently buckets every out-of-domain label into class 0. For the frozen binary
fixtures (labels 0/1) this is invisible, but it is a latent parity divergence
for any non-integer or negative class encoding, and it silently corrupts the
per-class vote rather than surfacing a malformed-label error.
**Fix:** Either validate the label explicitly
(`if !(target.is_finite() && target >= 0.0 && target.fract() == 0.0) { return
Err(CbError::OutOfRange(...)) }`) or document that labels are pre-binarized into
`[0, num_classes)` by the caller and assert that invariant, rather than silently
folding bad labels into class 0.

### WR-02: KNN class index clamped into range, masking parallel-array desync

**File:** `crates/cb-compute/src/embedding_calcers.rs:675`
**Issue:** `result.get_mut(class.min(self.feature_count.saturating_sub(1)))`
clamps a recorded class id to the last feature slot. If `target_classes` ever
holds a class `>= feature_count` (e.g. `num_classes` was under-specified at
`new`), the vote is silently attributed to the wrong (last) class instead of
erroring. The sibling regression arm and the `target_classes.get(id)` lookup
already error on desync; this clamp is the one place that swallows an
out-of-range class.
**Fix:** Replace the clamp with a checked branch that errors on
`class >= self.feature_count`, mirroring the `ok_or_else` pattern used two lines
above for the missing-id case.

### WR-03: `offline_text_features` does not validate `num_classes` against the calcer width / empty input

**File:** `crates/cb-train/src/estimated/online_text.rs:76-93`
**Issue:** Unlike `build_online_text_estimated_features` (which is the inert-on-
empty entry point), `offline_text_features` computes `width` from
`NaiveBayesState::new(num_classes).feature_count()` and allocates
`columns = vec![vec![0.0; n]; width]` with no `num_classes == 0` guard. With
`num_classes == 0`, `NaiveBayesState::feature_count()` returns 1 (the binary
branch), so the function allocates a width-1 column set, then `naive_bayes_compute`
correctly errors with `Degenerate("zero classes")` — but only on the first
compute, after allocation, and only if `texts` is non-empty. With empty `texts`
it returns a width-1 all-zero column with zero rows and NO error, which is a
silently-wrong shape for a zero-class request.
**Fix:** Add an explicit `if num_classes == 0 { return Err(CbError::Degenerate(
"offline_text_features: zero classes".into())) }` at the top, consistent with
the `bm25_compute`/`naive_bayes_compute` precondition.

### WR-04: `softmax_in_place` silently no-ops on all-`-inf` / NaN input instead of signaling

**File:** `crates/cb-compute/src/text_calcers.rs:285-301`
**Issue:** If every value is `-inf` (or the slice is empty), `max_value` stays
`NEG_INFINITY`, the `!max_value.is_finite()` guard returns early leaving `vals`
unchanged (un-normalized). If a value is `NaN`, the `v > max_value` comparison
is always false, so `max_value` may remain `-inf` and the same early-return
leaves NaN in place. The downstream `naive_bayes_compute` then emits raw
(un-softmaxed) log-probs or NaN as a "probability" feature with no error. For
the well-conditioned fixtures this never triggers, but a degenerate prefix
(e.g. a class with zero accumulated tokens producing `log(0)` = `-inf`) could
reach it.
**Fix:** Treat a non-finite max as an error (`CbError::Degenerate`) or
explicitly handle the all-`-inf` case by emitting a uniform distribution, rather
than silently returning the input unchanged. At minimum, document that the early
return is a deliberate no-op and why it cannot poison the fixtures.

### WR-05: `digitize_document_bigram` tokenizes but never lowercases-validates dictionary key consistency across dictionaries

**File:** `crates/cb-data/src/text/digitizer.rs:67-75`
**Issue:** `digitize_document_bigram` re-tokenizes with the passed
`tokenizer_options`, but the BiGram dictionary it applies was built (in
`build_bow_estimated_features`, `estimated_features.rs:112`) from a SEPARATE
`tokenize(t, tokenizer_options)` pass over the same texts. There is no
compile-time or runtime guarantee that the `tokenizer_options` used to build the
dictionary equals the one used to digitize. If a caller passes mismatched
options (e.g. dictionary built with lowercasing, digitize without), every bigram
silently misses and the BoW block becomes all-zero with no error — a
hard-to-diagnose parity failure.
**Fix:** This is an API-shape risk. Either carry the `TokenizerOptions` inside
the `Dictionary`/`BigramDictionary` so `apply`/`digitize` cannot diverge, or
document the invariant prominently and have `build_bow_estimated_features` be the
single source of both passes (it already is — so promote that to the only public
entry and mark the lower-level digitize functions `#[doc(hidden)]` or note the
coupling).

### WR-06: Brittle two-pass perl `calcer_encoding` hook can mutate the source before the awk fallback

**File:** `crates/cb-oracle/generator/build_instrumented_trainer.sh:416-425`
**Issue:** The first `perl -pi -e` block uses a `$cb_prev` previous-line trick
that is acknowledged as "brittle in -pi". Critically `$cb_prev` is never
assigned the actual previous line (`$cb_prev = $_ ne "" ? "" : $cb_prev;` only
ever sets it to `""` or leaves it), so the regex `$cb_prev =~ /featureCalcer\.Compute/`
can never match — the perl pass is effectively dead, yet it still rewrites the
file in place (`-pi`) on every invocation. The subsequent `grep -q` + awk
fallback then does the real work. Running a dead in-place rewrite over a C++
source before the working insertion is fragile: any future edit to the perl body
that *does* match would double-insert (the idempotency `grep -q 'cb_instr_calcer_encoding'`
guard wraps both, but the inner perl is inside that guard and runs first).
**Fix:** Delete the dead perl block entirely and keep only the deterministic awk
insertion (lines 428-437), which is already the documented working path. This is
offline run-once tooling, but a corrupted instrumented build silently produces
wrong fixtures.

### WR-07: `naive_bayes_log_prob` accumulates `class_tokens_count` in a non-sum_f64 running scalar that depends on token-iteration order

**File:** `crates/cb-compute/src/text_calcers.rs:250-276`
**Issue:** The module's parity contract (lib doc + this file's header) is that
every parity-critical float SUM routes through `cb_core::sum_f64`. The
`class_tokens_count` accumulation (`+= state.token_prior` per unseen token,
line 267) is a float running sum performed in `HashMap`/`TText` iteration order
outside `sum_f64`. The code comments justify this as "a running scalar matching
the upstream single accumulator," which is plausible because the increments are
all the identical constant `token_prior` (so order does not change the f64
result). This is correct *today* only because every increment is the same value;
it is a latent fragility — if `token_prior` ever became per-token, the result
would become order-dependent and silently diverge from `sum_f64` discipline.
**Fix:** Either route the unseen-token additions through `sum_f64`
(`token_prior * (unseen_count as f64)` computed from a counted set, which is
exact and order-free) or add an explicit assertion/comment that the constant-
increment property is what makes this safe and a test that pins it.

### WR-08: `build_dictionary` / `build_bigram_dictionary` comparator compares `Option` values, masking an out-of-bounds index as `Ordering::Equal`

**File:** `crates/cb-data/src/text/dictionary.rs:234-235` and
`crates/cb-data/src/text/bigram_dictionary.rs:182-183`
**Issue:** The sort comparator is
`match counts.get(r).cmp(&counts.get(l)) { Equal => tokens.get(l).cmp(&tokens.get(r)), ... }`.
Comparing `Option<&u64>` is correct only because `l`/`r` are guaranteed
in-bounds (built from `0..dictionary_size`). If that invariant ever broke, an
out-of-bounds `.get` returns `None`, and `None.cmp(&Some(_))` yields `Less`
silently — corrupting the deterministic `(count DESC, token ASC)` order that
SC-1 bit-exactness depends on, with no panic and no error. The checked-access
style is good, but here it converts a would-be-panic logic bug into a silent
ordering corruption.
**Fix:** Index directly (the indices are provably in-bounds: `0..size`) so a
future bug panics in debug instead of silently mis-sorting, or compare the
unwrapped values via `(*counts.get(r)?, ...)` with an explicit fallback that is
clearly unreachable. At minimum add a debug_assert that both gets are `Some`.

## Info

### IN-01: `LDA_DEFAULT_REG` constant is unused by the fixtures and never referenced in production

**File:** `crates/cb-compute/src/embedding_calcers.rs:54`
**Issue:** `LDA_DEFAULT_REG = 0.00005` is exported but the comment states the
fixtures pin `reg=0.05` and the value is always passed as a constructor argument.
It is re-exported from `lib.rs:38` but has no production caller — dead-ish public
surface that can mislead (a caller might assume it is the effective default).
**Fix:** Keep it (it documents the upstream default) but add `#[allow(dead_code)]`
context or a doc note that nothing in-tree uses it, or wire it as the actual
default in a builder so the constant is load-bearing.

### IN-02: Eigensolver divergence vs upstream CLAPACK is documented but ships as a silent ~3.9e-2 raw-projection gap

**File:** `crates/cb-compute/src/lda_linalg.rs:13-20` and module header
**Issue:** The LDA eigensolve is reference-faithful but diverges 4.9e-2 from
upstream's vendored CLAPACK `ssyev_`, yielding a ~3.9e-2 raw-projection
divergence that exceeds the ≤1e-5 parity bar. The justification (it does not
cross a split border, so the per-stage model oracle is byte-identical) is sound
for the frozen fixtures but is a corpus-specific argument. This is a documented,
deliberate tolerance escalation, not a bug — flagged for visibility because the
≤1e-5 parity claim has a calcer-level exception here that future LDA fixtures
(different clouds / non-binary) could expose.
**Fix:** No code change required; ensure the LDA tolerance carve-out is tracked
in the phase deferred-items / parity-exceptions ledger so a future corpus change
re-triggers the analysis.

### IN-03: Two narrow `Option` states for the online-text calcer is a state-validity smell

**File:** `crates/cb-train/src/estimated/online_text.rs:214-221`
**Issue:** `online_text_prefix` carries `nb_state: Option<..>` and
`bm_state: Option<..>` with the invariant "exactly one is `Some`", enforced only
by construction and re-checked via `ok_or_else` on every iteration. An enum
holding the active state would make the invalid (both-None / both-Some) states
unrepresentable and remove the per-iteration `Degenerate("missing ... state")`
guards.
**Fix:** Replace the two Options with an `enum ActiveState { NaiveBayes(NaiveBayesState), Bm25(Bm25State) }`.
Non-blocking refactor.

### IN-04: Magic batch-flush constants `128` / `32` are inline literals

**File:** `crates/cb-compute/src/embedding_calcers.rs:137`
**Issue:** `if self.base_size < 128 || self.additional_size >= 32` hard-codes the
upstream `IncrementalCloud` flush schedule thresholds as bare literals. They are
documented in the surrounding comment but not named constants, so a reader cannot
grep them and a parity audit cannot cross-reference them to the upstream source.
**Fix:** Promote to named `const` (e.g. `INCREMENTAL_CLOUD_BASE_FLUSH_FLOOR`,
`INCREMENTAL_CLOUD_ADDITIONAL_FLUSH_BATCH`) with the `lda.cpp` line reference.

### IN-05: `unknown_token_id` for an empty dictionary collides with the first valid id

**File:** `crates/cb-data/src/text/dictionary.rs:147-151` and
`crates/cb-data/src/text/bigram_dictionary.rs:84-86`
**Issue:** `unknown_token_id = len() + start_token_id`. For an empty dictionary
(`len()==0`, `start_token_id==0`) this is `0` — the same id a single-entry
dictionary would assign to its first token. Under `UnknownTokenPolicy::Insert`
over an empty dictionary, every unknown token maps to id `0`. The Skip policy
(the digitizer default) sidesteps this, so no in-tree path is affected, but the
`Insert` arm on a degenerate empty dictionary produces an ambiguous id with no
guard.
**Fix:** Document that `Insert` over an empty dictionary is undefined / not used,
or special-case the empty dictionary to error under `Insert`. Non-blocking
(matches the upstream `dictionary_size + StartTokenId` formula).
