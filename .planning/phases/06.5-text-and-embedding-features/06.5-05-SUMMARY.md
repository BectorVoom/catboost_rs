# Phase 6.5 Plan 05: LDA Embedding Calcer Summary

**One-liner:** First embedding calcer — LDA `TLinearDACalcer` (bit-faithful f32 IncrementalCloud scatter + reference-faithful generalized symmetric eigensolve + sgemv project), computed online over the TFold permutation (D-03), with per-stage model parity guaranteed via binarization stability despite a documented upstream-vendored-CLAPACK eigensolver divergence.

## What shipped

- **Task 1 (re-instrument & re-measure):** extended the LDA instrumentation with the env-gated `lda_scatter` (the exact `ssygst_` generalized-eigenproblem inputs, captured before `ssygst_` mutates them in place) and `lda_project` (per-document projected feature row) hooks; rebuilt the instrumented trainer incrementally; froze the new scatter+projection ground truth (`scatter_projection_gt.json`, thread_count=1, reproducible via `gen_lda_scatter_gt`). Re-measured and **localized the divergence to the upstream eigensolve, not the scatter.**
- **Task 2 (LDA calcer):** `cb_compute::embedding_calcers` — `IncrementalCloud` per-class scatter, `total_scatter`, `between_matrix` (classification + regression arms + reg diagonal), `LdaCalcer::fit` (eigensolve via `lda_linalg`) + `LdaCalcer::compute` (sgemv project). 8 unit tests; scatter locked ≤1e-5 vs the frozen GT.

## Deviations from Plan

### [Rule 3 - Blocking issue] Final-link skip after incremental lda.cpp edit
- **Found during:** Task 1 rebuild
- **Issue:** after editing `lda.cpp`, `ninja _catboost` recompiled `lda.cpp.o` + re-archived the `.global.a` but did NOT relink `lib_catboost.so` — the `*.global.a` whole-archive libs are tracked as *order-only* deps of the `.so` link edge, so the smoke fit silently ran the STALE trainer (new hooks never fired).
- **Fix:** `rm -f lib_catboost.so` + explicit `ninja <path>/lib_catboost.so`, then re-stage. Documented in `instrument_text_pipeline_README.md` build-bugs section + the build script.

### [Scoped out] likelihood=true probability columns
- The LDA feature row is 3 columns `[projection, prob0, prob1]`; the probability columns (CalculateGaussianLikehood + InverseMatrix) were scoped out (see key-decisions). The projection column alone answers the eigensolve parity question; the per-stage model oracle is gated via binarization stability.

## Documented tolerance (escalate-don't-weaken, human sign-off recorded)

`PROJECTION_TOL = 6e-2` applies to the **raw LDA projection vector ONLY**, cited in `lda_oracle_test.rs` (module doc) and here. It is NOT a weakening of the model per-stage oracle (which stays byte-exact, gated via binarization stability). Rationale: upstream's vendored-CLAPACK `ssyev_` iterate is non-reference and not reproducible by a reference-faithful eigensolver; a LAPACK crate would reproduce the same reference we already match, for zero parity gain and a supply-chain cost (A3). No `#[ignore]`, no fabricated values, no silent weakening.

## Known Stubs

- **LDA likelihood-probability columns** (`embedding_calcers.rs::LdaCalcer::compute`): returns only the `projection_dim` projected scalar(s); the `+NumClasses` probability columns (for `likelihood=true`) are not emitted. Intentional scope cut (see key-decisions); a probability-column oracle is a follow-up. The projection column is fully wired and oracle-gated.
