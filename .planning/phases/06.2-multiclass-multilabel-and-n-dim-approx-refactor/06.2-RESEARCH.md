## Summary

This phase has two oracle-locked deliverables on the single shared CPU train loop. **Deliverable 1** (Wave 0) is a pure mechanical generalization of the entire `cb-train`/`cb-compute` pipeline from a scalar per-object approx (`Vec<f64>`) to a dimension-major flat buffer (`approx[d*n+i]`, D-6.2-01), gated by re-running all ~40 existing scalar oracles green at dim=1 (D-04 hard checkpoint). **Deliverable 2** (Waves 1-3) implements MultiClass (softmax, the only cross-dimension-coupled loss), MultiClassOneVsAll, MultiLogloss, MultiCrossEntropy, and MultiQuantile on the stable N-dim foundation.

The central research finding is that **upstream itself preserves the dim=1 path bit-identically**: `approx_calcer.cpp:1093` dispatches `CalcLeafValuesSimple` when `approxDimension == 1 && !isMultiTarget`, and `CalcLeafValuesMulti*` only when dim>1 or the loss is a `TMultiDerCalcer`. The `TVector<TVector<double>>` layout is confirmed dimension-major (`[approxDim][doc]`) at `approx_calcer.cpp:1108` (`// [bodyTailId][approxDim][docIdxInPermuted]`), exactly matching D-6.2-01. This means the D-04 no-behavior-change checkpoint is genuinely achievable: a dim=1 dimension-major buffer is byte-identical to today's `Vec<f64>`, and routing dim=1 through `compute_leaf_deltas` over a single dimension preserves the existing `cb_core::sum_f64` reduction order.

The der/Hessian split confirms D-6.2-03: **softmax is the only coupled loss** (symmetric dense Hessian solved via LAPACK, `hessian.cpp:22-52`); MultiClassOneVsAll, MultiLogloss/MultiCrossEntropy, and MultiQuantile all use a **diagonal Hessian** (`hessian.cpp:55-69`, per-dimension division — i.e., the scalar Newton/Gradient/Exact leaf solve applied independently per dimension). MultiQuantile reuses 6.1's Exact weighted-α-quantile path verbatim per dimension.

**Primary recommendation:** Implement Wave 0 as a length-1 dimension-major buffer threaded through `compute_gradients` → bootstrap → `compute_leaf_deltas` → approx update → model storage, mirroring upstream's `approxDimension==1` short-circuit so the reduction order is provably unchanged; gate on the full scalar oracle suite before writing any softmax math. Then add the coupled softmax der/Hessian (Wave 1) and reuse the per-dimension scalar der for the separable losses (Waves 1-3). Pin every multilabel fixture to `leaf_estimation_iterations:1` and MultiQuantile to `leaf_estimation_method:Exact` to keep the leaf math a single deterministic solve.

### Pattern 1: Dimension-major short-circuit at dim=1 (the D-04 anchor)
**What:** Mirror upstream's `approxDimension == 1 && !isMultiTarget` dispatch so the dim=1 reduction path is the *same code* that exists today.
**When to use:** Wave 0 — the entire refactor.
**Example:**
```
// Source: catboost/private/libs/algo/approx_calcer.cpp:1093-1097
const bool isMultiTarget = dynamic_cast<const TMultiDerCalcer*>(&error) != nullptr;
if (approxDimension == 1 && !isMultiTarget) {
    CalcLeafValuesSimple(...);          // ← today's exact reduction order
} else {
    CalcLeafValuesMultiForAllLeaves(...);
}
```
Rust analogue: `compute_leaf_deltas` loops `for d in 0..approx_dimension` over `approx[d*n .. d*n+n]`; at `dim==1` that is one pass over `approx[0..n]` — byte-identical to the current scalar call. Keep `cb_core::sum_f64` per-dimension (never sum across dimensions in the leaf reduction).

### Pattern 2: Separable per-dimension der reuse
**What:** Diagonal-Hessian losses compute der independently per dimension, reusing the scalar elementwise kernel.
**When to use:** MultiClassOneVsAll, MultiLogloss, MultiCrossEntropy, MultiQuantile (Waves 1-3).
**Example:**
```
// MultiQuantile — K independent quantile dims (one alpha per dim)
// Source: catboost/private/libs/algo_helpers/error_functions.cpp:453-478
for idx in 0..quantile_count {
    let arg = target - approx[idx*n + i];
    der[idx*n+i] = if arg.abs() < delta { 0.0 }
                   else { weight * (if arg > 0.0 { alpha[idx] } else { -(1.0 - alpha[idx]) }) };
    der2[idx*n+i] = 0.0;  // QUANTILE_DER2_AND_DER3
}
```

### Pattern 3: Softmax coupled der + symmetric Hessian (the ONLY new coupled math)
See Code Examples below. The off-diagonal Hessian entries couple dimensions; solved via a dense symmetric linear system per leaf.

### Anti-Patterns to Avoid
- **Object-major buffer (`approx[i*dim+d]`):** breaks the dim=1 byte-identity AND diverges from upstream's per-dimension reduction grouping → D-04 checkpoint fails. Use dimension-major (D-6.2-01).
- **Summing der across dimensions in the leaf reduction:** the diagonal losses estimate each leaf dimension independently; cross-dim summation is only for the *split score* (Pitfall 7) and the *MVS norm*, never the leaf delta.
- **Storing leaf_values in dimension-major order in the model:** upstream `.cbm`/json stores **leaf-major** (`leaf0[d0..dk], leaf1[...]`) — see Pitfall 6. The training buffer is dimension-major; serialization transposes.
- **Treating MultiClass leaf estimation as `gradientIterations`-many steps:** the default is Newton with **1 iteration** (`catboost_options.cpp:108-110`), not the `defaultGradientIterations=10`. Pin `leaf_estimation_iterations:1`.

## Common Pitfalls

### Pitfall 1: dim=1 reduction-order drift breaks the D-04 checkpoint
**What goes wrong:** A "clean" rewrite of the leaf reduction (e.g., a single fused loop over `dim*n`) changes the order in which `cb_core::sum_f64` accumulates the dim=1 leaf sums, producing a ≤1e-5-but-nonzero diff against the frozen scalar fixtures.
**Why it happens:** Floating-point summation is not associative; the D-04 gate demands *byte-identical* (0.0 diff), not just ≤1e-5.
**How to avoid:** Mirror upstream's `approxDimension==1` short-circuit (Pattern 1). Keep the per-dimension reduction as an outer loop `for d in 0..dim { sum_f64(approx[d*n..d*n+n] slice) }` so at dim=1 the slice and order are exactly today's. CONTEXT "specific ideas": *any layout choice that perturbs reduction order at dim=1 is wrong.*
**Warning signs:** Existing scalar oracles show a tiny nonzero diff (e.g. 1e-12) after Wave 0 — that is a reduction-order regression, not noise.

### Pitfall 2: MultiClass uses Newton with 1 iteration, NOT gradientIterations=10
**What goes wrong:** Reading `defaultGradientIterations=10` (`catboost_options.cpp:110`) and running 10 leaf-refinement passes, diverging from upstream which runs **1 Newton iteration** for MultiClass/MultiClassOneVsAll.
**Why it happens:** `GetEstimationMethodDefaults` returns both newton AND gradient iteration counts; `SetLeavesEstimationDefault` (`catboost_options.cpp:316-322`) picks the one matching the chosen method. MultiClass method = Newton → `defaultNewtonIterations = 1`.
**How to avoid:** Pin `leaf_estimation_iterations:1` in MultiClass/OneVsAll fixtures (matches default). For MultiLogloss/MultiCrossEntropy the *default* is Newton with **10** iterations — pin the fixture to `leaf_estimation_iterations:1` to keep a single deterministic Newton solve (the 6.1 Poisson precedent: override the high default to 1).

### Pitfall 3: MultiQuantile leaf method is Exact, not Gradient (single-host CPU)
**What goes wrong:** Reading the `catboost_options.cpp:118-122` Gradient default and using Gradient leaf estimation, diverging from the actual default.
**Why it happens:** `SetLeavesEstimationDefault` has a `useExact` override (`catboost_options.cpp:289-301`): for MAE/MAPE/RMSPE/Quantile/GroupQuantile/**MultiQuantile** on single-host CPU (with no monotone constraints, no `ApproxOnFullHistory`), the method is overridden to **Exact**. So MultiQuantile defaults to Exact, reusing the 6.1 weighted-α-quantile path per dimension (D-6.2-05).
**How to avoid:** Pin MultiQuantile fixtures to `leaf_estimation_method:Exact` (matches the single-host CPU default). der2=0 per dimension (`error_functions.cpp:473`).
**Warning signs:** Leaf values diverge while der1 matches → wrong leaf estimator.

### Pitfall 4: Target must be the contiguous remapped class index (MultiClass)
**What goes wrong:** Feeding raw labels (e.g. `{2, 5, 9}`) as the target; `targetClass = static_cast<int>(target)` (`error_functions.h:716`) indexes the der vector out of range or into the wrong dimension.
**Why it happens:** Upstream `TLabelConverter` (`label_converter.cpp:142-145`) builds `LabelToClass` mapping sorted distinct labels → `[0, k)`, and the training target is the remapped index; the model stores `ClassToLabel` to recover original labels at predict time (`class_params`/`multiclass_params` model_info, verified empirically).
**How to avoid:** Build the label→index map before the loop; train on indices; store the label map in `cb-model` metadata. approx_dimension = `max(distinct_count, 2)` (`label_converter.cpp:142`).
**Warning signs:** Panic/out-of-range on der write, or class predictions permuted.

### Pitfall 5: Softmax Hessian L2/trace-epsilon regularization must be exact
**What goes wrong:** Solving the softmax Newton system with naive `l2Regularizer` only, omitting the `maxTraceElement * f32::epsilon` adjustment (`hessian.cpp:34-44`), producing leaf values off by >1e-5.
**Why it happens:** Upstream adjusts L2 to `Max(l2Regularizer, maxTraceElement * numeric_limits<float>::epsilon())`, subtracts it from the diagonal, negates, then `SolveLinearSystem`, then negates the result.
**How to avoid:** Transcribe `TSymmetricHessian::SolveNewtonEquation` (`hessian.cpp:22-52`) verbatim, including the `float`-precision epsilon (not `f64`).
**Warning signs:** MultiClass leaf values diverge at the 1e-6..1e-5 scale where MultiClassOneVsAll (diagonal) matches.

### Pitfall 6: Model leaf_values are LEAF-MAJOR, not dimension-major
**What goes wrong:** Serializing the dimension-major training buffer directly into `.cbm`/json `leaf_values`, producing a model whose predictions are scrambled across dimensions.
**Why it happens:** Empirically verified (catboost 1.2.10): a depth-2 (4-leaf) 3-class model has `leaf_values` length 12 ordered **leaf-major** `[leaf0_d0, leaf0_d1, leaf0_d2, leaf1_d0, ...]` (`leaf_weights` has 4 entries, one per leaf). The training buffer is dimension-major; serialization must transpose `leaf[d][l] → leaf_values[l*dim + d]`.
**How to avoid:** Transpose at the `cb-model::from_trained` boundary; `leaf_weights` stays one-per-leaf.
**Warning signs:** Predictions oracle diverges while staged-approx (training-side) matches.

### Pitfall 7: Split score for multi-dim needs a defined cross-dimension reduction
**What goes wrong:** The oblivious split score (`tree.rs`) is computed from a single per-object gradient column; for multi-dim, *which* column? Picking the wrong reduction diverges the **splits** stage.
**Why it happens:** Upstream multi-dim scoring sums the per-dimension der contributions into the leaf-stat histogram (the score uses summed der/der2 across dimensions, NOT a single dimension). `scoring.cpp` uses `GetApproxDimension`.
**How to avoid:** Research the exact multi-dim split-score reduction in `greedy_tensor_search.cpp`/`scoring.cpp` during Wave 1 planning (this RESEARCH confirms *that* it differs; the exact accumulation is a Wave-1 transcription task). In-scope MVP fixtures use `bootstrap_type:No`, `random_strength:0` so only the score reduction (not sampling) matters.
**Warning signs:** Splits stage diverges before leaf-values for multi-dim losses.

### Pitfall 8: SHAP is per-dimension and OUT of the per-stage train oracle
**What goes wrong:** Spending Wave-0 effort widening SHAP to N-dim when the per-stage train oracle (borders/splits/leaf-values/staged-approx/predictions) does not exercise it.
**Why it happens:** SHAP (`cb-model::shap.rs`) is a feature-importance path, not part of the training oracle. D-6.2-06's per-stage floor does not include SHAP.
**How to avoid:** Wave 0 must keep SHAP *compiling* against the new model shape (it reads `leaf_values`), but full N-dim SHAP parity is NOT a Wave-0/1-3 gate. Keep SHAP at dim=1 behavior or first-dimension; flag any N-dim SHAP work as out-of-scope (deferred to a later fstr pass, consistent with MODEL-05 being 6.6).

### MultiClass softmax der + symmetric Hessian
```cpp
// Source: catboost/private/libs/algo_helpers/error_functions.h:687-728  (TMultiClassError::CalcDersMulti)
const auto derRef = MakeArrayRef(*der);
CalcSoftmax(approx, derRef);                       // p[d] = softmax(approx)[d]
if (der2 != nullptr) {                             // Symmetric Hessian, packed lower-triangular
    int idx = 0;
    for (int dimY : xrange(approxDimension)) {
        const double derY = derRef[dimY];
        der2Ref[idx++] = derY * (derY - 1);        // diagonal:  p_y*(p_y - 1)
        for (int dimX : xrange(dimY + 1, approxDimension))
            der2Ref[idx++] = derY * derRef[dimX];  // off-diag:  p_y * p_x
    }
}
for (auto& value : derRef) value = -value;         // der[d] = -p[d]
derRef[(int)target] += 1;                          // der[targetClass] += 1   ⇒  δ(d=t) - p[d]
// (× weight if weight != 1, both der and der2)
```

## Open Questions

1. **Symmetric Hessian solver crate vs hand-roll for softmax Newton.**
   - What we know: upstream uses `SolveLinearSystem` (LAPACK, `catboost/private/libs/lapack/linear_system.h`) on a small dense symmetric system (size = class count, typically 3-10).
   - What's unclear: whether to add a Rust linear-algebra crate (latest-version per CLAUDE.md) or hand-roll a small Cholesky/LU with the exact trace-epsilon regularization.
   - Recommendation: prefer a small vetted solver; gate any new crate behind `checkpoint:human-verify` + the package-legitimacy gate. For k≤3 a transcribed closed-form is acceptable if it reproduces `hessian.cpp:22-52` bit-faithfully. Planner decides per the in-scope class count.

2. **Exact multi-dim split-score reduction (Pitfall 7 / A1).**
   - What we know: the score uses cross-dimension der accumulation into the leaf histogram; `GetApproxDimension` threads through `scoring.cpp`.
   - What's unclear: the precise per-candidate score formula across dimensions (sum of per-dim L2 scores vs summed der then single score).
   - Recommendation: Wave-1 transcription task — read `greedy_tensor_search.cpp` + `scoring.cpp` multi-dim path before implementing MultiClass split scoring; isolate as a checkpoint within Wave 1.

3. **Plan-to-wave mapping granularity (Claude's discretion).**
   - Recommendation: Wave 1 = one plan (MultiClass softmax + MultiClassOneVsAll share the diagonal-vs-symmetric Hessian seam); Wave 2 = one plan (MultiLogloss + MultiCrossEntropy are the *same* `TMultiCrossEntropyError` class — a single der path, two enum names); Wave 3 = one plan (MultiQuantile, reuses 6.1 Exact). Wave 0 may need 1-2 plans (buffer/runtime widening; model+oracle widening).
