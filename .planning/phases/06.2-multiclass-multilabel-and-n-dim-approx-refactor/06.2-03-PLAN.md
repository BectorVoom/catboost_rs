  <decision>Softmax per-leaf symmetric Hessian Newton solver: hand-rolled transcription vs a vetted Rust linear-algebra crate (Open Q1).</decision>
  <context>
    Upstream solves a small dense symmetric system per leaf (size = class count, in-scope fixtures 3-4 classes) via LAPACK `SolveLinearSystem` (`catboost/private/libs/lapack/linear_system.h`), with an exact L2/trace-epsilon regularization at `f32` precision (`hessian.cpp:34-44`) that MUST be reproduced bit-faithfully for ≤1e-5 parity (RESEARCH Pitfall 5). CLAUDE.md mandates latest crate versions and prohibits `unwrap()`. The system is tiny (k≤4 here), so a transcribed closed-form / small Cholesky is viable and adds no dependency; a vetted crate is the alternative but introduces a supply-chain surface (T-6.2-SC).
  </context>
  <options>
    <option id="hand-roll">
      <name>Hand-roll the small symmetric solve (transcribe hessian.cpp:22-52 verbatim)</name>
      <pros>No new dependency; no package-legitimacy gate; full control over the f32 trace-epsilon regularization for bit-faithful parity; trivial at k≤4.</pros>
      <cons>Must transcribe SolveLinearSystem semantics carefully; risk of parity drift for larger k (out of in-scope range this phase).</cons>
    </option>
    <option id="crate">
      <name>Add a vetted Rust linear-system crate (latest version)</name>
      <pros>Reuses a tested dense solver; satisfies the latest-crate-versions preference.</pros>
      <cons>Supply-chain surface (must run the package-legitimacy gate + this checkpoint stays blocking-human); must still reproduce upstream's exact trace-epsilon regularization on top of it — the crate does not do the catboost-specific adjustment.</cons>
    </option>
  </options>
