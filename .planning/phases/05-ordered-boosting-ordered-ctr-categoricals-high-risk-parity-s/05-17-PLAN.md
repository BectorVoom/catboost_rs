<upstream_findings>
Verified during planning (line-by-line read of the vendored catboost 1.2.10 source):

- `learn_context.cpp:490-595` is the fold-creation loop. With one persistent
  `Rand` (`TRestorableFastRng64`, member of `TLearnProgress`): `InitApproxes` runs
  FIRST (line 490), then learning folds `0..LearningFoldCount` are built via
  `BuildPlainFold` (Plain path, line 524-549) with `shuffle = foldIdx != 0` (line
  529) — so `Folds[0]` is the identity (zero shuffle draws) and folds
  `1..LearningFoldCount` are each shuffled; FINALLY the AveragingFold is built LAST
  (line 575-589) with `shuffle = IsAverageFoldPermuted` (= hasCtrs).
- THE CANDIDATE EXTRA-DRAW SOURCE (the 05-15 hypothesis, now localized): both
  `BuildPlainFold` and `BuildDynamicFold` call `InitOnlineEstimatedFeatures(...,
  rand)` (`fold.cpp:200-206`, `:298-304`) -> `CreateEstimatedFeaturesData(...,
  rand)` PER FOLD. If feature estimators draw on `rand`, every fold (including the
  identity Folds[0]) consumes additional draws beyond its Shuffle — which the
  current `create_folds` does NOT model. At pc=4 (`LearningFoldCount=3`) there are
  4 `InitOnlineEstimatedFeatures` calls before/around the averaging shuffle vs 2 at
  pc=2; that asymmetry is the prime suspect for why pc=2 matches but pc=4 does not.
  The harness MUST log `GetCallCount()` around EACH fold build (not just each
  Shuffle) to capture this.
- `TRestorableFastRng64` (`catboost/libs/helpers/restorable_rng.h`) wraps
  `TFastRng64` and increments a `CallCount` on every `GenRand()`/`Advance(delta)`;
  `GetCallCount()` returns it. This is the exact per-fold draw accounting the
  harness reads. `TFastRng64::GenRand()` (`util/random/fast.h:68-73`) is the
  primitive `cb-core::TFastRng64` already reproduces bit-exact.
- `InitPermutationData` (`fold.cpp:43-95`): `shuffle == false` -> `std::iota`
  (zero draws); `shuffle == true` -> `Shuffle(ObjectsGrouping, PermutationBlockSize,
  rand)`. For ungrouped data with `PermutationBlockSize == 1`
  (`objects_grouping.cpp:203` `CreateShuffledIndices(objectCount, rand, &indices)`
  -> `permutation.h:84-89` -> `util/random/shuffle.h` Fisher-Yates), each shuffled
  fold consumes exactly one Fisher-Yates pass (`n-1` `Uniform` draws) on the shared
  `rand`. The block-size path (`objects_grouping.cpp:207`) only triggers for
  grouped data; the tensor_ctr_e2e config (N=30, ungrouped) uses block size 1.

CONCLUSION: the missing accounting is per-fold non-Shuffle draws (most likely
`InitOnlineEstimatedFeatures`, and possibly target-classifier / starting-approx
draws). The harness reads `GetCallCount()` deltas per fold at pc=1,2,4 to expose
the exact rule; `create_folds` is then corrected to consume the SAME per-fold
draw counts so the pc=4 averaging shuffle lands on the upstream call-count,
yielding `[6,0,10,14]`.
</upstream_findings>
