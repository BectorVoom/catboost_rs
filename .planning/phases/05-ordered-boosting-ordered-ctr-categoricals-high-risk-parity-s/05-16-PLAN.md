<objective>
Close GAP 1 (ORD-02): the wiring test `ordered_structure_differs_from_plain`
FAILS at HEAD. After the 05-12 identity-`Folds[0]` upstream-faithful RNG change,
at `permutation_count=1` the Ordered structure search runs on the identity
(object-order) learning fold, so on this synthetic dataset (random_strength=0,
bootstrap=No) the per-segment L2 scores collapse to Plain-identical — the test's
assumption "Ordered splits ≠ Plain splits" is invalidated, not the
implementation. The Ordered branch is demonstrably ALIVE: `ordered_boost_e2e_oracle_test`
passes 2/2 ≤1e-5 and `greedy_tensor_search_oblivious_ordered` is wired at
boosting.rs:1416.

A failing test in the oracle suite is an unacceptable blocker for phase
completion. The boosting.rs ordered path selects the learning permutation via
`find(|f| !f.is_averaging)`, which returns `Folds[0]` — the IDENTITY for EVERY
`permutation_count`. Therefore re-keying the test's `permutation_count` alone
CANNOT make Ordered diverge from Plain: the ordered search would still consume the
identity fold. Changing which fold the ordered search consumes is a PRODUCTION
change and is OUT OF SCOPE for this gap-closure plan (the e2e oracle already
validates ORD-02 ≤1e-5). The correct fix is therefore to RETIRE the invalidated
structural-divergence sub-test in place, with an in-file rationale that points to
`ordered_boost_e2e_oracle_test` as the authoritative ORD-02 structural check,
while keeping the aliveness gates (`ordered_training_grows_a_full_finite_model`,
`plain_path_still_trains`) intact.

This plan depends on 05-15 only for sequencing hygiene (the multi-permutation
draw order is corrected and locked there); the retire itself touches no
production code.

Purpose: remove the only failing test at HEAD without weakening any genuine ORD-02
parity guarantee.

Output: a passing `ordered_boost_wiring_test` suite + an auditable deferred-decision
note co-located with the phase artifacts.
</objective>

<approach_decision>
PRIMARY: retire `ordered_structure_differs_from_plain` in place with a documented
rationale that delegates ORD-02 structural authority to
`ordered_boost_e2e_oracle_test`. Rationale (and why this — not re-keying — is the
real resolution):

- The boosting.rs ordered path selects the learning permutation via
  `find(|f| !f.is_averaging)` (boosting.rs:~1054), which returns `Folds[0]` =
  the IDENTITY fold for ALL `permutation_count`. So re-keying the test's
  `permutation_count` to >=2 does NOT change the learning fold the ordered search
  consumes — it would still run on the identity. A test-only re-key therefore
  CANNOT make the `assert_ne!` divergence hold; only an out-of-scope production
  change to fold selection could, and the e2e oracle already locks ORD-02 ≤1e-5.
- At the gated config, upstream itself builds the lone learning `Folds[0]` as the
  IDENTITY (`shuffle = foldIdx != 0`, `fold.cpp:54`), so Ordered structure search
  running on object order — and coinciding with Plain on a randomness-free
  synthetic dataset — is FAITHFUL behavior, not a bug. Asserting divergence is
  asserting a false premise.
- The Ordered branch is provably alive: `ordered_training_grows_a_full_finite_model`
  proves a real grown 5-tree model (not a Plain fall-through), and
  `ordered_boost_e2e_oracle_test` (2/2 PASS ≤1e-5 vs catboost 1.2.10) is the
  authoritative ORD-02 structural-correctness check. Retiring the invalidated
  wiring assertion loses NO genuine guarantee.

CONDITIONAL (option b) — re-key to `permutation_count>=2` with a passing
`assert_ne!` — is taken ONLY IF the read-first investigation shows (contrary to
the current reading of boosting.rs:~1054) that the ordered structure search
actually consumes a NON-identity learning fold for higher permutation_count. If
that proves true, parametrize `params` to the smallest pc yielding a non-identity
ordered learning fold and keep `assert_ne!` — but only if it PASSES with no
production change. Given the current code this is not expected to hold.

REJECTED (option a) — synthesize a new diverging dataset under the identity fold —
is the least upstream-faithful: it engineers a dataset to force divergence under a
regime where upstream itself does not diverge, masking rather than reflecting true
behavior.
</approach_decision>
