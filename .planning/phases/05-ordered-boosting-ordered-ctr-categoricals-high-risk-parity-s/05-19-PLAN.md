<objective>
Close bar (c) = SC-1 / ORD-01 at the production-default permutation_count=4: make the
pc=4 categorical train->predict path reproduce catboost 1.2.10 RawFormulaVal <=1e-5
across all objects and all 5 trees.

The root cause is PROVEN (05-18 live instrumentation + 05-BAR-C-S-SHUFFLE-RESEARCH.md;
do NOT re-derive). TWO mechanisms must be ported, both absent from cb-train today:

1. **The initial learn-set shuffle `S`.** Upstream applies a Fisher-Yates shuffle to the
   learn data BEFORE fold creation whenever `NeedShuffle` fires (cat features present, not
   has_time): `preprocess.cpp:183 ShuffleLearnDataIfNeeded` -> `NCB::Shuffle(grouping, 1, rand)`
   -> `CreateShuffledIndices` (`permutation.h:84` = std::iota + the shuffle.h Fisher-Yates,
   the SAME algorithm as cb-train's `shuffle_in_place`). `rand` is `TRestorableFastRng64(random_seed)`
   and the shuffle is its FIRST consumer (zero pre-draws, `train_model.cpp:1057-1058`). The
   averaging-fold online-CTR bins are therefore materialized in order `Q = S ∘ LearnPermutation`.
   cb-train skips `S`, producing the right pc=4 PARTITION but the WRONG leaf VALUES (the mixed
   cat buckets cat3/cat4 are interleaved differently). **Crucial:** cb-train's current
   `create_folds` cc=29/87 "pre-draw hack" advances the RNG to the same post-shuffle phase, so
   the FOLD PERMUTATIONS already match upstream — but the DATA is never shuffled. Applying `S`
   on the persistent stream consumes those very draws, so it SUBSUMES the hack (one unified fix,
   not a second compensating advance).

2. **Per-iteration structure-fold cycling.** Upstream picks the structure fold each iteration
   via `takenFold = Folds[Rand.GenRand() % foldCount]` (`train.cpp:208`); for pc=4 this is the
   cycle `[0,2,0,2,2]` giving per-tree structures `[A,B,A,B,B]`. cb-train currently uses a
   FIXED `Folds[0]` structure column for every tree (`boosting.rs` `find(|f| !f.is_averaging)`),
   so trees 1/3/4 get the wrong structure. This is the SECOND half of bar (c) and was omitted
   from the prior 05-19 draft (gsd-plan-checker blocker 1).

The CTR math itself is already bit-exact (Spike-001) and is OUT OF SCOPE — only the ORDER of
the data fed to the (unchanged) materialization changes.

Purpose: finalize the last open bar of Phase 5 (production default pc=4) without weakening any
oracle. Output: a derived `create_shuffled_indices` + integer-exact `S` oracle; a DE-RISK gate
proving S-order reproduces the self-consistent CTR bins through the unmodified materialization;
`S` applied in `train_inner` (subsuming the pre-draw hack); per-iteration structure-fold
cycling; predictions inverted to original order; the committed pc=4 e2e oracle <=1e-5.
</objective>
