# Phase 5 Plan 05: Ordered CTR + Ordered Boosting (ORD-02 / ORD-03 ordered half) Summary

**The anti-leakage heart of the phase, added LAST over the locked plain-CTR (05-04) and permutation/fold (05-03) substrates: ordered (per-permutation prefix) read-before-increment CTR and the ordered-boosting body/tail approximant (a tail document's delta estimated on the body prefix, never on itself), both per-object oracle-locked ‚Äî D-03 permutation gate FIRST, then per-object integer anchors exact + values ‚â§1e-5, plus the monotone / identity-permutation degeneration internal-consistency anchors (the A2-accepted indirect path for the per-iteration ordered approx).**

## Performance

- **Duration:** ~16 min
- **Completed:** 2026-06-14
- **Tasks:** 2
- **Files modified:** 18 (2 created, 16 modified)

## Accomplishments

- **Ordered CTR** (`ordered_ctr_per_permutation`, `ctr/online.rs`): the SAME
  read-before-increment prefix loop as 05-04's `online_ctr_prefix_binclf`, now
  computed UNDER a specific permutation (the per-fold order) ‚Äî the focused Wave-5
  delta over the whole-set Plain CTR (D-05/D-06). Returns `OrderedCtrPrefix`: the
  OBJECT-order per-object `good`/`total`/`value` plus the PERMUTATION-order running
  `(num, denom)` read at each learn step, with a `per_bucket_monotone` anchor (a
  fixed bucket's running counts must be monotone non-decreasing ‚Äî a non-monotone
  read betrays out-of-order accumulation / self-inclusion, the leakage signature).
- **Ordered boosting approximant** (`ordered_approx_delta_simple`, `boosting.rs`):
  the `UpdateApproxDeltasHistoricallyImpl` transcription (`approx_calcer.cpp:566-600`
  / `CalcApproxDeltaSimple` `:706`). Seeds a running per-leaf der/weight from the
  BODY prefix, then walks the TAIL rows in permutation order add-then-read,
  computing the running Gradient leaf average `leafDer / (leafWeight + l2)` per row
  ‚Äî a tail doc's delta never depends on itself, and body docs keep delta `0` (the
  estimation prefix). Body sums via `body_sum_weight` (`sum_f64`, D-08).
- **`EBoostingType{Plain,Ordered}`** + `BoostParams.boosting_type` pinned
  EXPLICITLY via `boosting_type_default()` (CPU default `Plain`; Ordered
  auto-select is GPU-only, Pitfall 6), propagated across all 13 `BoostParams`
  literals ‚Äî workspace stays compiling.
- **Two per-object oracles** locking the D-03 ‚Üí ordered-value sequence with the
  internal-consistency + degeneration anchors (see below).

## Deviations from Plan

**1. [Rule 3 ‚Äî fixture-sourcing] ordered_ctr fold-1 per-fold reseed.**
The plan's Task 1 gates BOTH committed fold permutations (`permutation_fold0`,
`permutation_fold1`) integer-exact (D-03). The continuous-stream multi-fold draw
order modeled by `cb_train::permutations`/`create_folds` (one persistent RNG across
folds, 05-03) reproduced fold-0 but NOT fold-1. Probing confirmed the
`ordered_ctr` offline harness drew each fold from a FRESH
`TFastRng64::from_seed(random_seed + foldIdx)` (fold-0 ‚Üê seed 0, fold-1 ‚Üê seed 1):
`fisher_yates_permutation(30, 1)` reproduces the committed `permutation_fold1`
EXACTLY. The D-03 contract is "our permutation reproduces upstream's"; the gate is
locked against that exact per-fold seeding. The continuous-stream draw order remains
the `create_folds` model for a single training run (an orthogonal axis from the
offline per-fold dump). **Resolution:** gate fold-0 ‚Üê seed 0 and fold-1 ‚Üê seed 1
(the per-fold reseed), documented in the test header.

**2. [Rule 3 ‚Äî D-09 oracle sourcing] ordered approx validated indirectly.**
The plan's Task 2 frames a per-iteration ordered-approx oracle + final ordered
prediction parity. The `ordered_boost` fixture commits only the permutation, the
body/tail boundaries, and the per-object ordered approx ‚Äî NOT the features/target
(stdin-fed, uncommitted, D-09; the 05-04 precedent). A literal full train‚Üípredict
ordered oracle is not runnable from committed artifacts. **Resolution (the approved
A2 residual + transcribe-then-self-oracle):** lock the ordered-approx MACHINERY via
structural anchors whose deltas are auditable by hand ‚Äî body/tail boundaries ==
committed, the no-leakage tail-delta property (body docs keep delta 0; a tail doc's
delta is a body-seeded running average that never depends on itself), and
identity-permutation degeneration toward the plain whole-set leaf average ‚Äî plus the
committed `ordered_approx_iter0` well-formedness (finite, per-object, bounded) as the
A2 indirect anchor.

**Total deviations:** 2 (both Rule 3 ‚Äî fixture-sourcing adapted to the committed
surface). **Impact:** No scope creep; ORD-02 + the ordered half of ORD-03 are locked
(D-03 permutation gate, per-object integer anchors exact, values ‚â§1e-5, no-leakage
structural + degeneration anchors). The residual (a full ordered train‚Üípredict
parity) is the accepted A2 risk, tracked behind the indirect anchors.

## Issues Encountered

- **fold-1 continuous-stream mismatch:** initial gate used the 05-03
  continuous-stream `permutations(N, 2, 0)` for both folds; fold-1 diverged at
  index 0 (expected 22, got 10). A seed probe localized it to a per-fold reseed
  (seed 1 reproduces fold-1 exactly) ‚Äî Deviation 1.
- **No committed fold inputs for ordered boost (D-09):** the ordered approximant
  could not be driven end-to-end from the fixture; locked via the structural +
  indirect anchors instead ‚Äî Deviation 2.
- **Disk pressure:** all verification scoped to per-crate `cargo test -p cb-train`
  (no cb-compute/MLIR/cubecl test-profile link); the link-failure risk flagged in
  the plan was not hit.
