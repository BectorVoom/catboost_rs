<objective>
De-risk the single biggest parity hazard in the ORD-05 leaf-value fix BEFORE
any value stage runs: the fold-permutation DRAW ORDER. Per the research
(05-CTR-LEAF-VALUE-RESEARCH.md, Open-question 1 — "the single biggest de-risk
item"), upstream catboost 1.2.10 builds the lone learning fold `Folds[0]` as the
IDENTITY (no shuffle, zero RNG draws — `shuffle = foldIdx != 0`,
learn_context.cpp:524 / fold.cpp:54) and the AveragingFold as the FIRST real
seeded Fisher-Yates draw (`IsAverageFoldPermuted = hasCtrs`,
learn_context.cpp:38-46/575-577). `cb-train::create_folds` currently shuffles
EVERY fold including `Folds[0]`, so its TFastRng64 draw stream is already
divergent from upstream — meaning the AveragingFold materialization (Plan 05-13)
would be computed under the WRONG permutation and silently fail the e2e gate.

Fix `create_folds` so the first learning fold is identity and the averaging fold
consumes the first draw, then prove the AveragingFold permutation byte-matches
upstream with a standalone integer-exact oracle — the D-03-style linchpin that
must pass before the leaf-value stage is allowed to run.

Purpose: A CTR partition computed under the wrong permutation is meaningless
(the same discipline as the D-03 permutation gate). Locking the draw order in
isolation here means a divergence in 05-13/05-14 isolates to the leaf-value
math, never an entangled permutation + leaf-value break.

Output: an identity-`Folds[0]` `create_folds`, a documented averaging-fold
permutation accessor, and a GREEN standalone draw-order oracle. Closes ORD-05
de-risk gate (no e2e gate yet — that is 05-14).
</objective>
