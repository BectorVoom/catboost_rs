<objective>
Close GAP 2 (WR-01 / ORD-01): the pre-averaging `GenRand` draw in `create_folds`
currently fires before the FIRST learning shuffle (`idx == 1`) via the
`first_real_shuffle` flag. That is correct ONLY for `permutation_count=1`, where
`learning_folds == 0` so `idx == 1` IS the averaging fold. For the production
default `permutation_count=4` (`learning_folds == 3`), the pre-draw fires before
the first of three learning shuffles; the averaging fold then draws at an
RNG call-count that was never validated against upstream. Every CTR and Ordered
train path under the default config is therefore unvalidated.

Move the pre-averaging draw to fire immediately before the AVERAGING fold's
shuffle (`idx == learning_folds`) for ALL `permutation_count` values, leaving the
`permutation_count=1` draw order byte-identical (because at `permutation_count=1`,
`idx == learning_folds == 1` is already where the draw fires today — both the old
`first_real_shuffle` guard and the new `idx == learning_folds` guard fire on the
same fold). Add an integer-exact `permutation_count>=2` draw-order oracle ANCHORED
on a REAL catboost 1.2.10 offline dump (committed as a fixture) so the
multi-permutation path is validated against UPSTREAM — not against the same RNG
primitives the implementation uses.

Purpose: a user training with default params (permutation_count=4) currently gets
an unvalidated permutation draw that may silently diverge from upstream parity.
This pins the correct upstream position and proves it with a catboost-anchored
oracle that can actually catch a wrong advance count.

Output: corrected `create_folds` pre-draw guard + a new multi-permutation
draw-order oracle test anchored on a committed catboost 1.2.10 fixture.
</objective>

<upstream_findings>
The pre-averaging draw position was settled by reading the vendored upstream
fold-creation loop and the Shuffle/RNG-consumption helpers:

- `catboost/private/libs/algo/learn_context.cpp:524` and `:529` — each learning
  fold `foldIdx` is built with `shuffle = foldIdx != 0`, so `Folds[0]` is the
  IDENTITY (no shuffle) and learning folds `1..LearningFoldCount` are shuffled.
- `catboost/private/libs/algo/learn_context.cpp:575-578` — the AveragingFold is
  built AFTER all learning folds with `shuffle = foldsCreationParams.IsAverageFoldPermuted`
  (`= hasCtrs`), so it is the LAST fold created and its shuffle is the last draw
  in the fold-creation sequence.
- `catboost/private/libs/algo/fold.cpp:43-95` (`InitPermutationData`) — when
  `shuffle == false` the function does a plain `std::iota` and consumes ZERO
  draws on `rand`; when `shuffle == true` it calls `Shuffle(..., rand)`.
- `catboost/libs/data/objects_grouping.cpp:191-224` (`NCB::Shuffle`) →
  `catboost/libs/helpers/permutation.h:83-89` (`CreateShuffledIndices`) →
  `util/random/shuffle.h:24-32` — each shuffled fold consumes exactly one
  Fisher-Yates pass (`n-1` draws) on the SHARED persistent `Rand`; there is no
  extra inter-fold `GenRand` inside this fold-creation loop itself.

CONCLUSION: upstream draw order is the learning shuffles in order (fold 0 = zero
draws, folds 1..learning_folds-1 = one Fisher-Yates each) THEN the averaging
shuffle. The single extra `GenRand` that 05-14 empirically validated against
catboost 1.2.10 (the "RNG call-count 1" pre-averaging advance that produces the
`[6,0,7,17]` partition at permutation_count=1) is tied to the AVERAGING fold's
construction, so it belongs immediately before the averaging shuffle — i.e. it
must be guarded on `idx == learning_folds`, NOT on the first learning shuffle.
At `permutation_count=1` (`learning_folds == 1`) these two positions coincide,
which is why the gated e2e gate stayed green while the multi-permutation path was
silently wrong. WR-01 flags that the "exactly one pre-averaging GenRand" advance
was only ever validated against upstream at permutation_count=1; for
permutation_count>1 it has NEVER been checked against catboost. This plan
generalizes the guard correctly AND anchors the pc>=2 averaging permutation
against a REAL catboost 1.2.10 dump (not against cb-core's own TFastRng64), so the
advance count is validated against upstream — which is the entire point of the gap.
</upstream_findings>
