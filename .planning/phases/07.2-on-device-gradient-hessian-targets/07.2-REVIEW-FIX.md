All fixes verified on real hardware: the full `pointwise_hist` suite (9/9) and
`gradient_gpu` self-oracle suite (23/23) PASS on ROCm/gfx1100 in-env. The default

## Fixed Issues

### WR-01: `launch_der_binary` / `launch_der_param` accept a length-mismatched (empty-approx, non-empty-target) input

**Files modified:** `crates/cb-backend/src/gpu_runtime.rs`
**Commit:** d2378bb
**Applied fix:** Added the `target.len() != approx.len()` shape guard BEFORE the
`approx.is_empty()` short-circuit in both `launch_der_binary` and `launch_der_param`,
returning `CbError::LengthMismatch` so these readback wrappers agree with their sibling
`*_into` / `*_handle` entry points on the malformed-input contract. (`launch_der_unary`
takes a single array and is unaffected.)

### WR-02: `launch_pointwise_hist2` reports `AtomicFinalizePath::HostSumFallback` for a run that executed the in-kernel f64 atomic merge

**Files modified:** `crates/cb-backend/src/gpu_runtime.rs`
**Commit:** 608ac0f
**Applied fix:** Added a distinct `AtomicFinalizePath::InKernelAtomicF64Unadvertised`
enum variant and report it (instead of `HostSumFallback`) when the device does not
advertise f64 atomic-add but the in-kernel atomic merge runs anyway (the gfx1100 case).
This stops the histogram report from colliding with the reduce helper's genuine
`HostSumFallback` (deterministic-host-sum) semantics. The empty short-circuit now reports
the same capability-derived path rather than `HostSumFallback`. The genuine host-sum
path in `launch_block_reduce_atomic_f64` and the `reduce.rs` consumer are unchanged.
**Note:** This changes the runtime-reported `AtomicFinalizePath` value on the histogram
path (a contract/reporting change, not a numeric-parity change). Histogram values are
unchanged — verified by the 9/9 ROCm `pointwise_hist` pass. Recommend a human confirm
no downstream consumer pattern-matched the old `HostSumFallback` value on the histogram
path (none found in-tree; all consumers only `{:?}`-print).

### WR-03: `max_divergence` test helper indexes `device[i]` over `baseline.len()` without a length precondition

**Files modified:** `crates/cb-backend/src/kernels/gradient_gpu.rs`
**Commit:** 54bc497
**Applied fix:** Replaced the raw-index loop with an `assert_eq!(device.len(),
baseline.len(), ...)` precondition + a `zip` iteration, matching the hardened
`kernels::scan`/`kernels::reduce` reporters (IN-02/IN-03). A short device buffer now
surfaces a clear length-mismatch panic instead of an opaque index-out-of-bounds.

### IN-01: `all_losses_device_resident_handoff` constructs a `device` binding only to discard it

**Files modified:** `crates/cb-backend/src/kernels/gradient_gpu.rs`
**Commit:** 2d86666
**Applied fix:** Deleted the unused top-level `device` binding and its trailing
`let _ = device;`; the `read_handle` helper already builds its own device/client. The
intent (single-runtime hand-off surface, read-backs after all launches) is preserved as
a plain comment.

### IN-02: Duplicated per-family launch geometry/upload block across the binary / half-byte histogram arms

**Files modified:** `crates/cb-backend/src/gpu_runtime.rs`
**Commit:** 81bca90
**Applied fix:** Extracted a single `launch_hist2_family!` macro (local to
`launch_pointwise_hist2_into`) that encapsulates the cindex/indices/der1/weight upload,
the channel-float `#[cfg(wgpu)]` f32 vs `#[cfg(not)]` f64 split, the binSums zero-init,
the launch, and the `return Ok(h)`. The binary, half-byte, and non-binary arms now each
call the macro (the non-binary arm passes the `bits` comptime arg as the only structural
difference). The channel dispatch and zero-init now live in exactly one place.
**Verification:** This touches production launch logic, so it was verified beyond a
syntax check — all 9 `pointwise_hist` tests (binary, half-byte, 8-bit, 5/6/7-bit,
whole-family, handoff, f32) PASS on ROCm/gfx1100, confirming byte-identical histograms.

### IN-03: Module-level doc inventory still describes the `gpu_runtime/{der,histogram}.rs` split as an open-ended planned follow-up

**Files modified:** `crates/cb-backend/src/gpu_runtime.rs`
**Commit:** 4f45815
**Applied fix:** Reworded the module inventory doc comment so the split reads as an
EXPLICIT tracked follow-up (scheduled refactor work) rather than an open-ended "planned
follow-up," so the inventory comment does not become permanently load-bearing. Doc-only
change (the review noted "no code change required this phase").
