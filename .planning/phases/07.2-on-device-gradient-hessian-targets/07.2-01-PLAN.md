<objective>
First MVP vertical slice of Phase 7.2: make the RMSE der1/der2 derivative path run
device-resident over the compile-time `SelectedRuntime` (rocm/wgpu/cuda/cpu), return
der1/der2 as device buffer HANDLES with no host fold (SC-3 / D-7.2-04), and self-oracle
the RMSE derivative against the `cb-compute::loss` CPU baseline on `rocm` (gfx1100).

This is the load-bearing seam slice: it generalizes the 7.1 `gpu_runtime.rs` launch
shell off `CpuRuntime` onto `SelectedRuntime` for the elementwise der family, defines the
handle-in → handles-out device-residency seam the 7.3 histogram kernels plug into, and
lands the all-backend self-oracle file the remaining loss families (Plans 02/03) extend.
RMSE is chosen because it exercises both a kernel-backed der1 (`gradient_kernel`) AND a
constant der2 (`-1.0`, no kernel) — proving both seam shapes in one slice.

After this plan, a user running the rocm der oracle can compute RMSE der1/der2
device-resident and see the reported divergence vs the CPU baseline — a real end-to-end
capability that did not exist before.

Purpose: Land the generic der launch seam + device-residency handle hand-off (the 7.2
deliverable that matters more than the arithmetic, per CONTEXT specifics).
Output: `launch_der_binary_handle` / `launch_der_binary` in `gpu_runtime.rs`,
`DerBinaryKernel` selector, the RMSE der2 const-handle helper, the new
`kernels/gradient_gpu.rs` self-oracle file, and the `gradient_gpu` module mount.

Honors binding decisions: D-7.2-01 (RMSE in the narrow start), D-7.2-03 (reuse the
existing `gradient_kernel` body — no new math), D-7.2-04 (device handles, no host fold),
D-7.2-05 (all cubecl coupling stays in cb-backend; cb-compute untouched), D-7.2-06 (oracle
REPORTS divergence, does not sign off the GPU-06 epsilon — that is 7.6), D-7.2-07 (per-crate
verification). der1/der2 stay UNWEIGHTED to match the CPU baseline (RESEARCH A1 / Pitfall 1).
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| host slice → device buffer | `client.create(Bytes::from_elems(approx.to_vec()))` copies a host `&[f64]` into a device handle; `ArrayArg::from_raw_parts(handle, n)` re-asserts the length `n` to the kernel |
| (no external/network/auth boundary) | in-process GPU compute over in-memory numeric arrays; no untrusted input, no persistence, no user-facing surface (RESEARCH Security Domain) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07.2-01 | Tampering | `unsafe { ArrayArg::from_raw_parts(out_handle, n) }` over-provisioned threads on non-cube-multiple n | mitigate | The reused `gradient_kernel` already guards every write with `if ABSOLUTE_POS < approx.len()`; the n=37 oracle fixture exercises the idle path (Pitfall 4). Output handle is allocated `n * size_of::<f64>()`, matching the kernel's bound |
| T-07.2-02 | DoS (numeric overflow) | launch geometry `n.div_ceil(CUBE_DIM)` / byte-size `n * size_of::<f64>()` | accept | `usize` arithmetic; der arrays are training-set-sized (bounded by available RAM, far below `usize::MAX`); no untrusted length crosses the seam. Documented as accepted, low risk |
| T-07.2-03 | Tampering (silent corruption) | `read_one` failure on the self-oracle wrapper | mitigate | Map `read_one` failure to `CbError::Degenerate` (WR-05, gpu_runtime.rs:128–131); never return a zero `Vec`/handle masquerading as a valid derivative |
| T-07.2-SC | Tampering | npm/pip/cargo installs | mitigate | No new packages installed this phase (RESEARCH Package Legitimacy Audit: none new); all deps vetted in 7.1. No `[ASSUMED]`/`[SUS]` packages |
</threat_model>
