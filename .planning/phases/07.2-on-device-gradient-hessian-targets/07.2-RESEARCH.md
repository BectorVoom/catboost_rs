# Phase 7.2: On-Device Gradient/Hessian & Targets - Research

**Researched:** 2026-06-20
**Domain:** GPU device-resident gradient/Hessian (der1/der2) kernels via CubeCL, structural parity with `catboost/cuda/targets/`
**Confidence:** HIGH

## Summary

Phase 7.2 makes the **already-authored** `#[cube]` pointwise derivative kernels in `cb-backend/src/kernels.rs` run **device-resident on the GPU runtimes** (rocm/wgpu/cuda), feeding the 7.3 histogram kernels without a host round-trip. The math is done — RMSE, Logloss/CrossEntropy, Quantile/MAE, and Focal grad+hess kernels already exist, are `generics-float`, and are verified bit-exact on `CpuRuntime` via the `kernels/gradient.rs` spike. **The load-bearing work is plumbing, not arithmetic**: generalize the der launch helpers (currently hard-coding `CpuRuntime`/`CpuDevice` in `cpu_runtime.rs`) over `R: Runtime` exactly as Phase 7.1 already did for `block_reduce`/`block_scan` in `gpu_runtime.rs`, return der1/der2 as **device buffer handles** (not host `Vec<f64>`), and define the device-side handle-in → handles-out seam so 7.3 plugs in with zero host fold.

The upstream structural-parity target (`pointwise_targets.cu`) is a one-thread-per-object elementwise kernel: bounds-guarded `i = blockIdx*blockDim + threadIdx`, weight folded as `der[i] = weight * target.Der(relev, val)`, der/der2 written only when the output pointer is non-null, and **the only reduction is for `functionValue`** — which D-7.2-02 defers. der/der2 carry no cross-object reduction, so D-03's atomic-finalize contract is a genuine no-op for this slice (confirmed: 7.1 found gfx1100/HIP at cubecl 0.10.0 lacks f64 atomic-add anyway, and the histogram phases are where atomics bite).

Two prior 7.1 risk flags are **already resolved**: the cubecl-hip-sys version skew is closed (`Cargo.lock` pins `7.1.5280200`, HIP 7.1, matching the box — not 7.2), and **plane ops work on gfx1100** (the 7.1 scan/reduce oracle ran the plane path with max divergence 0.000e0). The der kernels here use neither atomics nor planes (pure elementwise), so they are even safer than the 7.1 primitives.

**Primary recommendation:** Add a `gpu_runtime.rs` der-launch family — `launch_der_binary`, `launch_der_param`, `launch_der_unary` generic over `SelectedRuntime` — mirroring the 7.1 `launch_block_reduce_f64` shell (per-call client, `Bytes::from_elems`, typed-error read-back, `CUBE_DIM=32`, ceiling-division launch). Return device **handles** from an internal seam; expose a thin host-readback wrapper only for the self-oracle. Use **one `#[cube]` kernel per loss** (the kernels already exist as separate functions) selected by the host-side `match loss` — NOT a single comptime mega-kernel — because that mirrors upstream's `switch(loss)` template-instantiation structure (D-01) and the kernels are already written that way.

## User Constraints (from CONTEXT.md)

### Locked Decisions

**D-7.2-01 (loss scope):** Port the pointwise derivative family that upstream `pointwise_targets.cu` covers AND that already has a Rust CPU der1/der2 path + an authored `#[cube]` kernel: **RMSE/MSE** (`gradient_kernel`), **Logloss/CrossEntropy** (`logloss_gradient_kernel` + `logloss_hessian_kernel`), **Quantile/MAE** (`quantile_gradient_kernel`, der2≡0), and **Focal** (`focal_gradient_kernel` + `focal_hessian_kernel`). Structurally mirror upstream's dispatch-by-loss shape (`PointwiseTargetKernel` switching on `ELossFunction`). Start narrowest — the remaining pointwise regression losses (LogCosh/Lq/Huber/Expectile/Poisson/Tweedie/MAPE) are additive follow-ons on the same seam.

**D-7.2-02 (weights & functionValue):** Per-object weights are folded into der1/der2 at structural parity with upstream (`PointwiseTargetKernel` takes `weights`). The scalar `functionValue` (loss value) is **NOT computed in this slice** — the histogram path consumes der1/der2, not the loss value.

**D-7.2-03 (parity target):** Parity target is `pointwise_targets.cu(h)` + `pointwise_target_impl.{h,cpp}` — mirror its memory layout, blocking, and accumulation. The `#[cube]` kernel BODIES are done; 7.2 supplies the GPU launch helper + device residency, not the math. der/der2 are elementwise (no cross-object reduction), so D-03's atomic-finalize contract is a no-op here.

**D-7.2-04 (device residency):** der1/der2 stay device-resident — returned as device buffer **handles** (the per-call client + `bytemuck`/`Bytes::from_elems` device-memory pattern from `cpu_runtime.rs` / 7.1's primitives), consumable directly by the 7.3 histogram kernels with no host fold inserted. Define the device-side seam now (handle in → handles out).

**D-7.2-05 (boundary):** All cubecl coupling stays **inside `cb-backend`**. Do NOT widen the frozen `cb-compute` `Runtime`/`Float` traits unless unavoidable. `cb-compute` stays cubecl-free; `cb-core`/`cb-model` stay unchanged from Phase 3–6 form (SC-4).

**D-7.2-06 (self-oracle):** GPU der1/der2 self-oracle against the Rust CPU der path (`cb-compute/src/loss.rs` der1/der2 fns — the parity *baseline*, not the C++ oracle) on representative fixtures: random f32 + f64, non-trivial weights, edge cases (empty, single element, length not a multiple of block/cube size, large N). REPORT observed max abs/rel divergence — do NOT hard-code or sign off the GPU-06 epsilon (7.6's job). Tests in dedicated `_test.rs` files, run on `rocm` in-env.

**D-7.2-07 (verification):** Per-crate, honoring disk-pressure. `rocm` = build + run + self-oracle in-env on gfx1100; `wgpu` = build + run on host; `cuda` = build/compile only. `rocm` GPU tests run locally/manually, never in GitHub Actions.

### Pattern 1: Generalize the der launch helper over `SelectedRuntime` (the load-bearing 7.2 work)
**What:** The existing der launch helpers in `cpu_runtime.rs` hard-code `cubecl::cpu::CpuDevice` / `CpuRuntime`. The 7.1 `gpu_runtime.rs` already proved the generalization: use `<SelectedRuntime as cubecl::Runtime>::Device::default()` and `::client(&device)`, then launch `kernel::launch::<F, SelectedRuntime>(...)`. Mirror that EXACTLY.
**When to use:** Every der/der2 kernel launch in this phase.
**Example:**
```rust
// Source: cb-backend/src/gpu_runtime.rs (7.1 launch_block_reduce_f64) — the proven pattern to mirror
// (handle-returning variant for the histogram hand-off; the test path adds a read_one wrapper)
pub fn launch_der_binary_handle(
    approx: &[f64],
    target: &[f64],
    kernel: DerBinaryKernel,
) -> CbResult<cubecl::server::Handle> {
    let n = approx.len();
    let device = <SelectedRuntime as cubecl::Runtime>::Device::default();
    let client = <SelectedRuntime as cubecl::Runtime>::client(&device);

    let approx_handle = client.create(cubecl::bytes::Bytes::from_elems(approx.to_vec()));
    let target_handle = client.create(cubecl::bytes::Bytes::from_elems(target.to_vec()));
    let out_handle = client.empty(n * std::mem::size_of::<f64>());

    let num_cubes = n.div_ceil(CUBE_DIM).max(1);
    let count = CubeCount::Static(num_cubes as u32, 1, 1);
    let dim = CubeDim { x: CUBE_DIM as u32, y: 1, z: 1 };

    match kernel {
        DerBinaryKernel::RmseGradient => gradient_kernel::launch::<f64, SelectedRuntime>(
            &client, count, dim,
            unsafe { ArrayArg::from_raw_parts(approx_handle, n) },
            unsafe { ArrayArg::from_raw_parts(target_handle, n) },
            unsafe { ArrayArg::from_raw_parts(out_handle.clone(), n) },
        ),
        // ... LoglossGradient arm, etc. — same structure as cpu_runtime::launch_binary_f64
    }
    // For the histogram hand-off: return out_handle DIRECTLY (device-resident, no read-back).
    // For the self-oracle: a sibling wrapper calls client.read_one(out_handle) + maps WR-05 error.
    Ok(out_handle)
}
```
The unsafe `ArrayArg::from_raw_parts` consumes the (Clone) `Handle`; clone the output handle so the original is still readable/returnable after launch — exactly the 7.1 idiom.

### Pattern 2: Parametric kernels pass loss params as length-1 `Array<F>` (already established)
**What:** `quantile_gradient_kernel`, `focal_gradient_kernel`, `focal_hessian_kernel` take `alpha`/`gamma`/`delta` as length-1 `Array<F>` args read at index 0 — NOT scalar kernel args — to keep the kernel fully generic over `F: Float` (a generic scalar arg would require non-generic `F: ScalarArgType` bounds). The GPU launch helper must `client.create(Bytes::from_elems(vec![param]))` a length-1 device buffer per param.
**When to use:** Quantile/MAE and Focal launches.
**Example:** See `cpu_runtime::launch_focal_f64` / `launch_quantile_f64` for the existing pattern to generalize (they build `alpha_handle`/`gamma_handle` as length-1 arrays). Reuse verbatim, swapping `CpuRuntime` → `SelectedRuntime`.

### Pattern 3: Per-loss dispatch via host-side `match` (mirrors upstream `switch(ELossFunction)`)
**What:** Upstream `PointwiseTargetKernel` is a host-side `switch(loss)` that instantiates a per-loss target functor (`TRmseTarget`, `TQuantileTarget`, ...) and launches the SAME `PointwiseTargetImpl<TTarget>` template. The Rust analog is a host-side `match loss { Loss::Rmse => launch gradient_kernel, ... }` selecting the matching pre-written `#[cube]` function. This is D-01 structural parity AND already how `cpu_runtime::compute_gradients_one_dim` is written — extend that match, do not invent a comptime mega-kernel.
**When to use:** The dispatch entry point.

### Anti-Patterns to Avoid
- **Single comptime mega-kernel** selecting the loss formula inside one `#[cube]` fn: contradicts upstream's per-functor template structure (D-01), forces a unified arg signature across losses with different parametrics, and discards working oracle-passing kernels. Use per-loss kernels + host `match`.
- **Returning host `Vec<f64>` from the histogram hand-off path:** forces a device→host→device round-trip, violating SC-3 / D-7.2-04. Return device handles; read back only inside the self-oracle test.
- **Widening the `cb-compute` `Runtime`/`Float` trait** to carry a cubecl handle type: violates D-7.2-05 (cb-compute stays cubecl-free). Keep the handle seam backend-internal.
- **Hard-coding `CpuRuntime`/`CpuDevice` in the new GPU helpers:** the whole point is generality over `SelectedRuntime`. The cpu-only spike (`kernels/gradient.rs`) may name `CpuRuntime` (it's a cpu-gated test); production GPU helpers must not.
- **Folding weight inside the der kernel** to match upstream's `der[i] = weight * Der(...)`: the Rust CPU baseline (`compute_gradients`) returns UNWEIGHTED der (weight applied downstream in `boosting.rs`). The GPU kernel must match its oracle baseline, not upstream's in-kernel weighting. See Pitfall 1.
- **Warp/wave-size literals (32/64)** in any stride: D-09. (The der kernels are elementwise and use no strided reduction, so this is naturally satisfied — `CUBE_DIM=32` is launch geometry, not a reduction stride.)

## Common Pitfalls

### Pitfall 1: Weight folding — GPU kernel must match the CPU *baseline*, not upstream's in-kernel weighting
**What goes wrong:** Upstream `PointwiseTargetImpl` computes `der[i] = weight * target.Der(relev, val)` — weight folded IN-KERNEL. D-7.2-02 says "weights folded at structural parity with upstream." But the Rust CPU baseline the GPU is oracled against (`cb-compute::loss::rmse_der1` etc., called via `compute_gradients`) returns **UNWEIGHTED** der; the trait `compute_gradients(loss, approx, target, dim)` takes **no weights at all**. Weight is applied LATER in `cb-train/boosting.rs` (`der1 * weight`, confirmed at `boosting.rs:2939`/`1519`). If the GPU kernel folds weight in but the baseline doesn't, the self-oracle diverges by exactly the weight factor.
**Why it happens:** The CONTEXT wording mirrors upstream's signature; the actual Rust seam differs (weight is a downstream concern, not a `compute_gradients` input).
**How to avoid:** The 7.2 GPU der kernels produce **unweighted** der1/der2, byte-identical structure to the CPU baseline. Document that "weight folding at structural parity" is satisfied by the SAME downstream `der1 * weight` the CPU path uses — the kernel signature does not gain a `weights` array in this slice. If the planner wants literal in-kernel weighting for 7.3's benefit, that is a separate, explicit decision (the histogram phase needs `der1*weight` and `weight` per object) — confirm with the user before adding a weights arg, since it would force the self-oracle baseline to also weight. **Recommendation: keep der1/der2 unweighted in 7.2 to match the existing baseline; weight folding stays downstream.** `[ASSUMED — see Assumptions A1]`
**Warning signs:** Self-oracle divergence proportional to a non-unit weight vector; an oracle that passes only when all weights are 1.0.

### Pitfall 2: Returning host `Vec` defeats the device-residency deliverable (SC-3)
**What goes wrong:** The existing `compute_gradients` reads der1/der2 back to host `Vec<f64>`. If the 7.2 histogram hand-off reuses that path, der1/der2 go device→host and 7.3 must re-upload them — the exact host fold SC-3 / D-7.2-04 forbids.
**Why it happens:** Copy-pasting the cpu_runtime launch helper, which ends in `client.read_one(...)`.
**How to avoid:** Add a handle-returning internal seam (`launch_der_*_handle -> CbResult<Handle>`) that does NOT read back. The trait's `compute_gradients` may keep its `Vec` return for the boosting loop's CPU consumers; the histogram seam uses the handle path. Read back ONLY in the self-oracle test.
**Warning signs:** A `read_one` call on the histogram-feeding path; a `Vec<f64>` crossing between the der step and the histogram step.

### Pitfall 3: f64 atomic-add is unavailable on gfx1100/HIP at cubecl 0.10.0 — but der kernels don't need it
**What goes wrong:** A planner might assume the der path needs the D-03 atomic-finalize like the reduce primitive. It does not — der/der2 are elementwise (no cross-object reduction). 7.1 confirmed gfx1100/HIP at cubecl 0.10.0 LACKS f64 atomic-add (the `block_reduce_atomic` test took the `HostSumFallback` path; `run_to_run_spread=0.000e0`).
**Why it happens:** Conflating the elementwise der step with the histogram reduction step.
**How to avoid:** der kernels use NO atomics and NO planes — pure per-element writes. D-03's contract is explicitly a no-op for this slice (D-7.2-03). The atomic question returns in 7.3/7.4 (histogram accumulation).
**Warning signs:** Any `Atomic<F>` or `fetch_add` in a 7.2 der kernel; a plane op where none is needed.

### Pitfall 4: Non-cube-multiple lengths must idle surplus threads (bounds guard)
**What goes wrong:** Ceiling-division launch (`n.div_ceil(32)`) over-provisions threads; without the `if ABSOLUTE_POS < approx.len()` guard, surplus threads read/write out of bounds.
**Why it happens:** Forgetting the guard when wiring a new kernel.
**How to avoid:** Every existing der kernel ALREADY has the `if ABSOLUTE_POS < approx.len()` guard (verified in `kernels.rs`). The self-oracle must include a non-multiple-of-32 length (e.g., n=37, as the `kernels/gradient.rs` f64 spike already does) to exercise the idle path.
**Warning signs:** A test that only uses lengths that are multiples of 32; an out-of-bounds device error on odd lengths.

### Pitfall 5: Quantile/MAE/MAPE der2 is constant 0 — there is no hessian kernel to launch
**What goes wrong:** Trying to launch a `quantile_hessian_kernel` that doesn't exist (der2 ≡ 0 for Quantile/MAE/MAPE).
**Why it happens:** Assuming every loss has a der2 kernel.
**How to avoid:** For Quantile/MAE the dispatch fills `der2 = vec![0.0; n]` host-side (the `cpu_runtime::compute_gradients_one_dim` precedent: `Loss::Mae`/`Loss::Quantile` → `der2 = vec![0.0_f64; ...]`). For the device-handle path, allocate a zeroed device buffer (`client.create(Bytes::from_elems(vec![0.0_f64; n]))`) so 7.3 still receives a der2 handle. RMSE der2 is constant `-1.0` (no kernel either — fill a `-1.0` device buffer).
**Warning signs:** A missing-kernel compile error for `quantile_hessian_kernel`; an unhandled der2 path for RMSE/Quantile/MAE.

### Pitfall 6: Logloss/CrossEntropy share the EXACT der path — one kernel, two enum arms
**What goes wrong:** Writing a separate CrossEntropy kernel. Upstream has a dedicated `CrossEntropyImpl` (with `ELEMENTS_PER_THREAD=2` blocking and an optional border), but the Rust seam already collapses `Loss::Logloss | Loss::CrossEntropy` to the SAME `logloss_gradient_kernel` + `logloss_hessian_kernel` (D-09, confirmed in `compute_gradients_one_dim`). For this slice (no border, no functionValue), one kernel serves both.
**Why it happens:** Following upstream's two-kernel split literally.
**How to avoid:** Map both `Loss::Logloss` and `Loss::CrossEntropy` to the logloss grad/hess kernels (the established baseline). The upstream `CrossEntropyImpl` border / 2-elements-per-thread blocking is a perf/feature variant out of scope here (no border in the in-scope losses).
**Warning signs:** A duplicate CrossEntropy kernel; a `border` arg appearing in this slice.

### Self-oracle: GPU der vs CPU baseline, reporting divergence (mirror reduce/scan oracle)

// Source: cb-backend/src/kernels/reduce.rs + scan.rs (the established D-7.x-09 oracle shape)
// New file: kernels/gradient_gpu.rs — runs over SelectedRuntime (rocm in-env / wgpu host)
fn max_divergence(device: &[f64], baseline: &[f64]) -> (f64, f64) {
    let mut max_abs = 0.0_f64;
    let mut max_rel = 0.0_f64;
    for i in 0..baseline.len() {
        let abs = (device[i] - baseline[i]).abs();
        let rel = if baseline[i].abs() > 0.0 { abs / baseline[i].abs() } else { abs };
        max_abs = max_abs.max(abs);
        max_rel = max_rel.max(rel);
    }
    (max_abs, max_rel)
}

## Open Questions

1. **Does the 7.3 histogram seam consume `der1` or `der1 * weight`?**
   - What we know: upstream histograms accumulate the weighted der; the Rust `histogram_scatter_kernel` ALREADY computes `contrib[i] = der1[i] * weight[i]` (a separate kernel). The CPU `reduce_leaf_stats` folds `der1` (weight pre-folded in the unweighted path per its doc).
   - What's unclear: whether 7.2 should hand 7.3 an unweighted `der1` handle + a `weight` handle (letting `histogram_scatter_kernel` do the multiply), or a pre-weighted `der1*weight` handle.
   - Recommendation: Hand 7.3 the **unweighted der1/der2 handles** (matching the `compute_gradients` baseline and the self-oracle), plus reuse the existing `histogram_scatter_kernel` for the weight multiply in 7.3. This keeps 7.2's self-oracle clean (no weight in the baseline) and defers weight folding to where the histogram already does it. Planner should confirm this contract with the 7.3 plan.

2. **Exact cubecl 0.10.0 `Handle` type for the device-residency seam.**
   - What we know: 7.1 returns/holds `client.empty(...)` / `client.create(...)` handles and reads them with `client.read_one(handle)`. The handle is `Clone`.
   - What's unclear: the precise type name to put in the internal seam signature (`cubecl::server::Handle` vs a runtime-associated type).
   - Recommendation: Inspect the return type of `client.empty(...)` in the 7.1 code at implementation time and name the seam after it; keep the seam `pub(crate)` inside `cb-backend` (D-7.2-05). Low risk — the handle already flows correctly in 7.1.

### Known Threat Patterns for GPU kernel code
| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Out-of-bounds device read/write on non-cube-multiple lengths | Tampering / DoS | `if ABSOLUTE_POS < len` bounds guard (already in every kernel); CR-02-style `CUBE_DIM <= shmem` const assert where shared memory is used (not needed for elementwise der) |
