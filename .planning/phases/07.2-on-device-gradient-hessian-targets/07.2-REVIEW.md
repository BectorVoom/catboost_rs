# Phase 07.2: Code Review Report

**Reviewed:** 2026-06-20T00:00:00Z
**Depth:** standard
**Files Reviewed:** 3
**Status:** issues_found

## Summary

Reviewed the Phase-7.2 device-resident der1/der2 launch seam (`gpu_runtime.rs`,
production), the `#[cube]` kernels it dispatches (`kernels.rs`, production), and the
all-backend self-oracle (`gradient_gpu.rs`, `#[cfg(test)]`).

The launch geometry is generally well-guarded: the `CUBE_DIM <= BLOCK_REDUCE_SHMEM`
compile-time coupling (gpu_runtime.rs:76-80) is a real OOB guard, the `n > CUBE_DIM`
scan precondition is enforced with a typed error instead of returning a silently-wrong
result (gpu_runtime.rs:168-173), `param_pair()` is non-panicking (gpu_runtime.rs:601-612),
both per-loss `match` blocks are exhaustive over their enums, and the length-mismatch /
empty-handle / read-back-failure paths all surface typed `CbError`s rather than panicking
or producing a masquerading all-zero buffer. No `unwrap`/`expect`/`panic`/indexing appears
in the production files; the test file's `.unwrap()` usage is permitted by the `lib.rs:1`
`#[cfg(test)]` allow.

The findings below are correctness/robustness concerns rather than crashes or security
holes. The most material is WR-01: a kernel-vs-baseline boundary disagreement in the
Quantile/MAE gradient at `|val| == delta` that the kernel's own comment defends but that
contradicts the `cb-compute` baseline it is oracle-tested against. WR-02 and WR-03 are
launch-geometry integer-overflow concerns that are unreachable at current fixture sizes
but are not guarded. No Critical issues were found.

## Warnings

### WR-01: Quantile/MAE gradient kernel disagrees with the CPU baseline at the deadzone boundary `|val| == delta`

**File:** `crates/cb-backend/src/kernels.rs:88-101` (and the baseline `crates/cb-compute/src/loss.rs:166-175`)

**Issue:** The CPU baseline the self-oracle compares against returns `0` for the **closed**
deadzone `|val| < delta` (strict `<`), so at the exact boundary `|val| == delta` the
baseline returns `0`:

```rust
// cb-compute/src/loss.rs:166-175
if val.abs() < delta { 0.0 }
else if val > 0.0 { alpha }
else { -(1.0 - alpha) }
```

The kernel instead enters the **non-zero** branch on `F::abs(val) >= d`, so at
`|val| == delta` it returns the signed quantile weight (`alpha` or `-(1-alpha)`), NOT `0`:

```rust
// kernels.rs:94-101
let mut g = F::new(0.0);
if F::abs(val) >= d {                       // boundary |val| == d ENTERS here
    g = F::new(0.0) - (one - a);
    if val > F::new(0.0) { g = a; }
}
der1[ABSOLUTE_POS] = g;
```

This is a genuine device-vs-baseline parity divergence at `|val| == delta`. The kernel
comment (lines 88-93) explicitly asserts the `>= delta` boundary "returns the signed
quantile weight ... NOT 0", but that directly contradicts `quantile_der1`'s `< delta`
deadzone — the kernel and its oracle baseline disagree on the boundary value. Note that
the sibling `huber_gradient_kernel` (kernels.rs:316, `>= d` enters the *saturated* branch)
**is** consistent with `huber_der1` (`< delta` enters the *in-band* branch) because for
Huber the two branches are complementary around `< delta`; for Quantile the two are NOT
complementary (the deadzone is the `< delta` side), so the kernel's `>= d` flips the
boundary element relative to the baseline. The divergence is measure-zero in random
fixtures (hence the oracle passes), but it is a real correctness defect that will silently
produce a different gradient than `cb-compute` for any object landing exactly on the
boundary (e.g. `delta` chosen as an exact binary fraction with crafted residuals).

**Fix:** Make the kernel boundary match the baseline — use the strict complement so
`|val| == delta` stays in the deadzone (`g = 0`):

```rust
let mut g = F::new(0.0);
// Match cb-compute quantile_der1: deadzone is |val| < delta (closed at the boundary -> 0).
if F::abs(val) > d {                        // strict '>' so |val| == d stays 0
    g = F::new(0.0) - (one - a);
    if val > F::new(0.0) { g = a; }
}
der1[ABSOLUTE_POS] = g;
```

and correct the now-inaccurate comment at lines 88-93. (Alternatively, if the `>= delta`
semantics are intended to be authoritative, fix `cb-compute::quantile_der1` and re-derive
the upstream parity — but the kernel must not silently diverge from the baseline it is
oracle-tested against.)

### WR-02: Unchecked `usize -> u32` truncation of `num_cubes` in every launch helper

**File:** `crates/cb-backend/src/gpu_runtime.rs:114, 185, 269, 388, 511, 667` (`CubeCount::Static(num_cubes as u32, 1, 1)`)

**Issue:** `num_cubes = n.div_ceil(CUBE_DIM).max(1)` is a `usize`, cast to `u32` with `as`.
For `n > u32::MAX * CUBE_DIM` (≈ 1.37e11 elements) `num_cubes` exceeds `u32::MAX` and the
`as u32` silently truncates, launching far too few cubes and producing a silently-truncated
der buffer (the tail objects never get a thread; their `out_handle` slots stay
uninitialized device memory). This is the same masquerading-wrong-result class the file
otherwise guards against (the `read_one` typed-error path, the scan `n > CUBE_DIM` guard).
It is unreachable at current fixture sizes but is unguarded production code on a hot path
that explicitly markets "large N" support.

**Fix:** Reject the overflow with a typed error instead of truncating, e.g. a small helper
reused by all six launch sites:

```rust
let num_cubes_u32 = u32::try_from(num_cubes).map_err(|_| {
    CbError::Degenerate(format!(
        "launch geometry overflow: {num_cubes} cubes exceeds u32::MAX (n = {n})"
    ))
})?;
let count = CubeCount::Static(num_cubes_u32, 1, 1);
```

### WR-03: Unchecked `n * size_of::<f64>()` byte-size multiplication can overflow

**File:** `crates/cb-backend/src/gpu_runtime.rs:108, 183, 385, 508, 665` (`client.empty(n * std::mem::size_of::<f64>())` / `num_cubes * std::mem::size_of::<f64>()`)

**Issue:** The output-buffer byte size is computed as `n * 8` (or `num_cubes * 8`) in plain
`usize` arithmetic with no overflow check. On a 32-bit target this overflows for
`n > usize::MAX / 8` and (in release) wraps to a too-small allocation, after which the
kernel writes `der1[ABSOLUTE_POS]` past the end of the device buffer (device-side OOB). On
64-bit this is not reachable in practice, but the multiplication is unguarded and the
allocation size is load-bearing for memory safety. Pairs with WR-02 (same "geometry math
isn't range-checked" root).

**Fix:** Use checked multiplication and surface a typed error:

```rust
let out_bytes = n.checked_mul(std::mem::size_of::<f64>()).ok_or_else(|| {
    CbError::Degenerate(format!("output byte size overflow for n = {n}"))
})?;
let out_handle = client.empty(out_bytes);
```

### WR-04: `block_reduce_kernel` / `block_scan_kernel` fallback tree-reduce assumes `CUBE_DIM_X` is a power of two

**File:** `crates/cb-backend/src/kernels.rs:622-633` (`block_reduce_kernel` fallback), `kernels.rs:709-720` (`block_reduce_atomic_kernel` fallback)

**Issue:** The shared-memory tree reduction starts the stride at `s = CUBE_DIM_X / 2u32`
and halves (`s /= 2`) until `s == 0`. This Hillis-Steele/Blelloch halving only sums every
element when `CUBE_DIM_X` is a power of two. `CUBE_DIM_X` here equals the launch `CubeDim.x`,
which is pinned to `CUBE_DIM = 32` (a power of two) by every launch site, so it is correct
**today**. But the kernel takes `CUBE_DIM_X` as a runtime value and contains no
`debug_assert`/comptime guard that it is a power of two; if a future launch ever passes a
non-power-of-two cube width (e.g. 48), the `s /= 2` ladder silently drops the high
partials and returns a wrong sum with no error. This is a latent correctness trap in a
primitive that is explicitly documented as the parity-critical intra-cube fold.

**Fix:** Either document and enforce the power-of-two precondition at the launch seam (a
`const _: () = assert!(CUBE_DIM.is_power_of_two())` next to the existing
`CUBE_DIM <= BLOCK_REDUCE_SHMEM` assert in gpu_runtime.rs), or make the fallback reduction
round the initial stride up to the next power of two and bounds-check the partner index so
non-power-of-two widths still sum correctly. The compile-time assert is the cheaper fix and
matches the existing CR-02 coupling pattern.

## Info

### IN-01: `device_supports_f64_atomic_add` only checked in the atomic helper, not the comptime-selected plane path

**File:** `crates/cb-backend/src/gpu_runtime.rs:214-222, 112, 267`

**Issue:** `use_plane` is selected from `client.features().plane.contains(Plane::Ops)` and
the f64-atomic capability is queried separately in `launch_block_reduce_atomic_f64`. These
are independent queries; there is no single struct capturing "device capability profile,"
so a future kernel that needs both could drift (query one but not the other). Not a bug
today — purely a maintainability note. Consider a small `DeviceCaps { use_plane, f64_atomic }`
resolved once per client.

### IN-02: `max_divergence` test helper indexes `device[i]` assuming equal lengths

**File:** `crates/cb-backend/src/kernels/gradient_gpu.rs:31-45`

**Issue:** `max_divergence` iterates `0..baseline.len()` and indexes `device[i]`; if a
regression ever made the device buffer SHORTER than the baseline, this panics with an index
OOB instead of a descriptive assertion failure. Every caller does assert
`device.len() == baseline.len()` first, so this is test-only and currently safe (and
indexing is allowed in test code), but a `device.len() == baseline.len()` guard inside the
helper would turn a future truncation bug into a clear message instead of a panic in an
unrelated helper.

### IN-03: `cb_compute_is_cubecl_free` test does not actually verify the dependency graph

**File:** `crates/cb-backend/src/kernels/gradient_gpu.rs:936-954`

**Issue:** The test name and SC-4 framing imply it gates "cb-compute stays cubecl-free,"
but the body only calls two `cb_compute` functions and asserts they return finite values —
it cannot fail if cb-compute *did* pull in cubecl. The test's own comment is honest about
this (the authoritative gate is the `cargo tree` command in verification), so this is not a
correctness defect, but the test name over-promises relative to what it asserts. Consider
renaming to `cb_compute_der_baselines_are_pure_host_fns` so the assertion matches the name,
keeping the `cargo tree` check as the real SC-4 gate.
