# Phase 07.2: Code Review Report

**Reviewed:** 2026-06-20T09:24:46Z
**Depth:** standard
**Files Reviewed:** 3
**Status:** issues_found

## Summary

Reviewed the Phase 7.2 device-resident gradient/hessian der seam across three files:
`gpu_runtime.rs` (the `launch_der_*` launch helpers + the 7.1 reduce/scan and 7.3
histogram seams that share the module), `kernels.rs` (the `#[cube]` elementwise der
kernels), and `gradient_gpu.rs` (the all-backend self-oracle, a test file).

Correctness of the der math is sound. I traced every Phase-7.2 kernel against its
`cb-compute::loss` baseline:
- `gradient_kernel` (RMSE der1 = `target - approx`) — matches `rmse_der1`.
- `logloss_gradient_kernel` / `logloss_hessian_kernel` — sigmoid gradient/hessian
  match `logloss_der1`/`logloss_der2` structurally.
- `quantile_gradient_kernel` — the previously-deferred WR-01 `|val| == delta`
  boundary bug is **fixed**: the kernel's `F::abs(val) >= d` band is the exact
  complement of the CPU baseline's `val.abs() < delta` deadzone, so the boundary
  returns the signed quantile weight in both paths. MAE-as-Quantile{0.5,1e-6}
  parity holds.
- `focal_gradient_kernel` / `focal_hessian_kernel` — the `u*dv + du*v` chain and
  the `clamp(p, 1e-13, 1-1e-13)` saturation guard are transcribed faithfully from
  `focal_der1`/`focal_der2`; the saturated-logit (`±40`) finiteness path is exercised.

The compile-time guards (power-of-two `CUBE_DIM`, `CUBE_DIM <= BLOCK_REDUCE_SHMEM`,
histogram shmem coupling), the host-side length/overflow/value-range guards in the
7.3 histogram seam, and the "no `unwrap`/panic/indexing in production" discipline are
all upheld. No security issues, no injection/eval/unsafe-deserialization surface
(the `unsafe` blocks are all CubeCL `ArrayArg::from_raw_parts` launch ABI, correctly
length-matched to the allocated handles).

The findings below are robustness/consistency defects, not parity breaks: two sibling
public entry points disagree on a malformed-input contract, and an `AtomicFinalizePath`
enum value is reported with a name that contradicts the path actually taken.

## Warnings

### WR-01: `launch_der_binary` / `launch_der_param` accept a length-mismatched (empty-approx, non-empty-target) input that the sibling `*_handle` entry points reject

**File:** `crates/cb-backend/src/gpu_runtime.rs:501-503` (binary), `:813-815` (param)
**Issue:**
The host-readback wrappers short-circuit on `approx.is_empty()` and return
`Ok(Vec::new())` **before** any length validation:

```rust
pub fn launch_der_binary(approx: &[f64], target: &[f64], kernel: DerBinaryKernel) -> CbResult<Vec<f64>> {
    if approx.is_empty() {
        return Ok(Vec::new());   // <- returns Ok even if target = [x, y, ...]
    }
    ...
    let handle = launch_der_binary_into(&client, approx, target, kernel)?;
```

The handle entry point, by contrast, routes straight through `launch_der_binary_into`,
whose first action is the shape guard `if target.len() != n { return Err(LengthMismatch{..}) }`
(`:431-437`). So for the malformed input `(approx = [], target = [1.0])`:
- `launch_der_binary` → `Ok([])` (silently swallows the mismatch),
- `launch_der_binary_handle` → `Err(CbError::LengthMismatch{expected:0, actual:1})`.

The two public entry points for the SAME operation disagree on the contract. A caller
that validates via the readback wrapper and then hands off via the handle path (the
documented 7.3 seam) would see an inconsistent accept/reject. The same asymmetry exists
in `launch_der_param` vs `launch_der_param_handle`.
**Fix:** Validate length before the empty short-circuit, mirroring `*_into`, so both
entry points agree:

```rust
pub fn launch_der_binary(approx: &[f64], target: &[f64], kernel: DerBinaryKernel) -> CbResult<Vec<f64>> {
    if target.len() != approx.len() {
        return Err(CbError::LengthMismatch {
            column: "target".to_owned(),
            expected: approx.len(),
            actual: target.len(),
        });
    }
    if approx.is_empty() {
        return Ok(Vec::new());
    }
    ...
}
```

Apply the equivalent guard in `launch_der_param` (`:813`). (`launch_der_unary` takes a
single array, so it is not affected.)

### WR-02: `launch_pointwise_hist2` reports `AtomicFinalizePath::HostSumFallback` for a run that executed the in-kernel f64 atomic merge — a name that contradicts the path taken

**File:** `crates/cb-backend/src/gpu_runtime.rs:1338-1342` (and the enum at `:71-79`)
**Issue:**
On the histogram path the in-kernel `Atomic<F>::fetch_add` merge ALWAYS runs (there is
no host-sum fallback — the doc at `:923-927` says so explicitly). Yet the reported path
is derived from the device's *advertised* capability:

```rust
let path = if device_supports_f64_atomic_add(&client) {
    AtomicFinalizePath::InKernelAtomicF64
} else {
    AtomicFinalizePath::HostSumFallback   // <- but NO host sum ran; the kernel atomic-merged
};
```

On gfx1100 (the in-env oracle target) `device_supports_f64_atomic_add` returns `false`
even though HIP runs the f64 atomic, so this path reports `HostSumFallback` for a fill
that was finalized entirely on-device. `AtomicFinalizePath::HostSumFallback` is documented
elsewhere (`:76-78`) as "the portable Plan-01 shared-mem-partial + host `cb-core::sum_f64`
finalize ran instead" — which is categorically NOT what happened here. A consumer that
inspects `path == HostSumFallback` to decide whether a deterministic host sum occurred
(the contract the enum advertises, and that `launch_block_reduce_atomic_f64` actually
honors at `:299-353`) would draw the wrong conclusion. This is a misleading-API defect:
the same enum value means two different things depending on which helper returned it.
**Fix:** Do not overload `HostSumFallback` to mean "device advertised no f64 atomic but
the kernel atomic-merged anyway." Either return a distinct value for the
informational-capability report (e.g. add an `InKernelAtomicF64Unadvertised` variant or a
separate `AtomicCapability { advertised: bool }` type), or have `launch_pointwise_hist2`
report `InKernelAtomicF64` unconditionally (since that path always runs the in-kernel
atomic) and surface the advertised-capability bit separately. At minimum, rename the
histogram report so it cannot collide with the reduce helper's genuine fallback semantics.

### WR-03: `max_divergence` test helper indexes `device[i]` over `baseline.len()` without a length precondition, unlike the IN-03-fixed reduce/scan reporters

**File:** `crates/cb-backend/src/kernels/gradient_gpu.rs:31-45`
**Issue:**
```rust
fn max_divergence(device: &[f64], baseline: &[f64]) -> (f64, f64) {
    ...
    for i in 0..baseline.len() {
        let abs = (device[i] - baseline[i]).abs();   // <- device[i] unchecked
```
If `device.len() < baseline.len()` this panics with an index-out-of-bounds inside the
reporter rather than surfacing a clear length mismatch. The commit history for the sibling
files shows this exact pattern was deliberately hardened in 7.1/7.3 (`IN-02`/`IN-03`:
"zip device/baseline in max_divergence for structural length precondition"), but this
7.2 copy was left in the raw-index form (its own doc comment at `:29-30` even says it was
"Copied verbatim from the `kernels::scan` reporter" — i.e. copied from the pre-fix
version). Every current caller asserts `device.len() == approx.len()` first, so it does
not misfire today; the risk is a future caller (or a der that returns a short buffer on a
backend bug) producing an opaque panic instead of a diagnostic. This affects test
diagnosability/reliability, which is in-scope for test files.
**Fix:** Match the hardened sibling reporters — zip and assert the precondition:

```rust
fn max_divergence(device: &[f64], baseline: &[f64]) -> (f64, f64) {
    assert_eq!(device.len(), baseline.len(), "max_divergence requires equal-length slices");
    let mut max_abs = 0.0_f64;
    let mut max_rel = 0.0_f64;
    for (&d, &b) in device.iter().zip(baseline) {
        let abs = (d - b).abs();
        let rel = if b.abs() > 0.0 { abs / b.abs() } else { abs };
        max_abs = max_abs.max(abs);
        max_rel = max_rel.max(rel);
    }
    (max_abs, max_rel)
}
```

## Info

### IN-01: `all_losses_device_resident_handoff` constructs a `device` binding only to discard it with `let _ = device`

**File:** `crates/cb-backend/src/kernels/gradient_gpu.rs:850`, `:933`
**Issue:** `let device = <…>::Device::default();` is built at `:850`, never used for a
read-back (each block calls the local `read_handle` fn which builds its own device/client
at `:855-856`), and is finally silenced with `let _ = device;` at `:933`. The trailing
comment rationalizes it as "documenting the single-runtime hand-off surface," but it is
dead state kept alive only to dodge an unused-variable warning.
**Fix:** Delete the `:850` binding and the `:933` `let _ = device;`; the `read_handle`
helper already owns its device/client. The intent comment can stay as a plain comment.

### IN-02: Duplicated per-family launch geometry/upload block across the binary / half-byte histogram arms

**File:** `crates/cb-backend/src/gpu_runtime.rs:1069-1169` (binary + half-byte branches)
**Issue:** The binary and half-byte branches each repeat the same four-step block
(`hist2_binsums_len` → create cindex/indices handles → `#[cfg(wgpu)]` f32 vs
`#[cfg(not)]` f64 create+launch+`return Ok(h)`), differing only in the kernel name. The
module doc claims a single launch geometry (IN-02), and the `count`/`dim` geometry is
indeed shared, but the upload+launch+return boilerplate is triplicated (binary,
half-byte, non-binary). This is maintenance-fragile: a fix to the channel-dispatch or
the zero-init must be applied in three places.
**Fix:** Extract a small helper that the binary and half-byte arms (and ideally the
non-binary arm) call, so the channel-float `#[cfg]` split and the zero-init live in
exactly one place. Not a correctness issue — flagged for maintainability only.

### IN-03: Module-level doc inventory in `gpu_runtime.rs` still describes the planned `gpu_runtime/der.rs` + `gpu_runtime/histogram.rs` split as not-yet-done

**File:** `crates/cb-backend/src/gpu_runtime.rs:13-27`
**Issue:** The header notes "a single-responsibility split ... is the planned
follow-up; until then this doc is the authoritative inventory." The file is now ~1350
lines hosting three phases' seams (reduce/scan, der, histogram). This is acknowledged
technical debt, not a defect, but the single-file growth is large enough that the split
should be scheduled rather than left as an open-ended "planned follow-up." No code change
required this phase.
**Fix:** Track the `gpu_runtime/{der,histogram}.rs` split as an explicit follow-up item
so the inventory comment does not become permanently load-bearing.
