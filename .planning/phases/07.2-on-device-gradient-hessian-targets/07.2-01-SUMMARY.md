## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Same-client launch+read-back for the host-readback wrapper**
- **Found during:** Task 3 (running the rocm der oracle)
- **Issue:** The initial `launch_der_binary` constructed a NEW client to read back the handle returned by `launch_der_binary_handle` (which had its own client). On rocm/HIP this aborted in the HIP IO controller: `slice::from_raw_parts requires the pointer to be aligned and non-null` — a CubeCL `Handle` is bound to its originating client's allocator/stream, so reading it through a second freshly-constructed client is undefined. The canonical CubeCL idiom (basic-operations manual, the 7.1 `launch_block_reduce_f64`) keeps ONE client per op through read-back.
- **Fix:** Extracted the launch into a private `launch_der_binary_into(client, ...)` helper. `launch_der_binary_handle` constructs its client and returns the Handle; `launch_der_binary` constructs ONE client and uses it for BOTH the launch and the read-back. Single launch geometry preserved.
- **Files modified:** crates/cb-backend/src/gpu_runtime.rs
- **Verification:** rocm gfx1100 oracle 4/4 green; reported divergence 0.0e0.
- **Committed in:** a8f0f7a (Task 3 commit)
- **CubeCL guideline:** consulted `/home/user/Documents/workspace/cubecl_manual/manual/cubecl/Cubecl_basic_operations.md` (the canonical one-client launch+read pattern) before applying the fix, per AGENTS.md.

**2. [Rule 1 - Bug] Do not read back empty (zero-length) device handles in the oracle**
- **Found during:** Task 3 (the `rmse_der_edge_cases` empty case)
- **Issue:** Reading back a `const_der_handle(-1.0, 0)` (zero-length HIP buffer) via `client.read_one` aborted with the SAME `slice::from_raw_parts` null-pointer precondition (an empty device buffer has a null/zero-length pointer). Each test passed individually but the empty read-back aborted the process.
- **Fix:** The empty edge case now asserts `const_der_handle(-1.0, 0).is_ok()` (the handle CONSTRUCTS) WITHOUT reading the empty buffer back — mirroring the production seam, which short-circuits and never reads an empty handle. The empty der1 is still asserted empty via the host wrapper (which short-circuits before any device read).
- **Files modified:** crates/cb-backend/src/kernels/gradient_gpu.rs
- **Verification:** rocm gfx1100 full suite 4/4 green.
- **Committed in:** a8f0f7a (Task 3 commit)

## Issues Encountered
- The HIP IO controller's `slice::from_raw_parts` precondition fires on (a) cross-client handle reads and (b) empty-buffer reads. Both surfaced as non-unwinding aborts (SIGABRT), not catchable errors. Resolved by the two Rule-1 fixes above; the production read-back path now uses the originating client and never reads empty buffers.
