This document records the single signed-off GPU tolerance epsilon for the

## 1. The signed-off epsilon

> **ε = 1e-4** (single global GPU-06 gate, vs the **Rust CPU path**).

This is the **human-signed-off** value (D-7.6-07). Auto-mode captured only the
methodology for proposing it (D-7.6-01..04); the number itself was signed off by
the user during execution, and only the signed-off value is recorded here.

### Derivation (D-7.6-04)

`ε = roundUp(clean value strictly looser than the CPU path's 1e-5, over (observed_max + 3σ))`

- **observed_max** across ALL families = **7.451e-9** (the `score_split` family;
  see §5). Every other measured family was bit-exact (0.000e0).
- **3σ run-to-run variance** = **0.0** (the N=32 atomic-finalize variance loop
  measured stddev = 0.0 on this hardware; see §5 `reduce`).
- `observed_max + 3σ` = **7.451e-9**.
- Rounding UP to a clean value **strictly looser than 1e-5** (D-04) → **1e-4**.

ε = 1e-4 therefore sits **~5 orders of magnitude above** the observed global max
divergence (7.451e-9) and one clean order **looser** than the CPU path's own
1e-5 parity bar.

### The bit-exact-vs-conservative tension (deliberately preserved)

Every sub-phase self-oracle (7.1–7.5) and this phase's evidence harness observed
divergence at or near **bit-exact (0.000e0 to ~7.451e-9)** — far tighter than
D-03/D-04 anticipated when they predicted device atomics would force a
looser-than-1e-5 epsilon. We nonetheless record ε = 1e-4 (looser than 1e-5)
**despite** the bit-exact observations, because:

- **(a)** D-04 mandates an epsilon strictly looser than the CPU path's 1e-5;
- **(b)** observed bit-exactness may NOT hold for losses/fixtures not yet
  measured at the model level, for larger fixtures with greater atomic
  contention, or on **wave64** hardware (see §3);
- **(c)** headroom protects the gate from future flakiness (run-to-run
  float-summation-order variance from device atomics).

The observed-vs-signed-off gap is intentional, informed, and documented so the
gate has durable headroom rather than tracking a possibly-fragile bit-exact
observation.

## 2. Measured vs the Rust CPU path — NOT the C++ oracle (D-04)

The GPU-06 epsilon is measured across exactly one seam: the `cb-backend`
`SelectedRuntime` **rocm** arm vs the **Rust** `CpuRuntime` / `cb-core::sum_f64`
ordered-reduction baseline. It is **NOT** measured against the original C++
CatBoost CPU oracle. GPU-06 certifies that the Rust GPU path reproduces the Rust
CPU path within ε; the Rust-CPU-vs-C++-oracle parity (≤ 1e-5) is a separate,
already-established bar owned by the CPU phases.

## 3. Validation scope: wave32-native; wave64 gap explicitly NOT closed (D-09)

- **In scope (validated):** gfx1100 (RDNA3), **wave32-native** execution. All
  evidence in §5 was produced on this hardware.
- **Explicit, documented gap (NOT closed here):** **wave64** (CDNA / MI-series)
  execution. The kernel design stays wave-size-agnostic (D-09), but literal
  wave64 execution is out of reach on gfx1100. The ε = 1e-4 headroom is part of
  why the gate is robust against a future wave64 run differing from wave32 — but
  wave64 is **not** validated by this phase and remains an open gap, carried
  forward from `07-CONTEXT.md` D-09.
