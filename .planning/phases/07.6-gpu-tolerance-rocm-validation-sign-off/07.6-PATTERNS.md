#### Pattern A — the `max_divergence` helper (COPY VERBATIM)

**Analog:** `crates/cb-backend/src/kernels/gradient_gpu.rs` lines 34–49 (identical copy
also at `score_split.rs:81`, `pointwise_hist.rs:67`, `grow_loop.rs:60`). This is the
canonical abs/rel reporter every oracle uses — copy it, do not reinvent.

```rust
// crates/cb-backend/src/kernels/gradient_gpu.rs:34-49
fn max_divergence(device: &[f64], baseline: &[f64]) -> (f64, f64) {
    assert_eq!(
        device.len(),
        baseline.len(),
        "max_divergence requires equal-length slices"
    );
    let mut max_abs = 0.0_f64;
    let mut max_rel = 0.0_f64;
    for (&d, &b) in device.iter().zip(baseline) {
        let abs = (d - b).abs();
        let rel = if b.abs() > 0.0 { abs / b.abs() } else { abs };
        max_abs = max_abs.max(abs);
        max_rel = max_rel.max(rel);
    }
    (max_abs, max_rel)
}
```

#### Pattern B — the `#[cfg(not(feature="wgpu"))]` channel-gating BOUND constant (COPY VERBATIM)

**Analog:** `crates/cb-backend/src/kernels/pointwise_hist.rs` lines 57–65 (identical shape
at `score_split.rs:71–73` `SCORE_BOUND`, `grow_loop.rs:53–56` `LEAF_BOUND`). The device
channel is **f64 on rocm/cuda/cpu** (HIP/CUDA emulate the f64 atomic add → ~1e-9 band) and
**f32 on wgpu** (WGSL has no f64 atomics → ~1e-3 band). This is a REPORTED run-stable
bound that catches a wrong result, **NOT the GPU-06 epsilon** (which is the signed-off
documented number, never a test constant — see Anti-Patterns).

```rust
// crates/cb-backend/src/kernels/pointwise_hist.rs:57-65
/// The asserted run-stable divergence bound for the device channel. f64 on
/// rocm/cuda/cpu (~1e-9), f32 on wgpu (~1e-3). REPORTED, NOT the GPU-06 epsilon (7.6's job).
#[cfg(feature = "wgpu")]
const HIST_BOUND: f64 = 1e-3;
#[cfg(not(feature = "wgpu"))]
const HIST_BOUND: f64 = 1e-9;
```

**GPU-06 relevance (Pitfall 2):** 7.6 measures on the **rocm/f64 channel only**
(`--no-default-features --features rocm`). If a draft reports ~1e-3 it is on the wgpu/f32
channel and is NOT the rocm gate.

#### Pattern C — the N≥30 run-to-run variance loop (EXTEND with stddev)

**Analog:** `crates/cb-backend/src/kernels/reduce.rs` lines 258–342 (test
`block_reduce_atomic_finalize_matches_cpu_sum_and_reports_variance`). This is the existing
`runs=32` prototype. D-7.6-02 wants the SAME loop **plus standard deviation** and the
`observed_max + 3σ` headroom term. Three load-bearing pieces to preserve:

1. **The runs loop collecting device sums + the chosen `AtomicFinalizePath`:**
```rust
// crates/cb-backend/src/kernels/reduce.rs:269-276
let runs = 32; // D-7.6-02 floor is N>=30; bump as discretion allows
let mut sums: Vec<f64> = Vec::with_capacity(runs);
let mut path = AtomicFinalizePath::HostSumFallback;
for _ in 0..runs {
    let (sum, p) = launch_block_reduce_atomic_f64(&input).unwrap();
    sums.push(sum);
    path = p;
}
```

2. **The min/max/spread + max_abs/max_rel rollup (reduce.rs:279–295)** — keep verbatim,
   then ADD for D-7.6-02:
```rust
// NEW (D-7.6-02) — append after the existing min/max/spread/max_abs block:
let mean = cb_core::sum_f64(&sums) / runs as f64;
let variance =
    cb_core::sum_f64(&sums.iter().map(|s| (s - mean).powi(2)).collect::<Vec<_>>()) / runs as f64;
let stddev = variance.sqrt();
let headroom_input = max_abs + 3.0 * stddev; // observed_max + 3σ feeds the epsilon proposal
```

3. **The path-consistency assertion (reduce.rs:315–334)** — the critical Pitfall-3 guard
   that proves the GPU did NOT silently run on CPU / a deterministic host sum. Reuse it so
   a `0.0e0` divergence cannot pass as fake validation. On gfx1100 HIP does **not** advertise
   f64 atomic-add, so `AtomicFinalizePath::HostSumFallback` is the *expected* path; assert
   against the device's advertised capability rather than hard-coding either branch:
```rust
// crates/cb-backend/src/kernels/reduce.rs:315-334 (path-consistency guard)
let device = <crate::SelectedRuntime as Runtime>::Device::default();
let client = <crate::SelectedRuntime as Runtime>::client(&device);
let advertises_atomic = {
    let ty = <cubecl::prelude::Atomic<f64> as CubePrimitive>::as_type_native_unchecked();
    client.properties().atomic_type_usage(ty)
        .contains(cubecl::features::AtomicUsage::Add)
};
let expected_path = if advertises_atomic {
    AtomicFinalizePath::InKernelAtomicF64
} else {
    AtomicFinalizePath::HostSumFallback
};
assert_eq!(path, expected_path, /* silent mode-switch would otherwise pass (WR-02) */);
```

## Shared Patterns

### REPORTED-not-signed-off divergence convention
**Source:** every oracle module (`gradient_gpu.rs:15`, `pointwise_hist.rs:61`,
`grow_loop.rs:50`, `score_split.rs:71`).
**Apply to:** the new `gpu_tolerance.rs` module — its asserts use the generous run-stable
`1e-9`(f64)/`1e-3`(f32) BOUND to catch a wrong result; the GPU-06 epsilon is the documented
human-signed-off number (D-7.6-07), produced from the `[GPU-06 EVIDENCE]` printouts, NOT a
test constant.

### CPU parity baseline (D-04 — vs the Rust CPU path)
**Source:** `cb_core::sum_f64` (`crates/cb-core/src/reduction.rs:32`) +
`cb_compute::calc_average` / loss fns. Used read-only.
**Apply to:** every baseline in `gpu_tolerance.rs`. This is the frozen host fold the device
is compared against. Do NOT import `cb-train` (transcribe inline).

### The documented rocm gate invocation (GPU-03)
**Source:** D-7.6-05; CLAUDE.md per-crate + disk-pressure constraint.
**Apply to:** the doc + VERIFICATION sign-off. Run in-env on gfx1100 (orchestrator/human,
NOT a subagent, NOT CI):
```bash
cargo test -p cb-backend --no-default-features --features rocm -- --nocapture
```

### cb-train feature-unification landmine (Anti-Pattern — do NOT)
**Source:** `grow_loop.rs:29-36`.
**Apply to:** `gpu_tolerance.rs` must NOT add or transitively pull a `cb-train` dependency.
Doing so activates `cb-backend/cpu` alongside `rocm`, `SelectedRuntime` resolves to
`CpuRuntime`, and the "GPU" oracle silently runs on CPU reporting a meaningless `0.0`.
