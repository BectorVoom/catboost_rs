# Phase 7.6: GPU Tolerance, rocm Validation & Sign-off - Research

**Researched:** 2026-06-21
**Domain:** GPU/CPU fidelity measurement, rocm oracle gate formalization, epsilon sign-off (validation/documentation phase — NOT kernel authoring)
**Confidence:** HIGH

## Summary

Phase 7.6 closes the Phase 7 umbrella by turning the divergence numbers that 7.1–7.5
already **REPORT** into a single **SIGNED-OFF** GPU-06 epsilon, locking `rocm` on the in-env
gfx1100 as the sole GPU oracle gate (GPU-03), and writing down the wave32-scope / wave64-gap
limitation (D-09). This is a measurement / aggregation / documentation phase. **No new
kernels, no production-code change to `cb-core`/`cb-model`/`cb-compute`**, and per the
additive boundary, no change even to `cb-backend`'s production kernels — the only new code is
a measurement/aggregation harness living in `cb-backend`'s `#[cfg(test)]` oracle modules, plus
documentation files. [VERIFIED: codebase — every kernel oracle module is `#[cfg(test)] mod` in `kernels.rs`]

The single most important thing to get right: **observed reality is bit-exact, but D-04
mandates a conservative epsilon looser than the CPU path's 1e-5.** Every existing self-oracle
on the f64 (rocm/cuda/cpu) channel asserts a *run-stable bound of `1e-9`* and the project's
prior phases recorded actual divergence "REPORTED 0.0e0 to ~1e-9 / ~1e-14" — far tighter than
D-03/D-04 predicted. [VERIFIED: codebase — `const HIST_BOUND/LEAF_BOUND/SCORE_BOUND = 1e-9` under `not(feature="wgpu")`]. The plan must (a) aggregate the per-family REPORTED numbers, (b) add an N≥30 repeat
variance harness to characterize the D-03 atomic non-determinism budget, (c) measure the
end-to-end GPU-trained-vs-CPU-trained leaf values (the only numbers 7.5 left REPORTED-not-
signed-off), (d) propose a clean epsilon strictly looser than 1e-5 with documented headroom
over `(observed max + 3σ)`, and (e) **pause for a hard human sign-off (D-7.6-07)** before
recording the gate and flipping GPU-03/GPU-06.

**Primary recommendation:** Build one new `#[cfg(test)]` aggregation+variance oracle in
`cb-backend` (cloning the existing `reduce.rs` `runs=32` variance pattern and the `max_divergence`
helper used across all kernel oracles), run it via `cargo test -p cb-backend --no-default-features
--features rocm` in-env on gfx1100, then author the GPU tolerance doc + VERIFICATION sign-off
section and flip REQUIREMENTS GPU-03/GPU-06 — gated behind an explicit `checkpoint:human-verify`
for the epsilon number.

### Pattern 1: REPORTED divergence oracle (the existing convention — reuse, don't reinvent)
**What:** Run device kernel → read back → compute `max_divergence` vs a deterministic CPU
reference → `println!("[label n=..] REPORTED max abs_div=.. rel_div=..")` → assert against a
generous *run-stable* bound (`1e-9` f64 / `1e-3` f32), NOT the GPU-06 epsilon.
**When to use:** Every per-family re-measurement (D-7.6-01 tier 1).
**Example:**
```rust
// Source: crates/cb-backend/src/kernels/gradient_gpu.rs:35 (verbatim convention)
fn max_divergence(device: &[f64], baseline: &[f64]) -> (f64, f64) {
    assert_eq!(device.len(), baseline.len(), "max_divergence requires equal-length slices");
    let mut max_abs = 0.0_f64;
    let mut max_rel = 0.0_f64;
    for (&d, &b) in device.iter().zip(baseline) {
        let abs = (d - b).abs();
        let rel = if b.abs() > 0.0 { abs / b.abs() } else { abs };
        max_abs = max_abs.max(abs);
        max_rel = max_rel.max(rel);
    }
    (max_abs, max_rel)
}
```

### Pattern 2: N≥30 run-to-run variance harness (extend the existing `runs=32` prototype)
**What:** Loop the atomic-finalize device path N≥30 times over a multi-cube fixture, collect
sums/outputs, report `min`, `max`, `run_to_run_spread = max - min`, and **standard deviation**
(D-7.6-02 adds stddev to the existing min/max/spread).
**When to use:** The variance term that feeds epsilon headroom (`observed_max + 3σ`).
**Example:**
```rust
// Source: crates/cb-backend/src/kernels/reduce.rs:259 (existing prototype — extend with stddev)
let runs = 32; // D-7.6-02 floor is N>=30; bump as discretion allows
let mut sums: Vec<f64> = Vec::with_capacity(runs);
let mut path = AtomicFinalizePath::HostSumFallback;
for _ in 0..runs {
    let (sum, p) = launch_block_reduce_atomic_f64(&input).unwrap();
    sums.push(sum);
    path = p;
}
// existing: min_sum / max_sum / variance_spread = max_sum - min_sum / max_abs / max_rel
// ADD for D-7.6-02: mean = sum_f64(&sums)/runs;
//                   stddev = (sum_f64(&sums.iter().map(|s|(s-mean).powi(2))...)/runs).sqrt();
//                   headroom_input = max_abs + 3.0 * stddev;
println!("[family X] runs={runs} path={path:?} max_abs={max_abs:.3e} stddev={stddev:.3e} \
          observed_max_plus_3sigma={headroom_input:.3e}");
```
[VERIFIED: codebase — `reduce.rs:259` already does runs=32 + min/max/spread; only stddev is missing]

## Common Pitfalls

### Pitfall 1: Mistaking the `1e-9` run-stable assert for the GPU-06 epsilon
**What goes wrong:** Reporting `1e-9` (or `0.0e0`) as "the signed-off tolerance" — contradicting
D-04's mandate that the gate be *looser than the CPU path's 1e-5*.
**Why it happens:** The observed f64-channel divergence genuinely *is* ~1e-9 to bit-exact, so it
looks like the natural epsilon. [VERIFIED: codebase — f64 channel asserts `≤ 1e-9`]
**How to avoid:** Present BOTH numbers explicitly: observed (tight, ~bit-exact) AND proposed
(conservative, looser than 1e-5 per D-04). The proposed epsilon deliberately keeps headroom
*despite* the bit-exact observation (D-7.6-04 rationale: unmeasured losses/fixtures, larger
atomic contention, wave64). Make the gap the centerpiece of the sign-off (D-7.6-04).
**Warning signs:** A draft doc that proposes `ε = 1e-9` or `ε = 1e-5` (not strictly looser).

### Pitfall 2: Running on the wgpu (f32) channel and conflating its `1e-3` bound
**What goes wrong:** Measuring under `--features wgpu` yields ~1e-3 divergence (WGSL has no f64
atomics → f32 channel), which is NOT the rocm gate.
**Why it happens:** wgpu is the convenient no-special-hardware backend.
**How to avoid:** GPU-06 is **rocm-only** (D-06). Measure under `--no-default-features --features
rocm` on gfx1100. HIP supports/emulates f64 atomic-add, so the rocm channel is f64 (`1e-9` band).
[VERIFIED: codebase — `pointwise_hist.rs:58` "device channel is f64 on rocm/cuda/cpu ... f32 on wgpu"]
**Warning signs:** Reported divergence clustering near 1e-3 instead of 1e-9.

### Pitfall 3: `cpu` feature leaking in alongside `rocm` (feature unification)
**What goes wrong:** `SelectedRuntime` resolves to `CpuRuntime` even with `rocm` requested,
silently running the "GPU" oracle on CPU and reporting a meaningless 0.0 divergence.
**Why it happens:** `SelectedRuntime`'s cfg-chain is `rocm AND not(cpu) AND not(wgpu) AND
not(cuda)`; any dep that pulls `cb-backend/cpu` (notably `cb-train`) defeats it.
**How to avoid:** Use `--no-default-features --features rocm` (default is `cpu`), and never add a
`cb-train` dependency to `cb-backend`. [VERIFIED: codebase — `lib.rs` cfg-chain; `Cargo.toml` `default = ["cpu"]`]
**Warning signs:** Build pulls `cb-train`; or the rocm test passes with literally 0.0e0 and no
`AtomicFinalizePath::InKernelAtomicF64` reported.

### Pitfall 4: Full-workspace test invocation under disk pressure
**What goes wrong:** Link/compile fails or fills the disk (root ~100%, 1.1G free).
**Why it happens:** A full `cargo test` links every crate's test binary.
**How to avoid:** `cargo test -p cb-backend --no-default-features --features rocm` (per-crate),
and `cargo clean` the target dir if it exceeds 30 GB (AGENTS.md). [VERIFIED: `df -h /` 100%]

### Pitfall 5: The verifier subagent cannot run the GPU
**What goes wrong:** A spawned verifier reports the rocm sign-off as unverifiable / fails the gate.
**Why it happens:** GPU access (`/dev/kfd`) is in-env but a subagent can't drive it reliably —
the established pattern (memory: Phase 7.5) is the **orchestrator discharges the rocm run in-env**.
**How to avoid:** Plan the rocm measurement + the human epsilon sign-off as an orchestrator/human
step, not a verifier-subagent assertion. Build an explicit `checkpoint:human-verify` task.

### Pitfall 6: Asserting wave64 behavior on gfx1100
**What goes wrong:** Claiming wave64 validation when the box is wave32-native (RDNA3).
**Why it happens:** GPU-03's literal wording says "wavefront-64 safe."
**How to avoid:** Validate the *no-warp-size-assumptions* property on wave32 and **document the
wave64 (CDNA/MI-series) gap explicitly** (D-09) — do not assert it as closed.
