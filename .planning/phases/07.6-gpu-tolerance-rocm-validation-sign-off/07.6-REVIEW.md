## Summary

Phase 07.6 is a GPU-tolerance MEASUREMENT/sign-off phase. The only net-new code is
`crates/cb-backend/src/kernels/gpu_tolerance.rs` — a `#[cfg(test)]` test-harness module
that aggregates per-kernel-family GPU-vs-CPU divergence, runs an N=32 run-to-run variance
loop, and measures end-to-end GPU-vs-CPU leaf values on rocm/gfx1100. `kernels.rs` gained
a one-line `#[cfg(test)] mod gpu_tolerance;` declaration (verified correctly gated at
kernels.rs:2694-2695). `ci.yml` had a stale `cargo build --features rocm` step removed
(D-06: rocm is in-env-only).

I reviewed against the test-appropriate severity bar from the phase context: correctness of
the measurement logic, the path-consistency guard, and absence of the `cb-train` landmine.

**Positive verifications (no defect):**
- The `cb-train` landmine is correctly avoided: `gpu_tolerance.rs` imports only `cb_core`
  and `cb_compute` (both regular deps of `cb-backend`, confirmed in `Cargo.toml:23-24`);
  `cb-train`/`cb_train` appears only in comments. Feature-unification cannot pull `cb-backend/cpu`
  in alongside `rocm`.
- The `mod gpu_tolerance;` declaration is `#[cfg(test)]`-gated (kernels.rs:2694), so the
  module never enters the production build graph. Source/test separation is honored (standalone
  file, not an inline `mod tests`).
- The pairwise host reference (`add_pair_contrib`, gpu_tolerance.rs:213-234) matches the
  kernel's channel mapping VERBATIM in both branches: non-one-hot `histId = 2*ge+isSecondBin /
  2*gt+isSecondBin` (kernels.rs:1024-1041) and one-hot `is_ge = (b1 != b2)` (kernels.rs:1010-1022).
- All `cb_compute`/`cb_core` symbols consumed exist with the consumed contracts:
  `calc_average(sum_der, sum_weight, scaled_l2)` (leaf.rs:83/107), `scale_l2_reg`, `l2_split_score`,
  `rmse_der1`, `logloss_der2(_target unused)`, `sum_f64`. The leaf-value reference correctly passes
  `sum_f64(&w_seg)` as the `count` parameter per leaf.rs:107.
- All reductions are folded through `cb_core::sum_f64` (D-08), never `std::iter::sum`.
- The Test B path-consistency guard (gpu_tolerance.rs:647-653) is preserved verbatim and is
  the load-bearing anti-false-pass mechanism for the reduce family.

## Warnings

### WR-01: Tests A and C have no GPU-runtime path guard — a CPU-resolved build silently false-passes

**File:** `crates/cb-backend/src/kernels/gpu_tolerance.rs:467-574` (Test A),
`crates/cb-backend/src/kernels/gpu_tolerance.rs:682-772` (Test C)

**Issue:** The module's entire validity rests on `SelectedRuntime` resolving to the GPU
(rocm/gfx1100). The module doc (lines 16-25) and Test B (lines 647-653, via
`expected_atomic_finalize_path`) both explicitly defend against the "skipped/CPU-resolved GPU
test counted as passed" failure mode. But Tests A (`gpu06_per_family_aggregation_reports_evidence`)
and C (`gpu06_end_to_end_leaf_values_report_evidence`) carry NO equivalent guard. If the harness
is accidentally run under the default `cpu` feature (or feature unification flips the runtime),
every `max_divergence` in A and C becomes a CPU-vs-CPU comparison that is bit-exact (0.0), and
both tests PASS while measuring nothing — the precise fake-validation outcome the module was
written to prevent. The `channel=f64(rocm/gfx1100)` string in the println is HARD-CODED text,
not derived from the actual runtime, so the evidence line also lies in that case.

**Fix:** Add a cheap runtime assertion at the top of Tests A and C asserting the resolved
backend is the GPU channel, reusing the capability query already present in
`expected_atomic_finalize_path()` (lines 126-141). For example, assert the device advertises the
GPU runtime type name, or—mirroring Test B—assert the pointwise/score paths return one of the
in-kernel-atomic `AtomicFinalizePath` variants (`InKernelAtomicF64` /
`InKernelAtomicF64Unadvertised`) rather than `HostSumFallback`:
```rust
// Test A, pointwise_hist block — promote the printed path to an assertion:
assert_ne!(
    path, AtomicFinalizePath::HostSumFallback,
    "pointwise_hist resolved to a host-sum path — SelectedRuntime is not the GPU \
     (run --no-default-features --features rocm); a CPU-resolved build would false-pass"
);
```
For Test C (which has no path return), gate on the same `expected_atomic_finalize_path()` /
device-type check used in Test B before the structure assertions.

### WR-02: Pass/fail tolerance is an `abs OR rel` disjunction — a large absolute drift passes whenever the relative term is small

**File:** `crates/cb-backend/src/kernels/gpu_tolerance.rs:495-498, 514-517, 540-543, 565-568,
657-660, 766-770`

**Issue:** Every assertion is `rel <= TOL_BOUND || abs <= TOL_BOUND`. Because it is a
disjunction, a result passes if EITHER term clears the bound. For the large-magnitude families
(histogram sums and leaf values over n up to 10000, `der1` up to ~5000), this means an absolute
divergence far larger than `TOL_BOUND_F64 = 1e-9` still passes as long as the relative ratio is
small (e.g. baseline 5000, abs 1e-6 → rel 2e-10 passes). The harness is explicitly a REPORT bound
"that catches a WRONG result" (lines 29-34), but the disjunction widens the catch envelope: a
genuine but bounded GPU regression on a high-magnitude leaf could slip through while the printed
`observed_max_abs` evidence still records it. This is the conventional rel-OR-abs pattern, so it
is a deliberate design choice, not an outright bug — but it weakens the "catches a wrong result"
claim for the high-magnitude channels and is worth pinning before the Plan-02 epsilon is derived
from these same numbers.

**Fix:** Either document the disjunction's envelope explicitly at each assertion, or tighten to
a combined predicate where the magnitude warrants it, e.g. `assert!(abs <= TOL || rel <= TOL_REL)`
with a separately-justified `TOL_REL`, so the relative escape hatch cannot mask an unbounded
absolute drift on the large-baseline families. At minimum, since the evidence consumer (Plan 02)
reads `observed_max_abs`, ensure a high-abs/low-rel result cannot silently pass the gate that
feeds the epsilon proposal.

### WR-03: `make_fixture` / `make_pair_fixture` underflow and div-by-zero on degenerate `n_bins`/`n` are unguarded (latent, not reachable from current callers)

**File:** `crates/cb-backend/src/kernels/gpu_tolerance.rs:333-349` (`make_fixture`),
`crates/cb-backend/src/kernels/gpu_tolerance.rs:354-382` (`make_pair_fixture`)

**Issue:** `make_fixture` computes `((obj * n_bins) / n.max(1)).min(n_bins - 1)` — `n_bins - 1`
underflows (panics in debug) if `n_bins == 0`. `make_pair_fixture` guards `n_bins == 0` and
`n_objects == 0` for its own arithmetic, but `make_fixture` does NOT guard `n_bins == 0`. Today
every caller passes `n_bins = 32` and `n >= 1000`, so this is unreachable — but the two sibling
fixture helpers are inconsistent (one guards the degenerate case, the other does not), which is a
maintenance hazard if either is reused at a smaller scale.

**Fix:** Either add a `debug_assert!(n_bins > 0)` precondition to `make_fixture` to make the
contract explicit, or make the two helpers consistent (both guard or both assert). Since these are
test-only fixtures with fixed callers, a precondition assertion is sufficient.

## Info

### IN-01: Hard-coded `channel=f64(rocm/gfx1100)` evidence string is not derived from the actual runtime

**File:** `crates/cb-backend/src/kernels/gpu_tolerance.rs:490, 510, 536, 561, 631, 761`

**Issue:** The `[GPU-06 EVIDENCE] ... channel=f64(rocm/gfx1100)` text is a literal in every
println. Under wgpu the channel is actually f32 (the module itself documents this at lines 73-76
where `TOL_BOUND_F64` becomes `1e-3`), and under a CPU-resolved build it is neither rocm nor
gfx1100. The evidence line therefore mislabels the channel on any backend other than rocm/f64.

**Fix:** Derive the channel/device label from the resolved runtime (or from a `cfg!(feature)`
switch) rather than hard-coding it, so the emitted evidence matches the build that produced it.
This compounds WR-01: the hard-coded label removes the operator's last visual cue that a wrong
channel was measured.

### IN-02: `AtomicFinalizePath` evidence annotations for several families are hard-coded descriptive strings

**File:** `crates/cb-backend/src/kernels/gpu_tolerance.rs:493, 538, 563, 764`

**Issue:** The der_hess, pairwise_hist, score_split, and end_to_end families print
`AtomicFinalizePath=n/a(...)` / `in-kernel(global)` / `device-resident-...` as literal text rather
than a value returned by the launch. Only the pointwise (line 512, `{path:?}`) and reduce
(line 635, `{path:?}`) families print an actual returned path. The literal annotations cannot drift
out of sync with a launch that genuinely changed its finalize strategy.

**Fix:** Where the launch returns a path (or could), print the returned value; where it genuinely
has no atomic finalize, the literal is acceptable but should be commented as intentionally static.

### IN-03: Module-level `let _ = TOL_BOUND_F32;` discard is a workaround for an otherwise-unused const

**File:** `crates/cb-backend/src/kernels/gpu_tolerance.rs:571-573`

**Issue:** `TOL_BOUND_F32` (line 80) is referenced only via a `let _ =` discard at the end of
Test A to suppress a dead-code warning; it is never actually used as a tolerance in any assertion
in the f64 build. The accompanying comment explains it documents the wgpu channel's looser
magnitude, but a `const` that is only kept alive by a discard is effectively dead under the
default/rocm build.

**Fix:** Either use `TOL_BOUND_F32` in a wgpu-gated assertion path (so it is live where it
matters), or gate the const itself behind `#[cfg(feature = "wgpu")]` so it is not dead code under
the f64 channel and the discard can be removed.
