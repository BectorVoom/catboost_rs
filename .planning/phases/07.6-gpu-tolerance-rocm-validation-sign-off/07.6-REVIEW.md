# Phase 07.6: Code Review Report

**Reviewed:** 2026-06-21T00:00:00Z
**Depth:** standard
**Files Reviewed:** 2

## Summary

Reviewed the GPU-06 tolerance measurement harness (`gpu_tolerance.rs`, ~806 new
lines this phase) and the supporting `kernels.rs` (the IN-02 `cb_leaf_score_term`
extraction + the WR-01 `f32::MIN` sentinel + the WR-04 `partition_update`
in-kernel bounds guard). `gpu_tolerance.rs` is a `#[cfg(test)]` measurement
module, so the `unwrap()`/`expect()` prohibition (production-only, CLAUDE.md) does
NOT apply to it.

The module's own doc-comments (lines 27-34, 143-157) establish its single
load-bearing invariant: it MUST NOT silently report a fake `0.0` divergence under
the default `cpu` backend (which resolves `SelectedRuntime` to `CpuRuntime` and
collapses every comparison to bit-exact CPU-vs-CPU). The WR-01 fix added a
`gpu_backend_active()` SKIP guard to Test A and Test C — but **Test B was left
unguarded**, and its WR-02 path-consistency guard does NOT substitute for the
missing skip: under `cpu` the fallback path is self-consistent, so Test B passes
as a green, fake `rocm/gfx1100` evidence line. That is the BLOCKER below.

The `kernels.rs` refactor (Cosine numerator/denominator split via `cb_leaf_avg`,
the five comptime calcer arms folded into `cb_leaf_score_term`) is
behavior-preserving on inspection. No correctness defect found in the kernel code.

## Critical Issues

### CR-01: Test B (`gpu06_variance_with_stddev_reports_evidence`) is missing the `gpu_backend_active()` skip — emits a fake `rocm/gfx1100` PASS under the default `cpu` backend

**File:** `crates/cb-backend/src/kernels/gpu_tolerance.rs:611-686`

**Issue:** Tests A and C begin with the WR-01 anti-false-pass guard
(`if !gpu_backend_active() { eprintln!(...); return; }`, lines 487-493 and
709-717). Test B has NO such guard. Trace the `cpu`-backend path:

1. `launch_block_reduce_atomic_f64` (`gpu_runtime/mod.rs:320-336`) calls
   `device_supports_f64_atomic_add`; the CPU runtime does not advertise f64
   atomic-add, so it takes the `HostSumFallback` branch and returns
   `(cb_core::sum_f64(partials), AtomicFinalizePath::HostSumFallback)`.
2. Every one of the 32 runs returns the SAME deterministic host sum, so
   `max_abs == max_rel == 0.0` and `stddev == 0.0`.
3. `expected_atomic_finalize_path()` (lines 126-141) also returns
   `HostSumFallback` on the CPU client, so the WR-02 path-consistency
   `assert_eq!(path, expected_path)` PASSES.
4. The bound assertion (`max_rel <= TOL_BOUND_F64 || max_abs <= TOL_BOUND_F64`,
   lines 682-685) passes trivially at `0.0`.

So under `cargo test --workspace` (default `cpu`) Test B runs to completion,
asserts nothing meaningful, and prints
`[GPU-06 EVIDENCE] family=reduce channel=f64(rocm/gfx1100) ... observed_max_abs=0.000e0 ...`
(lines 655-661) — a green evidence line falsely labelled `rocm/gfx1100` that
measured CPU-vs-CPU. This is precisely the fake-validation outcome the module's
own doc (lines 22-25, 143-157) and the WR-01 fix exist to prevent. The WR-02
guard the test relies on (lines 667-678) only proves the *fallback was consistent*;
it does NOT prove a GPU ran, because `HostSumFallback` is also the LEGITIMATE
gfx1100 path (the module knows this — lines 152-154), so it cannot distinguish
"real gfx1100 host-fallback reduce" from "cpu-backend host sum."

**Fix:** Add the same skip guard Tests A/C use as the first statement of Test B:
```rust
#[test]
fn gpu06_variance_with_stddev_reports_evidence() {
    // WR-01: under the cpu backend this is a CPU-vs-CPU false-pass — SKIP.
    if !gpu_backend_active() {
        eprintln!(
            "[GPU-06] SKIP gpu06_variance_with_stddev: cpu backend active — \
             measurement requires a GPU feature (--no-default-features --features rocm)."
        );
        return;
    }
    // ... existing body ...
}
```
(Retain the WR-02 path-consistency assertion behind the guard — it remains a
valuable real-GPU check; it is just not a substitute for the skip.)

## Warnings

### WR-01: Evidence-line channel label is hardcoded `rocm/gfx1100` regardless of the active backend

**File:** `crates/cb-backend/src/kernels/gpu_tolerance.rs:515, 535, 561, 586, 656, 795`

**Issue:** Every `[GPU-06 EVIDENCE]` line prints the literal
`channel=f64(rocm/gfx1100)`. But `gpu_backend_active()` (lines 155-157) is true for
`wgpu` and `cuda` as well, and under `wgpu` the device channel is f32, not f64
(the module itself documents this — `TOL_BOUND_F64` is overridden to `1e-3` on
wgpu, lines 73-76). So a `wgpu` host run emits lines claiming
`channel=f64(rocm/gfx1100)` while actually measuring an f32 wgpu channel. The
module-doc rationale for printing the channel (lines 24-25: "so a wrong channel is
visible in the output") is defeated by hardcoding it — the output can never reveal
a wrong channel because it is a string literal, not the observed backend.

**Fix:** Derive the channel label from the active backend feature and the channel
width, e.g. a small helper:
```rust
fn channel_label() -> &'static str {
    #[cfg(feature = "rocm")] { return "f64(rocm/gfx1100)"; }
    #[cfg(feature = "cuda")] { return "f64(cuda)"; }
    #[cfg(feature = "wgpu")] { return "f32(wgpu)"; }
    #[allow(unreachable_code)] { "f64(cpu-fallback)" }
}
```
and interpolate `channel={}` from it in each evidence line.

### WR-02: `observed_max_plus_3sigma` for the single-shot families is just `observed_max_abs` with NO sigma term — the printed headroom understates real variance

**File:** `crates/cb-backend/src/kernels/gpu_tolerance.rs:517, 537, 563, 588, 797`

**Issue:** The module's stated deliverable (lines 9-12, 33-34) is an
`observed_max + 3σ` headroom term that feeds the Plan-02 epsilon proposal. Only
Test B computes a real `stddev` over N=32 runs (lines 652-653). Tests A and C are
single-shot: they print `observed_max_plus_3sigma={abs:.3e}` where the value is
literally `observed_max_abs` with `3σ = 0` (the kernels they exercise — pointwise
hist, pairwise hist, score-split, end-to-end leaf values — use in-kernel atomic
merges with documented run-to-run float-order non-determinism, so their true σ is
NOT zero). Labelling a single-shot maximum as `observed_max_plus_3sigma` feeds the
epsilon proposal a headroom number that is structurally too small for exactly the
families whose atomic merges are the dominant non-determinism source. The
`stddev=n/a(single-shot)` annotation documents the gap but the
`observed_max_plus_3sigma` field is still emitted as if it were the headroom.

**Fix:** Either (a) wrap the atomic-merge families (pointwise/pairwise hist,
end-to-end leaf values) in the same N=32 variance loop Test B uses and emit a real
`observed_max + 3σ`, or (b) rename the single-shot field to
`observed_max_no_sigma` (and have the Plan-02 consumer treat single-shot families
explicitly) so the epsilon proposal cannot silently consume a sigma-free max as a
3σ headroom.

### WR-03: `max_divergence` reports `abs` in the `max_rel` slot when the baseline element is `0.0`, conflating two metrics

**File:** `crates/cb-backend/src/kernels/gpu_tolerance.rs:95-100`

**Issue:** `let rel = if b.abs() > 0.0 { abs / b.abs() } else { abs };` — when the
baseline value is exactly `0.0`, the function substitutes the ABSOLUTE divergence
into the RELATIVE accumulator. The score-split and leaf-value families legitimately
produce `0.0` baseline entries (e.g. empty/degenerate leaves return `0.0` via the
`count > 0` guard; the centred der ramp in `make_fixture`, line 350, crosses zero).
For those elements the reported `observed_max_rel` is actually an absolute number,
which is then fed verbatim into the epsilon proposal alongside true relative
values. While copied from `gradient_gpu.rs` per the comment (lines 84-86), in the
divergence-measurement context this silently mixes units in the very number the
phase exists to produce.

**Fix:** Either skip zero-baseline elements from the relative accumulator
entirely, or track them in a separate `max_abs_at_zero_baseline` field so the
relative metric stays purely relative:
```rust
let rel = if b.abs() > 0.0 { abs / b.abs() } else { 0.0 }; // do not pollute max_rel
```
(`max_abs` already covers the zero-baseline divergence as the catch-all.)

## Info

### IN-01: `_ = TOL_BOUND_F32` no-op reference is a workaround for a dead-code warning rather than a use

**File:** `crates/cb-backend/src/kernels/gpu_tolerance.rs:596-598`

**Issue:** `TOL_BOUND_F32` (line 80) is declared and documented as the f32-channel
bound but is never actually asserted against — it is only kept alive via
`let _ = TOL_BOUND_F32;` at the end of Test A with a comment claiming it
"documents" the wgpu channel. A constant that is only `let _`-referenced is dead
for all practical purposes; the wgpu f32 channel is actually bounded by
`TOL_BOUND_F64` (overridden to `1e-3` on wgpu, lines 75-76), so `TOL_BOUND_F32` is
genuinely unused.

**Fix:** Remove `TOL_BOUND_F32` (lines 78-80) and the `let _ = TOL_BOUND_F32;`
discard (lines 596-598), or wire it into the der/hess f32-fixture family it claims
to bound.

### IN-02: `make_pair_fixture` carries dead `n_bins == 0` / `n_objects == 0` guards never reachable from any caller

**File:** `crates/cb-backend/src/kernels/gpu_tolerance.rs:380-396`

**Issue:** `make_pair_fixture` guards `if n_objects == 0` (lines 380-381) and
`if n_bins == 0` (line 389) with fallback `0` values, but the sole caller (Test A,
lines 550-551) always passes `n_objects = 1000`, `n_bins = 32`. The guards are
dead defensive code in a test-only fixture builder.

**Fix:** Drop the zero-guards and index directly, or add a `debug_assert!` at the
top documenting the fixture's non-empty precondition.

### IN-03: `add_pair_contrib` one-hot branch is never exercised by any test in this module

**File:** `crates/cb-backend/src/kernels/gpu_tolerance.rs:229-250`

**Issue:** `host_reference_pairwise_hist` is invoked only with `one_hot = false`
(line 557), so the `if one_hot { ... }` branch of `add_pair_contrib` (lines
237-241) is dead within this module. It is transcribed correctly (it matches the
`pairwise_hist_nonbinary_kernel` one-hot branch, lines 1010-1022), but as written
it is untested transcription — a future divergence between it and the kernel would
not be caught by this harness.

**Fix:** Either add a one-hot evidence sub-case to Test A's pairwise_hist family
(passing `one_hot = true` to both the launch and the host reference), or drop the
one-hot branch from the host helper until a one-hot family is measured here.
