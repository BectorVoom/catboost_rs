## Fixed Issues

### CR-01: Test B missing `gpu_backend_active()` skip — fake `rocm/gfx1100` PASS under `cpu`

**Files modified:** `crates/cb-backend/src/kernels/gpu_tolerance.rs`
**Commit:** 51dcaf4
**Applied fix:** Added the same `if !gpu_backend_active() { eprintln!(...); return; }`
anti-false-pass skip guard (used by Tests A/C) as the first statement of
`gpu06_variance_with_stddev_reports_evidence`. The WR-02 path-consistency assertion is
retained behind the guard. Under `cpu` the test now skips instead of emitting a green,
fake `rocm/gfx1100` evidence line that measured CPU-vs-CPU bit-exact `0.0`.

### WR-01: Evidence-line channel label hardcoded `rocm/gfx1100`

**Files modified:** `crates/cb-backend/src/kernels/gpu_tolerance.rs`
**Commit:** fbb28db
**Applied fix:** Added a `channel_label()` helper that derives the label from the active
backend feature (rocm -> `f64(rocm/gfx1100)`, cuda -> `f64(cuda)`, wgpu -> `f32(wgpu)`,
else `f64(cpu-fallback)`). Replaced the hardcoded `channel=f64(rocm/gfx1100)` literal in
all five evidence lines with the interpolated `channel={channel}`. A wgpu host run now
correctly reports its f32 channel instead of falsely claiming the rocm f64 channel.

### WR-02: Single-shot `observed_max_plus_3sigma` understates variance

**Files modified:** `crates/cb-backend/src/kernels/gpu_tolerance.rs`
**Commit:** eb15e61
**Applied fix:** Chose option (b) — renamed the single-shot field from
`observed_max_plus_3sigma` to `observed_max_no_sigma` for all single-shot families
(der_hess, pointwise_hist, pairwise_hist, score_split in Test A; end_to_end_leaf_values
in Test C). Test B (the real N=32 variance loop) keeps `observed_max_plus_3sigma` because
it computes a genuine `stddev`. The Plan-02 epsilon consumer can no longer silently
consume a sigma-free maximum as a 3σ headroom for the very atomic-merge families whose
run-to-run float-order non-determinism is the dominant variance source.

### WR-03: `max_divergence` puts abs into the rel slot at zero baseline

**Files modified:** `crates/cb-backend/src/kernels/gpu_tolerance.rs`
**Commit:** 5d42524
**Status:** fixed: requires human verification (logic/semantic change to the measurement)
**Applied fix:** Changed `let rel = if b.abs() > 0.0 { abs / b.abs() } else { abs };` to
`else { 0.0 }` so the relative accumulator stays purely relative; `max_abs` already
covers the zero-baseline divergence. Updated the function doc accordingly. This changes
the reported `observed_max_rel` for the score-split and leaf-value families (which emit
legitimate `0.0` baselines), so the measured numbers should be re-confirmed on a real
GPU run before they feed the Plan-02 epsilon proposal.

### IN-01: `TOL_BOUND_F32` only `let _`-referenced (dead)

**Files modified:** `crates/cb-backend/src/kernels/gpu_tolerance.rs`
**Commit:** 594255f
**Applied fix:** Removed the `TOL_BOUND_F32` constant, the `let _ = TOL_BOUND_F32;`

### IN-03: untested `add_pair_contrib` one-hot branch

**Files modified:** `crates/cb-backend/src/kernels/gpu_tolerance.rs`
**Commit:** 30cc83b
**Status:** fixed: requires human verification (new GPU measurement path)
**Applied fix:** Chose option (a) — added a `one_hot=true` pairwise sub-case to Test A's

evidence path must be confirmed by running the rocm suite in-env on gfx1100.
