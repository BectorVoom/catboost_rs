## Phase Boundary

Close the Phase 7 umbrella by establishing the **GPU-06 tolerance epsilon
empirically** on the in-env gfx1100 GPU, documenting it, and obtaining
**user sign-off**; and locking **`rocm` as the sole authoritative GPU oracle
gate** (GPU-03) with GPU tests run on `rocm` only — never in GitHub Actions.

phase. The GPU-01 kernel surface is already CLOSED (7.1–7.5: scan/reductions,
der/hess + targets, pointwise + pairwise histogram families, score calcers,
split selection, and the full device-resident grow loop all exist and
self-oracle on gfx1100). 7.6 measures what those kernels actually do vs the
Rust CPU path, proposes a defensible epsilon, gets it signed off, and writes
the validation scope down.

**Requirements in scope:** GPU-03, GPU-06.

**Explicitly NOT in scope:** new kernels, new losses, perf tuning, wgpu/cuda
validation, wave64 execution, any change to `cb-core`/`cb-model`/`cb-compute`.

## Implementation Decisions

All gray areas here are *how to measure and formalize*, not *what to build* —
the substantive fidelity decisions were locked in `07-CONTEXT.md`
(D-04 empirical-then-signed-off epsilon, D-06 rocm sole gate / never in CI,
D-07 wgpu+cuda compile-gated only, D-09 wave32 scope / wave64 gap). Those
carry forward unchanged and are NOT re-opened here.

### Divergence Measurement Coverage
- **D-7.6-01:** Measure divergence across the **full GPU surface**, two tiers:
  (1) **per-kernel-family** — re-run each existing self-oracle (der/hess,
  pointwise hist 5/6/7/8-bit + half-byte + binary, pairwise hist 4-channel +
  one-hot + 8-bit-atomics, scan/reduction, score calcers L2/Cosine/Solar/LOO/Sat,
  pairwise split scorer) capturing its abs/rel divergence vs the Rust CPU path;
  and (2) **end-to-end** — a GPU-trained vs CPU-trained model on representative
  fixtures comparing **per-tree structure (must be exact) and leaf values
  (within epsilon)**. The end-to-end leaf values were *REPORTED, not signed off*
  in 7.5 (07.5-03/04/06) — 7.6 is where they get measured for the gate.
  _[auto] Coverage — Selected: "per-kernel + end-to-end (full surface)" (recommended default) over end-to-end-only / per-kernel-only._

### Run-to-Run Variance Protocol
- **D-7.6-02:** Characterize the D-03 atomic-nondeterminism budget by **repeating
  each representative fixture N ≥ 30 times** and reporting **max observed abs
  divergence + standard deviation** per fixture. The variance term feeds the
  epsilon headroom (D-7.6-04). Single-run measurement is insufficient — the whole
  reason D-04 mandated an empirical epsilon is run-to-run float-summation-order
  variance from device atomics.
  _[auto] Variance — Selected: "N≥30 repeats, report max + stddev" (recommended default) over single-run / N=5._

### Epsilon Structure
- **D-7.6-03:** Propose **one single global signed-off epsilon** for the GPU-06
  gate (matching the requirement wording — "a separately-stated epsilon"), with
  the **per-family observed-divergence table documented as supporting evidence**.
  A single number is the gate; the per-family breakdown is the justification, not
  separate per-family thresholds. Tiered per-kernel epsilons are rejected as
  over-engineering for a sign-off gate.
  _[auto] Structure — Selected: "single global epsilon + evidence table" (recommended default) over tiered per-family thresholds._

### Epsilon Headroom & Derivation
- **D-7.6-04:** Derive the proposed epsilon as: **round UP to a clean,
  defensible value strictly looser than the CPU path's 1e-5** (honoring D-04),
  with headroom over `(observed max abs divergence + 3σ run-to-run variance)`.
  **Known tension to surface at sign-off:** every sub-phase self-oracle so far
  reported divergence at or near **bit-exact (0.0e0 to ~1e-14)** — far tighter
  than D-03/D-04 predicted. The proposed epsilon therefore deliberately keeps
  D-04's looser-than-1e-5 headroom **despite** the bit-exact observations,
  because: (a) D-04 mandates it; (b) observed bit-exactness may not hold for
  losses/fixtures not yet measured at the model level, for larger fixtures with
  more atomic contention, or on wave64 hardware; (c) headroom protects the gate
  from future flakiness. The derivation, the observed-vs-proposed gap, and the
  rationale are all documented so the user signs off an informed number.
  _[auto] Headroom — Selected: "conservative, looser-than-1e-5 per D-04, document the bit-exact gap" (recommended default) over "tight epsilon reflecting observed bit-exactness"._

### rocm Gate Formalization (GPU-03)
- **D-7.6-05:** Operationalize the GPU gate as a **documented manual in-env
  invocation** on gfx1100 (the established per-crate command form, e.g.
  `cargo test -p cb-backend --features rocm`), recorded as the authoritative GPU
  validation step. **Explicitly NOT wired into GitHub Actions** (standing ROCm CI
  constraint, D-06). `wgpu`/`cuda` stay **compile-gated only** (build-checked,
  never asserted as validation, D-07). Disk pressure on this box (root ~100%
  full) means **per-crate** test invocation, not full-workspace links.
  _[auto] Gate — Selected: "documented manual per-crate rocm run, no CI" (recommended default) over CI wiring / wgpu-as-gate._

### Documentation Deliverable
- **D-7.6-06:** Produce a concise **GPU tolerance / validation document** (a
  dedicated doc and/or the VERIFICATION.md sign-off section) recording: the
  signed-off epsilon and its derivation; that it is measured **vs the Rust CPU
  path, not the C++ CPU oracle** (D-04); the **wave32-native validation scope**
  on gfx1100 and the **explicit, documented wave64 (CDNA/MI-series) gap** that
  this phase does NOT close (D-09); and the rocm-only / never-in-CI gate policy
  (D-06). REQUIREMENTS.md GPU-03/GPU-06 get flipped to complete with the phase
  reference once signed off.
  _[auto] Docs — Selected: "dedicated tolerance doc + VERIFICATION sign-off + REQUIREMENTS update" (recommended default)._

### User Sign-off Gate (carried from D-04 — cannot be auto-resolved)
- **D-7.6-07:** SC-2's **user sign-off of the concrete epsilon is a hard human
  gate** that occurs during *execution/verification*, NOT at discuss time.
  `--auto` here captures the **methodology** for proposing the epsilon
  (D-7.6-01..04), not the number itself. The executor measures, proposes the
  concrete epsilon with its derivation, and **must pause for the user to sign
  off** before the gate value is recorded and GPU-06 is marked complete. Planner:
  build this human checkpoint into the plan explicitly — do not let auto-mode
  flow past it.

## Specific Ideas

- **The empirical surprise to resolve at sign-off:** D-03/D-04 anticipated that
  device atomics would force an epsilon *looser than 1e-5*, but every sub-phase
  observed **bit-exact or ~1e-14** divergence. The plan must present BOTH the
  observed (tight) reality AND the D-04-mandated conservative headroom, and let
  the user pick the signed-off number with that tension made explicit. This is
  the single most important thing to get right in 7.6.
- "match cuda in catboost" remains the umbrella steer; 7.6 adds no new fidelity
  scope — it certifies what 7.1–7.5 built.
</specifics>
