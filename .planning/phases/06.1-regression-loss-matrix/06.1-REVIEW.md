# Phase 06.1: Code Review Report

**Reviewed:** 2026-06-16
**Depth:** standard
**Files Reviewed:** 18
**Status:** issues_found

## Summary

Phase 6.1 adds the regression-loss matrix (LogCosh, Lq, Huber, Expectile,
Poisson, Tweedie, MAPE, Quantile) with scalar der1/der2 helpers, CubeCL
generics-float kernels, CPU dispatch, Exact-leaf optimizers, and the MSLE
eval-metric. The scalar math (`loss.rs`) is faithfully transcribed and well
unit-tested; the `Loss::validate` domain guards are correct and avoid `unwrap`.
However, the kernels are NOT bit-equivalent to their scalar counterparts at one
loss-function boundary, and the Exact-leaf dispatch silently mis-routes several
losses. The phase prompt flagged a prior latent Quantile-kernel boundary bug
(WR-01) and a silent zero-derivative swallow (WR-04) for verification — **the
Quantile/MAE kernel boundary bug is confirmed present and is promoted to a
BLOCKER** because it is a parity-correctness defect against a ≤1e-5 oracle bar
that the existing tests do not catch.

A focused source/test-separation check passed — every reviewed production file
keeps its tests in a sibling `_test.rs` / `#[path]` module, per CLAUDE.md.

## Critical Issues

### CR-01: Quantile/MAE gradient KERNELS disagree with the scalar `quantile_der1` at the exact deadzone boundary `|target-approx| == delta`

**File:** `crates/cb-backend/src/kernels.rs:67-71` (`mae_gradient_kernel`), `crates/cb-backend/src/kernels.rs:103-108` (`quantile_gradient_kernel`)
**Also:** `crates/cb-compute/src/loss.rs:166-175` (the reference scalar)

The scalar reference uses a strict-`<` deadzone, so the boundary value is OUTSIDE
the deadzone:

```rust
// loss.rs quantile_der1
let val = target - approx;
if val.abs() < delta { 0.0 }          // |val| == delta is NOT deadzone
else if val > 0.0 { alpha }           // val ==  delta -> alpha
else { -(1.0 - alpha) }               // val == -delta -> -(1-alpha)
```

The kernels invert the test, putting the boundary value INSIDE the deadzone:

```rust
// kernels.rs quantile_gradient_kernel (and mae_gradient_kernel, identical shape)
let mut g = F::new(0.0);              // deadzone default
if val > d { g = a; }                 // val ==  d  -> falls through to 0 (BUG)
else if val < F::new(0.0) - d { g = -(one - a); }  // val == -d -> 0 (BUG)
der1[ABSOLUTE_POS] = g;
```

For a residual exactly equal to `±delta`, the scalar returns `±alpha` /
`-(1-alpha)` but the kernel — which is what `compute_gradients` actually runs in
production — returns `0`. This is a real numeric divergence between the host
reference and the shipped compute path on a loss whose entire purpose is the
robust/quantile gradient, against a ≤1e-5 parity bar.

Why the test suite misses it: `cpu_runtime_test.rs:88-95` re-derives the EXPECTED
value with the same `val.abs() < delta` scalar form, so it agrees with the scalar
helper, not the kernel — but no test object has a residual of exactly `1e-6`, so
the boundary case is never exercised. The bug is latent only because the corpora
happen not to land on the boundary; default `delta = 1e-6` makes a boundary hit
unlikely-but-possible with real f32-widened residuals, and any future fixture or
user dataset that does hit it diverges.

**Fix:** Make the kernel boundary match the scalar `< delta` deadzone (use the
in-band complement `>= delta`, mirroring the correct Huber kernel at
`kernels.rs:323`):

```rust
// quantile_gradient_kernel
let abs_val = F::abs(val);
let mut g = F::new(0.0);
if abs_val >= d {                     // |val| >= delta is OUTSIDE the deadzone (== scalar `< delta` band)
    g = F::new(0.0) - (one - a);     // val < 0 arm
    if val > F::new(0.0) {
        g = a;                        // val > 0 arm
    }
}
der1[ABSOLUTE_POS] = g;
```

Apply the same `>= delta` band correction to `mae_gradient_kernel`
(`kernels.rs:67-71`). Then add a `cpu_runtime_test.rs` object with a residual
constructed to be exactly `delta` (e.g. `approx=0.0, target=1e-6`) asserting the
kernel matches `quantile_der1`/`mae_der1`, so the boundary is covered.

## Warnings

### WR-01: Exact-leaf dispatch silently routes Huber/Expectile/Poisson/Tweedie/MAPE to the quantile-median optimizer

**File:** `crates/cb-train/src/boosting.rs:749-762`

`compute_leaf_deltas`'s `Exact` arm dispatches only `LogCosh` to its own
optimizer; the catch-all `_` arm sends EVERY other loss to `exact_leaf_delta`
(the weighted alpha-quantile), defaulting `(alpha, delta)` to `(0.5, 1e-6)` for
anything that is not `Loss::Quantile`:

```rust
let (quantile_alpha, quantile_delta) = match loss {
    Loss::Quantile { alpha, delta } => (alpha, delta),
    _ => (QUANTILE_ALPHA, QUANTILE_DELTA),   // RMSE/Huber/Tweedie/... all land here
};
members.iter().map(|(r, w)| match loss {
    Loss::LogCosh => logcosh_exact_leaf_delta(r, w),
    _ => exact_leaf_delta(r, w, quantile_alpha, quantile_delta),  // wrong optimum for non-quantile losses
})
```

If any caller selects `LeafMethod::Exact` with, say, `Loss::Huber` or
`Loss::Expectile`, it silently computes the weighted MEDIAN of the leaf residuals
instead of that loss's true 1-D optimum — a wrong leaf value with no error.
Upstream restricts Exact to specific losses (the generator notes
`catboost_options.cpp:346` rejects Exact for most losses), and the current
fixtures pin Newton/Gradient for Wave-2, so this is not exercised today, but the
public `train` API accepts any `(loss, LeafMethod::Exact)` combination. This is
the WR-04 "silent wrong-derivative swallow" class: an unsupported combination
produces a plausible-but-wrong number rather than a typed error.

**Fix:** Reject unsupported `(loss, Exact)` combinations explicitly — gate via a
`validate_leaf_method(loss, method)` returning `CbError::OutOfRange`, or make the
`_` arm in `compute_leaf_deltas` error for losses without a defined Exact
optimizer. At minimum, restrict the quantile fallback to
`Loss::Mae | Loss::Quantile { .. }`.

### WR-02: `Lq` der2 produces non-finite hessians for `1 <= q < 2` near a zero residual, undetected by `validate`

**File:** `crates/cb-compute/src/loss.rs:252-255`, `crates/cb-backend/src/kernels.rs:287-301`, `crates/cb-compute/src/runtime.rs:186-192`

`Loss::validate` accepts any finite `q >= 1`, but `lq_der2 = -q*(q-1)*|r|^(q-2)`
diverges to `±inf` as `r -> 0` whenever `q < 2` (the `^(q-2)` exponent is
negative). The doc-comments acknowledge "Newton-clean only for `q >= 2`" and the
fixture pins `q = 2.0`, yet nothing prevents a caller from constructing
`Loss::Lq { q: 1.5 }` and selecting Newton, at which point a leaf with a
near-zero residual injects `inf`/`NaN` into the Newton denominator. The
`validate` doc claims it rejects parameters that "produce NaN/Inf derivatives" —
for `Lq` it does not.

**Fix:** Tighten `Loss::Lq` validation to require `q >= 2.0` (matching the only
tested/Newton-clean regime), or reject `q < 2` when the leaf method is Newton in
`validate`/`compute_leaf_deltas`.

### WR-03: All Wave-1/Wave-2 losses silently report the RMSE eval metric, masking divergence in the eval/overfit path

**File:** `crates/cb-train/src/metrics.rs:78-87`

`EvalMetric::for_loss` maps LogCosh, Lq, Huber, Expectile, Poisson, Tweedie,
MAPE, Quantile, and MAE all to `Self::Rmse`. The doc frames this as a deliberate
"parity-neutral RMSE eval surface," but the consequence is that when any of these
objectives runs with an eval set and the overfitting detector / `use_best_model`
(TRAIN-06) is active, the stop decision and best-iteration selection are driven
by an RMSE curve that does NOT match upstream's own-named metric for that loss.
For a Poisson or MAPE objective this can select a different best iteration than
catboost, breaking end-to-end parity silently (no error, and the wrong metric
still produces a monotone-looking curve).

**Fix:** When `params.eval_metric` is `None` for a loss whose upstream default
metric is not yet implemented, either require an explicit `eval_metric` (return
`CbError` if an eval set is supplied), or document loudly that eval-set-driven
early stopping is unsupported for these objectives in this phase. Do not silently
substitute RMSE for a different objective's default metric.

### WR-04: `mae_gradient_kernel` hardcodes `1e-6` / `0.5` as `F::new(...)`, losing precision under `f32` instantiation

**File:** `crates/cb-backend/src/kernels.rs:64-65`

```rust
let alpha = F::new(0.5);
let delta = F::new(1e-6);
```

`F::new(1e-6)` truncates `1e-6` to the kernel element type. Under the `f64`
instantiation used today this is fine, but the kernels are generic over
`F: Float` and the project explicitly plans `f32`/GPU instantiation in Phase 7.
At `f32`, `1e-6` and the residual comparison drift relative to the host scalar's
`f64` `1e-6`, compounding CR-01's boundary mismatch. The parametric
`quantile_gradient_kernel` avoids this by passing `delta`/`alpha` as device
arrays; `mae_gradient_kernel` should not carry a second, divergent hardcoded
path.

**Fix:** Drop `mae_gradient_kernel` and route `Loss::Mae` through
`launch_quantile_f64(approx, target, 0.5, 1e-6)` (the dispatch already proves
MAE == Quantile{0.5} in `quantile_alpha05_gradients_equal_mae`). This removes the
duplicate kernel, removes the hardcoded-literal precision hazard, and guarantees
MAE and Quantile{0.5} are bit-identical by construction rather than by test.

### WR-05: `read_one` failure is silently swallowed as an all-zeros derivative buffer

**File:** `crates/cb-backend/src/cpu_runtime.rs:130-133, 178-181, 210-213, 271-274, 411-414`

Every kernel read-back uses:

```rust
let bytes = client
    .read_one(out_handle)
    .unwrap_or_else(|_| cubecl::bytes::Bytes::from_elems(vec![0.0_f64; n]));
```

A device read failure is mapped to a buffer of zeros and returned as if it were a
valid derivative vector. Zero der1 means "no gradient" — the boosting loop would
silently produce a degenerate (all-zero-delta) tree rather than surfacing the
backend failure. `compute_gradients` returns `CbResult`, so the error channel
exists; swallowing the failure defeats it and violates the project's "no silent
failure" posture (the WR-04 silent-swallow class at the I/O boundary).

**Fix:** Propagate the read failure as a `CbError`. Change the launch helpers to
return `CbResult<Vec<f64>>` and `?` them in `compute_gradients`, rather than
`unwrap_or_else` to zeros.

### WR-06: `exact_leaf_delta` quantile sort uses `partial_cmp(...).unwrap_or(Equal)`, masking NaN residuals

**File:** `crates/cb-compute/src/leaf.rs:189`

```rust
elements.sort_by(|a, b| a.0.partial_cmp(&b.0).unwrap_or(std::cmp::Ordering::Equal));
```

If a residual is `NaN` (e.g. produced by the WR-02 `Lq` divergence, or a bad
input), `partial_cmp` returns `None` and the sort treats `NaN` as equal to
everything, yielding an arbitrary, unstable ordering and therefore an arbitrary
quantile — silently. The function documents a "stable sort over (value, weight)"
to match upstream's `StableSort`, but a `NaN` breaks the total order the stable
quantile relies on.

**Fix:** Assert/return an error when a residual is non-finite before sorting, or
filter/guard `NaN` explicitly. Tie this to the WR-02 validation fix.

## Info

### IN-01: `delta == 0` Quantile edge case diverges scalar-vs-kernel at `val == 0`

**File:** `crates/cb-compute/src/loss.rs:166-175`, `crates/cb-backend/src/kernels.rs:103-108`

With `delta == 0` (which `validate` permits), the scalar `val.abs() < 0.0` is
always false (no deadzone), but the kernel's `val > 0` / `val < 0` puts `val == 0`
exactly into the deadzone (returns 0). Folds into CR-01; noted separately because
it persists even after the `±delta` fix unless the band test handles the
`val == 0` case identically on both sides.

### IN-02: Magic cube dimension `32` duplicated across six launch helpers

**File:** `crates/cb-backend/src/cpu_runtime.rs:52, 163, 195, 238, 296, 358`

`let cube_dim = 32usize;` is repeated verbatim in every launch helper. Extract a
single `const CUBE_DIM: u32 = 32;` so the launch geometry lives in one place.

### IN-03: `compute_gradients` match arms duplicate the launch-and-fill-der2 pattern

**File:** `crates/cb-backend/src/cpu_runtime.rs:438-518`

The constant-der2 arms (Rmse `-1.0`; Mae/Quantile/Mape `0.0`) and the
gradient+hessian arms repeat near-identical structure. Not a correctness issue,
but consolidating the constant-der2 cases (and adopting WR-04's MAE→Quantile
routing) would shrink the dispatch and reduce the chance of a future arm drifting
from its scalar reference.
