# Phase 06.1: Code Review Report

**Reviewed:** 2026-06-16T05:52:36Z
**Depth:** standard
**Files Reviewed:** 9
**Status:** issues_found

## Summary

The regression-loss matrix (LogCosh, Lq, Huber, Expectile, Poisson, Tweedie, MAPE,
Quantile + MSLE metric) is a careful, well-cited transcription of upstream
`error_functions.h` / `optimal_const_for_loss.h`. I verified every der1/der2 and
both Exact-leaf solvers line-by-line against the vendored upstream source in
`catboost-master/` — the *mathematical formulas are all correct* and the sign /
tie / boundary conventions in the **scalar `*_der*` host functions** match upstream
exactly. The convention-enforcement (`Loss::validate`), error typing (no `unwrap`),
and source/test separation are all clean.

No Critical defects were found: there is no injection surface, no `unwrap`/`panic`
in new production code, no div-by-zero (every divisor is structurally `>= 1` or
guarded), and the exhaustive `Loss` matches are complete.

The findings below are correctness/robustness divergences that the 1e-5 oracle did
not catch because they live on measure-zero boundaries (exact deadzone edge) or at
sub-tolerance precision (dropped f32 narrowing). They are real divergences from
upstream and from this crate's own documented "source of truth," so they are
WARNING-classified rather than Info. The status is `issues` because WARNINGs exist.

The single most important finding is **WR-01**: the CubeCL gradient kernels for
MAE/Quantile — which are the code path that *actually trains* (training calls
`runtime.compute_gradients`, not the scalar `quantile_der1`) — disagree with both
upstream and the scalar host function at the deadzone boundary `|val| == delta`.

## Warnings

### WR-01: MAE/Quantile gradient KERNEL diverges from upstream (and from the scalar `quantile_der1`) at the deadzone boundary `|val| == delta`

**File:** `crates/cb-backend/src/kernels.rs:61-110` (`mae_gradient_kernel`, `quantile_gradient_kernel`)
**Issue:**
Upstream `TQuantileError::CalcDer` (`error_functions.h`) is:
`if (abs(val) < Delta) return 0; return (target-approx > 0) ? Alpha : -(1-Alpha)`.
The scalar host fn `quantile_der1` (loss.rs:166-175) transcribes this correctly:
at `val == delta` exactly, `|val| < delta` is **false**, so it returns `alpha`
(non-zero). But the kernels invert the branch structure:
```rust
let mut g = F::new(0.0);
if val > d        { g = a; }
else if val < -d  { g = -(1 - a); }
// val == d or val == -d falls through to g = 0
```
So at the exact boundary `val == delta` the kernel returns `0`, where upstream and
the scalar function return `alpha` (and at `val == -delta` it returns `0` where they
return `-(1-alpha)`). Training drives off `compute_gradients` → these kernels
(boosting.rs:1547), **not** the scalar `quantile_der1` (which is only exercised in
unit tests). The 1e-5 oracle passed only because no fixture residual lands exactly
on `±delta` (`delta = 1e-6`; an f64-exact tie is measure-zero). This is a latent
parity bug and a contradiction between the kernel and the cited source-of-truth.
**Fix:** Mirror upstream's structure — test the deadzone first, then the strict
sign — so the boundary value gets the non-zero gradient:
```rust
let one = F::new(1.0);
let a = alpha[0];
let d = delta[0];
let val = target[ABSOLUTE_POS] - approx[ABSOLUTE_POS];
// |val| < delta -> deadzone 0; else sign(val>0?alpha:-(1-alpha))
let mut g = F::new(0.0);
if F::abs(val) < d {
    g = F::new(0.0);
} else if val > F::new(0.0) {
    g = a;
} else {
    g = F::new(0.0) - (one - a);
}
der1[ABSOLUTE_POS] = g;
```
Apply the same correction to `mae_gradient_kernel` (alpha=0.5, delta=1e-6).

### WR-02: Exact-leaf solvers drop upstream's final `float` narrowing of the leaf delta

**File:** `crates/cb-compute/src/leaf.rs:165-233` (`exact_leaf_delta`), `crates/cb-compute/src/leaf.rs:262-310` (`logcosh_exact_leaf_delta`)
**Issue:**
Upstream `CalculateWeightedTargetQuantile` and `CalculateOptimalConstApproxForLogCosh`
both **return `float`** (`optimal_const_for_loss.h:69`, `:120`). The final value —
`q` after the `q -= delta` / `q += delta` adjustment, and `left` for LogCosh — is
narrowed to f32 on return. The Rust ports return `f64` and never apply that final
narrowing: `exact_leaf_delta` returns the f64 `quantile`, `logcosh_exact_leaf_delta`
returns the f64 `left`. The chosen quantile value itself is f32-exact (it came
through `f64::from(f32)`), but `q ± delta` and the bisection midpoint are full-f64,
so the leaf delta can differ from upstream by up to ~f32 epsilon (~1e-7). This is
under the 1e-5 bar today, but it is a genuine transcription gap from the documented
"transcribed verbatim from upstream" claim and erodes the parity margin.
**Fix:** Narrow the return through f32 to match upstream's `float` return type:
```rust
// exact_leaf_delta, final line:
f64::from(quantile as f32)
// logcosh_exact_leaf_delta, final line:
f64::from(left as f32)
```

### WR-03: `Loss::validate` omits upstream's `Quantile delta <= 1e-2` upper bound

**File:** `crates/cb-compute/src/runtime.rs:223-234` (`Loss::validate`, `Quantile` arm)
**Issue:**
Upstream `TQuantileError` asserts `Y_ASSERT(Delta >= 0 && Delta <= 1e-2)`
(`error_functions.h`, the two-arg ctor). The Rust guard checks only
`delta >= 0` and `delta.is_finite()` — the `delta <= 1e-2` ceiling is missing. A
caller passing `Quantile{alpha: 0.5, delta: 5.0}` would be accepted here but would
crash upstream's assert, so the Rust path silently trains a model upstream rejects
— a parity/validation gap (the threat register T-06.1.03-01 intent was to mirror
upstream's domain checks). No NaN results (the deadzone just grows), so this is a
WARNING not a BLOCKER.
**Fix:** Add the upper bound to the `Quantile` arm:
```rust
if !delta.is_finite() || delta < 0.0 || delta > 1e-2 {
    return Err(cb_core::CbError::OutOfRange(format!(
        "Quantile delta must be finite and in [0, 1e-2], got {delta}"
    )));
}
```

### WR-04: Kernel-launch read-back failures silently substitute all-zero derivatives instead of erroring

**File:** `crates/cb-backend/src/cpu_runtime.rs:130-133, 178-182, 210-214, 271-275, 314-318, 411-415`
**Issue:**
Every launcher ends with
`client.read_one(out_handle).unwrap_or_else(|_| Bytes::from_elems(vec![0.0_f64; n]))`.
If the device read-back fails, the function returns a buffer of all-zero
gradients/hessians and the caller (`compute_gradients`, which returns `CbResult`)
never learns the launch failed. All-zero der1 produces a degenerate (all-leaves-0)
tree rather than a loud error — a silent-corruption path that defeats the 1e-5
oracle's purpose (it would produce a wrong-but-finite model instead of failing).
`compute_gradients` already returns `CbResult`, so the error is propagatable. (This
fallback predates Plan 06.1 in the RMSE/Logloss launchers, but the three new
launchers — `launch_quantile_f64`, `launch_param_f64`, `launch_poisson_hessian` —
replicate it.)
**Fix:** Have the launchers return `CbResult<Vec<f64>>` and map a read failure to a
typed `CbError` (e.g. `CbError::Backend`/`Degenerate`) propagated by
`compute_gradients`, instead of fabricating zeros:
```rust
let bytes = client.read_one(out_handle)
    .map_err(|e| CbError::Degenerate(format!("kernel read-back failed: {e:?}")))?;
```

### WR-05: Poisson/Tweedie exp-link derivatives have no overflow guard on the raw approx

**File:** `crates/cb-compute/src/loss.rs:339-383` (`poisson_der1/2`, `tweedie_der1/2`), `crates/cb-backend/src/kernels.rs:422-500`
**Issue:**
`poisson_der1 = target - approx.exp()` and the Tweedie `e^((2-p)*approx)` term grow
unbounded with the raw approx. A large positive approx (which can accumulate over
many boosting rounds, or from a mis-scaled target) overflows f64 `exp` to `+inf`,
making der1/der2 `inf`/`NaN` that then poison the histogram and leaf reduction. The
threat register dispositioned T-06.1.02-01 as "accept" because the positive-target
fixture keeps the approx small and `compare_stage` NaN-rejects — but that protects
only the test, not a production caller training a real Poisson/Tweedie model. There
is no clamp on the exponent argument (upstream relies on `IsStoreExpApprox` storage
plus its own approx-update clipping, which this raw-inline path does not reproduce).
**Fix:** This is acceptable to defer if documented, but at minimum add a NaN/Inf
assertion or clamp the exponent argument (e.g. `approx.min(EXP_ARG_MAX)`) so an
overflow surfaces as a typed error rather than silently corrupting the reduction.
Track explicitly rather than leaving it as an implicit test-only invariant.

## Info

### IN-01: Scalar `*_der*` host functions are dead relative to the training path (only unit tests use them)

**File:** `crates/cb-compute/src/loss.rs:166-410` (all new `quantile_/lq_/huber_/expectile_/poisson_/tweedie_/mape_der*`)
**Issue:**
Training computes derivatives exclusively via the CubeCL kernels
(`compute_gradients`). The scalar `*_der1`/`*_der2` host functions are referenced
only from `loss_test.rs`. They are valuable as the documented oracle and as the
single-source-of-truth the kernels are meant to mirror, but the kernel-vs-scalar
divergence in WR-01 shows the two implementations can drift undetected because no
production code cross-checks them. Consider a debug-assert or a shared
test that feeds identical inputs through both paths across the deadzone/band
boundaries.
**Fix:** Add a boundary-grid equivalence test (kernel der1 vs scalar der1 at
`val ∈ {±delta, ±delta±ε, 0}`) so a future kernel edit cannot silently diverge.

### IN-02: `mae_gradient_kernel` hardcodes `delta = 1e-6` / `alpha = 0.5` as float literals

**File:** `crates/cb-backend/src/kernels.rs:64-65`
**Issue:**
`let alpha = F::new(0.5); let delta = F::new(1e-6);` duplicates the
`QUANTILE_ALPHA`/`QUANTILE_DELTA` constants defined in `loss.rs:150-152`. Since
`Loss::Mae` is exactly `Quantile{0.5, 1e-6}` (the summary's own MAE==Quantile{0.5}
decision), the MAE kernel could be removed entirely in favor of dispatching MAE
through `quantile_gradient_kernel` with the default arrays, eliminating the
duplicated magic numbers and a second boundary-bug site (WR-01 must currently be
fixed in two kernels).
**Fix:** Route `Loss::Mae` through `launch_quantile_f64(approx, target, 0.5, 1e-6)`
and delete `mae_gradient_kernel`, or at minimum reference the named constants.

### IN-03: MSLE `GetFinalError` divisor omits upstream's `+ 1e-38` denominator epsilon

**File:** `crates/cb-train/src/metrics.rs:190` (`Msle` arm)
**Issue:**
The cited upstream `GetFinalError = Stats[0] / (Stats[1] + 1e-38)`
(`metric.cpp:1899-1926`), but the Rust computes `sum_f64(&weighted_sq) / total_weight`
with no `+1e-38`. The `total_weight <= 0` guard already prevents div-by-zero, so the
result is numerically identical for any non-degenerate eval set; this is a faithful-
transcription nit, not a correctness bug, but it is a (sub-tolerance) deviation from
the quoted formula.
**Fix:** Either add `+ 1e-38` to the denominator to match the citation verbatim, or
adjust the doc comment to note the intentional simplification (the explicit
`total_weight > 0` guard makes the epsilon redundant).

### IN-04: `EvalMetric` derives `Eq` but is documented to grow float-carrying metrics

**File:** `crates/cb-train/src/metrics.rs:42` (`#[derive(... Eq)]`)
**Issue:**
`EvalMetric` currently has only unit variants (`Rmse`/`Logloss`/`Msle`) so `Eq` is
valid, but `Loss` in the same workspace deliberately drops `Eq` because it carries
`f64` params (`runtime.rs:46`). If a future parametric metric (e.g.
`Quantile{alpha}` eval) is added to `EvalMetric`, the `Eq` derive will fail to
compile and force a churned change. Minor consistency note, not a defect.
**Fix:** None required now; be aware `Eq` must drop if a float-carrying metric
variant is added (mirroring the `Loss` PartialEq-only decision).
