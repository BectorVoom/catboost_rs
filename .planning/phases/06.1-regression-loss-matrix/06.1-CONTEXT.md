# Phase 6.1: Regression-Loss Matrix - Context

**Gathered:** 2026-06-16
**Status:** Ready for planning

<domain>
## Phase Boundary

Implement every **named scalar** CatBoost regression loss on the **existing scalar boosting loop** (no N-dim refactor — that is 6.2), each oracle-locked ≤1e-5 vs upstream **catboost 1.2.10** (`thread_count=1`) per-stage (splits/leaves/staged-approx + final prediction). This is the narrowest, lowest-risk slice of the Phase 6 split (umbrella `06-CONTEXT.md` D-01/D-02), landed first so the N-dim refactor (6.2) arrives on stable, proven loss code.

**Requirement:** LOSS-03 (scalar portion).

**In scope — the NEW scalar losses (RMSE and MAE already exist from Phases 3/4):**
- **Quantile**(alpha, delta) — generalizes the existing `Loss::Mae` (MAE = Quantile α=0.5).
- **LogCosh**, **Lq**(q), **Huber**(delta), **Expectile**(alpha) — smooth losses with a real der2.
- **Poisson**, **Tweedie**(variance_power), **MAPE** — positive-domain / link-function count losses.
- **MSLE** — implemented as an **eval_metric only** (see D-6.1-06): upstream catboost 1.2.10 does **not** expose MSLE as a trainable objective (`enum_helpers.cpp:200,533-549`), so it has no training oracle. It is oracle-locked as a metric, not a training loss.

**NOT in this phase:**
- **MultiQuantile** — relocated to **6.2** (it is multi-output: approx dimension = number of quantiles, so it needs the N-dim machinery 6.2 builds). ROADMAP/REQUIREMENTS updated to reflect LOSS-03 = 6.1 scalar matrix + 6.2 MultiQuantile.
- The N-dim approx refactor, multiclass/multilabel, ranking, score-fns/uncertainty, text/embedding, advanced features — all later sub-phases.
- Any regression loss **not** explicitly named above ("etc." in the SC) → deferred-to-v2 (umbrella D-06), not silently in-scope.

</domain>

<decisions>
## Implementation Decisions

### Scope & Sequencing
- **D-6.1-01: MultiQuantile moves to 6.2.** MultiQuantile is multi-output (a vector of quantile predictions per object, approx_dimension = K), so it cannot ride 6.1's scalar loop without a throwaway ad-hoc path. It lands in 6.2 on the N-dim approx foundation instead. ROADMAP 6.1 SC + requirements, ROADMAP 6.2 SC#4 + requirements, the summary checklist, and the REQUIREMENTS LOSS-03 traceability row were all updated in this discussion. Rejected: keep-in-6.1-gated-on-6.2 (cross-phase dependency, muddier); implement-ad-hoc-now (throwaway multi-output code 6.2 rips out).
- **D-6.1-02: Grouped family waves, each with its own oracle gate** (additive narrowest-first, umbrella D-05/D-07). Provisional grouping (planner finalizes exact wave-to-plan mapping):
  - **Wave 1 — smooth (have der2):** LogCosh, Lq(q), Huber(delta), Expectile(alpha).
  - **Wave 2 — positive-domain / link losses:** Poisson, Tweedie(variance_power), MAPE, MSLE.
  - **Wave 3 — quantile family:** Quantile(alpha, delta) generalizing the existing `Loss::Mae`.
  Each wave is its own plan + its own ≤1e-5 oracle gate → small blast radius, unambiguous regression attribution. Rejected: single all-in-one plan (coarse gate, blurs which loss regressed); per-loss plan (≈10 plans, excess ceremony).

- **D-6.1-06: MSLE is descoped as a training objective; implemented as eval_metric only** (research finding, user-confirmed 2026-06-16). Upstream catboost 1.2.10 lists MSLE as metric-only — it is absent from `RegressionObjectives` (`enum_helpers.cpp:533-549`), tagged metric-only at `enum_helpers.cpp:200`, and has no `IDerCalcer`/`BuildError` training case; `loss_function=MSLE` throws upstream. Therefore **no training oracle can exist** for MSLE, and SC#1's ≤1e-5 training-parity bar is unachievable for it. 6.1 implements MSLE as an `eval_metric` (a valid upstream metric, oracle-locked as a metric) and drops it from the trainable-loss matrix. ROADMAP SC#1 updated to match. Rejected: implement-as-objective-anyway (no upstream oracle → violates the parity guarantee); drop-entirely (loses the valid metric parity).

### Loss-Parameter API Surface
- **D-6.1-03: Extend the established `Loss::Variant { params }` enum pattern** (`cb-compute::runtime::Loss`, already used by `Focal { alpha, gamma }`). Parametric losses carry their hyperparameters as enum-struct fields with **CatBoost-native names** and **defaults transcribed from `error_functions.h`**:
  - `Quantile { alpha, delta }` (α default 0.5, δ default 1e-6 — already pinned in the existing MAE/Quantile code), `Expectile { alpha }`, `Huber { delta }`, `Tweedie { variance_power }`, `Lq { q }`. Non-parametric: `LogCosh`, `Poisson`, `MAPE`, `MSLE`.
- **D-6.1-04: CatBoost-native loss-string parsing (`"Quantile:alpha=0.7"`) is a Phase-8 / Builder concern, layered on top** — not built in 6.1. The enum-variant-with-params is the *core* representation; 6.1 only needs the typed Rust surface oracle-tested. Rejected: a flat `LossParams` struct decoupled from the enum (diverges from the `Focal{...}` precedent).

### Leaf Estimation (parity-dictated — confirm during research)
- **D-6.1-05: Wire the existing Phase-3 (03-02) leaf-estimation methods per upstream default, do not invent new ones.** Non-smooth losses with `der2 = 0` (Quantile/MAE, and MAPE) use **Exact** leaf estimation (weighted α-quantile / weighted median); the smooth losses (LogCosh/Huber/Lq/Expectile/Poisson/Tweedie/MSLE) use **Newton**. **Dependency flag for research:** confirm the 03-02 Exact method handles a **weighted α-quantile with α ≠ 0.5** (today it is exercised as the median for MAE); generalizing Exact to arbitrary α is a likely 6.1 task, not a free reuse.

### Claude's Discretion (parity-dictated — research reads upstream and reproduces)
- Exact der1/der2 formulas, parameter defaults, prediction transforms (Poisson exp-link, MSLE log-space, MAPE scaling) for each named loss — transcribe from `error_functions.{h,cpp}`.
- The exact wave-to-plan mapping and how many losses share a single plan within each family wave.
- Which leaf-estimation method each individual loss defaults to (pin from upstream `leaf_estimation_method` defaults).
- Oracle fixture datasets/configs (reuse of the existing numeric corpus, per-loss param fixtures, default depth/lr/iterations) — Python-reachable per-stage floor (umbrella D-07; no C++ instrumentation for 6.1).

</decisions>

## Existing Code Insights

### Reusable Assets
- `cb-compute::runtime::Loss` (`runtime.rs:47`) — new losses attach here; `Focal { alpha, gamma }` shows the parametric-variant shape (D-6.1-03).
- `cb-compute::loss` — RMSE (`rmse_der1`/`rmse_der2`), MAE/Quantile (`mae_der1`, `der2=0`, α=0.5/δ=1e-6) already implemented; MAE is literally Quantile(0.5), so Quantile generalizes existing code.
- Phase-3 (03-02) leaf-estimation: Gradient / Newton / **Exact** (weighted median today) / Simple — reused per D-6.1-05.
- `cb-oracle` per-stage `compare_stage` ≤1e-5 harness + frozen fixtures.

### Established Patterns
- **Scalar-`approx` everywhere** (`approx: f64`) — 6.1 stays on this; the N-dim generalization is explicitly 6.2 (D-03/D-04). 6.1 must NOT pre-empt the refactor.
- **Per-fn upstream transcription with `error_functions.*` file:line citations** in `loss.rs` — the new losses follow this exact documentation style.
- **All parity-critical summation through `cb-core::sum_f64`** (Phase-2 D-07, D-08 CI-grep).
- **Source/test separation mandatory**; `thiserror` in libs; `unwrap()` banned in production.

### Integration Points
- New loss → `Loss` enum variant + der1/der2 dispatch → consumed by the train loop's leaf-estimation + score path → predicted via the existing scalar apply.
- Non-smooth losses (Quantile/MAE/MAPE) → Exact leaf estimation (weighted quantile) — the one spot that may need real new work (α≠0.5 generalization), not just a transcription.

</code_context>
