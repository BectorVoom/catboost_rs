## Fixed Issues

### CR-01: Quantile/MAE gradient kernels disagree with scalar at the deadzone boundary

**Files modified:** `crates/cb-backend/src/kernels.rs`, `crates/cb-backend/src/cpu_runtime_test.rs`
**Commit:** f03a5c3
**Status:** fixed: requires human verification (numeric boundary / oracle parity)
**Applied fix:** Replaced the inverted `val > d` / `val < -d` band in both
`quantile_gradient_kernel` and `mae_gradient_kernel` with the `|val| >= delta`
band complement (mirroring the correct `huber_gradient_kernel`), so the boundary
`|val| == delta` returns the signed quantile weight exactly as the scalar
`quantile_der1`/`mae_der1` do. The `g` variable now defaults to `0` (deadzone),
is set to the `val < 0` arm `-(1-alpha)` inside the band, then overwritten by the
`val > 0` arm `alpha` — which also makes the `val == 0, delta == 0` edge case
(IN-01) match the scalar's `else` branch. Added a `cpu_runtime_test.rs` boundary
regression test asserting the launched kernel matches `quantile_der1`/`mae_der1`
for residuals constructed exactly at `±delta`.
Human verification: confirm the existing Quantile / MAE oracle fixtures still
hold to ≤1e-5 and that no fixture residual now crosses the boundary differently.

### WR-01 / WR-02: Gate unsupported (loss, leaf_method) combinations

**Files modified:** `crates/cb-train/src/boosting.rs`
**Commit:** 22b9e56
**Status:** fixed
**Applied fix:** Added `validate_leaf_method(loss, method)` returning
`CbError::OutOfRange`, called at the central `train_*` chokepoint right after
`params.loss.validate()?`. It rejects (a) `LeafMethod::Exact` for any loss
without a defined Exact optimizer — only `LogCosh`, `Mae`, `Quantile` are
dispatched; every other loss previously fell through to the weighted-median
fallback silently (WR-01); and (b) `Lq{q<2}` with `LeafMethod::Newton`, whose
`-q*(q-1)*|r|^(q-2)` hessian diverges to inf/NaN near a zero residual (WR-02).
All existing fixtures (LogCosh/Quantile Exact; Lq{2.0}/Huber/Expectile Newton)
remain valid and pass the gate. The `compute_leaf_deltas` unit test in
`boosting_test.rs` calls the helper directly (below the gate) and is unaffected.

Note: WR-02's alternative "tighten `Loss::validate` to require `q >= 2`
globally" was deliberately NOT chosen — it would reject `q ∈ [1, 2)` even for
non-Newton leaf methods, narrowing the documented `Lq: q >= 1` contract. Gating
only the Newton + `q<2` combination is the more targeted fix the review offered.

### WR-05: Device read-back failure swallowed as all-zeros derivative buffer

**Files modified:** `crates/cb-backend/src/cpu_runtime.rs`
**Commit:** 5e208df
**Status:** fixed
**Applied fix:** Changed all five launch helpers (`launch_binary_f64`,
`launch_logloss_hessian`, `launch_poisson_hessian`, `launch_focal_f64`,
`launch_quantile_f64`, `launch_param_f64`) to return `CbResult<Vec<f64>>`,
replaced `read_one(...).unwrap_or_else(|_| zeros)` with
`read_one(...).map_err(|e| CbError::Degenerate(...))?`, and `?`-propagated each
call in `compute_gradients`. A backend read failure now surfaces on the existing
`CbResult` channel instead of masquerading as a valid no-gradient vector.

### WR-06: `exact_leaf_delta` quantile sort masks NaN residuals

**Files modified:** `crates/cb-compute/src/leaf.rs`
**Commit:** 85244a0
**Status:** fixed
**Applied fix:** Replaced
`partial_cmp(...).unwrap_or(std::cmp::Ordering::Equal)` (which treats a NaN as
equal to every element, collapsing the total order into an arbitrary, unstable
quantile) with `f32::total_cmp`, a true total order with deterministic NaN
placement. For all-finite inputs (the tested/oracle regime) `total_cmp` agrees
with `partial_cmp`, so upstream `StableSort` parity is preserved while the silent
nondeterminism is removed.

Note: the review's preferred variant (return `CbError` on a non-finite residual)
was NOT chosen because it requires changing `exact_leaf_delta`'s `-> f64`
signature to `-> CbResult<f64>`, cascading through all callers in the boosting
loop. The WR-01/WR-02 gate already prevents the main NaN source (Lq{q<2} Newton),
and `total_cmp` makes the remaining behavior deterministic without an API break.

### WR-04: `mae_gradient_kernel` hardcodes `1e-6`/`0.5`, losing precision under f32

**Files modified:** `crates/cb-backend/src/cpu_runtime.rs`, `crates/cb-backend/src/kernels.rs`
**Commit:** 0545aeb
**Status:** fixed: requires human verification (MAE compute path changed)
**Applied fix:** Routed `Loss::Mae` through
`launch_quantile_f64(approx, target, QUANTILE_ALPHA, QUANTILE_DELTA)` and deleted
the duplicate `mae_gradient_kernel` plus its `BinaryKernel::MaeGradient` variant
and match arm. `delta`/`alpha` now pass as length-1 `f64` device arrays (the
generics-float-correct path) instead of hardcoded `F::new(1e-6)`/`F::new(0.5)`,
removing the second divergent path that would drift under a future f32/GPU
instantiation. MAE and Quantile{0.5} are now bit-identical by construction,
guarded by the existing `quantile_alpha05_gradients_equal_mae` test.
Human verification: confirm the existing MAE oracle fixture (`leaf_methods/exact`)
still holds to ≤1e-5 with MAE now dispatched through the quantile kernel.

### IN-01: `delta == 0` Quantile edge case diverges scalar-vs-kernel at `val == 0`

**Files modified:** (resolved by CR-01) `crates/cb-backend/src/kernels.rs`
**Commit:** f03a5c3
**Status:** fixed (folded into CR-01)
**Applied fix:** The CR-01 `|val| >= delta` band with the `val < 0` default arm
handles `delta == 0, val == 0` identically to the scalar: `|0| >= 0` is true, the
`val > 0` test is false, so both return `-(1-alpha)`. No separate change needed;
the boundary test added in CR-01 also covers a `delta`-band control object.

### IN-02: Magic cube dimension `32` duplicated across six launch helpers

**Files modified:** `crates/cb-backend/src/cpu_runtime.rs`
**Commit:** 8a7645c
**Status:** fixed
**Applied fix:** Hoisted the repeated `let cube_dim = 32usize;` into a single
module-level `const CUBE_DIM: usize = 32;` and replaced all six occurrences with
`let cube_dim = CUBE_DIM;` (keeping the local name so the `div_ceil(cube_dim)` /
`cube_dim as u32` usages are unchanged).

## Skipped Issues

### WR-03: All Wave-1/Wave-2 losses silently report the RMSE eval metric

**File:** `crates/cb-train/src/metrics.rs:78-87`
**Reason:** skipped: requires a design / scope decision (and likely oracle
re-derivation). The RMSE eval surface is an explicitly documented, deliberate
"parity-neutral" choice; the proposed fixes (return `CbError` when an eval set is
supplied for these losses, or hard-document early-stopping as unsupported) are
behavioral/API changes that would likely break existing eval-set oracle tests
and depend on deciding which own-named metrics are in scope for this phase versus
deferred to a later one. Applying either speculatively risks regressing the
eval/overfit pipeline. Flagged for maintainer decision rather than auto-fixed.
**Original issue:** `EvalMetric::for_loss` maps all regression losses to
`Self::Rmse`; with an eval set + overfitting detector / `use_best_model` active,
the stop decision is driven by an RMSE curve that does not match upstream's
own-named metric, which can select a different best iteration than catboost and
break end-to-end parity silently.

### IN-03: `compute_gradients` match arms duplicate the launch-and-fill-der2 pattern

**File:** `crates/cb-backend/src/cpu_runtime.rs:438-518`
**Reason:** skipped: cosmetic readability refactor with no correctness impact
(the review states "Not a correctness issue"). Each match arm currently carries
valuable inline per-loss parity documentation (e.g. "der2 = 0, Newton undefined
— Pitfall 5"); consolidating the arms risks obscuring that per-loss provenance
for marginal gain. The WR-04 MAE→Quantile routing (already applied) addressed the
portion of IN-03 that touched correctness/dedup. Deferred as a low-priority
cleanup.
**Original issue:** The constant-der2 arms (RMSE `-1.0`; Mae/Quantile/Mape `0.0`)
and the gradient+hessian arms repeat near-identical structure.
