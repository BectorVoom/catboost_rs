## Summary

Phase 6.1 attaches the named scalar regression losses to the existing Phase-3/4/5 scalar boosting loop and oracle-locks each ≤1e-5 vs upstream catboost 1.2.10. The research's core value is **upstream-formula extraction + scope de-risking**, and three findings reshape the plan materially:

1. **MSLE is NOT a trainable objective in catboost 1.2.10.** It is registered only as a metric (`EMetricAttribute::IsRegression`, `enum_helpers.cpp:200`), is absent from the authoritative `RegressionObjectives` list (`enum_helpers.cpp:533-549`), and has no `IDerCalcer` case in the `BuildError` dispatch (`tensor_search_helpers.cpp:160-318` — it falls through to `default: not supported`). **You cannot generate a training/per-stage oracle for MSLE** — `loss_function=MSLE` throws upstream. The planner must descope MSLE-as-objective (treat as deferred / metric-only) rather than write an unreachable oracle. This is a BLOCKING scope correction to the CONTEXT in-scope list.

2. **D-6.1-05 is largely a FREE REUSE.** `cb_compute::leaf::exact_leaf_delta` (`leaf.rs:165-233`) ALREADY accepts arbitrary `alpha: f64, delta: f64` and implements the full upstream weighted-quantile + delta adjustment (`quantile.cpp` / `optimal_const_for_loss.h`), with tests at α=0.5, α=0 (min), and weighted quantiles (`leaf_test.rs:90-119`). The α≠0.5 generalization for Quantile is NOT new algorithm work — only the wiring in `compute_leaf_deltas` (`boosting.rs:709-723`, which hardcodes `QUANTILE_ALPHA`/`QUANTILE_DELTA`) needs to be parameterized to pass the loss's α/δ.

3. **Upstream's default `leaf_estimation_method` per loss diverges sharply from D-6.1-05's assumption** (`catboost_options.cpp:40-239`). MAE/MAPE/Quantile/LogLinQuantile default to **Gradient** (not Exact); **LogCosh defaults to Exact** (not Newton); Lq defaults to Gradient if q<2 else Newton; Expectile defaults to Newton with 5 iterations; Poisson to Newton with 10 iterations. Because `leaf_method` is an explicit per-config `BoostParams` field (`boosting.rs:117`) and the existing MAE oracle fixture pins `leaf_estimation_method:"Exact"` explicitly (`fixtures/leaf_methods/exact/config.json`), the safest 6.1 strategy is to **pin `leaf_estimation_method` and `leaf_estimation_iterations:1` explicitly in every fixture config** so cb-train's single-iteration leaf path matches — rather than reproducing the auto-default switch.

The der1/der2 formulas for every trainable in-scope loss are short, all isExpApprox-aware, and all transcribe cleanly in the existing `loss.rs` style. The one genuinely-new piece of machinery is the **exp-link family**: Poisson (and LogLinQuantile, if in scope) are `IsStoreExpApprox=true` (`approx_updater_helpers.h:60-72`); cb-train stores raw approx (never implemented exp-approx storage), so Poisson must compute `exp(rawApprox)` inside its der functions (the same way Logloss computes `sigmoid` inline) and predict via the `Exponent` prediction type. **Tweedie is NOT exp-approx** (`CB_ENSURE(isExpApprox==false)`, `error_functions.h:1644`) — it applies `exp()` inside its der formula directly.

**Primary recommendation:** Group into three waves matching D-6.1-02 but with the leaf-method corrections below; descope MSLE-as-objective; treat Quantile-family Exact-α as a wiring change (free), and Poisson exp-link as the single new-machinery item; pin `leaf_estimation_method` + `leaf_estimation_iterations:1` in every fixture.

## Per-Loss der1/der2, Parameters, Transforms, Leaf-Method

All formulas transcribed from `catboost-master/catboost/private/libs/algo_helpers/error_functions.h` (catboost 1.2.10). `der1 = CalcDer`, `der2 = CalcDer2`. Upstream applies the per-object weight AFTER the derivative (`error_functions.cpp:46-59`), so cb-compute helpers take the unweighted scalar form (matching the existing RMSE/MAE pattern). **`approx` below is the value `CalcDer` receives**: for exp-approx losses it is `exp(rawApprox)`; cb-train stores raw approx, so the equivalent raw-form is noted.

### Wave 1 — smooth (real der2), Newton/Exact/Gradient leaf

| Loss | der1 | der2 | Params (default) | Leaf method (upstream default) | Citation |
|------|------|------|------------------|-------------------------------|----------|
| **LogCosh** | `-tanh(approx - target)` | `-1/cosh²(approx - target)` | none | **Exact** (newtonIter=1) ⚠️ not Newton | `error_functions.h:405-425`; leaf `catboost_options.cpp:65-70` |
| **Lq(q)** | `q·sign(target-approx)·\|approx-target\|^(q-1)` | `-q·(q-1)·\|target-approx\|^(q-2)` | `q` **MANDATORY, no default**, `q≥1` | **Gradient if q<2 else Newton** | `error_functions.h:539-568`; leaf `catboost_options.cpp:83-93`; param `loss_description.cpp:195-200` |
| **Huber(delta)** | `\|diff\|<δ ? diff : (diff>0?δ:-δ)`, `diff=target-approx` | `\|diff\|<δ ? -1 : 0` | `delta` **MANDATORY, no default** | **Newton** (newtonIter=1) | `error_functions.h:1596-1632`; leaf `catboost_options.cpp:187-192`; param `loss_description.cpp:202-207` |

### Wave 3 — quantile family

| Loss | der1 | der2 | Params (default) | Leaf method | Citation |
|------|------|------|------------------|-------------|----------|
| **Quantile(alpha, delta)** | `val=target-approx; \|val\|<δ ? 0 : (val>0 ? α : -(1-α))` | `0` | `alpha` **0.5**, `delta` **1e-6** | **Gradient** (upstream default) — BUT existing MAE oracle uses **Exact** explicitly | `error_functions.h:457-498`; leaf `catboost_options.cpp:113-124`; dispatch `tensor_search_helpers.cpp:178-186` |

**Notes:**
- **Lq maxDerivativeOrder:** Lq sets `maxDerivativeOrder = q>=2 ? 3 : 1` (`error_functions.h:545`). For **q<2, der2 is not computed** (only der1), which is why upstream forces **Gradient** leaf (`catboost_options.cpp:87-89`). For 6.1, prefer fixtures with **q≥2** (Newton-clean der2) OR q<2 with Gradient leaf; document which per fixture.
- **MAPE's `max(1.f, Abs(target))`:** the `1.f` is a **float** literal (`error_functions.h:620`) — the divisor is `max(1.0_f32_widened, |target|)`. Transcribe as `f64::max(1.0, target.abs())`; the difference vs an f32 max is below 1e-5 for non-pathological targets but pin the exact form.
- **der2=0 losses (MAE, MAPE, Quantile):** Newton is undefined (denominator = scaledL2 alone). `newton_leaf_delta` already guards `denom>0` → returns the gradient-equivalent. Upstream routes these to **Gradient** leaf; the existing MAE oracle uses **Exact**. Pick ONE per fixture and pin it.

## Exp-Link (Poisson) — the one new-machinery item

[VERIFIED: error_functions.cpp:13-61 + approx_updater_helpers.h:60-72]

Upstream stores `approx = exp(rawApprox)` for `IsStoreExpApprox` losses (**Poisson, LogLinQuantile**, plus Logloss/CrossEntropy/PairLogit/YetiRank). `CalcDer` receives the already-exponentiated approx. The approx-UPDATE path uses `UpdateApprox<UseExpApprox>` which **multiplies** (`approx *= exp(delta)`) rather than adds.

**cb-train does NOT implement exp-approx storage.** It stores raw approx even for Logloss (`loss.rs:21` "stored approx is the raw logit") and computes `sigmoid(rawApprox)` inline. The same equivalence works for Poisson:
- Raw-form der1 = `target - exp(rawApprox)`, der2 = `-exp(rawApprox)` — compute `exp()` inside the der kernel (exactly the inline-`sigmoid` precedent).
- The additive raw-approx update stays unchanged (boosting loop adds `lr·leafDelta` to raw approx).
- **Final prediction transform = `Exponent`** (`cb-model` PredictionType::Exponent already exists, oracle-locked Plan 04-02, `RawFormulaVal` exp'd). The per-stage StagedApprox oracle compares RAW approx; the Predictions stage compares `exp(rawApprox)`.

**Risk:** confirm via fixture whether catboost reports staged approx as raw or exp for Poisson — the existing `staged_layout` note says "raw approximant / logit", so StagedApprox is RAW; Predictions need the Exponent transform. Validate empirically with the first Poisson fixture (Open Question Q1).

**Tweedie** stays on raw approx with `exp()` inside its der formula — no exp-approx machinery, no Exponent predict transform (Tweedie's prediction IS the raw approx; its der already accounts for the link). Confirm Tweedie's prediction_type in the fixture.

### Pattern 1: Parametric Loss Enum Variant (D-6.1-03)
**What:** Each parametric loss carries its hyperparameters as enum-struct fields, CatBoost-native names.
**When to use:** All of Quantile/Expectile/Huber/Tweedie/Lq.
**Example** (the existing Focal precedent, `runtime.rs:63-68`):
```rust
// Source: crates/cb-compute/src/runtime.rs:63
Focal {
    alpha: f64,   // class-balance weight
    gamma: f64,   // focusing exponent
},
// New (transcribe defaults from error_functions.h):
Quantile { alpha: f64, delta: f64 },     // default 0.5, 1e-6
Huber    { delta: f64 },                  // MANDATORY upstream — no default
Tweedie  { variance_power: f64 },         // MANDATORY upstream — 1<p<2
Lq       { q: f64 },                      // MANDATORY upstream — q>=1
Expectile{ alpha: f64 },                  // default 0.5
```
**Note:** `Loss` intentionally does NOT derive `Eq` (it carries `f64`) — `PartialEq` only (`runtime.rs:46`). Keep this.

### Pattern 2: Inline-link der (Poisson exp / Logloss sigmoid)
**What:** Compute the link transform inside the der function instead of storing transformed approx.
**Example** (`loss.rs:54-78` sigmoid precedent → Poisson analog):
```rust
// Source pattern: crates/cb-compute/src/loss.rs:65 (logloss_der1)
pub fn poisson_der1(approx: f64, target: f64) -> f64 {
    // error_functions.h:666 — der1 = target - approxExp; cb-train stores raw,
    // so approxExp = exp(approx).
    target - approx.exp()
}
pub fn poisson_der2(approx: f64, _target: f64) -> f64 {
    -approx.exp()   // error_functions.h:670
}
```

### Pattern 3: Exact leaf with loss α/δ (D-6.1-05 wiring fix)
**What:** Pass the loss's α/δ to `exact_leaf_delta` instead of the MAE-hardcoded constants.
**Current** (`boosting.rs:721`):
```rust
.map(|(r, w)| exact_leaf_delta(r, w, QUANTILE_ALPHA, QUANTILE_DELTA))  // hardcoded 0.5, 1e-6
```
**Needed:** thread the active loss's `(alpha, delta)` through `compute_leaf_deltas` so a `Quantile{alpha:0.7}` config uses 0.7. `exact_leaf_delta` already handles it (`leaf.rs:165`, tests `leaf_test.rs:106-119`). This is the ONLY "real code" change in the quantile family.

### Anti-Patterns to Avoid
- **Implementing exp-approx storage** to mirror upstream's `UpdateApprox<true>` — unnecessary; the inline-`exp()` der + additive raw update + Exponent predict is mathematically equivalent and matches the existing Logloss approach. Hand-rolling exp-approx storage would diverge from the established pattern and risk the StagedApprox oracle.
- **Writing an MSLE training oracle** — it cannot exist (not a trainable objective). Do not author a fixture that will throw upstream.
- **Reproducing the leaf-method auto-default switch** in cb-train — pin `leaf_estimation_method` explicitly per fixture instead (matches the existing MAE-Exact fixture pattern); far smaller blast radius.
- **Raw float sums** in der/leaf code — D-08 CI-grep ban; route through `cb_core::sum_f64` (der1/der2 are scalar-only, no sum at stake).

## Common Pitfalls

### Pitfall 1: Treating MSLE as in-scope
**What goes wrong:** Authoring an MSLE training oracle; `loss_function=MSLE` throws "provided error function is not supported" upstream.
**Why:** MSLE is a metric, not an objective (`enum_helpers.cpp:200`; absent from `RegressionObjectives`).
**How to avoid:** Descope MSLE-as-objective; if LOSS-03 text demands "MSLE", document it as metric-only / v2-deferred and surface to the user (this contradicts the CONTEXT in-scope list — A1 in Assumptions Log).
**Warning signs:** No `model.json` can be produced for MSLE training.

### Pitfall 2: Wrong leaf_estimation_method default
**What goes wrong:** Assuming LogCosh=Newton or MAE=Exact by default; predictions diverge ~1e-2.
**Why:** Upstream defaults: LogCosh=**Exact**, MAE/MAPE/Quantile=**Gradient** (`catboost_options.cpp:65,113`).
**How to avoid:** PIN `leaf_estimation_method` explicitly in every fixture config (as `fixtures/leaf_methods/exact/config.json` already does); make cb-train's `BoostParams.leaf_method` match the fixture verbatim. Do not rely on the auto-default.
**Warning signs:** Leaf values off by a consistent factor vs the upstream model.json.

### Pitfall 3: leaf_estimation_iterations > 1
**What goes wrong:** Expectile (newtonIter=5), Poisson (newtonIter=10) default to multi-step leaf refinement; cb-train only implements **1 iteration** (`boosting.rs:17`).
**Why:** cb-train's leaf path is single-step; multi-iteration refinement is NOT implemented.
**How to avoid:** PIN `leaf_estimation_iterations: 1` in every fixture (the existing MAE/regression fixtures already do). If a loss MUST be tested at its multi-iteration default for parity, that is NEW work — flag to planner.
**Warning signs:** Expectile/Poisson leaf values diverge even with correct der + method.

### Pitfall 4: Poisson StagedApprox raw-vs-exp confusion
**What goes wrong:** Comparing the Predictions stage against raw approx, or StagedApprox against exp'd approx.
**Why:** Poisson is exp-approx upstream but cb-train stores raw; StagedApprox = raw, Predictions = `exp(raw)`.
**How to avoid:** StagedApprox oracle compares raw; Predictions oracle applies the `Exponent` transform first. Confirm the fixture's `staged_layout` note (Q1).
**Warning signs:** A constant `exp`-factor offset in one stage but not the other.

### Pitfall 5: der2=0 + Newton selection
**What goes wrong:** Selecting Newton for MAE/MAPE/Quantile → denominator is scaledL2 alone (or 0), NaN/inf.
**Why:** der2=0 for these (`error_functions.h:491,623`).
**How to avoid:** Use Gradient or Exact for der2=0 losses (upstream uses Gradient by default; `newton_leaf_delta` guards denom>0 → 0). Pin per fixture.
**Warning signs:** NaN leaf values.

### Pitfall 6: Lq q<2 has no der2
**What goes wrong:** Requesting Newton for Lq(q<2); only der1 is computed (`maxDerivativeOrder=1`, `error_functions.h:545`).
**Why:** `|target-approx|^(q-2)` diverges as approx→target for q<2.
**How to avoid:** Prefer q≥2 fixtures (Newton-clean) or q<2 with Gradient leaf (upstream default).
**Warning signs:** Lq der2 → inf near the residual zero.

### Pitfall 7: MAPE float-literal divisor
**What goes wrong:** Using `f64::max(1.0, |target|)` when upstream uses `Max(1.f, Abs(target))` with a float `1.f` and an f32 `Abs(target)`.
**Why:** `error_functions.h:620` — `1.f` and the `Abs` operate in the float domain before widening.
**How to avoid:** Transcribe carefully; for |target| not near 1.0 the f32/f64 max agree well within 1e-5, but pin the form and validate per fixture.
