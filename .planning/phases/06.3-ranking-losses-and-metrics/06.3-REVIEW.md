# Phase 6.3: Code Review Report

**Reviewed:** 2026-06-17T00:00:00Z
**Depth:** standard
**Files Reviewed:** 14
**Status:** issues_found

## Summary

Phase 6.3 adds eight ranking losses (QueryRMSE, QuerySoftMax, PairLogit/Pairwise, LambdaMart,
YetiRank/Pairwise, StochasticRank) and nine eval-only ranking metrics (NDCG, DCG, MAP, MRR, ERR,
PFound, PrecisionAt, RecallAt, QueryAUC) as a Rust rewrite of CatBoost gradient boosting.

Two BLOCKER-level defects were found that cause incorrect behavior with non-uniform sample weights:

1. **CR-01**: `calc_dcg_metric_diff` (StochasticRank NDCG gradient) uses unnormalized position
   weights in `doc_diff` while the cumulative prefix arrays (`cum_sum`, `cum_sum_up`,
   `cum_sum_low`) were built from normalized weights (divided by `ideal_dcg`). The two terms in
   `doc_diff + mid_diff` are on mismatched scales for the NDCG metric arm. Upstream reads
   `posWeights[oldPos]` directly (the same normalized vector used to build the cumulative arrays);
   the Rust code recomputes `1.0 / ndcg_denominator(pos)` instead, introducing a factor of
   `ideal_dcg` error in the NDCG gradient.

2. **CR-02**: The boosting loop applies `der1 * weight` universally to ALL losses, but QueryRMSE
   and QuerySoftMax return derivatives that are already multiplied by per-object weight inside
   their per-group derivative functions. At the uniform-weight (w=1.0) oracle fixtures the double
   multiplication is invisible, but any weighted training scenario will produce incorrect gradients
   for these two losses.

Three WARNING-level D-08 discipline violations exist where raw float accumulation loops were used
instead of the project-mandated `cb_core::sum_f64`. The author documented all three as "parity
contract" assertions, but the project rule admits no exception path for library-internal
summations.

No `unwrap()`/`expect()`/`panic!`/unchecked indexing on numeric buffers was found in the reviewed
source files. `sorted_indices` in `ranking_metrics.rs` and the grouped derivative dispatch in
`ranking_der.rs` are both safe. Panic-safety, division-by-zero guards, and group-boundary
handling are correct throughout.

## Critical Issues

### CR-01: `calc_dcg_metric_diff` uses unnormalized position weights for NDCG `doc_diff` term

**File:** `crates/cb-compute/src/ranking_der.rs:904-906`
**Issue:** Upstream `CalcDCGMetricDiff` (error_functions.cpp:1233-1234) reads
`oldWeight = posWeights[oldPos]` and `newWeight = posWeights[newPos]` directly from the
`posWeights` vector — the SAME normalized vector used to construct `cumSum`/`cumSumUp`/`cumSumLow`
(confirmed at error_functions.cpp:1438-1450 and lines 1233-1234 of the upstream source). For the
NDCG arm, `compute_dcg_pos_weights` normalizes each entry by `1/ideal_dcg` (ranking_der.rs:668-
671). The Rust `calc_dcg_metric_diff` instead recomputes raw `1.0 / ndcg_denominator(old_pos)`
(line 904) and `1.0 / ndcg_denominator(new_pos)` (line 905) — the pre-normalization values. As a
result, `doc_diff = doc_gain * (new_weight - old_weight)` is off by a factor of `ideal_dcg`
relative to `mid_diff`, which was derived from cumulative arrays built with the normalized weights.
The mixed-scale sum `doc_diff + mid_diff` (line 916) is incorrect for any group with
`ideal_dcg != 1.0`. The DCG arm (non-NDCG) is unaffected because `is_ndcg=false` skips the
normalization step.

**Fix:** Pass `pos_weights` into `calc_dcg_metric_diff` and read the weights from it directly,
mirroring upstream:

```rust
// Add pos_weights parameter:
fn calc_dcg_metric_diff(
    old_pos: usize,
    new_pos: usize,
    target_slice: &[f64],
    order: &[usize],
    pos_weights: &[f64],          // <-- add this
    cum_sum: &[f64],
    cum_sum_up: &[f64],
    cum_sum_low: &[f64],
) -> f64 {
    let doc_gain = ndcg_numerator(
        target_slice
            .get(order.get(old_pos).copied().unwrap_or(0))
            .copied()
            .unwrap_or(0.0),
    );
    // Use the same normalized vector that built the cumSum arrays (upstream line 1233-1234).
    let old_weight = pos_weights.get(old_pos).copied().unwrap_or(0.0);
    let new_weight = pos_weights.get(new_pos).copied().unwrap_or(0.0);
    let doc_diff = doc_gain * (new_weight - old_weight);
    // ... rest unchanged ...
}
```

And update every call site (lines 788-796) to pass `&pos_weights` as the new fourth argument.

### CR-02: Ranking derivatives double-weighted in `boosting.rs` for QueryRMSE and QuerySoftMax

**File:** `crates/cb-train/src/boosting.rs:2211-2227`
**Issue:** After the grouped-derivative path returns `ders` (lines 2177-2209), the code computes
`weighted_der1[idx] = ders.der1[idx] * weights[idx % n]` unconditionally for all losses (lines
2218-2227). However, `queryrmse_der` (loss.rs:583-584) returns `der1 = (target - approx -
query_avrg) * weight` — already multiplied by the object's weight — and `querysoftmax_der`
(loss.rs:617) returns `der1 = beta * (-sumWTargets * p + weight * target)` — also weight-folded.
The author's comment at line 2185-2187 acknowledges "the querywise der is ALREADY weighted" but
then does not skip the re-multiplication for those losses. At the uniform-weight test corpus
(w=1.0 throughout) the product `already_weighted * 1.0` is numerically identical to the correct
value, masking the error. Any weighted training scenario with QueryRMSE or QuerySoftMax will
receive squared-weight gradients and produce wrong tree structures and leaf values.

**Fix:** Skip the weight re-multiplication for grouped (ranking) losses whose derivatives are
already weight-folded. The cleanest approach is to bypass `weighted_der1` for the grouped path
entirely and use `ders.der1` directly as the "weighted" buffer:

```rust
let weighted_der1: Vec<f64> = if group_spans.is_some() {
    // Grouped ranking ders (QueryRMSE, QuerySoftMax, etc.) already fold the
    // per-object weight into der1 inside the per-group derivative function.
    // Re-multiplying would double-weight. Use as-is.
    ders.der1.clone()
} else {
    // Pointwise losses: apply the per-object weight here.
    ders.der1
        .iter()
        .enumerate()
        .map(|(idx, &d)| {
            let i = idx % n;
            let w = weights.get(i).copied().unwrap_or(1.0);
            d * w
        })
        .collect()
};
```

Note: PairLogit and LambdaMart derivatives are NOT weighted by per-object weight (the pair weight
is folded into the competitor structure, not the object weight). If any future grouped loss is
added that does NOT pre-weight its derivatives, this branching logic must be updated. Consider
adding a boolean flag on the loss discriminant or returning a `WeightFolded` marker from
`compute_gradients_grouped`.

## Warnings

### WR-01: `make_zero_average` in `pairwise_leaves.rs` uses raw `+=` instead of `sum_f64`

**File:** `crates/cb-train/src/pairwise_leaves.rs:205-208`
**Issue:** CLAUDE.md (D-08 rule) requires all parity-critical float summation to route through
`cb_core::sum_f64`. The `make_zero_average` function uses a raw sequential `average += v` loop.
The comment at line 198-199 claims "A sequential accumulation matching upstream's `average +=
res[i]` loop order (the parity contract is the loop order)." No exceptions to D-08 are documented
in CLAUDE.md. The CI grep ban on raw float accumulation will catch this.

**Fix:**
```rust
fn make_zero_average(res: &mut [f64]) {
    let n = res.len();
    if n == 0 {
        return;
    }
    let average = cb_core::sum_f64(res) / n as f64;
    for v in res.iter_mut() {
        *v -= average;
    }
}
```

If sequential summation order is genuinely required for bit-exact parity against upstream, that
requirement must be documented with a cross-referenced upstream line citation AND an exception
approved in the project summation-discipline policy. As written, no such exception pathway exists.

### WR-02: `lambdamart_ideal_ndcg` uses raw `score +=` instead of `sum_f64`

**File:** `crates/cb-compute/src/ranking_der.rs:611-613`
**Issue:** The same D-08 violation as WR-01. `lambdamart_ideal_ndcg` accumulates via raw
`score += ndcg_numerator(t) / ndcg_denominator(pos)` in a sequential loop. The comment at
line 604-606 again invokes "parity contract is the loop order" without a valid exception in
CLAUDE.md.

**Fix:**
```rust
fn lambdamart_ideal_ndcg(target_slice: &[f64], query_top_size: usize) -> f64 {
    let mut sorted: Vec<f64> = target_slice.to_vec();
    sorted.sort_by(|a, b| b.partial_cmp(a).unwrap_or(std::cmp::Ordering::Equal));
    let terms: Vec<f64> = sorted
        .iter()
        .enumerate()
        .take(query_top_size)
        .map(|(pos, &t)| ndcg_numerator(t) / ndcg_denominator(pos))
        .collect();
    cb_core::sum_f64(&terms)
}
```

### WR-03: `stochasticrank_oracle.cpp` centering uses `std::accumulate` vs Rust's `sum_f64`

**File:** `crates/cb-oracle/generator/stochasticrank_oracle.cpp:178`
**Issue:** The oracle computes the shift-centering average via `std::accumulate(shifted.begin(),
shifted.end(), 0.0)` (sequential left-to-right IEEE 754 sum), while the Rust implementation
(ranking_der.rs:723) uses `sum_f64(&shifted)` which may use a different accumulation order
(pairwise, compensated, or reordered). For the 3-doc `mu=0.0` fixture the sum happens to be
exactly 0.0, masking any divergence. For groups with more than ~8 docs and non-zero mu, or
with approxes that do not cancel exactly, the two orderings can diverge by one or more ULP in
the centering average, shifting every `shifted[d]` value and in turn shifting all per-doc noise
scores. Downstream fixture values become inconsistent at last-ULP precision, violating the ≤1e-5
oracle parity guarantee for non-trivial configurations.

**Fix:** Use sequential IEEE 754 left-to-right summation in the oracle to match whatever order
`sum_f64` uses in Rust — or change the oracle to call the same `sum_f64` semantics explicitly:

```cpp
// Replace std::accumulate with a left-to-right sequential accumulation that
// matches cb_core::sum_f64's documented accumulation order:
double avrg = 0.0;
for (double s : shifted) avrg += s;
avrg /= static_cast<double>(count);
```

If `sum_f64` uses pairwise summation internally, the oracle must be updated to match that order.
The precedent is that `sum_f64` order is the SOURCE OF TRUTH (D-08); the oracle must adapt to it,
not the reverse. An integration test running both the C++ oracle and the Rust implementation on a
non-trivial group (>4 docs, non-zero mu) would catch this divergence.
