## Summary

Reviewed the 06.3 gap-closure round (plans 10-14): the YetiRank per-tree
multi-fold RNG seed-plumbing subsystem (`yetirank.rs`, `boosting.rs`), the
StochasticRank Monte-Carlo der + index hardening (`ranking_der.rs`), the

--lib` fails to compile (`error: could not compile cb-train ... due to 8

2026-06-17** (see CR-01 below); `cargo clippy -p cb-train --lib --no-deps` now
exits 0 and the YetiRank oracles stay ≤1e-5.

`ranking_der.rs` is clean under clippy and carefully bounds-checked throughout.
The remaining findings are latent correctness risks around the dual persistent
RNGs and fixture-scoped draw-count assumptions in the YetiRank seeder, plus
minor quality items.

## Critical Issues

### CR-01: `yetirank.rs` production code violates workspace-denied `indexing_slicing` — `cargo clippy -p cb-train` fails to compile

**Status:** RESOLVED (2026-06-17). All 8 raw `[]` sites converted to bounds-checked
access matching the `ranking_der.rs`/`add_weight` discipline: the
`competitors_weights` matrix read/push loop now iterates by reference
(`.iter().enumerate()` + `competitors.get_mut(winner)`); the three
`gen_rand_ui64_vec(_, 1)[0]` seed reads use `.first().copied().unwrap_or(seed)`
guarded by a `debug_assert_eq!(len, 1)`; and the two `permutation[doc_id-/]`
reads use `.get().copied().unwrap_or(0)`. No `unwrap()/expect()/panic!`
introduced. `cargo clippy -p cb-train --lib --no-deps` exits 0 (0
indexing_slicing errors). `cargo test -p cb-train` fully green including the
YetiRank end-to-end per-stage oracle (≤1e-5), the YetiRank RNG draw-log oracle,
and the YetiRankPairwise variants — no parity regression.

**File:** `crates/cb-train/src/yetirank.rs:142-145,302,334,335,376,377`
**Issue:** The workspace pins `clippy::indexing_slicing = "deny"`
(`Cargo.toml:14`) and `cb-train` opts in (`crates/cb-train/Cargo.toml:7-8`).
The exemption is `#![cfg_attr(test, allow(...))]` — **test code only**. The
following production sites use raw indexing and are rejected:

- `yetirank.rs:142-143`: `competitors_weights[winner][loser]` (the `for winner in 0..query_size { for loser in 0..query_size {` matrix read)
- `yetirank.rs:145`: `competitors[winner].push(...)`
- `yetirank.rs:302`: `gen_rand_ui64_vec(deriv_base, 1)[0]`
- `yetirank.rs:334`: `gen_rand_ui64_vec(learnfold_base, 1)[0]`
- `yetirank.rs:335`: `gen_rand_ui64_vec(learnfold_rs0, 1)[0]`
- `yetirank.rs:376-377`: `permutation[doc_id - 1]` / `permutation[doc_id]`

Confirmed by running `cargo clippy -p cb-train --lib --no-deps`:
```
error: indexing may panic
   --> crates/cb-train/src/yetirank.rs:143:44
   ...
error: could not compile `cb-train` (lib) due to 8 previous errors
```
This is doubly egregious because the module header (lines 80-82, 99) and the
`add_weight`/`stable_sort_desc_by_key` helpers explicitly claim a bounds-checked
`.get()`/`get_mut()` discipline, but the new hot-loop code does not honor it.
Although every index is provably in-range for the current caller (sizes are
loop-bounded by `query_size`/hardcoded `1`), the lint is a hard build gate and
the crate ships broken.

**Fix:** Replace each raw index with the `.get()/.get_mut()/.copied().unwrap_or`
pattern already used elsewhere in the same file. Examples:
```rust
// 302/334/335 — size is hardcoded 1, but satisfy the lint:
let deriv_recalc_seed = gen_rand_ui64_vec(deriv_base, 1)
    .first()
    .copied()
    .unwrap_or(deriv_base);

// 142-145 — iterate by reference instead of indexing:
for (winner, row) in competitors_weights.iter().enumerate() {
    for (loser, &cw) in row.iter().enumerate() {
        let w_f32 = query_weight_f32 * cw / denom;
        if w_f32 != 0.0 {
            if let Some(out_row) = competitors.get_mut(winner) {
                out_row.push(Competitor { id: loser, weight: f64::from(w_f32) });
            }
        }
    }
}

// 376-377 — bounds-checked reads:
let first = permutation.get(doc_id - 1).copied().unwrap_or(0);
let second = permutation.get(doc_id).copied().unwrap_or(0);
```
After fixing, `cargo clippy -p cb-train --lib --no-deps` must exit 0.

## Warnings

### WR-01: Two independent persistent RNGs both model the single upstream `LearnProgress->Rand` — silently desyncs YetiRank + bootstrap/random_strength

**File:** `crates/cb-train/src/boosting.rs:1828-1833,2267,2303`
**Issue:** `train_inner` now constructs two distinct `TFastRng64` instances both
seeded from `params.random_seed`:
- `rng` (`boosting.rs:2267`) — the bootstrap / random-strength / per-level draw
  stream, advanced when `draws_active` (`bootstrap_type != No || random_strength
  != 0`).
- `YetiRankTreeSeeder.rng` (`boosting.rs:1828`, seeded `1833`) — which itself
  *also* reproduces the per-level Rsm/score/Box-Muller draws (`yetirank.rs:307-319`).

Upstream has exactly one `LearnProgress->Rand`. When YetiRank is combined with
`bootstrap_type != No` or `random_strength != 0`, both Rust RNGs consume from
independent copies of the same seed and the seeder additionally re-models the
same per-level draws the main `rng` consumes — so neither stream can match the
single interleaved upstream stream. The `YetiRankTreeSeeder` doc (`yetirank.rs:225`)
silently assumes `boosting_type=Plain, bootstrap=No`; nothing enforces it. Any
non-fixture YetiRank config will train to wrong (un-oracle-able) values with no
error. This passes today only because the YetiRank fixtures pin `bootstrap=No,
random_strength=0`.
**Fix:** Either (a) drive both the bootstrap draws and the YetiRank per-tree
seeds from a *single* shared `&mut rng` so the draw order is one stream, or
(b) reject the unsupported combination explicitly, e.g.
```rust
if is_yetirank
    && (!matches!(params.bootstrap_type, EBootstrapType::No)
        || params.random_strength != 0.0)
{
    return Err(CbError::Degenerate(
        "YetiRank seeding currently supports only bootstrap=No, random_strength=0".into(),
    ));
}
```
so a mis-configured run fails loudly instead of producing silently-wrong models.

### WR-02: YetiRank per-tree seed draw-count is hardcoded to float features — desyncs the RNG on CTR / one-hot corpora

**File:** `crates/cb-train/src/boosting.rs:1824-1827`; `crates/cb-train/src/yetirank.rs:307-319`
**Issue:** `yetirank_n_candidate_features` counts only non-empty
`feature_borders` (float features). The seeder's per-level loop
(`yetirank.rs:308-318`) draws exactly `n_features` Rsm `gen_rand_real1()` calls
and `n_features` Box-Muller normals per level — modelling only `OneFeature`
float candidate sublists. If a YetiRank fit ever has CTR or one-hot candidates
(which add their own candidate sublists and thus their own per-level draws), the
seeder under-counts the draws, the persistent RNG phase drifts, and every
subsequent per-group query seed is wrong from the first tree — failing the
≤1e-5 oracle with no diagnostic. The comment (`boosting.rs:1820-1823`) documents
the float-only corpus assumption but it is unenforced.
**Fix:** Compute the candidate-sublist count from the actual candidate set the
tree search will enumerate (float + CTR + one-hot), not just float borders; or
assert the YetiRank corpus is float-only and reject otherwise. At minimum, gate
with a debug assertion that the seeder's assumed `n_features` equals the real
candidate count used by the tree search on tree 0.

### WR-03: `gen_rand_ui64_vec(_, 1)[0]` becomes a panic-on-empty trap if the size argument is ever made dynamic

**File:** `crates/cb-train/src/yetirank.rs:302,334,335`
**Issue:** Beyond the lint violation in CR-01, these three sites take `[0]` of a
`Vec` whose size is currently the literal `1`. The pattern is fragile: any
future refactor that threads a dynamic `size` (e.g. a real multi-block trainer,
which the surrounding docs explicitly anticipate at `yetirank.rs:186-208`) would
turn this into an out-of-bounds panic on a zero-size vector. The seam was
introduced precisely to support multi-block, so the latent trap lives in the
change that motivates it.
**Fix:** Use `.first().copied().unwrap_or(<sentinel>)` (also resolves CR-01 for
these three sites), making the empty-vec case total rather than panicking.

### WR-04: Learning-fold approx update silently no-ops on an out-of-range span and lacks a standalone seed-derivation guard

**File:** `crates/cb-train/src/boosting.rs:2993-3010,3050-3055`; `crates/cb-train/src/yetirank.rs:329-336`
**Issue:** The new YetiRank learning-fold block re-samples competitors off
`learn_approx`, recomputes the grouped der, and adds a Newton leaf delta back
into `learn_approx`. Two robustness gaps: (1) `learn_approx.get(*begin..*end)
... .unwrap_or_default()` (line ~2999) silently yields an empty `raw_approx` if
the span is out of range, which `sample_pairs` turns into an empty competitor
set — the group is silently dropped from that tree's learning-fold update with no
error, diverging the *next* tree's gradient. (2) The block's correctness rests
entirely on the cited `train.cpp:420 + approx_calcer.cpp:1147` two-layer seed
derivation (`yetirank.rs:329-336`), where learnfold has one *more*
`GenRandUI64Vector` layer than deriv — an easy-to-break asymmetry verified only
against the single fixture corpus, with no standalone unit guard.
**Fix:** Return `CbError::Degenerate` (not `unwrap_or_default`) when a YetiRank
group span is out of range — a silently-empty learning-fold group is a
correctness bug, not a recoverable default. Add a focused unit test pinning
`derive_per_tree_query_seeds` (deriv) vs the learnfold two-layer chain against
the instrumented per-group first-Gumbel draws so a future edit to either layer
fails fast.

**Issue:** `derive_query_seeds` (168) re-implements `GenRandUI64Vector(1, seed)`
