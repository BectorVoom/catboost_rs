# Phase 06.3: Code Review Report

**Reviewed:** 2026-06-17
**Depth:** standard
**Files Reviewed:** 18
**Status:** issues_found

## Summary

Adversarial re-review of the Phase 06.3 ranking-loss / ranking-metric surface: the
grouped der seam (`ranking_der.rs`), per-loss derivative scalars (`loss.rs`), the
pairwise split-scoring + Cholesky leaf path (`pairwise_scoring.rs`,
`pairwise_leaves.rs`), the YetiRank RNG sampler + per-tree seeder (`yetirank.rs`),
the StochasticRank Monte-Carlo der (`ranking_der.rs::stochastic_rank_group_der`), the
nine eval-only ranking metrics (`ranking_metrics.rs`, `metrics.rs`), `QueryInfo`
construction, the PCG RNG port (`rng.rs`), the CubeCL backend dispatch
(`cpu_runtime.rs`), and the oracle generator scripts (`.cpp`/`.py`/`.sh`).

Code discipline is strong: no `unwrap`/`expect`/`panic`/`unreachable` in production
source; no embedded `#[cfg(test)] mod tests { … }` (every test file is an external
`#[path]` shim — the project-sanctioned separation; the `generator/*_test.rs` files
are discovered through `tests/*_test.rs` shims, not duplicated); parity SUMs route
through `cb_core::sum_f64` with documented per-cell scatter exceptions; bounds-guarded
`.get`/`.get_mut` on every trainer-trust-boundary index. The `Pair` struct
(`cb-data/src/pool.rs:22`) has NO weight field, so `query_info.rs:241` hardcoding
`weight: 1.0` is CORRECT, not a defect. The instrumented-trainer shell script uses
`set -euo pipefail`, an `ERR` trap, a disk-headroom gate, and an env-gated no-op sink —
no command-injection or unquoted-variable hazards surfaced.

No BLOCKER-class correctness or security defect was proven. The findings below are
parity-edge and maintainability concerns. The parity-edge warnings are latent — not
reachable by the pinned `top == -1`, single-thread, `mu = 0` corpus — but are real
defects in the general API surface that the oracle does not currently lock.

## Warnings

### WR-01: `top`-sentinel handling is inconsistent across ranking metrics; a non-`-1` negative `top` silently zeroes MRR/ERR

**File:** `crates/cb-train/src/ranking_metrics.rs:106, 222, 241, 275`
**Issue:** The "full group" sentinel is decoded three different ways:
- `clamp_top` (`:106`) and `pfound_group` (`:222`): `top < 0` → full group (any negative).
- `err_group` (`:241`): `top == -1` → full group; any other negative passes through `top.max(0)` → `0` depth (returns `0`).
- `mrr_group` (`:275`): `top == -1` → full group; any other negative makes `max_pos = (size).min(top)` NEGATIVE, so the `pos <= max_pos` loop body never runs and the function returns `0.0` even when a relevant doc exists.

For a caller passing any negative `top` other than `-1` (e.g. `-2`), MRR/ERR return a
degenerate `0`, while PrecisionAt/RecallAt/MAP/PFound/DCG treat it as the full group.
The corpus only uses `top == -1` and `top == 2`, so this is not fixture-reachable, but
it is a real divergence in the public metric API.
**Fix:** Decode the sentinel in one place. Gate on `top < 0` uniformly:
```rust
// mrr_group depth/max_pos and err_group depth:
let max_pos: i64 = if top < 0 { size as i64 } else { (size as i64).min(top) };
let depth = if top < 0 { size } else { size.min(top as usize) };
```

### WR-02: per-sample heap allocations inside the StochasticRank Monte-Carlo hot loop

**File:** `crates/cb-compute/src/ranking_der.rs:777-793`
**Issue:** `stochastic_rank_group_der` allocates six `Vec`s (`noise`, `scores`,
`cum_sum`, `cum_sum_up`, `cum_sum_low`, `order`) on EVERY Monte-Carlo sample, inside a
loop run `num_estimations` times, per group, per tree, and re-run a second time for the
leaf-value der recompute (`boosting.rs:2855`). Their lengths are sample-invariant
(`count` and `count + 1`). Memory efficiency is a first-class project constraint
(CLAUDE.md). With `num_estimations == 1` (the corpus default) the impact is bounded,
but the pattern scales with `num_estimations × groups × trees × 2`.
**Fix:** Hoist the buffers above the `for _sample` loop and refill/reset them per
iteration (`fill(0.0)` / re-iota) instead of reallocating.

### WR-03: the "top window" decode casts a negative `i64` to `usize` in metrics that lack the `top < 0` guard pattern

**File:** `crates/cb-train/src/ranking_metrics.rs:108-111, 222, 241`
**Issue:** The safe idiom (`pfound_group`, `clamp_top`) guards `top < 0` BEFORE the
`top as usize` cast. `err_group` (`:241`) instead casts `top.max(0) as usize` (safe by
clamping), and `mrr_group` works in `i64` (WR-01). The inconsistency means the cast
hazard is mitigated three different ways. Any future "top-k" metric that copies the
`else { size.min(top as usize) }` shape WITHOUT a preceding `top < 0` guard would cast
a negative `i64` to a near-`usize::MAX` value and silently request the full group. The
current call sites are individually safe; the divergent patterns are fragile.
**Fix:** Route every top-window decode through a single `clamp_top(top, size)` so the
negative-cast guard lives in exactly one place (folds WR-01 + WR-03 into one helper).

### WR-04: LambdaMart plain-`DCG` der divides by the ideal DCG but no committed fixture locks it

**File:** `crates/cb-compute/src/ranking_der.rs:526-552, 616-627`
**Issue:** In `lambdamart_group_der`, the `LambdaMartMetric::Dcg | Ndcg` arm
unconditionally computes `ideal_score = lambdamart_ideal_ndcg(...)` and divides
`delta = dcg_num * dcg_den / ideal_score` for BOTH metrics. The corpus pins NDCG
(the LambdaMart default), so the plain-`Dcg` path is asserted ("DCG/NDCG share the
same `CalcDersNDCG` arm") but NOT oracle-locked by a committed fixture. If upstream's
plain-DCG lambda arm does not divide by IDCG, the plain-DCG gradient is scaled wrong
and would ship undetected.
**Fix:** Either add a `LambdaMart:metric=DCG` corpus fixture to lock the plain-DCG der
≤1e-5, or add a code comment citing the exact `error_functions.cpp` line proving the
`Dcg` arm also divides by `CalcIdealMetric`.

### WR-05: MRR uses a bespoke unsorted tie-break that diverges from the module's shared `compare_docs`

**File:** `crates/cb-train/src/ranking_metrics.rs:277-285`
**Issue:** `mrr_group` counts the first-relevant rank by walking objects in ORIGINAL
index order and incrementing `pos` when `a > max_relevant_approx || (a ==
max_relevant_approx && t <= border)`. Every other metric in the module sorts via the
shared `sorted_indices`/`compare_docs` tie-break (approx desc, target asc, stable
index). MRR's hand-rolled rule is a DIFFERENT tie-break and scans unsorted. It may be
a faithful transcription of `metric.cpp:6035-6060`, but on approx TIES at the relevant
boundary (which the corpus's random approx vector can produce) the bespoke rule can
diverge from the shared comparator, and a future maintainer is likely to "correct" it
to `compare_docs` and break parity.
**Fix:** Add a unit assertion that `mrr_group` agrees with a `sorted_indices`-based
reference on an input with deliberate approx ties at the relevant boundary, and/or pin
the exact upstream tie-rule line in a code comment so the bespoke counting is not
mistaken for a bug.

## Info

### IN-01: StochasticRank der recomputes the per-group mean over the already-centered slice

**File:** `crates/cb-compute/src/ranking_der.rs:745, 889`
**Issue:** `approx_mean = sum_f64(approx_slice) / count` (`:745`) centers
`approx_slice`; then `avrg_approx = sum_f64(approx_slice) / count` (`:889`) is
recomputed over the now-centered slice (structurally ~0). The second `O(count)`
`sum_f64` plus the `zero_mean` re-centering are near-no-ops.
**Fix:** Use the centered `approx_slice` directly as the zero-mean vector and drop the
second reduction, or add a comment explaining the residual-f64-mean recompute is kept
for byte-parity.

### IN-02: `derive_query_seeds` inlines the `GenRandUI64Vector(1, ·)` body instead of calling the existing helper

**File:** `crates/cb-train/src/yetirank.rs:176-180, 186-189`
**Issue:** `derive_query_seeds` (`:176`) inlines `TFastRng64::from_seed(random_seed)
.gen_rand()` for the block seed, duplicating the `GenRandUI64Vector(1, seed)`
semantics that `gen_rand_ui64_vec` (`:186`) already encodes. Cosmetic duplication of
the seed-derivation contract.
**Fix:** Route the block-seed draw through `gen_rand_ui64_vec(random_seed, 1)` so the
contract lives in one place (behavior already identical).

### IN-03: CubeCL launch helpers duplicate ~25 lines of device/handle/read-back boilerplate per kernel arm

**File:** `crates/cb-backend/src/cpu_runtime.rs:44-136, 156-225, 230-446`
**Issue:** Each launcher repeats the device/client/handle/`CubeCount`/`CubeDim`
construction and the identical `read_one` + `bytemuck::cast_slice` epilogue verbatim
(the WR-05 read-back comment is copy-pasted five times). Pre-existing (not new to
06.3) and out of v1 performance scope, but raises maintenance cost and the risk of an
inconsistent future fix.
**Fix:** Extract `read_back_f64(client, out_handle)` and `launch_geometry(n)` helpers.

### IN-04: the standalone StochasticRank oracle harness uses `mu = 0.5` while the trained corpus uses `mu = 0.0`

**File:** `crates/cb-oracle/generator/stochasticrank_oracle.cpp:258-263`; `crates/cb-oracle/generator/gen_ranking_fixtures.py:168`
**Issue:** The standalone self-oracle pins `mu = 0.5` (non-zero, to exercise the
centering path) and logs only the noise STREAM, while the Python-trained corpus pins
the StochasticRank default `mu = 0.0` with `metric=DCG`. The non-zero-`mu` der path is
gated ONLY by `stochasticrank_centering_test.rs`, not by an end-to-end der oracle —
the end-to-end ≤1e-5 gate covers `mu = 0` only. Intentional and documented, but a
reader could mistake `mu = 0.5` for the corpus value.
**Fix:** Add a header note cross-referencing that `mu != 0` is unit-gated only and the
end-to-end der oracle covers `mu = 0` (documentation clarity; no code change).
