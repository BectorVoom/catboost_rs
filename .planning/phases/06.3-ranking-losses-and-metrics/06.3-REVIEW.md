# Phase 06.3: Code Review Report

**Reviewed:** 2026-06-17
**Depth:** standard
**Files Reviewed:** 16
**Status:** issues_found

## Summary

Reviewed the LOSS-04 ranking-loss closure: the new pairwise split-scorer
(`cb-compute/src/pairwise_scoring.rs`), its tree-search wiring
(`cb-train/src/tree.rs`), the StochasticRank querywise der + per-query centering
(`cb-compute/src/ranking_der.rs`), the YetiRank/StochasticRank per-tree RNG
seeders (`cb-train/src/yetirank.rs`, `boosting.rs`), the RNG primitive
(`cb-core/src/rng.rs`), the oracle tests, and the offline fixture generators.

The code is unusually well-documented, bounds-checked throughout, and free of
prohibited constructs: **no `unwrap`/`expect`/`panic!`/`unreachable!` and no raw
panicking indexing in any production source file** (grep-verified). Every FFI/
trust-boundary index goes through `.get`/`.get_mut`. The oracle tests carry **no
`#[ignore]` and no weakened tolerances** — the ≤1e-5 / integer-exact bars hold.
The self-oracle in `pairwise_scoring_test.rs` independently re-derives the score
with a *different* (Gaussian-elimination) solver, which is good adversarial
design.

No BLOCKER/Critical defects were proven. The findings below are correctness-
adjacent robustness gaps (silent-fallback masking, a couple of summation-
discipline deviations from the project's `sum_f64` rule, and seed-derivation
fragility) plus quality items. The two most material are **WR-01** (silent
`unwrap_or(0)` seed fallbacks that mask a per-group desync) and **WR-02** (raw
`+=` reductions that bypass the mandatory `cb_core::sum_f64` ordered fold).

## Warnings

### WR-01: Silent `unwrap_or(0)` query-seed fallback masks a per-group RNG desync

**File:** `crates/cb-train/src/boosting.rs:2392`, `:2802`, `:3169`
**Issue:** In all three per-tree competitor re-sample sites the per-group inner
seed is fetched with `seeds.deriv.get(gi).copied().unwrap_or(0)` (and likewise
`seeds.leafval` / `seeds.learnfold`). If the seeds vector is ever shorter than
the group list (a length desync between `YetiRankTreeSeeder` `group_count` and
the live `group_spans`), the affected group silently samples its competitors
from seed `0` instead of erroring. Seed `0` is a *valid-looking* but wrong seed,
so the gradient/leaf values would diverge with no signal — exactly the failure
mode the WR-04 fix at `:2382` was introduced to prevent for spans. The seeder is
constructed with `yetirank_groups.len()` / `group_spans` length, so the indices
*should* always be in range, but this is a trust-coupling between two separately
maintained lengths.
**Fix:** Surface a typed error instead of substituting seed 0, mirroring the
`ok_or_else` span guard already adopted two lines above:
```rust
let query_seed = *seeds.deriv.get(gi).ok_or_else(|| {
    CbError::Degenerate(format!(
        "YetiRank deriv re-sample: no per-group seed for group {gi} \
         (seeder group_count {} != live group count)", seeds.deriv.len()
    ))
})?;
```

### WR-02: Raw `+=` reductions bypass the mandatory `cb_core::sum_f64` ordered fold

**File:** `crates/cb-compute/src/ranking_der.rs:674-678` (`compute_dcg_pos_weights` ideal_dcg); `:521`, `:571`, `:523` (`lambdamart_group_der`)
**Issue:** Project convention (CLAUDE.md D-08) requires every floating-point
reduction in a numeric kernel to route through `cb_core::sum_f64` for
deterministic left-to-right order. `compute_dcg_pos_weights` accumulates
`ideal_dcg += ndcg_numerator(t) * pos_weights[pos]` with a raw `+=` over the
sorted targets — a cross-position reduction that is not a documented "fixed-order
scatter." (The `lambdamart_group_der` `sum_der1 -= 2.0 * antigrad` accumulation
*is* documented as a parity-order scatter, so it is borderline-acceptable; the
`ideal_dcg` reduction has no such justification in its doc-comment.) The values
will agree with upstream *only if* upstream also accumulates in this exact order;
the comment claims it mirrors `CalcDCG` but does not pin the order as the parity
contract the way the sanctioned scatter sites do.
**Fix:** Collect the per-position terms and reduce through `sum_f64`, matching the
sibling `lambdamart_ideal_ndcg` (`:616-627`) which already does this correctly:
```rust
let terms: Vec<f64> = sorted.iter().enumerate().take(query_top_size)
    .map(|(pos, &t)| ndcg_numerator(t) * pos_weights.get(pos).copied().unwrap_or(0.0))
    .collect();
let ideal_dcg = sum_f64(&terms);
```
If the raw order is genuinely load-bearing for parity, add the same explicit
"summation order IS the parity contract" doc-comment the scatter sites carry so
the deviation is auditable rather than incidental.

### WR-03: Infallible `uniform()` silently returns 0 for `bound == 0`

**File:** `crates/cb-core/src/rng.rs:247-250`
**Issue:** `uniform(bound)` calls `try_uniform(bound).unwrap_or(0)`. For
`bound == 0` (the upstream `Y_ABORT_UNLESS(max > 0)` precondition violation) it
returns `0` — a *valid index value* — rather than signalling the programming
error. A caller that computes `bound` from a possibly-empty collection
(`uniform(len)` with `len == 0`) gets index 0 back and may then index an empty
slice elsewhere. The doc-comment acknowledges this, but a `0`-returning degenerate
path is a latent correctness trap distinct from a panic.
**Fix:** Prefer steering all callers to `try_uniform` and removing the infallible
variant, or `debug_assert!(bound != 0, ...)` inside `uniform` so the degenerate
call is caught in test/debug builds without panicking in release. At minimum,
audit every `uniform()` call site to confirm `bound > 0` is structurally
guaranteed.

### WR-04: `add_at`/`add_to` silently drop out-of-range scatter writes

**File:** `crates/cb-compute/src/pairwise_scoring.rs:433-437` (`add_at`), `:584-588` (`add_to`)
**Issue:** Unlike `decrement_smaller`/`decrement_greater` (which return
`CbError::OutOfRange` on a bad cell), `add_at` and `add_to` silently no-op when
`r`/`c`/`idx` are out of range (`if let Some(cell) = ... { *cell += delta; }`).
The doc-comment argues every index is provably in `0..2*leaf_count`, which is true
for the current callers — but it means a future indexing bug in the
weight-matrix assembly would *silently lose* a weight contribution and produce a
subtly-wrong score rather than failing loudly. The two scatter helpers in the
same file take the opposite (fail-loud) stance, so the file is internally
inconsistent about how to treat "provably-in-range" trainer indices.
**Fix:** Either make these `CbResult` and propagate (consistent with the
`decrement_*` helpers), or add a `debug_assert!` on the bound so a regression is
caught in tests:
```rust
fn add_at(weight_sum: &mut [Vec<f64>], r: usize, c: usize, delta: f64) {
    debug_assert!(weight_sum.get(r).is_some_and(|row| c < row.len()),
        "add_at index ({r},{c}) out of range");
    if let Some(cell) = weight_sum.get_mut(r).and_then(|row| row.get_mut(c)) { *cell += delta; }
}
```

### WR-05: `score_pairwise_feature` substitutes bucket 0 for a missing feature value

**File:** `crates/cb-train/src/tree.rs` (`score_pairwise_feature`: `values.get(obj).copied().unwrap_or(0.0_f32)`)
**Issue:** When building `bucket_of`, a missing per-object feature value falls back
to `0.0_f32`, which buckets the object at bucket 0 (passes no split). If the
feature column is ever short relative to `n_objects` (a SoA length mismatch), the
affected objects are silently mis-bucketed into bucket 0 rather than the trainer
erroring — silently corrupting the pairwise der-sums/statistics and thus the split
score. For the current callers the columns are full-length, so this is latent.
**Fix:** Validate `values.len() == n_objects` once at function entry and return
`CbError::OutOfRange` on mismatch, rather than per-object silent substitution.

### WR-06: `derive_query_seeds` vs `derive_per_tree_query_seeds` are non-equivalent look-alikes

**File:** `crates/cb-train/src/yetirank.rs:160-221`
**Issue:** Two `pub` seed-derivation functions exist that are NOT equivalent for
`group_count == 1` (`derive_per_tree_query_seeds` adds an extra
`GenRandUI64Vector` block layer, as its own doc-comment warns).
`derive_query_seeds` is the standalone self-oracle chain;
`derive_per_tree_query_seeds` is the live trainer path. They have near-identical
signatures `(u64, usize) -> Vec<u64>`, so nothing prevents a future caller from
binding the wrong one — and a mis-pick would silently desync the per-query Gumbel
stream (a ≤1e-5 oracle failure that is expensive to localize, per the module's own
Pitfall 1 warning).
**Fix:** Make the distinction unmissable: rename to encode call context
(`derive_query_seeds_single_block` vs `derive_query_seeds_per_tree`), or gate the
standalone variant behind `pub(crate)`/`#[cfg(test)]` if only the self-oracle
consumes it, so the trainer cannot accidentally bind it.

## Info

### IN-01: `try_uniform` rejection loop has no documented termination bound

**File:** `crates/cb-core/src/rng.rs:228-240`
**Issue:** `try_uniform` loops until `rand < randmax`. The rejection region is at
most `(bound-1)/2^64`, so termination is overwhelmingly likely, but the loop has
no documented worst-case bound. This matches upstream `GenUniform` and is not a
real hang risk; noted for completeness.
**Fix:** Add a one-line comment noting the rejection region is `< bound/2^64` so
the unbounded loop is justified.

### IN-02: Magic constant `1.000_001_f32` Gumbel guard is undocumented at the literal

**File:** `crates/cb-train/src/yetirank.rs:119`
**Issue:** `1.000_001_f32 - u` is the upstream div-by-zero guard, explained in the
surrounding comment but inlined as a bare literal — if the value ever changes it
lives in two places (comment + literal).
**Fix:** Hoist to a named `const GUMBEL_DENOM_GUARD: f32 = 1.000_001;` alongside
`MAGIC_CONST`.

### IN-03: Standalone StochasticRank RNG oracle uses a different (bare) seed than the trainer

**File:** `crates/cb-train/tests/stochasticrank_oracle_test.rs:262`
**Issue:** `stochasticrank_rng_draw_log_oracle` hardcodes `group_seed = 5`
(generator unit `random_seed 5 + group 0`), while the end-to-end test uses
`RANDOM_SEED = 20_260_617`. The two are intentionally decoupled (one gates the
sampler, the other the plumbing), but the bare `5` with only a comment is easy to
misread as a copy-paste bug.
**Fix:** Name the constant (`const STANDALONE_GENERATOR_SEED: u64 = 5;`) and
cross-reference the generator that froze it.

### IN-04: `BucketPairWeightStatistics::merge` is dead for the single-thread path

**File:** `crates/cb-compute/src/pairwise_scoring.rs:70-85`
**Issue:** `merge` is provided "to keep the struct a faithful analogue" but the
doc-comment states the single-thread library never splits ranges, so it is never
called in production — only carried as API surface (a documented
speculative-generality smell).
**Fix:** Either drop it until the multi-thread range split lands, or add a unit
test so it is at least covered.

### IN-05: `gen_metric` trains a throwaway PairLogit model whose output is unused

**File:** `crates/cb-oracle/generator/gen_ranking_fixtures.py:234-262`
**Issue:** `gen_metric` fits a PairLogit model and evaluates the metric over the
same pool, but `gen_metrics_eval` (the LOSS-05 path actually consumed by the Rust
oracle) computes the metric over a FIXED approx via `eval_metric` with NO trained
model. `gen_metric` appears to be a superseded earlier path — likely dead
generator code (offline-only, low risk, but confusing for the next maintainer
regenerating fixtures).
**Fix:** Remove `gen_metric` / the `--metric` flag if `gen_metrics_eval` /
`--metrics-eval` is canonical, or comment-mark `gen_metric` deprecated.

## Notes on items explicitly checked and found clean

- **No prohibited constructs:** grep-verified zero `unwrap()`/`expect()`/`panic!`/
  `unreachable!`/`todo!`/`unimplemented!` and no raw panicking indexing in the four
  new/changed production source files.
- **Source/test separation:** the `#[cfg(test)] #[path = "..._test.rs"] mod`
  footers in `lib.rs` / `pairwise_scoring.rs` / `tree.rs` link EXTERNAL sibling
  test files — the permitted structure, NOT an embedded `mod tests` body.
- **Oracle parity discipline:** no `#[ignore]`, no weakened tolerances; the
  `l2_leaf_reg: 1e-20` and f32-rounded `learning_rate` values are fixture-matching
  pins, not weakenings. The per-tree RNG oracle asserts integer-exact call counts.
- **Bucket/split consistency:** `float_bucket_of` (count of `value > border`) is
  consistent with `FeatureMatrix::passes_float` (`value > border`) — a doc passes
  the split at border `j` iff `bucket > j`; NaN buckets to 0 and passes nothing in
  both, so they agree.
- **Divide-by-zero guards:** StochasticRank `approx_mean` / `mean` / SFA
  projection all short-circuit on `count <= 1` (so `count >= 2`); `query_avrg`,
  softmax share, YetiRank `permutation_count`, and `ideal_dcg`/`approxes_norm_sqr`
  all guard their denominators.
- **RNG `call_count`:** `from_seed` uses a separate derivation engine that does not
  pollute the constructed rng's counter; `advance(delta)` increments by `delta`
  consistent with `gen_rand` draws.
