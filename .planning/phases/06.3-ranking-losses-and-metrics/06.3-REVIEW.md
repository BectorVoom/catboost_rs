## Summary

The gap-closure delta is disciplined: no `unwrap`/`expect`/`panic` in production
source, bounds-checked `.get()` on the grouped input, and source/test separation is
respected (every test rides a `#[path]` shim or a `tests/` integration file). The
core fixes read correctly and are backed by targeted regression tests that
distinguish the pre-fix from the post-fix value:

- **CR-01** (`calc_dcg_metric_diff` reads the normalized `pos_weights`) is correct
  and well-tested (`calc_dcg_metric_diff_reads_normalized_pos_weights_ndcg`).
- **CR-02** (grouped der already weight-folds; `weighted_der1` must not re-multiply)
  is correct, with a non-uniform-weight regression test that fails against the old code.
- **WR-01** (`make_zero_average` via `sum_f64`) is a faithful order-preserving refactor.

The serious problems are in **verification coverage**, not the delta math: the two
oracle tests that prove the headline PairLogit pair-weight wiring actually reaches
≤1e-5 parity are both `#[ignore]`d, and the StochasticRank RNG ground-truth fixture
is serialized at ~6 decimal digits, too coarse to back the ≤1e-5 "f64-exact" gate its
own header claims. There are also latent weight-handling and indexing concerns in
paths the (uniform-weight) fixtures do not reach.

## Critical Issues

### CR-01: StochasticRank oracle serializes f64 ground truth at ~6 decimals, defeating the ≤1e-5 gate it claims

**File:** `crates/cb-oracle/generator/stochasticrank_oracle.cpp:197-202`
**Issue:** The instrumented generator writes the `noise` and `score` ground-truth
stream into the committed JSONL fixture via `std::to_string(double)`:

```cpp
",\"noise\":" + std::to_string(noise[d]) + "}");
...
",\"score\":" + std::to_string(scores[d]) + "}");
```

`std::to_string(double)` formats as if by `%f` with the default precision of 6 — it
emits exactly 6 digits **after the decimal point** and silently truncates the rest.
The file header (line 28-29) states this fixture "freezes the noise stream as the RNG
ground truth the Rust oracle … gates integer-/f64-exact," and the StochasticRank
parity bar is ≤1e-5 (CLAUDE.md). For any `score` whose magnitude exceeds ~1 (e.g.
`shifted[d] + sigma·noise[d]` with graded relevances) 6 digits past the decimal leaves
under 1e-5 of usable precision, and at magnitude ≥10 the 1e-5 budget is already gone.
The self-oracle deliberately uses `%.17g` for stderr (line 229), making the gap stark:
the *fixture* the Rust test compares against is the low-precision one. A fixture that
cannot represent the ground truth to better than ~1e-6 absolute (worse at scale)
cannot certify a ≤1e-5 parity claim.
**Fix:** Serialize with full round-trip precision:

```cpp
char buf[32];
std::snprintf(buf, sizeof(buf), "%.17g", noise[d]);
LogLine(... + ",\"noise\":" + buf + "}");
```

Apply the same to the `score` line, then regenerate the committed fixture
(RUN-ONCE/COMMIT) so the JSONL carries 17 significant digits.

## Warnings

### WR-01: Headline PairLogit / PairLogitPairwise oracle tests are `#[ignore]`d — the gap-closure deliverable is unverified

**File:** `crates/cb-train/tests/pairlogit_oracle_test.rs:128-130`,
`crates/cb-train/tests/pairlogit_pairwise_oracle_test.rs:128-130`
**Issue:** Both end-to-end oracle tests that would prove the 06.3-09 pair-weight
wiring (`calc_pairwise_weights`, `eff_weights`, the Cholesky pairwise-leaf path)
reaches ≤1e-5 parity are annotated `#[ignore]`. The rationale is honest (the
assembled per-leaf der2 still diverges ~6× per the comment at lines 115-127, pending
an instrumented trainer oracle), but the practical effect is that the central
deliverable of the phase ships with **no executing parity test**. The supporting unit
tests verify the building blocks in isolation; the integrated path is known-divergent
and untested.
**Fix:** Either (a) add a non-ignored partial-parity test that gates the stages that
*do* pass today (the comment notes Splits now advance to index 6) so regressions in
the wired portion are caught, or (b) record explicitly in phase verification that
PairLogit parity is unverified and is NOT a closed gap.

### WR-02: `calc_dcg_metric_diff` indexes `cum_sum`/`cum_sum_up`/`cum_sum_low` with raw `[]` despite the project unchecked-index ban

**File:** `crates/cb-compute/src/ranking_der.rs:903-909`
**Issue:** The CR-01 fix correctly switched the position-weight reads to
bounds-checked `pos_weights.get(...).unwrap_or(0.0)` (lines 899-900), citing "CLAUDE.md
unchecked-index ban; T-06.3-06-01." But the next lines still index the prefix-sum
arrays with raw subscripts:

```rust
let old_mid = cum_sum[old_pos] - cum_sum[new_pos];
let new_mid = cum_sum_low[old_pos] - cum_sum_low[new_pos];
...
let old_mid = cum_sum[new_pos + 1] - cum_sum[old_pos + 1];
let new_mid = cum_sum_up[new_pos + 1] - cum_sum_up[old_pos + 1];
```

These are in-range for the current `0..count` caller (the `cum_sum*` are length
`count+1`), but this is the exact "trust the caller" indexing pattern the same diff
removed two lines above. A future `top`/`query_top_size` change that desyncs the index
range would panic instead of returning a typed error — inconsistent with the fix's own
stated discipline.
**Fix:** Use `.get(..).copied().unwrap_or(0.0)` for the `cum_sum*` reads too, matching
the `pos_weights` treatment in the same function.

### WR-03: YetiRank Gradient leaf passes unit weights, dropping the pairwise `sumWeight` upstream uses

**File:** `crates/cb-train/src/boosting.rs:2658-2667`
**Issue:** On the grouped path the leaf-delta call forces `leaf_weights_for_deltas =
unit_weights` whenever `group_spans.is_some()`. The comment justifies this for the
**Newton** leaf (der2 already folds the pair weight, so re-weighting would
double-count — correct). But `YetiRank` (non-`Pairwise`) rides the **Gradient** leaf,
where the delta is `sumDer / (sumWeight + scaledL2)` and upstream
`UsesPairsForCalculation` makes that `sumWeight` the **pairwise** weight
(`bt.PairwiseWeights`), not the document count. With unit weights the Gradient
denominator becomes `docCount + scaled_l2`, while `scaled_l2` was itself computed from
`Σ eff_weights` (the pairwise total at line 2452-2457) — so the denominator mixes a
document-count `sumWeight` with a pairwise-weight-scaled L2, internally inconsistent
and divergent from upstream for YetiRank Gradient leaves. Latent (YetiRank E2E is
deferred), but a concrete correctness gap in the wired path.
**Fix:** Branch the leaf `sumWeight` on the leaf method (or on
`uses_pairwise_weights(loss)`), not uniformly on `group_spans.is_some()`: pass
`eff_weights` for the Gradient leaf, unit weights only for Newton.

### WR-04: `assert_single_weight_invariant` distinguish-guard threshold is mismatched to its assertion

**File:** `crates/cb-train/tests/grouped_weight_regression_test.rs:209-219`
**Issue:** The "squared path is detectably wrong" check fires whenever
`(w - 1.0).abs() > f64::EPSILON && p.abs() > EXACT_TOL` (EXACT_TOL = 1e-12), then
asserts `(p*w - p).abs() > 1e-6`, i.e. `|p|·|w-1| > 1e-6`. The entry guard only
requires `|p| > 1e-12`, which does not imply `|p|·|w-1| > 1e-6`. For a der `p` between
~1e-12 and ~1e-6 with `w` near 1, the inner assertion could fail though the test logic
is sound — a latent false-negative. The current corpus produces `p ~ O(0.1)` so it
passes today, but the guard threshold is mismatched to the assertion threshold.
**Fix:** Gate on the same scale as the assertion:
```rust
if (w - 1.0).abs() > f64::EPSILON && (p * (w - 1.0)).abs() > 1e-6 { ... }
```

### WR-05: StochasticRank instrumented fixture is generated with `mu = 0`, the exact masking case WR-03 documents

**File:** `crates/cb-oracle/generator/stochasticrank_oracle.cpp:242`
**Issue:** `main()` pins `const double mu = 0.0;` for the single instrumented unit.
The WR-03 centering test header (`stochasticrank_centering_test.rs:13-15`) states the
"3-doc mu=0 fixture has an exactly-zero shifted sum, so it masks any ordering
divergence." The C++ generator that produces the committed ground truth therefore uses
precisely the degenerate input that cannot exercise the centering / accumulation-order
behavior WR-03 is about. The Rust test compensates with an in-Rust mu≠0 replica, but
the *generated fixture itself* never stresses the centering path.
**Fix:** Add (or switch to) a second instrumented unit with `mu != 0` and >4 docs so
the committed fixture's centering subtraction is non-trivial and the generated stream
is gated against the Rust reproduction on a non-masking input.

## Info

### IN-01: Model `leaf_weights` now stores pairwise weight sums for pairwise losses — confirm against upstream model.json

**File:** `crates/cb-train/src/boosting.rs:2686`
**Issue:** `accumulate_leaf_weights(&leaf_value_leaf_of, &eff_weights, n_leaves)` now
folds the **pairwise** weights into the model's `leaf_weights` for
`uses_pairwise_weights` losses. For unweighted pointwise models this field is
documented (RESEARCH Pitfall 1 / A4) as the per-leaf document count. Upstream does
store the fold's effective (pairwise) weights for these losses, so this is likely
correct — but it changes the field semantic for ranking models, feeds SHAP /
PredictionValuesChange, and is not covered by an executing test (the PairLogit oracle
is ignored).
**Fix:** Document the pairwise-loss semantic on `ObliviousTree::leaf_weights` and add a
`leaf_weights` assertion to the PairLogit oracle when it is un-ignored.

### IN-02: WR-03 generator hand-transcribes the `sum_f64` fold; duplication is intentional but untracked

**File:** `crates/cb-oracle/generator/stochasticrank_oracle.cpp:184-187`
**Issue:** The centering average uses an inline `for (double s : shifted) avrg += s;`
loop, justified as "std::accumulate could be reordered." That is a sound C++ judgment
for a dependency-free oracle (it cannot link `cb_core::sum_f64`), but the parity
contract now lives as a duplicated loop in three places (oracle, Rust centering, Rust
test). Drift would be silent.
**Fix:** None required; keep the loop and rely on the existing WR-03 test as the
binding cross-check.

### IN-03: `lambda_query_top_size` fallback branch is effectively dead

**File:** `crates/cb-compute/src/ranking_der.rs:594-601`
**Issue:** `usize::try_from(top).unwrap_or(doc_count).min(doc_count)` — `top` is
guaranteed `> 0` (or `-1`, handled by the first branch) by `Loss::validate`, and the
`> doc_count` case is also handled by the first branch, so the
`try_from(...).unwrap_or(doc_count)` fallback is unreachable. Harmless, just slightly
misleading as "clamp defensively."
**Fix:** Optional/cosmetic: drop the redundant `.unwrap_or(doc_count)`.
