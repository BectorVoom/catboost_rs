# Phase 06.3 Plan 09: PairLogit Pair-Weight Normalization + Oracles Summary

PairLogit / PairLogitPairwise split-scoring and Newton-L2 scaling now ride the upstream
per-object PAIRWISE weights (`bt.PairwiseWeights`) instead of the uniform per-object sample
weight, mirroring `UsesPairsForCalculation` / `scoring.cpp:275-279` / `CalcDeltaNewtonBody`;
the two per-stage oracle tests are authored and committed, gated behind the still-deferred
PairLogit leaf-der2 parity (06.3-03) that requires the instrumented catboost trainer.

## What was built

### Task 1 — `bt.PairwiseWeights` wiring (commit `de78521`)

- `uses_pairwise_weights(loss)` selects exactly the `UsesPairsForCalculation` losses
  (`enum_helpers.cpp:502` = `IsYetiRankLossFunction || IsPairLogit`): PairLogit,
  PairLogitPairwise, YetiRank, YetiRankPairwise.
- `calc_pairwise_weights(groups, n)` transcribes upstream `CalcPairwiseWeights`
  (`approx_updater_helpers.h:74-89`): for each winner→loser competitor edge, add
  `competitor.weight` to BOTH the winner's and the loser's per-object slot →
  `pw[obj] = Σ competitor.weight` over incident pairs.
- In the per-iteration boosting loop, `eff_weights = calc_pairwise_weights(...)` for
  pairwise losses (else the per-object `weights`, byte-identical). `eff_weights` feeds:
  - `score_weights` (the split-scoring histogram `sumWeight`, `scoring.cpp:276-279`
    `hasPairwiseWeights ? bt.PairwiseWeights : fold.LearnWeights`);
  - `scaled_l2 = scale_l2_reg(l2, Σ eff_weights, n)` (the Newton denominator's
    `l2 * sumAllWeights/docCount`, `online_predictor.h:126`).
  - The Newton leaf der2 keeps UNIT weights for the grouped der (the der already folds
    `competitor.weight`; re-weighting would double-count — the der2 analogue of CR-02).
- RED-first unit tests in `boosting_test.rs`: predicate selectivity, pairwise-sum over
  both endpoints (non-uniform vs the old `1.0`), weighted/empty-group handling.

### Task 2 — PairLogit + PairLogitPairwise per-stage oracles (commit `ae6eb67`)

- `pairlogit_oracle_test.rs` / `pairlogit_pairwise_oracle_test.rs` mirror the
  queryrmse/lambdamart oracle template, gating `compare_stage(Splits|LeafValues|
  StagedApprox|Predictions)` ≤1e-5 against the frozen `ranking_corpus/PairLogit{,Pairwise}`
  fixtures. PairLogit pins Newton leaf + l2=3; PairLogitPairwise pins Gradient/Cholesky
  pairwise leaf + l2=5; both boosting_type Plain, Cosine score, random_seed 20260617.
- Both `#[ignore]`'d with an explicit deferral (see Deviations); NO tolerance weakened.

### Oracle parity numbers (why the oracles stay deferred)

PairLogit oracle progress: the `bt.PairwiseWeights` wiring advanced the Splits match from
**index 4 → index 6** (more split borders now match). The residual divergence is the
LEAF-der2 reduction. At iter-0 tree-0 the `TPairLogitError` der is bit-verified-identical
to upstream (leaf-0 `sumDer = 2`), yet the Newton leaf deltas diverge:

| leaf | sumDer | sumDer2 (ours) | required denom (expected) | our denom |
|------|--------|----------------|---------------------------|-----------|
| 0    | 2.0    | -2.0           | 7.47                      | 5.5       |
| 1    | -2.5   | -1.25 / -2.25  | 3.47                      | ~4.8      |
| 3 (obj 10, single cross-leaf pair) | 0.5 | -0.25 | 23.3 | 3.75 |

The per-leaf required denominators are mutually inconsistent with any `-sumDer2 + C`
(leaf 3 needs ~23 from a single pair at p=0.5 — impossible from a plain per-object der2
sum), so the per-leaf SumDer2 (or a coupling the pointwise path omits) genuinely differs
upstream. Pinning ≤1e-5 needs the instrumented trainer's per-leaf SumDer/SumDer2 log.

## Deviations from Plan

### Auto-fixed / re-diagnosed

**1. [Rule 1 - Bug] Root cause re-diagnosed: pairwise-weight WIRING, not Competitor.weight**
- **Found during:** Task 1 (empirical fixture inspection + leaf-sum instrumentation).
- **Issue:** the plan (following deferred-items' hypothesis) located the fix in
  `build_query_info`'s `Competitor.weight`. But `pairs.npy` is `(7,2)` winner/loser only
  (no weight column → all pair weights `1.0`), and upstream sets `Competitor.Weight =
  pair.Weight` verbatim (`data_providers.cpp:327-329`). The actual gap is that PairLogit's
  split-scoring `sumWeight` + L2 must read `bt.PairwiseWeights`, which cb-train fed as the
  uniform per-object `1.0`.
- **Fix:** added `uses_pairwise_weights` + `calc_pairwise_weights` and wired the pairwise
  weight vector into scoring + `scaled_l2` (boosting.rs). `Competitor.weight` /
  `build_query_info` are UNCHANGED (correct as-is).
- **Files:** `crates/cb-train/src/boosting.rs`, `crates/cb-train/src/boosting_test.rs`.
- **Commit:** `de78521`.

### Escalated (Rule 4 — deferred, not weakened)

**2. [Rule 4] PairLogit/PairLogitPairwise leaf-der2 parity remains DEFERRED**
- **What found:** even with the pairwise-weight wiring, the PairLogit leaf-der2 reduction
  diverges (table above) in a way no global L2/pairwise scaling reconciles. This is the
  precise blocker 06.3-03 deferred.
- **Why deferred:** resolving it requires the instrumented catboost 1.2.10 trainer oracle
  (per-leaf SumDer/SumDer2 log), INFEASIBLE this session (no clang/toolchain, disk
  pressure — MEMORY.md `catboost-instrumented-trainer-build`). Per D-6.3-03b
  escalate-don't-weaken (06.3-04 precedent), the oracle tolerance is NOT weakened.
- **Action:** the two oracle tests are committed and `#[ignore]`'d with a documented
  deferral; removing `#[ignore]` runs the full ≤1e-5 gate the moment the leaf-der2 parity
  lands. `deferred-items.md [06.3-03]` updated with the refined root cause + the
  reproduction table.
