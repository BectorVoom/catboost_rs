<objective>
Implement the **pairwise split-scoring subsystem** in `cb-compute` as a pure
library — the missing component isolated by the 06.3-13/06.3-14 verification as
the SPLIT-SELECTION divergence that defers both `PairLogitPairwise` (gap #1) and
`YetiRankPairwise` (gap #2). This is the Rule-4 architectural piece per
LOSS-04: `*Pairwise` (`IsPairwiseScoring`) losses score candidate splits through
upstream's `TPairwiseScoreCalcer` / `CalculatePairwiseScore`
(`pairwise_scoring.cpp`), NOT the pointwise der histogram the current split path
reuses.

This plan ports ONLY the `OneFeature` (float-feature) path — the float-only
frozen ranking corpus needs no BinarySplits / ExclusiveBundle / FeaturesGroup
branches. It adds NO tree-search wiring (that is Plan 06.3-16) and freezes NO
fixtures — it lands the scored-candidate primitive and self-oracles it against a
hand-derived reference so the wiring plan inherits a verified component.

Purpose: One subsystem unblocks BOTH deferred `*Pairwise` losses; isolating it
as a pure library keeps blast radius small (no boosting/tree churn here) and lets
the per-split score be unit-validated against the already-oracle-locked Cholesky
leaf solver before any end-to-end fixture is involved.
Output: `crates/cb-compute/src/pairwise_scoring.rs` + its `_test.rs`, exported
from `cb-compute/src/lib.rs`.
</objective>
