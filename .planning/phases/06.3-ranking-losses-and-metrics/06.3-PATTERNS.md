## Pattern Assignments

### `crates/cb-compute/src/loss.rs` — new ranking der functions (Wave A/B)

**Analog:** itself — the existing scalar der-function block (`loss.rs:30-411`) and the coupled multi-output der `softmax_ders` (`loss.rs:478-499`).

**Pattern to mirror — verbatim transcription with `error_functions.h:line` doc citation, unweighted scalar return.** Every der fn here transcribes a single upstream formula, documents the `error_functions.{h,cpp}:line`, and returns the *unweighted* scalar(s) (the caller multiplies the per-object weight afterward). Example (`loss.rs:339-352`, Poisson — the closest precedent for the exp-approx ranking losses):

```rust
/// Poisson first derivative for one object: `der1 = target - exp(approx)` over the
/// RAW approx.
/// `error_functions.h:657-676` — `TPoissonError::CalcDer` ...
#[must_use]
pub fn poisson_der1(approx: f64, target: f64) -> f64 {
    target - approx.exp()
}
```

- **QueryRMSE / QuerySoftMax** der (per-group, raw approx) → these are *per-group* not per-object, so the group-loop wrapper lives in `ranking_der.rs` (below); the inner per-object formula mirrors a scalar der fn here. Transcribe from `error_functions.h:871-933` (QueryRMSE) and `error_functions.cpp:531-577` (QuerySoftMax). QuerySoftMax must max-shift before `exp` exactly like `calc_softmax` does (`loss.rs:424-448`, `eval_processing.h:18` — Security V5 NaN guard).
- **PairLogit** der over Competitors (exp-approx) → mirror the coupled accumulation shape of `softmax_ders` (`loss.rs:478-499`), but accumulate winner/loser der pairs. Transcribe from `error_functions.h:849-866` (RESEARCH § Code Examples gives the verbatim loop).
- **`isExpApprox` decision** → follow the `Poisson` precedent documented at `loss.rs:328-352`: cb-train stores RAW approx and computes `exp()` INLINE in the der (the inline-sigmoid/inline-exp link), rather than maintaining an exp-approx buffer. Document per-loss which buffer the staged-approx oracle compares (RESEARCH Pitfall 3).

**Validation pattern** (`runtime.rs:283-392`): every new param-carrying ranking `Loss` variant adds a `match` arm to `Loss::validate` returning `cb_core::CbError::OutOfRange` (no `unwrap`/`panic`). Mirror the `Quantile`/`Tweedie` arms (`runtime.rs:316-338`) for `Sigma > 0`, `decay ∈ [0,1]`, `permutations >= 1` (RESEARCH Security V5).

### `crates/cb-compute/src/runtime.rs` — ranking `Loss` variants (all waves)

**Analog:** itself — the `Loss` enum (`runtime.rs:55-259`), `EScoreFunction` (`:404-413`), `Derivatives` (`:433-441`), `Runtime::compute_gradients` (`:469-475`).

**Pattern to mirror — `Loss::Variant { params }` with a full doc-comment carrying the upstream citation + default + leaf-method note.** New variants (`QueryRmse`, `QuerySoftMax { lambda, beta }`, `PairLogit`, `PairLogitPairwise`, `YetiRank { permutations, decay }`, `YetiRankPairwise { .. }`, `LambdaMart { metric, sigma, top }`, `StochasticRank { metric, sigma, mu, num_estimations }`) attach exactly like the existing `MultiQuantile { alpha: Vec<f64>, delta }` (`runtime.rs:250-258`) — which already carries owned non-`Copy` params and is the precedent that dropped `Copy` from `Loss` (`runtime.rs:47-55`). Each variant's doc-comment must follow the existing style (`runtime.rs:226-258`): der formula, `error_functions.h:line`, default values, leaf-estimation-method note, predictions form.

**Grouped der seam dispatch** → the existing `compute_gradients(loss, approx, target, approx_dimension)` (`runtime.rs:469-475`) is pointwise/N-dim. The grouped ranking path needs `group_id`/`pairs` which this trait signature does not carry. Either widen the trait or add a sibling grouped entry (`compute_gradients_grouped`) that `cb-train::boosting` calls for ranking losses — keep the existing non-ranking signature byte-identical (D-04 no-regression on shipped scalar/N-dim oracles). The actual per-group/per-pair reduction lives in `ranking_der.rs`, not in `cb-backend`.

### `crates/cb-compute/src/ranking_der.rs` (NEW) — the grouped der seam (Wave A, the design hinge)

**Analog:** `crates/cb-compute/src/leaf.rs` (ordered-reduction + per-leaf gather discipline) and `loss.rs::softmax_ders` (coupled der accumulation `loss.rs:478-499`).

**Pattern to mirror — `QueryInfo`-shaped slicing + `cb_core::sum_f64` for every per-group/per-pair accumulation.** Mirror `TQueryInfo` (`query.h:19-44`): `QueryInfo { begin, end, weight, subgroup_id: Vec<u32>, competitors: Vec<Vec<Competitor>> }`, `Competitor { id, weight }` (RESEARCH Pattern 1). The seam slices `approx[begin..end]` per group; per-pair losses read `competitors`. Every cross-object/cross-pair sum routes through `cb_core::sum_f64` in upstream's iteration order (group index asc, doc index asc — RESEARCH Pitfall 4), exactly as `boosting.rs` does for leaf gather (`boosting.rs:1000-1050`):

```rust
// boosting.rs:1040-1050 — the per-leaf ordered gather + reduce pattern to mirror per-group
let sum_der: Vec<f64> = (0..k)
    .map(|d| sum_f64(&der1_members[leaf][d]))
    .collect();
...
let delta = solve_symmetric_newton(&sum_der, &sum_der2, scaled_l2);
```

The QueryRMSE `queryAvrg` and QuerySoftMax `Σ expApprox·w` normalizers are per-group reductions — route them through `cb_core::sum_f64`. Guard empty-group / zero-weight division (`queryCount > 0`, `sumWeightedTargets > 0`) returning 0, never dividing (RESEARCH Security V5 — mirror the `metrics.rs:145-149` `total_weight <= 0` degenerate guard).

### `crates/cb-train/src/query_info.rs` (NEW) — build the grouped view from Pool (Wave A)

**Analog:** `crates/cb-data/src/pool.rs` — the `group_id()` (`:184`), `subgroup_id()` (`:190`), `pairs()` (`:196`) accessors + `Pair` type (`:19-27`).

**Pattern to mirror — consume Pool columns read-only, assert contiguous+unique group runs.** `Pool::pairs()` returns `&[Pair]` with `winner_id`/`loser_id` already validated `< n_rows` at ingest (`pool.rs:30-36`). Build `Vec<QueryInfo>` by scanning `group_id()` for contiguous equal runs (mirror upstream `GroupSamples` `query.h:65` — assert grouped+unique, surface a typed `CbError` not a panic, RESEARCH Don't-Hand-Roll). Explicit pairs (`PairLogit`) map `Pool.pairs` → per-group `Competitors` (`data_providers.cpp:315-340`); YetiRank-sampled pairs come from `yetirank.rs`.

**Reference signature shape** (`pool.rs:182-198`):

```rust
/// Per-object group id (empty when ungrouped).

pub fn group_id(&self) -> &[u64] { &self.group_id }
/// Ranking pairs (empty for non-pairwise data).
```

### `crates/cb-train/src/yetirank.rs` (NEW) — RNG-stream pair sampling (Wave C, randomized crux)

**Analog:** `crates/cb-core/src/normal.rs` (`std_normal` — a bit-for-bit RNG-draw-order transcription) + `crates/cb-core/src/rng.rs` (`TFastRng64`, `gen_rand` `:175`, `gen_rand_real1` `:195`).

**Pattern to mirror — transcribe the exact RNG draw count + order, doc-cite each `GenRand`/`GenRandReal1` call, validate against the C++-instrumented log.** `normal.rs` is the canonical precedent: it documents the upstream `util/random/normal.h:11-24` draw loop line-by-line and reproduces it through `TFastRng64::gen_rand_real1`. YetiRank must reproduce the 2-level seed derivation (`yetirank_helpers.cpp:347-393`): block seed → per-query `gen_rand()` re-seed → per-permutation Gumbel noise `u = rng.gen_rand_real1(); expApprox[d] *= u/(1.000001-u)` (RESEARCH Pattern 3). All pairwise-weight accumulation routes through `cb_core::sum_f64`. The Rust `TFastRng64` call sequence must match the `CB_INSTRUMENT_LOG` sequence *before* the der is checked (RESEARCH Pitfall 1).

`cb-core` already exports everything needed (`cb-core/src/lib.rs:24-26`): `std_normal`, `sum_f64`/`sum_f32_in_f64`, `TFastRng64`.

### `crates/cb-train/src/pairwise_leaves.rs` (NEW) — Cholesky pairwise leaf solve (`*Pairwise` only, Wave B)

**Analog:** `crates/cb-compute/src/leaf.rs:184-296` — `solve_symmetric_newton` + `cholesky_solve`. **A dense SPD Cholesky solver already exists in-house** (resolves RESEARCH Open Question 1 — do NOT add a linear-algebra crate).

**Pattern to mirror — reuse/extend the existing `cholesky_solve` (`leaf.rs:251`).** The existing solver does `a = L·Lᵀ`, forward/back substitution, returns `None` on a non-positive pivot so the caller falls back to zeros instead of a NaN (`leaf.rs:251-296`). The pairwise leaf path (`pairwise_leaves_calculation.cpp:9-101`) builds an `(n-1)×(n-1)` SPD system with `diagReg`/`nonDiagReg` priors, a `systemSize==2` closed-form special case, and `MakeZeroAverage` (RESEARCH § Code Examples). Transcribe the matrix assembly + the 2×2 special case + `MakeZeroAverage`, then call the existing `cholesky_solve` (or a thin generalization of it) for the general case; route the `weightSums`/`derSums` accumulation through `cb_core::sum_f64`.

**Gate the path** on a transcribed `is_pairwise_scoring(loss)` predicate (only `{YetiRankPairwise, PairLogitPairwise, QueryCrossEntropy}`, `enum_helpers.cpp:469-475`); `*Pairwise` fixtures must pin `boosting_type=Plain` (`IsPlainOnlyModeLoss`). Do NOT fuse with the pointwise leaf estimators in `boosting.rs::compute_leaf_deltas` (`boosting.rs:869-972`) — keep them separate (RESEARCH Pitfall 2 / Anti-Patterns).

### `crates/cb-train/src/metrics.rs` — ranking metrics (Wave D, eval-only)

**Analog:** itself — `EvalMetric` enum (`metrics.rs:42-55`), `EvalMetric::eval` (`:116-205`), `EvalMetricHistory` (`:213-256`), `EvalMetric::for_loss` (`:66-104`).

**Pattern to mirror — extend `EvalMetric` with variant-with-params; widen `eval` with group structure; route every fold through `cb_core::sum_f64`.** The existing `eval` (`metrics.rs:116-205`) already shows the discipline: length/empty/weight validation up front returning `CbError::Degenerate` (`:117-149`), per-object weighted folds collected into a `Vec`, then `sum_f64(&fold) / total_weight` (`:154-178`). Ranking metrics add `group_id`/`subgroup_id` + `top`/`k`/`border`/`decay` params via the established variant-with-params pattern (mirror how `Loss::Quantile { alpha, delta }` carries params). Add a grouped overload `eval_grouped(approx, target, weights, group_id, subgroup_id, params)` so the non-ranking flat path (`:116`) stays byte-identical (D-04 — RESEARCH Pattern 5). Existing fold shape to mirror (`metrics.rs:154-163`):

```rust
let weighted_sq: Vec<f64> = approx.iter().zip(target.iter()).enumerate()
    .map(|(i, (&a, &t))| { let d = a - t; weight_at(i) * d * d })
    .collect();
Ok((sum_f64(&weighted_sq) / total_weight).sqrt())
```

Transcribe `CompareDocs` (predicted desc → target → stable index) ONCE and share across DCG/NDCG/Precision/Recall/MAP/MRR/ERR (RESEARCH Anti-Patterns / Pitfall 6); `k = (top<0 || size<top) ? size : top`. `EvalMetricHistory` (`:213-256`) is unchanged — ranking metrics push into the same per-set curves.

**Source:** `crates/cb-core/src/reduction.rs:32` (`sum_f64`), `:45` (`sum_f32_in_f64`); exported `cb-core/src/lib.rs:25`.
**Apply to:** EVERY new per-group / per-pair / cross-group accumulation in `ranking_der.rs`, `yetirank.rs`, `pairwise_leaves.rs`, and the ranking arms of `metrics.rs`. CI greps for raw float `+` in reductions (D-08 ban). Iterate in upstream order (group asc, doc asc).
