1. **LOSS-04 тАФ Ranking losses:** YetiRank(/Pairwise), PairLogit(/Pairwise), QueryRMSE, QuerySoftMax, LambdaMart, StochasticRank. Each attaches at the existing `Loss` enum and computes its der per query-group / per-pair, then rides the established train loop тЖТ leaf-estimation тЖТ apply path.

## Implementation Decisions

All decisions were auto-selected in `--auto` mode to the recommended option. They are dominated by constraints already locked upstream of this discussion тАФ umbrella **D-07** (selective C++ instrumentation for randomized losses), **D-08** (instrumented builds under disk pressure), the project's memory-efficiency-first goal, the тЙд1e-5 parity bar, and the family-wave/per-stage-oracle pattern carried from 6.1 (D-6.1-02) and 6.2 (D-6.2-02). None are open design questions; they are the parity-/scope-dictated reading of a fixed scope.

### Wave / Plan Structure
- **D-6.3-01: Family-wave plans, each with its own тЙд1e-5 per-stage oracle gate.** Split the surface so blast radius stays small and regression attribution is unambiguous (carried from 6.1 D-6.1-02 / 6.2 D-6.2-02). Suggested grouping (planner finalizes exact plan-to-wave mapping):
  - **Wave A тАФ grouped seam + deterministic pointwise-per-group losses:** the per-query-group gradient/eval plumbing (D-6.3-03) plus **QueryRMSE** and **QuerySoftMax** (deterministic, Python-reachable).
  - **Wave B тАФ pairwise deterministic losses:** **PairLogit(/Pairwise)** and **LambdaMart** (deterministic given pairs; Python-reachable).
  - **Wave C тАФ randomized losses:** **YetiRank(/Pairwise)** and **StochasticRank** тАФ the RNG-stream losses that get C++ instrumentation (D-6.3-02).
  - **Wave D тАФ ranking metrics:** NDCG, DCG, MAP, MRR, ERR, PFound, PrecisionAt, RecallAt, QueryAUC (eval-only; D-6.3-05).
  - Waves are independent except Waves AтАУC depend on the grouped seam landing first. Rejected: one-mega-plan (defeats per-loss regression attribution); per-loss plans (excess ceremony).

### Oracle Floor & C++ Instrumentation (D-07 / D-08)
- **D-6.3-02: C++ instrumentation ONLY for the randomized losses (YetiRank(/Pairwise), StochasticRank); everything else rides the Python-reachable per-stage oracle.** Per umbrella D-07 ("selective, not universal"): the deterministic losses (PairLogit, QueryRMSE, QuerySoftMax, LambdaMart) and **all** ranking metrics have a clean Python-reachable ground truth, so they ride the transcribe-then-self-oracle + per-stage Python-reachable parity (borders/splits/leaf-values/staged approx + final predictions, тЙд1e-5). YetiRank/StochasticRank draw from RNG streams with **no clean Python-reachable ground truth** тАФ they get the C++-instrumented harness (umbrella D-07's explicit candidates; SC-3). Rejected: instrument-everything (wasted instrumentation budget on deterministic losses); instrument-nothing (leaves the randomized RNG streams unverifiable тАФ fails SC-3).
- **D-6.3-03b: Instrumented builds treat link/disk feasibility as a first-class risk (D-08).** Reuse the **persisted `/tmp` clang-18 + lld-18 + built `_catboost` toolchain** (see `instrumented-trainer-toolchain-persists` + `catboost-instrumented-trainer-build` memories) via **incremental rebuild**, not a fresh full build. Each instrumented-harness task: smallest instrumented unit, free disk before/after, explicit feasibility-probe that **escalates rather than silently expanding scope**. Instrumentation is RUN-ONCE/COMMIT (fixtures frozen, generators run OFFLINE) тАФ same precedent as Phase-5 ORD-01.

### Per-Group / Per-Pair Gradient Plumbing
- **D-6.3-03: Ranking losses compute der per query-group / per-pair, reading the existing `Pool` `group_id` / `subgroup_id` / `pairs` fields through a new grouped seam in the gradient path.** The `Pool` already carries `group_id()`, `subgroup_id()`, `pairs()` (DATA layer, shipped). The gradient dispatch (`Runtime::compute_gradients` / `loss.rs` der) currently operates pointwise on the flat approx; ranking losses need to aggregate **within a query group** (QueryRMSE/QuerySoftMax/listwise) or **over pairs** (PairLogit/LambdaMart/YetiRank). Add a grouped der seam that slices approx/target/weights by group boundaries and computes the per-object gradient contribution upstream-faithfully тАФ **transcribed** from `yetirank_helpers.*` / `pairwise_*` / `querywise_*`, not invented. All accumulation routes through `cb-core::sum_f64` (D-08 CI-grep ban). Rejected: a parallel ranking-only gradient pipeline (duplication / divergence тАФ violates the single-code-path discipline D-03 carried from 6.2).

### Pair-Source Modes
- **D-6.3-04: Support BOTH explicit pairs and group-derived sampled pairs.** PairLogit consumes **explicit `Pool.pairs`** (user-supplied or weighted pairs). YetiRank **generates pairs from `group_id` + target** via the upstream sampling procedure (the randomized part тАФ D-6.3-02 instrumentation covers its RNG stream). The pairwise der path is shared; only the pair *source* differs. Pair generation is transcribed from upstream, not approximated. Rejected: explicit-pairs-only (fails YetiRank); derived-pairs-only (fails PairLogit's explicit-pair contract).

### Ranking-Metric Eval Seam
- **D-6.3-05: Widen `EvalMetric::eval` to receive group structure; ranking metrics are eval-only (no der).** Today `EvalMetric::eval(approx, target, weights)` (`cb-train/src/metrics.rs:116`) has no group plumbing. NDCG/DCG/MAP/MRR/ERR/PFound/PrecisionAt/RecallAt/QueryAUC are **per-query-group** quantities, so the eval seam gains `group_id` / `subgroup_id` (and `top` / `k` params for the @k metrics via the established `EvalMetric` variant-with-params pattern). Metrics carry no gradient тАФ they extend `EvalMetric` / `EvalMetricHistory` only. Rejected: shoehorning group structure through a side-channel (brittle); computing metrics outside the EvalMetric seam (breaks the established eval/early-stopping path).

### Claude's Discretion (parity-dictated тАФ research reads upstream and reproduces)
- Exact der1/der2 formulas + leaf-estimation-method defaults per ranking loss (transcribe from `error_functions.{h,cpp}` + `yetirank_helpers.*` / `pairwise_*` / `querywise_*` + options defaults).

- The exact plan-to-wave mapping and how many losses/metrics share a plan within Waves AтАУD.

## Specific Ideas

- **The grouped der seam is the design hinge of LOSS-04.** Everything ranking-loss-specific funnels through one new seam that slices approx/target/weights by group/pair boundaries and computes per-object gradient contributions; it must not perturb the existing non-ranking pointwise locks (D-04-style no-regression discipline on the shipped scalar/N-dim oracles). Get this seam transcribed from `querywise_*` / `pairwise_*` before any individual loss.
- **YetiRank and StochasticRank are the only RNG-stream losses** тАФ isolate their randomized pair-sampling / weighting and validate against the C++-instrumented ground truth (`CB_INSTRUMENT_LOG`). Everything else (PairLogit, QueryRMSE, QuerySoftMax, LambdaMart) is deterministic and Python-reachable. This keeps the instrumentation surface minimal and disk-pressure-bounded (D-08).
- **PairLogit's pairwise leaf calculation differs from pointwise leaf estimation** тАФ upstream has a dedicated `pairwise_leaves_calculation`. Confirm whether the pairwise losses need that path vs. the existing leaf estimators during research.
