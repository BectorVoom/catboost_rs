# Phase 06.3 Plan 07: CR-02 weighted_der1 Double-Weight Fix Summary

**Closed the CR-02 BLOCKER: `boosting.rs` now branches `weighted_der1` on `group_spans.is_some()` so QueryRMSE/QuerySoftMax grouped ders (already weight-folded) are used as-is instead of being re-multiplied by the per-object weight — eliminating squared-weight gradients in any non-uniform-weight ranking training.**

## Accomplishments

- **CR-02 BLOCKER closed (boosting.rs:2241):** `weighted_der1` now branches on `group_spans.is_some()`. The grouped (ranking) path uses `ders.der1.clone()` directly — QueryRMSE's der already carries `* weight` (`(target - approx - queryAvrg) * weight`) and QuerySoftMax's carries `weight` through both the `weight * target` term and the softmax probability numerator. Re-multiplying by `weights[i]` (the old unconditional behavior) double-weighted the gradient (squared weights → corrupt split scores and leaf values). The pointwise branch keeps the exact `der1 * weight` map with `idx % n` (byte-identical at `approx_dimension == 1`, D-04).
- **Comment clarity:** Updated the grouped-der dispatch comment (boosting.rs:2185) and the `weighted_der1` comment block to explain WHY the grouped path is not re-weighted (would double-weight) and to point to the regression test.
- **Regression test:** Added `grouped_weight_regression_test.rs` with non-uniform per-object weights `[2.0, 0.5, 1.5, 1.0, 0.25]` over 2 groups. It gates the single-weighting invariant against the public `cb_compute::calc_ders_for_queries` grouped seam (the exact buffer the trainer routes into `weighted_der1`): (1) grouped seam der == an independently-computed single-weighted reference (≤1e-12); (2) grouped seam der differs from the squared-weight value (`der * weight`) for every non-unit-weight object — so the test would FAIL against the pre-CR-02 unconditional `der1 * weight` code. Covers both `Loss::QueryRmse` and `Loss::QuerySoftMax`.

## Decisions Made

- **Gate against the public der seam, not an end-to-end trainer fixture.** No instrumented catboost TRAINER build is in scope (LOSS-04 truths #5/#7 stay deferred — see Next Phase Readiness). The grouped seam `calc_ders_for_queries` is exactly the buffer the trainer uses as `weighted_der1` on the grouped path, so asserting (a) seam der == single-weighted reference and (b) seam der != `der * weight` for non-unit weights is sufficient to gate the branch invariant and would fail against the pre-CR-02 code.
- **Pointwise branch preserved byte-for-byte.** The `else` branch keeps the existing `idx % n` / `weights.get(i).copied().unwrap_or(1.0)` map verbatim (D-04 dim=1 byte-identity, T-06.3-07-02 no unchecked indexing).
