## Summary

Phase 6.3 adds the **grouped** dimension that all prior loss work (6.1 scalar, 6.2 N-dim) did not need: per-query-group aggregation (QueryRMSE/QuerySoftMax/LambdaMart/StochasticRank) and per-pair aggregation (PairLogit/YetiRank). Upstream concentrates the entire ranking-loss surface in **one abstraction**: `IDerCalcer::CalcDersForQueries(queryStart, queryEnd, approxes, targets, weights, queriesInfo, ders, randomSeed, executor)` (`error_functions.h`). Every ranking loss is a subclass overriding that method; the `queriesInfo: TVector<TQueryInfo>` carries group boundaries `[Begin,End)`, per-group `Weight`, optional `SubgroupId`, and the `Competitors` adjacency (winner→{loser,weight}) that both explicit pairs and YetiRank-sampled pairs populate. This is the exact shape of the **grouped der seam** D-6.3-03 calls for — the Rust seam should mirror `TQueryInfo` + `CalcDersForQueries`, not invent a new structure.

Two of eight losses are randomized: **YetiRank/YetiRankPairwise** (samples `permutations` orderings per group by perturbing exp-approxes with Gumbel/Gauss noise, then accumulates pairwise weights — `yetirank_helpers.cpp`) and **StochasticRank** (Monte-Carlo gradient estimate via `NumEstimations` Gaussian-noise samples per group — `error_functions.cpp:1008`). Both draw from `TFastRng64` (already ported, INFRA-05) with a precise, documented seed-derivation and draw order — these are the only two losses needing the D-07 C++-instrumented ground truth. The six remaining losses and all nine metrics are deterministic and Python-reachable.

Two leaf-estimation paths exist: the **pointwise** path (QueryRMSE/QuerySoftMax/LambdaMart/StochasticRank — der per object, ride the existing Gradient/Newton leaf estimators) and the dedicated **pairwise leaf-calculation** path (`PairLogitPairwise`/`YetiRankPairwise`/`QueryCrossEntropy` — `IsPairwiseScoring`==true — solve a Cholesky linear system over pairwise weight sums, `pairwise_leaves_calculation.cpp`). The non-`Pairwise` variants (`PairLogit`, `YetiRank`) use the **pointwise** leaf estimators with `isExpApprox==true` storage. This pairwise-vs-pointwise split is the single biggest structural decision of the phase.

**Primary recommendation:** Land the grouped der seam first (Wave A), modeling it on `TQueryInfo` + `CalcDersForQueries`, with `isExpApprox` storage handled per-loss. Transcribe each der/metric verbatim with the file:line citations below; route all per-group/per-pair sums through `cb-core::sum_f64`. Defer the pairwise Cholesky leaf path until Wave B (it is only needed by the `*Pairwise` variants). Treat the YetiRank/StochasticRank RNG draw-order as the parity crux and build the C++-instrumented harness against the **smallest** instrumented unit (one query group, `permutations`/`NumEstimations`=small, single thread) — but verify the persisted `/tmp` toolchain still exists FIRST (it was not found this session).

## Per-Loss Der Matrix (the LOSS-04 contract)

> All der formulas transcribed from `error_functions.{h,cpp}` + `yetirank_helpers.cpp`, catboost 1.2.10. `isExpApprox` = whether the stored approx is `exp(rawApprox)` (PairLogit/YetiRank) vs raw (querywise). Leaf method = the upstream default leaf-estimation path. "Pairwise leaf" = needs the Cholesky `pairwise_leaves_calculation` path (`IsPairwiseScoring`).

| Loss | Error type | isExpApprox | der1 / der2 | Pair source | Leaf path | Citation |
|------|-----------|-------------|-------------|-------------|-----------|----------|
| **QueryRMSE** | Querywise | false | `der1 = (target - approx - queryAvrg) * w`, `der2 = -1 * w`; `queryAvrg = Σ_g (target-approx)·w / Σ_g w` (per group) | n/a (group) | pointwise | `error_functions.h:871-933` |
| **QuerySoftMax** | Querywise | false | per-group softmax over `Beta*approx` (max-shifted): `p = expApprox·w / Σ expApprox·w`; `der1 = Beta·(-sumWTargets·p + w·target)`, `der2 = Beta·sumWTargets·(Beta·p·(p-1) - LambdaReg)`; if `sumWTargets<=0` ders=0 | n/a (group) | pointwise | `error_functions.cpp:531-577`; defaults `lambda`/`beta` in `loss_description.cpp` |
| **PairLogit** | Pairwise | true | per winner doc, over its Competitors: `p = expApprox[competitor]/(expApprox[competitor]+expApprox[doc])`; `winnerDer += w·p`, `loserDer1 -= w·p`; `winnerDer2 += w·p(p-1)`, `loserDer2 += w·p(p-1)` | explicit `Pool.pairs` | pointwise (PairLogit) | `error_functions.h:823-868` |
| **PairLogitPairwise** | Pairwise | true | same der as PairLogit | explicit pairs | **pairwise Cholesky** (`IsPairwiseScoring`) | `error_functions.h:823`; `pairwise_leaves_calculation.cpp` |
| **LambdaMart** | Querywise | false | per-group lambda gradient toward a target metric (NDCG default-ish; supports DCG/NDCG/MRR/ERR/MAP): sort by approx, for each ordered pair with `firstTarget>secondTarget`: `delta = (dcgNum·dcgDen)/idealScore` (norm option `/= 0.01+|approxDiff|`); `antigrad = -Sigma·delta/(1+exp(Sigma·approxDiff))`; accumulate ±antigrad and hessian. **der2 maxDerivativeOrder=1** but Newton hessian filled in CalcDersNDCG | n/a (group, derived from target order) | pointwise | `error_functions.cpp:589-941`; ctor `:1325` |
| **StochasticRank** | Querywise | false | **Monte-Carlo**: shift approx by `-Sigma·Mu·target`, center; for `NumEstimations` samples add `Sigma·StdNormal(rng)` noise, sort, accumulate metric-position-weight gradients; averaged. Supports DCG/NDCG/PFound/FilteredDCG/ERR/MRR | n/a (group) | pointwise | `error_functions.cpp:943-1090+`; ctor `:1409` |
| **YetiRank** | Pairwise (Order) | true | der = PairLogit der over **sampled** Competitors (pairs generated per-group by `UpdatePairsForYetiRank`); recomputed each iteration | **YetiRank-sampled** (RNG) | pointwise (YetiRank) | `yetirank_helpers.cpp:347-418`; der via PairLogit `error_functions.h:823` |
| **YetiRankPairwise** | Pairwise (Order) | true | same sampled pairs | YetiRank-sampled (RNG) | **pairwise Cholesky** (`IsPairwiseScoring`) | `yetirank_helpers.cpp`; `pairwise_leaves_calculation.cpp` |

**Key structural facts:**
- `IsPairwiseScoring(loss)` ⇒ `{YetiRankPairwise, PairLogitPairwise, QueryCrossEntropy}` only. These three use the Cholesky pairwise leaf path AND pairwise split scoring. [VERIFIED: enum_helpers.cpp:469-475]
- `IsPlainOnlyModeLoss` ⇒ the `*Pairwise` variants force `boosting_type=Plain` (no Ordered boosting). [VERIFIED: enum_helpers.cpp:460-467] — relevant: ranking `*Pairwise` fixtures must pin `boosting_type=Plain`.
- `UsesPairsForCalculation` ⇒ `{YetiRank*, PairLogit*}` — these four consume the `Competitors` adjacency. [VERIFIED: enum_helpers.cpp:503]
- PairLogit/YetiRank store `expApprox` (isExpApprox==true, `CB_ENSURE(isExpApprox==true)` in ctor `error_functions.h:828`); querywise losses store raw approx (isExpApprox==false). The der formulas above read whichever the loss stores.

|--------|------------------------------------------|----------|----------------------|----------|

### Pattern 2: isExpApprox storage per loss

**What:** PairLogit/YetiRank store `exp(rawApprox)`; querywise losses store raw approx. The der reads the stored form.
**When to use:** When wiring each loss's der + the staged-approx oracle.
**Note:** This mirrors the Phase-6.1 Poisson/Tweedie precedent (STATE.md: "cb-train stores RAW approx + computes exp() INLINE in the der"). ### Pattern 3: YetiRank pair sampling (the randomized crux)

**What:** Per group, per boosting iteration: draw `permutations` (default 10) orderings by perturbing the exp-approx with noise (Gumbel default), stable-sort descending, accumulate pairwise weights, divide by `permutations`.
**RNG draw order (the parity contract):**
```cpp
// Source: yetirank_helpers.cpp:347-393 (UpdatePairsForYetiRank) + :305-345 (per query)
// 1. randomSeeds = GenRandUI64Vector(blockCount, randomSeed)   // per thread-block
// 2. per block: TFastRng64 rand(randomSeeds[blockId]);

// 4. inside per-query: TFastRng64 rand(randomSeed);  // re-seeded from the GenRand() above
```

### Pattern 4: StochasticRank Monte-Carlo gradient

**What:** Per group: shift approx by `-Sigma·Mu·target`, center to zero mean; for `NumEstimations` samples, add `Sigma·StdNormal(rng)` noise, stable-sort desc, accumulate metric-position-weight gradients; average.
**RNG draw order:**
```cpp
// Source: error_functions.cpp:1008-1090 (CalcDersForSingleQuery)
// seed = randomSeed + queryIndex  (from CalcDersForQueries :997/:1003)
// TFastRng64 rng(seed);
// for sample in 0..NumEstimations:
//    for docId: noise[docId] = StdNormalDistribution<double>(rng); scores = shifted + Sigma·noise
//    StableSort order by scores desc
//    posWeights = Compute{DCG|PFound|ERR}PosWeights(...)  // metric-dependent
//    ... accumulate gradient
```
**Differs from YetiRank:** StochasticRank is querywise (pointwise leaf, no Competitors/pairs) and estimates the gradient *directly* via noise; YetiRank generates *pairs* and rides the PairLogit der. StochasticRank's noise is Gaussian (`StdNormalDistribution`); YetiRank's default is Gumbel. [VERIFIED: error_functions.cpp:1008-1062]

### Pattern 5: Widened EvalMetric eval seam (group structure)

**What:** `EvalMetric::eval` today is `(approx, target, weights)` (`metrics.rs:116`). Ranking metrics need `group_id`/`subgroup_id` and `top`/`k`/`border`/`decay` params.
**When to use:** All 9 ranking metrics.
**Recommendation:** Add a grouped overload (or widen the signature) `eval_grouped(approx, target, weights, group_id, subgroup_id, params)` so non-ranking metrics keep the existing flat path byte-identical (D-04 no-regression discipline). Ranking-metric variants carry their params via the established `EvalMetric::Variant { top, .. }` pattern. [CITED: metrics.rs:116; CONTEXT D-6.3-05]

### Anti-Patterns to Avoid

- **Fusing pointwise and pairwise leaf estimation into one path.** Only `*Pairwise` variants use the Cholesky solve; the rest use the existing pointwise estimators. Keep them separate (mirror upstream `IsPairwiseScoring` branch).
- **Re-implementing the metric sort per metric.** The doc-comparator tie-break (predicted desc → target → stable index) is shared; transcribe `CompareDocs` once.
- **Drawing RNG in a different order/granularity than upstream.** The 2-level seed derivation (block seed → per-query GenRand → per-query rng) is load-bearing; a single flat rng will diverge. Instrument to confirm.
- **Summing per-group/per-pair contributions with raw `+`.** D-08 CI-grep bans it; route through `cb-core::sum_f64`.
- **Storing the wrong approx form.** PairLogit/YetiRank are isExpApprox==true; comparing a RAW staged-approx oracle against an exp-stored buffer (or vice-versa) silently fails ≤1e-5.

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Per-group/per-pair summation | A new accumulator | `cb-core::sum_f64` / `sum_f32_in_f64` | Order-locked f64 parity; D-08 CI-grep ban. |
| Group boundary detection | Ad-hoc scan | Mirror upstream `GroupSamples` (`query.h:65`): contiguous equal `group_id` runs, assert grouped+unique | Upstream asserts group ids are contiguous and unique; matching the assertion prevents silent mis-grouping. |
| RNG stream | A fresh PRNG | `cb-core::TFastRng64` (INFRA-05, bitstream-validated) | The randomized losses MUST match this exact stream. |

| YetiRank/StochasticRank ground truth | Guessing the RNG draw order | C++-instrumented harness (`CB_INSTRUMENT_LOG`) on the persisted `/tmp` toolchain | D-6.3-02 — no clean Python-reachable ground truth for the RNG streams. |

The phase's whole risk surface collapses to: (1) build that structure faithfully, (2) transcribe each der, (3) match the RNG draw order for the two randomized losses, (4) add the Cholesky leaf path for the three `*Pairwise` variants. Inventing parallel structures multiplies the divergence surface.

## Common Pitfalls

### Pitfall 1: RNG draw-order divergence (YetiRank/StochasticRank)
**What goes wrong:** Rust draws noise in a different order/count than upstream → sampled pairs differ → der differs → fails ≤1e-5 from tree 0.
**Why it happens:** Upstream's 2-level seed derivation (per-block `GenRandUI64Vector` → per-query `GenRand()` → re-seeded per-query rng) is non-obvious; YetiRank uses Gumbel (`GenRandReal1`) while StochasticRank uses Gaussian (`StdNormalDistribution`).
**How to avoid:** Build the C++-instrumented harness FIRST for these two; log every RNG call (count + value) and the sampled pairs/weights. Reproduce single-thread (`blockCount=1`). Confirm the Rust `TFastRng64` call sequence matches the logged sequence before checking the der.
**Warning signs:** der matches for `permutations=1`/`NumEstimations=1` but diverges at the default → seed re-derivation across samples is wrong.

### Pitfall 2: Pairwise vs pointwise leaf path mis-routing
**What goes wrong:** Using pointwise leaf estimation for a `*Pairwise` loss (or vice-versa) → leaf values diverge.
**Why it happens:** Only `IsPairwiseScoring` losses (`YetiRankPairwise`/`PairLogitPairwise`/`QueryCrossEntropy`) use the Cholesky solve; `PairLogit`/`YetiRank` (no `Pairwise` suffix) use pointwise.
**How to avoid:** Gate on a transcribed `is_pairwise_scoring(loss)` predicate; `*Pairwise` fixtures must also pin `boosting_type=Plain` (`IsPlainOnlyModeLoss`).
**Warning signs:** splits match but leaf values are off → wrong leaf path.

### Pitfall 3: isExpApprox staging mismatch
**What goes wrong:** The staged-approx oracle compares a RAW buffer against an exp-stored buffer.
**Why it happens:** PairLogit/YetiRank are isExpApprox==true; querywise losses are raw.
**How to avoid:** Decide per-loss what the staged approx buffer holds; make the oracle's `StagedApprox` stage compare the same form upstream stores. Mirror the 6.1 Poisson "RAW staged + Exponent prediction" precedent where applicable.
**Warning signs:** predictions match but staged approx fails (or vice-versa).

### Pitfall 4: Group-block summation order
**What goes wrong:** Summing across groups in a different order than upstream → last-ULP drift accumulates past 1e-5 over many groups/iterations.
**Why it happens:** Upstream parallelizes per-query but the *reduction* into the metric/leaf is order-sensitive at f64.
**How to avoid:** Route every cross-group and within-group sum through `cb-core::sum_f64` in upstream's iteration order (group index ascending, doc index ascending). Single-thread fixtures (`thread_count=1`) eliminate the parallel-reduction nondeterminism.
**Warning signs:** small groups pass, large corpora drift.

### Pitfall 5: LambdaMart hessian with maxDerivativeOrder=1
**What goes wrong:** LambdaMart's ctor sets `IDerCalcer(false, 1, ...)` (max order 1) yet `CalcDersNDCG` fills `Der2` (hessian). Choosing the wrong leaf-estimation method (Gradient vs Newton) diverges.
**Why it happens:** The der order declaration and the actual hessian fill disagree; upstream's default leaf method for LambdaMart must be pinned.
**How to avoid:** Read the upstream default leaf-estimation method for LambdaMart (`leaf_estimation_method` default per loss in `options/`) and pin it per fixture; transcribe whether der2 is consumed.
**Warning signs:** leaf values off by a hessian factor.

### Pitfall 6: Metric `top=-1` and tie-break semantics
**What goes wrong:** Treating `top=-1` as literal k, or sorting ties differently from `CompareDocs`.
**Why it happens:** `DefaultTopSize=-1` means "use full group"; `nth_element`/`StableSort` with the doc-comparator break ties by target then stable index.
**How to avoid:** `k = (top<0 || size<top) ? size : top` (`precision_recall_at_k.cpp:9`); transcribe `CompareDocs` exactly.
**Warning signs:** @k metrics pass for distinct predictions, fail when predictions tie.
