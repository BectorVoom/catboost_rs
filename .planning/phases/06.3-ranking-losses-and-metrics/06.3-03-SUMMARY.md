## Decisions Made

- **Pairwise der scatter order is the parity contract.** PairLogit/LambdaMart der scatter winner/loser contributions with `+=`/`-=` in upstream's exact loop order rather than reordering through `sum_f64`; the per-group normalizers (`idealScore`, `MakeZeroAverage`) likewise mirror upstream's own sequential float accumulation. (The `sum_f64` discipline applies to the per-leaf `derSums` reductions, which it does use.)
- **Reused the vendored Cholesky** via a thin `pub fn pairwise_cholesky_solve` over the private `leaf.rs::cholesky_solve` — no linear-algebra crate (RESEARCH Open Q1).
- **`newton_leaf_delta` negative-denominator fix (Rule 1):** see Deviations.
- **LambdaMart MRR/ERR/MAP der arms deferred** — only the NDCG default is fixture-reachable this wave; the executor ships no unverifiable der math (the arms are reachable-as-zero no-ops, gated by `validate`, with a follow-up note).

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] `newton_leaf_delta` clamped legitimate negative denominators to zero, zeroing every LambdaMart leaf**
- **Found during:** Task 3 (LambdaMart oracle)
- **Issue:** `newton_leaf_delta` returned `0.0` whenever `-sum_der2 + scaled_l2 <= 0`. That guard was safe for the regression/binary losses (NON-positive `der2` ⇒ denominator always `> 0`), but LambdaMart fills a STRICTLY POSITIVE Newton hessian (`Sigma²·delta·σ(1-σ)`), so its denominator is legitimately NEGATIVE and upstream `CalcDeltaNewtonBody` (`online_predictor.h:162-170`) divides by it. The clamp zeroed every LambdaMart leaf → all-zero tree-0 leaves.
- **Fix:** divide verbatim for every non-zero denominator (positive OR negative); guard ONLY the exact-zero case (empty leaf with no L2) to avoid a `0/0` NaN. Updated the existing `newton_guards_degenerate_denominator_returns_zero` test (renamed `..._only_exact_zero_denominator`) to assert the negative-denominator division.
- **Files modified:** `crates/cb-compute/src/leaf.rs`, `crates/cb-compute/src/leaf_test.rs`
- **Verification:** LambdaMart oracle now GREEN ≤1e-5 (all 4 stages); all shipped regression/binary/QueryRMSE/QuerySoftMax oracles + 113 cb-compute lib tests stay green (their denominators are unchanged, always `> 0`).
- **Committed in:** `bef767d` (Task 3 commit)

### Deferred Work (not auto-fixable within scope)

**PairLogit / PairLogitPairwise per-stage oracles — DEFERRED (pair-weight normalization parity gap).**
- **Found during:** Task 3 (PairLogit oracle authoring).
- **Issue:** PairLogit tree-0 leaf values diverge from catboost 1.2.10. Probed with both leaf methods on an isolating depth-1/l2=0/lr=1 tree: the transcribed `sumDer` MATCHES upstream, but BOTH the Gradient `sumWeight` and the Newton `-sumDer2` are scaled by a NON-uniform per-pair weight (implied `sumWeight = [72/7, 36/7]` vs raw incidence counts `[8, 6]`; the `7` denominator == total pair count). catboost applies a group-relative pair-weight normalization to explicit `Pool.pairs` that `build_query_info`'s pair→`Competitor.weight` mapping (Plan 01 seam) sets to a flat `1.0`. The der1/der2 FORMULAS and the Cholesky pairwise-leaf solve are independently unit-test-verified (the latter bit-exact vs `pairwise_leaves_calculation_ut.cpp`).
- **Decision:** per the fix-attempt limit and the no-fake-pass integrity rule, the PairLogit/PairLogitPairwise machinery (der, Cholesky leaf, routing, predicates) LANDS and is unit-tested green, the frozen fixtures are committed, and the two end-to-end oracle tests are DEFERRED (NOT shipped failing, NO `#[ignore]`, NO weakened tolerance). Full analysis + the follow-up plan (transcribe the upstream pair-weight normalization into `build_query_info`, then gate the two oracles against the already-frozen fixtures) is in `deferred-items.md`.
