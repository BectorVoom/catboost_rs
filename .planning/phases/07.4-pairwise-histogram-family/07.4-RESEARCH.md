# Phase 7.4: Pairwise Histogram Family - Research

**Researched:** 2026-06-20
**Domain:** GPU compute kernels (CubeCL / cubecl-hip on AMD gfx1100) — porting the CatBoost CUDA `pairwise_hist*` 4-channel object-pair histogram FILL family at structural parity, device-resident over `R: Runtime`/`F: Float`, with CUDA-style in-kernel atomic accumulation. The pairwise sibling of Phase 7.3 (pointwise `hist2`).
**Confidence:** HIGH for the upstream structure (read directly from the vendored `.cu/.cuh`), the cubecl 0.10.0 atomic/shared-mem/comptime surface (already verified + exercised by the shipped 7.3 kernels), and the as-implemented 7.3 seam this phase mirrors. MEDIUM for the exact wave-agnostic shared-memory pair-line geometry per bit-width (a D-09-constrained design choice finalized + validated empirically on gfx1100 in execution). The GPU-06 epsilon is deliberately NOT signed off here (7.6).

## Summary

Phase 7.4 ports the **fill** half of CatBoost's *pairwise* (object-pair) histogram kernels — `ComputePairwiseHistogramOneByte{5,6,7}Bits`, `ComputePairwiseHistogramOneByte8BitAtomics`, `ComputePairwiseHistogramHalfByte`, `ComputePairwiseHistogramBinary`, plus the **one-hot** overlay of the 5/6/7/8-bit paths — to run device-resident over `R: Runtime`/`F: Float`. This is the GPU analog of the pairwise/ranking split path (the histogram surface for PairLogitPairwise / YetiRankPairwise landed in 6.3).

The decisive structural fact, confirmed by reading every family's `.cu/.cuh`: the pairwise histogram is **4-channel, not 2-channel, and single-stat (weight only), not (der1, weight)**. Where 7.3's `hist2` accumulates per single object `(Σ der1, Σ weight)` into 2 interleaved channels per (feature, bin), the pairwise family accumulates per **object pair** `(i, j)` a single per-pair `weight` into **4 channels** per (feature, bin/fold) — `histId ∈ {0,1,2,3}` encoding the 2×2 combination of "object i's bin leq/ge the split border" × "object j's bin leq/ge", selected by `TCmpBins::Compare(i, bin1, bin2, flag)`. The final per-block merge is `histogram[4 * FirstFoldIndex + 4 * fold + histId] += val` across all six families — so the device-resident output handle's line size is `histLineSize = 4 * totalBinFeatures`, the load-bearing seam contract for 7.5 (versus 7.3's `2 * totalBinFeatures`). There is **no der1/target channel** in this phase: the only per-pair value is the pairwise weight from the ranking derivative path.

The six families are genuinely distinct structural kernels (different shared-memory line sizes and bin-decomposition bit math), exactly as 7.3's three were — D-7.4-02's "distinct families, not one mega-kernel" matches upstream's own file split. The 5/6/7-bit non-binary paths share one template differing only by max-fold/line-size (a `#[comptime] bits` arg, identical mechanism to the shipped `pointwise_hist2_nonbinary_kernel`). The **8-bit-atomics path is structurally distinct** because a 256-bin × 32-slice line does not fit the per-block shared-memory budget the sub-8-bit paths use, so upstream accumulates via a per-thread cached-bin run + **direct global `atomicAdd`** — a different memory structure (D-7.4-02/04). **One-hot is a `#[comptime] one_hot: bool` overlay** that swaps the `Compare` predicate (`bin1 >= bin2 == flag` → `bin1 == bin2`), not a separate hand-written kernel — the cleanest CubeCL expression and the resolution of the research-flag item.

**Primary recommendation:** Build narrowest-first per D-7.4-01 across **5 plans**: Plan A lands the **pairwise device seam + the one-byte non-binary path parameterized to a comptime `bits` (5/6/7)** as a `#[cube]` per-pair atomic-merge fill (mirror the shipped 7.3 direct-global-atomic MVP — defer upstream's per-block shared-mem pre-reduce as a perf follow-up), consuming a device-resident `pairs` buffer + per-pair `weight` + `cindex` + partition, and self-oracle it against a fresh ordered `cb_core::sum_f64` host reference grounded by a PairLogitPairwise pair/weight fixture; **freeze the `histLineSize = 4 * totalBinFeatures` 4-channel layout here**. Plan B adds the **8-bit-atomics** distinct family. Plan C adds **half-byte**. Plan D adds **binary**. Plan E layers the **one-hot `#[comptime]` overlay** onto the bit-width paths. Do NOT port `BuildBinaryFeatureHistograms` / `UpdatePairwiseHistograms` / `ScanPairwiseHistograms` or `split_pairwise.cuh` (that is 7.5). Clone `kernels/pointwise_hist.rs` → `kernels/pairwise_hist.rs` and reuse the entire 7.3/7.2 harness, capability-gate, typed-error, and no-readback machinery verbatim.

### Pattern 1: per-pair 4-channel fill — the core structural difference from 7.3
**What:** The kernel iterates over PAIRS, not objects. For each pair `(i, j)` with weight `w`, for each feature `f`: read `bin1 = cindex[f*n + i]`, `bin2 = cindex[f*n + j]`; compute `histId = Compare(bin1, bin2, flag, one_hot)`; atomic-add `w` into the **4-channel** cell. The 4 channels encode the 2×2 (i leq/ge border)×(j leq/ge border) combination — upstream's `(invBin1&invBin2) | (invBin1&bin2)<<8 | (bin1&invBin2)<<16 | (bin1&bin2)<<24` (binary path) and the `tmp = (Compare? 0 : 8) + f; offset = tmp + (bin<<5) + flag` (non-binary path) both decompose to "which of 4 channels gets the weight." There is **no der1 channel** — `weight` is the only per-pair value.

**When to use:** All four families. The MVP fill (per the shipped 7.3 precedent) does a direct per-pair global `Atomic<F>::fetch_add` into the 4-channel cell; upstream's per-block shared-mem pre-reduce is the perf follow-up.

**Example (the shipped 7.3 non-binary kernel — the precise mechanism to adapt object→pair, 2-channel→4-channel):**
```rust
// Source: crates/cb-backend/src/kernels.rs:621-681 (pointwise_hist2_nonbinary_kernel — SHIPPED 7.3)
#[cube(launch)]
pub fn pointwise_hist2_nonbinary_kernel<F: Float>(
    der1: &Array<F>, weight: &Array<F>,
    cindex: &Array<u32>, indices: &Array<u32>,
    bin_sums: &Array<Atomic<F>>,
    n_features: u32,
    #[comptime] bits: u32,
) {
    let n_bins = comptime!((1u32 << bits) as usize);
    let n = indices.len();
    let stride = CUBE_COUNT * (CUBE_DIM as usize);   // topology-derived, NO warp literal (D-09)
    let mut i = ABSOLUTE_POS;
    while i < n {
        let obj = indices[i] as usize;
        let d = der1[obj]; let w = weight[obj];
        let mut feature = 0usize;
        while feature < (n_features as usize) {
            let bin = cindex[feature * n + obj] as usize;
            let cell = (feature * n_bins + bin) * 2usize;   // 2 channels (7.3)
            bin_sums[cell].fetch_add(d);                     // ch0 = Σ der1
            bin_sums[cell + 1usize].fetch_add(w);            // ch1 = Σ weight
            feature += 1usize;
        }
        i += stride;
    }
}
```

```rust
#[cube(launch)]
pub fn pairwise_hist_nonbinary_kernel<F: Float>(
    pair_i: &Array<u32>, pair_j: &Array<u32>,   // two parallel u32 arrays = uint2 pairs (Open Q1)
    pair_weight: &Array<F>,                      // per-pair weight (the ONLY value; no der1)
    cindex: &Array<u32>,
    bin_sums: &Array<Atomic<F>>,                 // 4-channel global histogram, atomic-merged (D-03)
    n_features: u32,
    #[comptime] bits: u32,                       // 5/6/7 — drives n_bins = 1<<bits
    #[comptime] one_hot: bool,                   // swaps Compare predicate (research-flag resolution)
) {
    let n_bins = comptime!((1u32 << bits) as usize);
    let n_pairs = pair_weight.len();
    let stride = CUBE_COUNT * (CUBE_DIM as usize);
    let mut p = ABSOLUTE_POS;
    while p < n_pairs {
        let oi = pair_i[p] as usize;
        let oj = pair_j[p] as usize;
        let w = pair_weight[p];
        let mut feature = 0usize;
        while feature < (n_features as usize) {
            let bin1 = cindex[feature * n_pairs_or_n + oi] as usize;  // NOTE: cindex stride is over OBJECTS (n), not pairs — see Open Q2
            let bin2 = cindex[feature * n_pairs_or_n + oj] as usize;

            // upstream writes into the (feature, max(bin1,bin2)/fold) cell; the exact fold/bin the
            // channel attaches to follows split_properties_helpers (resolve in Plan A, see Open Q2).
            let cell = (feature * n_bins + /*fold/bin*/ bin) * 4usize + hist_id;
            bin_sums[cell].fetch_add(w);
            feature += 1usize;
        }
        p += stride;
    }
}
```

### Pattern 2: comptime one-hot overlay (the research-flag resolution)
**What:** Upstream selects one-hot via the template type `TCmpBinsOneByteTrait<OneHotPass>::TCmpBins`, whose `Compare` is `bin1 >= bin2 == flag` (non-one-hot) vs `IsOneHot ? bin1 == bin2 : bin1 >= bin2 == flag` (one-hot). **When to use:** Plan E layers `one_hot` onto the non-binary (Plan A) and 8-bit-atomics (Plan B) kernels. Half-byte and binary do not take a one-hot overlay upstream (no `*_one_hot.cu` for them).

### Pattern 3: 8-bit-atomics as a distinct global-atomic family (D-7.4-02/04)
**What:** At 8 bits the working line is 256 bins × (up to 32 slices), which does not fit the per-block shared-memory budget the 5/6/7-bit paths use. Upstream's `TEightBitHistogram` keeps a tiny per-thread cache (`CachedBinsLeq/Ge[8]`, `CachedSumsLeq/Ge[8]`) and **flushes to the global histogram via `atomicAdd` on every cache miss + a final `Reduce()` flush** — true global atomics, no block shared-mem histogram. For the CubeCL MVP this collapses cleanly to the SAME direct-per-pair-global-`Atomic<F>::fetch_add` the non-binary MVP uses (the cache is a perf optimization over the same atomic structure) — i.e. the 8-bit-atomics kernel is structurally a `bits = 8` non-binary kernel that ALWAYS uses global atomics (no shared-mem pre-reduce reserved), which the shipped 7.3 MVP already does for every width. The distinct FAMILY is preserved (a separate `#[cube]` kernel + a separate launch arm), faithful to upstream's separate `.cu`, even though the MVP bodies converge.

**When to use:** Plan B. Keep it a separate kernel symbol + launch arm (D-7.4-02 structural parity) even though the MVP body mirrors the non-binary one with `bits = 8`; document that upstream's per-thread cached-bin run is the perf follow-up.

### Anti-Patterns to Avoid
- **A 2-channel (Σ der1, Σ weight) layout copied from 7.3:** the pairwise histogram is **4-channel, weight-only**. `histLineSize = 4 * totalBinFeatures`, index `(feature * n_bins + bin) * 4 + histId`. Copying 7.3's `* 2 + channel` is the #1 way to silently break the 7.5 seam (Pitfall 2).
- **Reusing the 7.2 der1/der2 handle as kernel input:** there is no der path in 7.4 (D-7.4-03 is a DISTINCT seam). The only per-pair value is `weight` from `ranking_der`. Feeding der1 here is a category error.
- **One mega-kernel with runtime bit-count/one-hot branches:** violates D-7.4-02/D-01. Use four `#[cube]` families + comptime `bits` + comptime `one_hot`.
- **A literal `32`/`64`/`& 31`/`tiled_partition<32>`/`<16>` in any stride/slice offset:** violates D-09. Upstream's `tiled_partition<16>`, `(threadIdx.x & 6)`, `512 * (threadIdx.x >> 5)`, `1024 * (threadIdx.x / 32)` are wave32/warp-tile-specific and MUST be re-expressed wave-agnostically (`CUBE_COUNT`/`CUBE_DIM`/`PLANE_DIM`/`UNIT_POS`). The shipped 7.3 MVP avoids them entirely by using a grid-stride per-pair loop + direct global atomics — adopt the same.
- **Reading the histogram handle back on the hand-off path:** violates D-7.4-03 / the no-readback seam (and the HIP empty-buffer read-back hazard). Read back ONCE in `_test.rs` only.
- **Modifying `cb-compute/src/pairwise_scoring.rs` to serve the GPU path:** violates D-7.4-08. The host reference lives in the `cb-backend` `_test.rs`.
- **Hard-coding a float type or signing off the GPU-06 epsilon:** violates AGENTS.md generics-float and D-7.4-05. Kernels are `<F: Float>`; the oracle REPORTS, 7.6 signs off.
- **f64 `Atomic` channel on wgpu:** WGSL has no f64 atomics — the shipped 7.3 produced an all-zero histogram on wgpu until the channel was backend-dispatched to f32. Reuse the shipped backend-dispatched-channel machinery verbatim (Pitfall 1).

## Common Pitfalls

### Pitfall 1: f64 in-kernel atomic-add is absent on wgpu and not advertised on HIP/gfx1100
**What goes wrong:** A blind `Atomic<f64>::fetch_add` produces an all-zero histogram on wgpu (WGSL has no f64 atomics) — the shipped 7.3 Plan 01 hit exactly this. On gfx1100, `device_supports_f64_atomic_add` returns **false** (HIP does not ADVERTISE f64 atomic-add) yet the in-kernel `Atomic<f64>::fetch_add` **runs correctly** (HIP emulates it).
**Why it happens:** AMD RDNA3 / WGSL atomic-float support differs by type and backend.
**How to avoid:** Reuse the shipped 7.3 machinery verbatim: dispatch an **f32** binSums channel on the wgpu arm and the **f64** channel on rocm/cuda/cpu, with `read_pair_binsums_f64` upcasting the f32 channel so the 7.5 seam + oracle see a uniform f64 layout; cfg-gate the oracle bound (f32 ~1e-3 on wgpu, f64 ~1e-9 elsewhere). Treat the `device_supports_f64_atomic_add` report as informational about the ADVERTISED capability, not the actual execution path (the f64 kernel runs the live atomic on rocm regardless).
**Warning signs:** `rel_div = 1.0` (off by the full value) on wgpu; a self-oracle that passes on rocm but errors on wgpu.
`[VERIFIED: 07.3-01-SUMMARY.md Deviation 2 + "Notable observation (for 7.6 epsilon sign-off)"]`

### Pitfall 2: the pairwise binSums layout is 4-channel — not 2-channel like 7.3
**What goes wrong:** If the device histogram handle uses 7.3's 2-channel `(feature*n_bins+bin)*2+channel` layout, the 7.5 pairwise scorer (`split_pairwise.cuh`, which consumes 4 channels per fold) cannot read it; and the per-pair channel writes overrun/alias.
**Why it happens:** Every pairwise family's final merge is `histogram[4 * FirstFoldIndex + 4 * fold + histId]` with `histId ∈ {0,1,2,3}` — confirmed across binary (`pairwise_hist_binary.cu:179`), non-binary (`pairwise_hist_one_byte_5bit.cuh:355`), half-byte (`pairwise_hist_half_byte.cu:265`), and 8-bit (`atomicAdd(Histogram + offset…)` with the same 4-way decomposition). `histLineSize` is passed as a count and the kernels multiply by 4 (`histLineSize * 4ULL`).
**How to avoid:** In Plan A, FREEZE `PAIR_HIST_CHANNELS = 4` and `histLineSize = 4 * totalBinFeatures`; single-tree collapse index `(feature * n_bins + bin) * 4 + histId`, buffer length `n_features * n_bins * 4`. Document the exact formula in `launch_pairwise_hist_handle` (the shipped 7.3 `gpu_runtime.rs:928` FROZEN-layout doc-comment is the template). Keep it byte-stable across Plans B/C/D/E. The `fullPass=false` multi-part offset (`((blockIdx.z+1)<<depth)|blockIdx.y`) is a 7.5 forward dependency, documented and NOT filled here.
**Warning signs:** The 7.5 planner needing to re-pack the histogram; the oracle indexing not matching the kernel write index; buffer length `…* 2` anywhere.
`[VERIFIED: pairwise_hist_binary.cu:179, pairwise_hist_one_byte_5bit.cuh:355, pairwise_hist_half_byte.cu:265 — all "4 * bfStart + 4 * fold + histId"]`

### Pitfall 3: the cindex stride is over OBJECTS, but the loop is over PAIRS
**What goes wrong:** A naive port indexes `cindex[feature * n_pairs + oi]` (using the pair count as the stride). The pair indices `oi`/`oj` are OBJECT ids; `cindex` is laid out per object (`cindex[feature * n_objects + obj]`). Using `n_pairs` as the feature-major stride reads the wrong cell.
**Why it happens:** 7.3's loop variable `i` is over objects and the stride is `n = indices.len()`; here the loop is over pairs (`p < n_pairs`) but the cindex stride must stay `n_objects`. The kernel therefore needs `n_objects` as a separate input from `n_pairs` (unlike 7.3 where they coincide).
**How to avoid:** Pass `n_objects` explicitly (it is `cindex.len() / n_features`, validated host-side via the checked-product guard) and index `cindex[feature * n_objects + oi]`. Validate `pair_i[p] < n_objects` and `pair_j[p] < n_objects` host-side (the value-range-guard pattern from the shipped `launch_pointwise_hist2_into:1057`).
**Warning signs:** Off-by-stride histogram values that scale with the pair/object ratio; oracle divergence that grows with `n_pairs`.

### Pitfall 4: upstream's pair-warp bin math is wave32/tile-specific — must be re-expressed wave-agnostically (D-09)
**What goes wrong:** Directly porting `tiled_partition<32>` / `tiled_partition<16>` / `(threadIdx.x & 6)` / `512 * (threadIdx.x >> 5)` / `1024 * (threadIdx.x / 32)` / `groupTile.thread_rank() < 8` bakes in wave32 and the 16/32-lane tile structure, breaking D-09.
**Why it happens:** The CUDA pairwise kernels distribute the 4-channel write across a 16- or 32-lane tile to avoid bank conflicts in the shared-mem histogram — a perf structure, not the accumulation semantics.
**How to avoid:** The MVP (per the shipped 7.3 direct-global-atomic approach) sidesteps the tile entirely: a grid-stride per-pair loop computes `histId` and does a direct `Atomic<F>::fetch_add` into the global 4-channel cell — no tile, no warp literal, structurally faithful to "atomic merge into the global histogram." The tile-distributed shared-mem pre-reduce is the documented perf follow-up.
**Warning signs:** A `32`/`64`/`16`/`& 31`/`& 6`/`tiled_partition`/`thread_rank` literal in a stride or slice offset; a kernel that only passes at exactly 32 threads.

### Pitfall 5: never `read_one` an empty (or hand-off) device buffer on the production path
**What goes wrong:** `read_one` on a zero-length HIP buffer dereferences a null/zero-length pointer; any read-back on the hand-off path violates D-7.4-03.
**Why it happens:** Documented in the shipped 7.2/7.3 edge-case tests and the MEMORY note ("Handle bound to its client, never read 0-len").
**How to avoid:** Empty input (`n_pairs == 0` or `n_features == 0` or `n_bins == 0`) short-circuits to `client.empty(0)` with NO launch and NO read-back (the shipped `launch_pointwise_hist2_into:1078`). The self-oracle reads back ONCE, only for non-empty fixtures, through the SAME client that allocated the handle.
**Warning signs:** A HIP panic on the empty/n_pairs=0 fixture; a read-back through a freshly-constructed client.

### Pitfall 6: the cubecl-hip-sys 7.2 vs HIP 7.1 version skew
**What goes wrong:** The lock carries HIP-7.2 sys headers vs the box's HIP 7.1.1 runtime — a potential symbol/ABI mismatch.
**Why it happens:** Flagged in 7.1/7.2/7.3; not yet observed to break (all three ran on rocm in-env). The sys `build.rs` auto-detects the HIP patch — do NOT manually pin `cubecl-hip-sys/hip_xxx` (it panics; `cb-backend/Cargo.toml` note).
**How to avoid:** Proceed on the current pin. If a runtime HIP symbol error appears, the registry cache has `cubecl-hip-sys-7.1.5280200` as a fallback pin. Document, don't pre-emptively re-pin.
**Warning signs:** A HIP runtime "undefined symbol"/version error that did NOT occur in 7.1/7.2/7.3.

## Open Questions

1. **Device encoding of `pairs` (`uint2`-like vs two parallel `u32` arrays).**
   - What we know: upstream is `uint2* pairs`; CubeCL has no first-class `uint2` device type in the project's usage; the shipped kernels read `&Array<u32>` for `cindex`/`indices`.
   - What's unclear: nothing blocking — both encodings are correct.
   - Recommendation: two parallel `u32` arrays (`pair_i`, `pair_j`) — idiomatic, no device struct, index math stays `usize`. Decide + freeze in Plan A (D-7.4-03 discretion).

2. **The exact fold/bin the 4-channel write attaches to + the clean `Compare→histId` reduction.**
   - What we know: the final merge is `4 * FirstFoldIndex + 4 * fold + histId`; the `AddPair` decomposes bins via `tmp = (Compare? 0:8) + f; offset = tmp + (bin<<5) + flag` (non-binary) / the 2×2 OR (binary).
   - What's unclear: the cleanest per-pair `(bin, histId)` expression equivalent to the warp-tile-distributed `AddPair` (the tile is a perf structure, not semantics).
   - Recommendation: in Plan A, reduce `AddPair` to its accumulation SEMANTICS (each pair contributes `w` to exactly the 4 cells its `(bin1, bin2, Compare)` selects), write directly with `Atomic::fetch_add`, and let the self-oracle (a wrong mapping diverges immediately) validate it. This is the genuinely-new logic (A6).

3. **Raw 4-channel fill vs the `BuildBinaryFeatureHistograms` (w00,w01,w10,w11) transform.**
   - What we know: the FILL writes raw `hist0..hist3`; `BuildBinaryFeatureHistograms` (pairwise_hist.cu:14-83) POST-transforms them into `(w00,w01,w10,w11)` using `partWeight` — that runs AFTER the fill.
   - What's unclear: whether 7.5 wants the raw fill or the transformed histogram from the 7.4 handle.
   - Recommendation: 7.4 emits the RAW 4-channel fill (D-7.4-06 defers `BuildBinaryFeatureHistograms` to 7.5). Document this in the seam doc; flag the transform as a 7.5 forward dependency. (A1.)

4. **`parallelStreamCount` / `fullPass` / multi-part partition depth for the single-tree fill.**
   - What we know: upstream uses `numBlocks.y/.z` over parts + `parallelStreams` for occupancy; the `fullPass=false` path computes a multi-part histogram offset `((blockIdx.z+1)<<depth)|blockIdx.y`.
   - What's unclear: how much partition depth the single-tree fill needs (vs the multi-part grow-loop = 7.5).
   - Recommendation: single-tree fill needs only `partCount = 1`, full-pass (the shipped 7.3 A2 precedent). Defer the `fullPass=false` multi-part offset to 7.5; document it as a forward dependency. `parallelStreamCount` is a host occupancy knob, not a semantic — the MVP launches one grid (D-7.4-03 discretion).

### Known Threat Patterns for {GPU compute kernel, Rust + cubecl}
| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Out-of-bounds device read/write (malformed pair index or bin) | Tampering / DoS (UB) | Host-side value-range guards (`pair_i/j < n_objects`, `cindex < n_bins`) + length/overflow guards BEFORE launch (the shipped CR-01/WR-04 block); device `p < n_pairs` grid-stride guard. |
