# Phase 7.4: Code Review Report

**Reviewed:** 2026-06-20
**Depth:** standard
**Files Reviewed:** 3
**Status:** issues_found

## Summary

Reviewed the Phase 7.4 GPU pairwise-histogram kernel family added since `8e5996d`:
the four `#[cube(launch)]` kernels (`pairwise_hist_nonbinary_kernel`,
`pairwise_hist_8bit_atomics_kernel`, `pairwise_hist_half_byte_kernel`,
`pairwise_hist_binary_kernel`) in `kernels.rs`, the eight `launch_pairwise_hist*`
host seams in `gpu_runtime.rs`, and the self-oracle harness in
`kernels/pairwise_hist.rs`.

Overall the additive surface is disciplined: the `(feature * n_bins + bin) * 4 +
histId` index math is consistently `* 4` (Pitfall 2 honored), the host-side guard
block validates lengths, value ranges, AND the `n_features*n_objects` /
`n_features*n_bins*4` products via `checked_mul` before any device index is formed,
the empty short-circuit avoids launching/reading an empty buffer, and the production
launch helpers contain no `unwrap`/`expect`/`panic`/indexing — the no-panic rule is
honored. No out-of-bounds histogram write or integer-overflow path survived the
host-side guards on any of the four families. No BLOCKER-class defect found.

The findings below are quality/robustness issues: a structurally tautological
one-hot self-oracle that cannot catch a wrong one-hot mapping, a `num_cubes as u32`
truncation shared with the rest of the file, a `1 << bits` shift that is unguarded
against large `bits`, and substantial doc/guard-block duplication across the four
launch arms.

## Warnings

### WR-01: One-hot self-oracle is tautological — cannot catch a wrong one-hot channel mapping

**File:** `crates/cb-backend/src/kernels/pairwise_hist.rs:122-151`, `crates/cb-backend/src/kernels.rs` (`pairwise_hist_nonbinary_kernel` / `pairwise_hist_8bit_atomics_kernel` one-hot branches)
**Issue:** The host reference (`add_pair_contrib`, one-hot branch) and the device
kernels implement the IDENTICAL one-hot formula: `is_ge = (b1 != b2)`, write `b1` at
`2*is_ge` twice and `b2` at `2*is_ge+1` twice. The `one_hot` test
(`pairwise_hist.rs:702-880`) compares device-vs-host and additionally asserts
one-hot ≠ non-one-hot. But because both sides share the formula, the one-hot
device-vs-host comparison passes for ANY one-hot mapping the kernel happens to use —
including a wrong one. The only real signal is "one-hot differs from non-one-hot,"
which a trivially-different-but-still-wrong mapping also satisfies. The module header
(line 134) and kernel comments openly note "Plan E refines the exact one-hot fold,"
i.e. the current one-hot semantics are admittedly provisional and the oracle does not
pin them to any external/upstream truth. This is the same self-oracle class as the
non-one-hot path, but for non-one-hot the host reference is a genuinely independent
ordered re-derivation; for one-hot the two implementations are copies of one formula,
so the parity assertion has near-zero defect-detection power.
**Fix:** Either (a) derive the one-hot host reference from an independent source
(upstream `split_properties_helpers.cuh:261` `Compare = (bin1 == bin2)` expanded by
hand, not the same `is_ge`/`push_cell` shortcut the kernel uses), or (b) explicitly
mark the one-hot path as NOT-yet-oracle-validated in the test and gate the SC claim on
Plan E, so a reader does not mistake the passing `one_hot` test for proof of one-hot
correctness. At minimum, add an assertion against a hand-computed expected histogram
for one small fixed one-hot fixture (3-4 pairs) so the channel mapping is pinned to a
literal, not to its own twin.

### WR-02: `1 << bits` is unguarded against large `bits` before the family check

**File:** `crates/cb-backend/src/kernels.rs` (`pairwise_hist_nonbinary_kernel`, `let n_bins = comptime!((1u32 << bits) as usize);`); `crates/cb-backend/src/gpu_runtime.rs:225` (`n_bins != (1usize << bits)`)
**Issue:** In `launch_pairwise_hist_into` the family validation `bits < 5 || bits > 7
|| n_bins != (1usize << bits)` runs AFTER the value-range guards, and it evaluates
`1usize << bits` directly. On a 64-bit host `bits >= 64` makes `1usize << bits`
undefined-behavior-adjacent (Rust: shift overflow panics in debug, is masked/wraps in
release) — a malformed `bits` from a caller can yield a wrong comparison value in
release rather than a clean rejection. The kernel side `comptime!((1u32 << bits))`
is comptime so a bad `bits` is a JIT/build error there, but the host guard is the
production input-validation chokepoint and should reject out-of-range `bits` with a
typed error before performing any shift. The existing `bits < 5 || bits > 7` check is
short-circuited correctly today (it is `||`-first), so this is latent rather than
live — but the ordering depends on `||` evaluation and the `n_bins != (1 << bits)`
term is one edit away from being reachable with a huge `bits`.
**Fix:** Hoist the `bits` range check to its own statement before computing any shift:
```rust
if !(5..=7).contains(&bits) {
    return Err(CbError::Degenerate(format!(
        "pairwise_hist non-binary fill expects bits in {{5,6,7}}, got bits={bits}"
    )));
}
if n_bins != (1usize << bits) {
    return Err(CbError::Degenerate(format!(
        "n_bins ({n_bins}) must equal 1 << bits ({})", 1usize << bits
    )));
}
```
so the shift only runs once `bits` is known to be in `5..=7`.

### WR-03: `num_cubes as u32` truncation can silently under-cover the grid for very large `n_pairs`

**File:** `crates/cb-backend/src/gpu_runtime.rs:1537-1538, 1788-1789, 2003-2004, 2213-2214`
**Issue:** Each pairwise launch arm computes `num_cubes = n_pairs.div_ceil(CUBE_DIM)`
then `CubeCount::Static(num_cubes as u32, 1, 1)`. For `n_pairs` large enough that
`num_cubes > u32::MAX` (≈ 137 billion pairs at CUBE_DIM=32) the `as u32` truncates,
launching far fewer cubes than intended. The kernel's grid-stride loop
(`stride = CUBE_COUNT * CUBE_DIM`, `while p < n_pairs`) still covers every pair
correctly because the stride is derived from the actual launched topology, so this is
a performance/scaling cliff, not a correctness defect — and the truncation pattern is
shared verbatim with every pre-existing launch helper in the file (lines 175, 246,
…), so it is not specific to 7.4. Flagged because the pairwise path is the first to
loop over `n_pairs` (which is O(n_objects²) for full pairwise ranking and thus the
most likely arg to grow large), making the truncation more reachable here than in the
per-object pointwise paths.
**Fix:** Cap or reject defensively, e.g. `let num_cubes = n_pairs.div_ceil(CUBE_DIM).max(1).min(u32::MAX as usize) as u32;`
(the grid-stride loop already absorbs the cap), or surface a `CbError::OutOfRange`
when `num_cubes > u32::MAX as usize`. Since this is a shared pattern, consider a small
helper `fn cube_count_for(n: usize) -> CubeCount` reused by all launch arms.

### WR-04: Four near-identical guard blocks duplicated across the pairwise launch arms (drift risk)

**File:** `crates/cb-backend/src/gpu_runtime.rs:137-296` (`launch_pairwise_hist_into`), `400-541` (`8bit_into`), `615-754` (`half_byte_into`), `826-964` (`binary_into`)
**Issue:** The four `*_into` functions repeat the same ~60-line block almost verbatim:
the `pair_j`/`pair_weight` length checks, the `checked_mul` overflow guards, the
`cindex` length check, the three value-range `find(...)` guards, the empty
short-circuit, the launch geometry, the three input-handle `create`s, and the
backend-split `#[cfg(feature = "wgpu")]` dispatch. The only real differences are the
comptime `n_bins`, the family-specific `n_bins != X` degenerate check, the `one_hot`
arg presence, and the dispatched kernel symbol. This is code duplication of the most
hazardous kind: a future fix to a guard (e.g. WR-02 above, or a tightened overflow
check) must be applied four times, and the binary/half-byte arms have already silently
diverged in one place — their value-range guard checks `cindex < n_bins` while the
kernel additionally masks (`& 15` / `& 1`); the comment claims this keeps them in
"lock-step" but it makes the mask dead code, so a future relaxation of the host guard
would change behavior in a way the masked kernel hides. The IN-02 note in the docs
claims the geometry lives "in ONE place," which is true per-family but not across
families.
**Fix:** Extract the shared guard+geometry into one helper that takes the family-specific
bits as parameters, e.g.
`fn prepare_pairwise_launch(client, pair_i, pair_j, pair_weight, cindex, n_objects, n_bins, n_features) -> CbResult<PairwiseLaunchPlan>`
returning the validated handles + geometry, and a small per-family `match`/macro for
the kernel-symbol dispatch (the existing `launch_hist2_family!` macro precedent at
`gpu_runtime.rs:1234` shows this pattern is already used for the pointwise family).
This collapses the four guard copies to one and makes WR-02's fix a one-line change.

## Info

### IN-01: Binary/half-byte nibble/bit mask is dead code given the host-side value guard

**File:** `crates/cb-backend/src/kernels.rs` (`pairwise_hist_half_byte_kernel` `& nibble_mask`, `pairwise_hist_binary_kernel` `& bit_mask`)
**Issue:** The host guard (`gpu_runtime.rs:678`, `889`) rejects any `cindex` value
`>= n_bins` (16 or 2), so by the time the kernel runs, every bin is already `< 16` /
`< 2` and the `& 15` / `& 1` mask is a no-op. The mask faithfully mirrors upstream's
nibble/bit extraction, so keeping it is defensible for structural parity, but it is
currently unreachable-effect code and a reader may assume it is load-bearing
protection against out-of-range bins (it is not — the host guard is). 
**Fix:** Add a one-line comment that the mask is upstream-structural parity, not a
bounds guard (the bounds guard is host-side), or note it in the kernel doc so the
two are not conflated as redundant safety.

### IN-02: `one_hot` swap sanity check inspects only feature 0 of `cindex`

**File:** `crates/cb-backend/src/kernels/pairwise_hist.rs:825-829`
**Issue:** The `has_unequal` precondition reads `cindex[oi] != cindex[oj]` (raw,
no feature offset), which inspects feature 0 only. This is correct for the intent
("at least one bin1 != bin2 pair for feature 0", per the comment) because feature 0's
slice starts at offset 0, but it relies on the implicit `feature == 0` offset rather
than the explicit `cindex[feature * n_objects + obj]` stride used everywhere else in
the file. A reader cross-checking against `host_reference_pairwise_hist`'s stride
math could misread this as a stride bug.
**Fix:** Make the offset explicit for clarity: `cindex[0 * n_objects + oi] != cindex[0 * n_objects + oj]`,
or loop over a chosen feature with the same `feature * n_objects + obj` form the
reference uses.

### IN-03: Doc comment claims one-hot mirrors upstream `Compare = (bin1 == bin2)` but implements a coarser fold

**File:** `crates/cb-backend/src/kernels/pairwise_hist.rs:119-139`; `crates/cb-backend/src/kernels.rs` (one-hot branch comments)
**Issue:** The header text says one-hot "replaces the predicate with `bin1 == bin2`"
and cites `split_properties_helpers.cuh:261`, but the actual implementation collapses
both flag passes onto a single slot keyed by `is_ge = (b1 != b2)` and writes each cell
twice — which is the team's own provisional fold, not a transcription of the upstream
`Compare`. The doc itself later admits "Plan E refines the exact one-hot fold," so the
two statements are in mild tension and a reader may over-trust the upstream-citation
framing.
**Fix:** Soften the upstream citation to "approximates / placeholder for the upstream
one-hot `Compare`, refined in Plan E" so the provisional status is unambiguous at the
point of the formula, consistent with WR-01.
