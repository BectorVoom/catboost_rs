## Phase Boundary

The fourth additive GPU sub-phase of the Phase 7 umbrella, and the **pairwise
sibling of 7.3**. It ports the **pairwise (object-pair) histogram kernel
family** — `catboost/cuda/methods/kernel/pairwise_hist*` — to run
**device-resident** on the GPU runtimes, generic over `R: Runtime`/`F: Float`,
accumulating with CUDA-style **in-kernel atomics** (umbrella D-03), completing
the GPU histogram surface alongside the pointwise family of 7.3.

Where 7.3's `hist2` accumulates per **single object** into a 2-channel
(Σ der1, Σ weight) histogram, the pairwise family accumulates per **object
pair** `(i, j)` (upstream's `uint2* pairs`) into the pairwise histogram lines
that the pairwise/ranking split scorer (`split_pairwise.cuh`) reduces — the GPU
analog of the CPU pairwise path already in
`cb-compute/src/pairwise_scoring.rs`. This is the histogram surface for the
**pairwise/ranking losses** (PairLogitPairwise, YetiRankPairwise) landed in
Phase 6.3.

The umbrella `07-CONTEXT.md` (D-01…D-09), `07.1-CONTEXT.md` (D-7.1-01…10),
`07.2-CONTEXT.md` (D-7.2-01…07), and **`07.3-CONTEXT.md` (D-7.3-01…07)** are the
**binding shared context**: full **structural-parity** fidelity bar (D-01), the
**in-kernel atomic** determinism contract with f64 finalization (D-03), the
**empirical GPU-06 epsilon signed off only in 7.6**, **`rocm`-only validation
never in GitHub Actions** (D-06), the **wave-size-agnostic** policy / no
warp-size constants (D-09), and the **device-resident histogram fill seam +
self-oracle harness pattern** that 7.3 established (D-7.3-03/04/05). Those are
**already locked and NOT re-decided here.** This phase resolves only the
HOW-level choices specific to the **pairwise** histogram fill and its
self-oracle.

**In scope:** GPU-01 (the pairwise-histogram slice only) — the `pairwise_hist`
family ported at structural parity: the **general one-byte non-binary path**
(5/6/7-bit), the **8-bit-atomics path** (a structurally DISTINCT family —
upstream `pairwise_hist_one_byte_8bit_atomics.cu`), the **half-byte** path
(`pairwise_hist_half_byte.cu`), the **binary** path
(`pairwise_hist_binary.cu`), and the **one-hot variants** of the 5/6/7/8-bit
paths (`*_one_hot.cu`) — all generic over `R`/`F`, device-resident, self-oracled
against the Rust CPU pairwise baseline. Consumes a device-resident `pairs` +
per-pair `weight` (+ `cindex` + partition) input; produces a device-resident
pairwise-histogram (`binSums`-equivalent) buffer handle.

**Out of scope (owned by later sub-phases):** the post-fill
`UpdatePairwiseHistograms` / `BuildBinaryFeatureHistograms` /
`ScanPairwiseHistograms` scan-update steps and the **pairwise split scorer**
(`split_pairwise.cuh`) integration into the full device-resident tree-grow loop
→ **7.5**; the **empirical GPU-06 epsilon sign-off + `rocm` oracle-gate
lock-in** → **7.6**. The pointwise `hist2` family is 7.3 (done). 7.4 delivers
the pairwise FILL + its device handle, nothing downstream of it.

## Implementation Decisions

All decisions below were auto-resolved to the recommended default in `--auto`
mode. They are HOW-level choices within the boundary the umbrella + 7.1 + 7.2 +

### Variant coverage & in-phase sequencing (SC-1)
- **D-7.4-01:** The **full** `pairwise_hist` family is in scope (SC-1 requires
  the general path plus 5/6/7-bit, half-byte, binary, 8-bit-atomics, and the
  one-hot variants all run on `rocm`). Sequence them **narrowest-first across
  plans**, mirroring the 7.1/7.2/7.3 "resist porting the whole set up front"
  pattern: land the general **one-byte non-binary** path (5/6/7-bit) + the
  pairwise device seam first, then the **8-bit-atomics** distinct family, then
  **half-byte**, then **binary**, then layer the **one-hot** specialization onto
  the bit-width paths it applies to. Every variant must land in THIS phase — the
  sequencing is plan decomposition, not a scope cut.

### Kernel-family decomposition + bit-width / one-hot specialization (research-flagged)
- **D-7.4-02:** Structurally mirror upstream's **distinct kernel families**
  rather than one mega-kernel (D-01), matching upstream's own file split:
  (a) **one-byte non-binary** 5/6/7-bit sharing a template
  (`pairwise_hist_one_byte_{5,6,7}bit.cuh`) with the **bit-count carried as a
  `#[comptime]` parameter** (histogram line size / shared-mem blocking / unroll
  — NOT a runtime branch); (b) **8-bit-atomics** as a **structurally separate
  family** (`pairwise_hist_one_byte_8bit_atomics.cuh`) — at 8 bits the histogram
  line does not fit the per-block shared-memory budget, so upstream accumulates
  via **true global atomics**, a different memory structure that must be mirrored
  distinctly (D-03); (c) **half-byte** (`pairwise_hist_half_byte.cu`); (d)
  **binary** (`pairwise_hist_binary.cu`). The **one-hot** variant is a
  **`#[comptime]` overlay** on the 5/6/7/8-bit paths (upstream's `*_one_hot.cu`
  instantiations), not a separate hand-written kernel per width. **The exact
  CubeCL comptime-vs-monomorphization expression is the research-flag item** —
  leave the precise mechanism to RESEARCH.md, subject to D-09 (no warp
  constants) and AGENTS.md generics-float. Reuse the 7.3 comptime-bits
  precedent in `pointwise_hist2_nonbinary_kernel`.

### Pairwise input/output seam (SC-1 / hand-off to 7.5)
- **D-7.4-03:** The pairwise fill seam is **distinct from the 7.2/7.3 pointwise
  der1/der2 seam.** It consumes a **device-resident `pairs` buffer** (upstream
  `uint2* pairs` — object-pair indices) + a **device-resident per-pair `weight`
  buffer** (the pairwise weights produced by the pairwise/ranking loss
  derivative path) + the **quantized-bin `cindex`** + **partition** layout,
  mirroring upstream's `(features, featureCpu, compressedIndex, pairs, pairCount,
  weight, partition, partCount, histLineSize, fullPass, histogram,
  parallelStreamCount)` signature. Output is a **device-resident pairwise
  histogram buffer (handle)** — upstream's `binSums` — consumable by the 7.5
  pairwise scorer + grow loop with **no host round-trip**. Define this seam
  (pairs + pair-weight + bins in → histogram handle out) now so 7.5 plugs in
  without a round-trip. The exact device encoding of `pairs` (single `uint2`-like
  buffer vs two parallel `u32` index arrays) is Claude's discretion, subject to
  D-09 / generics-float.

### Accumulation strategy — shared-memory blocking + in-kernel atomics (D-03)
- **D-7.4-04:** Mirror upstream's pairwise accumulation: per-block
  **shared-memory histogram** accumulation over the block's pair slice for the
  sub-8-bit paths, then an **atomic merge** into the global histogram; the
  **8-bit-atomics and one-hot paths accumulate via true global atomics**
  directly (their lines exceed the shared-memory budget). Both honor D-03's
  **in-kernel-atomic determinism contract with f64 finalization**, reuse the 7.1
  atomic-finalize building block (`block_reduce_atomic_kernel` /
  `launch_block_reduce_atomic_f64`) and the wave-agnostic plane/shared-mem
  fallback (D-09 — shared-mem size a `#[comptime]` const, stride from
  `CUBE_DIM_X`/`PLANE_DIM`, never a literal 32/64). This is the **real on-device
  fill**, distinct from the scatter-only `histogram_scatter_kernel` (D-02
  baseline). D-03/D-09 are **locked by the umbrella — restated here, not
  reopened.**

### CPU self-oracle baseline & fixtures (SC-2)
- **D-7.4-05:** Self-oracle the device-resident pairwise histogram against a
  **host reference pairwise histogram** built with the existing **ordered**
  `cb-core::sum_f64` reduction (same gather-then-`sum_f64` shape 7.3 used,
  generalized to per-pair accumulation). The existing CPU pairwise path —
  `cb-compute/src/pairwise_scoring.rs` (`calculate_pairwise_score`,
  `reference_pairwise_score` in its test) — is the **parity reference** for the
  pairwise structure (SC-2: "match the Rust CPU pairwise path … including a
  pairwise/ranking loss"). Fixtures: random f32 + f64, non-trivial per-pair
  weights, edge cases (empty pair set, single pair, size not a multiple of the
  block/cube, large pair count), exercised **across the covered bit-widths +
  one-hot** (binary, half-byte, 5/6/7-bit, 8-bit-atomics, one-hot), **including a
  pairwise/ranking-loss-derived pair/weight fixture** (PairLogitPairwise /
  YetiRankPairwise). The harness **REPORTS** observed max absolute + relative
  divergence (run-to-run atomic variance absorbed) as the current tolerance; it
  does **NOT** hard-code or sign off the GPU-06 epsilon — that is 7.6's job
  (D-7.2-06 / D-7.3-04 pattern). Tests live in dedicated `_test.rs` /
  test-module files (source/test separation), run on `rocm` in-env. Clone the
  `kernels/pointwise_hist.rs` oracle harness into `kernels/pairwise_hist.rs`.

### Scan/update deferral to 7.5 (boundary of the FILL)
- **D-7.4-06:** The post-fill `UpdatePairwiseHistograms`,
  `BuildBinaryFeatureHistograms`, and `ScanPairwiseHistograms` steps — and the
  pairwise **split scorer** (`split_pairwise.cuh`) — are **7.5's** job (mirrors
  7.3 deferring `ScanPointwiseHistograms`/`UpdatePointwiseHistograms`). 7.4
  delivers the **FILL + the device-resident histogram handle** and defines the
  seam; it does NOT integrate scan/update or scoring into the grow loop.

### Verification gating (carried from D-7.3-06, restated)
- **D-7.4-07:** Verification is **per-crate**, honoring the standing
  disk-pressure constraint (root ~100% full; full-workspace test links fail):

  gate (no NVIDIA hardware — D-07). `rocm` GPU tests run **locally/manually,
  never in GitHub Actions** (umbrella D-06).

### Boundary preservation (carried from D-7.3-07, restated)
- **D-7.4-08:** All cubecl coupling stays **inside `cb-backend`** (D-7.1-04 /
  D-7.2-05 / D-7.3-07). Do **not** widen the frozen `cb-compute`
  `Runtime`/`Float` traits; prefer backend-internal launch helpers. `cb-compute`
  stays cubecl-free; `cb-core`/`cb-model` stay byte-unchanged from Phase 3–6 form

## Specific Ideas

- Pairwise histograms accumulate over **object pairs** `(i, j)` (upstream
  `uint2* pairs`), not single objects — the fundamental difference from 7.3's
  `hist2`. The pairwise weight/derivative structure comes from the
  pairwise/ranking loss path (Phase 6.3: PairLogitPairwise, YetiRankPairwise).
- Upstream's **8-bit-atomics** path is structurally distinct precisely because an
  8-bit pairwise histogram line does not fit the shared-memory budget — it uses
  **global atomics** directly. This must be mirrored as a separate family
  (D-7.4-02), not folded into the sub-8-bit shared-mem template.
- Upstream selects the variant by **feature bit-width** (binary → half-byte →
  5/6/7-bit non-binary → 8-bit-atomics), with **one-hot** as an orthogonal
  overlay on the bit-width paths (`*_one_hot.cu`).
- The post-fill `Update/BuildBinaryFeature/Scan PairwiseHistograms` and the
  pairwise split scorer (`split_pairwise.cuh`) are **7.5** — 7.4 delivers the
  fill + its device handle.
</specifics>
