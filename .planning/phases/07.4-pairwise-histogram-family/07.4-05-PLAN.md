## Phase Goal

**As a** Rust developer embedding the GPU pairwise/ranking training path with one-hot categorical features, **I want to** fill a device-resident 4-channel pairwise histogram for the one-hot variants of the 5/6/7-bit and 8-bit features on rocm via a comptime predicate overlay, **so that** the full upstream variant set runs device-resident with verifiable parity against the CPU pairwise path.

## Artifacts this phase produces (new symbols — exclude from drift verification)

This plan adds: the `one_hot=true` dispatch path through the EXISTING `launch_pairwise_hist_handle` (5/6/7-bit) and `launch_pairwise_hist_8bit_handle` (8-bit) — plumbing the comptime flag to the host helper if Plan A/B did not yet expose it on the public signature; the `one_hot` test in `kernels/pairwise_hist.rs`. It REUSES the Plan A/B kernels (which already carry the `#[comptime] one_hot: bool` arg), the `host_reference_pairwise_hist` (which already threads `one_hot`), `make_pair_fixture`, `read_pair_binsums_f64`, and the backend-dispatched channel. No NEW kernel symbol is created — one-hot is a comptime overlay (D-7.4-02).
