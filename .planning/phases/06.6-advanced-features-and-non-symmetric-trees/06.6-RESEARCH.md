## Summary

Both priority gates are **resolved decisively** against the installed catboost 1.2.10 `.venv` and the vendored upstream source, and both change phase scope:

1. **Gate 1 (monotone × non-symmetric):** Upstream 1.2.10 **explicitly rejects** monotone constraints under every non-symmetric policy. Fitting `monotone_constraints=[1,0]` with `grow_policy=Lossguide` or `Depthwise` raises `monotonic_constraint_utils.h:42: "Monotone constraints are unsupported for non-symmetric trees yet"`. `grow_policy=Region` raises `greedy_tensor_search.cpp:1953(.dispatch ~1995): "GrowPolicy Region is unimplemented for CPU."` even WITHOUT monotone. **Therefore: monotone is OBLIVIOUS-ONLY this phase, and Region is OUT OF SCOPE entirely (no CPU ground truth exists).** D-6.6-07's "non-symmetric monotone variant" must be recorded as an escalated gap, not built. FEAT-06 narrows to **Lossguide + Depthwise**.

2. **Gate 2 (SAGE reachability):** `EFstrType.SageValues` exists in 1.2.10 AND `get_feature_importance(type='SageValues', data=pool)` returns a per-feature vector. SAGE is **deterministic across repeated calls AND reproducible for a fixed `random_seed`** (verified: same seed → bit-identical, different seed → differs). **Fallback (a) seed-match strict is achievable purely through the Python API** — no C++ instrumentation required. SAGE stays IN scope.

The non-symmetric engine is the load-bearing risk. Upstream uses ONE shared leaf-wise scoring core (`leafwise_scoring.cpp`) and two policy growers (`GreedyTensorSearchDepthwise`, `GreedyTensorSearchLossguide`) dispatched in `GreedyTensorSearch`. The trained structure serializes to a flat node array (`TreeSplits` per-node + `NonSymmetricStepNodes` {LeftSubtreeDiff, RightSubtreeDiff} + `NonSymmetricNodeIdToLeafId`), and the apply path is a simple pointer-walk: start at `TreeStartOffsets[t]`, `diff = (val>=splitIdx)?RightSubtreeDiff:LeftSubtreeDiff`, `index += diff`, stop at `diff==0`, leaf = `NonSymmetricNodeIdToLeafId[index]`. The FlatBuffers bindings already exist in Rust (`TNonSymmetricTreeStepNode`, parsed-then-discarded). **Critical serialization finding:** non-symmetric models export JSON under a DIFFERENT top-level key (`"trees"`, recursive nested-node `{split,left,right}` / `{value,weight}`) vs oblivious (`"oblivious_trees"`, flat arrays) — the oracle harness `model_json` parser needs a new code path. `.cbm` round-trips bit-exact (verified max-abs-diff 0.0).

**Primary recommendation:** Build in the locked order. Wave block A = monotone-oblivious + penalties (Python-oracle, both leaf-projection and the two penalty insertion points transcribed verbatim). Wave block B = the non-symmetric engine gate: a `TreeVariant::NonSymmetric` parallel to oblivious, the two leaf-wise growers behind one policy-parameterized core, the flat-node serialization (cbm + json), and the pointer-walk apply — all gated on tree kind so the oblivious oracles stay byte-identical. Wave block C = generalize fstr/SHAP recursion to non-symmetric + LossFunctionChange + ShapInteractionValues + PredictionDiff + SAGE (seed-match via Python). Wave block D = recursive feature selection (LAST). Record monotone-non-symmetric and Region as escalated gaps with the exact upstream guard citations as proof.

### Pattern 1: Non-symmetric apply (pointer-walk)
**What:** Flat node-array traversal; no depth/bit math.
**When to use:** `apply.rs` for `TreeVariant::NonSymmetric`.
**Example:**
```rust
// Source: catboost-master/catboost/libs/model/cpu/evaluator_impl.cpp:726-742 [CITED]
// (scalar CalcNonSymmetricTrees path)
let mut index = tree_start_offsets[tree_id];
loop {
    let step = step_nodes[index];                 // {left_subtree_diff, right_subtree_diff}
    let split = repacked_bins[index];             // feature + border (split_idx)
    let feature_value = bin_features[split.feature_index]; // ^= xor_mask if cat one-hot
    let diff = if feature_value >= split.split_idx { step.right_subtree_diff }
               else { step.left_subtree_diff };
    index = (index as i64 + diff as i64) as usize;
    if diff == 0 { break; }                        // TerminalMarker == 0
}
let leaf_value_idx = node_id_to_leaf_id[index];   // index into flat leaf_values
```

### Pattern 2: Depthwise grower (level-order)
**What:** Build the tree level by level; every leaf at the current level finds its best split; all splits at a level applied; up to `max_depth` levels. Children appended in `(left, right)` pairs; `id%2==0` even/odd handling drives the subtract-trick.
**When to use:** `grow_policy=Depthwise`.
**Reference:** `greedy_tensor_search.cpp:1509-1700` `GreedyTensorSearchDepthwise`. Key loop: `for curDepth in 0..MaxDepth { candidatesContexts=SelectFeaturesForScoring; for leaf in curLevelLeafs { CalculateStats; if gain<1e-9 continue; AddSplit; push left/right to nextLevelLeafs } }`.

### Pattern 3: Lossguide grower (best-gain priority queue)
**What:** A max-priority-queue of `(leaf, gain, best_candidate, stats)`. Pop the best-gain leaf, split it, push its two children's best candidates back. Stop when queue empty OR `leaf_count >= max_leaves`. `gain<1e-9` and `leafDepth[leaf] < max_depth` gate splits.
**When to use:** `grow_policy=Lossguide`.
**Reference:** `greedy_tensor_search.cpp:1806-1938` `GreedyTensorSearchLossguide`. Note the documented penalty caveat (lines 1875-1882): queued candidates are NOT recalculated when a feature first becomes used — upstream accepts this imprecision "for performance and code simplicity"; the Rust port MUST replicate it (do not recompute) to stay bit-exact.

### Pattern 4: Monotone isotonic leaf projection (oblivious only)
**What:** After computing leaf deltas, for each monotonic linear order run Pool-Adjacent-Violators (PAVA) isotonic regression over `currLeafValues + deltas`, weighted by leaf weights (`SumWeights+λ` for Gradient, `-SumDer2+λ` for Newton).
**When to use:** `cb-compute/leaf.rs`, oblivious trees with non-empty `monotone_constraints`.

### Anti-Patterns to Avoid
- **Refactoring oblivious apply into a general node-graph traversal:** D-6.6-05 forbids it. The oblivious flat-array path is locked; non-symmetric is a SEPARATE branch.
- **Recomputing Lossguide queue penalties when a feature first becomes used:** upstream deliberately does NOT (lines 1875-1882). Replicating the "bug" is required for parity.
- **Building monotone for non-symmetric trees:** there is NO upstream ground truth (it throws). Fabricating a fixture is prohibited (D-6.6-07).
- **Implementing Region:** CPU-unimplemented upstream. No ground truth.
- **Parsing non-symmetric JSON via the oblivious `"oblivious_trees"` path:** non-symmetric uses `"trees"` with a recursive nested-node schema — a distinct parser.

## Common Pitfalls

### Pitfall 1: Treating Region as in-scope
**What goes wrong:** Planning a Region grower; no fixture can be generated.
**Why it happens:** Region is in `EGrowPolicy` and the Python `grow_policy` accepts the string, but CPU training throws `"GrowPolicy Region is unimplemented for CPU."` (`greedy_tensor_search.cpp` dispatch, verified at runtime).
**How to avoid:** Scope FEAT-06 to Lossguide + Depthwise. Record Region as an escalated gap (Phase 7 GPU may add it; upstream itself only has it on GPU).
**Warning signs:** A plan task named "Region grower" or a `grow_policy=Region` fixture.

### Pitfall 2: Building non-symmetric monotone
**What goes wrong:** No ground truth; fitting throws.
**Why it happens:** D-6.6-07 anticipated it; confirmed `monotonic_constraint_utils.h:42` `CB_ENSURE_INTERNAL(monotoneConstraints.empty(), "...unsupported for non-symmetric trees yet")`.
**How to avoid:** Monotone is OBLIVIOUS-ONLY. Add a guard in `cb-train` that errors (typed `CbError`) on `monotone_constraints` + non-symmetric `grow_policy`, mirroring upstream's rejection.
**Warning signs:** Any task combining `monotone_constraints` with `grow_policy in {Lossguide, Depthwise, Region}`.

### Pitfall 3: Non-symmetric JSON parsed as oblivious
**What goes wrong:** `model_json` parser returns empty/garbage; oracle silently mis-compares.
**Why it happens:** Oblivious models export `"oblivious_trees"` (flat `splits`/`leaf_values`); non-symmetric export `"trees"` (recursive `{split, left, right}` nodes, `{value, weight}` leaves) — a structurally different schema (verified).
**How to avoid:** Add a distinct parser branch keyed on the top-level container. Convert the nested form to the flat `{tree_splits, step_nodes, node_id_to_leaf_id, leaf_values}` triple (or parse the `.cbm` FlatBuffers directly, which is unambiguous).
**Warning signs:** A non-symmetric oracle test that passes trivially / compares zero-length vectors.

### Pitfall 4: Lossguide queue penalty recomputation
**What goes wrong:** Splits diverge from upstream when penalties are active under Lossguide.
**Why it happens:** Upstream deliberately does NOT recompute queued candidate penalties when a feature first becomes used (documented, lines 1875-1882).
**How to avoid:** Replicate the imprecision exactly — penalize at enqueue time only.
**Warning signs:** Penalized-Lossguide oracle drift even though oblivious penalties pass.

### Pitfall 5: SHAP/fstr scale==1 assumption under non-symmetric
**What goes wrong:** SHAP recursion assumes `scale==1` (`CB_ENSURE_SCALE_IDENTITY`) and consumes oblivious `leaf_weights` indexed in forward-bit order; the non-symmetric node graph has no forward-bit leaf ordering and weights attach to leaf nodes via `node_id_to_leaf_id`.
**Why it happens:** The existing `calc_subtree_weights` (`shap.rs:206`) walks a perfect binary tree of depth `tree_depth`; a non-symmetric tree is an arbitrary binary graph.
**How to avoid:** Generalize the subtree-weight computation to the actual node graph (recurse on `step_nodes` children rather than bit-indexing). Keep the oblivious path unchanged; branch on variant.
**Warning signs:** Non-symmetric SHAP fails local-accuracy (sum of SHAP + bias ≠ prediction).

### Pitfall 6: Penalty sign/placement confusion
**What goes wrong:** Applying all three penalties the same way.
**Why it happens:** They differ. `FeatureWeights` is a **multiplicative gain factor** in `SelectBestCandidate` (`score *= GetCatFeatureWeight(...)`; then `gain *= GetSplitFeatureWeight(...)`). `FirstFeatureUsePenalty` and `PerObjectFeaturePenalty` are **subtractive**, applied in a separate `PenalizeBestSplits` pass (`score -= ...`), scaled by `penalties_coefficient`. Per-object penalty counts objects whose feature is unused (whole-fold for symmetric; per-leaf for non-symmetric).
**How to avoid:** Transcribe `feature_penalties_calcer.cpp` faithfully; respect the two distinct insertion points. For the symmetric path, per-object penalty = `penalty_per_object * doc_count` when the feature is globally unused, else 0.
**Warning signs:** Penalty oracle passes for `feature_weights` but fails for `first_feature_use_penalties`/`per_object_feature_penalties`.

## Open Questions

1. **pc/threading determinism for non-symmetric leaf values**
   - What we know: oblivious leaf values depend on fold permutation/RNG draw order (Phase 5 pain); non-symmetric uses the same fold machinery. `thread_count=1` is fixed.
   - What's unclear: whether the leaf-wise growers consume the RNG draw stream identically to the oblivious path (CalcScoreStDev / per-candidate score draws appear shared, but the subtract-trick ordering differs).
   - Recommendation: in Wave block B Wave-0, generate the simplest possible Depthwise fixture (no CTR, no bootstrap, `random_strength=0`) and lock splits FIRST, before leaf values; escalate to the instrumented trainer only if the draw stream diverges.

2. **ShapInteractionValues output shape `(n_obj, n_feat+1, n_feat+1)`**
   - What we know: verified shape includes a `+1` (bias/expected-value row/col).
   - What's unclear: exact indexing convention of the bias slot vs upstream `shap_interaction_values.cpp`.
   - Recommendation: capture a tiny oracle and reverse-map the bias index empirically before coding the recursion.
