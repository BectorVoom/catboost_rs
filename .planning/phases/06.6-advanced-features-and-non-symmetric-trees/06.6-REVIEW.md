# Phase 06.6: Code Review Report

**Reviewed:** 2026-06-18
**Depth:** standard
**Files Reviewed:** 28
**Status:** issues_found

## Summary

Reviewed the FEAT-03/04/05/06 + MODEL-05 + D-12 production source for phase 06.6
(advanced features + non-symmetric trees). The numeric/parity discipline is
generally strong: bounded `.get` access, `cb_core::sum_f64` reductions, checked
casts, and the previously-flagged CR-01 (`as u16` step-node diff → now
`u16::try_from`, tree.rs:1163-1178) and CR-02 (interior-node `u32::MAX` sentinel,
tree.rs:1143/1155) are both fixed correctly and completely.

The headline NEW defect is a CLAUDE.md hard-rule violation that is also a latent
correctness bug: a cluster of bare `Vec` index operations (`[id]`, `[g]`,
`[node]`) in the FEAT-06 leaf-wise grower (`tree.rs`), in `json.rs`, and in
`feature_selection.rs` violate the workspace `indexing_slicing = "deny"` lint and
the project `unwrap`/panic prohibition spirit. They are reachable on crafted /
malformed inputs and would either fail `cargo clippy` or panic at runtime.

Secondary findings center on the non-symmetric tree path: the eval-set /
early-stopping / `use_best_model` machinery in the boosting loop only operates on
the oblivious `trees` vector and silently no-ops for non-symmetric models
(CR-03 + a newly-observed `use_best_model` truncation gap). These are consistent
with the verifier's triaged latent items; the `use_best_model` truncation is a
distinct manifestation worth flagging.

The known-deferred items (CR-03 eval-set no-op, CR-04 non-sym CTR index-0, WR-02
`.cbm`→json non-identity round-trip, WR-04 shap recursion fall-through) were all
confirmed present and are classified Info/known-deferred below. WR-04 was
verified to be benign (the fall-through descends only the non-zero side of a
one-sided halt node, which is the correct upstream behaviour).

## Critical Issues

### CR-01: Bare slice indexing in production violates `indexing_slicing = "deny"` (panic on malformed input)

**File:** `crates/cb-train/src/tree.rs:973,990,991,1012,1090,1095,1155,1186,1187`; `crates/cb-model/src/json.rs:275`; `crates/cb-train/src/feature_selection.rs:252,326`
**Issue:** The workspace denies `clippy::indexing_slicing` (`/Cargo.toml:14`) and
CLAUDE.md prohibits panics in production. The FEAT-06 leaf-wise grower and the
json flatten path use direct `Vec` indexing instead of checked `.get`:

- `tree.rs:973` `let depth = node_depth[id] + 1;`
- `tree.rs:990` `nodes[id] = BuiltNode::Interior { … };`
- `tree.rs:991` `node_split[id] = Some(bs.split);`
- `tree.rs:1012` `let docs = node_docs[leaf].clone();`
- `tree.rs:1090` `if node_depth[node] >= max_depth`
- `tree.rs:1095` `&node_docs[node],`
- `tree.rs:1155` `node_id_to_leaf_id[id] = u32::MAX;`
- `tree.rs:1186-1187` `node_to_leaf[id] = …; node_id_to_leaf_id[id] = …;`
- `json.rs:275` `node_id_to_leaf_id[id] = u32::try_from(leaf_id)…`
- `feature_selection.rs:252,326` `(0..n_features).filter(|&g| retained[g])`

In the grower/json cases the indices are constructed in-bounds by the surrounding
loop, so they will not panic in practice today — but they are exactly the kind of
unchecked access the deny-lint exists to prevent, and any future refactor of the
node-registration order turns a silent in-bounds assumption into an OOB panic.
More importantly, `cargo clippy` (the project's stated quality gate) will reject
these, so the code does not meet the project's own bar. This is a CLAUDE.md hard
rule, hence Critical.
**Fix:** Replace each with checked access in the established style of the file,
e.g. tree.rs:973:
```rust
let depth = node_depth.get(id).copied().unwrap_or(0) + 1;
```
tree.rs:990-991 (the do_split closure already owns `id < nodes.len()` by
construction — make it explicit):
```rust
if let Some(slot) = nodes.get_mut(id) {
    *slot = BuiltNode::Interior { split: bs.split, left, right };
}
if let Some(slot) = node_split.get_mut(id) {
    *slot = Some(bs.split);
}
```
tree.rs:1012 / 1095:
```rust
let docs = node_docs.get(leaf).cloned().unwrap_or_default();
```
json.rs:275:
```rust
if let Some(slot) = node_id_to_leaf_id.get_mut(id) {
    *slot = u32::try_from(leaf_id).map_err(|_| ModelError::Deserialize(
        "non-symmetric leaf id exceeds u32".to_owned()))?;
}
```
feature_selection.rs:252,326:
```rust
let local_to_global: Vec<usize> =
    (0..n_features).filter(|&g| retained.get(g).copied().unwrap_or(false)).collect();
```

## Warnings

### WR-01: `use_best_model` never truncates non-symmetric models

**File:** `crates/cb-train/src/boosting.rs:4013-4017`
**Issue:** After the boosting loop, `use_best_model` truncates only the oblivious
`trees` vector:
```rust
if params.use_best_model {
    if let Some(best) = best_model.best_iteration() {
        trees.truncate(best + 1);
    }
}
```
For a non-symmetric model (Lossguide / Depthwise) every tree is pushed into
`non_symmetric_trees` (boosting.rs:3957) and `trees` stays empty, so
`trees.truncate(...)` is a no-op and the model is NOT truncated to the best
iteration. A `use_best_model=true` non-symmetric run therefore keeps every grown
tree, silently producing a different (over-fit) model than requested. This is the
non-symmetric sibling of the triaged CR-03 eval-set no-op but a DISTINCT code
path (the best-iteration tracker can still be populated via CR-03's eval approx
if that is ever fixed, yet this truncation would remain broken).
**Fix:** Truncate whichever vector holds the model's trees:
```rust
if params.use_best_model {
    if let Some(best) = best_model.best_iteration() {
        trees.truncate(best + 1);
        non_symmetric_trees.truncate(best + 1);
    }
}
```
(Long-term, the eval-approx update at boosting.rs:3971-3980 must also cover
`non_symmetric_trees` — see IN-01 / CR-03.)

### WR-02: `predict_raw_multi_cat` non-symmetric `n_leaves` divides a possibly-ragged buffer without validation

**File:** `crates/cb-model/src/apply.rs:520-529`
**Issue:** The multi-dim non-symmetric arm computes
`n_leaves = tree.leaf_values.len() / dim.max(1)` and reads
`d.saturating_mul(n_leaves).saturating_add(leaf)`. If `leaf_values.len()` is not
a multiple of `dim` (a malformed loaded model), `n_leaves` is a floor and the
per-dimension stride is wrong, so dimension `d > 0` reads values belonging to a
different leaf rather than failing or contributing 0.0. The oblivious arm has the
same shape but is documented (and the loader enforces the multiple); the
non-symmetric arm has no such enforcement in `reconstruct_non_symmetric`
(cbm.rs slices by `distinct_leaves * dim` but never re-checks the per-tree
`leaf_values.len() % dim == 0`). The result is a plausible-but-wrong prediction
rather than a detected error.
**Fix:** Either validate `tree.leaf_values.len() % dim == 0` at load time and
return `ModelError::Deserialize` on a remainder, or guard the read so a non-zero
remainder contributes 0.0 explicitly:
```rust
let n_leaves = if dim == 0 || tree.leaf_values.len() % dim != 0 {
    0 // ragged buffer -> contribute nothing rather than mis-stride
} else {
    tree.leaf_values.len() / dim
};
```

### WR-03: `predict_raw_multi` / `predict_raw_multi_cat` drop per-dimension bias (RMSEWithUncertainty)

**File:** `crates/cb-model/src/apply.rs:394-396,483`
**Issue:** `predict_raw_multi_cat` seeds EVERY output dimension with the single
scalar `model.bias` (= dim-0 bias). The code itself documents (apply.rs:384-393,
479-482) that this is WRONG for a genuine per-dimension-bias model such as
RMSEWithUncertainty whose `scale_and_bias[1] = [mean, 0.5*log(var)]`: the dim-1
log-scale starting bias is silently dropped, yielding a wrong
`exp(2*log-scale)` variance. The mitigation (`predict_raw_multi_biased`) exists
but is NOT what the public `predict`/`predict_proba` facade calls, so a loaded
RMSEWithUncertainty model predicted through the normal surface is numerically
wrong on dim 1. This is the verifier-adjacent WR; flagging because the divergence
is silent (no error, plausible output) and the model `Model::bias` is a single
scalar with no per-dim field.
**Fix:** Carry the per-dimension bias on the canonical `Model` (a
`Vec<f64>` populated from `scale_and_bias[1]` in json.rs / cbm.rs) and have
`predict_raw_multi_cat` seed slot `d` with `bias[d]` (falling back to `0.0`),
collapsing `predict_raw_multi_biased` into the single path. Until then, route
multi-dim-bias losses through `predict_raw_multi_biased` at the facade.

### WR-04: `interaction_dfs` / shap treat a pure-leaf node's sentinel split as real feature 0

**File:** `crates/cb-model/src/fstr.rs:413-458`; `crates/cb-model/src/shap.rs:716-723`
**Issue:** A pure-leaf node stores a placeholder `Float { feature: 0, border:
NEG_INFINITY }` (json.rs:269-272, tree.rs:1184). In `interaction_dfs`, after the
terminal-node accumulation (fstr.rs:413-437) the code reads the node's split
feature (fstr.rs:440) and gets `Some(0)` for that placeholder rather than `None`;
the subsequent descend loop is a no-op only because both children equal
`node_idx`. The same placeholder reaches `shap_recurse_non_symmetric`
(shap.rs:716) and `map_node_id_to_is_go_right` (shap.rs:641-650). Today the
placeholder is harmless because feature `0` is never actually consulted on a
zero-diff side, but it is fragile: feature index `0` is a REAL feature, so any
future change that treats a leaf node's split feature as live (e.g. recording the
combination class before the diff check) would attribute spurious interaction /
SHAP mass to feature 0. The placeholder should be unambiguously non-float.
**Fix:** Short-circuit on `step_nodes[node] == (0, 0)` before reading the split
feature in `interaction_dfs` and `shap_recurse_non_symmetric`:
```rust
if matches!(tree.step_nodes.get(node_idx), Some(&(0, 0))) {
    return; // pure leaf: no split feature to descend on
}
```

### WR-05: `feature_selection` schedule rounding can under-count survivors without surfacing it

**File:** `crates/cb-train/src/feature_selection.rs:111-129,240-248,313-320`
**Issue:** `calc_num_to_eliminate_by_steps` produces `Vec<u32>` rounded counts and
the caller clamps each step to the remaining budget
(`.min(n_to_eliminate_total - eliminated_features.len())`). If the rounded
schedule sums to FEWER than `n_to_eliminate_total` (a rounding shortfall on
adversarial cardinalities), the loop exits with `selected_features.len() >
num_features_to_select` and NO error — the function silently returns more
survivors than requested. The remainder-carry (`rem = precise - r`) only corrects
the LAST step if iteration count and rounding cooperate; there is no post-loop
assertion that the budget was met.
**Fix:** After the elimination loop, assert the invariant and error on a
shortfall:
```rust
if eliminated_features.len() != n_to_eliminate_total {
    return Err(CbError::Degenerate(format!(
        "feature-selection schedule eliminated {} of {} (rounding shortfall)",
        eliminated_features.len(), n_to_eliminate_total)));
}
```
(or eliminate the residual greedily in a final pass before reporting).

## Info

### IN-01: CR-03 (known-deferred) — eval-set / early-stopping silently no-op for non-symmetric trees

**File:** `crates/cb-train/src/boosting.rs:3971-4007`
**Issue:** `tree_eval_contribution` is consumed via `trees.last()`
(boosting.rs:3972), which is `None` for a non-symmetric model, so the eval-set
approx is never updated and the overfitting detector / best-model tracker never
see real metrics. This matches the verifier's triaged CR-03 and is classified
Info/known-deferred. Note it compounds WR-01.
**Fix:** Branch the eval-approx update on the tree variant, applying a
`leaf_index_nonsym`-equivalent contribution for `non_symmetric_trees.last()`.

### IN-02: CR-04 (known-deferred) — non-symmetric CTR split serialized as global index 0

**File:** `crates/cb-model/src/cbm.rs:181-185`
**Issue:** In the non-symmetric serialize path a node whose split is not a float
split (`split.as_float()` is `None`, i.e. a CTR split) falls through to
`tree_splits.push(0)`, indistinguishable from a pure-leaf filler. Matches the
verifier's triaged CR-04; non-symmetric CTR models are out of the in-scope
fixture set. Classified Info/known-deferred.
**Fix:** When non-symmetric CTR support lands, route CTR splits through the
`ctr_data` model-parts region (as the oblivious path documents at cbm.rs:243-250)
rather than emitting a `0` placeholder.

### IN-03: WR-02 (known-deferred) — `.cbm` → json round-trip is non-identity for one-sided non-symmetric nodes

**File:** `crates/cb-model/src/json.rs:333-393`
**Issue:** `unflatten_non_symmetric` expands a one-sided halt node `(d, 0)` /
`(0, d)` into a real json leaf child (json.rs:372-381) because a json node cannot
be both a leaf and an interior split. Re-flattening via
`flatten_non_symmetric` then mints a NEW distinct leaf, so node/leaf counts
differ from the original compact `.cbm`. Predictions remain correct (the leaf
VALUE is duplicated, not changed); only the structural representation differs.
Matches the verifier's triaged WR-02. Classified Info/known-deferred.
**Fix:** Document the non-identity explicitly in `save_json`/`load_json`, or have
the json schema carry the compact one-sided form so the round-trip is structural-
identity.

### IN-04: WR-04 verification — shap non-symmetric fall-through descent is correct (no fix needed)

**File:** `crates/cb-model/src/shap.rs:699-709,768,786`
**Issue:** The triaged "shap recursion fall-through without return" was verified:
`shap_recurse_non_symmetric` distributes SHAP at a terminal/one-sided node and
then continues to descend ONLY the non-zero side (the descent guards at
shap.rs:768,786 require `go_node_idx != node_idx` / `skip_node_idx != node_idx`).
For a one-sided halt node this is the correct upstream behaviour (the node both
halts on the zero side and branches on the non-zero side); for a pure `(0,0)`
leaf neither side recurses, so there is no double-counting. The genuine residual
risk in this neighbourhood is the placeholder-feature fragility tracked as WR-04;
the descent control flow itself is sound and needs no change.
