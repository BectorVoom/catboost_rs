# Phase 06.6: Code Review Report

**Reviewed:** 2026-06-18
**Depth:** standard
**Files Reviewed:** 10
**Status:** issues_found

## Summary

Phase 06.6 introduces Lossguide/Depthwise non-symmetric tree growing, the PAVA
isotonic projection for monotone constraints, new `BoostParams` fields, `.cbm`/json
serde for non-symmetric trees, PVC/SHAP recursion for the non-symmetric node graph,
and a feature-selection subsystem. Most code is well-structured with consistent
checked-access patterns and good depth-bounding. Four blockers were found: two
involve incorrect integer casts or sentinel mismatches that corrupt the produced
model (one from overflow, one from a sentinel contract mismatch introduced by the
new grower), and two involve silent data loss (eval-set update bypassed for
non-symmetric models, CTR splits silently serialized as the wrong split). Four
warnings cover lint violations, debug-only invariant guards, and a
non-identity-preserving serde roundtrip.

## Critical Issues

### CR-01: Unchecked `as u16` truncation in step-node diff encoding

**File:** `crates/cb-train/src/tree.rs:1148-1149`
**Issue:** The finalization loop computing `step_nodes` casts child-minus-parent
node-id differences with bare `as u16`:

```rust
let ld = (*left - id) as u16;
let rd = (*right - id) as u16;
```

`Depthwise` policy with `max_depth = 16` can produce up to `2^17 - 1 = 131071`
nodes (`MAX_DEPTH` = 16, and `max_leaves` defaults to `max_leaves_default()` which
allows 64 leaves). A root node's right child can have id up to 65535 (just over the
u16 limit). Any child id >= 65536 wraps silently: diff wraps to a small wrong value
or to 0. A diff of 0 means "halt here" in the apply walk — so an interior node with
a diff that wraps to 0 is misclassified as a terminal leaf, causing the apply walk
to read a leaf value from an interior node's `node_id_to_leaf_id` slot (which holds
`0`, leaf 0) rather than continuing the walk. All objects that should descend through
that branch silently read leaf 0's value. The resulting model produces wrong
predictions with no error. The identical conversion in `crates/cb-oracle/src/model_json.rs`
correctly uses `u16::try_from(...)`, which is the fix pattern.

**Fix:**
```rust
// In the BuiltNode::Interior arm of the finalization loop:
let ld = u16::try_from(*left - id).map_err(|_| {
    CbError::SchemaVersion(format!(
        "non-symmetric step diff {} exceeds u16 (node count {}); \
         reduce max_depth or max_leaves",
        *left - id,
        node_count
    ))
})?;
let rd = u16::try_from(*right - id).map_err(|_| {
    CbError::SchemaVersion(format!(
        "non-symmetric step diff {} exceeds u16 (node count {}); \
         reduce max_depth or max_leaves",
        *right - id,
        node_count
    ))
})?;
step_nodes.push((ld, rd));
```

### CR-02: Interior-node `node_id_to_leaf_id` sentinel mismatch corrupts `.cbm` save

**File:** `crates/cb-train/src/tree.rs:1137` / `crates/cb-model/src/cbm.rs:197-201`
**Issue:** The grower initializes every slot of `node_id_to_leaf_id` to `0`:

```rust
let mut node_id_to_leaf_id: Vec<u32> = vec![0; node_count];
```

Interior nodes (`BuiltNode::Interior`) never write to this vec; their slots stay
`0`. But the `.cbm` serializer (`cbm.rs:197-201`) treats `u32::MAX` as the
sentinel for interior nodes that must not contribute to the leaf count:

```rust
let distinct_leaves = tree
    .node_id_to_leaf_id
    .iter()
    .filter(|&&id| id != u32::MAX)   // interior nodes should have MAX
    .count();
```

For a Rust-trained non-symmetric model, interior nodes have `0` instead of
`u32::MAX`, so they are counted as additional leaf slots. `distinct_leaves`
is over-counted by the number of interior nodes. The serializer then emits
`distinct_leaves * dim` leaf-value entries (line 216-220), reading
out-of-bounds from `tree.leaf_values` (the `.ok_or_else` catches that and
returns an error) or producing a corrupt blob with garbage leaf values.
The apply walk itself is not affected when the model stays in memory
(the apply code only reads `node_id_to_leaf_id` on diff==0 halt nodes,
which are always actual leaf nodes in a well-formed grower output), but
any model saved with `cbm::save` and then loaded will decode incorrectly.

**Fix:** Fill interior-node `node_id_to_leaf_id` slots with `u32::MAX` in the
finalization loop:

```rust
BuiltNode::Interior { split, left, right } => {
    splits.push(*split);
    let ld = ...;
    let rd = ...;
    step_nodes.push((ld, rd));
    // Mark as interior (never a halt node); cbm serializer uses u32::MAX sentinel.
    node_id_to_leaf_id[id] = u32::MAX;
}
```

### CR-03: Eval-set approximant never updated for non-symmetric models

**File:** `crates/cb-train/src/boosting.rs:3949-3980`
**Issue:** The per-iteration eval-set approx update, overfitting detection, and
`use_best_model` logic all pivot on `trees.last()` (the ObliviousTree vector):

```rust
if grown.step_nodes.is_empty() {
    trees.push(ObliviousTree { ... });
} else {
    non_symmetric_trees.push(NonSymmetricTree { ... });
}

// ...
if has_test {
    if let Some(tree) = trees.last() {   // always None for non-symmetric path
        // eval_approx update — silently never runs
    }
}
```

For a non-symmetric model (`EGrowPolicy::Lossguide` or `EGrowPolicy::Depthwise`)
`trees` is always empty; `trees.last()` always returns `None`; the inner block
never executes. Consequences:
1. `eval_approx` is never updated, so the eval-set metric is always the bias-only
   prediction for every iteration.
2. `best_model.add_error` / `detector.add_error` never receive a finite value;
   `use_best_model` never triggers and the model keeps all iterations regardless.
3. Early stopping via `is_need_stop()` never fires.

This is silent data loss: the trainer completes without error but eval-set
metrics and early stopping are silently broken for non-symmetric models.

**Fix:** Extract the leaf contribution into a per-variant helper and apply it
unconditionally, or route through the `non_symmetric_trees` last entry:

```rust
if has_test {
    // Route through whichever variant was just pushed.
    let contrib_fn: Box<dyn Fn(&FeatureMatrix, usize) -> f64> =
        if let Some(t) = trees.last() {
            let t = t.clone(); // or use an index
            Box::new(move |em, obj| tree_eval_contribution(&t, em, obj))
        } else if let Some(t) = non_symmetric_trees.last() {
            let t = t.clone();
            Box::new(move |em, obj| ns_tree_eval_contribution(&t, em, obj))
        } else {
            // No tree grown this iteration (degenerate); skip.
            continue; // or handle gracefully
        };
    for (set_idx, approx_col) in eval_approx.iter_mut().enumerate() {
        if let Some(em) = eval_matrices.get(set_idx) {
            for (obj, a) in approx_col.iter_mut().enumerate() {
                *a += contrib_fn(em, obj);
            }
        }
    }
    // ... metric / detector logic unchanged ...
}
```

Alternatively, factor both paths through a common `eval_contribution` enum.

### CR-04: CTR split in non-symmetric tree silently serialized as wrong float split

**File:** `crates/cb-model/src/cbm.rs:181-185`
**Issue:** In `build_core_blob`, the per-node split serialization has an `else`
arm that pushes global split index `0` for CTR splits:

```rust
} else {
    // CTR split: not yet implemented for non-symmetric trees.
    tree_splits.push(0);
}
```

Index `0` in the global split table corresponds to the first border of float
feature 0. On reload, this is deserialized as a `ModelSplit::Float` for that
feature and border. A model saved and then reloaded will have all CTR splits
in non-symmetric trees silently replaced by the first float-feature's first
border split. Predictions will differ from the original model without error.
Although CTR splits in non-symmetric trees are currently prevented by upstream
validation (CTR candidates are only generated on the oblivious `greedy_tensor_search`
path), the comment and code imply it is expected to be reached in the future.
The correct behavior is to return an error rather than silently corrupt the
serialized model.

**Fix:**
```rust
} else {
    return Err(ModelError::SchemaVersion(
        "CTR splits in non-symmetric trees are not yet supported for .cbm \
         serialization; use oblivious (SymmetricTree) grow_policy for CTR models"
            .to_owned(),
    ));
}
```

## Warnings

### WR-01: Direct `[id]` indexing in `do_split` closure violates `indexing_slicing` deny-lint

**File:** `crates/cb-train/src/tree.rs:973, 990-991`
**Issue:** Three direct index operations appear in the `do_split` closure inside
`leaf_wise_grower`:

```rust
let depth = node_depth[id] + 1;
// ...
nodes[id] = BuiltNode::Interior { split: bs.split, left, right };
node_split[id] = Some(bs.split);
```

`lib.rs:1` sets `#![cfg_attr(test, allow(clippy::indexing_slicing))]`, meaning
`indexing_slicing` is denied in production code. These three accesses are in
production code. They are valid by construction (all ids are allocated by the
grower and fit within the corresponding vecs), but the lint is a hard project
rule enforced for a reason — a future refactor that changes the allocation order
could introduce an OOB panic without any compiler warning.

**Fix:** Use `.get` / `.get_mut` with `ok_or_else`:
```rust
let depth = node_depth.get(id)
    .copied()
    .ok_or_else(|| CbError::SchemaVersion("node depth index OOB".to_owned()))?
    + 1;
// ...
*nodes.get_mut(id)
    .ok_or_else(|| CbError::SchemaVersion("node index OOB".to_owned()))? =
    BuiltNode::Interior { split: bs.split, left, right };
*node_split.get_mut(id)
    .ok_or_else(|| CbError::SchemaVersion("node split index OOB".to_owned()))? =
    Some(bs.split);
```

### WR-02: `.cbm` → `model.json` → `.cbm` roundtrip is not identity-preserving

**File:** `crates/cb-model/src/json.rs` (`unflatten_non_symmetric`)
**Issue:** `unflatten_non_symmetric` expands one-sided halt nodes `(d, 0)` or
`(0, d)` into interior nodes with a synthetic leaf child. A one-sided halt node
in a `.cbm`-loaded model occupies a single flat slot in `step_nodes`; after the
json roundtrip it becomes an interior node with an extra child slot inserted.
The resulting flat encoding differs from the original. Predictions are identical
(the synthetic leaf returns the same value), but the wire form changes.

This matters for any workflow that does `.cbm` → `Model` → `json` → `Model` →
`.cbm` and expects byte-identical output, or for tests that compare serialized
forms. The discrepancy could also surface as a validation failure in a strict
schema checker.

**Fix:** Document the non-identity behavior explicitly in the `unflatten_non_symmetric`
docstring, and add a test that asserts prediction identity across the roundtrip
(as a minimum contract). If wire-identity is required, the serializer must
preserve one-sided halt nodes rather than expanding them.

### WR-03: `debug_assert!` for invariants that already have runtime guards

**File:** `crates/cb-train/src/feature_selection.rs:117-118`
**Issue:**
```rust
debug_assert!(n_to_select <= n_for_select);
debug_assert!(steps >= 1);
```
These assertions fire only in debug builds. The identical invariants are
enforced again by `return Err(...)` guards at lines 173 and 205. The
`debug_assert!` is dead code in release builds and misleadingly implies the
checks are in the right place. Mixing `debug_assert!` with proximate runtime
guards is inconsistent and can erode confidence in the error handling.

**Fix:** Remove the `debug_assert!` lines and rely solely on the runtime error
guards; they are already correct and fire in release builds.

### WR-04: `shap_recurse_non_symmetric` terminal-node check does not emit SHAP before early return

**File:** `crates/cb-model/src/shap.rs:699-723`
**Issue:** When the walk reaches a terminal node (`go_node_idx == node_idx ||
skip_node_idx == node_idx`), the code correctly calls
`update_shap_by_feature_path`. However, immediately after that block, the code
continues to the `let Some(combination_class) = ...` guard:

```rust
// Terminal node for this object (the chosen — or skipped — side has diff 0).
if go_node_idx == node_idx || skip_node_idx == node_idx {
    let leaf_id = ...;
    let leaf_value = ...;
    update_shap_by_feature_path(...);
}
// This node's split combination class; a CTR / pure-leaf node has none — stop.
let Some(combination_class) = tree.tree_splits.get(node_idx)... else {
    return;
};
```

For a terminal leaf node (both diffs zero, `step_nodes == (0, 0)`), both
conditions `go_node_idx == node_idx` AND `skip_node_idx == node_idx` hold.
After `update_shap_by_feature_path` the `combination_class` guard returns
early because a leaf node's split is a placeholder (`Split { feature: 0, border: 0.0 }`)
and `float_feature()` on that placeholder still succeeds (feature index 0, border 0.0
is a valid float split). The recursion therefore continues to compute SHAP
contributions for children — but `go_node_idx == node_idx` causes the child
recursion at line 768 to call `shap_recurse_non_symmetric` with the SAME
`node_idx`, leading to unbounded recursion.

In practice the depth bound at line 679 (`node_idx >= subtree_weights.first().map_or(0, Vec::len)`)
stops the recursion because the parent's `node_idx` is already a valid slot and
the child's `go_node_idx == node_idx` passes the same bound check, recursing
until `condition_feature_fraction` drives the fractions to zero via the coefficient
multiplication. Whether this actually terminates in finite steps depends on floating-
point rounding, not a hard structural bound. If `hot_coefficient` happens to be
exactly 1.0 (all weight in one subtree), the fractions never decay and the
recursion proceeds until the stack exhausts.

**Fix:** Early-return after the terminal-node SHAP update:
```rust
if go_node_idx == node_idx || skip_node_idx == node_idx {
    let leaf_id = tree.node_id_to_leaf_id.get(node_idx).copied().unwrap_or(u32::MAX) as usize;
    let leaf_value = tree.leaf_values.get(leaf_id).copied().unwrap_or(0.0);
    update_shap_by_feature_path(
        &feature_path,
        leaf_value,
        average_tree_approx,
        condition_feature_fraction,
        shap_by_feature,
    );
    return;  // Terminal: do not recurse into children.
}
```

## Info

### IN-01: Redundant `.abs()` in class label deduplication

**File:** `crates/cb-train/src/boosting.rs:1073` (approximately)
**Issue:** `(*a - *b).abs() == 0.0` is used to deduplicate float class labels.
The `.abs()` is redundant: `x == 0.0` and `x.abs() == 0.0` are equivalent for
finite floats, and the subtraction result is exactly 0.0 when the two floats
are bit-equal. The `.abs()` adds a CPU instruction and implies the comparison
is checking magnitude rather than equality.

**Fix:** Replace with `(*a - *b) == 0.0` or simply `a == b`.

### IN-02: Off-by-one in apply loop bound (benign extra iteration)

**File:** `crates/cb-model/src/apply.rs:245`
**Issue:** The walk loop is bounded by `0..=node_count` (inclusive), giving
`node_count + 1` iterations rather than `node_count`. A valid acyclic graph
terminates before `node_count` steps; the extra iteration is harmless (it just
returns `None` on the final overflow attempt). However it slightly misrepresents
the invariant documented in the comment ("more than `node_count` steps means a
cyclic graph").

**Fix:** Change to `0..node_count` to match the stated invariant. A walk
requiring exactly `node_count` steps without halting is already a cycle.

### IN-03: Dead `unwrap_or(0)` fallback on infallible conversion

**File:** `crates/cb-model/src/cbm.rs:816`
**Issue:**
```rust
let gv = usize::try_from(raw_leaf_id).unwrap_or(0);
```
`raw_leaf_id` is a `u32`. On any 32-bit or 64-bit target, `u32 -> usize` via
`try_from` is infallible (`usize` is at least 32 bits). The `unwrap_or(0)` branch
is dead code. If the intent was to fall back to leaf 0 on a hypothetical 16-bit
target, that would be a silent data error rather than a defensive default.

**Fix:** Use `usize::from(raw_leaf_id)` (which is infallible and does not compile
on targets where `usize < u32`) to make the platform assumption explicit and
eliminate the dead fallback.
