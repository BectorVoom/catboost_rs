# Phase 06.6 Plan 08: Recursive Feature Selection (Gate-D, FEAT-05, LAST) Summary

**`cb_train::select_features(...)` — the recursive feature-elimination loop mirroring `recursive_features_elimination.cpp` (the per-step `CalcNumEntitiesToEliminateBySteps` schedule, the sub-matrix re-index + global remap, the survivor partition), generic over an injected `ImportanceRanker` so the orchestration stays in `cb-train` source (D-6.6-03, modules over crates) without the `cb-model -> cb-train` build-graph cycle — with the `{selected_features, eliminated_features}` partition oracle-locked (discrete set/order equality) vs catboost 1.2.10 `select_features(...)` for BOTH the SHAP backend (`RecursiveByShapValues`) and the FeatureEffect backend (`RecursiveByPredictionValuesChange`, shared with `RecursiveByLossFunctionChange`), the real Gate-C importances supplied through the cycle-exempt cb-model dev-dependency.**

## Accomplishments
- **`select_features(...)`** in the NEW `crates/cb-train/src/feature_selection.rs` module: per step, train via `cb_train::train` on the current candidate sub-matrix (only the still-retained columns, re-indexed `0..n_local`), rank with the injected `ImportanceRanker`, eliminate the `numToEliminate[step]` lowest-scoring STILL-SELECTABLE features (always-retained features outside `features_for_select` are never eliminated, mirroring `selectSet.Features.contains`), repeat until `num_features_to_select` survive. `train_final_model=false` returns the partition without a final fit; `=true` runs (and discards) one final survivor retrain to mirror the upstream control flow.
- **`EFeaturesSelectionAlgorithm`** (3 variants) + **`FeatureSelectionResult { selected_features, eliminated_features }`** + the **`ImportanceRanker`** callback type, exported from `lib.rs`.
- **`calc_num_to_eliminate_by_steps`** transcribing `CalcNumEntitiesToEliminateBySteps` (geometric fraction `p=(k/n)^(1/steps)`, `preciseValue = n*p^step*(1-p)` with the carried rounding remainder so the rounded per-step counts sum EXACTLY to `n-k`).
- **Typed validation (T-06.6-19)**: `CbError::OutOfRange` on out-of-range / duplicate `features_for_select` indices, `num_features_to_select > features_for_select.len()`, `steps == 0`, and a `feature_borders` length mismatch; checked `.get` access throughout; the loop is bounded by `steps` / the total-to-eliminate (T-06.6-20, no unbounded loop).
- **`feature_selection_oracle_test`** (2 tests, neither `#[ignore]`d): drives `select_features` end-to-end with the REAL Gate-C importances — `shap_values` (the SHAP backend, ranked by RMSE loss-change-on-removal) and `prediction_values_change` (the FeatureEffect backend) — and asserts `{selected_features, eliminated_features}` EXACTLY match the captured catboost 1.2.10 partition.
- **Oracle fixture** (offline, `.venv` catboost 1.2.10, `thread_count=1`, seed=0): `gen_fixtures.py` + `X.npy`/`y.npy` (200x4, signal coef 3,2,0.3,0) + an independent complete per-feature quantile border grid + the full-feature `model.json` + the captured `{selected,eliminated}` partitions for both backends. Both backends: selected={0,1}, eliminated=[2,3].

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] cb-train SOURCE cannot consume cb-model importances directly (build-graph cycle)**
- **Found during:** Task 1 (designing the importance-backend call)
- **Issue:** The plan frames `feature_selection.rs` (a cb-train SOURCE module per D-6.6-03) as consuming the cb-model importance methods (`shap_values` / `prediction_values_change` / `loss_function_change`). But `cb-model` depends on `cb-train` in the workspace build graph (`cb_model::Model::from_trained(&cb_train::Model, ...)`), so cb-train source naming a cb-model type would be a dependency cycle (cargo would refuse to compile). The plan's integration note acknowledged the importances "live in cb-model and are already committed" but did not resolve the directionality.
- **Fix:** Made the retrain->rank->eliminate loop GENERIC over an injected `ImportanceRanker` closure. The parity-bearing orchestration (per-step schedule, sub-matrix re-index/remap, survivor partition) stays in cb-train source; the importance computation is supplied by the caller. The oracle test injects the REAL Gate-C importances via cb-model, which is a cycle-exempt DEV-dependency of cb-train — so the full FEAT-05 path (train -> lift -> importance -> eliminate) is exercised end-to-end and oracle-locked. "Modules over crates" (D-6.6-03) is honored (no new crate); the importance methods are consumed (just through injection rather than a direct source call).
- **Files modified:** `crates/cb-train/src/feature_selection.rs`, `crates/cb-train/tests/feature_selection_oracle_test.rs`
- **Commit:** `a55c2c4`, `0602be6`

**Total deviations:** 1 auto-fixed Rule-3 blocking issue (the build-graph direction forced the injected-ranker design; no scope or fidelity weakening — the real importances are oracle-locked end-to-end).
