# Phase 6.6: Advanced Features & Non-Symmetric Trees - Context

**Gathered:** 2026-06-18
**Status:** Ready for planning

<domain>
## Phase Boundary

Deliver the **advanced-feature surface + a second (non-symmetric) tree engine** so all of it matches upstream **catboost 1.2.10** (`thread_count=1`) ≤1e-5, per-stage. This is the largest and riskiest slice of Phase 6. Five requirement strands → four ROADMAP success criteria:

1. **FEAT-03 (SC-1):** Monotone constraints (per-feature +1/-1/0).
2. **FEAT-04 (SC-1):** Feature penalties + per-object penalties (`feature_weights`, `first_feature_use_penalties`, `per_object_feature_penalties`).
3. **FEAT-06 (SC-3):** Alternative grow policies — Lossguide / Depthwise / Region — producing **true non-symmetric trees** with full train + non-symmetric apply + `.cbm`/json round-trip oracle-locked ≤1e-5 (D-10; touches `cb-train` AND `cb-model`, wiring into existing `TNonSymmetricTree*` bindings). The load-bearing structural risk.
4. **FEAT-05 (SC-2):** Recursive feature selection by PredictionValuesChange / LossFunctionChange / ShapValues.
5. **MODEL-05 + MODEL-03/D-12 (SC-4):** Advanced fstr — ShapInteractionValues, PredictionDiff, SAGE — and the deferred **LossFunctionChange importance** (completes MODEL-03, Phase-4 D-12).

**In scope:** monotone constraints (both tree engines, pending upstream-support confirmation), feature/per-object penalties, the unified non-symmetric grow-policy engine (Lossguide/Depthwise/Region) with non-symmetric apply + serialization round-trip, recursive feature selection, and all advanced fstr types (extended to non-symmetric trees).
**Out of scope (deferred — see below):** GPU kernels for any of the above (Phase 7, CubeCL), Python/PyO3 surface for these params (Phase 8). LossFunctionChange remains a CPU/Rust-trait item; Python callback objectives stay in Phase 8.
</domain>

<decisions>
## Implementation Decisions

### Sequencing & risk gating
- **D-6.6-01 (symmetric-features-first):** Ship the features that ride the **existing oblivious grower** first — monotone constraints (symmetric variant) + feature/per-object penalties — to de-risk and build momentum, BEFORE the non-symmetric engine. Mirrors 6.5's incremental gating intent (but inverted: small-features-first here, since the engine is the risk, not a prerequisite).
- **D-6.6-02 (engine as its own gate):** The non-symmetric grow-policy engine (FEAT-06, D-10) is its own **multi-wave gate** after the symmetric features. Non-symmetric apply + `.cbm`/json round-trip is part of this gate (not a follow-on).
- **D-6.6-03 (fstr before feature selection):** Advanced fstr + LossFunctionChange land **before** FEAT-05. FEAT-05's recursive elimination explicitly consumes PVC / LossFunctionChange / ShapValues importance, so it is the **LAST** strand and **blocks on fstr completion**. No oracle-locking FEAT-05 until those importance methods exist.
- **Resulting order:** symmetric features (monotone-sym + penalties) → non-symmetric engine gate (train + apply + round-trip) → non-symmetric monotone variant + advanced fstr (extended to non-symmetric) → FEAT-05 feature selection.

### Grow-policy engine (FEAT-06, D-10)
- **D-6.6-04 (unified leaf-wise engine):** ONE non-symmetric grower mirroring upstream `leafwise_scoring.cpp`, with the split queue / expansion strategy **parameterized by policy** (Lossguide = best-loss leaf, Depthwise = level order, Region = region split). Not three separate growers — matches upstream's shared core, minimizes divergence risk.
- **D-6.6-05 (parallel model variant, oblivious untouched):** Add the non-symmetric tree as a **separate tree variant** (e.g. `enum TreeVariant`/parallel array) reusing the existing `TNonSymmetricTree*` FlatBuffers bindings (schema present today, apply path is currently dead code). The **symmetric train / apply / `.cbm` / json path stays BYTE-IDENTICAL** — the project's established no-regression discipline (gate non-symmetric apply on tree kind). Do NOT refactor oblivious into a general traversal (would risk regressing the locked oblivious oracles).

### Monotone constraints (FEAT-03)
- **D-6.6-06 (transcribe verbatim):** Monotone enforcement = upstream's isotonic projection over leaf values (`monotonic_constraint_utils`), transcribed verbatim; summations route through `cb_core::sum_f64` (D-08).
- **D-6.6-07 (both tree engines — verify-first):** Implement monotone constraints for **both** the oblivious AND the non-symmetric engines. **Research MUST first confirm upstream catboost 1.2.10 actually supports monotone constraints under Lossguide/Depthwise/Region.** If upstream does NOT support a given combination, there is **no ground truth to oracle against** → escalate-don't-weaken (record the gap, do not fabricate a fixture or weaken tolerance), and scope monotone to the supported engines only. The non-symmetric monotone variant **depends on the engine gate (D-6.6-02) landing first**.

### Feature / per-object penalties (FEAT-04)
- **D-6.6-08 (Python-reachable strict):** `feature_weights`, `first_feature_use_penalties`, `per_object_feature_penalties` are public training params in catboost 1.2.10 → **generate fixtures exercising all three** and oracle-lock ≤1e-5 against Python (D-06 path). Only fall back to the instrumented trainer (D-07) if a per-stage internal proves not Python-reachable. Note: the existing `cat_feature_weight()` in `tree.rs` covers only CTR projections — float-feature + per-object penalties are new.

### Advanced fstr (MODEL-05) + LossFunctionChange (MODEL-03 / D-12)
- **D-6.6-09 (LossFunctionChange completes here):** The deferred Phase-4 D-12 LossFunctionChange importance lands this phase, oracle-locked ≤1e-5 against Python (`get_feature_importance(type='LossFunctionChange', data=pool)`).
- **D-6.6-10 (fstr extends to non-symmetric):** Generalize the TreeSHAP / PVC / Interaction recursion AND the new fstr types (ShapInteractionValues, PredictionDiff) to **non-symmetric trees**, and oracle-lock **≥1 non-symmetric fstr case**. Upstream computes fstr on non-symmetric models too — oblivious-only would leave a parity gap.
- **D-6.6-11 (SAGE seed-match strict):** SAGE is sampling-based/stochastic. Reproduce upstream's RNG seeding **exactly** via the persistent instrumented C++ trainer (`CB_INSTRUMENT_LOG`, 6.3 randomized-loss discipline) so sampled importances match ≤1e-5. **Research MUST first confirm `EFstrType::SageValues` is exposed in catboost 1.2.10's Python API.** Fallbacks, in order of preference: (a) seed-match strict; (b) documented weakened tolerance with structural/statistical oracle (6.4 D-6.4-06 precedent) only if seed-matching is impractical; (c) defer SAGE to a later phase if not reachable at all — never fabricate ground truth.

### Oracle & layering discipline (carried forward — not re-litigated)
- **No new crate; modules over crates** (06.3-15 / 6.5 D-05): grow-policy in `cb-train/{tree,boosting}.rs`, monotone/penalty hooks in `cb-train` + `cb-compute/leaf.rs`, feature selection as a NEW `cb-train` module, fstr in `cb-model/{fstr,shap}.rs`, non-symmetric decode in `cb-model/{cbm,json,apply}.rs`.
- **Oracle sourcing:** Python-reachable (catboost 1.2.10 `.venv`) first; instrument the **persistent** trainer (`/tmp/cb_build313` + clang-18) for thin internals (D-06/D-07). Escalate-don't-weaken; never `#[ignore]`, never loosen tolerance, never fabricate. Vendored `catboost-master/` instrumentation patches stay UNCOMMITTED (D-09/D-12).
- **Summation discipline:** all reductions via `cb_core::sum_f64` (D-08); only documented upstream-order scatter cells use raw `+=`.

### Claude's Discretion
- Exact wave/plan granularity within each gate (sequencing principles D-6.6-01..03 are fixed; granularity is flexible).
- Exact in-memory shape of the non-symmetric tree variant (enum vs parallel array) — D-6.6-05 fixes the *property* (oblivious byte-identical, reuse FlatBuffers bindings), not the field layout.
- Recursive-feature-elimination loop structure (`recurse_eliminate`) and whether each importance backend is a separate plan.
- Whether monotone-symmetric and penalties are one plan/wave or split.
</decisions>

## Existing Code Insights

### Reusable Assets
- `crates/cb-model/src/generated/model_generated.rs` — `TNonSymmetricTreeStepNode` + `NonSymmetricStepNodes`/`NonSymmetricNodeIdToLeafId` FlatBuffers fields ALREADY parsed (but discarded) — the apply path is dead code waiting to be wired (D-6.6-05).
- `crates/cb-model/src/{fstr,shap}.rs` — `prediction_values_change()`, `interaction()`, full oblivious TreeSHAP (`shap_values()`) already oracle-locked; generalize the recursion to non-symmetric (D-6.6-10). NB: SHAP currently assumes `scale == 1` (CB_ENSURE_SCALE_IDENTITY) and needs `leaf_weights` captured at train time.
- `crates/cb-train/src/tree.rs::cat_feature_weight()` — existing `(1 + count/maxCount)^(-model_size_reg)` penalty multiplier for CTR projections; the pattern to extend to float-feature + per-object penalties (D-6.6-08).
- Persistent instrumented catboost 1.2.10 trainer (`/tmp/cb_build313` + clang-18, from 06.3-10/17 / 06.5-01) with `CB_INSTRUMENT_LOG` hooks — reuse for SAGE RNG seed-match (D-6.6-11) and any thin penalty internals.
- `crates/cb-oracle/src/{fixture,compare,model_json}.rs` — per-stage oracle harness (borders/splits/leaf/staged/prediction) in place; note text/embedding models forbid JSON export but non-symmetric tabular models DO round-trip `.cbm`/json.

### Established Patterns
- **Keep the locked path byte-identical**: gate new behavior on a discriminator (here: tree kind / grow_policy), as done for `has_ctr` in Phase 5 (D-6.6-05).
- **Unified, policy-dispatched core** over forked implementations (mirrors upstream shared scoring) (D-6.6-04).
- **Escalate-don't-weaken** on thin/stochastic ground truth: instrument the C++ trainer OFFLINE/RUN-ONCE or seed-match, never fabricate / `#[ignore]` / loosen tolerance (D-6.6-11, 06.3/06.5 discipline).
- **No new crate** unless layering forces it — modules over crates (06.3-15 / 6.5 D-05).

### Integration Points
- `BoostParams` (cb-train) gains `grow_policy`, `monotone_constraints`, penalty fields → flows into `tree.rs` grower (symmetric + new leaf-wise) and `cb-compute/leaf.rs` (constrained leaves).
- Non-symmetric grower output → new `TreeVariant` in `cb-model` → `cbm.rs`/`json.rs` decode → `apply.rs` traversal (gated on tree kind) → `fstr.rs`/`shap.rs` recursion.
- FEAT-05 wraps the `cb-train` train loop in an outer recursive-elimination loop consuming `cb-model` importance methods (blocks on fstr, D-6.6-03).
