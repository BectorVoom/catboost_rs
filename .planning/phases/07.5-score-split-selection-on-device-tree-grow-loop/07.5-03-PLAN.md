Land the **full single-tree device-resident grow loop** (D-05 / D-7.5-02) — the genuinely-new host-light driver that mirrors upstream `oblivious_tree_doc_parallel_structure_searcher.cpp:63-158`. Per depth it chains fill (7.3) → scan/update (Plan B) → score+argmin (Plan A) → ONE O(1) ~16-byte BestSplit read-back → partition-split → partition-update, over persistent device buffers threaded through ONE `ComputeClient`. At the leaves it reads back only the `2^depth` part-stats and computes leaf values via `cb-compute::calc_average`.

This is the core integration vertical slice and the central risk of Phase 7.5 (SC-3). The cross-oracle in `kernels/grow_loop.rs` asserts the GPU tree's STRUCTURE matches `cb-train::greedy_tensor_search_oblivious` EXACTLY (the strict bar) and REPORTS leaf-value divergence (the reported tolerance, signed off only in 7.6).

Two genuinely-new kernels land here beyond the driver: `partition_split_kernel` (forward-bit leaf convention == `leaf_index`, parity-critical per Pitfall 6) and `partition_update_kernel` (per-partition reduce == upstream `UpdatePartitionProps`).

Purpose: realize D-05 — a real device-resident loop, NOT a host-orchestrated kernel-offload hybrid — and lock per-tree structure parity.
Output: `partition_split_kernel`, `partition_update_kernel`, `launch_partition_split`, `launch_partition_update`, `grow_oblivious_tree`, and the `grow_loop::single_tree` cross-oracle.

## Phase Goal (MVP — user story)

**As a** developer training CatBoost-rs on a GPU, **I want to** grow a complete oblivious tree entirely device-resident on rocm with only O(1)-per-level host crossings, **so that** I get a GPU-trained tree whose structure exactly matches the CPU path and whose leaf values are within the reported GPU tolerance.

After this plan: the full single-tree device loop runs on rocm AND cross-oracles end-to-end against the CPU `greedy_tensor_search_oblivious` (structure exact, leaf values reported) — the first complete working vertical slice of GPU training.

## Artifacts this phase produces (this plan's portion)

- `partition_split_kernel<F: Float>` (`#[cube]`, `kernels.rs`) — in-place device doc-routing reorder; assigns bit `i` for split `i` (FORWARD bit order, consistent with CPU `leaf_index` `idx |= 1<<i`); bulk routing never crosses to host.
- `partition_update_kernel<F: Float>` (`#[cube]`, `kernels.rs`) — per-partition Σweight/Σtarget reduce (== upstream `UpdatePartitionProps`), in-kernel atomic + f64 finalize.
- `launch_partition_split`, `launch_partition_update` (+ private `_into`) (`gpu_runtime.rs`) — single-client seam, typed guards.
- `grow_oblivious_tree(&client, ...) -> CbResult<GrownTree>` (`gpu_runtime.rs`) — the host-light per-depth driver; persistent handles; O(1) BestSplit read-back per level; final 2^depth part-stats read-back; host leaf value via `calc_average` formula transcription. Returns split sequence + leaf values.
- `crates/cb-backend/src/kernels/grow_loop.rs` (NEW) — cross-oracle harness; `single_tree` test.
- `#[cfg(test)] mod grow_loop;` mount in `kernels.rs`.

| Boundary | Description |
|----------|-------------|
| host driver → device launches | fixture handles + lengths + depth/scaled_l2 cross into the per-depth launch geometry; no external/untrusted input |
| device → host (O(1) per level) | only the ~16-byte BestSplit descriptor and (once) the 2^depth part-stats cross back; bulk data device-resident |
