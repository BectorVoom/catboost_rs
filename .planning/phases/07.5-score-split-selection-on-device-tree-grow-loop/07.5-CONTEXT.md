## Phase Boundary

The **integration** sub-phase of the Phase 7 umbrella: it turns the device-resident
**histogram FILL** kernels of 7.3 (pointwise) and 7.4 (pairwise) into a complete,
**device-resident tree-grow loop**. It ports the **score calcers + split selection**
(`catboost/cuda/methods/kernel/score_calcers.cuh`, `pointwise_scores.cu`,
`split_pairwise.cu`) and wires the whole oblivious-tree training pass —
**candidate generation → histogram fill → scan/update → score → split
selection → partition/leaf update** — to run on-GPU with **minimal host↔device
round-trips**, mirroring upstream CUDA's host-light train loop
(`gpu_data/` + `train_lib/`). This is the umbrella **D-05** decision realized, and
it **closes the GPU-01 kernel surface**.

The umbrella `07-CONTEXT.md` (D-01…D-09) and the four prior sub-phase CONTEXTs
(`07.1`…`07.4`) are the **binding shared context**: the **structural-parity**
fidelity bar (D-01), the **in-kernel atomic** determinism contract with f64
finalization (D-03), the **full device-resident tree-grow loop** mandate (**D-05**
— central to THIS phase: "not a kernel-offload hybrid with host orchestration"),
the **empirical GPU-06 epsilon signed off only in 7.6**, **`rocm`-only validation
never in GitHub Actions** (D-06), `wgpu`/`cuda` **compile-gated** (D-07), and the
**wave-size-agnostic** policy (D-09). The **device-resident histogram handles** of
7.3/7.4 ([[phase74-pairwise-histogram-outcome]]) are the **inputs** this loop
consumes. All of the above are **already locked and NOT re-decided here.** This
phase resolves only the HOW-level choices specific to scoring, split selection,
and the on-device loop orchestration.

**In scope:** GPU-01 (the full on-device-loop integration slice) —
1. **Score calcers** (`score_calcers.cuh`): the score functions the oblivious
   symmetric-tree path needs, structurally mirrored, matching the existing CPU
   reference in `cb-compute/src/score.rs` (`l2_split_score`, `cosine_split_score`,
   `multi_dim_split_score`, and the Solar/LOO/Sat variants already implemented
   there). **L2** and **Cosine** are the default/primary path.
2. **Split selection** (`pointwise_scores.cu` `ComputeOptimalSplits`): reduce the
   device-resident histograms to per-candidate scores and pick the best split,
   device-resident.
3. The **post-fill scan/update** transforms deferred from 7.3/7.4 —
   `ScanPointwiseHistograms`/`UpdatePointwiseHistograms` (pointwise) and
   `ScanPairwiseHistograms`/`BuildBinaryFeatureHistograms`/`UpdatePairwiseHistograms`
   (pairwise) — land here as the bridge from FILL handle → scorer.
4. The **pairwise split scorer** (`split_pairwise.cu`) for the pairwise/ranking
   path, consuming the 7.4 4-channel handle.
5. The **device-resident grow loop** (D-05): candidate gen → fill → scan/update →
   score → argmax split → partition/leaf update over persistent device buffers,
   producing a complete GPU-trained oblivious tree / model.

**Out of scope (deferred):** the **empirical GPU-06 epsilon sign-off + `rocm`
oracle-gate lock-in** → **7.6** (this phase REPORTS divergence, does not sign off).
**Non-symmetric trees and the ordered-boosting GPU path** are NOT the MVP parity
surface — the MVP is the **symmetric oblivious + Plain-boosting** path that matches
the CPU default; non-symmetric/ordered GPU grow is an additive follow-on only if a
downstream consumer needs it. CTR/text/embedding GPU paths are out of scope (CPU
remains the reference for those).

## Implementation Decisions

All decisions below were auto-resolved to the recommended default in `--auto`

### Score-calcer + split-selection scope (SC-1)
- **D-7.5-01:** Port the **score calcers** at structural parity with
  `score_calcers.cuh`, covering the score functions already in the CPU reference
  `cb-compute/src/score.rs` — **L2/NewtonL2** and **Cosine/NewtonCosine** as the
  default/primary path, plus the **SolarL2 / LOOL2 / SatL2** variants that
  `multi_dim_split_score` already dispatches. The CPU `score.rs` functions are the
  **parity oracle** (read-only). Split selection mirrors `pointwise_scores.cu`
  `ComputeOptimalSplits`: reduce device histograms → per-candidate score → argmax,
  device-resident. The **pairwise split scorer** (`split_pairwise.cu`) is included
  for the pairwise/ranking path and consumes the 7.4 4-channel handle
  ([[phase74-pairwise-histogram-outcome]]). Sequence narrowest-first across plans
  (pointwise L2/Cosine end-to-end first, then variants, then pairwise), mirroring
  the 7.1…7.4 incremental pattern — every listed scorer lands in THIS phase.

### Device-resident grow-loop orchestration (D-05 — RESEARCH-FLAGGED)
- **D-7.5-02:** The **full tree-grow loop runs device-resident** (umbrella D-05):
  candidate generation → histogram fill (7.3/7.4 handles) → scan/update → score →
  argmax split → partition update → leaf-value estimation, over **persistent
  device buffers** with **minimal host↔device round-trips** — **NOT** a
  host-orchestrated per-kernel-offload hybrid (SC-2). **The exact CubeCL
  expression of this is the roadmap's research-flag item** — the kernel
  launch-graph / sequencing, persistent-buffer lifetime management, and whether
  **argmax + partition** run as on-device kernels vs a thin bounded host
  reduction are left to **RESEARCH.md + planning**, subject to D-05 (device-
  resident, host-light), D-03 (in-kernel atomics + f64 finalize), and D-09
  (wave-agnostic). MVP target: a single GPU-trained oblivious tree, then a full
  multi-tree boosting pass.

### Scan/update integration (deferred from 7.3 + 7.4)
- **D-7.5-03:** The post-fill transforms explicitly deferred by 7.3 and 7.4 land
  here as the FILL-handle → scorer bridge: `ScanPointwiseHistograms` /
  `UpdatePointwiseHistograms` (pointwise) and `ScanPairwiseHistograms` /
  `BuildBinaryFeatureHistograms` / `UpdatePairwiseHistograms` (pairwise),
  structurally mirroring `pointwise_scores.cu` / `split_pairwise.cu`. They consume
  the FROZEN 7.3 2-channel and 7.4 4-channel device handles unchanged (no host
  round-trip inserted at the seam).

### Boundary placement & additive constraint (SC-4)
- **D-7.5-04:** All cubecl coupling stays **inside `cb-backend`** (D-7.1-04 carried
  through 7.4). **`cb-core` / `cb-model` / `cb-compute` stay byte-unchanged from
  Phase 3–6 form** (SC-4 — note SC-4's frozen list does NOT include `cb-train`).
  The device-resident grow loop is built **self-contained in `cb-backend`** and
  **cross-oracled against** the CPU reference; do **not** entangle `cb-train`/
  `cb-compute` with cubecl. The CPU parity references —
  `cb-train/src/tree.rs` (`greedy_tensor_search_oblivious`,
  `select_best_candidate`, `leaf_wise_grower`), `cb-compute/src/score.rs`,
  `cb-compute/src/leaf.rs`, `cb-compute/src/pairwise_scoring.rs` — are **read-only
  oracles**. Whether a thin **cubecl-free** integration hook is later added to
  `cb-train` is deferred; the MVP keeps the device loop in `cb-backend`.

### CPU cross-oracle (SC-3)
- **D-7.5-05:** Verify a **complete GPU-trained tree/model matches the Rust
  CPU-trained result** within the **reported** GPU tolerance on representative
  fixtures — **per-tree structure** (the split feature+bin decision sequence) AND
  **leaf values**. Cross-oracle the `cb-backend` device loop against the
  `cb-train` CPU `greedy_tensor_search_oblivious` + `cb-compute` leaf estimation
  on the SAME fixture/seed. The harness **REPORTS** observed structure-match and
  max abs/rel leaf-value divergence; it does **NOT** hard-code or sign off the
  GPU-06 epsilon (that is 7.6's job — the D-7.2-06 / D-7.3-04 / D-7.4-05 pattern).
  Fixtures: small synthetic datasets, **multiple trees** (to exercise the
  multi-tree run), f32 + f64, plus a **pairwise/ranking-loss fixture** (exercises
  the 7.4 pairwise scorer). Self-oracle harness clones the
  `kernels/pairwise_hist.rs` / `kernels/pointwise_hist.rs` report-divergence shape.

### Non-determinism budget across a multi-tree run (RESEARCH-FLAGGED)
- **D-7.5-06:** The D-03 in-kernel-atomic accumulation order is non-deterministic
  and its float noise **compounds across a full multi-tree boosting run**
  (roadmap research-flag, second item). Decision: **per-tree STRUCTURE
  (integer split decisions) must match the CPU reference exactly** on the fixtures;
  **leaf VALUES match within the reported float tolerance**. If atomic noise is
  large enough to flip a split argmax (changing structure), that is the tolerance
  boundary to **characterize and REPORT** — 7.6 signs off the epsilon. Mirror
  upstream's determinism approach (e.g. f64 finalization, stable argmax
  tie-breaking) to keep structure decisions robust against atomic jitter.

### Verification gating (carried from D-7.4-07, restated)
- **D-7.5-07:** Verification is **per-crate**, honoring the disk-pressure
  constraint (root ~100% full; full-workspace test links fail): `rocm` gets build +
  **run + cross-oracle** in-env on gfx1100; `wgpu` gets build + run on host;
  `cuda` gets **build/compile-only** (D-07, no NVIDIA hardware). `rocm` GPU tests
  run **locally/manually, never in GitHub Actions** (D-06).

## Specific Ideas

- The pointwise score calcers reduce to per-leaf `AddLeaf(sum, weight)` → score;
  the CPU equivalents are exactly `l2_split_score` / `cosine_split_score` /
  `multi_dim_split_score` in `cb-compute/src/score.rs` — use them as the bit-level
  parity oracle.
- D-05 is explicit that this must be a real device-resident loop ("not a
  kernel-offload hybrid with host orchestration") — the research-flag exists
  precisely because expressing that host-light loop in CubeCL (persistent buffers,
  launch sequencing) is the genuinely-new design work of this phase.
- SC-3 requires matching **per-tree structure AND leaf values** — structure
  (integer split decisions) is the strict bar; leaf values ride the reported float
  tolerance (D-7.5-06).
- The scan/update transforms here are the ones 7.3 and 7.4 explicitly deferred —
  this phase is where the FILL handles finally feed a scorer.
</specifics>
