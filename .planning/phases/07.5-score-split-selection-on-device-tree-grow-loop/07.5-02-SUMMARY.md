# Phase 07.5 Plan 02: Scan/Update Bridge (ScanPointwiseHistograms) Summary

**Device-resident scan/update over the FROZEN 7.3 2-channel histogram handle: a per-(feature,channel)-cube inclusive prefix-sum (reusing `block_scan_kernel` verbatim) turns per-bin (Σder, Σweight) into cumulative "left-of-border" leaf stats, consumed IN PLACE with no host round-trip at the FILL→scan seam, cross-oracled bit-exact (0.000e0) on gfx1100 (rocm) against the host ordered `sum_f64` prefix over n_bins in {2,16,32}.**

## Accomplishments
- `scan_update_pointwise_kernel<F: Float>` (generics-float, `kernels.rs`) — an inclusive prefix-sum over the per-feature bin axis that REUSES `block_scan_kernel`'s wave-agnostic within-plane `plane_inclusive_sum` + Hillis-Steele cross-plane carry VERBATIM (no hand-rolled scan, no warp/wave-size literal — D-09). One cube per `(feature, channel)`; `CUBE_POS` decodes `feature = CUBE_POS/2`, `channel = CUBE_POS%2`; reads the FROZEN 2-channel `(feature*n_bins+bin)*2+channel` layout and writes the cumulative left-of-border stat back into the same layout.
- `launch_scan_update_pointwise` (+ private `_into`, `gpu_runtime.rs`) — fills the FROZEN 7.3 binSums handle via `launch_pointwise_hist2_into` and consumes that `Handle` IN PLACE (no host round-trip at the FILL→scan seam, D-7.5-03); single `&client` threading so the read-back uses the allocating client; reads back ONLY the cumulative result (UPCAST f32→f64 on wgpu via the FROZEN `read_binsums_f64` path).
- `kernels/score_split.rs::scan` self-oracle (cloned report-divergence shape): `cumulative_matches_host_ordered_reference` (device cumulative vs the host ordered `sum_f64` inclusive prefix) + `over_cube_dim_bins_is_typed_error_not_silent_truncation` (the scope guard) — both GREEN on the gfx1100 rocm oracle; the cumulative is BIT-EXACT (0.000e0 abs/rel) with the host reference.
- Typed guards (STRIDE register): empty short-circuit (no 0-len read), `n_bins > CUBE_DIM` → `CbError::OutOfRange` (the tracked cross-cube-carry follow-up), `checked_mul` overflow → `CbError::OutOfRange`, read-back failure → `CbError::Degenerate`.

## Decisions Made
- **Verbatim reuse of `block_scan_kernel` (plan mandate):** the scan kernel transcribes the within-plane `plane_inclusive_sum` + the Hillis-Steele cross-plane carry over per-plane partials exactly as in `block_scan_kernel`, with `inclusive = true` fixed (the scorer reads `left = scan[b]` as the left-of-AND-including-border cumulative). No new scan was hand-rolled.
- **One cube per (feature, channel) axis:** `num_cubes = n_features * HIST_CHANNELS`; the kernel decodes the axis from `CUBE_POS`. This keeps each per-feature bin-axis scan inside a single cube (the `block_scan_kernel` precondition) while scanning both channels (Σder, Σweight) in one launch.
- **Cumulative output reuses the FROZEN binSums layout + read path:** the output buffer is `n_features * n_bins * HIST_CHANNELS` in the same `(feature*n_bins+bin)*2+channel` order, sized via `hist2_binsums_len_checked` and read back via `read_binsums_f64`, so the f32(wgpu)/f64(rocm/cuda) channel split and overflow guard are inherited from the FROZEN 7.3 helpers (one place).
- **`CUBE_POS` is `usize` in this CubeCL version:** the feature/channel decode uses `usize` literals (`CUBE_POS / 2usize`), matching the existing kernels' `best_gain[CUBE_POS]` usage (a `u32` literal failed to compile — fixed during GREEN).

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Test n_bins set restricted to FROZEN 7.3 FILL families (excluded n_bins=1)**
- **Found during:** Task 2 (running the scan test on rocm)
- **Issue:** The plan's Task 1 behavior listed `n_bins=1` as an edge case. The scan/update consumes the FROZEN 7.3 FILL handle, and `launch_pointwise_hist2_into` only supports `n_bins` in {2 (binary), 16 (half-byte), 32/64/128/256 (one-byte non-binary)} — it rejects `n_bins=1` with `CbError::Degenerate("...expects n_bins in {32,64,128,256}...got 1")`. The test panicked on the `n_bins=1` fixture.
- **Fix:** Restricted the passing test's `n_bins` to the FILL families `<= CUBE_DIM`: {2, 16, 32}. The `n_bins=1` degenerate single-border case is not a producible FILL handle, so it cannot reach the scan; the `<= CUBE_DIM` boundary (`n_bins=32 == CUBE_DIM`) is still exercised, and `> CUBE_DIM` (`n_bins=64`) is covered by the typed-error scope guard.
- **Files modified:** `crates/cb-backend/src/kernels/score_split.rs`
- **Verification:** `cargo test -p cb-backend --no-default-features --features rocm score_split::scan` → 2/2 pass on gfx1100, bit-exact.
- **Committed in:** `52326ee` (Task 2 commit)

**2. [Rule 3 - Blocking] `CUBE_POS` indexed with `usize` literals**
- **Found during:** Task 2 (first GREEN compile)
- **Issue:** `CUBE_POS / 2u32` / `CUBE_POS % 2u32` failed to compile (`no implementation for usize / u32`): `CUBE_POS` is a `usize` topology intrinsic in this pinned CubeCL version (the existing `find_optimal_split_kernel` indexes `best_gain[CUBE_POS]` directly).
- **Fix:** Used `usize` literals (`CUBE_POS / 2usize`, `CUBE_POS % 2usize`).
- **Files modified:** `crates/cb-backend/src/kernels.rs`
- **Committed in:** `52326ee` (Task 2 commit)

## Tracked Follow-up (EXPLICIT — not a silent cut)

**Cross-cube scan carry for features with `> CUBE_DIM` bins (RESEARCH A1 / Open Q1).** This plan implements the single-cube `n_bins <= CUBE_DIM` scan path. Features with `> CUBE_DIM` bins — specifically the 8-bit (256-bin) and 7-bit (128-bin) / 6-bit (64-bin) one-byte non-binary families the FROZEN 7.3 FILL supports — need the cross-cube scan carry (each cube's running offset added to the next), the documented 7.2/7.3 forward dependency that `block_scan_kernel` / `launch_block_scan_f64` also defer. Until it lands, `launch_scan_update_pointwise` returns `CbError::OutOfRange` for `n_bins > CUBE_DIM` (verified by `over_cube_dim_bins_is_typed_error_not_silent_truncation`). This is a forward dependency for the full device grow loop (Plan 07.5-03/C) and the score variants (Plan 07.5-05/E) when run on wide-bin features, and is the next prerequisite for scaling the bin axis past CUBE_DIM = 32.

## Issues Encountered

**Pre-existing (out-of-scope) cubecl-cpu/wgpu atomic limitation (carried from Plan A).** The FROZEN 7.3 histogram FILL uses in-kernel `Atomic<F>::fetch_add`, which cubecl 0.10.0 does not implement for the `cpu` (`atomic<f64>`) or `wgpu`-host (`atomic<f32>`) compiler visitors. The scan/update consumes that FILL handle, so the scan test RUN is gated to the gfx1100 `rocm` oracle (D-7.5-07 / D-06), where it passes bit-exact; `wgpu` and `cuda` BUILD cleanly (compile-gated per the plan). This affects the histogram FILL subsystem, not the scan logic, and is recorded for the histogram/toolchain track (not this plan's scope; FIX ATTEMPT LIMIT respected).
