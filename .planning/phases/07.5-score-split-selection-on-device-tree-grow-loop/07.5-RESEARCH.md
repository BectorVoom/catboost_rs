# Phase 7.5: Score/Split Selection & On-Device Tree-Grow Loop - Research

**Researched:** 2026-06-20
**Domain:** GPU device-resident tree-grow orchestration in CubeCL (score calcers, split selection, scan/update transforms, partition update, leaf estimation) at structural parity with `catboost/cuda/methods`
**Confidence:** HIGH (upstream CUDA + CPU oracle + existing 7.1–7.4 seam all read directly in-session; CubeCL launch-model claims CITED from manual)

## Summary

Phase 7.5 is the integration sub-phase that turns the 7.3 (2-channel pointwise) and 7.4 (4-channel pairwise) device-resident histogram FILL handles into a complete device-resident oblivious-tree grow loop. The genuinely-new design work (the roadmap research flag) is NOT new kernel math — every score formula, leaf-value formula, and tie-break rule already exists bit-exact in the FROZEN CPU oracle (`cb-compute/src/{score,leaf,pairwise_scoring}.rs`) and in upstream CUDA (`score_calcers.cuh`, `pointwise_scores.cu`, `split_pairwise.cu`). The new work is the **loop orchestration**: expressing upstream's host-light per-depth sequence (fill → scan/update → score → argmin → partition-split → next depth, with leaf estimation at the leaves) in CubeCL's launch model, keeping device buffers persistent across launches and reading back only a tiny best-split descriptor per level.

The decisive finding is that upstream's own GPU loop (`oblivious_tree_doc_parallel_structure_searcher.cpp:63-158`) is **NOT** a pure on-device loop and was never intended to be: it is a host-driven loop where, per depth, the host submits device kernels (`SubmitCompute` = histogram, `ComputeOptimalSplit` = scan+score+argmin) and reads back exactly ONE `TBestSplitProperties` scalar struct (~16 bytes: FeatureId, BinId, Score, Gain) to take the integer split decision, then issues a device `Split` that updates the device-resident partition/subsets in place. The heavy data (histograms, partition stats, doc routing, derivatives) never leaves the device. This is precisely what D-05 means by "device-resident, minimal host↔device round-trips, NOT a host-orchestrated per-kernel-offload hybrid" — the distinction is between (a) reading back full histograms/partitions every level to score on the host (the forbidden hybrid) and (b) reading back only the O(1) best-split descriptor per level while all bulk data stays device-resident (the mandated design, and exactly what upstream does). **The MVP must mirror (b).** A truly host-launch-free single mega-kernel is neither what upstream does nor expressible cleanly in CubeCL's launch model, and is not what D-05 requires.

**Primary recommendation:** Build the loop as a thin Rust host driver in `cb-backend` that, per depth, calls the existing 7.3/7.4 FILL handle helpers then new `launch_scan_update_*` / `launch_find_optimal_split_*` / `launch_partition_update` / `launch_partition_split` device helpers — all operating on persistent `Handle` device buffers bound to ONE `ComputeClient` for the whole tree — reading back only a small best-split descriptor per level and the final per-partition leaf stats. Sequence plans narrowest-first: pointwise L2 single-split → scan/update integration → full single-tree device loop cross-oracled → multi-tree boosting pass → Cosine + score variants → pairwise scorer + ranking fixture.

### Pattern 1: Score calcer as a `#[comptime]`-selected in-kernel accumulator
**What:** Each upstream calcer (`TL2ScoreCalcer`, `TCosineScoreCalcer`, `TSolarScoreCalcer`, `TLOOL2ScoreCalcer`, `TSatL2ScoreCalcer`) is a tiny struct with `NextFeature`/`AddLeaf(sum,weight)`/`GetScore`/`Combine`. The CPU oracle already encodes each formula bit-exact. In CubeCL, express the calcer choice as a `#[comptime]` enum/flag (the 7.1 `use_plane` / 7.3 comptime-bits precedent), so the per-leaf `AddLeaf` arithmetic is monomorphized — no runtime branch in the inner leaf loop.
**When to use:** the `find_optimal_split` kernel, one comptime specialization per score function.
**Example (formula source — bit-exact CPU oracle, transcribe into the `#[cube]` AddLeaf):**
```rust
// Source: crates/cb-compute/src/score.rs (FROZEN oracle) — transcribe these into the device AddLeaf.
// L2 AddLeafPlain:   avg = sum/(weight + L2);  score += avg * sum         (CITED score.rs:39-55)
// Cosine:            DP += avg*sum; D2(seed 1e-100) += avg*avg*weight; score = DP/sqrt(D2)  (CITED score.rs:73-85)
// SolarL2:  weight>1e-20 ? (-sum*sum)*(1+2*ln(weight+1))/weight : 0       (CITED score.rs:182-197)
// LOOL2:    adj=weight>1?weight/(weight-1):0; adj*=adj; weight>0?adj*(-sum*sum)/weight:0   (CITED score.rs:202-223)
// SatL2:    adj=weight>2?weight*(weight-2)/(weight*weight-3*weight+1):0; weight>0?adj*(-sum*sum)/weight:0 (CITED score.rs:228-247)
```
Note the upstream GPU calcers (`score_calcers.cuh`) keep `Score` in `float` for L2/Solar/LOO/Sat but `double` for Cosine's `Score`/`DenumSqr`; the CPU oracle folds everything in `f64` via `sum_f64`. For SC-3 STRUCTURE parity the device path should fold in **f64** (D-03 f64-finalize) — the `float`-precision upstream detail is a 7.6 epsilon concern, not a structure concern.

### Pattern 2: Host-light per-depth loop with O(1) read-back (the D-05 realization)
**What:** Mirror `oblivious_tree_doc_parallel_structure_searcher.cpp:63-158` exactly: the host loop issues device launches per depth and reads back ONLY the `TBestSplitProperties` descriptor (`FeatureId, BinId, Score, Gain`) to make the integer decision; partition/histogram/doc-routing stay device-resident. Leaf estimation reads back only the `2^depth` part-stats once at the end.
**When to use:** the `grow_oblivious_tree` driver in `gpu_runtime.rs`.
**Example (the upstream loop skeleton, the structural reference):**
```cpp
// Source: oblivious_tree_doc_parallel_structure_searcher.cpp:63-158 (CITED)
for (depth = 0; depth < MaxDepth; ++depth) {
    Gather(groupedByBinObservations, observations, subsets.Indices);   // device
    featuresScoreCalcer->SubmitCompute(subsets, groupedByBinObservations);   // device hist fill
    featuresScoreCalcer->ComputeOptimalSplit(reducedPartStats, ..., scoreBeforeSplit, ...);  // device scan+score+argmin
    bestSplitProp = featuresScoreCalcer->ReadOptimalSplit();   // <-- O(1) read-back, the ONLY per-level crossing
    bestSplit = ToSplit(FeaturesManager, bestSplitProp);       // host integer decision
    TSubsetsHelper::Split(target, ..., bestSplit.FeatureId, bestSplit.BinIdx, &subsets);  // device partition split
    structure.Splits.push_back(bestSplit);
}
leaves = EstimateLeaves(ReadReduce(subsets.PartitionStats));   // one part-stats read-back at the end
```

### Pattern 3: Scan/update over bins reusing the existing `block_scan_kernel`
**What:** `ScanPointwiseHistograms` is a prefix-sum over the per-feature bin axis turning per-bin (Σder, Σweight) into cumulative "left of border b" stats so a candidate split at border b reads `left = scan[b]`, `right = total - scan[b]` (exactly what `FindOptimalSplitSingleFoldImpl` does: `weightRight = part.Weight - weightLeft`, `:259-263`). The existing `block_scan_kernel` / `launch_block_scan_f64` already does an inclusive/exclusive cube-local scan; the bin axis per feature is small (≤256 bins ≤ CUBE_DIM bound after the cross-cube-carry note), so the scan fits the documented single-cube precondition.
**When to use:** the FILL-handle → scorer bridge (D-7.5-03).

### Pattern 4: Stable deterministic argmin tie-break matching upstream (structure robustness, D-7.5-06)
**What:** upstream `FindOptimalSplit`'s block reduction breaks ties by **lower bin-feature index** (`gains[tid] == gains[tid+s] && indices[tid] > indices[tid+s]` keeps the lower index, `pointwise_scores.cu:140-141`). The CPU oracle uses strict first-wins on score in ascending (feature, border) order (`select_best_candidate`, `tree.rs:291-302`). These must agree: the device argmin must keep the **lowest (feature,bin) index on a tie**, which equals the CPU's first-wins on the same iteration order. Encode this tie-break in the device reduce so atomic-jitter on a near-tie still resolves to the same integer split (structure determinism).
**When to use:** the `find_optimal_split` / `select_best_split` block reduction.

### Anti-Patterns to Avoid
- **Reading the full score/histogram/partition buffer to the host each level** to argmin or partition on the host. This is the host-orchestrated hybrid D-05/SC-2 forbids. Only the O(1) best-split descriptor (and the final part-stats) may cross.
- **Allocating fresh device buffers each depth from a fresh client.** Persistent handles must be bound to ONE `ComputeClient` for the whole tree (a `Handle` is bound to its originating client — see the rule documented in `launch_der_binary_into`). Re-resolving `Runtime::client(&device)` for the same device returns the cached pooled client, but explicitly threading one client through the loop is safer and clearer.
- **Comparing `>=` in the split tie-break.** Strict `>` (first-wins) is load-bearing for structure parity (`tree.rs:295`, Pitfall 1).
- **Folding scores in `f32` on the device.** D-03 mandates f64 finalization; f32 folds widen atomic jitter enough to flip a near-tie argmax and break STRUCTURE parity (D-7.5-06).
- **Hard-coding wave size (32/64) anywhere in the reduce stride.** D-09 — shared-mem size is a `#[comptime]` const; stride derives from `CUBE_DIM_X`/`PLANE_DIM`.

## Common Pitfalls

### Pitfall 1: Device argmin tie-break diverges from CPU first-wins → structure mismatch
**What goes wrong:** the GPU picks a different equal-gain split than the CPU, so the tree STRUCTURE differs (SC-3 hard fail) even though leaf values are within tolerance.
**Why it happens:** the block-reduce argmin uses a different tie rule, or folds in f32 so a true tie becomes a tiny non-tie that resolves arbitrarily.
**How to avoid:** mirror upstream's "keep lowest index on equal gain" (`pointwise_scores.cu:140-141`), fold scores in f64, and verify the device argmin equals `select_best_candidate` over the same ascending (feature,bin) candidate order on every fixture.
**Warning signs:** structure-match fails on a fixture with two near-equal candidate gains; flipping f32→f64 or the tie comparator changes the chosen split.

### Pitfall 2: Atomic-order jitter flips a near-tie argmax across a multi-tree run (D-7.5-06)
**What goes wrong:** D-03 non-deterministic atomic summation makes a candidate's score vary run-to-run by ~1e-9 (f64); on a fixture where two candidates' gains are within that jitter, the chosen split flips, and because each tree's residuals feed the next, the divergence compounds.
**Why it happens:** in-kernel atomic add order is not fixed (the accepted D-03 cost).
**How to avoid:** (a) f64 finalization keeps jitter ~1e-9 so genuine ties are vanishingly rare; (b) the cross-oracle fixtures must be designed so the CPU's chosen splits have a clear gain margin (no artificial exact ties) — then structure is robust; (c) if a fixture DOES sit on an argmax-flip boundary, the harness REPORTS it (the structure-flip is the tolerance boundary D-7.5-06 names) and does NOT sign off — 7.6 owns the epsilon.
**Warning signs:** running the same fixture twice yields different structures; a candidate gain margin smaller than the reported leaf-value divergence.

### Pitfall 3: `read_one` on a handle through the wrong client → HIP UB
**What goes wrong:** reading the best-split descriptor (or part-stats) via a client of a different device/runtime than the one that allocated the handle violates a `slice::from_raw_parts` precondition in the HIP IO controller.
**Why it happens:** constructing a fresh client mid-loop for read-back instead of reusing the loop's client.
**How to avoid:** thread ONE `ComputeClient` through the whole tree's loop (the documented `launch_der_binary_into` rule); allocate, launch, and read every handle through it. Never read a 0-length handle (prior-phase MEMORY: "Handle bound to its client, never read 0-len").
**Warning signs:** intermittent HIP errors on read-back; crashes only under `rocm`, not the host CPU backend.

### Pitfall 4: f64 atomics unadvertised on gfx1100 → wrong `AtomicFinalizePath` assumption
**What goes wrong:** gfx1100 EXECUTES the f64 in-kernel atomic but `device_supports_f64_atomic_add` returns `false` (the documented `InKernelAtomicF64Unadvertised` case). A naive helper that branches to a host-sum fallback on `!advertised` would insert a host round-trip mid-loop, breaking device-residency.
**Why it happens:** the capability bit and the actual execution disagree on HIP/gfx1100 (`gpu_runtime.rs:86-93`).
**How to avoid:** the histogram/score/partition device reductions must ALWAYS run the in-kernel atomic (like `launch_pointwise_hist2_handle` does), reporting the path as informational for 7.6 — NOT use it as a host-fallback selector. wgpu (no f64 atomics) is the only place the channel type drops to f32 (RESEARCH A1, 7.3).
**Warning signs:** a host read-back appears in the per-depth loop on rocm; `AtomicFinalizePath::HostSumFallback` returned where device residency was expected.

### Pitfall 5: Pairwise scorer is structurally heavier than pointwise (a linear system, not a calcer)
**What goes wrong:** treating the pairwise split scorer like the pointwise calcer underestimates it — `split_pairwise.cuh` exposes `MakePairwiseDerivatives` (builds a per-leaf linear system `linearSystem` from the 4-channel pairwise histogram), `SelectBestSplit` (argmin over scores), `UpdateBinsPairs`/`FillPairBins`/`ZeroSameLeafBinWeights`/`FillPairDer2Only` (pair-routing + der prep). The score comes from solving the linear system, mirroring `cb-compute/src/pairwise_scoring.rs` (`calculate_pairwise_score`, `pairwise_cholesky_solve`).
**Why it happens:** the pointwise path is a simple per-leaf scalar fold; the pairwise path is a small dense linear-system solve per candidate.
**How to avoid:** sequence the pairwise scorer LAST (D-7.5-01 narrowest-first); port `MakePairwiseDerivatives` → solve (Cholesky, as in `pairwise_scoring.rs`) → `SelectBestSplit`, cross-oracled against `cb-compute/src/pairwise_scoring.rs`'s `reference_pairwise_score`. Use the 7.4 4-channel handle as input unchanged.
**Warning signs:** trying to reuse the pointwise `find_optimal_split` kernel for the pairwise path; missing the linear-system step.

### Pitfall 6: Partition-split doc routing must stay device-resident and match CPU `leaf_index`
**What goes wrong:** the per-object leaf assignment (`leaf_of`) computed device-side disagrees with CPU `leaf_index` (forward bit order, split i → bit i, `tree.rs:272-280`), so leaf membership and thus leaf values differ.
**Why it happens:** upstream's partition update (`TSubsetsHelper::Split`) and the CPU forward-bit leaf index use the same convention but it must be replicated exactly on device; an off-by-one in bit order silently permutes leaves.
**How to avoid:** the device `partition_split_kernel` must assign bit `i` for split `i` (forward order) consistent with `leaf_index`; verify per-object `leaf_of` matches the CPU on every fixture (it is part of the SC-3 structure check).
**Warning signs:** leaf VALUES diverge by far more than the reported float tolerance while individual leaf SUMS look right — a sign leaves are permuted.

## Open Questions

1. **Cross-cube scan carry for features with >CUBE_DIM bins.**
   - What we know: `launch_block_scan_f64` is single-cube only (n ≤ CUBE_DIM=32); 8-bit features have 256 bins.
   - What's unclear: whether MVP fixtures stay ≤32 bins, or the scan-update plan must first land the cross-cube carry (the documented 7.2/7.3 forward dependency).
   - Recommendation: scope the first scan-update plan to ≤CUBE_DIM bins (low-bit-width fixtures), and add a cross-cube-carry sub-task before the 8-bit path; keep the carry as an explicit tracked dependency, not a silent cut.

2. **argmax + partition: device kernel vs thin bounded host reduction (the D-7.5-02 discretion).**
   - What we know: upstream does argmin on-device (block reduce) and reads back the O(1) descriptor; partition-split is on-device.
   - What's unclear: whether the MVP first plan does the across-block argmin on device or reads back the small `resultSize` per-block winners and finishes the argmin host-side (still O(blocks), not O(features*bins)).
   - Recommendation: MVP may finish the across-block argmin host-side over the small per-block descriptor array (O(blocks) read-back is well within "minimal round-trips"); push the across-block reduce fully on-device only if a reviewer requires it. Per-leaf partition-split MUST stay on device (bulk routing).

3. **Pairwise linear-solve location.**
   - What we know: CPU `pairwise_scoring.rs` uses a Cholesky solve per candidate; upstream `MakePairwiseDerivatives` builds the system on device.
   - What's unclear: whether the small per-leaf dense solve is cleanly expressible in `#[cube]` or better done on a bounded host read of the assembled systems.
   - Recommendation: build the system on device (the heavy histogram reduction); the small solve may be host-side over read-back systems if `#[cube]` linear algebra is awkward — decide in the pairwise plan, cross-oracled against `reference_pairwise_score`.
