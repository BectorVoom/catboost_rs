# Phase 07.5: Code Review Report

**Reviewed:** 2026-06-21
**Depth:** standard
**Files Reviewed:** 4
**Status:** issues_found

## Summary

Reviewed the Phase-7.5 device-resident score/split + tree-grow-loop surface in `cb-backend`
(CubeCL): the `find_optimal_split_kernel` (L2/Cosine/Solar/LOO/Sat comptime arms),
`scan_update_pointwise`/`scan_update_pairwise`, `partition_split`/`partition_update`, the
pairwise scorer (`pairwise_hist_*` / `pairwise_make_derivatives` / `select_best_split` + the
host Cholesky path), and the host-light `grow_oblivious_tree` / `grow_boosting_pass` /
`grow_oblivious_tree_pairwise` drivers, plus the `score_split.rs` / `grow_loop.rs` oracle
harnesses.

Overall the real risk surface (device-buffer index math, host-side value-range guards before
device stores, overflow-checked products, typed-error read-backs, the deterministic
lowest-index argmin tie-break, single-`ComputeClient` handle threading) is handled carefully
and consistently. Host-side guards in `launch_pointwise_hist2_into`, `launch_pairwise_hist_into`,
and `launch_pairwise_make_derivatives_into` validate object-id and bin VALUE ranges before any
device store, which is the load-bearing defense for the unmasked kernels — and it is present and
correct. The `(feature*n_bins+bin)*2+chan` pointwise and `*4+histId` pairwise index math matches
the host references bit-for-bit, and the candidate/leaf/2^depth products are uniformly
`checked_mul`/`checked_shl` guarded.

No BLOCKERS found. The findings below are latent correctness gaps (most reachable only via the
not-yet-exercised one-hot / wgpu-f32 / non-power-of-two paths), one cross-kernel sentinel-type
inconsistency, and quality issues. None of them break the fixture-reachable L2/Cosine paths the
suite exercises in-env.

## Warnings

### WR-01: Pairwise one-hot fill double-adds the weight into a single cell (unverified, no oracle)

**File:** `crates/cb-backend/src/kernels.rs:1010-1022` (and the duplicated 8-bit-atomics arm at `:1129-1140`)
**Issue:** In the non-one-hot path each bin contributes the pair weight to TWO DISTINCT channels
(`2*ge` and `2*gt`):
```rust
bin_sums[b1_base + 2*ge].fetch_add(w);
bin_sums[b1_base + 2*gt].fetch_add(w);
```
The one-hot path instead collapses both flags to a single `is_ge` slot and then `fetch_add`s the
SAME cell twice:
```rust
let cell1 = base + b1*4 + 2*is_ge;
let cell2 = base + b2*4 + 2*is_ge + 1;
bin_sums[cell1].fetch_add(w);
bin_sums[cell1].fetch_add(w);   // same cell, added twice
bin_sums[cell2].fetch_add(w);
bin_sums[cell2].fetch_add(w);   // same cell, added twice
```
Accumulating `2*w` into one cell is only correct if upstream's one-hot `Compare` genuinely
double-weights that bucket; there is no self-oracle for `one_hot == true` (every test and every
grow-loop call passes `false`), so this arm is unverified. The doubled-add to an identical cell
is a structural divergence from the non-one-hot sibling and reads like a copy-paste artifact
rather than a deliberate `2*w` accumulation. This is latent (Plan-E-only, not fixture-reachable
today) but will silently produce a wrong pairwise histogram if one-hot is ever enabled without an
oracle.
**Fix:** Add a `one_hot == true` self-oracle in `kernels/score_split.rs::pairwise` (or
`pairwise_hist.rs`) that cross-checks against the upstream one-hot `Compare` semantics, and
confirm whether the intended accumulation is `2*w` into one cell or `w` into two cells. If the
former, leave a comment justifying the double-add; if the latter, write to the two distinct
channels as the non-one-hot arm does.

### WR-02: Cosine denominator can produce NaN on the wgpu f32 channel (1e-100 seed underflows)

**File:** `crates/cb-backend/src/kernels.rs:1935-1938`
**Issue:** The Cosine arm seeds the denominator with `F::new(1e-100)` and divides
`numerator / denominator.sqrt()`:
```rust
let mut denominator = F::new(1e-100);
denominator += left_avg * left_avg * left_w;
denominator += right_avg * right_avg * right_w;
score = numerator / denominator.sqrt();
```
On the wgpu arm `F` is `f32`, where `1e-100` rounds to `0.0`. For a fully-degenerate candidate
(both leaves empty → both `avg == 0`), the denominator becomes exactly `0.0`, `numerator` is
`0.0`, and `0.0 / sqrt(0.0)` is `NaN`. The 1e-100 seed exists precisely to keep this finite, but
it cannot do so in f32. `score_split.rs::variants::degenerate_leaf_guards_yield_finite_not_nan`
runs over `SelectedRuntime` and asserts `s.is_finite()` for every candidate, so under the `wgpu`
feature this guard can fire (or the score silently NaNs in production). The L2/Solar/LOO/Sat arms
gate their division behind `weight > 0` / threshold checks and do not have this hole.
**Fix:** Use an f32-representable seed for the wgpu channel (e.g. `F::new(1e-30)` or a
`#[cfg(feature = "wgpu")]`-selected epsilon), or guard `if denominator > 0 { ... } else { score = 0.0 }`
before the division so a degenerate Cosine candidate yields `0.0`, matching the documented
finite-not-NaN contract.

### WR-03: `find_optimal_split_kernel` minimal-score sentinel (`f32::MIN`) understates the f64 channel range

**File:** `crates/cb-backend/src/kernels.rs:1858` and `:2561` (`select_best_split_kernel`)
**Issue:** The in-kernel argmin seeds the running best with
`let minimal_score = F::new(f32::MIN);` (≈ -3.4e38) even on the f64 channel, while the host
across-block finalize (`gpu_runtime.rs:1557`, `:4142`) seeds with `f64::NEG_INFINITY`. The
Solar/LOO/Sat arms produce scores of the form `(-sum*sum)*adjust/weight`, whose magnitude scales
with `sum²` over a leaf — for a large-N dataset with large summed gradients on the f64 channel, a
score MORE negative than `-3.4e38` is representable in f64 but would fail the kernel's
`score > my_gain` test against the `f32::MIN` seed, so that candidate would never become `my_idx`
and the cube would report `my_idx == n_candidates` (no candidate). In the common case every
candidate clears the sentinel and the argmax (highest score) is unaffected, but the sentinel is a
silent floor that does not match the f64 score range the kernel claims to support. The
clear-margin fixtures keep scores well above the floor, so this is latent, not fixture-reachable.
**Fix:** Seed the kernel sentinel from the channel float's true minimum rather than `f32::MIN`
(e.g. a comptime-selected `f64::MIN` on the non-wgpu channel, or document that the f64 channel's
usable score range is clamped to `> f32::MIN` and assert it host-side).

### WR-04: `partition_update` parity depends entirely on host `leaf_of < n_parts`, but the grow loop never re-validates the device-produced `leaf_of`

**File:** `crates/cb-backend/src/kernels.rs:2300-2325` (kernel) and `gpu_runtime.rs:2238-2246` (driver call)
**Issue:** `partition_update_kernel` does an UNGUARDED atomic store
`part_stats[part * 2].fetch_add(d)` where `part = leaf_of[obj]` is read from a device buffer. The
kernel doc and the `launch_partition_update_into` doc both say the host "validated
`leaf_of[obj] < n_parts` before launch" — but in the grow-loop driver the `leaf_of` handle is
PRODUCED on-device by `partition_split_kernel` and is NEVER read back or validated before being
fed to `partition_update` (D-05 forbids reading it back). The bound therefore holds only by
construction: `partition_split` sets bit `level` for `level in 0..depth`, so `leaf_of < 2^depth ==
n_parts`. That invariant is correct today for `depth == 1`, but the "host validated the range"
claim in the docs is false for the grow-loop path — nothing validates it; it is an unchecked
structural invariant. If the depth>1 forward dependency ever lands and the bit-assignment changes,
an out-of-range `leaf_of` becomes an out-of-bounds device atomic store (UB) with no guard.
**Fix:** Either (a) correct the kernel/launch docs to state the bound is an unchecked structural
invariant of `partition_split` (not a host-validated value range), or (b) bound `n_parts` such
that `part_stats` is allocated for the full `leaf_of` u32 range, or (c) clamp `part` in-kernel
(`if part < n_parts`) — preferred, since it makes the atomic store provably in-bounds without a
host read-back.

### WR-05: `score_split.rs::max_divergence` `debug_assert_eq!` lets a length mismatch silently truncate in release test builds

**File:** `crates/cb-backend/src/kernels/score_split.rs:81-92` and `grow_loop.rs:60-71`
**Issue:** `max_divergence` guards equal lengths with `debug_assert_eq!(device.len(), baseline.len())`
then `zip`s the two slices. In a release-profile test run (`cargo test --release`) the
`debug_assert` is compiled out, and `zip` silently truncates to the shorter slice — so a device
buffer that is shorter than the baseline (e.g. a truncated read-back) would report an artificially
small divergence and the oracle could PASS on wrong data. The callers do separately
`assert_eq!(len, len)` before calling `max_divergence` in most cases, but not all (the
`partition_update` and multi-tree aggregate paths call it after a length `assert_eq!`, so they are
covered; the helper itself is the weak link if reused). This is test-harness robustness, not
production code, but it weakens the oracle that is the phase's correctness evidence.
**Fix:** Make the length precondition a hard `assert_eq!` inside `max_divergence` (it runs only in
tests, where panics/asserts are allowed), or return `(f64::INFINITY, f64::INFINITY)` on a length
mismatch so a truncated read-back fails the bound rather than passing.

## Info

### IN-01: `unreachable!()` in production `launch_pointwise_hist2_into`

**File:** `crates/cb-backend/src/gpu_runtime.rs:1236-1239`
**Issue:** The function ends with `unreachable!("launch_hist2_family! returns ...")` after a macro
that always `return`s. CLAUDE.md prohibits `unwrap()` in production and the file's own doc-comments
repeatedly assert "no `unwrap`/`expect`/`panic`/indexing." `unreachable!()` is a panic macro; it is
genuinely unreachable here (the macro expands to a `return Ok(h)` from whichever `#[cfg]` arm is
active), so it cannot fire, but it is a panic-shaped construct in a file that claims to have none.
**Fix:** Replace with a non-panicking tail, e.g. return a typed
`Err(CbError::Degenerate("unreachable: launch_hist2_family did not return".into()))`, keeping the
"no panic in production" contract literally true.

### IN-02: Magic `2` / `4` channel literals inlined in kernel index math despite named constants

**File:** `crates/cb-backend/src/kernels.rs:672`, `:791`, `:913` (`* 2usize`); `:1009`, `:1035`, `:2396` (`* 4usize`)
**Issue:** The host side names the channel counts (`HIST_CHANNELS = 2`, `PAIR_HIST_CHANNELS = 4`)
and the module docs stress "naming the 2/4 removes the magic literal." The kernels nonetheless
hard-code `* 2usize` / `* 4usize` / `+ 1usize` in the cell-index arithmetic. Since `#[cube]`
kernels cannot import the host `usize` consts directly this is partly unavoidable, but the literal
`2`/`4` in the interleave stride is exactly the Pitfall-2 hazard the docs warn about (a `*2` vs
`*4` confusion across the pointwise/pairwise seams).
**Fix:** Introduce comptime kernel-local constants (`const CHANNELS: usize = 2;`) at the top of
each kernel and use them in the index math, so the stride is named in the one place a future
edit would touch.

### IN-03: `gpu_runtime.rs` is 4401 lines hosting four phases' launch seams

**File:** `crates/cb-backend/src/gpu_runtime.rs:1-4401`
**Issue:** The file's own module doc (IN-03, already tracked) notes it has grown to host the 7.1
reduce/scan, 7.2 der, 7.3 pointwise-hist, 7.4 pairwise-hist, and now 7.5 score/split + grow-loop
seams. At 4401 lines with ~30 public launch functions it is hard to navigate and review; the
single-responsibility split into `gpu_runtime/{der,histogram,score,grow}.rs` is already flagged as

**Fix:** Proceed with the already-scheduled module split (the doc names it); no new action beyond

### IN-04: Cosine/Solar/LOO/Sat score arms duplicate large literal-by-literal blocks in the kernel

**File:** `crates/cb-backend/src/kernels.rs:1905-2014`
**Issue:** The five comptime score arms in `find_optimal_split_kernel` each re-derive the
left/right `avg` and per-leaf term with near-identical structure (the `if left_w > 0` /
`if right_w > 0` ladders repeat verbatim across L2 and Cosine; the Solar/LOO/Sat arms repeat the
`(-sum*sum)*adjust/weight` shape). Because each arm is comptime-monomorphized this is not a
runtime cost, but it is ~110 lines of copy-paste where a left/right helper would halve the surface
and reduce the chance of a left-only edit (a classic asymmetry bug).
**Fix:** Factor a comptime `#[cube]` helper `fn leaf_term(sum, w, lambda, score_fn) -> F` applied
to left and right, so the per-leaf arithmetic lives once and the left/right symmetry is structural.
