## Fixed Issues

### WR-06: `max_divergence` truncates on length mismatch under release profile

**Files modified:** `crates/cb-backend/src/kernels/score_split.rs`, `crates/cb-backend/src/kernels/grow_loop.rs`
**Commit:** `675c4d8`
**Applied fix:** Replaced the `debug_assert_eq!(device.len(), baseline.len())` (compiled
out under release) with an unconditional length check in both copies of the helper; a
mismatch now returns a sentinel `(f64::INFINITY, f64::INFINITY)` divergence so any caller
threshold check fails loudly instead of silently comparing only the common prefix.
Test-only helper; verified via `cargo check -p cb-backend --tests` (cpu).

### WR-04: `partition_update_kernel` unguarded `part_stats[leaf_of[obj]*2]` store

**Files modified:** `crates/cb-backend/src/kernels.rs`
**Commit:** `cbd6c64`
**Applied fix:** Added an in-kernel `if part * 2 + 1 < part_stats.len()` guard around the
two atomic `fetch_add` stores (matching the scan kernel's `cell < bin_sums.len()`
precedent at `kernels.rs:2131`), and corrected the kernel doc which had claimed a
host-side `leaf_of < n_parts` validation that does not actually run (`leaf_of` is produced
device-side by `partition_split_kernel` with no read-back). Pure additive defensive guard;
for depth==1 (the MVP path) the index is always in bounds so it is a no-op for the verified
oracle. **Requires human verification** (device atomic path; not GPU-runnable in-env).

### WR-01: Finite `f32::MIN` argmin sentinel can drop a real candidate for negative-score calcers

**Files modified:** `crates/cb-backend/src/kernels.rs`
**Commit:** `3cb6b00`
**Applied fix:** Changed the argmin seed from `F::new(f32::MIN)` (finite, ~-3.4e38) to
`F::new(f32::NEG_INFINITY)` in both `find_optimal_split_kernel` and
`select_best_split_kernel`. A SolarL2/LOOL2/SatL2 candidate score more negative than
`f32::MIN` previously failed `score > my_gain`, kept `my_idx = n_candidates`, and was
discarded host-side — a latent device-vs-CPU argmin disagreement. The `-inf` seed matches
the CPU oracle's `f64::NEG_INFINITY` and casts to `-inf` in both the f32 (wgpu) and f64
channels, so every finite candidate wins on the first compare. Empty argmin lanes still
hold `(-inf, n_candidates)` and are correctly discarded; they never displace a real
(finite) candidate. **Requires human verification** (kernel logic; not GPU-runnable
in-env, though L2/Cosine — the in-env oracle calcers — are unaffected since their scores
are >=0).

### WR-03 / IN-01: Pairwise winner reported f32-reduced gain; decorative device round-trip

**Files modified:** `crates/cb-backend/src/gpu_runtime.rs`
**Commit:** `b5fc434`
**Applied fix:** In `select_best_split_over_scores`, the across-block argmax now compares
and tie-breaks over the exact host-resident f64 `scores` Vec (looked up by the candidate
index each device block nominates) instead of the device-reduced `best_gains` (which run
in f32 on wgpu and could collapse two host-distinct f64 scores into one f32 tie, flipping
the split versus the CPU oracle's strict-`>` over f64). The device-reduced gains are now
explicitly a structural-only artifact (`let _ = &best_gains;`), addressing IN-01's
"decorative round-trip" observation without removing the device kernel (which the review
framed as an optional larger refactor). **Requires human verification** (pairwise
split-selection on a parity path; not GPU-runnable in-env).

### WR-02: `launch_der_*_into` launches `f64` unconditionally; breaks on a true wgpu backend

**Files modified:** `crates/cb-backend/src/gpu_runtime.rs`
**Commit:** `62e36d4`
**Applied fix:** Took the reviewer-sanctioned typed-reject alternative. The der binary,
unary, and parametric `_into` seams are f64-typed end-to-end (handle layout,
`empty(n * size_of::<f64>())`, `cast_slice::<u8, f64>` read-backs, and the
`grow_boosting_pass_into` f64 read-back); WGSL has no `f64` type so a genuine wgpu backend
cannot JIT them. Each seam now returns a descriptive `CbError::OutOfRange` under
`#[cfg(feature = "wgpu")]` (before any handle is created) instead of surfacing an opaque
JIT crash. `#[cfg_attr(feature = "wgpu", allow(unused_variables, unreachable_code))]` keeps
the wgpu build warning-clean without affecting the rocm/cuda/cpu f64 path. Verified to
compile cleanly under BOTH `cpu` (default) and `--features wgpu`.

> Note: the full f32-channel split (mirroring the histogram seams) was deliberately NOT
> taken — it would touch the der upload, launch, read-back, AND the
> `grow_boosting_pass_into` f64 read-back across the 7.2-owned der path, is unverifiable
> in-env (no wgpu device run; rocm is f64-only), and a partial version would leave wgpu in
> a subtly-wrong state worse than an honest typed reject.

## Skipped Issues

### WR-05: Trailing `border = n_bins-1` no-op split is scored and argmin-eligible (pointwise)

**File:** `crates/cb-backend/src/kernels.rs:1869-1892`, `crates/cb-backend/src/gpu_runtime.rs:1572`
**Reason:** skipped — fix would risk the verified bit-exact in-env oracle parity.
The review itself states "Device and CPU oracle agree (no parity break)"; the trailing
no-op border is currently scored and argmin-eligible identically on BOTH the device kernel
and the host reference, so the in-env rocm oracle passes precisely because both sides
include it. The "force the trailing candidate's score to the sentinel" fix would have to
be applied to BOTH the kernel's `scores[c]`/argmin AND `host_reference_scores` in exact
lock-step, and the per-candidate `scores` buffer is compared element-wise via
`max_divergence` — a sentinel of `-inf` on both sides produces `(-inf) - (-inf) = NaN` in
that comparison (fragile; relies on `f64::max`'s NaN-handling). The alternative (bounding
the loop to `n_bins-1`) shifts the `n_candidates = n_features * n_bins` geometry that is
wired into buffer allocation, the kernel index decode, and the host winner decode
(`gpu_runtime.rs:1572`). Per the environment constraint to not force unverifiable changes
that risk the verified parity (the test suite cannot be linked in-env due to ENOSPC, and
GPU runs require rocm), this upstream-fidelity-only nicety (no proven parity break) is left
for a dedicated change that can be validated against the in-env rocm oracle.
**Original issue:** Pointwise enumerates `0..n_bins`; the last border puts all bins
left/right empty — a non-split that can win, diverging from upstream and from the pairwise
path which correctly uses `n_splits = n_bins-1`.
