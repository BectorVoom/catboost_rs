<objective>
Establish the Phase-4 structural prerequisites that every later plan depends on: capture per-leaf training-document weights in the boosting loop (currently absent and required by SHAP / PredictionValuesChange / Interaction — RESEARCH Pitfall 1), re-home the canonical serializable `Model` into `cb-model` carrying `leaf_weights` and `float_feature_borders`, extend the `cb-oracle::model_json` parser with `leaf_weights`, install `flatc` and commit the generated FlatBuffers bindings (D-01), and stage the offline fixture-generation manifest (D-13).

Purpose: SHAP, PredictionValuesChange, and Interaction all divide by / weight on leaf weights; without them every downstream fstr/SHAP path silently returns zeros (RESEARCH Pitfall 1). The canonical `Model` is the substrate all of serialize / apply / explain operate on. This plan unblocks Waves 2-5.

Output: leaf-weights capture in `cb-train`, `cb-model::Model`, extended `model_json` parser, committed `generated/` bindings, an oracle test locking Rust leaf weights vs upstream, and the fixture-generation manifest.
</objective>
