# Phase 17 Research: Model Export ‚Äî ONNX (EXPORT-01 slice)

## Research Summary

- **Phase goal (full Phase 17)**: export a float-only trained model to ONNX and CoreML matching upstream's guards/correctness, with unsupported (categorical/CTR/text/embedding) models rejected by a typed error `[PROJECT: ROADMAP.md@a82289c Phase 17]`.
- **This research's scope**: EXPORT-01 only ‚Äî ONNX export of a float-only, oblivious, identity-scale model via `ai.onnx.ml` `TreeEnsembleRegressor`/`TreeEnsembleClassifier` (+`ZipMap`), with a typed rejection guard. CoreML (EXPORT-02) and the ORT oracle harness (EXPORT-03) are explicitly out of scope for this pass; findings needed for those later slices are flagged where relevant.
- **Recommended approach**: add a new `export` (sub)module inside the existing `cb-model` crate (NOT a new `cb-export` crate ‚Äî see Crate-Placement Decision), following the exact same "read-only analysis surface over `Model`" shape already established by `fstr.rs`/`shap.rs`/`partial_dependence.rs`. Hand-build the `onnx::ModelProto` tree exactly as upstream's `onnx_helpers.cpp` does (ir_version 3, `ai.onnx.ml` opset 2), using `prost` + a committed, generated-once `onnx.proto` binding (mirroring the project's existing "commit flatc output, never hand-edit" convention for `cb-model/src/generated/`).
- **Most important constraints**:
  - Do NOT hold ONNX export to the ‚â§10‚Åª‚Åµ double bar ‚Äî this is the one phase with a relaxed, export-specific float32 tolerance `[PROJECT: ROADMAP.md@a82289c Phase 17 Context]`.
  - Pin the opset to match upstream exactly: `ir_version=3`, `opset_import{domain="ai.onnx.ml", version=2}` `[VERIFIED: WEB raw.githubusercontent.com/catboost/catboost/master/catboost/libs/model/model_export/onnx_helpers.cpp, accessed 2026-07-17]`.
- **Highest-risk findings** (all elaborated below):
  1. `cb_model::Model` has **no loss-function/objective metadata at all** ‚Äî there is no way, today, to distinguish a binary-classifier model from a regression model (both are `approx_dimension == 1` with an empty `class_to_label`). Upstream's `IsClassifierModel` resolves exactly this ambiguity by reading `ModelInfo["params"]["loss_function"]`, which this port does not store anywhere (`cb_model::Model`, `catboost-rs::Model` facade ‚Äî neither carries it). **This blocks a literal `TreeEnsembleClassifier`+`ZipMap` implementation until a classifier/loss hint is threaded through** (see Open Questions).
  2. `cb_model::Model`'s `ModelSplit` enum has exactly two variants, `Float` and `Ctr` ‚Äî there is **no distinct one-hot-categorical split representation** at the canonical-model layer, and **no text/embedding-feature concept anywhere on `Model`**. This actually SIMPLIFIES the EXPORT-01 guard versus upstream's four separate checks: in this codebase, "float-only" reduces to a single structural predicate (see Recommended Architecture).
  3. This port has **no `ScaleAndBias`/`Scale` concept on `Model`** at all (only a scalar `bias: f64`, equivalent to upstream's `Bias` with `Scale` permanently == 1.0). The "identity-scale" guard is therefore vacuously satisfied by every `Model` value that can currently be constructed ‚Äî worth implementing defensively, but it cannot currently reject anything.
  4. No ONNX/protobuf-writing crate currently exists in the workspace (`prost`/`protobuf` absent from every `Cargo.toml` and `Cargo.lock`) `[VERIFIED: LOCAL grep across crates/*/Cargo.toml, Cargo.lock]`. A new dependency is required; recommendation below.
  5. The `.planning/codebase/STACK.md` vs `ARCHITECTURE.md` crate-placement "disagreement" cited in the roadmap **cannot be corroborated against the current files** ‚Äî neither file mentions onnx/coreml/cb-export today, and git history shows only one commit ever touched either file (unrelated, a security-audit doc) `[VERIFIED: LOCAL git log --all -- .planning/codebase/STACK.md .planning/codebase/ARCHITECTURE.md]`. Treat the roadmap's framing as historical/aspirational context, not a live discrepancy to reconcile from current file content.

## Phase Requirements

### In Scope (this research pass)
- **EXPORT-01**: export a float-only, oblivious, identity-scale model to ONNX via `ai.onnx.ml` `TreeEnsembleRegressor`/`TreeEnsembleClassifier` (+`ZipMap`); categorical/CTR/text/embedding models rejected with a typed error mirroring upstream's guard `[PROJECT: ROADMAP.md@a82289c Phase 17 Success Criterion 1]`.

## Current Project Architecture

### Relevant subsystems and boundaries
- `crates/cb-model/src/model.rs` ‚Äî the canonical, serializable `Model` (re-homed from `cb-train` per Phase 4). This is the type any export function must consume `[CODEGRAPH: crates/cb-model/src/model.rs]`.
- `crates/cb-model/src/apply.rs` ‚Äî the pure-Rust CPU oblivious-tree apply path; documents the exact binarization/leaf-index/accumulate conventions an ONNX-equivalent traversal must reproduce numerically (`RawFormulaVal = bias + Œ£_trees leaf_values[leaf]`).
- `crates/cb-model/src/{fstr,shap,partial_dependence}.rs` ‚Äî the three existing precedents for "read-only analysis surface consuming `&Model`, exposed via `lib.rs` `pub use`, typed `thiserror` error enum, oracle-tested against a pinned upstream fixture." ONNX export should follow this shape verbatim.
- `crates/catboost-rs/src/model.rs` ‚Äî the published facade `Model { inner: cb_model::Model }`; wraps every `cb-model` capability (predict/shap/fstr/partial_dependence/save_cbm/save_json) with facade-level error mapping into `catboost-rs::CatBoostError`. An export entry point would eventually get a facade method here (`Model::export_onnx`), but wiring it is **out of scope for the EXPORT-01 slice itself** unless the spec author chooses to include it.
- `crates/catboost-rs-py/src/{estimator,classifier,regressor,ranker}.rs` ‚Äî PyO3 bindings; every facade capability gets a thin `py.detach`-wrapped method per estimator (`partial_dependence_py` is the newest precedent). Export would follow the same shape in a later slice.
- `crates/cb-oracle/{fixtures,generator}/` ‚Äî the committed-fixture oracle-harness convention; `fixtures/partial_dependence/gen_fixtures.py` is the most recent precedent for a Python-script-driven, pinned-catboost-version fixture generator.

### Existing data/control flow (apply path, the substrate export must replicate)
1. **Binarize** (`apply.rs:binarize_feature`): bin index = count of borders `b` with `raw > b` (strict `>`) `[CODEGRAPH: crates/cb-model/src/apply.rs:49-51]`.
2. **Leaf index**: forward bit order over the tree's ordered `splits` ‚Äî split `i` contributes bit `i` (`cb_train::leaf_index`) `[CODEGRAPH: crates/cb-model/src/apply.rs doc comment]`.
3. **Accumulate**: `raw = bias + Œ£_trees leaf_values[leaf]`, routed through `cb_core::sum_f64` (deterministic ordered summation, D-08) `[CODEGRAPH: crates/cb-model/src/apply.rs]`.

### Existing reusable implementations
- `Model.float_feature_borders: Vec<Vec<f64>>` ‚Äî per-float-feature ascending borders; directly reusable to derive ONNX's `nodes_values`/`nodes_featureids` per split.
- `ObliviousTree.splits: Vec<ModelSplit>`, `.leaf_values: Vec<f64>`, `.leaf_weights: Vec<f64>` ‚Äî everything an ONNX tree-walk needs is already on the type; no new tree-representation work is needed, only a new *serializer* consuming it.
- `Model.bias: f64` ‚Äî directly maps to ONNX's `base_values` (see Recommended Architecture ‚Äî Bias/Scale handling).
- The `thiserror`-based `ModelError`/`PdpError` pattern (`error.rs`, `partial_dependence.rs`) ‚Äî the template for a new `ExportError` (or `OnnxExportError`) enum.

### Current conventions and patterns
- Every (de)serializer / analysis module is a private `mod` in `cb-model/src/lib.rs`, re-exported via a flat `pub use module::{Type, function};` block ‚Äî no nested public module paths are exposed to consumers `[VERIFIED: LOCAL crates/cb-model/src/lib.rs:14-49]`.
- Generated/vendored code lives under `src/generated/`, is committed, and is NEVER hand-edited; the crate's own doc comment states this explicitly for the FlatBuffers bindings `[VERIFIED: LOCAL crates/cb-model/src/lib.rs:51-89]`. This is the direct precedent for how to bring in ONNX protobuf types (see Standard Stack).
- Doc comments cite the exact upstream C++ source file/line as "source of truth" before describing an algorithm (`RESEARCH Pattern N` convention) ‚Äî the spec/plan for export should do the same, citing `onnx_helpers.cpp`/`model_exporter.cpp` line-anchored quotes captured in this research.

## Dependency Analysis

- **Transitive/peer**: `prost`'s own transitive deps (`bytes`, `prost-derive`) are standard, widely-used, no conflict with existing workspace deps (`bytemuck` etc. are unrelated).
- **Runtime/build/system**: none added to the *build* graph if bindings are committed (see above); `protoc`/`protox` is a one-time, out-of-CI generation-time tool (parallel to the `flatc` precedent, which likewise is not invoked by `cargo build`).
- **Compatibility/migration**: none ‚Äî this is new, additive functionality with no existing ONNX code to migrate.
- **Removals**: none.
- **Legitimacy audit for `prost`**: maintained by `tokio-rs` (the Tokio project org), extremely widely adopted across the Rust ecosystem (gRPC/protobuf de facto standard), actively released (0.14.4 as latest stable) `[VERIFIED: CRATES.IO api/v1/crates/prost]`. ## Recommended Architecture and Implementation Pattern

### Prescribed approach
Add `crates/cb-model/src/export/onnx.rs` (new submodule; a sibling `export/mod.rs` or flat `export_onnx.rs` is an equally acceptable naming choice for the plan/spec author) that:
1. Runs the typed guard (see below) against `&Model`.
2. On success, hand-builds a `prost`-generated `onnx::ModelProto` value field-by-field, transcribing `onnx_helpers.cpp`'s `InitMetadata` + `ConvertTreeToOnnxGraph` + `AddTree` **verbatim in structure** (same field names/order is not required, but same semantics/values are).
3. Serializes via `prost::Message::encode` to `Vec<u8>` (or writes directly to a `Write`r), returned to the caller ‚Äî mirroring `save_cbm`'s `(model: &Model, path) -> Result<(), ModelError>` shape, or a `to_onnx_bytes(&Model) -> Result<Vec<u8>, OnnxExportError>` pure-bytes shape (spec author's call; `save_cbm`/`save_json` both take a path, so following that shape for consistency is reasonable, but an in-memory `Vec<u8>` return is also defensible since ONNX Runtime can load from bytes and the oracle test harness will want bytes for a temp-file round trip anyway).

### Component responsibilities
- **Guard** (`fn guard_exportable(model: &Model) -> Result<(), OnnxExportError>`): the single chokepoint mirroring `SerializeFullModelToOnnxStream`'s checks, adapted to what this codebase's `Model` can actually represent (see below) ‚Äî call this BEFORE touching any tree data, exactly as upstream calls its `CB_ENSURE`s before `ConvertTreeToOnnxGraph`.
- **Graph builder** (`fn build_onnx_graph(model: &Model, is_classifier: bool) -> onnx::GraphProto`, or similar): the `AddTree`/`ConvertTreeToOnnxGraph` transcription.
- **Metadata builder** (`fn build_metadata(model: &Model) -> onnx::ModelProto` fields): `ir_version`, `opset_import`, `producer_name`/`producer_version`.

### Data and control flow
1. Guard rejects a non-oblivious model (`!model.non_symmetric_trees.is_empty() || !model.region_trees.is_empty()`), a CTR/categorical model (any `ModelSplit::Ctr` in any tree, OR `model.ctr_data.is_some()`), and ‚Äî defensively, for forward-compatibility ‚Äî a non-identity scale, once/if this port ever grows a `Scale` concept.
2. For each oblivious tree, walk `depth in 0..tree.splits.len()`, but **in REVERSED index order**: `tree.splits[tree.splits.len() - 1 - depth]` is the split node emitted for ONNX depth `depth` (root = the LAST element of `splits`). This reproduces upstream's `TreeSplits[treeStart + (treeSize - 1 - depth)]` reversal exactly `[VERIFIED: WEB onnx_helpers.cpp AddTree]`, and is consistent with this port's own documented forward-bit-order convention (`splits[i]` contributes bit `i` of the leaf index ‚Äî the LOWEST-index split is evaluated LAST/closest to the leaves, so it must map to the DEEPEST ONNX depth, and the HIGHEST-index split is the root) `[CODEGRAPH: crates/cb-model/src/apply.rs doc comment "forward bit order... split i contributes bit i"]`. **This reversal is a load-bearing pitfall** ‚Äî getting it backwards silently produces a structurally-valid but semantically-wrong tree (every prediction still "runs," just against the wrong feature/border per level).
3. Each ONNX node uses the standard complete-binary-tree indexing `nodeIdx`, `2*nodeIdx+1` (false-child), `2*nodeIdx+2` (true-child), branch mode `BRANCH_GT` (`value > border`, matching this port's own `binarize_feature`'s strict `>` semantics exactly ‚Äî no translation needed) `[VERIFIED: WEB onnx_helpers.cpp AddTree]`.
4. Leaf nodes (the final `2^depth` node ids) are filled with `nodes_modes = "LEAF"` and dummy split fields; `leaf_values` are read out in-order ‚Äî this port's `ObliviousTree.leaf_values` is ALREADY in "canonical forward-bit-order," which is the SAME sequential order the ONNX complete-binary-tree leaf walk visits, so **no reordering is needed**, only a straight iteration.
5. `nodes_featureids` = the float-feature index used by the split (`ModelSplit::Float(Split{feature,..})`), used DIRECTLY with no remap ‚Äî for a model that has ALREADY passed the float-only guard, this port's `Split.feature` is the same index space `predict_raw`'s per-feature SoA columns use, which for a float-only model coincides with the flat/original input-column index (there is no categorical interleaving to offset around) `[CODEGRAPH: crates/cb-model/src/apply.rs, crates/cb-model/src/partial_dependence.rs "Float-only scope (index space)" doc comment]`.
6. `base_values` (bias): populate ONLY if `model.bias != 0.0` (mirrors upstream's `!scaleAndBias.IsZeroBias()` gate ‚Äî `TTreesAttributes` only allocates the `base_values` attribute when there IS a bias, so an all-zero-bias model must omit the attribute entirely, not emit a zero) `[VERIFIED: WEB onnx_helpers.cpp ConvertTreeToOnnxGraph + TTreesAttributes constructor]`. For a single-dimension classifier, upstream emits a symmetric `[-bias, +bias]` pair into `base_values` (`class_ids` 0 and 1) ‚Äî this is the binary-classifier-specific asymmetric-bias trick and must be reproduced if/when the classifier path is implemented.
7. Regressor: `op_type="TreeEnsembleRegressor"`, `post_transform="NONE"`, `n_targets=1`, single output `"predictions"`.
8. Classifier (deferred pending the loss-function-hint decision, see Open Questions): `op_type="TreeEnsembleClassifier"`, `post_transform="LOGISTIC"` (1-dim) or `"SOFTMAX"` (>1-dim), a `label` output (int64 or string per `classlabels_int64s`/`classlabels_strings` attribute ‚Äî this port's `class_to_label: Vec<f64>` gives numeric labels only, so `classlabels_int64s` is the only reachable branch unless string labels are added elsewhere), a `probability_tensor` intermediate, then a SEPARATE `ZipMap` node (same `ai.onnx.ml` domain) consuming `probability_tensor` and producing the final `probabilities` output.

### Error, security, and failure behavior
- Every rejection path returns a typed `OnnxExportError` variant (no panic, no `unwrap`/`expect`, no raw indexing ‚Äî workspace-`deny`d lints apply here as everywhere) mirroring the `PdpError`/`ModelError` convention.
- No untrusted input parsing is involved in export (it's read-only over an already-validated in-memory `Model`), so the main security surface is: never silently drop a CTR/non-symmetric/region tree into a corrupted-but-"successful" ONNX file. The guard MUST run to completion before any byte is written (exactly upstream's ordering: guard checks precede `InitMetadata`/`ConvertTreeToOnnxGraph`).

## Do Not Hand-Roll
- **Do not reimplement protobuf wire encoding** ‚Äî use `prost`'s derive-generated `Message::encode`, not a hand-written varint/length-delimited writer.
- **Do not reimplement the oblivious apply/binarize semantics** ‚Äî reuse `Model.float_feature_borders` and the existing documented `binarize_feature` semantics (`raw > border`, strict) as the source of truth for what `BRANCH_GT` must mean; do not derive a new threshold convention.
- **Do not build a second CTR/categorical detector** ‚Äî if a "does this model use CTR/categorical" check is ever needed elsewhere, reuse whatever the guard function here settles on (iterate `ModelSplit::Ctr` across `oblivious_trees`/`non_symmetric_trees`/`region_trees` + `ctr_data.is_some()`) rather than inventing a parallel detector; consider hoisting it to a small shared helper in `model.rs` (e.g. `Model::is_float_only(&self) -> bool`) that BOTH the ONNX and future CoreML guards can call.
- **Do not vendor/hand-transcribe the ONNX `.proto` schema by hand** ‚Äî generate the Rust bindings once from the official ONNX project's `.proto` file via `prost-build`/`protoc` (or `protox`), then commit the output, exactly as `flatc --rust` output is committed today.

## Common Pitfalls and Risks

1. **Reversed split-order pitfall** (HIGH risk). Trigger: naively walking `tree.splits[depth]` in forward index order when building ONNX depth levels. Consequence: every internal-node test attaches to the wrong (feature, border) pair ‚Äî the exported graph is structurally valid (loads fine in ORT) but numerically WRONG, and this would only surface as a failure once EXPORT-03's numeric oracle runs, not at export time. Prevention: implement the `splits[len-1-depth]` reversal explicitly, matching upstream `[VERIFIED: WEB onnx_helpers.cpp AddTree]`, and add a dedicated unit test asserting the mapping on a hand-constructed depth-2+ tree with distinct borders per level (not just an oracle-fixture round trip, so a regression is caught at the STRUCTURE level, independent of numeric drift). Verification: a new `cb-model` unit/integration test comparing `nodes_featureids`/`nodes_values` per depth against the model's own `splits` in the expected reversed order.
2. **Missing classifier/regressor signal** (HIGH risk, blocks classifier support). Trigger: attempting to implement `TreeEnsembleClassifier` for a 1-dim model without a loss-function hint. Consequence: cannot correctly choose `TreeEnsembleRegressor` vs `TreeEnsembleClassifier`+`ZipMap`+`LOGISTIC` for the single most common CatBoost case (binary Logloss classification) ‚Äî the model TYPE (`cb_model::Model`) genuinely cannot answer this question today. Prevention: either (a) scope this EXPORT-01 slice to `TreeEnsembleRegressor` only (any 1-dim model exports as a regressor; multiclass exports as a classifier since dimension>1 is unambiguous), explicitly deferring "binary-as-classifier" to a follow-up slice once a hint mechanism exists, or (b) add an explicit `is_classifier: bool` (or `ExportKind` enum) parameter to the export function signature that the CALLER (who DOES know ‚Äî the facade layer at `fit()` time, or the loader if `model.json`'s stored params are ever threaded through) supplies. Verification: this is a SPEC decision, not a code-verification item ‚Äî flag prominently for the spec author.
3. **Zero-bias `base_values` omission** (MEDIUM risk). Trigger: always emitting a `base_values` attribute (even `[0.0]`) regardless of whether `model.bias == 0.0`. Consequence: a structurally-different attribute set than upstream produces for a zero-bias model; likely harmless numerically (ONNX Runtime treats a `[0.0]` base_value as a no-op) but breaks any BYTE-level/structural comparison against upstream's own export in EXPORT-03. Prevention: gate `base_values` emission on `model.bias != 0.0`, matching `TTreesAttributes`'s conditional attribute allocation. Verification: a test asserting the `base_values` attribute is ABSENT (not present-and-zero) for a zero-bias fixture.
4. **Flat-feature-index assumption for `nodes_featureids`** (MEDIUM risk, only relevant if this port's index-space assumption is ever violated). Trigger: a future extension that allows a model with categorical features intermixed with float features to still be "float-only" for SOME feature slots ‚Äî in that case `Split.feature` (float-feature-local index) would diverge from the "original flat column index" ONNX expects. Consequence: silently wrong `nodes_featureids`. Prevention: for THIS slice, the float-only guard already forbids any categorical/CTR presence, so `Split.feature == flat_feature_index` holds by construction; document this invariant explicitly in the export code so a future loosening of the guard doesn't silently break it. Verification: an oracle/unit test on a model whose float features are NOT feature 0..N-1 contiguous from the pool's perspective (none such exists in current fixtures) is not reachable today ‚Äî note as a structural assumption, not a currently-testable gap.
5. **Legitimacy/staleness of a naive `onnx-protobuf` dependency choice** (MEDIUM risk if the planner picks it without the audit). Trigger: choosing the crates.io `onnx-protobuf` crate for expedience. Consequence: single-maintainer, ~2-year-stale, `rust-protobuf`-family dependency inconsistent with the rest of the ecosystem and the project's own legitimacy-audit bar. Prevention: use `prost` + a self-generated, committed binding instead (see Standard Stack / Dependency Analysis). Verification: N/A (a sourcing decision, not a runtime check) ‚Äî flag in code review / plan sign-off.
6. **Non-oblivious guard gap** (MEDIUM risk). Trigger: forgetting to check `non_symmetric_trees`/`region_trees` emptiness (only checking for `ModelSplit::Ctr` presence). Consequence: a Depthwise/Lossguide/Region model with ALL-float splits would incorrectly pass a guard that only checks split TYPE, not tree VARIANT, and the graph-builder (written only for the oblivious complete-binary-tree shape) would then either panic (denied by lints, so it would have to be a checked-indexing failure surfacing some other way) or silently emit a wrong structure. Prevention: guard checks BOTH conditions ‚Äî no CTR splits AND `non_symmetric_trees.is_empty() && region_trees.is_empty()`. Verification: a rejection test per tree-variant fixture (`fixtures/non_symmetric/*`, `region`-tree fixtures if any exist) asserting the typed error, not a panic.
7. **Opset drift** (LOW risk, but flagged since the roadmap explicitly calls out "pin the opset"). Trigger: generating the vendored `.proto` from a very recent ONNX release and then ALSO defaulting to a recent/"latest" opset version number instead of the upstream-matching `ai.onnx.ml` v2 / `ir_version` 3. Consequence: the exported graph, while valid ONNX, no longer matches what upstream CatBoost emits structurally, complicating the EXPORT-03 comparison story and potentially exercising less-tested code paths in ORT's tree-ensemble op implementations. Prevention: hardcode `ir_version=3` and `ai.onnx.ml` opset `2` in the exporter regardless of which `.proto` schema release generated the Rust message types.

## Crate-Placement Decision

**Recommendation: a submodule inside `cb-model`** (e.g. `crates/cb-model/src/export/onnx.rs`), NOT a new `cb-export` crate.

Rationale:
- The roadmap's cited "STACK.md vs ARCHITECTURE.md disagreement" is **not present in the current files** ‚Äî both were checked directly and neither mentions onnx/coreml/export, and git history confirms neither file has ever been edited to say so (`git log --all` shows exactly one commit touching either file, an unrelated security-audit doc) `[VERIFIED: LOCAL .planning/codebase/STACK.md, ARCHITECTURE.md content + git log --all]`. There is therefore no LIVE, textual disagreement to reconcile ‚Äî only the roadmap's own paraphrase of one, which cannot be independently confirmed.
- Every precedented "add a new read-only analysis surface over `Model`" addition in this codebase (`fstr.rs`, `shap.rs`, `partial_dependence.rs`, plus the (de)serializers `cbm.rs`/`json.rs`) is a `cb-model` submodule, never a new crate, despite some of them (SHAP, interaction fstr) being substantially more complex than a straightforward tree-walk serializer. Consistency with this pattern is a strong default.
- The workspace's `members = ["crates/*"]` glob makes crate-vs-submodule mechanically equivalent effort ‚Äî the deciding factor is code organization, and submodule wins on consistency.
- A new Cargo dependency (`prost`) is not disqualifying: `cb-model` ALREADY unconditionally depends on `flatbuffers` and `serde_json` for its OTHER two serialization formats (`.cbm`, `model.json`) with no feature-gating ‚Äî adding a third format's dependency (`prost`, for `.onnx`) the same (unconditional) way is consistent, though the planner MAY choose to gate it behind a new Cargo feature (e.g. `onnx-export`) if minimizing default compile surface for pure-inference consumers matters more than convention-consistency; this project already has an established feature-flag mechanism in `cb-model/Cargo.toml` (the backend passthrough), so adding one more named feature is low-friction either way.
- CoreML (EXPORT-02, later) will need its OWN protobuf schema (`contrib/libs/coreml/*.proto` upstream) ‚Äî grouping BOTH exporters under one `cb-model/src/export/{onnx.rs, coreml.rs}` mirrors upstream's own `catboost/libs/model/model_export/` directory, which co-locates `onnx_helpers.cpp` and `coreml_helpers.cpp` as siblings in ONE library, not two.

If the planner disagrees (e.g., to keep `cb-model`'s dependency footprint minimal for pure-inference embedders), the fallback is a new `cb-export` crate depending on `cb-model` (read-only, one-directional) ‚Äî mechanically trivial given the glob-based workspace, but breaks the established "one crate per phase 4-era layer" file-organization pattern. Either choice is workable; this research's job is to flag that the STATED disagreement source cannot be verified, and recommend the submodule as the lower-friction, higher-consistency default.
