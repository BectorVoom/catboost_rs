## User Constraints (from CONTEXT.md)

### Locked Decisions

- **D-7.1-01:** Replace each `()` placeholder with its CubeCL runtime type via a dedicated `#[cfg(feature = ...)]` arm mirroring the live `cpu` arm (`pub type SelectedRuntime = cubecl::cpu::CpuRuntime;`). One arm per backend feature; preserve the mutual-exclusion `not(...)` cfg chain so exactly one alias resolves. **Zero runtime dispatch** тАФ compile-time `type` alias only, never a runtime `match`.
- **D-7.1-02:** The CubeCL runtime feature flags (`hip` for ROCm, `wgpu`, `cuda`) are gated on the matching `cb-backend` Cargo feature so a `cpu`-only build never compiles GPU backends. Resolve the exact per-feature cubecl feature set in planning.
- **D-7.1-03:** `cb-compute` stays **cubecl-free**; `cb-core`/`cb-model` stay **unchanged** from Phase 3тАУ6. All cubecl coupling remains inside `cb-backend` only.
- **D-7.1-04:** Primitives are `#[cube]` kernels + launch helpers **inside `cb-backend`** (extend `kernels.rs` / `kernels/` and the `cpu_runtime.rs` launch-helper pattern). Do **not** widen the `cb-compute` `Runtime`/`Float` traits unless a primitive genuinely must be expressed through the trait seam тАФ prefer backend-internal helpers.
- **D-7.1-05:** Primitives are **generic over `F: Float`** using `generics-float` (no hard-coded float types). Parity-critical reduction *finalization* follows the f64 contract where the CPU baseline does; the in-kernel-atomic GPU divergence allowed by umbrella D-03 is the documented exception.
- **D-7.1-06:** The 7.1 slice ports ONLY the foundational surface: block-level **inclusive/exclusive scan** + block/plane **reduction with atomic finalize**, mirroring `cuda_util/kernel/{scan,reduce}.cu(h)` + `cuda_lib`'s reduce/scan. `segmented_scan*`, `segmented_sort*`, `partitions_reduce` are **DEFERRED** to the sub-phase that first needs them.
- **D-7.1-07:** Reductions follow CUDA's in-kernel atomic/plane structure (umbrella D-03), accepting non-deterministic float summation order; run-to-run variance is *reported* in 7.1, *signed off* as the GPU-06 epsilon only in 7.6.
- **D-7.1-08:** Wave-level reductions via CubeCL **plane (subgroup) ops**, no warp-size constant. Shared-memory tree-reduce fallback (width from runtime/comptime, never literal 32/64) if plane ops absent at pinned cubecl. **(Research flag тАФ resolved below: plane ops ARE present at 0.10.0.)**
- **D-7.1-09:** Self-oracle vs the Rust CPU path on representative f32/f64 inputs incl. edge cases (empty, single element, length not a multiple of block/wave size, large N). **Report** observed max abs/rel divergence; do NOT hard-code or sign off the GPU-06 epsilon (7.6's job). Tests in dedicated `_test.rs` files, run on `rocm` in-env.
- **D-7.1-10:** Per-crate verification (disk pressure, root ~100% full): `rocm` = build + run + self-oracle in-env on gfx1100; `wgpu` = build + run on host; `cuda` = build/compile only. `rocm` GPU tests run locally/manually, **NEVER in GitHub Actions**.

## Summary

Phase 7.1 has two deliverables that are well-scoped and de-risked by this research. **(1) Runtime wiring** is mechanical: the three `()` placeholders in `crates/cb-backend/src/lib.rs:38/42/46` become `cubecl::hip::HipRuntime`, `cubecl::wgpu::WgpuRuntime`, and `cubecl::cuda::CudaRuntime`, each gated on its `cb-backend` feature, which in turn forwards to the cubecl facade features `hip`/`wgpu`/`cuda`. All three runtime types, their device types (`AmdDevice`, `WgpuDevice`, `CudaDevice`), and the facade feature names are confirmed present in the locked cubecl 0.10.0 sources. **(2) Device primitives** are `#[cube]` block-scan + block-reduce kernels added to `cb-backend/src/kernels.rs` (plus launch helpers in a new `gpu_runtime.rs` mirroring `cpu_runtime.rs`), generic over `F: Float`, wave-size-agnostic via CubeCL plane ops.

**The single biggest risk тАФ the cubecl-hip тЖФ ROCm 7.1.1 version skew тАФ is RESOLVED.** The lock currently pins `cubecl-hip-sys 7.2.5321100`, but `cubecl-hip 0.10.0`'s manifest only requires `cubecl-hip-sys >= 7.1.5280200`. Both the 7.1.5280200 and 7.2.5321100 sys crates ship a `hip_52802` feature, and the sys `build.rs` **auto-detects** the system HIP patch from `hipconfig --version` (the box reports `7.1.52802` тЗТ patch `52802`, `hipconfig` is on PATH at `/opt/rocm/bin/hipconfig`) and sets `hip_52802` itself. So the version skew is cosmetic: either sys crate will bind to the box's HIP 7.1.1 runtime correctly. The recommended action is to pin the sys crate to `=7.1.5280200` (the version cubecl-hip declares) for reproducibility, via a one-line `cargo update --precise`. No spike is strictly required to *unblock* тАФ but a 1-task build smoke-test (`cargo build -p cb-backend --no-default-features --features rocm`) is the cheap confirmation.

For wave-agnostic reductions, **plane ops exist at 0.10.0** (`plane_sum`, `plane_inclusive_sum`, `plane_exclusive_sum`, `plane_broadcast`, `plane_shuffle_xor`, `PLANE_DIM`, `UNIT_POS_PLANE`), and `client.features().plane.contains(Plane::Ops)` lets the host pass a `#[comptime] use_plane: bool` so the kernel compiles a plane path or a shared-memory tree-reduce fallback with no device-side branch and no literal warp size тАФ directly satisfying D-7.1-08. The CUDA parity target is `kernel_helpers.cuh::FastInBlockReduce` (shared-mem tree-reduce down to warp size, then `WarpReduce` via `__shfl_down_sync`) and `inplace_scan.cuh::InplaceInclusiveScan` (Hillis-Steele shared-memory scan). Note upstream's *host-facing* `ScanVector`/`Reduce` delegate to NVIDIA CUB; the **hand-written in-kernel** primitives are the structural-parity reference, not CUB.

**Primary recommendation:** Pin `cubecl-hip-sys = "=7.1.5280200"`, wire the three runtime aliases, then build the thinnest vertical slice (block reduction тЖТ self-oracle on rocm) FIRST, scan SECOND. Express wave reductions through `plane_*` ops with a `#[comptime]`-gated shared-memory fallback. Use f64 in-kernel for the parity-critical primitive (CPU-baseline contract) and report observed divergence; do not chase the GPU-06 epsilon here.

### Pattern 2: Generic launch helper over `SelectedRuntime` (D-7.1-04)

**What:** A single generic helper that constructs the client, transfers buffers, launches a `#[cube]` kernel, and reads back тАФ parameterized over the runtime so the SAME code path serves cpu/wgpu/cuda/rocm. This generalizes the per-runtime-hard-coded `launch_binary_f64` in `cpu_runtime.rs` (which writes `cubecl::cpu::CpuRuntime` literally) by substituting `crate::SelectedRuntime`.

**When to use:** Every primitive launch helper in `gpu_runtime.rs`.
**Example:**

```rust
// Source: pattern derived from crates/cb-backend/src/cpu_runtime.rs:46-138 (verified existing)
use cubecl::prelude::*;
use crate::SelectedRuntime;

fn launch_block_reduce_f64(input: &[f64], use_plane: bool) -> cb_core::CbResult<Vec<f64>> {
    let n = input.len();
    let device = <SelectedRuntime as cubecl::Runtime>::Device::default(); // AmdDevice/WgpuDevice/...
    let client  = <SelectedRuntime as cubecl::Runtime>::client(&device);

    let in_handle  = client.create(cubecl::bytes::Bytes::from_elems(input.to_vec()));
    let num_cubes  = n.div_ceil(CUBE_DIM).max(1);
    let out_handle = client.empty(num_cubes * std::mem::size_of::<f64>());

    block_reduce_kernel::launch::<f64, SelectedRuntime>(
        &client,
        CubeCount::Static(num_cubes as u32, 1, 1),
        CubeDim { x: CUBE_DIM as u32, y: 1, z: 1 },
        unsafe { ArrayArg::from_raw_parts(in_handle,  n) },
        unsafe { ArrayArg::from_raw_parts(out_handle.clone(), num_cubes) },
        use_plane,  // #[comptime] тАФ resolved from client.features().plane.contains(Plane::Ops)
    );

    let bytes = client.read_one(out_handle)
        .map_err(|e| cb_core::CbError::Degenerate(format!("device read-back failed: {e:?}")))?;
    Ok(bytemuck::cast_slice::<u8, f64>(&bytes).to_vec())
}
```

### Pattern 3: Wave-agnostic reduce тАФ plane op with comptime shared-memory fallback (D-7.1-08, D-09)

**What:** Mirror `FastInBlockReduce`: tree-reduce in shared memory down to plane width, then a plane reduction тАФ but select the plane path vs a pure shared-mem fallback at *comptime* so no warp-size literal and no device branch appears.
**When to use:** The block-reduction primitive's intra-cube fold.
**Example:**

```rust
// Source: cubecl comptime_specialization.md (verified pattern) + Cubecl_plane.md + Cubecl_shared_memory.md
use cubecl::prelude::*;

#[cube(launch)]
pub fn block_reduce_kernel<F: Float>(
    input: &Array<F>,
    output: &mut Array<F>,
    #[comptime] use_plane: bool,
) {
    let tid = UNIT_POS;
    let mut acc = F::new(0.0);
    if ABSOLUTE_POS < input.len() {
        acc = input[ABSOLUTE_POS];
    }

    if use_plane {
        // Wave-size-agnostic: PLANE_DIM is the runtime plane width, never a literal.
        let plane_total = plane_sum(acc);            // one value per plane
        // (combine per-plane partials via shared memory if CUBE_DIM > PLANE_DIM тАФ
        //  width read from PLANE_DIM, still no literal 32/64)
        if UNIT_POS_PLANE == 0 {
            output[CUBE_POS] = plane_total;          // or atomic-add into a global accumulator
        }
    } else {
        // Fallback: shared-memory tree reduce (mirrors cubecl_reduce_sum.md), width = CUBE_DIM_X
        let mut shared = SharedMemory::<F>::new(CUBE_DIM); // CUBE_DIM is a comptime const, NOT 32 literal
        shared[tid as usize] = acc;
        sync_cube();
        let mut s = CUBE_DIM_X / 2;
        while s > 0 {
            if tid < s {
                let v = shared[(tid + s) as usize];
                shared[tid as usize] += v;
            }
            sync_cube();
            s /= 2;
        }
        if tid == 0 { output[CUBE_POS] = shared[0usize]; }
    }
}
```

> **`CUBE_DIM` as a `SharedMemory::new` size:** the size must be comptime-known. The project already uses `const CUBE_DIM: usize = 32` in `cpu_runtime.rs:36`. Pass the cube width as a `#[comptime] cube_dim: u32` arg or use the `CUBE_DIM` topology constant тАФ but the shared-memory array size literal must be a const expression. This is a real constraint to design around (see Pitfall 3).

**What:** When reducing across cubes into a single scalar, each cube's representative `fetch_add`s its partial into a global `Atomic<F>` accumulator тАФ matching CUDA's in-kernel atomic adds and accepting the non-deterministic summation order (reported, not signed off).

### Pattern 5: Block scan (Hillis-Steele) mirroring `InplaceInclusiveScan`

**What:** Inclusive scan = shared-memory stride-doubling (`val += data[tid - s]`, `s = 1,2,4,...`) with `sync_cube()` between stages тАФ the exact structure of upstream `inplace_scan.cuh::InplaceInclusiveScan`. Exclusive scan = inclusive then shift, or use `plane_exclusive_sum` directly per plane. Prefer `plane_inclusive_sum`/`plane_exclusive_sum` for the intra-plane segment, shared-mem for cross-plane carry.
**Example:**

```rust
// Source: cubecl-core-0.10.0/src/frontend/plane.rs:242/275 + inplace_scan.cuh structure
#[cube(launch)]
pub fn block_scan_kernel<F: Float>(input: &Array<F>, output: &mut Array<F>, #[comptime] inclusive: bool) {
    let val = if ABSOLUTE_POS < input.len() { input[ABSOLUTE_POS] } else { F::new(0.0) };
    let scanned = if inclusive { plane_inclusive_sum(val) } else { plane_exclusive_sum(val) };
    if ABSOLUTE_POS < input.len() { output[ABSOLUTE_POS] = scanned; }
    // NOTE: this scans WITHIN a plane only. Cross-plane / cross-cube carry needs a second
    // shared-memory pass (block-level partials) тАФ mirror InplaceInclusiveScan's stride loop.
}
```

### Anti-Patterns to Avoid

- **Literal warp/wave size (`32`/`64`) anywhere in a kernel** тАФ violates D-09. Use `PLANE_DIM` / `CUBE_DIM` const. The shared-mem array *size* may be a comptime const (e.g. the project's `CUBE_DIM = 32` launch geometry), but the *reduction stride width* must derive from `CUBE_DIM_X`/`PLANE_DIM`, not a literal.
- **Runtime `match` over backend** тАФ violates GPU-02. Selection is the `cfg` `type` alias only.
- **Widening the `cb-compute` `Runtime` trait** to add a `scan`/`reduce` method тАФ violates D-7.1-03/04. Keep primitives as `cb-backend`-internal launch helpers.
- **`if`-as-expression in `#[cube]`** тАФ CubeCL requires the if-as-STATEMENT pattern (the existing kernels all follow this; see `Cubecl_conditionals.md`). Initialize a `mut` then overwrite in the branch.
- **Calling a normal Rust fn from inside `#[cube]`** тАФ only `#[cube]` sub-functions are legal inside a kernel (see the error-solution guide).
- **Assuming atomic-float is universally available** тАФ see Pitfall 4. On wgpu it needs `VK_EXT_shader_atomic_float`; gate with comptime + a shared-mem-only finalize fallback.

## Common Pitfalls

### Pitfall 1: Workspace `cubecl` pin forces `features=["cpu"]` onto every backend build
**What goes wrong:** The root `Cargo.toml:34` pins `cubecl = { version = "0.10.0", features = ["cpu"] }`. Feature unification means a `--features rocm` build STILL compiles the cpu backend (and may even resolve `SelectedRuntime` to the cpu arm because `cpu` is enabled), defeating GPU-02's "cpu-free rocm build" intent.
**Why it happens:** Cargo feature unification is additive across the workspace; a feature pinned on a shared dep can't be turned off downstream.
**How to avoid:** Remove the unconditional `features=["cpu"]` from the workspace `cubecl` dep; move backend selection into `cb-backend/Cargo.toml` features (`cpu = ["cubecl/cpu"]`, `rocm = ["cubecl/hip"]`, тАж). Build with `--no-default-features --features rocm`.
**Warning signs:** `cargo tree -e features -p cb-backend --features rocm` shows `cubecl/cpu` enabled; `SelectedRuntime` resolves to `CpuRuntime` under a rocm build.

### Pitfall 2: `cubecl-hip-sys` build.rs panics if `hipconfig` is missing or `hip_xxx` is set manually
**What goes wrong:** Build fails with a panic ("Could not find 'hipconfig'") or ("HIP_XXX feature detected!").
**Why it happens:** The sys build.rs shells out to `hipconfig` (`exec_hipconfig`) and forbids manual feature pinning.
**How to avoid:** Ensure `hipconfig` is on PATH (it is: `/opt/rocm/bin/hipconfig`). Never put `cubecl-hip-sys/hip_xxx` in any feature list. If building in a constrained env, export `HIP_PATH=/opt/rocm-7.1.1` / `ROCM_PATH=/opt/rocm-7.1.1`.
**Warning signs:** build.rs panic mentioning `hipconfig` or `HIP_XXX`.

### Pitfall 3: `SharedMemory::new(size)` requires a comptime-const size
**What goes wrong:** `SharedMemory::<F>::new(dynamic_var)` fails to compile тАФ size must be known at JIT comptime.
**Why it happens:** Shared memory is allocated at kernel-compile time; the size cannot be a runtime value.
**How to avoid:** Use the project's `const CUBE_DIM: usize` (or a `#[comptime] cube_dim` arg) as the array size. The *reduction stride* uses `CUBE_DIM_X` (topology). Pad out-of-range lanes with `F::new(0.0)` (see `cubecl_reduce_sum.md`) so non-multiple-of-cube lengths stay correct.
**Warning signs:** "size must be known at compile-time" / `ExpandElementTyped` errors (see error-solution guide).

### Pitfall 4: Atomic-float add is not universally supported across backends
**What goes wrong:** `Atomic::<F>::fetch_add` on a float compiles but fails/UB on a backend lacking hardware atomic-float (notably wgpu without `VK_EXT_shader_atomic_float`; f64 atomics even rarer).
**Why it happens:** Atomic-float is a hardware/extension feature. `fetch_add` exists for `Inner::Scalar: Numeric` in the frontend, but lowering depends on backend capability. HIP/ROCm supports f32 atomic-add natively; f64 atomic-add is emulated/slower.
**How to avoid:** Gate the atomic-finalize path behind a comptime capability flag; provide a shared-memory-only finalize (one partial per cube, host sums the small partial vector) as the portable fallback. For the **rocm oracle path**, prefer f64 in-kernel for parity (the CPU-baseline contract, D-7.1-05) and accept that f64 atomic-add may be slow тАФ correctness over speed in 7.1. For **wgpu build+run**, the shared-mem finalize avoids the atomic-float extension dependency entirely.
**Warning signs:** wgpu validation error about unsupported atomic operation; rocm f64 atomic perf cliff (not a correctness issue here).

### Pitfall 5: Plane ops absent тЗТ must fall back, but the kernel must still compile on cpu/wgpu
**What goes wrong:** `plane_inclusive_sum` may be a no-op/unsupported on the CPU runtime or a wgpu adapter without subgroups, silently returning wrong values.
**Why it happens:** Plane ops require `Plane::Ops` capability; the CPU runtime and some wgpu adapters lack it.
**How to avoid:** Always query `client.features().plane.contains(Plane::Ops)` and pass `#[comptime] use_plane`. The fallback (shared-mem tree-reduce / shared-mem Hillis-Steele scan) is the correct path when `use_plane == false`. The self-oracle (D-7.1-09) catches a wrong fallback. The **rocm oracle is the gate**; cpu/wgpu only need to build+run, so the fallback's correctness on them is exercised by the same oracle harness.
**Warning signs:** Oracle divergence only on the non-plane path; plane path correct.

### Pitfall 6: `read_one` vs `read` API and f64 cast
**What goes wrong:** Reading back the wrong byte width or using the wrong client method.
**Why it happens:** cubecl 0.10.0 exposes `client.read_one(handle)` (single buffer) тАФ the existing `cpu_runtime.rs:134` uses exactly this. `cast_slice::<u8, f64>` requires the out buffer be f64-sized.
**How to avoid:** Mirror `cpu_runtime.rs` verbatim: `client.empty(num * size_of::<f64>())`, launch with `::<f64, SelectedRuntime>`, `client.read_one`, `bytemuck::cast_slice::<u8, f64>`.
**Warning signs:** length mismatch panics, garbage values.

## Open Questions

1. **Does f64 atomic-add lower correctly on the HIP/ROCm backend at cubecl 0.10.0?**
   - What we know: `fetch_add` exists for `Numeric` scalars; HIP supports f32 atomic-add natively; f64 is emulated on most AMD GPUs.
   - What's unclear: whether cubecl-hip 0.10.0 emits a working f64 atomic-add or errors.
   - Recommendation: The vertical-slice reduce kernel should be authored with a **shared-memory partial + host-side f64 sum** finalize as the default (no atomic-float dependency), and the atomic-finalize variant added as a SECOND kernel behind a comptime flag. This keeps Task 1 unblocked regardless of f64-atomic support, while still delivering the D-03 atomic structure. Resolve empirically in the slice.

2. **Cross-plane carry for the block scan.**
   - What we know: `plane_inclusive_sum` scans within a plane; a full block scan needs a second pass over per-plane partials (mirroring `InplaceInclusiveScan`'s stride loop).
   - What's unclear: cleanest CubeCL expression of the per-plane-partial carry.
   - Recommendation: For the 7.1 foundational slice, scope the scan primitive to **single-cube / within-plane** correctness first (oracle on N тЙд CUBE_DIM), then add the cross-cube carry as a documented follow-up within 7.1 if time allows, else note as the first 7.2/7.3 dependency. The block-reduce primitive is the higher-priority deliverable (histograms need reduce before scan).
