All fixes were verified with `cargo check -p cb-backend --tests` (default `cpu`
feature), which compiled clean. The atomic-kernel and capability-assertion test code
compiles under `cpu` but only executes on the `rocm` in-env oracle, consistent with
the existing atomic oracle convention.

## Fixed Issues

### WR-01: In-kernel atomic-reduce kernel is unexercised on the in-env gfx1100 oracle

**Files modified:** `crates/cb-backend/src/kernels/reduce.rs`
**Commit:** c95df3f
**Applied fix:** Added a `run_atomic_reduce` helper that launches
`block_reduce_atomic_kernel` DIRECTLY (bypassing the capability gate in
`launch_block_reduce_atomic_f64`) plus a new test
`block_reduce_atomic_kernel_direct_matches_cpu_sum` that drives the in-kernel
`acc[0].fetch_add` finalize unconditionally (32 runs, both plane and shared-mem
paths). This exercises the atomic kernel on gfx1100 even though HIP does not
advertise f64 atomic-add, matching the reviewer's recommended fix.
**Requires human verification:** the oracle math is a self-oracle; confirm the
direct-launch test actually passes on the in-env rocm device (it runs on `rocm`,
not `cpu`).

### WR-02: Capability fallback invites a silent atomic-to-deterministic mode switch

**Files modified:** `crates/cb-backend/src/kernels/reduce.rs`, `crates/cb-backend/src/gpu_runtime.rs`
**Commit:** ed9b8bc
**Applied fix:** The capability-gated oracle test now asserts the returned
`AtomicFinalizePath` matches the device's advertised f64 atomic-add capability, so a
silent fallback on a device expected to support atomics surfaces as a test failure.
Added a "Caller contract (best-effort atomic, NOT guaranteed atomic)" doc section to
`launch_block_reduce_atomic_f64` warning callers not to discard the returned path.

### WR-03: gpu_runtime.rs mixes three phases' surface in one module

**Files modified:** `crates/cb-backend/src/gpu_runtime.rs`
**Commit:** 29b23af
**Applied fix:** Took the reviewer's interim alternative ("update the module doc to
list the actual contents") rather than the risky full module split. Added a "Module
contents (WR-03)" doc section inventorying the Phase 7.1 reduce/scan primitives, the
Phase 7.2 der seam, and the Phase 7.3 histogram seam, and noting the planned
`gpu_runtime/der.rs` + `gpu_runtime/histogram.rs` split as follow-up.
**Note:** the full single-responsibility module split remains a future refactor.

### WR-04: Reduce/atomic kernels assume a power-of-two CUBE_DIM

**Files modified:** `crates/cb-backend/src/gpu_runtime.rs`, `crates/cb-backend/src/kernels.rs`
**Commit:** 9d59afd
**Applied fix:** Added `const _: () = assert!(CUBE_DIM.is_power_of_two(), ...)` in
`gpu_runtime.rs` alongside the existing `CUBE_DIM <= BLOCK_REDUCE_SHMEM` guard, so a
future non-power-of-two launch width is a compile error rather than a silent dropped
top element. Documented the precondition next to `BLOCK_REDUCE_SHMEM` in `kernels.rs`.

### IN-01: Dead sync_cube() after the cross-plane carry read in block_scan_kernel

**Files modified:** `crates/cb-backend/src/kernels.rs`
**Commit:** cfaf928
**Applied fix:** Removed the post-read `sync_cube()` and replaced it with a comment
explaining the read depends on the Hillis-Steele loop's trailing barrier, not a
barrier after the read.
**Requires human verification:** this removes a device barrier in a `#[cube]` kernel.
The reviewer states it is unneeded on the single-plane oracle path; confirm the scan
oracle still passes on the in-env rocm device.

### IN-02: max_divergence indexes device[i] over baseline.len() instead of zipping

**Files modified:** `crates/cb-backend/src/kernels/scan.rs`
**Commit:** 54c2366
**Applied fix:** Rewrote `max_divergence` to `for (d, b) in device.iter().zip(baseline)`
with a `debug_assert_eq!` length precondition, matching the already-hardened sibling
(commit 252c33a / 07.3 IN-03).

### IN-03: Oracle assert tolerances are hand-tuned magic literals

**Files modified:** `crates/cb-backend/src/kernels/reduce.rs`, `crates/cb-backend/src/kernels/scan.rs`
**Commit:** cf48494
**Applied fix:** Hoisted the f32/f64 relative and absolute tolerances into named
`const`s (`F32_REL_TOL`, `F32_ABS_TOL`, `F64_REL_TOL`, `F64_ABS_TOL_TIGHT`,
`F64_ABS_TOL_LARGE_N`) in each oracle module and replaced every bare literal
assertion with the named const, so the 7.6 epsilon sign-off edits one place per
module. Values preserved exactly (1e-3 f32, 1e-9 f64 rel, 1e-12 tight, 1e-6 large-N).

### IN-04: Stale Cargo.toml comment calls GPU arms "inert placeholders until Phase 7"

**Files modified:** `crates/cb-backend/Cargo.toml`
**Commit:** a31699e
**Applied fix:** Updated the dependency comment to state that as of Phase 7.1 the
`wgpu`/`cuda`/`rocm` arms resolve to real CubeCL runtimes
(`WgpuRuntime`/`CudaRuntime`/`HipRuntime`), matching the `lib.rs` `SelectedRuntime`
module docs. TOML re-validated.

### IN-05: Empty-input short-circuits of the production launch helpers are unverified

**Files modified:** `crates/cb-backend/src/kernels/reduce.rs`, `crates/cb-backend/src/kernels/scan.rs`
**Commit:** e6c5fa2
**Applied fix:** Added an assertion that `launch_block_reduce_f64(&[])` returns
`Ok(vec![])` to the reduce edge-case test, and a new
`block_scan_empty_input_short_circuits` test asserting `launch_block_scan_f64(&[], _)`
returns `Ok(vec![])` for both inclusive and exclusive flags — exercising the
production empty short-circuits rather than only the host baseline.
