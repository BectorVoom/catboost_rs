reduce variant with the observed run-to-run variance reported.

that 7.2–7.5 build on (D-7.1-06).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| host → device | Host-validated f64/f32 buffers cross to the GPU; scan/atomic kernels read them by index. |
| device → host | Scan output and the atomic-accumulator scalar read back via `client.read_one`; a failed read must not masquerade as data. |
| cross-cube atomic | Multiple cubes `fetch_add` into one global accumulator; order is non-deterministic (accepted per D-03). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-7.1-01 | Tampering/DoS (V5 input validation) | `block_scan_kernel` / atomic reduce device read | mitigate | Bounds-guard `ABSOLUTE_POS < input.len()` on every device read; zero-pad out-of-range lanes; host validates buffer length before launch. |
| T-7.1-02 | Tampering | `gpu_runtime.rs` scan/atomic read-back | mitigate | Map `client.read_one` errors to typed `CbError::Degenerate` (WR-05); never a silent zero/all-zero buffer. |
| T-7.1-05 | Information disclosure (non-determinism) | atomic-finalize cross-cube add | accept | Run-to-run float-order variance is inherent to D-03 in-kernel atomics; REPORTED in the oracle, signed off as the GPU-06 epsilon only in 7.6. Not a correctness defect; accepted per umbrella D-03. |
| T-7.1-04 | Memory safety | `unsafe { ArrayArg::from_raw_parts }` / `Atomic` out buffer | mitigate | Confined to `cb-backend` launch helpers; lengths derived from validated host buffers; atomic accumulator is a fixed length-1 buffer. |
| T-7.1-SC | Tampering | crates installs | accept | No new packages introduced in this plan (extends Plan-01 code); RESEARCH Package Legitimacy Audit already verified all cubecl crates OK. |
</threat_model>
