<domain>
## Phase Boundary

The first additive GPU sub-phase of the Phase 7 umbrella. It delivers two things:

1. **Runtime wiring** — replace the inert `()` placeholders in `cb-backend`'s
   `SelectedRuntime` cfg-alias (`crates/cb-backend/src/lib.rs:36-46`) with the
   real CubeCL GPU runtime types for `wgpu`, `cuda`, and `rocm`, keeping
   selection purely compile-time (zero runtime `match` over backends, GPU-02).
2. **Device primitives** — the wave-size-agnostic **scan + reduction + device
   memory** primitives that every later GPU kernel (7.2–7.5) builds on, ported
   at structural parity from `catboost/cuda/cuda_lib` + `cuda_util` using
   CUDA-style in-kernel atomic/plane reductions (D-03 of the umbrella).

The umbrella `07-CONTEXT.md` (D-01…D-09) is the binding shared context — the
fidelity bar (full structural parity), determinism contract (match CUDA
in-kernel atomics), validation strategy (`rocm`-only, never in GitHub Actions),
and wave-size policy (strictly wave-agnostic, no warp-size constants) are
**already locked there and are NOT re-decided here**. This phase only resolves
the HOW-level implementation choices specific to wiring + primitives.

**In scope:** GPU-02 (compile-time backend selection), GPU-04 (`wgpu`
builds/runs on dev machines), GPU-05 (`cuda` compile-gated), GPU-01 (the scan +
reductions slice only — grad/hess, histograms, score/split, and the on-device
grow loop are owned by 7.2–7.5).

**Out of scope (owned by later sub-phases):** gradient/hessian + targets (7.2),
pointwise histograms (7.3), pairwise histograms (7.4), score/split + the full
device-resident tree-grow loop (7.5), the empirical GPU-06 epsilon sign-off and
the `rocm` oracle gate lock-in (7.6).

## Implementation Decisions

All decisions below were auto-resolved to the recommended default in `--auto`
mode. They are HOW-level choices within the boundary the umbrella already
locked; none reopen D-01…D-09. Renumbered with a `7.1-` prefix to avoid
collision with the umbrella's D-01…D-09.

### Runtime cfg-arm wiring (GPU-02/04/05)
- **D-7.1-01:** Replace each `()` placeholder with its CubeCL runtime type via a
  dedicated `#[cfg(feature = ...)]` arm, mirroring the existing `cpu` arm
  (`pub type SelectedRuntime = cubecl::cpu::CpuRuntime;`). One arm per backend
  feature; the mutual-exclusion `not(...)` cfg chain already in `lib.rs` is
  preserved so exactly one alias resolves. **Zero runtime dispatch** — selection
  stays a compile-time `type` alias, never a runtime `match`.
- **D-7.1-02:** The CubeCL runtime **feature flags** (`hip` for ROCm, `wgpu`,
  `cuda`) are gated on the matching `cb-backend` Cargo feature so a `cpu`-only
  build never compiles GPU backends. The workspace currently pins
  `cubecl = { version = "0.10.0", features = ["cpu"] }`
  (`Cargo.toml:34`); research/planning resolves the exact per-feature cubecl
  feature set (the lock already carries `cubecl-hip 0.10.0` transitively).
- **D-7.1-03:** `cb-compute` stays **cubecl-free** and `cb-core`/`cb-model`
  stay **unchanged** from their Phase 3–6 form (additive boundary, SC-1). All
  cubecl coupling remains inside `cb-backend` only.

### Primitive ownership & boundary
- **D-7.1-04:** Scan/reduction/memory primitives are implemented as `#[cube]`
  kernels + launch helpers **inside `cb-backend`** (extending the existing
  `kernels.rs` / `kernels/` pattern and the `cpu_runtime.rs` launch-helper
  pattern). The abstract `Runtime`/`Float` traits in `cb-compute` are **not
  widened** unless a primitive genuinely must be expressed through the trait
  seam — keep the trait frozen, prefer backend-internal helpers (additive
  boundary).
- **D-7.1-05:** Primitives are **generic over `F: Float`** and use
  `generics-float` (no hard-coded float types, per AGENTS.md). Parity-critical
  reduction *finalization* still follows the project's f64 contract where the
  CPU baseline does; the GPU divergence allowed by umbrella D-03 (in-kernel
  atomics) is the deliberate, documented exception.

### Primitive slice & parity structure (D-03)
- **D-7.1-06:** The 7.1 primitive slice ports the **foundational** scan + reduce
  surface that downstream kernels depend on: block-level **inclusive/exclusive
  scan** and block/plane **reduction with atomic finalize**, mirroring the
  structure of `catboost/cuda/cuda_util/kernel/{scan,reduce}.cu(h)` and
  `cuda_lib`'s `reduce`/`scan`. Specialized variants —
  `segmented_scan*`, `segmented_sort*`, `partitions_reduce` — are **deferred to
  the sub-phase that first needs them** (histogram/score phases), not ported
  speculatively here.
- **D-7.1-07:** Reductions follow CUDA's **in-kernel atomic/plane** structure
  (umbrella D-03), accepting non-deterministic float summation order; the
  resulting run-to-run variance is *reported* in 7.1 and *signed off* as the
  GPU-06 epsilon only in 7.6.

### Wave-size-agnostic expression (D-09)
- **D-7.1-08:** Wave-level reductions are expressed through CubeCL **plane
  (subgroup) ops** rather than any hard-coded warp/wave-size constant, so the
  same kernel is correct on wave32 (gfx1100, in-env) and wave64 silicon. If the
  pinned cubecl 0.10.0 lacks a needed plane op, fall back to a **shared-memory
  tree-reduction** that reads its width from a runtime/`comptime` value — still
  no literal 32/64 constant. (Research flag: confirm plane-op coverage at
  0.10.0.)

### Self-oracle harness (SC-4)
- **D-7.1-09:** Primitives self-oracle against the **Rust CPU path** on
  representative inputs — random f32 and f64 vectors plus edge cases (empty,
  single element, length not a multiple of the block/wave size, large N). The
  harness **reports** the observed max absolute and relative divergence as the
  current tolerance; it does **not** hard-code or sign off the GPU-06 epsilon
  (that is 7.6's job). Tests live in dedicated `_test.rs` files (source/test
  separation), run on `rocm` in-env.

### Build-gating verification (disk-pressure constraint)
- **D-7.1-10:** Verification is **per-crate**, honoring the standing
  disk-pressure constraint (root ~100% full; full-workspace test links fail):
  `rocm` gets build + **run + self-oracle** in-env on gfx1100; `wgpu` gets
  build + run on the host (no ROCm/CUDA needed, GPU-04); `cuda` gets **build/
  compile only** behind its feature gate (no NVIDIA hardware here — untested
  locally, D-07). `rocm` GPU tests are run **locally/manually, never in GitHub
  Actions** (standing constraint, umbrella D-06).

### Claude's Discretion
- Concrete kernel decomposition, block/launch geometry, shared-memory sizing,
  the exact CubeCL plane-op API used, and the precise per-feature cubecl feature
  set are left to research + planning — subject to umbrella D-01 (structural
  parity), D-03 (atomic reductions), D-09 (no warp-size constants), and the 7.1
  decisions above.

<specifics>
## Specific Ideas

- "match cuda in catboost" — the umbrella's guiding instruction. For 7.1 it
  means the scan/reduction primitive *structure* mirrors
  `cuda_util/kernel/{scan,reduce}.cu(h)`, not merely the numeric result.
- Start narrowest: wire the runtimes + the foundational scan/reduce slice and
  self-oracle them; resist porting the whole `cuda_util` primitive set up front
  (segmented/sort variants follow demand in 7.3–7.5).
</specifics>

## Deferred Ideas

- **Segmented scan / segmented sort / partitions_reduce primitives** — ported
  later, in the histogram/score sub-phases that first consume them (D-7.1-06).

  existing per-call client pattern for primitives.

  (D-09) but real wave64 hardware is out of reach here.
- **`wgpu`/`cuda` as validation gates** — explicitly not this phase (D-07);
  `wgpu` = build+run only, `cuda` = compile only.
