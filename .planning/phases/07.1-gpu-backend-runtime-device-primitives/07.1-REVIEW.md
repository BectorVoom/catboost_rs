# Phase 07.1: Code Review Report

**Reviewed:** 2026-06-20
**Depth:** standard
**Files Reviewed:** 7
**Status:** issues_found

## Summary

Reviewed the Phase-7.1 GPU backend runtime alias (`lib.rs`), the generic launch
helpers (`gpu_runtime.rs`), the `#[cube]` kernels (`kernels.rs`), and the two
self-oracles (`reduce.rs`, `scan.rs`). The project rules (compile-time backend
selection, generics-float kernels, wave-size-agnostic strides, no production
`unwrap`/indexing, typed read-back errors) are largely honored — the `cfg`-gated
`type SelectedRuntime` alias has zero runtime `match`, kernels are `F: Float`
generic, and read-back failures map to `CbError::Degenerate`.

The dominant defect is a **silent correctness gap in `launch_block_scan_f64`**:
it launches `ceil(n/CUBE_DIM)` cubes but `block_scan_kernel` only scans WITHIN a
single cube, so for `n > 32` it returns mathematically wrong prefix sums with no
error and no clamp — and the oracle only exercises `n <= 24`, so the bug is
latent. A second BLOCKER is an out-of-bounds device write risk in the reduce/scan
shared-memory paths when the launched `CUBE_DIM_X` exceeds the comptime
`BLOCK_REDUCE_SHMEM` size. Several warnings cover the focal-loss `pt -> 0`
domain hazard, the atomic accumulator not being re-zeroed across the oracle's 32
runs, and the lq hessian `q < 2` singularity.

## Critical Issues

### CR-01: `launch_block_scan_f64` silently returns wrong results for `n > CUBE_DIM`

**File:** `crates/cb-backend/src/gpu_runtime.rs:140-177`
**Issue:** The helper computes `num_cubes = n.div_ceil(CUBE_DIM).max(1)` (line 150)
and launches that many cubes (line 155). But `block_scan_kernel`
(`kernels.rs:761-838`) performs the prefix scan only WITHIN a single cube — its
own doc (`kernels.rs:748-754`) states it "is correct WITHIN a single cube
(N ≤ CUBE_DIM)" and that the cross-cube carry is "NOT performed here". For any
`n > 32`, cube 1 scans elements 32..63 starting from 0 (ignoring the running
total of cube 0), etc. The returned vector is the wrong prefix sum, yet the
function reports `Ok(...)` with no error and no clamp. The contract violation is
invisible because the `scan.rs` oracle only tests `n <= 24` (single cube), so CI
never hits the broken multi-cube path. This is exactly the "silent wrong result"
failure the phase rules forbid (a scan that masquerades as valid).

**Fix:** Enforce the documented single-cube precondition with a typed error
instead of silently launching multiple cubes:
```rust
let n = input.len();
if n == 0 {
    return Ok(Vec::new());
}
if n > CUBE_DIM {
    return Err(CbError::Degenerate(format!(
        "launch_block_scan_f64 supports n <= {CUBE_DIM} until the cross-cube \
         carry lands (Open Q2, 7.2/7.3); got n = {n}"
    )));
}
// ...num_cubes is now always 1...
```
(Document the limit in the public doc comment as a hard precondition, not just a
scope note.)

### CR-02: Reduce/scan shared-memory write can exceed `BLOCK_REDUCE_SHMEM` if launched cube width != 32

**File:** `crates/cb-backend/src/kernels.rs:619-633` (and `706-719`, `791-794`)
**Issue:** `BLOCK_REDUCE_SHMEM` is a fixed comptime `32` (line 22). The fallback
tree reduction writes `shared[tid as usize]` for every unit `tid in 0..CUBE_DIM_X`
(line 620) and reads `shared[(tid + s) as usize]` with `s = CUBE_DIM_X / 2` (lines
622, 625). `CUBE_DIM_X` is the RUNTIME launched cube width, not the comptime
shared size. The two are kept equal only by convention (every caller passes
`CubeDim { x: 32 }`). If any caller — now or in 7.2/7.3 — launches with
`x > 32`, units `tid >= 32` write `shared[tid]` out of the 32-slot allocation
(device-side OOB write / UB), and the plane path's `partials[PLANE_POS]`
(line 599) has the same exposure when the plane count exceeds 32. Nothing in the
kernel asserts `CUBE_DIM_X <= BLOCK_REDUCE_SHMEM`.

**Fix:** Make the coupling explicit and defensive. Either (a) size the shared
memory from a single shared comptime that the launch geometry also consumes so
they cannot drift, or (b) add an early bounds guard in the kernel:
```rust
// guard every shared write/read against the comptime allocation
if (tid as usize) < BLOCK_REDUCE_SHMEM {
    shared[tid as usize] = acc;
}
```
At minimum, add a `debug_assert!`/comptime check in the launch helpers that
`dim.x as usize <= BLOCK_REDUCE_SHMEM` so a future wider launch fails loudly
rather than corrupting device memory.

## Warnings

### WR-01: Focal kernels evaluate `ln(pt)` / `powf(1-pt, ...)` where the negative-label clamp leaves `pt` un-floored

**File:** `crates/cb-backend/src/kernels.rs:147-160` and `186-202`
**Issue:** `p` is clamped to `[1e-13, 1-1e-13]` (line 147). For the positive label
`pt = p` (floored away from 0), but for the negative label `pt = 1 - p`
(lines 151/192). When `p` is at its clamp ceiling `1-1e-13`, `pt = 1e-13` > 0, OK.
But `factor = powf(1 - pt, gamma)` (line 158) with `pt` near 1 gives `powf` of a
near-zero base — and `F::powf(0, gamma)` for non-integer gamma plus `ln(pt)` are
the genuine hazards. The clamp keeps `pt` in `[1e-13, 1-1e-13]`, so `ln(pt)` and
`powf(1-pt, .)` are finite, which is the intended mitigation — however the doc
claims the clamp protects `ln`/`powf` for BOTH branches, and that is only true
because `pt` is always one of `p` or `1-p`, both inside the clamp. This holds, but
it is fragile and undocumented that `1 - pt` (line 158/198) is also bounded. Flag
for an explicit test asserting finiteness at saturated logits for BOTH labels.
**Fix:** Add an oracle case driving `approx -> +/-inf` (saturated logit) for both
`target=0` and `target=1` and assert `der1`/`der2` are finite; document that the
single clamp on `p` transitively bounds `pt` and `1-pt`.

### WR-02: Atomic accumulator is re-allocated per call but the oracle's 32 reruns rely on that — and a non-zero residual would silently corrupt the sum

**File:** `crates/cb-backend/src/gpu_runtime.rs:235` and `kernels/reduce.rs:208-215`
**Issue:** `launch_block_reduce_atomic_f64` allocates a fresh zero-initialized
accumulator each call (line 235), which is correct. But the only thing keeping the
oracle's 32 repeated calls correct is that fresh allocation; there is no assertion
that `acc` starts at 0.0 on the device. If a backend ever recycled the handle
(`client.create` returning pooled non-zeroed memory), every run would accumulate
on top of stale data and the `fetch_add` finalize would silently over-count. The
test loop (reduce.rs:211-215) reuses the same `input` and overwrites `path` each
iteration but never checks that successive sums are self-consistent before the
aggregate assertion.
**Fix:** After `client.create(...vec![0.0_f64])`, this is fine, but add a guard
test that explicitly reads back the accumulator equals the single-shot result, and
consider zeroing via `client.empty` + an explicit memset kernel if pooling is ever
suspected. At minimum assert per-run that `(sum - baseline).abs()` is bounded
inside the loop (reduce.rs:211) so a stale-accumulator run fails immediately, not
just in aggregate.

### WR-03: `lq_hessian_kernel` produces `inf`/`NaN` for `q < 2` via `powf(|r|, q-2)` at `r == 0`

**File:** `crates/cb-backend/src/kernels.rs:281-294`
**Issue:** `pow_term = F::powf(abs_loss, qv - F::new(2.0))` (line 291). For
`q < 2` the exponent `q-2` is negative, so when `target == approx` (`abs_loss == 0`)
this is `0^(negative) = +inf`, and `der2 = -q*(q-1)*inf`. The doc (lines 276-279)
says this is "Newton-clean only for `q >= 2`" and that the fixture pins `q = 2.0`,
but the kernel accepts any `q` from the length-1 array with no guard. A caller
passing `q = 1.5` gets `inf` hessians that poison the host-side Newton step. The
gradient kernel (lq_gradient, line 264) has the same `powf(abs_loss, q-1)` hazard
for `q < 1`.
**Fix:** Either document `q >= 2` as a hard precondition validated host-side
before launch (return `CbError` for `q < 2`), or clamp `abs_loss` to a small floor
before the negative-exponent `powf` (matching the focal clamp discipline) so the
kernel never emits non-finite hessians.

### WR-04: `block_reduce_kernel` plane path reads `partials[p]` for un-written slots when `num_planes` underestimates active planes

**File:** `crates/cb-backend/src/kernels.rs:597-613` (and atomic twin 690-704)
**Issue:** `partials` SharedMemory is written only by `UNIT_POS_PLANE == 0`
representatives (line 598-600), keyed by `PLANE_POS`. Unit 0 then folds
`partials[0..num_planes]` where `num_planes = ceil(CUBE_DIM_X / PLANE_DIM)`
(line 606). The shared array is never zero-initialized, so if the actual number of
planes that wrote a slot is fewer than `num_planes` (e.g. a partial final plane
whose `UNIT_POS_PLANE == 0` lane is an idle out-of-range lane that still ran the
write — which it does, since the zero-pad happens at load, line 584-587, but the
plane-representative write is unconditional for `UNIT_POS_PLANE==0`), the slot
holds a valid (possibly zero-padded) plane_total. This is correct at
`CUBE_DIM == PLANE_DIM` (1 plane), but the reliance on every `0..num_planes` slot
being written by a live representative is unverified for the multi-plane case the
fallback is meant to generalize.
**Fix:** Zero-initialize the `partials` shared array (or only fold over slots
proven written) before the `sync_cube`, and add an oracle case on hardware where
`CUBE_DIM > PLANE_DIM` (e.g. force `use_plane` with a 64-wide cube on wave32) to
prove the multi-plane fold, rather than relying on the wave32 `num_planes == 1`
collapse.

### WR-05: `device_supports_f64_atomic_add` uses `as_type_native_unchecked` — capability mismatch surfaces as wrong path, not error

**File:** `crates/cb-backend/src/gpu_runtime.rs:184-192`
**Issue:** `<Atomic<f64> as CubePrimitive>::as_type_native_unchecked()` (line 187)
is an `_unchecked` API. If the runtime cannot represent the native f64 atomic type,
this can return a placeholder type and the subsequent `atomic_type_usage(ty)`
query may report `Add` capability that the kernel then cannot actually honor,
causing `block_reduce_atomic_kernel`'s `fetch_add` (kernels.rs:728) to fail or
miscompile at launch rather than falling back. The fallback is gated on this query
being correct.
**Fix:** Confirm against the CubeCL atomic capability API that the checked variant
exists; if `_unchecked` is the only path, wrap the atomic launch so a launch/JIT
failure is caught and routed to `HostSumFallback` (defense in depth), rather than
trusting the capability bit alone.

## Info

### IN-01: `histogram_scatter_kernel` is only ever exercised under `cpu`, contradicting the "all backends" mount intent

**File:** `crates/cb-backend/src/kernels.rs:868-869`
**Issue:** `mod scatter` is gated `#[cfg(all(test, feature = "cpu"))]` and the
harness names `CpuRuntime` directly (scatter.rs:28), so the scatter kernel — though
compiled under every backend — has zero GPU-path test coverage, unlike `reduce`
and `scan` which run over `SelectedRuntime`. Not a bug, but the scatter primitive
is unverified on rocm/wgpu.
**Fix:** Consider migrating the scatter harness to `SelectedRuntime` like the
reduce/scan oracles so it runs in-env on gfx1100.

### IN-02: Repeated `device`/`client` construction and identical `count`/`dim` blocks across the three launch helpers

**File:** `crates/cb-backend/src/gpu_runtime.rs:86-102, 146-160, 218-244`
**Issue:** The `Device::default()` + `client(&device)` + `CubeCount::Static` +
`CubeDim { x: CUBE_DIM, y:1, z:1 }` boilerplate is duplicated verbatim in all
three helpers (and again in both oracles). Minor duplication; a small
`fn launch_geometry(n: usize) -> (CubeCount, CubeDim, usize)` would centralize it
and reduce the risk of one helper drifting (relevant given CR-01/CR-02).
**Fix:** Extract a shared geometry helper.

### IN-03: `read_one` error is formatted with `{e:?}` — opaque for callers

**File:** `crates/cb-backend/src/gpu_runtime.rs:118, 175, 257`
**Issue:** Read-back failures are correctly mapped to `CbError::Degenerate`
(good — satisfies the typed-error rule), but the message embeds the raw `Debug`
of the backend error, which may be terse/opaque. Acceptable, but consider a
distinct `CbError` variant (e.g. `CbError::Device`) so callers can distinguish a
device-transport failure from a numeric degeneracy.
**Fix:** Add a dedicated device-error variant if the error taxonomy allows.

## Orchestrator Resolution (2026-06-20, code-review gate)

**Both BLOCKERs fixed** in commit `8a3b1eb`:
- **CR-01 (fixed):** `launch_block_scan_f64` now returns `CbError::Degenerate` for `n > CUBE_DIM` instead of silently launching multiple cubes and returning wrong prefix sums. The documented single-cube precondition (cross-cube carry = 7.2/7.3 forward dep, Open Q2) is now enforced in code, not just prose.
- **CR-02 (fixed):** `BLOCK_REDUCE_SHMEM` made `pub(crate)`; a `const _: () = assert!(CUBE_DIM <= kernels::BLOCK_REDUCE_SHMEM)` compile-time coupling now makes any future launch-width/shared-mem drift a **build error** rather than a device-side OOB write.

**Post-fix verification (authoritative rocm gate, D-7.1-10 — gfx1100):**
- `kernels::scan` rocm oracle: **4/4 pass**.
- `kernels::reduce` rocm oracle (incl. atomic-finalize variant + variance report): **4/4 pass**.
- All four feature builds compile (`cpu`/`wgpu`/`cuda`/`rocm`); cpu test-target compiles.

**Finding recorded (not a kernel bug):** The `scan`/`reduce` oracles **fail/hang on the cubecl `CpuRuntime` backend** while passing on the real gfx1100 GPU. Root cause is the experimental cubecl CPU-JIT runtime's incomplete support for the plane/shared-mem primitives — the identical kernels are numerically correct on the authoritative rocm gate. Per the phase's locked validation strategy (D-06 / D-7.1-10: rocm is the SOLE GPU oracle; cpu/wgpu/cuda are compile-gated build targets), this is a documented CpuRuntime limitation, not a phase blocker. The Rust-CPU oracle baseline is `cb-core::sum_f64`, which is unaffected.

The 5 WARNINGs / 3 INFO remain as advisory (latent, non-gate-reachable) follow-ups for 7.2–7.6.
