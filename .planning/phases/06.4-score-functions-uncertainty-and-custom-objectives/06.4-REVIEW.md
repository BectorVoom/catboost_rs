# Phase 06.4: Code Review Report

**Reviewed:** 2026-06-18
**Depth:** standard
**Files Reviewed:** 23
**Status:** issues_found

## Summary

Reviewed the Phase 06.4 surface: the five GPU-only `EScoreFunction` variants
(LOSS-09), RMSEWithUncertainty (LOSS-08), the uncertainty prediction types
(LOSS-06), and the `CustomObjective`/`CustomMetric` trait pair (LOSS-07), plus
the Builder facade wiring. The code is heavily commented, defensive (checked
`.get`, no `unwrap` in production), and most numeric paths are oracle-gated.

The most serious finding is a **silent correctness defect in the Newton score
functions**: `EScoreFunction::NewtonL2` / `NewtonCosine` are documented to
require a der2-hessian-filled `sum_weight` slot (`reduce_leaf_stats_newton`),
but no production call path ever produces such stats — every split scorer feeds
the first-order weight-count reduction. As a result the two Newton score
functions silently compute their FIRST-ORDER counterparts, and the
`reduce_leaf_stats_newton` primitive built for them is dead code. These score
functions are not reachable through the shipped Builder facade (which pins Cosine
and does not surface `score_function`), and their tests are self-oracles that
construct der2-filled stats by hand and so never exercise the broken wiring —
which is exactly why the gap is invisible to the current suite.

Secondary findings concern an inconsistency in the MultiQuantile backend error
handling (silently swallows a device failure, contradicting the WR-05 discipline
applied everywhere else), a builder/doc mismatch on the custom-objective default
leaf method, and several robustness / dead-code items.

## Critical Issues

### CR-01: Newton score functions silently degrade to first-order; der2-fill primitive is dead code

**File:** `crates/cb-train/src/tree.rs:50-67`, `crates/cb-train/src/tree.rs:417-449`, `crates/cb-compute/src/histogram.rs:146-170`

**Issue:** `split_score` and its doc-comment (tree.rs:46-49) state that for
`EScoreFunction::NewtonL2` / `NewtonCosine` "the caller must supply `leaves`
whose `sum_weight` slot already carries the summed positive der2 hessian (via
`cb_compute::reduce_leaf_stats_newton`)". But the only producer of per-leaf
stats on every production scoring path is `multi_dim_candidate_score`
(tree.rs:434-447), which unconditionally calls
`reduce_leaf_stats(leaf_of, der1_d, weight, n_leaves)` — the first-order reducer
whose `sum_weight` is the per-object weight count, NOT the der2 hessian. A
workspace grep confirms `reduce_leaf_stats_newton` is referenced only by its own
definition, the `pub use` re-export, and a doc-comment — it is never invoked in
any non-test code:

```
crates/cb-compute/src/histogram.rs:146  (definition)
crates/cb-compute/src/lib.rs:34         (re-export)
crates/cb-train/src/tree.rs:48          (doc-comment only)
```

Consequences:
1. Selecting `NewtonL2` produces the exact L2 score (NewtonL2 reuses the L2
   formula on first-order stats), and `NewtonCosine` produces the exact Cosine
   score — the second-order distinction is silently lost. A user asking for a
   Newton split score gets a Gradient/first-order split score with no error.
2. `reduce_leaf_stats_newton` is dead code shipped as a public API.
3. The `score_functions_test.rs` "self-oracle" hides this: it constructs
   `der2_filled` `LeafStats` by hand (score_functions_test.rs:208-221, 240-256)
   and asserts `NewtonL2(stats) == L2(stats)` — which is trivially true for ANY
   stats and does not test that the production scoring path actually fills der2.

**Fix:** Either wire the der2-hessian fill into the scoring path when
`score_function` is a second-order function, or remove the unsupported variants
from the trainable surface and delete the dead primitive. Minimum viable wiring:

```rust
// in multi_dim_candidate_score, when score_function is second-order:
fn multi_dim_candidate_score(/* ... */ der2: &[f64], /* ... */) -> f64 {
    let per_dim_leaves: Vec<Vec<LeafStats>> = (0..approx_dimension)
        .map(|d| {
            let base = d * n_objects;
            let der1_d = der1.get(base..base + n_objects).unwrap_or(&[]);
            if score_function.is_second_order() {
                let der2_d = der2.get(base..base + n_objects).unwrap_or(&[]);
                let wd2: Vec<f64> = der2_d.iter().zip(weight).map(|(&d2, &w)| d2 * w).collect();
                reduce_leaf_stats_newton(leaf_of, der1_d, &wd2, n_leaves)
            } else {
                reduce_leaf_stats(leaf_of, der1_d, weight, n_leaves)
            }
        })
        .collect();
    multi_dim_split_score(score_function, &per_dim_leaves, scaled_l2)
}
```

If the intent is "these five are GPU-only and not trainable on CPU yet", then
`validate_leaf_method` / `Loss::validate` (or a `score_function` gate) should
REJECT `SolarL2`/`NewtonL2`/`NewtonCosine`/`LOOL2`/`SatL2` at train time rather
than silently produce a wrong (or first-order) split score, and the dead
`reduce_leaf_stats_newton` should be deleted (CLAUDE.md: "Dead code must be
deleted, not commented out" — the same principle applies to never-called
exports).

## Warnings

### WR-01: MultiQuantile backend swallows a device read-back failure as a zero gradient

**File:** `crates/cb-backend/src/cpu_runtime.rs:877-880`

**Issue:** Every other launcher in this file propagates a CubeCL read-back
failure as a typed `CbError` (the WR-05 discipline, e.g. lines 134-136, 444-446)
specifically because "mapping it to a zero buffer would masquerade as a valid
all-zero derivative vector, silently producing a degenerate (no-gradient) tree".
`compute_multiquantile_gradients` violates exactly that rule:

```rust
match launch_quantile_f64(approx_d, target, alpha_d, delta) {
    Ok(der1_d) => der1.extend_from_slice(&der1_d),
    Err(_) => der1.extend(std::iter::repeat(0.0_f64).take(n)),   // swallows the error
}
```

A device failure on dimension `d` silently yields an all-zero gradient for that
quantile dimension, training a degenerate tree instead of surfacing the failure.

**Fix:** Make `compute_multiquantile_gradients` return `CbResult<(Vec<f64>, Vec<f64>)>`
and propagate the launch error with `?`, matching the other launchers and the
WR-05 rationale already pasted verbatim into this file.

### WR-02: Custom objective default leaf method is Gradient (ignores der2), contradicting documented Newton default

**File:** `crates/cb-compute/src/custom.rs:52-54`, `crates/cb-train/src/boosting.rs:939-1016`, `crates/catboost-rs/src/builder.rs:99`

**Issue:** The `CustomObjective` trait doc states "the default leaf-estimation
method for a custom objective is Newton (der2 is provided), matching the
RMSE/Logloss defaults" (custom.rs:52-54), and `runtime.rs:489` repeats "the
default leaf method is Newton/1-iter (der2 is provided)". But there is no
`Loss::Custom` arm in `validate_leaf_method`, and the Builder default
`leaf_method` is `LeafMethod::Gradient` (builder.rs:99). So a `Loss::Custom`
trained through the Builder uses **Gradient**, which ignores the user-supplied
der2 entirely. The user provides a hessian that is silently discarded, and the
documented Newton default never takes effect through the public facade.

**Fix:** Either (a) make the leaf method default to Newton when
`loss == Loss::Custom` (mirroring the multiclass/RMSEWithUncertainty gates in
`validate_leaf_method`), or (b) correct the trait/enum docs to state that the
custom objective uses the caller-selected leaf method (Gradient by default) and
that der2 is consumed only under Newton. The current state is a behavior/doc
contradiction that will surprise any user relying on der2.

### WR-03: RMSEWithUncertainty per-dim bias is not stored in the Model; scalar `Model.bias` drops dim-1

**File:** `crates/cb-train/src/boosting.rs:2017-2039`, `crates/cb-train/src/boosting.rs:3498-3504`, `crates/cb-model/src/apply.rs:320-373`

**Issue:** RMSEWithUncertainty starts from a genuine per-dimension bias
`[mean, 0.5*log(var)]` (boosting.rs:2025-2036), but the trained `Model` stores
only the scalar `bias` (= dim-0 mean); the dim-1 log-scale starting bias is
dropped (boosting.rs:2021-2024 acknowledges this). The standard apply entry
points (`predict_raw_multi` / `predict_raw_cat`) seed EVERY dimension with that
one scalar (apply.rs:401), which is WRONG for dim 1. The only correct path is
`predict_raw_multi_biased`, and the per-dim bias must be re-read from
`model.json`'s `scale_and_bias[1]` OUTSIDE the Model abstraction (the oracle test
does exactly this at uncertainty_predict_test.rs:60-73). A caller who loads an
RMSEWithUncertainty model and calls the ordinary `predict_raw_multi` will get a
silently wrong dim-1 (and therefore a wrong `exp(2*log-scale)` variance) with no
error. The model abstraction does not faithfully round-trip a multi-dim-bias
model.

**Fix:** Store a per-dimension bias on `Model` (e.g. `bias_per_dim: Vec<f64>`
defaulting to `[bias]`) so `predict_raw_multi` is correct for any multi-dim-bias
model, and have the loader populate it from `scale_and_bias[1]`. Until then, the
ordinary multi-dim apply functions should at least be documented as unsafe for
RMSEWithUncertainty, and ideally the loader should refuse to drop a non-trivial
per-dim bias silently.

### WR-04: `apply_prediction_type(RmseWithUncertainty, ...)` returns empty on an odd-length input instead of erroring

**File:** `crates/cb-model/src/predict.rs:130-143`

**Issue:** When the dim-major approx length is odd (not `2*n`), the
`RmseWithUncertainty` arm returns `Vec::new()` (predict.rs:132-134). An empty
output silently propagates downstream (e.g. into a stage comparison or a Python
binding) as "no predictions" rather than signalling the shape error. This is the
same class of "degenerate-masquerades-as-valid" issue the backend WR-05 fixes
explicitly avoid; here `apply_prediction_type` has no error channel
(`-> Vec<f64>`), so the malformed input is swallowed.

**Fix:** Give the uncertainty transforms a fallible signature
(`-> Result<Vec<f64>, ModelError>`) or have callers validate the length is even
before invoking. At minimum, document that an empty return means a shape
violation so consumers can detect it.

### WR-05: `EvalMetric::eval` and `eval_grouped` disagree on the missing-weight fallback

**File:** `crates/cb-train/src/metrics.rs:261-267`, `crates/cb-train/src/metrics.rs:427-433`

**Issue:** Both functions validate `weights.len() == approx.len()` up front, then
build a `weight_at` closure. In `eval` the closure uses `unwrap_or(0.0)`
(metrics.rs:265); in `eval_grouped` it uses `unwrap_or(1.0)` (metrics.rs:431).
After the length check these branches are normally unreachable, but the
divergence is a latent hazard: if either length guard is ever relaxed or a future
caller passes a short weight slice, the two metric paths would weight the missing
object differently (0.0 vs 1.0), producing inconsistent metrics. The fallback for
the same logical condition should be identical.

**Fix:** Pick one convention (the upstream "missing weight == 1.0" is the natural
one) and use it in both closures, or assert the invariant once and document that
the closure is total under it.

### WR-06: `build_class_remap` uses `partial_cmp(...).unwrap_or(Equal)`; NaN labels collapse silently

**File:** `crates/cb-train/src/boosting.rs:842-847`

**Issue:** `build_class_remap` sorts class labels with
`a.partial_cmp(b).unwrap_or(std::cmp::Ordering::Equal)` and dedups with
`(*a - *b).abs() == 0.0`. A NaN label makes `partial_cmp` return `None` → treated
as `Equal`, yielding a non-total order and a potentially mis-sorted /
mis-deduplicated class map; `NaN - NaN` is NaN so the dedup keeps duplicate NaNs.
The doc says "NaN is not expected in a class target", but nothing rejects it, so a
malformed multiclass target produces a silently corrupt `class_to_label` (and
therefore wrong predicted labels) rather than a typed error.

**Fix:** Reject a non-finite label in `build_class_remap` (or in the multiclass
target validation) with a typed `CbError`, consistent with the no-NaN-poisoning
discipline applied to the custom-objective der path (cpu_runtime.rs:834-839).

### WR-07: Score-function self-oracles are vacuous for the Newton variants and weak for the others

**File:** `crates/cb-compute/tests/score_functions_test.rs:205-264`

**Issue:** `newton_l2_reuses_l2_formula_on_der2_filled_stats` and
`newton_cosine_reuses_cosine_formula_on_der2_filled_stats` assert
`NewtonL2(stats).to_bits() == L2(stats).to_bits()` on hand-built stats. Because
`multi_dim_split_score` maps `NewtonL2 -> numerator` and `L2 -> numerator`
identically (score.rs:131) and `NewtonCosine -> Cosine` identically
(score.rs:134), these assertions are TRUE for any input and test nothing about
the second-order behavior — they cannot detect CR-01. The SolarL2/LOOL2/SatL2
tests (lines 57-197) also recompute the production formula verbatim
(score_functions_test.rs:64-75 mirrors score.rs:179-183), so they only catch a
transcription typo, not a wrong-stats wiring bug. The D-6.4-06 weakened-oracle
caveat is honestly disclosed, but the tests give false confidence that the
Newton variants do something second-order.

**Fix:** Add a test that drives the actual production scoring path
(`multi_dim_candidate_score` / a candidate split over real der1/der2 buffers) and
asserts that `NewtonL2` differs from `L2` when der2 != weight-count — this would
have caught CR-01.

## Info

### IN-01: `multi_dim_split_score` numerator/denominator term vectors are not pre-sized

**File:** `crates/cb-compute/src/score.rs:120`, `crates/cb-compute/src/score.rs:173-239`

**Issue:** `num_terms` (score.rs:120) and the `*_terms` helpers
(score.rs:174, 194, 220) start from `Vec::new()` and grow via `push` inside a
nested loop, whereas the Cosine denominator pre-sizes with
`Vec::with_capacity(num_terms.len() + 1)`. Inconsistent and adds avoidable
reallocations.

**Fix:** Pre-size with `Vec::with_capacity(per_dim_leaves.iter().map(Vec::len).sum())`.

### IN-02: `random_score_instance` / `score_st_dev` perturbation helpers appear unused by the in-scope facade

**File:** `crates/cb-compute/src/score.rs:300-315`

**Issue:** The `random_strength` perturbation helpers are exercised only by the
perturbed tree-search path; the Builder default `random_strength` is `0.0`
(builder.rs:101) and the facade never sets it non-zero. Not a defect, but worth
confirming these have a non-self-oracle test that exercises a non-zero
`random_strength` end to end.

**Fix:** No change required; ensure a perturbed-path oracle exists.

### IN-03: `compute_softmax_gradients` packs an `n * k*(k+1)/2` Hessian buffer even for diagonal-only callers

**File:** `crates/cb-backend/src/cpu_runtime.rs:683-721`

**Issue:** Reviewed for correctness only (out of perf scope). The packed
symmetric Hessian path is correct and guarded; no action.

### IN-04: `solar_l2_terms` ignores `scaled_l2` (correct per CUDA source, but easy to misread)

**File:** `crates/cb-compute/src/score.rs:173-188`

**Issue:** `solar_l2_terms` (and LOOL2/SatL2) compute `(-sum*sum)/weight`
directly and never use the `scaled_l2` argument that `multi_dim_split_score`
threads in — correct per `score_calcers.cuh`, but the unused regularizer in a
function whose siblings use it is a readability trap.

**Fix:** A one-line comment at the `solar_l2_terms` call site noting "L2 reg is
not applied in the GPU Solar/LOO/Sat calcers" would prevent a future "bug fix"
that wrongly adds it.

### IN-05: Custom-objective oracle test standardize uses raw `iter().sum()` (test-only, acceptable)

**File:** `crates/cb-train/tests/custom_objective_oracle_test.rs:93-99`

**Issue:** `standardize` uses `y.iter().sum::<f64>()` and `.powi(2).sum()` rather
than `cb_core::sum_f64`. This is test-only setup (not a parity-critical
production fold), so the D-08 rule does not apply, but it diverges from the
project's summation discipline and could introduce a sub-1e-5 ordering difference
versus the fixture's NumPy standardization on a larger corpus.

**Fix:** None required for the current tiny corpus; note for larger fixtures.
