## Implementation Decisions

All decisions were auto-selected in `--auto` mode to the recommended option. They are dominated by constraints already locked upstream of this discussion — umbrella **D-07** (selective C++ instrumentation; here the selection is *none*, everything is Python-reachable), **D-09** (Python custom-objective callback deferred to Phase 8), the project's memory-efficiency-first goal, the ≤1e-5 parity bar, the single-code-path discipline (D-6.2-03), and the family-wave/per-stage-oracle pattern carried from 6.1 / 6.2 / 6.3. None are open design questions; they are the parity-/scope-dictated reading of a fixed scope.

### Wave / Plan Structure
- **D-6.4-01: Family-wave plans, each with its own ≤1e-5 per-stage oracle gate** (carried from 6.1 D-6.1-02 / 6.2 D-6.2-02 / 6.3 D-6.3-01) — keeps blast radius small and regression attribution unambiguous. Suggested grouping (planner finalizes exact plan-to-wave mapping):
  - **Wave A — Score functions (LOSS-09 / SC-1):** SolarL2, NewtonL2, NewtonCosine, LOOL2, SatL2 extend `EScoreFunction`; each split-score function oracle-locked ≤1e-5 (split selection identical to upstream). Independent of B/C.
  - **Wave B — Uncertainty (LOSS-08 + LOSS-06 uncertainty types / SC-2):** `RMSEWithUncertainty` loss (2-dim, rides the 6.2 N-dim spine) → virtual ensembles → the three uncertainty `PredictionType`s. Internally ordered: loss first, then ensemble aggregation, then the prediction-type transforms.
  - **Wave C — Custom objective/metric Rust trait (LOSS-07 Rust half / SC-3):** the trait pair + its dispatch seam + the self-oracle against a Rust reference loss.
  - Waves are largely independent; the planner may sequence A→B→C or parallelize A and C. Rejected: one-mega-plan (defeats per-strand regression attribution); per-function plans (excess ceremony).

### Oracle Floor & C++ Instrumentation (D-07 / D-08)
- **D-6.4-02: NO new C++ instrumentation — the entire phase rides the Python-reachable per-stage oracle.** Everything in 6.4 is **deterministic** and **Python-API-reachable**: score functions are pure split-score math (set `score_function=...`, read upstream splits/leaves); `RMSEWithUncertainty` is a deterministic 2-dim loss; virtual ensembles + the uncertainty prediction types are exposed by catboost's Python API (`virtual_ensembles_predict` / `predict(prediction_type='RMSEWithUncertainty' | 'TotalUncertainty' | 'VirtEnsembles')`); the custom-objective trait is **self-oracled** against a Rust reference (below). This is the explicit contrast with 6.3 — D-07's "selective, not universal" instrumentation here selects **nothing**, so the 6.3 disk-pressure / `/tmp` toolchain risk (D-08) does not apply. Rejected: instrumenting any 6.4 work (no RNG stream lacks a Python-reachable ground truth).

### Score-Function Seam (LOSS-09)
- **D-6.4-03: Extend the existing `EScoreFunction` enum in `cb-compute`; add hessian-weighted accumulation to the existing score seam for the Newton variants. Single code path — no parallel scorer.** SolarL2/NewtonL2/NewtonCosine/LOOL2/SatL2 join `Cosine`/`L2` in `runtime.rs:778`; the split-score calc (`score.rs` `multi_dim_split_score` / `l2_split_score` / `cosine_split_score`) gains the `der2`-weighted (Newton) and leave-one-out / saturated variants, transcribed verbatim from upstream `score_calcers.h` / `score_calcer.cpp`. The Newton variants consume the per-object hessian (`Derivatives.der2`) the boosting loop already computes — confirm during research how the score seam receives `der2` without perturbing the shipped L2/Cosine locks (D-04-style no-regression on every existing split oracle). Rejected: a parallel ranking-only/score-only scoring path (duplication / divergence — violates the single-code-path discipline D-6.2-03 carried from 6.2/6.3).

### Uncertainty Architecture (LOSS-08 + LOSS-06 uncertainty types)
- **D-6.4-04: `RMSEWithUncertainty` is a 2-dim loss on the 6.2 N-dim spine; virtual ensembles slice the trained model; the three uncertainty prediction types extend `cb-model::PredictionType`.** `RMSEWithUncertainty` predicts **two** outputs (mean + log-variance / scale) and attaches as a `Loss` variant computing 2-dim der1/der2 through the existing N-dim dispatch (transcribed from `error_functions.*` `TRMSEWithUncertaintyError`); leaf estimation rides the N-dim leaf path. **Virtual ensembles** partition the trained tree sequence into ensemble sub-models and aggregate their per-ensemble predictions (transcribe the slicing/count defaults from upstream `virtual_ensembles` / `model_estimated_features` apply path). The three uncertainty `PredictionType`s — `RMSEWithUncertainty`, `VirtEnsembles`, `TotalUncertainty` — extend the `PredictionType` enum (`predict.rs:36`) and `apply_prediction_type`; `TotalUncertainty` decomposes into data + knowledge uncertainty across the virtual ensemble (transcribe the decomposition from upstream). Rejected: a separate uncertainty-only model type (fragments the apply path — the prediction-type transform seam already exists and is the upstream-faithful home).

### Custom Objective / Metric Rust Trait (LOSS-07 Rust half)
- **D-6.4-05: Define a Rust trait pair — `CustomObjective` (per-object der1/der2, mirroring upstream `calc_ders_range`) and `CustomMetric` (eval) — plugged into the existing `Loss` / `EvalMetric` dispatch via a trait-object seam, designed so the Phase-8 PyO3 callback wraps the SAME trait; oracle-tested against a Rust-defined reference loss.** The trait methods mirror upstream's user-objective contract (`IDerCalcer` / the Python `calc_ders_range(approxes, targets, weights) -> [(der1, der2)]` + `evaluate(...) -> (sum, weight)` callback shape) so the Phase-8 PyO3 bridge is a thin adapter, not a redesign (D-09). It attaches to the `Loss` enum — which **already dropped `Copy`/`Eq`** (`runtime.rs:43-55`, for the parametric variants) — via a trait-object-carrying variant; the planner/research resolves the `Clone`/`PartialEq`-derive-vs-`dyn` tension (e.g. an `Arc<dyn CustomObjective>` newtype with manual or pointer-equality impls). **Oracle:** reimplement a built-in loss (e.g. RMSE or Logloss) AS a custom objective in Rust and assert it reproduces that built-in's existing known-good per-stage oracle ≤1e-5 — a self-oracle that proves the trait dispatch is faithful without needing a new upstream fixture for the mechanism itself. Rejected: a generic type parameter threaded through the train stack (viral, perturbs every existing monomorphization / the shipped locks); a closure-only API (doesn't wrap cleanly for the PyO3 callback — fails the SC-3 "designed for clean PyO3 wrap" intent).

## Specific Ideas

- **The custom-objective trait is the design-forward hinge of this phase.** Unlike LOSS-09/08 (pure upstream transcription), LOSS-07's Rust trait shape is a *design* choice constrained by one hard requirement: the Phase-8 PyO3 callback must wrap it cleanly (D-09). So mirror upstream's Python callback contract (`calc_ders_range(approxes, targets, weights) -> [(der1, der2)]`, custom-metric `evaluate(...) -> (sum, weight)`) in the Rust method signatures — the PyO3 layer then becomes a thin adapter, not a redesign. Prove the dispatch is faithful by self-oracling: reimplement a built-in (RMSE/Logloss) as a custom objective and match its existing known-good oracle ≤1e-5.
- **No instrumentation budget this phase** — the explicit lesson carried from 6.3: 6.3 spent the `/tmp` clang-18 toolchain on YetiRank/StochasticRank RNG streams; 6.4 has zero RNG-stream losses, so every fixture is plain offline catboost==1.2.10. Keep the instrumentation surface at zero (D-6.4-02).
- **NewtonL2 / NewtonCosine need `der2` in the split score** — the existing L2/Cosine scores use only `sum_weighted_delta` / `sum_weight`. Confirm during research how the hessian threads into `multi_dim_split_score` without perturbing the shipped Cosine/L2 split locks (the 05-19 Task A parity gap was exactly an L2-vs-Cosine split divergence — do not regress it).
- **RMSEWithUncertainty is 2-dimensional** — it predicts mean + (log-)variance, so it rides the 6.2 N-dim spine, not the scalar loop. Its prediction-type transform and the virtual-ensemble decomposition (`TotalUncertainty` = data + knowledge uncertainty) are the subtle transcription targets.
