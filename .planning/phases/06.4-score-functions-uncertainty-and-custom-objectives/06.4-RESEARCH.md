**Researched:** 2026-06-18

## Summary

This phase has **three independent strands** mapping to Waves A/B/C, and research surfaced a **decisive, scope-altering finding for Wave A** that the planner MUST resolve before planning:

1. **Wave A (LOSS-09 score functions) ‚Äî the five named functions SolarL2, NewtonL2, NewtonCosine, LOOL2, SatL2 are GPU-ONLY in upstream catboost.** The CPU split-score factory `MakePointwiseScoreCalcer` (`score_calcers.h:106-115`) hard-`CB_ENSURE(false, "Only Cosine and L2 score functions are supported for CPU.")` for anything but Cosine/L2, and validation `oblivious_tree_options.cpp:143-146` rejects them at config time unless `task_type==GPU`. **Verified live:** `CatBoostRegressor(score_function="SolarL2", task_type="CPU")` raises `oblivious_tree_options.cpp:146: Only Cosine and L2 score functions are supported for CPU.` for all five. **This means the D-6.4-02 oracle strategy ("set `score_function=...`, read upstream CPU splits/leaves") is IMPOSSIBLE for these five on CPU.** Their formulas live only in CUDA (`cuda/methods/kernel/score_calcers.cuh`). See the "Wave A reachability blocker" decision the planner must make below.

2. **Wave B (LOSS-08 + LOSS-06 uncertainty) ‚Äî fully CPU-Python-reachable and deterministic.** `RMSEWithUncertainty` trains on CPU (`thread_count=1`), `predict(prediction_type=...)` and `virtual_ensembles_predict(...)` all work and are bit-reproducible across runs (verified live). Every formula transcribed at file:line below. This wave is unblocked and is the bulk of the deliverable value.

3. **Wave C (LOSS-07 custom-objective Rust trait) ‚Äî self-oracled, no upstream fixture needed.** The Python contract (`calc_ders_range(approxes, targets, weights) -> [(der1, der2)]`, `calc_ders_multi`, `CustomMetric.evaluate -> (error, weight)` + `get_final_error` + `is_max_optimal`) is documented in `core.py:4867` and mirrored by the upstream `IDerCalcer` contract (`error_functions.h:27`). The `Arc<dyn>`-in-derived-enum mechanism is the one real design question.

**Primary recommendation:** Plan Waves B and C as specified (both unblocked). **For Wave A, do NOT plan a CPU oracle against the five GPU-only functions** ‚Äî they cannot be reached through catboost 1.2.10 CPU/Python. Recommended Wave-A scope: (a) implement the five formulas transcribed from the CUDA `.cuh` as additive `EScoreFunction` variants behind the existing single scorer seam, and (b) oracle them by **transcribe-then-self-oracle** (assert the CPU Rust formula reproduces the CUDA arithmetic at chosen leaf-stat inputs, hand-computed), NOT against a catboost CPU training run. Flag this as an Open Question for `/gsd-discuss-phase` ‚Äî it is a genuine scope correction, not Claude's discretion.

## Strand 1 ‚Äî LOSS-09 Score Functions (Wave A)

### ‚öÝÔ∏è Reachability blocker (READ FIRST)

The five functions are **GPU-only** in catboost 1.2.10. Evidence:

- `score_calcers.h:106-115` ‚Äî CPU `MakePointwiseScoreCalcer` switch has only `Cosine` and `L2` arms; `default` is `CB_ENSURE(false, "Only Cosine and L2 score functions are supported for CPU.")`.
- `oblivious_tree_options.cpp:143-146` ‚Äî validation: `CB_ENSURE(TaskType.Get() == ETaskType::GPU || EqualToOneOf(ScoreFunction, EScoreFunction::Cosine, EScoreFunction::L2), "Only Cosine and L2 score functions are supported for CPU.")`.
- The five formulas exist ONLY in CUDA: `catboost/cuda/methods/kernel/score_calcers.cuh` + dispatch `catboost/cuda/methods/kernel/pointwise_scores.cu:484-527`.
- **Live test (this session, `.venv` catboost==1.2.10, CPU, thread_count=1, Plain boosting):** `L2`‚ÜíOK, `Cosine`‚ÜíOK; `SolarL2`/`NewtonL2`/`NewtonCosine`/`LOOL2`/`SatL2` ‚Üí all raise `oblivious_tree_options.cpp:146`.

**Consequence:** D-6.4-02's CPU Python oracle ("set `score_function=...`, read upstream splits/leaves") cannot produce ground truth for these five. The planner must choose a fallback (see Open Q1). The project's CPU-only, `thread_count=1`, no-instrumentation constraints (CLAUDE.md, D-6.4-02) leave only **transcribe-then-self-oracle** (assert the Rust port reproduces the documented CUDA arithmetic on chosen leaf-stat inputs), which is weaker than a training-stage oracle but is the same family as the existing `cb-oracle` primitive fixtures.

### Exact upstream formulas (CUDA, `score_calcers.cuh`)

All operate per leaf on `(sum, weight)` where `sum = SumWeightedDelta` and `weight = SumWeight` of the leaf (Newton variants feed `der2`-derived stats; see below). Higher score = better split (note GPU returns NEGATED scores internally; preserve the sign convention used by the existing CPU `multi_dim_split_score`, which returns higher-is-better).

| Function | Per-leaf term (`AddLeaf(sum, weight)`) | Source | 2nd-order? |
|----------|----------------------------------------|--------|-----------|
| **SolarL2** | `weight > 1e-20 ? (-sum*sum) * (1 + 2*ln(weight + 1.0)) / weight : 0` | `score_calcers.cuh:22-24` | no |
| **L2** (shipped) | `weight > 1e-20 ? (-sum*sum)/(weight + Lambda) : 0` (MetaExponent=1 path) | `score_calcers.cuh:53-56` | no |
| **LOOL2** | `adjust = weight>1 ? weight/(weight-1) : 0; adjust*=adjust; weight>0 ? adjust*(-sum*sum)/weight : 0` | `score_calcers.cuh:83-87` | no |
| **SatL2** | `adjust = weight>2 ? weight*(weight-2)/(weight*weight-3*weight+1) : 0; weight>0 ? adjust*(-sum*sum)/weight : 0` | `score_calcers.cuh:114-117` | no |
| **Cosine** (shipped) | `mu = weight>0 ? sum/(weight+lambda) : 0; Score += sum*mu; DenumSqr += weight*mu*mu`; final `Score/sqrt(DenumSqr)` (DenumSqr seeded `1e-10` on GPU vs `1e-100` on CPU `score_calcers.h:52`) | `score_calcers.cuh:152-168` | no |
| **NewtonL2** | **Same `TL2ScoreCalcer` as L2** (`pointwise_scores.cu:504-510`) | reuses L2 calcer | **yes** |
| **NewtonCosine** | **Same `TCosineScoreCalcer` as Cosine** (`pointwise_scores.cu:512-521` and `:458-461`) | reuses Cosine calcer | **yes** |

### How `der2` threads in (the D-6.4-03 / Pitfall-2 question, ANSWERED)

The Newton vs non-Newton distinction is **NOT a different score formula** ‚Äî NewtonL2 reuses `TL2ScoreCalcer`, NewtonCosine reuses `TCosineScoreCalcer`. The difference is **what statistics fill `sum` and `weight`**:
- `IsSecondOrderScoreFunction` (`enum_helpers.cpp:830-847`) returns `true` only for `NewtonL2`/`NewtonCosine`.
- For second-order score functions, the histogram's `weight` slot is the **summed hessian (`der2`)** instead of the object count / `SumWeight`, and `sum` is the summed gradient. (This is the standard Newton-step "gradient¬≤/hessian" gain ‚Äî `(-sum*sum)/(hessianSum + lambda)`.) For first-order functions the same calcer is fed `SumWeight` as `weight`.

**Implication for the Rust seam (single-code-path, no-regression):** the existing `LeafStats { sum_weighted_delta, sum_weight }` already carries the two slots `multi_dim_split_score` needs. For the Newton variants, `sum_weight` must be populated with the **summed `Derivatives.der2`** (negated to a positive hessian ‚Äî RMSE der2 is `-1`, Logloss der2 is `-p(1-p)`, both ‚â§0; the Newton denominator wants `+hessian`) instead of the object/weight count. **The `LeafStats` struct does not change; only the upstream histogram fill changes per score function.** Cosine/L2 (first-order) keep feeding `SumWeight` ‚Üí byte-identical to today's shipped locks (D-04 anchor preserved). This is exactly how upstream avoids a parallel scorer.

### Defaults (`catboost_options.cpp` / `oblivious_tree_options.cpp`)
- CPU default `score_function` = **L2** (`useL2ScoreFunction` branch) for MultiClass/MultiClassOneVsAll/RMSEWithUncertainty under Lossguide (`catboost_options.cpp:980-990`); the general CPU oblivious-tree default is **Cosine** (`oblivious_tree_options.cpp:22`, already encoded in `EScoreFunction::default()` = Cosine, `runtime.rs:781`). The five new functions have NO CPU default ‚Äî they are opt-in GPU-only.
- `IsPlainOnlyModeScoreFunction` (`enum_helpers.cpp:877-882`): every function EXCEPT Cosine/NewtonCosine forbids ordered boosting. (Project is Plain-path anyway; note for completeness.)

### Rust integration points (Wave A)
- `crates/cb-compute/src/runtime.rs:778` ‚Äî `EScoreFunction` enum gains `SolarL2`, `NewtonL2`, `NewtonCosine`, `LOOL2`, `SatL2`. Keep `#[derive(Debug, Clone, Copy, PartialEq, Eq, Default)]` (these are unit variants ‚Äî no params, so the derives are safe, unlike `Loss`).
- `crates/cb-compute/src/score.rs:110` `multi_dim_split_score` ‚Äî add the five per-leaf transforms as additional `match` arms. **All accumulation MUST route through `cb_core::sum_f64`** (CLAUDE.md / D-08 grep). The `-sum*sum` numerator and per-function `adjust`/`ln`/`mu` factors are per-leaf scalars folded through `sum_f64`, mirroring the existing L2/Cosine arms.
- `crates/cb-train/src/tree.rs` + histogram fill ‚Äî for `NewtonL2`/`NewtonCosine`, populate the `sum_weight` leaf-stat slot with the summed `der2` hessian (via `cb_core::sum_f64`) rather than the weight count. **Confirm the existing fill path is the ONLY place this branches**, so L2/Cosine are untouched (no-regression gate against every shipped split oracle ‚Äî the 05-19 Task A L2-vs-Cosine lock).

## Strand 2 ‚Äî LOSS-08 + LOSS-06 Uncertainty (Wave B)

**All CPU-Python-reachable and deterministic** (verified live: `RMSEWithUncertainty` fit on CPU thread_count=1; `predict(prediction_type='RMSEWithUncertainty')` ‚Üí 2 cols; `virtual_ensembles_predict(prediction_type='TotalUncertainty', virtual_ensembles_count=5)` ‚Üí 3 cols; `'VirtEnsembles'` ‚Üí (n, V, 2); bit-reproducible across two runs with `random_seed=42`, both with and without `posterior_sampling`).

### RMSEWithUncertainty loss ‚Äî der1/der2 (`error_functions.h:280-313`)

`TRMSEWithUncertaintyError : TMultiDerCalcer(EHessianType::Diagonal)`, `dim = 2` (mean = `approx[0]`, log-scale = `approx[1]`):

```
diff = target[0] - approx[0]
prec = FastExp(-2 * approx[1])          // FastExpWithInfInplace on (-2*approx[1])
der[0] = weight * diff
der[1] = weight * (diff*diff * prec - 1)
// diagonal hessian:
der2[0] = -weight
der2[1] = -2 * weight * diff*diff * prec
```

It is the ngboost "natural gradient" (regular gradient √ó Fisher info). `Y_ASSERT(target.size() == 1)` ‚Äî single scalar target, two approx dims.

- **Leaf-estimation default:** Newton, `defaultNewtonIterations=1`, `defaultGradientIterations=1` (`catboost_options.cpp:77-82`). Diagonal hessian ‚Üí per-dimension scalar Newton step (no dense symmetric solve, unlike MultiClass). Pin `leaf_estimation_method: Newton`, `leaf_estimation_iterations: 1` in fixtures.
- **Rides the 6.2 N-dim spine** exactly: 2-dim dimension-major der1/der2 buffers (`Derivatives`, `runtime.rs:806`), N-dim leaf estimation per dimension. `EHessianType::Diagonal` = the separable per-dim path the 6.2 MultiClassOneVsAll/MultiLogloss already use.

### Rust integration (loss):
- `crates/cb-compute/src/runtime.rs:56` `Loss` ‚Äî add `RmseWithUncertainty` variant (unit; no params). `Loss` already dropped `Copy`/`Eq` so this is additive.
- `crates/cb-compute/src/loss.rs` ‚Äî add a 2-dim der arm computing the above through the existing N-dim dispatch (precedent: the diagonal multi-dim helpers `multi_crossentropy_ders`, `multiclass_onevsall_ders`). **Route the `diff*diff` and weight products through `cb_core::sum_f64` where they accumulate into leaf stats.** Note `FastExp` ‚Üî `f64::exp` is within 1e-5 (established A2 precedent in `predict.rs:Exponent`).

### Virtual ensembles ‚Äî model slicing (`apply.cpp:526-600`)

`ApplyVirtualEnsembles(model, dataset, end, virtualEnsemblesCount, &rawValues)`:
- `evalPeriod = end / (2 * virtualEnsemblesCount)` (integer division). `CB_ENSURE(evalPeriod > 0 && evalPeriod*virtualEnsemblesCount < end, "Not enough trees ...")`.
- `begin = end - evalPeriod * virtualEnsemblesCount`.
- **First ensemble** = prediction on trees `[0, begin)` (i.e. trees `[0, end/2)` region) ‚Üí `baseApprox`, copied as ensemble 0's seed.
- Then iterate `vEnsembleIdx = 0..`; each step applies trees `[begin, begin+evalPeriod)` (`InternalRawFormulaVal`), adds to the running ensemble approx, and for all but the last ensemble copies the running sum forward to seed the next ensemble (the `copyerLambda` with `copyToNextEnsemble=true`).
- **Unshrink:** `unshrinkCoef = pow(1 - actualShrinkCoef, lastTreeIdx - end)`; `actualShrinkCoef = model.GetActualShrinkCoef()` (= `model_shrink_rate * learning_rate`; **0 by default**, so `unshrinkCoef = 1` in the default config ‚Äî the project can implement the default-path first and gate non-zero shrink as v2 if it complicates parity). `CB_ENSURE(actualShrinkCoef in [0,1))`.
- Result `rawValues` has `virtualEnsemblesCount * approxDimension` rows (for RMSEWithUncertainty, `approxDimension=2` ‚Üí `2*V` rows: per ensemble a (mean, log-scale) pair).
- **Default `virtual_ensembles_count = 10`** (`core.py:3005,3027`). `ntree_end=0` ‚Üí use all trees.

### Uncertainty prediction-type transforms (`eval_helpers.cpp:422-470`, `PrepareEval`)

| PredictionType | Output cols | Transform | Source |
|----------------|-------------|-----------|--------|
| **RMSEWithUncertainty** (single-model predict, no VE) | 2 | col0 = `approx[0]` (mean, identity); col1 = `CalcSquaredExponent(approx[1]) = exp(2*approx[1])` (variance) | `eval_helpers.cpp:422-427`; `eval_processing.h:47` |
| **VirtEnsembles** | `V √ó 2` | `result = approx` (the `2V` VE rows); for RMSEWithUncertainty, every odd row (the log-scale dims, idx 1,3,5,...) ‚Üí `exp(2*x)` in place | `eval_helpers.cpp:428-444` |
| **TotalUncertainty** (RMSEWithUncertainty) | 3 | `[meanApprox, knowledgeUncertainty, dataUncertainty]` via `CalcRegressionUncertaitny` | `eval_helpers.cpp:445-455` |

**`CalcRegressionUncertaitny` (`eval_helpers.cpp:209-269`)** ‚Äî with `dimShift=2` (RMSEWithUncertainty has both mean & log-scale per ensemble):
```
for each object:
  mean = (1/V) * Œ£_{e=0..V-1} approx[e*2][obj]               // mean of ensemble means
  meanApprox[obj] = mean
  knowledgeUncertainty[obj] = (1/V) * Œ£_e (approx[e*2][obj] - mean)^2   // variance of means (epistemic)
  dataUncertainty[obj] = (1/V) * Œ£_e exp(2 * approx[e*2+1][obj])        // mean of exp(2*log-scale) (aleatoric)
```
So **TotalUncertainty output = [mean, knowledgeUncertainty (var of ensemble means), dataUncertainty (mean predicted variance)]**. (The classification path `CalcClassificationUncertainty`/`CalcMulticlassUncertainty` is entropy-based and out of scope ‚Äî only RMSEWithUncertainty regression uncertainty is needed for SC-2.)

### Rust integration (uncertainty prediction):
- `crates/cb-model/src/predict.rs:36` `PredictionType` ‚Äî add `RmseWithUncertainty`, `VirtEnsembles`, `TotalUncertainty`. **Caveat:** the enum currently derives `Copy, Eq` (`predict.rs:35`) and these unit variants keep that intact (no params) ‚Äî safe.

RMSEWithUncertainty single-model is a pure 2-col transform; VirtEnsembles/TotalUncertainty need the **VE approx matrix** (the `2V`-row buffer), so they cannot be a pure `&[f64]`‚Üí`Vec<f64>` over a single approx ‚Äî they need a VE-aware entry point.

## Strand 3 ‚Äî LOSS-07 Custom Objective / Metric Rust Trait (Wave C)

### Upstream contract shape

**Python callback contract (`core.py:4867` + catboost tutorials):** the user passes an object as `loss_function=` that provides:
- `calc_ders_range(approxes, targets, weights) -> list[(der1, der2)]` ‚Äî per-object first & second derivative (scalar loss).
- `calc_ders_multi(approxes, target, weight) -> (list[der1], matrix[der2])` ‚Äî multi-dim variant.
For `eval_metric=` a custom-metric object provides:
- `evaluate(approxes, target, weight) -> (error_sum, weight_sum)` ‚Äî accumulated numerator & denominator.
- `get_final_error(error, weight) -> float` ‚Äî final reduction (e.g. `error/weight`).
- `is_max_optimal() -> bool` ‚Äî direction.

**Upstream C++ `IDerCalcer` (`error_functions.h:27-182`):** the internal contract is `CalcDersRange(start, count, approxes, targets, weights, ders[], ...)` filling `TDers{Der1, Der2}` per object, and `CalcDersMulti(approx, target, weight, der, der2)` for multi-dim. The Python callback is adapted onto this. **Mirror the Python signatures** (not the C++ ones) in the Rust trait so the Phase-8 PyO3 wrap (D-09) is a thin adapter.

### Recommended Rust trait pair

```rust
// crates/cb-compute (new module, e.g. custom.rs)
pub trait CustomObjective: Send + Sync {
    /// Per-object (der1, der2) over a contiguous range ‚Äî mirrors
    /// Python `calc_ders_range(approxes, targets, weights) -> [(der1, der2)]`.
    fn calc_ders_range(
        &self,
        approxes: &[f64],
        targets: &[f64],
        weights: &[f64],   // empty slice == all weights 1.0 (upstream convention)
        ders: &mut [(f64, f64)],
    ) -> CbResult<()>;
}

pub trait CustomMetric: Send + Sync {
    /// (error_sum, weight_sum) ‚Äî mirrors Python `evaluate`.
    fn evaluate(&self, approxes: &[f64], target: &[f64], weight: &[f64]) -> CbResult<(f64, f64)>;
    fn get_final_error(&self, error: f64, weight: f64) -> f64;
    fn is_max_optimal(&self) -> bool;
}
```

### The `dyn`-in-derived-enum mechanism (the one real design question ‚Äî ANSWERED)

`Loss` derives `Clone + PartialEq` (`runtime.rs:55`). A bare `Box<dyn CustomObjective>` field breaks BOTH (`dyn Trait` is neither `Clone` nor `PartialEq`). Recommended solution ‚Äî an **`Arc<dyn>` newtype wrapper with manual `Clone` (cheap Arc bump) + pointer-equality `PartialEq`**:

```rust
#[derive(Clone)]
pub struct CustomObjectiveHandle(pub Arc<dyn CustomObjective>);

impl PartialEq for CustomObjectiveHandle {
    fn eq(&self, other: &Self) -> bool {
        Arc::ptr_eq(&self.0, &other.0)   // identity equality ‚Äî two distinct closures are never "equal"
    }
}
impl std::fmt::Debug for CustomObjectiveHandle {
    fn fmt(&self, f: &mut std::fmt::Formatter<'_>) -> std::fmt::Result {
        f.write_str("CustomObjectiveHandle(<dyn>)")
    }
}
```

Then add `Loss::Custom(CustomObjectiveHandle)` ‚Äî `Loss`'s `#[derive(Debug, Clone, PartialEq)]` now works because the field type implements all three manually. **`Arc` (not `Box`)** is required so `Clone` is cheap and the same objective instance is shared across the train loop's by-value/`.clone()` call sites (the 6.2 refactor already converted these to `&Loss`/`.clone()`). Same pattern for `EvalMetric::Custom(CustomMetricHandle)` ‚Äî but note `EvalMetric` currently derives `Copy` (`metrics.rs:54`); **adding a `Custom` variant forces dropping `Copy` from `EvalMetric`** (Arc is not Copy), a one-time cross-crate ripple identical to the 6.2 `Loss` `Copy`-drop. Flag this for the planner: it's a mechanical refactor, not a behavior change.

### Dispatch seam
- `crates/cb-compute/src/loss.rs` ‚Äî the der dispatch gains a `Loss::Custom(h) => h.0.calc_ders_range(...)` arm. ## Common Pitfalls

### Pitfall 1: Wave A's five score functions are GPU-only ‚Äî no CPU Python ground truth
**What goes wrong:** Planning a CPU oracle (`score_function="SolarL2"`, read upstream splits) ‚Äî it raises `oblivious_tree_options.cpp:146`. **Why:** upstream restricts CPU to Cosine/L2. **Avoid:** plan Wave A as transcribe-then-self-oracle of the CUDA `.cuh` formulas (assert Rust port == documented CUDA arithmetic on chosen leaf-stat inputs); do NOT plan a catboost-CPU training fixture for these five. **Warning sign:** any fixture generator that calls `model.fit(..., score_function=<one of the five>)` on CPU.

### Pitfall 2: Newton der2 sign when filling the score histogram (D-6.4-03)
**What goes wrong:** feeding the raw `Derivatives.der2` (which is ‚â§0 for RMSE/Logloss) into the Newton denominator `(-sum¬≤)/(weight+Œª)` flips the gain sign or divides by a negative. **Why:** upstream's "weight" slot for second-order score functions is the **positive** summed hessian. **Avoid:** negate `der2` to a positive hessian when populating `sum_weight` for NewtonL2/NewtonCosine; verify against the upstream Newton-gain sign convention. **Warning sign:** Newton splits ranking opposite to L2 on a config where they should agree.

### Pitfall 3: regressing the shipped L2/Cosine split locks (the 05-19 Task A gap)
**What goes wrong:** threading `der2` into `multi_dim_split_score` perturbs the Cosine/L2 (first-order) accumulation order. **Why:** the 05-19 parity gap was exactly an L2-vs-Cosine split divergence ‚Äî any change to the shared `sum_f64` fold order regresses it. **Avoid:** first-order functions keep feeding `SumWeight` (untouched code path); the Newton branch only changes the histogram FILL, not the score fold. Run every existing split oracle as a no-regression gate.

### Pitfall 4: RMSEWithUncertainty 2nd-dimension leaf estimation
**What goes wrong:** estimating both leaf dims with the mean (dim-0) Newton step, or using a dense symmetric solve. **Why:** the hessian is `EHessianType::Diagonal` ‚Äî each dim gets an INDEPENDENT scalar Newton step (`der2[0]=-weight`, `der2[1]=-2¬∑weight¬∑diff¬≤¬∑prec`), NOT MultiClass's dense solve. **Avoid:** route through the 6.2 diagonal per-dim leaf path (the MultiClassOneVsAll/MultiLogloss precedent), not the MultiClass `solve_symmetric_newton`. **Warning sign:** dim-1 (log-scale) leaf values diverging from upstream.

### Pitfall 5: virtual-ensemble slicing boundaries & `evalPeriod` integer division
**What goes wrong:** off-by-one in `begin`/`evalPeriod`, or float `evalPeriod`. **Why:** `evalPeriod = end / (2*V)` is INTEGER division; `begin = end - evalPeriod*V`; ensemble 0 seeds from trees `[0, begin)`, then each ensemble adds `evalPeriod` trees and (except the last) copies the running sum forward (`copyerLambda`). **Avoid:** transcribe `apply.cpp:540-599` exactly; `CB_ENSURE(evalPeriod>0 && evalPeriod*V < end)` (need ‚â• `2V+1` trees for `V` ensembles). Default `V=10`. **Warning sign:** "Not enough trees in model" or VE counts off by one tree.

### Pitfall 6: `CalcSquaredExponent` is `exp(2x)`, not `exp(x)¬≤` literally or `x¬≤`
**What goes wrong:** using `exp(x)` for the variance transform. **Why:** upstream doubles in place THEN FastExp: `CalcSquaredExponentInplace` does `x *= 2; FastExp(x)` ‚Üí `exp(2x)` (`eval_processing.h:40-43,47`). The log-scale dim is `0.5¬∑log(variance)`, so `variance = exp(2¬∑approx[1])`. **Avoid:** `(2.0*x).exp()` for the variance/dataUncertainty transforms. **Warning sign:** variance column off by a square-root factor.

### Pitfall 7: the `dyn`-in-derived-enum + `EvalMetric` Copy drop
**What goes wrong:** `Loss::Custom(Box<dyn ...>)` won't compile (`Clone`/`PartialEq`), or adding `EvalMetric::Custom` while `EvalMetric` still derives `Copy`. **Why:** `dyn Trait` isn't `Clone`/`PartialEq`/`Copy`. **Avoid:** `Arc<dyn>` newtype with manual `Clone`+`Debug`+pointer-eq `PartialEq` (`Arc::ptr_eq`); drop `Copy` from `EvalMetric` (one-time ripple, mirrors the 6.2 `Loss` Copy-drop). **Warning sign:** cross-crate "expected `EvalMetric`, found `&EvalMetric`" errors after the variant lands.

### Pitfall 8: summation routing (CI-grep ban)
**What goes wrong:** a raw `iter().sum()` or `0.0`-seeded fold in new score/uncertainty accumulation. **Why:** CLAUDE.md / D-08 CI-grep requires every parity-critical float sum through `cb_core::sum_f64`/`sum_f32_in_f64`. **Avoid:** the SolarL2 `ln` factor, NewtonL2 hessian fill, VE `mean`/`var`/`dataUncertainty` Œ£ all route through `sum_f64`. **Warning sign:** CI grep failure on the new modules.
