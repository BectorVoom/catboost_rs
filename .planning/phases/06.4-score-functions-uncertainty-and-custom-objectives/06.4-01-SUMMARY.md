# Phase 06.4 Plan 01: Wave-A Score Functions (LOSS-09) Summary

Added the five remaining split-score functions — `SolarL2`, `NewtonL2`, `NewtonCosine`,
`LOOL2`, `SatL2` — to the single `EScoreFunction`/`multi_dim_split_score` seam, with the
Newton variants' positive-hessian histogram fill primitive, self-oracled against the
hand-computed CUDA `score_calcers.cuh` arithmetic (D-6.4-06 transcribe-then-self-oracle, NOT
a ≤1e-5-vs-upstream-CPU lock — these five are GPU-only upstream).

## What was built

- **`EScoreFunction` (runtime.rs):** 5 new unit variants. The full `#[derive(Debug, Clone,
  Copy, PartialEq, Eq, Default)]` set is retained (unit variants carry no params), so the
  enum did NOT regress its `Copy + Eq` derives.
- **`multi_dim_split_score` (score.rs):** 5 new match arms.
  - `SolarL2` / `LOOL2` / `SatL2` — first-order per-leaf scalars transcribed verbatim from
    `score_calcers.cuh` (`:22-24`, `:83-87`, `:114-117`), each folded through
    `cb_core::sum_f64` (helpers `solar_l2_terms` / `loo_l2_terms` / `sat_l2_terms`).
  - `NewtonL2` reuses the `L2` arm verbatim; `NewtonCosine` reuses the `Cosine` arm verbatim
    (`pointwise_scores.cu:504-521`). The second-order distinction is the FILL, not the math.
- **`reduce_leaf_stats_newton` (histogram.rs, NEW, exported):** the Newton histogram fill —
  `sum_weighted_delta` is the gradient sum; `sum_weight` carries the summed **positive**
  hessian `Σ(-der2*weight)` (der2 is ≤0 for RMSE/Logloss → negated, Pitfall 2). `LeafStats`
  struct is UNCHANGED; the fold reuses `reduce_leaf_der2` (sum_f64, D-08).
- **`tree.rs::split_score`:** now dispatches all 7 variants. `Cosine`/`L2` stay on the
  dedicated scalar calcers (byte-identical hot path); the five new variants route through the
  one `multi_dim_split_score` seam at dim=1.
- **`crates/cb-compute/tests/score_functions_test.rs` (NEW):** the transcribe-then-self-oracle
  (9 tests) — all 5 functions + boundary (weight=1 LOOL2, weight=2 SatL2) + degenerate
  (weight=0 SolarL2 finiteness) + a non-vacuous SolarL2≠L2 guard. Leads with the D-6.4-06
  weakened-oracle caveat doc-comment.

## D-6.4-06 weakened-oracle caveat (NON-NEGOTIABLE)

The five functions are GPU-only in catboost 1.2.10 (CPU rejects them at
`oblivious_tree_options.cpp:146`). There is NO upstream-CPU training ground truth. The oracle
is **transcribe-then-self-oracle**: the Rust port reproduces the hand-computed CUDA arithmetic
on chosen leaf-stat inputs (≤1e-12, machine-epsilon). This is a documented, user-confirmed
weakening — it must never be reported as a ≤1e-5-vs-upstream lock. The caveat is stated
verbatim in the test file's module doc-comment.

## Deviations from Plan

### Scope clarification (in-scope, documented as a decision)

**Newton live-search der2 threading deferred to the GPU path (Phase 7).** Task 3 specifies the
Newton hessian fill and a no-regression gate. The reusable primitive (`reduce_leaf_stats_newton`)
and the full 7-variant `split_score` dispatch are shipped and unit-tested. Threading `der2`
through the entire live PLAIN boosting greedy-search call chain (boosting → greedy_tensor_search
→ select_level → candidate scoring) was NOT wired, because:
1. Per D-6.4-06 these five functions are GPU-only upstream with NO CPU training path —
   a live CPU boosting run with `score_function=NewtonL2` is unreachable and non-oracle-able.
2. The existing PLAIN/ordered search functions thread only `der1`/`weight` (no `der2` parameter);
   adding der2 to that chain would be untestable scaffolding for a GPU-only feature and risk
   regressing the shipped Cosine/L2 locks (Pitfall 3).

The Newton FILL building block is in place for the Phase-7 GPU wiring; the formula reuse
(NewtonL2≡L2, NewtonCosine≡Cosine on der2-filled stats) is self-oracled. No Rule 1-4 deviation
otherwise; the plan executed as written.

## Known Stubs

None. The five functions are fully implemented and self-oracled. The Newton live-search der2
plumbing is an explicit Phase-7-GPU deferral (above), not a stub feeding empty/placeholder data
to any reachable CPU path.
