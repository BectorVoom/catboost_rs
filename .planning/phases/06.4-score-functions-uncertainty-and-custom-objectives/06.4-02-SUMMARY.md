# Phase 06.4 Plan 02: Wave-B RMSEWithUncertainty Loss (LOSS-08) Summary

Added `RMSEWithUncertainty` as a 2-dimensional DIAGONAL-hessian `Loss` variant riding
the shipped 6.2 N-dim approx spine (D-6.4-04), oracle-locked ≤1e-5 against an
OFFLINE-generated catboost 1.2.10 training fixture (D-6.4-02 full Python-reachable
per-stage oracle). End-to-end: a Rust caller trains RMSEWithUncertainty on CPU
(`thread_count=1`, Newton/1-iter) and the per-stage splits / 2-dim leaf-values /
2-dim staged-approx / 2-dim predictions all match upstream.

## What was built

- **`Loss::RmseWithUncertainty` (runtime.rs):** an additive unit variant
  (`approx_dimension = 2`; `Loss` already dropped `Copy`/`Eq`) + the no-param
  `validate` arm.
- **`rmse_with_uncertainty_ders` (loss.rs, exported):** the 2-dim diagonal der
  helper transcribed VERBATIM from `error_functions.h:280-313` — `diff=t-a0`,
  `prec=(-2·a1).exp()`, der1 = `[w·diff, w·(diff²·prec−1)]`, der2-diag =
  `[−w, −2·w·diff²·prec]`. Unit-tested against the formula on hand-picked
  `(a0,a1,t,w)` triples (1e-12) + the `diff==0` boundary.
- **`compute_rmse_uncertainty_gradients` (cpu_runtime.rs):** a dedicated
  `compute_gradients` dispatch arm. RMSEWithUncertainty's der is COUPLED across the
  two dims (the log-scale der reads `approx[0]`), so it cannot run as `K` independent
  per-dim kernel launches; this arm assembles the dimension-major `[mean(0..n),
  log-scale(n..2n)]` diagonal `der1`/`der2`. The DIAGONAL hessian then feeds the
  EXISTING 6.2 per-dim scalar Newton leaf path (the MultiClassOneVsAll/MultiLogloss
  diagonal branch), NOT the MultiClass dense solve (Pitfall 4). A defensive
  single-dim arm rejects it with a typed `CbError`.
- **`rmse_uncertainty_starting_approx` (boosting.rs):** the per-dimension
  optimal-constant starting bias `[mean(target), 0.5·log(var(target))]`. RMSEWithUncertainty
  ALWAYS starts from this (even `boost_from_average=false` — `train_model.cpp:858`);
  `mean`/`var` are f32-truncated to match upstream's `float`-returning
  `CalculateWeightedTarget*`, then `0.5·log(var)` widens to f64
  (`optimal_const_for_loss.h:225-229`). Both Σ route through `cb_core::sum_f64` (D-08).
  The approx buffer init uses this per-dim bias for RmseWithUncertainty (every other
  loss keeps the scalar `bias`).
- **`loss_approx_dimension` → 2**, **`validate_leaf_method`** gates it to Newton,
  **`EvalMetric::for_loss`** maps it to the parity-neutral RMSE eval surface.
- **Offline generator + frozen fixture:** `rmse_uncertainty_fixture.py` (run ONCE
  vs `.venv` catboost==1.2.10, `thread_count=1`, seed 0; 11 trees ≥ 2V+1 for the
  Plan-03 V=5 reuse, depth 3, Newton/1-iter, model_shrink_rate=0,
  boost_from_average=False) → frozen `model.json`/`staged.npy`/`predictions.npy`/
  `config.json` under `crates/cb-oracle/fixtures/rmse_uncertainty/`.
- **`rmse_uncertainty_oracle_test.rs` (NEW):** the per-stage ≤1e-5 oracle —
  Splits / LeafValues (dim-major → leaf-major) / StagedApprox (dim-major →
  object-major, 2-dim) / Predictions (RAW 2-dim).

## Key finding (the load-bearing one)

The fixture model bias is `[0.3151967525, 1.361554006]`, which equals
`[mean(y), log(std(y))] = [mean(y), 0.5·log(var(y))]`. RMSEWithUncertainty applies
this optimal-constant starting approx **even with `boost_from_average=false`** —
`train_model.cpp:858` has an explicit non-BoostFromAverage `else` branch that calls
`CalcOptimumConstApprox` only for RMSEWithUncertainty. Without replicating it, the
staged approx (and every downstream stage) would diverge from tree 0. This is what
makes the diff coupling and the diagonal der reproduce upstream bit-for-bit at the
≤1e-5 bar.
