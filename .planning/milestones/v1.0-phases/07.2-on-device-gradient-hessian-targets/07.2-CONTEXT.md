## Phase Boundary

The second additive GPU sub-phase of the Phase 7 umbrella. It makes the
**gradient/Hessian (der1/der2) and target/derivative path run device-resident**
on the GPU runtimes, ported at structural parity from
`catboost/cuda/targets/`, so derivatives feed the histogram kernels (7.3/7.4)
**without a host round-trip**.

The umbrella `07-CONTEXT.md` (D-01ŌĆ”D-09) and `07.1-CONTEXT.md` (D-7.1-01ŌĆ”10)
are the binding shared context: full **structural-parity** fidelity bar,
**in-kernel atomic** determinism contract (D-03), **empirical GPU-06 epsilon
signed off only in 7.6**, **`rocm`-only validation never in GitHub Actions**,
and the **wave-size-agnostic** policy (no warp-size constants). Those are
**already locked and are NOT re-decided here.** This phase only resolves the
HOW-level choices specific to making the derivative kernels device-resident
and integrating them with the histogram hand-off.

**In scope:** GPU-01 (the gradient/hessian + targets slice only) ŌĆö der1/der2
kernels generic over `R: Runtime`/`F: Float`, device-resident, structurally
mirroring `catboost/cuda/targets/kernel/pointwise_targets.cu(h)` +
`pointwise_target_impl.{h,cpp}`, self-oracled against the Rust CPU der path.

**Out of scope (owned by later sub-phases):** pointwise histograms (7.3),
pairwise histograms (7.4), score/split + the full device-resident tree-grow
loop (7.5), the empirical GPU-06 epsilon sign-off + `rocm` oracle-gate lock-in
(7.6). The **querywise / pairwise / multiclass derivative kernels** in
`targets/kernel/` (`query_rmse`, `query_softmax`, `query_cross_entropy`,
`pair_logit`, `pfound_f`, `yeti_rank_pointwise`, `dcg`, `multilogit`) are
**deferred to the sub-phases that first consume them** (pairwise ŌåÆ 7.4; the
querywise/ranking GPU derivative path is its own follow-up), not ported
speculatively here.

## Implementation Decisions

All decisions below were auto-resolved to the recommended default in `--auto`
mode. They are HOW-level choices within the boundary the umbrella + 7.1
already locked; none reopen D-01ŌĆ”D-09 or D-7.1-01ŌĆ”10. Numbered with a `7.2-`
prefix to avoid collision.

### Loss / derivative scope for this slice (GPU-01 grad/hess)
- **D-7.2-01:** Port the **pointwise** derivative family that upstream
  `pointwise_targets.cu` covers AND that already has a Rust CPU der1/der2 path
  + an authored `#[cube]` kernel in `cb-backend/src/kernels.rs`: **RMSE/MSE**
  (`gradient_kernel`), **Logloss/CrossEntropy** (`logloss_gradient_kernel` +
  `logloss_hessian_kernel`), and the already-authored **Quantile/MAE**
  (`quantile_gradient_kernel`, der2ŌēĪ0) and **Focal**
  (`focal_gradient_kernel` + `focal_hessian_kernel`) kernels. Structurally
  mirror upstream's **dispatch-by-loss** shape (`PointwiseTargetKernel`
  switching on `ELossFunction`). Start narrowest (the 7.1 "resist porting the
  whole set up front" pattern) ŌĆö the remaining pointwise regression losses
  (LogCosh/Lq/Huber/Expectile/Poisson/Tweedie/MAPE) are additive follow-ons
  on the same seam if/when a downstream GPU consumer needs them.
- **D-7.2-02:** **Per-object weights** are folded into der1/der2 at structural
  parity with upstream (`PointwiseTargetKernel` takes `weights`). The scalar
  **`functionValue`** (loss value) upstream also emits is **NOT computed in
  this slice** ŌĆö the histogram path consumes der1/der2, not the loss value;
  add it only when a downstream consumer (metrics on device) needs it.

### Structural-parity target & kernel shape (D-01)
- **D-7.2-03:** Parity target is
  `catboost/cuda/targets/kernel/pointwise_targets.cu(h)` +
  `pointwise_target_impl.{h,cpp}` ŌĆö mirror its **memory layout, blocking, and
  accumulation** (relevs/weights/predictions ŌåÆ der/der2 over a flat object
  range). The existing `#[cube]` kernel **bodies** in `kernels.rs` are the
  derivative formulas; the 7.2 work is the **GPU launch helper + device
  residency**, not re-deriving the math. Elementwise der1/der2 carry **no
  cross-object reduction**, so D-03's atomic-finalize contract is a no-op for
  this slice (it bites in the histogram phases); the f64-finalization contract
  is irrelevant where the op is elementwise.

### Device residency & histogram hand-off (SC-3)
- **D-7.2-04:** der1/der2 outputs stay **device-resident** ŌĆö returned as device
  buffer **handles** (the per-call client + `bytemuck`/`Bytes::from_elems`
  device-memory pattern from `cpu_runtime.rs` / 7.1's primitives), consumable
  **directly by the 7.3 histogram kernels with no host fold inserted** between
  derivatives and histograms. Define the device-side seam now (handle in ŌåÆ
  handles out) so 7.3 plugs in without a round-trip.
- **D-7.2-05:** All cubecl coupling stays **inside `cb-backend`** (mirror
  D-7.1-04). Do **not** widen the frozen `cb-compute` `Runtime`/`Float`
  traits unless a derivative genuinely must cross the trait seam; prefer
  backend-internal launch helpers. `cb-compute` stays cubecl-free;
  `cb-core`/`cb-model` stay unchanged from Phase 3ŌĆō6 form (SC-4 / additive
  boundary).

### Self-oracle & tolerance (SC-2)
- **D-7.2-06:** GPU der1/der2 self-oracle against the **Rust CPU der path**
  (`cb-compute/src/loss.rs` der1/der2 fns ŌĆö the parity *baseline*, not the C++
  oracle) on representative fixtures: random f32 + f64, non-trivial weights,
  and edge cases (empty, single element, length not a multiple of the
  block/cube size, large N). The harness **reports** observed max absolute +
  relative divergence (run-to-run atomic variance from D-03 absorbed) as the
  current tolerance; it does **NOT** hard-code or sign off the GPU-06 epsilon
  ŌĆö that is 7.6's job. Tests live in dedicated `_test.rs` files (source/test
  separation), run on `rocm` in-env.

### Build-gating verification (disk-pressure constraint)
- **D-7.2-07:** Verification is **per-crate**, honoring the standing
  disk-pressure constraint (root ~100% full; full-workspace test links fail):
  `rocm` gets build + **run + self-oracle** in-env on gfx1100; `wgpu` gets
  build + run on host (GPU-04, no ROCm/CUDA needed); `cuda` gets **build/
  compile only** behind its feature gate (no NVIDIA hardware ŌĆö D-07). `rocm`
  GPU tests run **locally/manually, never in GitHub Actions** (umbrella D-06).

### Claude's Discretion
- Concrete kernel decomposition, block/launch geometry, shared-memory sizing,
  how the per-loss dispatch is expressed in CubeCL (`comptime` selector vs one
  kernel per loss), the exact device-buffer handle type handed to the
  histogram seam, and the precise representative fixture set are left to
  research + planning ŌĆö subject to umbrella D-01 (structural parity), D-03
  (atomic reductions where they apply), D-09 (no warp-size constants), and the
  7.2 decisions above.

## Specific Ideas

- "match cuda in catboost" ŌĆö the umbrella's guiding instruction. For 7.2 it
  means the derivative kernel *structure* mirrors
  `targets/kernel/pointwise_targets.cu(h)` (dispatch-by-loss, weights,
  flat-range der/der2), not merely the numeric result.
- Start narrowest: make the **already-authored** pointwise der1/der2 kernels
  device-resident and wire the histogram hand-off; resist porting the
  querywise/pairwise/multiclass derivative set up front (those follow demand
  in 7.4 and the ranking follow-up).
- The der/der2 ops are elementwise ŌĆö the histogram hand-off (handle-in ŌåÆ
  handles-out, no host fold) is the load-bearing 7.2 deliverable, more than the
  arithmetic.
</specifics>
