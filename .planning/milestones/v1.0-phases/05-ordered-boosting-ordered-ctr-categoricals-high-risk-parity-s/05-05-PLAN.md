<objective>
The anti-leakage heart of the phase: Ordered CTR (ORD-03 ordered path) and Ordered boosting (ORD-02). Per D-05/D-06 these are added LAST among the value layers, on top of locked plain CTR (05-04) and locked permutation/fold machinery (05-03), so a per-object divergence localizes to the ordering math alone — never an entangled three-way break.

This is where the per-object intermediate oracle (D-02) earns its keep: a prefix-boundary off-by-one or a compute-after-increment bug is exactly the silent-leakage signature this slice's per-object oracle exists to catch. Per the approved transcribe-then-self-oracle mechanism, the per-iteration ordered approximant is validated INDIRECTLY (final-prediction parity + internal consistency + identity-permutation degeneration — the accepted A2 residual risk).

Purpose: Lock ordered CTR + ordered boosting ≤1e-5 with per-object intermediates, closing ORD-02 and the ordered half of ORD-03.
Output: ordered accumulation in `ctr/online.rs`, ordered approximant path in `boosting.rs`, and the ordered_ctr + ordered_boost per-object oracles.
</objective>

RESEARCH Pattern 1 (read-before-increment — the no-leakage property):
For each doc in permutation order: READ accumulated (goodCount, totalCount) for its bucket, compute CTR, THEN increment the bucket with this doc's own class. Computing AFTER incrementing leaks the doc's own label.

RESEARCH Pattern 3 (body/tail ordered prefix): leaf deltas estimated on the body, applied to the tail; tail doc's approximant never depends on itself. approx_calcer.cpp:566-600 (UpdateApproxDeltasHistoricallyImpl), :706 (CalcApproxDeltaSimple).
