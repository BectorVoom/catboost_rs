<objective>
Close the single open BLOCKER (CR-01) from 03-VERIFICATION.md: the random_strength
split-score perturbation magnitude (`scoreStDev`) is computed over the
SAMPLED/control-masked derivative vector (`score_weighted_der1`) instead of the
full, un-sampled AveragingFold weighted derivatives (`weighted_der1`). Upstream
`CalcDerivativesStDevFromZeroPlainBoosting` (greedy_tensor_search.cpp:92-107)
operates on the full fold — exactly as the LEAF path in this same file already
does (boosting.rs leaf path uses `&weighted_der1`). The masked vector zeroes
control-false entries while `derivatives_std_dev_from_zero` still divides by the
full length `n`, biasing `scoreStDev` low whenever `bootstrap_type != No` is
combined with `random_strength != 0`, producing wrong split scores -> wrong tree
structure -> failing the <=1e-5 gate.

The bug is currently MASKED by the oracle suite: every existing `random_strength`
fixture pins `bootstrap_type=No`, so `score_weighted_der1 == weighted_der1`. The
fix is therefore untestable until a cross-scenario fixture exists. This plan
builds that fixture FIRST (so it fails RED against the buggy code), then applies
the one-line fix (GREEN).

Purpose: TRAIN-05 (regularization) is only PARTIALLY SATISFIED until the
random_strength + sampling interaction is correct and oracle-locked. This is the
phase that claims random_strength; the fix belongs here, not a later phase.

Output: a new cross-scenario oracle fixture
(`regularization/random_strength_bernoulli`), a first-tree oracle test gating it
<=1e-5, and the corrected `boosting.rs` std-dev input.

WR-06 (n-from-slice-length coupling in score.rs) is assessed OUT OF SCOPE — see
the "WR-06 disposition" note in Task 2. The mandatory, gating change is the
`&weighted_der1` fix only.
</objective>
