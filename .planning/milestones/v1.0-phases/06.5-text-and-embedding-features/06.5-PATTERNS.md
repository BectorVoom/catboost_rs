## Pattern Assignments

### `crates/cb-train/src/estimated/online_text.rs` & `online_embedding.rs` (online estimator, ordered prefix) — D-03 LOAD-BEARING

**Analog:** `crates/cb-train/src/ctr/online.rs` (read it in full; it is THE pattern to mirror).

This is the single most important analog. The four target-aware calcers (NaiveBayes, BM25, LDA, KNN) replicate the read-before-update prefix loop verbatim, swapping the per-bucket `[N0,N1]` count for the per-class frequency table / incremental scatter / HNSW insertion.

**Read-before-update prefix loop** (`ctr/online.rs:284-317`, the exact structure to mirror):
```rust
for &doc_i in permutation {
    let doc = doc_i as usize;
    // ... checked .get(doc) for bins / class ...
    let elem = counts.get(bucket);
    // READ the prefix counts BEFORE incrementing (online_ctr.cpp:303-304).
    let (n0, n1) = elem.map_or((0, 0), |e| (e[0], e[1]));
    // ... compute the calcer output from prefix state, write to value[doc] ...
    // INCREMENT after read (learn set): ++N[targetClass[doc]].
    if let Some(elem) = counts.get_mut(bucket) {
        if let Some(c) = elem.get_mut(class) { *c += 1; }
    }
}
```
For text calcers the upstream loop to transcribe (RESEARCH Pattern 1) is `base_text_feature_estimator.h:59-64`: `Compute(...text, line...)` (READ prefix state) THEN `calcerVisitor.Update(target.Classes[line], text, &featureCalcer)` (write). Column-major output layout `learnFeatures[f * samplesCount + line]`.

**Object-order vs permutation-order result split** (`ctr/online.rs:325-336` `OnlineCtrPrefix` / `:413-423` `OrderedCtrPrefix`): online estimators return per-object vectors indexed by `doc` (NOT permutation position), and may also expose the per-step running state for a monotone internal-consistency anchor (`:425-455` `per_bucket_monotone`) — reuse this shape for per-prefix instrumented oracle checks (RESEARCH SC-2 per-prefix intermediate).

**Error/parity discipline** (`ctr/online.rs:44-45, 163-180`): `use cb_core::{CbError, CbResult};`, length-mismatch → `CbError::Degenerate`, out-of-range permutation index → `CbError::Degenerate`. Integer counts are EXACT integer accumulation (do NOT route through `sum_f64`); only FLOAT reductions (NaiveBayes LogProb, BM25 score, textLen, LDA scatter sums) route through `cb_core::sum_f64` (D-04/D-08). See the module-doc "Parity discipline" block (`:31-42`).

**Learn permutation source** (`crates/cb-train/src/fold.rs`): the `Fold { permutation: Vec<i32>, .. }` from `create_folds` (`fold.rs:181-256`) is the SAME permutation CTRs/ordered-boosting use. Online estimators take `&fold.permutation`; do NOT generate a new permutation (RESEARCH "Don't Hand-Roll"; `fold.cpp:181 GetLearnPermutationArray`). The dictionary is built ONCE (target-independent, offline); only the per-class calcer state is per-fold/prefix (RESEARCH Anti-Pattern "Recomputing the dictionary per fold").

### `crates/cb-compute/src/text_calcers.rs` & `embedding_calcers.rs` & `lda_linalg.rs` (pure numeric primitive)

**Analog:** `crates/cb-compute/src/score.rs` (read 1-85) — exact for the math-module style.

**Module-doc + source-of-truth header** (`score.rs:1-23`): every function cites the upstream `.cpp` line, states the `sum_f64` routing rule, and documents which raw `+=` cells are sanctioned upstream-order scatter (D-04 "only documented upstream-order scatter cells use raw `+=`").

**Imports + sum_f64 routing** (`score.rs:25-27, 49-55`):
```rust
use cb_core::{std_normal, sum_f64, TFastRng64};
// ...
pub fn l2_split_score(leaves: &[LeafStats], scaled_l2: f64) -> f64 {
    let terms: Vec<f64> = leaves.iter().map(|&s| add_leaf_plain(s, scaled_l2)).collect();
    sum_f64(&terms)   // collect-then-sum_f64, never a zero-seeded fold (D-08)
}
```
Apply this `collect Vec<f64> → sum_f64` shape to: NaiveBayes `textLen`/`value` accumulation, BM25 per-class `scores[c]`, LDA scatter/GEMM reductions, KNN neighbor mean. Math accumulates in `double` internally, stored as `f32` feature (RESEARCH Pitfall 6); `Softmax` operates on `double` then downcasts (`helpers.h:33-47`).

**Calcer output-width discipline** (RESEARCH Pitfall 5 — transcribe exactly): NaiveBayes `BaseFeatureCount(nClasses) = nClasses>2 ? nClasses : 1`; BM25 `= nClasses`; BoW `= NumTokens`; LDA `= ProjectionDim (+nClasses if probs)`; KNN `= nClasses` (clf) / `1` (reg). Getting width wrong shifts every downstream feature index.

**BoW lockstep walk** (RESEARCH Code Examples §BoW, `bow.cpp:7-21`): walks `TText` and active feature ids BOTH sorted ascending — order is load-bearing; mirror the two-pointer merge. BoW is the ONLY target-independent calcer (no online wiring).

**LDA/KNN landmines** (RESEARCH Pitfall 1/2, A1/A2): `lda_linalg.rs` has NO existing analog for the eigensolver — the closest reduction-style sibling is `pairwise_scoring.rs`. Plan an instrumented-GT spike (D-07) BEFORE committing to hand-rolled f32 vs LAPACK-binding; any LAPACK/`nalgebra` crate is gated behind `checkpoint:human-verify` (A3). KNN must NOT use a third-party HNSW crate (parity fail) — exact `online_hnsw` port or instrumented neighbor-id GT.

### `crates/cb-data/src/text/tokenizer.rs` & `dictionary.rs` & `digitizer.rs` & `text.rs`

**Analog:** `crates/cb-data/src/cat_hash.rs` (read 1-60) — exact for the cb-data string-input, verbatim-from-C++, checked-access module style.

**Module-doc transcription-source header** (`cat_hash.rs:1-29`): cite the exact upstream file+line (`tokenizer.cpp`, `dictionary_builder.cpp`, `text.h`), warn against crate-sourcing a generic implementation (a generic `cityhash`/`tokenizers` crate "would produce different tail mixing and silently break parity" — same risk applies to a generic tokenizer/dictionary; transcribe, don't crate-source).

**Checked-access + wrapping/bounds discipline** (`cat_hash.rs:31-51`): `use cb_core::{CbResult, CbError};`, every slice access via `.get(off..off+n)` (never raw index), no `unwrap`/`expect`/`panic` (V5 Input Validation: empty text / unknown token / k>N / dim mismatch → typed `CbError`, never panic — replicate upstream `CB_ENSURE`).

**Deterministic dictionary sort** (RESEARCH Pattern 3, `dictionary_builder.cpp:146-162`) — the load-bearing reproducibility: sort indices by `(count DESC, token-string ASC)`, filter strict `count >= OccurrenceLowerBound`, truncate to `MaxDictionarySize`, assign ids `StartTokenId++`. The deterministic-comparator + `PerfectHash` remap pattern already exists in `cat_hash.rs` (PerfectHash) and `borders.rs:209 select_borders_greedy_logsum` (deterministic select); model the comparator on the borders-select discipline.

**TText sorted-RLE** (RESEARCH Pattern 2, `text.h:155-164`): `Vec<(tokenId:u32, count:u32)>` sorted ASC by tokenId, built by sort-then-RLE-collapse. Small upstream-shaped data type — model on `ctr/online.rs::TCtrHistory` (`online.rs:58-88`: `#[derive(..)]` struct + `new`/`increment` checked methods).

**Defaults to pin** (RESEARCH Pitfall 3, A4 — VERIFY `wc -l` of the actual learn split): `MaxDictionarySize = 50000`, `OccurrenceLowerBound = (learnPoolSize < 1000 ? 1 : 5)` (rotten_tomatoes 1000 rows → 5), unknown-token policy = Skip, EndOfSentence = Skip. ByDelimiter default `Delimiter=" "`, `SkipEmpty=true`, lowercasing.

**Module wiring** — `crates/cb-data/src/text/mod.rs`: copy the `#[path = "..."] pub mod ...` layout from `crates/cb-train/src/ctr/mod.rs:31-40`; declare in `cb-data/src/lib.rs` alongside `mod cat_hash;` and re-export the public surface (`lib.rs:27-46`).

## Shared Patterns

### Deterministic float reduction (D-04 / D-08)
**Source:** `cb_core::sum_f64` (`crates/cb-core/src/reduction.rs:32`), usage `crates/cb-compute/src/score.rs:49-55`.
**Apply to:** every float summation in all 5 calcers (NaiveBayes LogProb/textLen, BM25 scores/norms, LDA scatter/GEMM, KNN mean) and the online prefix float state. `collect Vec<f64> → sum_f64`, never a zero-seeded `.fold`/`.sum()`. Only documented upstream-order scatter cells use raw `+=` (LDA `IncrementalCloud::Update` GEMM rank cells).

### Read-before-update ordered prefix (D-03 leakage control)
**Source:** `crates/cb-train/src/ctr/online.rs:284-317` (loop), `fold.rs:181-256` (permutation).
**Apply to:** NaiveBayes, BM25, LDA, KNN online estimators. Compute output from prefix state THEN update with this line's label — per-line, over `&fold.permutation` in object→value indexing. NOT BoW (target-independent).

### Verbatim-from-C++ module doc + checked access (INFRA-02 / V5)
**Source:** `crates/cb-data/src/cat_hash.rs:1-51`, `crates/cb-compute/src/score.rs:1-23`.
**Apply to:** every new source module — header cites upstream `file:line`, states summation routing, warns against crate-sourcing; all slice access via `.get(..)`/`.get_mut(..)`, typed `CbError` on malformed input (empty text, unknown token, k>N, embedding-dim mismatch), no `unwrap`/`expect`/`panic`/raw-index.

### `#[path=]` submodule wiring
**Source:** `crates/cb-train/src/ctr/mod.rs:31-40`.
**Apply to:** `cb-data/src/text/mod.rs`, `cb-train/src/estimated/mod.rs`; declare in the crate `lib.rs` (`cb-data/src/lib.rs:27-46`, `cb-compute/src/lib.rs:21-79` for `_test` sibling declarations) and re-export the public surface.

### Source/test separation (INFRA-06 — mandatory)
**Source:** `crates/cb-train/src/ctr/online_test.rs` (sibling `_test.rs`), `crates/cb-oracle/tests/per_stage_oracle_test.rs` (integration `tests/`).
**Apply to:** ALL new test code. Unit tests → `<module>_test.rs` sibling declared `#[cfg(test)] mod <module>_test;` in `lib.rs`; oracle tests → `crates/cb-oracle/tests/*_oracle_test.rs`. NO inline `#[cfg(test)] mod tests`.
