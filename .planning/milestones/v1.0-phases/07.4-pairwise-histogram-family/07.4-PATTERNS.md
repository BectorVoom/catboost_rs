### `crates/cb-backend/src/kernels.rs` ŌĆö the 4 `#[cube]` pairwise families (kernel, batch/transform)

**Analog:** `pointwise_hist2_nonbinary_kernel` (lines 621-681), `pointwise_hist2_half_byte_kernel` (758), `pointwise_hist2_binary_kernel` (880), `block_reduce_atomic_kernel` (1049-) ŌĆö ALL in this same file. Consts: `HIST_SHMEM` (line 40), `HALF_BYTE_BINS` (689), `BLOCK_REDUCE_SHMEM` (29).

**Comptime-bits + grid-stride + atomic-merge skeleton to adapt** ŌĆö the SHIPPED non-binary kernel (lines 621-681). The pairwise non-binary kernel keeps this skeleton verbatim and changes EXACTLY three things: (1) loop over PAIRS not objects, (2) read TWO bins per feature, (3) 2-channel ŌåÆ 4-channel via `Compare ŌåÆ histId`:
```rust
// SHIPPED 7.3 ŌĆö crates/cb-backend/src/kernels.rs:621-681 (the exact template)
#[cube(launch)]
pub fn pointwise_hist2_nonbinary_kernel<F: Float>(
    der1: &Array<F>, weight: &Array<F>,
    cindex: &Array<u32>, indices: &Array<u32>,
    bin_sums: &Array<Atomic<F>>,
    n_features: u32,
    #[comptime] bits: u32,
) {
    let n_bins = comptime!((1u32 << bits) as usize);   // line 634 ŌĆö comptime n_bins
    let n = indices.len();                              // line 640 ŌĆö cindex stride = n
    let stride = CUBE_COUNT * (CUBE_DIM as usize);      // line 649 ŌĆö topology, NO 32/64 literal (D-09)
    let mut i = ABSOLUTE_POS;
    while i < n {                                       // line 651 ŌĆö grid-stride idle-guard (T-7.1-01)
        let obj = indices[i] as usize;
        let d = der1[obj]; let w = weight[obj];
        let mut feature = 0usize;
        while feature < n_features_usize {
            let bin = cindex[feature * n + obj] as usize;  // line 671 ŌĆö ONE bin per (object, feature)
            let cell = (feature * n_bins + bin) * 2usize;  // line 672 ŌĆö 2 CHANNELS (7.3)
            bin_sums[cell].fetch_add(d);                   // line 674 ŌĆö ch0 = ╬Ż der1
            bin_sums[cell + 1usize].fetch_add(w);          // line 675 ŌĆö ch1 = ╬Ż weight
            feature += 1usize;
        }
        i += stride;
    }
}
```

**Pairwise adaptation (the NEW logic ŌĆö D-7.4-03 seam + the 4-channel Compare mapping):**
- Replace `der1/weight/indices` inputs with `pair_i: &Array<u32>`, `pair_j: &Array<u32>` (two parallel u32 arrays = `uint2 pairs`, RESEARCH Open Q1 recommendation), and `pair_weight: &Array<F>` (the ONLY per-pair value ŌĆö NO der1; D-7.4-03).
- Loop bound is `pair_weight.len()` (= `n_pairs`); but the cindex stride MUST stay `n_objects` (passed as a SEPARATE input or `cindex.len()/n_features`) ŌĆö **Pitfall 3**: pair indices `oi/oj` are OBJECT ids; indexing `cindex[feature * n_pairs + oi]` is the #1 stride bug. Read `bin1 = cindex[feature * n_objects + oi]`, `bin2 = cindex[feature * n_objects + oj]`.
- `cell = (feature * n_bins + bin) * 4usize + hist_id` ŌĆö **4 channels**, where `hist_id Ōłł {0,1,2,3}` from the comptime-pruned Compare predicate (`#[comptime] one_hot: bool`): non-one-hot `(bin1 >= bin2) == flag`; one-hot `bin1 == bin2` (`split_properties_helpers.cuh:240/260`; RESEARCH Pattern 2 / A6). Which fold/bin the cell attaches to + the exact `Compare ŌåÆ histId` arithmetic is the genuinely-new piece to finalize in Plan A against `compute_pair_hist_loop.cuh::AddPair` + `split_properties_helpers.cuh` ŌĆö validated immediately by the self-oracle (a wrong mapping diverges).
- `bin_sums[cell].fetch_add(w)` ŌĆö single weight value per channel (D-03 atomic merge; same `Atomic<F>::fetch_add` as line 674).

**8-bit-atomics family** (D-7.4-02 distinct family): a SEPARATE `#[cube]` symbol `pairwise_hist_8bit_atomics_kernel<F>` + a separate launch arm, even though the MVP body is structurally `bits = 8` non-binary with global atomics (RESEARCH Pattern 3 / A2). Upstream's per-thread `CachedBinsLeq/Ge` cache is documented as a perf follow-up; the MVP uses the SAME direct-per-pair `fetch_add` the shipped 7.3 MVP uses for every width.

**Half-byte / binary families**: adapt `pointwise_hist2_half_byte_kernel` (758) and `pointwise_hist2_binary_kernel` (880) the same way (loop pairs, two bins, 4 channels). These take NO one-hot overlay upstream (RESEARCH Pattern 2). Half-byte: fixed `HALF_BYTE_BINS = 16` const (line 689 precedent ŌĆö a bin count, NOT a warp literal); binary: 2-bin. Both mask their nibble/bit raw (`& 15`/`& 1`) per the shipped precedent.

**Reserved shared-mem const** (perf follow-up, NOT the MVP): add a `pub(crate) const PAIR_HIST_SHMEM` next to `HIST_SHMEM` (line 40) only if/when the per-block shared-mem pre-reduce lands; the MVP direct-global-atomic fill needs no shmem (RESEARCH A2 / D-7.4-04). Couple any new shmem const to the launch geometry with the `const _: () = assert!(...)` pattern (gpu_runtime.rs precedent).

**Generics-float + bounds-guard + comptime discipline** (AGENTS.md / D-09): every kernel `<F: Float>`; grid-stride `while p < n_pairs` idle-guard; `#[comptime] bits`/`#[comptime] one_hot` resolved host-side (JIT-pruned, zero device divergence); if-as-STATEMENT only; NO literal `32`/`64`/`16`/`& 31`/`& 6`/`tiled_partition`/`thread_rank` in any stride/offset (**Pitfall 4** ŌĆö upstream's tile distribution is perf, not semantics; the grid-stride MVP sidesteps it entirely).

### host-reference 4-channel pairwise histogram (private fn INSIDE `kernels/pairwise_hist.rs`; test utility)

**Gather-then-`cb_core::sum_f64` (the SHIPPED 2-channel shape to generalize):**
```rust
// SHIPPED 7.3 ŌĆö crates/cb-backend/src/kernels/pointwise_hist.rs:105-152
use cb_core::sum_f64;                                          // line 53 ŌĆö the ONLY sanctioned ordered reduction
fn host_reference_hist2(der1, weight, cindex, indices, n_bins, n_features) -> Vec<f64> {
    let n = der1.len();
    let mut delta_members:  Vec<Vec<f64>> = vec![Vec::new(); n_features * n_bins];
    let mut weight_members: Vec<Vec<f64>> = vec![Vec::new(); n_features * n_bins];
    for feature in 0..n_features {
        for &obj in indices.iter() {                          // ascending object order = the kernel's walk
            let obj = obj as usize;
            let bin = cindex[feature * n + obj] as usize;
            assert!(bin < n_bins, "...");                     // line 131 ŌĆö same invariant as the host guard
            let cell = feature * n_bins + bin;
            delta_members[cell].push(der1[obj]);
            weight_members[cell].push(weight[obj]);
        }
    }
    let mut out = vec![0.0_f64; n_features * n_bins * 2];      // 2 channels (7.3)
    for feature in 0..n_features { for bin in 0..n_bins {
        let cell = feature * n_bins + bin;
        let base = (feature * n_bins + bin) * 2;
        out[base]     = sum_f64(&delta_members[cell]);        // line 147 ŌĆö ordered, NOT .sum()
        out[base + 1] = sum_f64(&weight_members[cell]);
    }}
    out
}
```
**Pairwise generalization** (RESEARCH "fresh host-reference"): allocate `4 * n_features * n_bins` member vecs; iterate PAIRS in ascending pair order; per (feature) compute `bin1 = cindex[feature*n_objects + oi]`, `bin2 = cindex[feature*n_objects + oj]`; compute `hist_id` with the SAME Compare predicate (and `one_hot` flag) the kernel uses; `members[(feature*n_bins + bin)*4 + hist_id].push(pair_weight[p])`; fold each cell via `sum_f64` (ordered, D-05 ŌĆö NEVER naive `.sum()`). The `assert!(bin < n_bins)` invariant carries (line 131). A wrong channel mapping diverges immediately against this reference (validates A6).
