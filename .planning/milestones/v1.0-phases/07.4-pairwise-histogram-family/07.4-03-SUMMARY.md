# Phase 7.4 Plan 03: Half-byte Pairwise Histogram Family Summary

**The structurally DISTINCT half-byte (4-bit, 16-bin x 4-channel) pairwise histogram fill runs device-resident on rocm — a separate `#[cube]` symbol + separate launch arm with a FIXED 16-bin comptime line (the `HALF_BYTE_BINS` precedent) and nibble-masked bins, NO one-hot overlay, reusing the Plan A FROZEN 4-channel layout and self-oracle harness unchanged, bit-exact (0.0 divergence) against the ordered host reference across empty/1/37/large-N and the device-resident hand-off.**

## Accomplishments
- `pairwise_hist_half_byte_kernel<F>` — the DISTINCT half-byte family (D-7.4-02): a SEPARATE `#[cube(launch)]` symbol with `n_bins` fixed at `comptime!(HALF_BYTE_BINS)` (16) and the read bins masked to the nibble (`& comptime!((HALF_BYTE_BINS as u32) - 1)`, the shipped 7.3 half-byte precedent). It accumulates each per-pair weight into the FROZEN 4-channel cell via direct `Atomic<F>::fetch_add` over the non-one-hot Compare->histId mapping. The half-byte family takes NO `one_hot` arg — upstream `pairwise_hist_half_byte.cu` has no one-hot variant (RESEARCH Pattern 2) — so the kernel hard-codes the non-one-hot `(ge, gt)` flag-collapsed writes (exactly the `else`-branch of the non-binary kernel).
- The 16-bin device-resident seam: `launch_pairwise_hist_half_byte_handle`/`_into` (no readback on the hand-off), `launch_pairwise_hist_half_byte` readback wrapper — clones of the Plan B 8-bit arm with `n_bins` pinned to `HALF_BYTE_BINS` (16, validated host-side), no `one_hot` parameter, and the half-byte kernel dispatched, reusing `pair_hist_binsums_len`, the full guard block (LengthMismatch / OutOfRange / `checked_mul` overflow / value-range / empty short-circuit), the backend-dispatched channel (f64 on rocm/cuda/cpu, f32 on wgpu), and the no-readback seam UNCHANGED.
- The `half_byte` self-oracle (n_bins=16, n_objects=64 so bins span the full 4-bit range): empty (no launch/read-back), n_pairs 1/37/10000, and the device-resident hand-off — all asserting against `host_reference_pairwise_hist(..., n_bins=16, one_hot=false)` and REPORTING max abs/rel divergence.
- Cross-backend gate green: rocm run+self-oracle (bit-exact), wgpu build+host run (f32 channel, also bit-exact on the host fixture), cuda compile-only; SC-4 boundary held (cb-compute cubecl-free, cb-compute/cb-core/cb-model byte-unchanged).

## Decisions Made
- **Separate symbol + arm with no one-hot (D-7.4-02):** kept `pairwise_hist_half_byte_kernel` and `launch_pairwise_hist_half_byte_*` fully distinct from the non-binary/8-bit paths — structural parity with upstream's own file split. The MVP body equals the non-binary kernel's non-one-hot path with the nibble mask and n_bins fixed at 16; the half-byte family takes NO one-hot overlay (no upstream one-hot .cu).
- **`n_bins` fixed at comptime HALF_BYTE_BINS, no `bits` runtime arg:** the half-byte family IS 4-bit/16-bin, so the launch arm takes no `bits` parameter and the kernel uses `comptime!(HALF_BYTE_BINS)`; the host arm validates `n_bins == HALF_BYTE_BINS` (`CbError::Degenerate` otherwise).
- **Nibble mask (& 15) on the read bins:** carried from the shipped 7.3 pointwise half-byte precedent; the host reference indexes RAW, so the host-side value-range guard (`cindex < 16`) keeps the kernel and reference in lock-step.
- **FROZEN 4-channel layout reused verbatim:** index `(feature*16+bin)*4+histId`, buffer length `n_features*16*4` via `pair_hist_binsums_len` — never `* 2` (Pitfall 2).
- **n_objects=64 in the fixture:** chosen `> 16` so the deterministic feature-major bin spread exercises the full 4-bit range (bins 0..15).
