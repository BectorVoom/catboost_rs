Close the architectural gap 08-07 surfaced: the facade `fit()` was hardwired to the cpu-only `cb_backend::CpuBackend` (the sole `cb_compute::Runtime` impl), so `catboost-rs` could not even COMPILE under `--features rocm/wgpu/cuda` and the per-backend wheels could not be built.

Make the facade training path GENERIC over the CubeCL backend per the user's directive ("I will use wgpu and cuda, so implement generics") — not a rocm-only patch. Introduce a single `GpuBackend` that implements the 2-method `cb_compute::Runtime` trait generically over `SelectedRuntime` (the feature-switched `cubecl::Runtime` alias already in `cb-backend/src/lib.rs`: cpu->CpuRuntime, wgpu->WgpuRuntime, cuda->CudaRuntime, rocm->HipRuntime), computing derivatives on-device through the EXISTING, oracle-validated Phase-7.2 der seam. Wire `builder.rs` to select `CpuBackend` vs `GpuBackend` by feature, and add `wgpu`/`cuda` passthrough to the Python crate. Then build and validate the rocm wheel in-env, discharging the deliverable 08-07 deferred.

This is an MVP-scope genericization: GPU-accelerated DERIVATIVES through the proven host training loop (`cb_train::train<R: Runtime>` is already fully generic — no concrete backend in its loop). The fuller device-resident grow loop (Phase-7 `grow_oblivious_tree`/`grow_boosting_pass`, depth>1 partition-aware histograms, Newton, multiclass GPU der) stays a documented forward dependency — out of scope here, exactly as Phase-8 CONTEXT scopes GPU kernel work out.

- `compute_gradients`: reuse CpuBackend's exact shape validation (approx_dimension == 0 reject, non-divisible length reject, per-object `target.len()==n` check, empty short-circuit). Then dispatch by `loss`:
