## Warnings

### WR-01 — Finite `f32::MIN` argmin sentinel can drop a real candidate for negative-score calcers
**Files:** `kernels.rs:1858`, `kernels.rs:2561`

The argmin seeds `minimal_score = F::new(f32::MIN)` (finite ≈ -3.4e38). L2/Cosine scores are ≥0 so they always beat it, but SolarL2/LOOL2/SatL2 produce strictly negative terms (`kernels.rs:1951/1975/2003`); a candidate more negative than `f32::MIN` fails `score > my_gain`, keeps `my_idx = n_candidates`, and the host discards it (`gpu_runtime.rs:1562`). The CPU oracle uses `f64::NEG_INFINITY` (`score_split.rs:228`, `score.rs:32`). Latent device-vs-CPU argmin disagreement on a degenerate Solar/LOO/Sat fixture.

**Fix:** track "have I seen any candidate" per lane (update on first real candidate) or use the most-negative representable channel value.

### WR-02 — `launch_der_binary_into` launches `f64` unconditionally; breaks on a true wgpu backend
**Files:** `gpu_runtime.rs:481-498`, `gpu_runtime.rs:611-621`, `gpu_runtime.rs:768-822`

The der seams launch `::<f64, SelectedRuntime>` with no `#[cfg(wgpu)]` f32 split (unlike every other seam in the file). `grow_boosting_pass_into:2500-2509` reads back as `cast_slice::<u8,f64>`. WGSL has no `f64` type, so a genuine wgpu backend fails to JIT — contradicting the "every backend" claim.

**Fix:** add the f32 channel split or `compile_error!`/typed-reject under wgpu.

### WR-03 — Pairwise winner reports the f32-reduced gain instead of the available f64 score; near-tie flip risk
**Files:** `gpu_runtime.rs:4156-4164` vs `gpu_runtime.rs:1571-1581`

Pointwise uses `scores.get(best_c)` (f64); pairwise stores `best_score as f32` from the device reduce, and on wgpu the reduce ran in f32 (`gpu_runtime.rs:4101`). Two host-distinct f64 scores collapsing to one f32 tie on-device resolve by index — potentially a different split than the CPU f64 strict-`>`. The exact f64 `scores` exist host-side before the reduce.

**Fix:** resolve the winner over host f64 `scores` (see IN-01).

### WR-04 — `partition_update_kernel` unguarded `part_stats[leaf_of[obj]*2]` store; grow loop never validates the device-produced `leaf_of`
**Files:** `kernels.rs:2300-2325`, `gpu_runtime.rs:1946-2004`, `gpu_runtime.rs:2238`

Doc claims "host validated `leaf_of[obj] < n_parts`," but `leaf_of_h` is produced on-device by `partition_split_kernel` and passed to update with no read-back/re-validation. Correct by construction for depth==1, but the safety argument names a validation that does not run; the depth>1 forward dependency could drift into an OOB device atomic (UB).

**Fix:** add in-kernel `if part*2+1 < part_stats.len()` guard (matching `kernels.rs:2131`) and correct the doc.

### WR-05 — Trailing `border = n_bins-1` no-op split is scored and argmin-eligible (pointwise)
**Files:** `kernels.rs:1869-1892`, `gpu_runtime.rs:1572`

Pointwise enumerates `0..n_bins`; the last border puts all bins left / right empty — a non-split that can win. Device and CPU oracle agree (no parity break), but diverges from upstream and from the pairwise path which correctly uses `n_splits = n_bins-1` (`gpu_runtime.rs:3987`).

**Fix:** bound the pointwise loop to `0..n_bins-1` in kernel + both references, or force the trailing candidate's score to the sentinel.

### WR-06 — `max_divergence` truncates on length mismatch under release profile
**Files:** `score_split.rs:81-92`, `grow_loop.rs:60-71`

Length guarded only by `debug_assert_eq!`, compiled out in release; a shorter device buffer silently compares the prefix and reports spuriously low divergence, masking a truncated-read-back fault. Callers assert separately (belt-and-suspenders), but the helper is unsafe to reuse.

**Fix:** unconditional length check inside the helper.

## Info

### IN-01 — `select_best_split_kernel` is a decorative device round-trip
`select_best_split_kernel` is an O(n_candidates) device round-trip over already-host-resident pairwise scores, then still finishes argmax host-side (`gpu_runtime.rs:4140-4154`); decorative and adds the WR-03 f32 collapse.
**Fix:** select directly over host f64 `scores`.

### IN-02 — `find_optimal_split_kernel` copy-paste calcer arms
One ~240-line function with five copy-pasted comptime calcer arms (`kernels.rs:1835-2070`); ~10 duplicated guard ladders.
**Fix:** factor a per-leaf `#[cube]` calcer term.

### IN-03 — `gpu_runtime.rs` size
`gpu_runtime.rs` (4400+ lines) now also hosts the entire 7.5 seam; the tracked IN-03 split predates it.
**Fix:** include the 7.5 seams in that refactor.

### IN-04 — Duplicated fixture builders
Five byte-identical fixture builders duplicated across `grow_loop.rs:95-115/705-727/979-1009` and `score_split.rs:126-154/868-902`; drift would desync the oracles.
**Fix:** shared fixture helper.

### IN-05 — Scan seam implicit precondition
The scan seam relies implicitly on the fill path's `cindex` value-range guards (`gpu_runtime.rs:1672-1735`); correct today but a future external `bin_sums` caller would bypass them.
**Fix:** document the precondition on the kernel.
