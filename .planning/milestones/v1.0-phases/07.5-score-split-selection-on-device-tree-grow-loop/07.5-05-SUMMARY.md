# Phase 07.5 Plan 05: Cosine/Solar/LOO/Sat Device Score Calcers + Cosine Grow-Loop Summary

Completed the device **score-calcer family**: `find_optimal_split_kernel` gains the **Cosine/NewtonCosine, SolarL2, LOOL2, SatL2** comptime arms ALONGSIDE the Plan-A L2 arm, each transcribed bit-for-bit from the FROZEN `cb-compute/src/score.rs` and folded in the f64 device channel (D-03). A `score_fn` selector is threaded through the Plan-A launch seam and the Plan-C grow-loop driver. Cross-oracled on gfx1100 (rocm, wave32): every variant's device score matches its `cb-compute` reference (`cosine_split_score` / `multi_dim_split_score` Solar/LOO/Sat) to f64 rel-div ~2e-16 (Cosine) / exactly 0.0 (Solar/LOO/Sat), the device argmin under each fn picks the SAME `(feature, bin)` as the CPU reference EXACTLY (STRUCTURE), every guard ladder yields finite scores (never NaN/Inf) on degenerate leaves, AND a depth-1 device tree grown with `SCORE_FN_COSINE` has the SAME split sequence + per-object `leaf_of` as the inline CPU Cosine greedy reference. Every score function D-7.5-01 lists (L2 from Plan A + the four here) now lands in this phase — not a "v1 with L2 only".

## Deviations from Plan

**1. [Rule 3 - Blocking] `score_fn` threaded through `grow_oblivious_tree` and `grow_boosting_pass` pinned to `SCORE_FN_L2`**
- **Found during:** Task 2 (wiring the Cosine grow-loop check).
- **Issue:** Plan C's `grow_oblivious_tree(_into)` and Plan 04's `grow_boosting_pass(_into)` both called `launch_find_optimal_split_pointwise_into` with the hardcoded L2 arm and had no `score_fn` parameter. The Cosine grow-loop check needs `grow_oblivious_tree` to score with Cosine.
- **Fix:** Added a `score_fn: u32` parameter to `grow_oblivious_tree(_into)` (forwarded to the launch seam); `grow_boosting_pass_into` passes `SCORE_FN_L2` explicitly so the Plan-04 multi-tree boosting pass (L2-only, out of this plan's score-fn scope) keeps its public signature and behavior unchanged. The four Plan-A `score_split` L2 tests and the existing Plan-C `grow_loop` L2 tests were updated to pass `SCORE_FN_L2` (the new required arg). No behavior change for any existing path.
- **Rule:** Rule 3 (blocking — the existing seam could not select Cosine).
- **Files modified:** `crates/cb-backend/src/gpu_runtime.rs`, `crates/cb-backend/src/kernels/score_split.rs`, `crates/cb-backend/src/kernels/grow_loop.rs`
- **Commits:** `be9c707` (seam + L2-test threading), `06413d0` (grow-loop)

**2. [Rule 2 - Correctness] Typed `CbError::OutOfRange` on an unknown `score_fn` selector**
- **Found during:** Task 1 (threading the selector).
- **Issue:** A `score_fn` value outside {L2, Cosine, Solar, LOO, Sat} would launch a kernel that monomorphizes NO arm, leaving every candidate score at the zero-init value (a silent wrong-arm / garbage-score path).
- **Fix:** `launch_find_optimal_split_pointwise_into` validates `score_fn` against the five known selectors and returns a typed `CbError::OutOfRange` BEFORE any launch (no silent dispatch).
- **Rule:** Rule 2 (correctness / input validation at the host-device boundary).
- **Files modified:** `crates/cb-backend/src/gpu_runtime.rs`
- **Commit:** `be9c707`

**Total deviations:** 2 auto-fixed (1 blocking, 1 correctness). No scope creep; the multi-tree boosting pass and its public signature are untouched.
