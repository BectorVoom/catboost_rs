# Phase 07.5 Plan 03: Score/Split Selection — Host-Light Single-Tree Grow Loop Summary

The host-light, device-resident single-tree grow loop (`grow_oblivious_tree`) — the first complete vertical slice of GPU training: it grows one oblivious tree entirely device-resident over `SelectedRuntime`, chaining the FROZEN 7.3 histogram fill -> Plan-A device score + deterministic argmin -> ONE O(1) `BestSplit` read-back -> the host integer split decision (strict first-wins) -> the Plan-C `partition_split` (forward-bit doc-routing) -> `partition_update` (per-partition Σ der1 / Σ weight reduce), over persistent handles threaded through ONE `ComputeClient`, then reads back ONLY the `2^depth` part-stats at the leaves and computes leaf values via the FROZEN `cb_compute::calc_average`. Cross-oracled in `kernels/grow_loop.rs::single_tree` against an inline transcription of `cb_train::greedy_tensor_search_oblivious` / `leaf_index`: the GPU tree STRUCTURE (split `(feature, bin)` sequence + per-object `leaf_of`) matches the CPU EXACTLY (the SC-3 strict bar) and the leaf-value divergence is REPORTED.

## Deviations from Plan

**1. [Plan scope clarification — depth==1 MVP, depth>1 typed forward dependency]**
- **Found during:** Task 2 driver design.
- **Issue:** The plan's `single_tree` behavior text references a "fixed depth" tree. The only device split scorer that exists (`find_optimal_split_kernel`, Plan A) scores a SINGLE binary split (left/right) over the FROZEN 7.3 whole-dataset (`partCount==1`) histogram. A depth>1 oblivious level must score each candidate over the current `2^level` partitions (the per-partition / `fullPass=false` histogram, upstream `SubmitCompute(subsets,...)`), which the FROZEN whole-dataset fill cannot express — and that partition-aware fill is explicitly documented as a forward dependency in `launch_pointwise_hist2_handle` (RESEARCH A2).
- **Fix:** The MVP grows a depth-1 oblivious tree (a single split / stump) — the genuinely-complete vertical slice the existing kernels support EXACTLY with the strict O(1)-per-level read-back (level 0 has one partition, so the whole-dataset stump score IS the exact CPU level-0 score). `grow_oblivious_tree` is written as a general `for level in 0..depth` loop so the partition-aware path slots into the per-level score step unchanged; `depth > 1` surfaces a typed `CbError::OutOfRange` naming the partition-aware-histogram forward dependency rather than silently scoring a stump and mislabeling it a deep tree (which would be a wrong-structure fabrication). The `single_tree::depth_gt_one_is_tracked_forward_dependency` test pins this scope error so the cut is testable, not silent.
- **Rule:** Rule 2 (correctness — refusing a wrong-structure fabrication) + explicit tracked deferral.
