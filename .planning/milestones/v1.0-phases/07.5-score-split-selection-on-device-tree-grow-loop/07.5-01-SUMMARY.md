## Decisions Made
- **L2 score transcription (D-03 f64-finalize):** the kernel transcribes `avg = count>0 ? sum/(count+scaled_l2) : 0; score += avg*sum` verbatim from `cb-compute/src/score.rs::l2_split_score` + `leaf.rs::calc_average`, folded in the f64 device channel (f32 on wgpu only). The `count>0` guard prevents NaN/Inf on empty leaves (T-07.5-01-05).
- **`scaled_l2` as a length-1 `Array<F>`:** a `#[cube(launch)]` scalar argument must be a concrete `CubeElement`, not the generic `F` (the codebase passes float data via `Array<F>`); discovered as a CubeCL build error, resolved per the conditionals/comptime manual and the existing kernel idiom.
- **Host-side across-block argmin (RESEARCH Open Q2):** the per-block winner is reduced on-device; the (single-cube here) across-block argmin is finished host-side over the small O(blocks) descriptor array with the SAME lowest-index tie-break — well within "minimal round-trips", bulk histogram never leaves the device.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] Transcribed `select_best_candidate` inline instead of importing `cb-train`**
- **Found during:** Task 2 (running the score/argmin tests on rocm)
- **Issue:** The plan instructed `use cb_train::{select_best_candidate, Candidate}` as the read-only split-winner oracle. Adding `cb-train` as a dev-dependency pulls `cb-train`'s own `cb-backend = {path}` dependency (default features = `cpu`) into the test build graph. Cargo feature unification then activates `cb-backend/cpu` ALONGSIDE the requested `rocm`/`wgpu`/`cuda` feature, so `SelectedRuntime` resolves to the CpuRuntime (cpu wins the mutual-exclusion cfg chain in `lib.rs`). The CpuRuntime lacks `Atomic<f64>`/`Atomic<f32>` (cubecl-cpu 0.10.0 `not yet implemented: atomic<f64>`), so the FROZEN 7.3 histogram fill could not run at all and every score read back as zero (abs-div 0.125, rel 1.0).
- **Fix:** Reverted the `cb-train` dev-dep and the Cargo.lock entry; transcribed the tiny, frozen, well-documented strict-`>` first-wins / lowest-index algorithm inline in the test file as `reference_best_split`, citing `cb-train/src/tree.rs:291-302`. This is the SAME pattern `pointwise_hist.rs::host_reference_hist2` already uses to GENERALIZE a frozen `cb-compute` reduction in the test file without importing/modifying it. The cross-oracle still validates against the EXACT documented CPU semantics.
- **Files modified:** `crates/cb-backend/Cargo.toml` (dev-dep reverted), `crates/cb-backend/src/kernels/score_split.rs`, `Cargo.lock`
- **Verification:** After the revert, `cargo tree -p cb-backend --no-default-features --features rocm -e features` shows ONLY the `rocm` feature on `cb-backend` (no `cpu`); `SelectedRuntime = HipRuntime`; all 3 tests pass on gfx1100.
- **Committed in:** `c2aab5d` (Task 2 commit)

## Issues Encountered

**Pre-existing (out-of-scope) cubecl-cpu/wgpu atomic limitation.** The cubecl 0.10.0 toolchain in-env does not implement `Atomic<f64>` (cpu) or `Atomic<f32>` (wgpu host run) in its CPU compiler visitor, so the FROZEN 7.3 histogram fill — and therefore this score/split seam that consumes it — cannot RUN under the `cpu` or `wgpu`-host paths (the existing `kernels::pointwise_hist` tests fail identically; verified untouched by this plan via `git diff`). This affects the histogram FILL subsystem, not the score/split logic. The authoritative validation gate per D-7.5-07 / D-06 is the gfx1100 `rocm` oracle, where all 3 tests pass. `wgpu` and `cuda` BUILD cleanly (per the plan's compile-gated verification). Recorded as a deferred item for the histogram/toolchain track (not this plan's scope; FIX ATTEMPT LIMIT respected).
