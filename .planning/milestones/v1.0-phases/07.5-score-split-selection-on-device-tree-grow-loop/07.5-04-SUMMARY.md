# Phase 07.5 Plan 04: Score/Split Selection — Device-Resident Multi-Tree Plain-Boosting Pass Summary

The device-resident MULTI-TREE Plain-boosting pass (`grow_boosting_pass`) — the first complete multi-iteration GPU training slice: it loops the 07.5-03 host-light `grow_oblivious_tree` over a full Plain-boosting run, threading ONE `ComputeClient` through the whole run, scaling each tree's leaf values by `learning_rate`, updating the running per-object approx from the just-grown tree's already-read-back `leaf_of` + leaf values (reusing an existing D-05 crossing — no NEW bulk read-back), and recomputing the residual gradient `der1` DEVICE-SIDE between trees via the 7.2 der seam (`launch_der_binary_into` RMSE gradient = `target - approx`). Cross-oracled in `kernels/grow_loop.rs::multi_tree` against an inline transcription of the CPU `cb_train::boosting` Plain skeleton (NOT imported — the D-7.5-04 landmine): EVERY tree's STRUCTURE (split `(feature, bin)` + per-object `leaf_of`) matches the CPU multi-tree reference EXACTLY across the whole run (the SC-3 strict bar), the per-tree + aggregate leaf-value divergence is REPORTED, and an N-run stability check REPORTS the D-7.5-06 non-determinism budget (run-to-run structure stability + leaf-value drift).

## What this plan delivered

**Task 1 (`8578d7c`)** — `gpu_runtime.rs`:
- `GrownModel` — the multi-tree driver return: `trees: Vec<GrownTree>` (each with `learning_rate`-scaled leaf values) + the `learning_rate` scale + an `iterations()` accessor.
- `grow_boosting_pass` + private `grow_boosting_pass_into(&client, ...)` — the device-resident multi-tree driver. Loops `grow_oblivious_tree_into` over `iterations` on ONE threaded client; per tree: scales leaf values by `learning_rate`, updates the running approx (`approx[i] += leaf_value[leaf_of[i]]`) from the already-read-back routing, and recomputes `der1` device-side via the 7.2 seam for the next iteration. Typed guards: `LengthMismatch` on `weight`, `OutOfRange` on `2^depth` overflow, `Degenerate` on der read-back failure / truncated read-back (T-07.5-04-01/02/04). No production unwrap/panic/indexing.

**Task 2 (`be05b18`)** — `kernels/grow_loop.rs`:
- `multi_tree` module: `make_boosting_fixture` (target-based clear-gain-margin regression fixture), `cpu_boosting_pass` (the inline CPU multi-tree Plain-boosting reference — `der1 = target - approx`, depth-1 stump via the existing `cpu_best_stump`, `calc_average` leaf delta, `learning_rate`-scaled leaf values, running-approx update — transcribed, NOT imported).
- `matches_cpu_multi_tree_boosting` — per-tree split + `leaf_of` EXACT vs the CPU reference across the whole run (n in {1, 37, 1000}, 5 iterations); per-tree + aggregate leaf-value divergence REPORTED.
- `run_to_run_structure_stability_reported` — the D-7.5-06 non-determinism budget: re-runs the device pass 4× and REPORTS run-to-run structure stability + max leaf-value drift; asserts structure stays stable on the clear-margin fixture (any flip would be the REPORTED 7.6 epsilon input, not a sign-off here).

## Deviations from Plan

**1. [Rule 2 — correctness / explicit tracked deferral] Device der recompute read back per tree rather than threaded as a handle into the next tree's score path**
- **Found during:** Task 1 driver design.
- **Issue:** The plan calls for derivatives "recomputed device-side between trees via the 7.2 der seam (no host residual round-trip beyond the existing O(1)/leaf read-backs)." The 7.2 seam (`launch_der_binary_into`) produces the der as a DEVICE HANDLE, but the existing histogram/score path (`launch_pointwise_hist2_into` / `launch_find_optimal_split_pointwise_into`, 07.5-01/03) consumes a HOST `der1: &[f64]` slice — there is no handle-fed der->score path. A fully handle-threaded der->score loop would require a histogram-fill-from-der-handle seam, which is the SAME tracked forward-dependency shape as the depth>1 partition-aware histogram (07.5-03).
- **Fix:** The der recompute runs DEVICE-SIDE on the threaded client via `launch_der_binary_into` (RMSE gradient) and is read back ONCE per tree — the SAME crossing class as the existing per-tree `leaf_of` / part-stats read-backs (D-05: no bulk histogram/partition/doc-routing crosses, only the O(1)/leaf data). The running-approx update reuses the tree's already-read-back `leaf_of` + leaf values (no NEW bulk crossing). This is the faithful device-resident MVP within the existing host-slice seams; the handle-fed der->score path is surfaced as the same forward dependency as depth>1.
- **Files modified:** crates/cb-backend/src/gpu_runtime.rs
- **Commit:** 8578d7c

**2. [Plan scope clarification — RMSE-from-zero start, depth-1, consistent with 07.5-03]**
- **Found during:** Task 1/2.
- **Issue:** The plan does not pin the starting approx or the per-tree depth; both must match between the device pass and the CPU cross-oracle for the strict SC-3 bar.
- **Fix:** The MVP starts from an all-zero approx (RMSE-from-zero; `boost_from_average`/target-mean bias OUT of scope) so the initial residual is `target`, and loops DEPTH-1 trees (07.5-03 scope UNCHANGED — `grow_oblivious_tree_into` surfaces `CbError::OutOfRange` for depth>1). The CPU `cpu_boosting_pass` reference uses the SAME zero start + depth-1 so the structure comparison is apples-to-apples; depth>1 is NOT faked.
- **Files modified:** crates/cb-backend/src/gpu_runtime.rs, crates/cb-backend/src/kernels/grow_loop.rs
- **Commit:** 8578d7c, be05b18

## Forward dependencies surfaced (tracked, not silent)

- **Handle-fed der -> score path** — a fully device-resident boosting loop with NO per-tree der read-back requires a histogram-fill-from-der-HANDLE seam (the existing fill consumes a host der1 slice). Same shape as the depth>1 partition-aware-histogram forward dependency (07.5-03). The current pass recomputes der device-side and reads it back once per tree (the existing D-05 crossing class).
- **boost_from_average / target-mean bias** — the MVP starts from a zero approx; the per-loss optimum constant-approx start (`CalcOptimumConstApprox`, the model bias) is a follow-up.
- **Newton der2 leaf estimation** — RMSE der2 is the constant -1.0 (folded); the L2 score path uses Σ der1 / Σ weight. The per-partition-histogram Newton der2 score is the depth>1 / score-variant follow-up.

None. The der-handle-fed score path and depth>1 are typed, tested forward dependencies (the depth>1 path surfaces `CbError::OutOfRange`, exercised by the 07.5-03 `depth_gt_one_is_tracked_forward_dependency` test), not stubs returning empty/placeholder data.
