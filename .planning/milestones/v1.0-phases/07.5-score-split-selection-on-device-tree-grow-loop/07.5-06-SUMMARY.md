## RESEARCH Open Q3 — Cholesky-Solve Location Decision (REQUIRED)

**Decision: the small per-leaf dense SPD Cholesky solve + `CalculateScore` run HOST-SIDE over the BOUNDED assembled per-(feature,bucket) statistics; the bulk pairwise histogram + der-sum scatter + scan stay device-resident.**

Rationale (documented per the plan's `<output>` requirement):
1. **Transcribe, don't invent.** The FROZEN `cb_compute::pairwise_cholesky_solve` + `calculate_pairwise_score` are the parity oracle. A `#[cube]` dense linear-algebra solve would *re-derive* (not transcribe) that primitive — a fragile, oracle-breaking path. The host solve calls the FROZEN `calculate_pairwise_score` VERBATIM.
2. **Minimal round-trips (D-05).** The heavy device-resident work is the per-object der-sum scatter (`pairwise_make_derivatives_kernel`), the per-pair 4-channel pairwise histogram fill (FROZEN 7.4), and the 4-channel scan (`scan_update_pairwise_kernel`). Only the BOUNDED `O(n_features * n_bins)` der-sum descriptor crosses to host. The per-leaf system is tiny (`2*leaf_count == 2` at the depth-1 root), so the solve is O(1)-class.
3. **The FROZEN 7.4 4-channel handle IS consumed device-resident** in `launch_pairwise_split_score_into` (the `launch_pairwise_hist_into` fill runs over the same pairs, threading `bits`/`one_hot`, before the score). Its correctness is independently proven by `scan_matches_reference`.

This is exactly the "bounded host solve over the small per-leaf system is acceptable" path the plan's `<critical_project_rules>` sanctions.

## Tracked Follow-up (EXPLICIT — not a silent cut)

- **Pairwise cross-cube scan carry for `n_bins > CUBE_DIM` (8-bit / 256-bin features).** The `scan_update_pairwise_kernel` is single-cube (`n_bins <= CUBE_DIM`), inheriting Plan B's scope; `launch_scan_update_pairwise` surfaces a typed `CbError::OutOfRange` for `n_bins > CUBE_DIM` (verified by `scan_matches_reference`'s `n_bins=64` arm). The cross-cube carry is the SAME 7.2/7.3/7.4 forward dependency `block_scan_kernel` / the pointwise scan defer. Not reachable by the 5/6/7-bit FROZEN 7.4 FILL family used here.
- **Partition-aware (leaf_count > 1 / depth > 1) pairwise der-sum + pair-weight assembly.** The depth-1 MVP scores the single root leaf; a deep pairwise tree needs the per-partition assembly (the SAME class as the pointwise grow loop's depth>1 guard). Surfaced as a typed `CbError::OutOfRange` (`depth_gt_one_is_tracked_forward_dependency`).
