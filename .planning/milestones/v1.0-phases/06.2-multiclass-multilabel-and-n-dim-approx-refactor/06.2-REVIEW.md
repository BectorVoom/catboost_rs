# Phase 06.2: Code Review Report

**Reviewed:** 2026-06-16
**Depth:** standard
**Files Reviewed:** 16
**Status:** issues_found

## Summary

Reviewed the N-dim approx refactor plus the multiclass / multilabel / MultiQuantile
loss machinery for phase 6.2. The dimension-major buffer indexing (`buf[d*n+i]`),
the softmax max-subtraction, the hand-rolled symmetric-Hessian Cholesky solve, the
class-index remap bounds, the MultiQuantile alpha/delta validation, and the
serialization leaf-major↔dim-major transposes were all examined against the
CLAUDE.md parity and safety rules.

The core elementwise der math, validation, and the dim=1 byte-identity discipline
are solid and well-guarded (checked `.get`, no raw indexing, no `unwrap` in
production, sums routed through `cb_core::sum_f64`). However two genuine
correctness defects surfaced at the **boundaries** between the new dim-major
training buffers and pre-6.2 scalar-only subsystems that were NOT updated for
`approx_dimension > 1`:

1. The public **apply / predict path (`predict_raw`) collapses every multi-output
   model to dimension 0** — it has no notion of `approx_dimension`, so multiclass /
   multilabel / MultiQuantile predictions are silently wrong on the published
   surface (CR-01).
2. The **bootstrap / sampling and `random_strength` std-dev are fed the dim-major
   derivative buffer**, so any sampler other than `No` (and any non-zero
   `random_strength`) draws `dim*n` times and masks per-(dim,object) instead of
   per-object — corrupting both the sampling semantics and the RNG phase (CR-02).

Both are latent for the in-scope committed fixtures (which pin `bootstrap_type=No`,
`random_strength=0`), but they are real defects that ship in the public API and
would corrupt the moment a user enables sampling on a multi-output model or calls
`predict` on one.

Several lower-severity issues (a doc/config contradiction in the oracle generator,
unused/over-broad fallbacks, silent error swallowing in MultiQuantile launch) are
recorded below.

## Critical Issues

### CR-01: `predict_raw` ignores `approx_dimension` — multi-output models predict only dimension 0

**File:** `crates/cb-model/src/apply.rs:200-228` (and `:244-284`)
**Issue:** `predict_raw_one` / `predict_raw` produce exactly ONE value per object:
`model.bias + sum_f64(leaf_contributions)`, where each tree contributes
`tree.leaf_values.get(leaf)` with `leaf ∈ [0, n_leaves)`. For a phase-6.2 multi-
output model, `tree.leaf_values` is the DIMENSION-MAJOR buffer of length
`approx_dimension * n_leaves`, so `tree.leaf_values.get(leaf)` only ever reads
**dimension 0** (`leaf_values[0*n_leaves + leaf]`); dimensions `1..approx_dimension`
are never read, and the bias is added as a scalar. There is no multi-dimensional
apply function anywhere in `cb-model` (`grep` for `approx_dimension` in `apply.rs`
returns nothing). Consequently `apply_multiclass_prediction` in `predict.rs` — which
expects a correct dim-major `approx[d*n+i]` — has **no producer** on the public
surface: every `Model::predict` / `predict_raw` call on a MultiClass /
MultiClassOneVsAll / MultiLogloss / MultiCrossEntropy / MultiQuantile model returns
silently-wrong, single-dimension scores. This defeats the entire phase for the
published prediction path.
**Fix:** Add a dimension-aware apply that emits the dim-major raw approx, e.g.:
```rust
pub fn predict_raw_multi(model: &Model, feature_values: &[Vec<f32>]) -> Vec<f64> {
    let dim = model.approx_dimension.max(1);
    let n = feature_values.first().map_or(0, Vec::len);
    let mut out = vec![model.bias; dim * n]; // per-dim bias when plumbed
    for tree in &model.oblivious_trees {
        let n_leaves = tree.leaf_values.len() / dim;
        for obj in 0..n {
            let leaf = leaf_index_for(tree, feature_values, obj);
            for d in 0..dim {
                if let (Some(a), Some(&lv)) =
                    (out.get_mut(d * n + obj), tree.leaf_values.get(d * n_leaves + leaf))
                {
                    *a += lv;
                }
            }
        }
    }
    out
}
```
and route `predict_raw` for `approx_dimension == 1` (byte-identical) while the
multi-output path returns the dim-major buffer that `apply_multiclass_prediction`
consumes. Until this exists, no in-scope test may claim the multi-output
Predictions stage passes through the production apply path.

### CR-02: bootstrap, MVS lambda and `random_strength` std-dev consume the dim-major derivative buffer (wrong `n`)

**File:** `crates/cb-train/src/boosting.rs:1981-2010, 2027-2034`
**Issue:** `weighted_der1` is length `approx_dimension * n` (dim-major). It is passed
unmodified to:
- `bootstrap(params.bootstrap_type, &weighted_der1, …)` (`:1981`), which sets
  `n = derivatives.len()` (`bootstrap.rs:391`). For Bernoulli, `set_sampled_control`
  draws `dim*n` uniforms (one per dim-major slot) instead of `n`, and the resulting
  `control` mask is indexed per-(dim,object), not per-object — so the score-fold mask
  applied in `score_weighted_der1`/`score_weights` (`:1999-2010`) drops fractional
  dimensions of objects and advances the persistent RNG by `dim*n` draws, breaking
  the documented per-object sampling and the RNG-phase parity contract. For MVS,
  `mvs_lambda(&weighted_der1, …)` and `mvs_sample_weights` likewise compute
  mean-grad / thresholds over `dim*n` elements.
- `score_st_dev(params.random_strength, &weighted_der1, model_length)` (`:2030`),
  whose `derivatives_std_dev_from_zero` divides `sum(wd²)` by
  `weighted_der1.len() == dim*n` (`score.rs:166-172`), not `n` — so `scoreStDev` is
  scaled by `1/sqrt(dim*n)` instead of `1/sqrt(n)`, a parity break for any non-zero
  `random_strength` on a multi-output model.

All in-scope multiclass/multilabel/MultiQuantile fixtures pin `bootstrap_type=No`
and `random_strength=0`, so this is currently masked — but the code path is reachable
through the public `BoostParams` and corrupts the moment either knob is enabled on a
multi-output loss. Upstream samples / measures std-dev per OBJECT, sharing the
sample weight across dimensions (the same per-object weight the leaf path already
shares at `:1965`).
**Fix:** Reduce the derivative buffer to a per-object vector before sampling /
std-dev, or pass `n` explicitly. E.g. derive a per-object representative
(`weighted_der1[0*n .. n]` for the std-dev/sample path is NOT correct either — upstream
uses the full multi-target gradient norm), so compute the per-object aggregate
upstream uses (`CalcDerivativesStDevFromZero` over the multi-dim gradient) and feed
`bootstrap` a length-`n` per-object derivative. Gate the bootstrap/std-dev call so
`derivatives.len()` is always `n`, not `dim*n`.

## Warnings

### WR-01: `multi_dim_candidate_score` silently substitutes the whole buffer on a bad stride

**File:** `crates/cb-train/src/tree.rs:409-415`
**Issue:** `der1.get(base..base + n_objects).unwrap_or(der1)` falls back to the ENTIRE
`der1` buffer when the slice is out of range (e.g. `der1.len()` not a multiple of
`n_objects`). That silently feeds a wrong-length, wrong-dimension slice into
`reduce_leaf_stats`, producing a plausible-but-wrong score instead of surfacing the
shape error. The same `.unwrap_or(&[])` / `.unwrap_or(der1)` pattern recurs in
`cpu_runtime.rs:708` (`approx.get(d*n..d*n+n).unwrap_or(&[])`).
**Fix:** The caller already validates the shape upstream; assert it here with a
checked path that returns an empty slice (scoring 0) or, preferably, thread a
`CbResult` so a stride mismatch is a typed `CbError` rather than a silent fallback to
a different dimension's data.

### WR-02: MultiQuantile per-dimension launch failure is silently swallowed to a zero gradient

**File:** `crates/cb-backend/src/cpu_runtime.rs:714-717`
**Issue:** `compute_multiquantile_gradients` maps a per-dimension
`launch_quantile_f64(...)` `Err(_)` to `der1.extend(std::iter::repeat(0.0).take(n))`.
This is exactly the failure mode WR-05 (the other launchers) was fixed to AVOID: a
device read-back failure masquerades as a valid all-zero derivative for that
dimension, silently producing a degenerate (no-gradient) tree for that quantile
instead of surfacing the backend error. `compute_gradients` already returns
`CbResult`, so the error channel exists.
**Fix:** Propagate the error:
```rust
let der1_d = launch_quantile_f64(approx_d, target, alpha_d, delta)?;
der1.extend_from_slice(&der1_d);
```
and change `compute_multiquantile_gradients` to return `CbResult<(Vec<f64>, Vec<f64>)>`
(the surrounding dispatch in `compute_gradients` is already `?`-friendly).

### WR-03: oracle generator config `target_rule` contradicts the actual multilabel target code

**File:** `crates/cb-oracle/generator/gen_fixtures.py:2816-2821` vs `:2899-2902`
**Issue:** `_multilabel_target()` thresholds all three labels on `y`:
`l0=(y>q[0]), l1=(y>q[1]), l2=(y>q[2])` with `q = quantile(y, [0.25, 0.5, 0.75])`
(matching the docstring "three NESTED binary thresholds on `y`"). But the committed
`config.json` `target_rule` string states
`"label0=(y>median(y)), label1=(X0>median(X0)), label2=(X1>median(X1))"` — a
completely different rule (per-feature thresholds). Anyone regenerating or porting
the fixture from the recorded `target_rule` will produce a DIFFERENT label matrix
than the one the committed `model.json` was trained on, breaking the ≤1e-5 oracle
lock. Since the Rust comparison harness for these losses is not yet present, this
latent contradiction is undetected.
**Fix:** Replace the `target_rule` string with the rule the code actually uses:
`"label_k = (y > quantile(y, [0.25,0.5,0.75])[k]); 3 nested binary {0,1} columns"`.

### WR-04: multiclass/multilabel bootstrap & sampling lengths assume per-object — no shape guard

**File:** `crates/cb-train/src/boosting.rs:1999-2010`
**Issue:** `score_weights` zips per-object `weights` (len `n`) with `sample_weights`
and `control` (len `dim*n` after CR-02). The `zip` silently truncates to `n`, so
`score_weights` happens to come out length `n` for `bootstrap_type=No` (all entries
identical), masking CR-02 for that one case. This implicit reliance on zip-truncation
is fragile: the moment `sample_weights` length changes meaning (CR-02 fix), the
truncation hides the mismatch instead of flagging it.
**Fix:** After fixing CR-02 so `sample_weights`/`control` are length `n`, add a
`debug_assert_eq!(sample_weights.len(), n)` (or a typed length check) so a future
dimension-major regression is caught rather than silently truncated.

### WR-05: `softmax_ders` does not validate `target_class < k`; OOB silently produces an all-`-p` gradient

**File:** `crates/cb-compute/src/loss.rs:478-499`; producer `crates/cb-backend/src/cpu_runtime.rs:643`
**Issue:** `softmax_ders` documents (and implements) that an out-of-range
`target_class` leaves `der1` at `-p[d]` for every dimension (no `+1`), relying on the
caller to range-check. The caller `compute_softmax_gradients` derives
`target_class = target.get(i).copied().unwrap_or(0.0) as usize` (`cpu_runtime.rs:643`)
directly from the float target with NO bounds check at the call site. The boosting
loop DOES remap the target to `[0,k)` before this (`boosting.rs:1561-1573`), so the
normal path is safe — but a negative `f64` target casts to `0` (saturating) and a
target ≥ k silently yields the no-`+1` degenerate gradient rather than an error. The
T-6.2-01 range guard is asserted in comments but not enforced at this boundary.
**Fix:** Validate `target_class < k` in `compute_softmax_gradients` /
`compute_onevsall_gradients` and return `CbError::OutOfRange` on violation (they are
already inside the `CbResult`-returning `compute_gradients`), so a mismatched
(target, remap) pair fails loudly instead of training a degenerate softmax.

### WR-06: `build_class_remap` mixes a total-order sort with an `abs()==0.0` dedup (NaN / -0.0 hazard)

**File:** `crates/cb-train/src/boosting.rs:730-735`
**Issue:** `build_class_remap` sorts with `partial_cmp(...).unwrap_or(Equal)` then
dedups with `dedup_by(|a, b| (*a - *b).abs() == 0.0)`. `partial_cmp` returning
`None` on a NaN label collapses to `Equal`, giving an unstable sort order (the same
hazard the Exact-leaf code explicitly fixed with `total_cmp` at `leaf.rs:360-368`),
and `(a-b).abs()==0.0` treats `+0.0` and `-0.0` as equal (usually desired) but will
also silently merge two labels that differ only past f64 round-off if a caller passes
near-duplicate float labels. The companion `remap_target_to_class` uses
`(l - t).abs() == 0.0` for the inverse lookup, so a label that sorted/deduped under
one comparison but fails the position search yields `CbError::OutOfRange`. For the
in-scope integer-tercile labels this is safe, but the inconsistency is a latent
parity/robustness gap.
**Fix:** Use `f64::total_cmp` for the sort (matching the leaf.rs precedent) and dedup
on `total_cmp(...) == Equal` (or bit-equality `a.to_bits() == b.to_bits()`), and use
the SAME equality predicate in `remap_target_to_class` so sort, dedup, and lookup are
consistent.

## Info

### IN-01: `multi_dim_split_score` Cosine denominator can be fed a negative `avg`-coupled numerator without guard

**File:** `crates/cb-compute/src/score.rs:110-145`
**Issue:** Mirrors the scalar `cosine_split_score`: `numerator / denominator.sqrt()`
where `denominator` is seeded `1e-100`. The seed prevents div-by-zero, and the math
faithfully transcribes upstream, so this is not a bug — but a comment noting that a
negative numerator (legitimate for a loss-reducing split) yields a negative score
that the strict `>` first-wins still ranks correctly would aid the next reader.
**Fix:** Add a one-line note that the score may be negative by design and the seed is
the only div-by-zero guard.

### IN-02: dead/garbled doc comment fragment in `loss_approx_dimension` header

**File:** `crates/cb-train/src/boosting.rs:687-699`
**Issue:** The doc block above `loss_approx_dimension` opens with the tail of
`starting_approx`'s doc ("Compute the starting approx … folded through … `sum_f64`")
immediately followed by the `loss_approx_dimension` description, with no separating
item — the `starting_approx` fn is actually defined later at `:764`. The two doc
comments were spliced during an edit; the result is a misattributed comment.
**Fix:** Move the "Compute the starting approx …" sentences back onto
`starting_approx` (`:764`) and leave only the `approx_dimension` description here.

### IN-03: `_multilabel_target` docstring also drifts from `config["target_rule"]`

**File:** `crates/cb-oracle/generator/gen_fixtures.py:2806-2821`
**Issue:** Related to WR-03: the docstring correctly says "three NESTED binary
thresholds on `y` at its 0.25 / 0.5 / 0.75 quantiles," but `label_count`/`target_rule`
in the emitted config disagree. Aligning all three (docstring, code, config string)
removes the ambiguity entirely.
**Fix:** Single source of truth — derive the `target_rule` config string from the
same quantile expression the code evaluates.

### IN-04: redundant per-dimension `der2 = 0` allocation comment vs. constant-0 reuse

**File:** `crates/cb-backend/src/cpu_runtime.rs:705-707`
**Issue:** `compute_multiquantile_gradients` allocates `der2 = vec![0.0; dim*n]` and
the per-dimension Mae/Quantile arms each allocate their own `vec![0.0; n]` der2
(`:490, :499, :545`). These constant-zero hessian allocations are correct but
repeated; a shared `zero_hessian(len)` helper (or documenting that the constant-0
vec is intentional, the "Mae precedent") would reduce duplication. Not a correctness
issue.
**Fix:** Optional: extract a `const-0 der2` helper, or leave as-is with the existing
"Mae precedent" note.
