<objective>
Wave 0, Part 1 of 2 (the pure mechanical N-dim refactor тАФ NO new loss math). Widen the `cb-compute` compute seam so the per-object approx is a **dimension-major flat buffer** (`approx[d*n_objects+i]`, D-6.2-01) and `Runtime::compute_gradients` carries an `approx_dimension`. At `approx_dimension=1` the dimension-major buffer reduces bit-identically to today's scalar `Vec<f64>` тАФ this equivalence is what makes the D-04 hard checkpoint (re-locked in Plan 06.2-02) provable.

This plan does NOT add any multiclass/multilabel/MultiQuantile math. It only:
1. threads `approx_dimension` through `Runtime::compute_gradients` and the `CpuBackend` impl,
2. drops the `Copy` derive on `Loss` (the cross-crate ripple that the Wave-3 `MultiQuantile { alpha: Vec<f64> }` variant forces тАФ surface it now, before any new variant, so it is a one-time refactor), and
3. keeps the existing scalar der1/der2 helpers in `loss.rs` unchanged, called per-dimension over dim-major slices.

Purpose: establish the N-dim compute boundary with zero behavior change at dim=1, isolating refactor risk from new-loss risk (umbrella D-03/D-04).
Output: a non-Copy `Loss`, an `approx_dimension`-carrying `compute_gradients`, and a per-dimension `CpuBackend` dispatch тАФ all green against the existing `cb-compute`/`cb-backend` unit + loss-oracle tests at dim=1.

NOTE: gsd-tools CLI is ABSENT on this machine (verified). STATE.md / ROADMAP.md / REQUIREMENTS.md updates are MANUAL, as in every prior phase since 02.
</objective>
