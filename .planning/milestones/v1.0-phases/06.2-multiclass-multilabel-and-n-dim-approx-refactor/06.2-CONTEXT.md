## Phase Boundary

Two sequenced deliverables, on the **single shared CPU train loop**, each oracle-locked ≤1e-5 vs upstream **catboost 1.2.10** (`thread_count=1`) per-stage (splits / leaves / staged approx + final prediction):

1. **N-dim approx refactor (D-03):** generalize the entire train loop so the per-object approx is **always a vector** (matching upstream `TVector<TVector<double>>`). Scalar losses become the **dim=1 degenerate case** on the *same* code path — no parallel scalar/multi-dim duplication. Everything scalar today (`cb-compute::loss` der1/der2 take `approx: f64`/`&[f64]`; `cb-train` bootstrap / leaf-estimation / `tree` / SHAP assume one dimension) is widened.

2. **Multiclass / multilabel + MultiQuantile losses (LOSS-02 + LOSS-03 tail):** on the stable N-dim foundation, implement **MultiClass** (softmax), **MultiClassOneVsAll**, **MultiLogloss**, **MultiCrossEntropy**, and **MultiQuantile** (the multi-output member of LOSS-03 relocated from 6.1) — each passing its own per-stage oracle ≤1e-5.

**HARD CHECKPOINT (D-04):** the pure mechanical refactor re-runs **ALL ~40 existing scalar oracles green at dim=1 BEFORE any multiclass math is written.** This is a non-negotiable gate that isolates refactor risk from new-loss risk.

**Requirements:** LOSS-02 (full); LOSS-03 (MultiQuantile only — the multi-output member; scalar matrix already shipped in 6.1).

**NOT in this phase** (later sub-phases / deferred per umbrella D-06): ranking losses/metrics (6.3), score functions / uncertainty / custom objectives (6.4), text/embedding features (6.5), advanced fstr / non-symmetric trees / monotone constraints (6.6), GPU backends (Phase 7), any Python bindings (Phase 8). Any loss/metric NOT named above → deferred-to-v2, not silently in-scope.

## Implementation Decisions

All decisions below were auto-selected in `--auto` mode to the recommended option. They are dominated by two non-negotiable constraints already locked upstream of this discussion — the umbrella **D-03** (single N-dim code path) and **D-04** (no-behavior-change checkpoint) — plus the project's memory-efficiency-first goal and the ≤1e-5 parity bar. None of these are open design questions; they are the parity-/efficiency-dictated reading of a fixed scope.

### N-Dim Approx Memory Layout
- **D-6.2-01: Dimension-major flat buffer.** The per-object approx is one contiguous `Vec<f64>` indexed `approx[d * n_objects + i]` (outer = approx dimension, inner = object), mirroring upstream's `TVector<TVector<double>>` where the outer vector is the dimension. Single allocation, dimension-contiguous so per-dimension reductions stay cache-friendly and route through `cb-core::sum_f64` in the same order upstream accumulates. **dim=1 is bit-identically today's `Vec<f64>` per-object buffer** — that equivalence is what makes the D-04 checkpoint provable. Rejected: `Vec<Vec<f64>>` (per-dimension allocations, fragmented, extra indirection — violates memory-efficiency-first); object-major flat (`approx[i * dim + d]`) (diverges from upstream reduction grouping, complicates per-dimension leaf reductions).

### Refactor Wave Structure & the D-04 Gate
- **D-6.2-02: Wave 0 is a pure mechanical refactor with a hard re-lock gate, before any new loss math.** Sequencing (planner finalizes exact plan-to-wave mapping):
  - **Wave 0 — N-dim refactor (NO new math):** widen approx → length-`dim` vector everywhere (loss der1/der2 signatures, `Runtime::compute_gradients`, bootstrap, leaf estimation, `tree`, SHAP/apply, model bias/leaf-values storage). **Gate: ALL existing scalar oracles re-run green at dim=1.** Multiclass code does not start until this is green. (D-04 hard checkpoint.)
  - **Wave 1 — multiclass:** MultiClass (softmax), MultiClassOneVsAll.
  - **Wave 2 — multilabel:** MultiLogloss, MultiCrossEntropy.
  - **Wave 3 — MultiQuantile** (LOSS-03 multi-output member).
  - Each loss wave is its own plan + its own ≤1e-5 per-stage oracle gate → small blast radius, unambiguous regression attribution (consistent with 6.1 D-6.1-02). Rejected: refactor-and-multiclass-in-one-wave (defeats D-04 risk isolation); per-loss plans (excess ceremony).

### Gradient / Runtime API Generalization
- **D-6.2-03: Generalize the single coarse op; one code path; reuse scalar der per-dimension where the loss is separable.** `Runtime::compute_gradients` carries an `approx_dimension` and operates on the dimension-major buffer; scalar losses run with `dim=1` through the identical path (D-03 — no parallel pipeline). **Separable** multi-dim losses (MultiClassOneVsAll, MultiLogloss, MultiCrossEntropy, MultiQuantile — each output dimension independent) **reuse the existing scalar elementwise der1/der2 helpers in `loss.rs` per-dimension.** Only **MultiClass softmax** (dimensions coupled through the softmax normalizer) gets a dedicated cross-dimension der/Hessian path. Rejected: a separate multi-dim Runtime/trait (duplication, divergence risk).

### approx_dimension Derivation & Leaf Estimation
- **D-6.2-04: dimension source per loss; leaf math transcribed from upstream, not invented.**
  - MultiClass / MultiClassOneVsAll: `approx_dimension` = **distinct class count** derived from the integer class labels.
  - MultiLogloss / MultiCrossEntropy: `approx_dimension` = **label-set width** (number of binary label columns).
  - MultiQuantile: `approx_dimension` = **number of quantiles** (length of the `alpha` vector).
  - Leaf-estimation method per dimension follows the **upstream `leaf_estimation_method` default** for each loss (transcribe, confirm during research). The exact N-dim dimensionality model, the softmax per-leaf Newton Hessian, and the class-count plumbing into model metadata are **parity-dictated → Claude's discretion / research reads `approx_calcer.cpp` and reproduces.**

### MultiQuantile Representation
- **D-6.2-05: MultiQuantile = K independent Quantile dimensions.** Reuses the scalar `Quantile` der + the **Exact weighted-α-quantile** leaf path (generalized in 6.1, D-6.1-05) per dimension; the per-quantile `alpha` values are a `Vec<f64>` parameter on the loss variant (extends the established `Loss::Variant { params }` enum pattern, D-6.1-03). No softmax coupling — each quantile output is independent. Rejected: a bespoke multi-output Quantile path decoupled from the scalar Quantile der (throwaway divergence).

### Oracle Floor
- **D-6.2-06: Python-reachable per-stage parity (D-07 default floor); NO C++ instrumentation.** Multiclass/multilabel/MultiQuantile leaf math is deterministic and Python-reachable, so — like 6.1 — they ride the transcribe-then-self-oracle + per-stage Python-reachable oracle (borders/splits/leaf-values/staged N-dim approx + final predictions, all ≤1e-5 vs upstream catboost 1.2.10, `thread_count=1`). The proactive C++-instrumentation budget (umbrella D-07) is reserved for 6.3 randomized ranking losses, not spent here. Disk-pressure constraint (D-08) does not bite this phase.

<specifics>
## Specific Ideas

- **Bit-identical dim=1 equivalence is the design hinge.** The dimension-major flat buffer must reduce to exactly today's scalar `Vec<f64>` at dim=1 so the D-04 checkpoint (all ~40 scalar oracles green) is a true no-behavior-change proof, not an approximate one. Any layout choice that perturbs reduction order at dim=1 is wrong.
- **Softmax is the only coupled loss** — isolate its cross-dimension der/Hessian; everything else (OneVsAll, MultiLogloss, MultiCrossEntropy, MultiQuantile) is per-dimension-independent and reuses scalar der. This keeps the new-math surface minimal.

</specifics>
