# Phase 06.3 Plan 06: ranking_der.rs gap-closure (CR-01 NDCG scale + WR-02 D-08) Summary

Closed the two `ranking_der.rs` gaps from 06.3-VERIFICATION.md / 06.3-REVIEW.md: the **CR-01 BLOCKER** (StochasticRank NDCG gradient scale mismatch — `calc_dcg_metric_diff` now reads `old_weight`/`new_weight` from the normalized `pos_weights` vector, mirroring upstream `posWeights[oldPos]/posWeights[newPos]`) and the **WR-02 WARNING** (`lambdamart_ideal_ndcg` D-08 raw-accumulation — now routes through `cb_core::sum_f64`). Both edits are in the single co-owned file; all 122 cb-compute lib tests, the lambdamart + stochasticrank oracles, and the full cb-train suite stay green with no oracle regression.

## What Was Built

### Task 1 — CR-01: normalized pos_weights into calc_dcg_metric_diff (commit 6cbe2eb)

- Added a `pos_weights: &[f64]` parameter to `calc_dcg_metric_diff` (after `order`, before `cum_sum`), matching the 06.3-REVIEW.md CR-01 signature.
- Replaced the two raw recomputations `let old_weight = 1.0 / ndcg_denominator(old_pos);` / `let new_weight = 1.0 / ndcg_denominator(new_pos);` with bounds-checked reads `pos_weights.get(old_pos).copied().unwrap_or(0.0)` / `...(new_pos)...` — the SAME normalized vector that built `cum_sum`/`cum_sum_up`/`cum_sum_low`. This puts `doc_diff` and `mid_diff` on one consistent `1/ideal_dcg` scale (upstream `error_functions.cpp:1233-1234`).
- Deleted the misleading multi-line "OFFLINE-closure precision note" comment block; rewrote the doc-comment to cite the normalized-vector read.
- Updated the SOLE call site in `stochastic_rank_group_der` to pass `&pos_weights` (already in scope, bound at the cum_sum construction site).
- Added the RED-first regression test `calc_dcg_metric_diff_reads_normalized_pos_weights_ndcg` (graded relevance `[3,2,1]`, `ideal_dcg` clearly != 1.0, non-perfect order; asserts the normalized result AND that it differs from the pre-fix raw-denominator value) plus the `calc_dcg_metric_diff_dcg_arm_unchanged` guard.

### Task 2 — WR-02: lambdamart_ideal_ndcg through cb_core::sum_f64 (commit c7d4baa)

- Replaced the `let mut score = 0.0; for ... { score += ndcg_numerator(t)/ndcg_denominator(pos); }` loop with a collected `terms: Vec<f64>` reduced by `cb_core::sum_f64(&terms)`. The descending stable sort is unchanged.
- Rewrote the doc-comment: dropped the "parity contract is the loop order" exception language, cited D-08 routing through `sum_f64` (kept the `error_functions.cpp:925-935` citation).
- Added `lambdamart_ideal_ndcg_equals_sum_f64_of_terms` (asserts equality to the explicit per-position-terms `sum_f64`) and `lambdamart_ideal_ndcg_trivial_windows` (empty -> 0.0, top-window-0 -> 0.0, single element -> sole term).
