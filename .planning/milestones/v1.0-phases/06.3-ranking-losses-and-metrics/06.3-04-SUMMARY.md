# Phase 06.3 Plan 04: YetiRank / YetiRankPairwise / StochasticRank Summary

**One-liner:** The two randomized RNG-stream ranking losses (YetiRank's 2-level
`TFastRng64` seed + Gumbel-noise pair sampling, StochasticRank's `std_normal`
Monte-Carlo DCG gradient) land der+sampler+variants+boosting-wiring; the RNG draw
order — the parity crux — is validated bit-for-bit against TWO standalone OFFLINE
instrumented generators that self-oracle against the bitstream-locked
`cb-core::TFastRng64`, with the end-to-end trainer fixture cleanly DEFERRED
(escalate-don't-weaken) on an infeasible instrumented-trainer build.

## What shipped

- **`Loss::YetiRank{permutations,decay}` / `Loss::YetiRankPairwise{permutations,
  decay}` / `Loss::StochasticRank{metric,sigma,mu,num_estimations}`** +
  `StochasticRankMetric{Dcg,Ndcg}` — doc-cited RNG draw order, range validation
  (`permutations>=1`, `decay∈[0,1]`, `sigma>0`, `mu>=0`, `num_estimations>=1` →
  `CbError::OutOfRange`), upstream defaults (`permutations=10`, `decay=0.85`,
  `sigma=1`, `mu=0`, `num_estimations=1`, `nu=0.01`, `lambda=1`).

- **`yetirank.rs` sampler** — transcribes `GenerateYetiRankPairsForQuery` +
  `CalcWeightsClassic` + `AddNoise` Gumbel + the 2-level seed derivation verbatim
  (`yetirank_helpers.cpp:146-393`, `restorable_rng.cpp:3-9`), draw-order-faithful
  to `cb-core::TFastRng64::gen_rand_real1`. Sampled competitors ride the existing
  PairLogit der.

- **StochasticRank DCG/NDCG Monte-Carlo der** (`ranking_der.rs`) — Stage-1
  shift/center, Stage-2 `std_normal` noise + score sort + DCG cumulative
  statistics + metric-diff/density gradient, Stage-3 SFA orthogonalization
  (`error_functions.cpp:1008-1256`). `der2=0` (Gradient leaf); all reductions via
  `sum_f64`; per-group reseed `random_seed + group_index`.

- **Boosting wiring** — YetiRank/YetiRankPairwise RE-SAMPLE competitors each
  iteration from the current approx (fixed 2-level-derived seeds) injected into
  the grouped der; YetiRankPairwise routes to the Cholesky pairwise leaf
  (`is_pairwise_scoring`/`is_plain_only` extended, Plain enforced).

- **Two OFFLINE instrumented generators** (`yetirank_oracle.cpp` /
  `stochasticrank_oracle.cpp`) — zero catboost includes, compile clean (g++ C++20
  `-Wall`), and **self-oracle bit-for-bit** against `cb-core::TFastRng64` /
  `std_normal` (validated: `block_seed=12283622132691337806`,
  `std_normal(0)=[0.6337,-0.5284,-0.4408]`, identical to Rust). RNG-draw ground
  truth frozen as committed fixtures.

- **Three per-stage oracles** gating the RNG draw log ≤1e-5: YetiRank/
  YetiRankPairwise sampled competitor weights (`0.192/0.098250/0.083250`) + the
  2-level query seed (`446232438382340955`, parsed as u64) + StochasticRank
  Gaussian noise stream — all reproduced bit-for-bit by the Rust sampler/der. NO
  `#[ignore]`, NO weakened tolerance.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Corrected the transcribed C++ Lcg32 to match rng.rs exactly**
- **Found during:** Task 3 (the generator self-oracle)
- **Issue:** The initial C++ `transcribed::Lcg32` used pcg_basic-style double-step
  seeding + mix-old-state + a 16-bit FixSeq mask + reversed 64-bit combine, which
  DIVERGED from the oracle-locked Rust port (`block_seed` and `std_normal` draws
  differed). The self-oracle CAUGHT it — exactly its purpose.
- **Fix:** Re-transcribed to match `crates/cb-core/src/rng.rs` verbatim
  (`X = seed` directly + Iterate-then-Mix, 31-bit FixSeq mask, `low|high<<32`).
  Self-oracle then agreed bit-for-bit.
- **Files modified:** yetirank_oracle.cpp, stochasticrank_oracle.cpp
- **Commit:** 6c59fde

**2. [Rule 1 - Bug] u64 query seed parsed as u64, not through f64**
- **Found during:** Task 4 (yetirank oracle)
- **Issue:** Parsing the 64-bit query seed through `f64 as u64` truncated it
  (`...955` → `...928`, the >53-bit mantissa loss), failing the seed match.
- **Fix:** Parse the seed field as `u64` directly.
- **Files modified:** yetirank_oracle_test.rs
- **Commit:** 3890089

**Note:** the plan's `files_modified` listed `crates/cb-compute/src/loss.rs` for
the Loss variants; the `Loss` enum actually lives in `runtime.rs` (the der
PRIMITIVES are in `loss.rs`). The variants + validate landed in `runtime.rs`; the
StochasticRank der landed in `ranking_der.rs` (the grouped seam) — both
consistent with the existing ranking-loss layering. No behavior change.
