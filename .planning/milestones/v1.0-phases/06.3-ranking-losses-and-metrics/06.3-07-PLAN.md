<objective>
Close the **CR-02 BLOCKER** from 06.3-VERIFICATION.md / 06.3-REVIEW.md: `boosting.rs` applies
`weighted_der1 = der1 * weight` unconditionally for ALL losses, including QueryRMSE/QuerySoftMax,
whose grouped derivative functions already fold the per-object weight into `der1` (and `der2`).
At the uniform-weight (w=1.0) oracle fixtures the double multiplication is invisible; any weighted
training scenario produces squared-weight gradients, wrong tree structures, and wrong leaf values.

Fix per 06.3-REVIEW.md CR-02: branch the `weighted_der1` computation on `group_spans.is_some()`. For
the grouped (ranking) path use `ders.der1` as-is (already weight-folded); for the pointwise path keep
the existing per-object `der1 * weight` map. Add a regression test with non-uniform per-object weights
that the existing w=1.0 fixtures mask.

Purpose: Restore QueryRMSE/QuerySoftMax gradient correctness for non-uniform per-object weights.
Output: A branched `weighted_der1` in boosting.rs + a non-uniform-weight gradient regression test.

Deferral note (for verify-phase): this plan does NOT close the deferred YetiRank/YetiRankPairwise/
StochasticRank end-to-end TRAINER fixtures (truth #5) nor SC-3/D-07 instrumented-harness validation
(truth #7). LOSS-04 stays partially open after this gap-closure round.
