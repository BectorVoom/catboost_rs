# Phase 07.1: Code Review Report

**Reviewed:** 2026-06-20
**Depth:** standard
**Files Reviewed:** 6
**Status:** issues_found

## Summary

The phase ships the GPU-backend runtime aliases (`SelectedRuntime` per backend) and
the Phase-7.1 device primitives: `block_reduce_kernel` (plane / shared-mem fallback),
`block_reduce_atomic_kernel` (in-kernel atomic finalize), and `block_scan_kernel`
(inclusive/exclusive), plus the `launch_*` host helpers and `rocm` self-oracles. The
two large files also carry forward-phase (7.2 der seam, 7.3 histogram) code that is in
scope because it lives in the reviewed files.

Adversarial assessment: the core reduce/scan kernel math and the host-side guard
discipline (typed errors over panics, overflow-checked products, single-cube scan
precondition enforced) are sound — I traced the plane fold, the shared-mem tree-reduce,
and the Hillis-Steele cross-plane carry and found no correctness defect on the wave32
single-plane path the oracles exercise. No BLOCKER-class correctness or security defect
was provable.

The findings below are real coverage and robustness gaps. The most consequential is a
**test-coverage hole**: on the phase's primary validation hardware (gfx1100), the
capability gate routes the atomic-finalize helper to the host-sum fallback, so the
in-kernel atomic reduce kernel — a headline deliverable of commit 4916c8e — is never
actually executed by the in-env oracle.

## Warnings

### WR-01: In-kernel atomic-reduce kernel is unexercised on the in-env gfx1100 oracle

**File:** `crates/cb-backend/src/gpu_runtime.rs:258-274` (gate); `crates/cb-backend/src/kernels/reduce.rs:197-253` (oracle)
**Issue:** `launch_block_reduce_atomic_f64` gates the in-kernel atomic path behind
`device_supports_f64_atomic_add(&client)` (line 270). The code's own comment at
`gpu_runtime.rs:1293` documents that on gfx1100 HIP returns `false` for advertised f64
atomic-add ("HIP does not ADVERTISE f64 atomic-add even though it runs it"). Consequently
the oracle test `block_reduce_atomic_finalize_matches_cpu_sum_and_reports_variance` takes
the `HostSumFallback` branch (gpu_runtime.rs:271-273) on the in-env rocm device and
reports `path=HostSumFallback`. `block_reduce_atomic_kernel` (kernels.rs:1042-1102) — the
`acc[0].fetch_add(cube_partial)` cross-cube finalize that is the entire point of commit
4916c8e — is therefore never launched on the hardware the phase claims validates it. The
"run-to-run variance" the test purports to observe is deterministic (host sum), so the
test passes for the wrong reason and gives no signal on the atomic kernel.
**Fix:** Add a test path that launches `block_reduce_atomic_kernel` directly (as
`run_reduce` does for the non-atomic kernel) rather than going through the
capability-gated helper, so the atomic kernel is exercised on gfx1100 regardless of the
*advertised* capability. Alternatively, since the comment asserts the device runs f64
atomics despite not advertising them, drive the atomic kernel unconditionally in the
oracle and use the capability query only for the production fallback decision. Document in
the SUMMARY that the production helper's atomic branch is dead on gfx1100 so downstream
phases do not assume it is covered.

### WR-02: Capability fallback in `launch_block_reduce_atomic_f64` invites a silent atomic-to-deterministic mode switch

**File:** `crates/cb-backend/src/gpu_runtime.rs:258-274`
**Issue:** The helper returns `(f64, AtomicFinalizePath)`, but the `HostSumFallback` path
(lines 271-273) recomputes via `launch_block_reduce_f64` + `cb_core::sum_f64`. A future
caller that ignores the returned `AtomicFinalizePath` (easy: `let (sum, _) = ...`) gets a
silent semantic divergence — the "atomic" entry point quietly produces a *deterministic
host* sum. This is exactly the silent-mode-switch the rest of the file guards against
(e.g. WR-05 read-back errors are made loud). The risk is latent, not yet triggered, but
the API shape invites it.
**Fix:** Either rename to make the dual behavior explicit
(`launch_block_reduce_best_effort_atomic_f64`) or have the oracle/consumers assert on the
returned path, so a fallback on a device that is *expected* to support atomics surfaces as
a failure rather than passing silently.

### WR-03: `gpu_runtime.rs` mixes three phases' surface (7.1 reduce/scan, 7.2 der seam, 7.3 histogram) in one 1309-line module

**File:** `crates/cb-backend/src/gpu_runtime.rs:1-1309` (whole file)
**Issue:** The module doc (lines 1-10) scopes this file to the Phase-7.1 reduce/scan
primitives, but the file also hosts the entire 7.2 der seam (`DerBinaryKernel`,
`DerUnaryKernel`, `DerParamKernel`, six `launch_der_*` fns) and the 7.3 pointwise
histogram seam (`launch_pointwise_hist2*`, `read_binsums_f64`). The phase-7.1 review
boundary is diluted and the file has grown past a maintainable single-responsibility size;
the `kernels::{...}` import at lines 41-46 pulls in 11 kernels, most unrelated to 7.1.
Maintainability defect, not correctness — but it makes per-phase review and future
bisection materially harder, and the module doc no longer matches the contents.
**Fix:** Split the der seam and the histogram seam into sibling modules (e.g.
`gpu_runtime/der.rs`, `gpu_runtime/histogram.rs`), leaving `gpu_runtime.rs` with only the
reduce/scan primitives its doc describes. Until the split lands, update the module doc to
list the actual contents.

### WR-04: Reduce/atomic kernels assume a power-of-two `CUBE_DIM_X`; the shared-mem tree-reduce silently drops elements otherwise

**File:** `crates/cb-backend/src/kernels.rs:994-1002` (reduce); `crates/cb-backend/src/kernels.rs:1081-1089` (atomic)
**Issue:** The fallback tree-reduce halves the stride (`s = CUBE_DIM_X / 2; ... s /= 2`).
This is correct only when `CUBE_DIM_X` is a power of two: for a non-power-of-two cube width
the integer-halved stride never covers the top element(s), silently dropping them from the
sum. Today `CUBE_DIM = 32` (gpu_runtime.rs:69) is hard-coded and a power of two, and
`BLOCK_REDUCE_SHMEM = 32` matches, so the bug is latent. But nothing asserts or documents
the precondition; a future launch-geometry change to a non-power-of-two width would produce
a silently wrong reduction with no compile-time or run-time guard.
**Fix:** Document the power-of-two precondition next to `BLOCK_REDUCE_SHMEM` and add a
`const _: () = assert!(CUBE_DIM.is_power_of_two());` guard in `gpu_runtime.rs` alongside the
existing `CUBE_DIM <= BLOCK_REDUCE_SHMEM` assertion (gpu_runtime.rs:77-81), so any future
drift is a compile error rather than a silent wrong sum.

## Info

### IN-01: Dead `sync_cube()` after the cross-plane carry read in `block_scan_kernel`

**File:** `crates/cb-backend/src/kernels.rs:1200-1204`
**Issue:** `carry` is read from `partials[PLANE_POS - 1]` (lines 1201-1203) and then
`sync_cube()` is called at line 1204 — *after* the read. The barrier the read depends on
already executed at line 1193 (the trailing sync of the Hillis-Steele loop). The post-read
sync synchronizes nothing relevant to `carry` and adds a needless cube-wide barrier on the
scan hot path.
**Fix:** Remove the `sync_cube()` at line 1204 (it is not needed for correctness on the
single-plane oracle path).

### IN-02: `max_divergence` indexes `device[i]` over `baseline.len()` instead of zipping

**File:** `crates/cb-backend/src/kernels/scan.rs:88-102`
**Issue:** The loop `for i in 0..baseline.len() { device[i] ... }` panics if
`device.len() < baseline.len()`. Callers assert equal length first, and this is test code
(indexing is permitted by the `lib.rs:1` `#[cfg(test)]` allow), so it is not a production
defect. But the analogous helper elsewhere was already hardened to zip with an explicit
length precondition (commit 252c33a, IN-03 in phase 07.3); this one remains inconsistent.
**Fix:** `for (d, b) in device.iter().zip(baseline) { ... }` to make the length
precondition structural and match the already-hardened sibling.

### IN-03: Oracle `assert!` tolerances are hand-tuned magic literals repeated across tests

**File:** `crates/cb-backend/src/kernels/reduce.rs:121,144,181,250`; `crates/cb-backend/src/kernels/scan.rs:116,138,157`
**Issue:** The "generous run-stable" bounds (`1e-3` f32, `1e-9` f64, `1e-6`/`1e-12`
absolute) are repeated as bare literals across seven assertions in two files. The module
docs explain the rationale, but the values are not named constants, so a future epsilon
policy change (the 7.6 sign-off the docs defer to) must edit each call site and risks
drift between the reduce and scan oracles.
**Fix:** Hoist the f32/f64 relative and absolute tolerances into named `const`s shared by
the oracle modules so the 7.6 sign-off edits one place.

### IN-04: Stale `Cargo.toml` comment calls the GPU arms "inert placeholders until Phase 7"

**File:** `crates/cb-backend/Cargo.toml:13-16`
**Issue:** The dependency comment (lines 13-14) still says "The `wgpu`/`cuda`/`rocm` arms
remain inert placeholders until Phase 7", but `lib.rs:51-65` (this phase) already resolves
those arms to real CubeCL runtimes. The comment now contradicts the implemented state and
can mislead a future reader into thinking the GPU arms are no-ops.
**Fix:** Update the comment to state that as of Phase 7.1 the `wgpu`/`cuda`/`rocm` arms
resolve to real runtimes, matching the `lib.rs` module docs (lib.rs:4-8).

### IN-05: Empty-input short-circuits of the production launch helpers are unverified

**File:** `crates/cb-backend/src/kernels/reduce.rs:157-161`; `crates/cb-backend/src/kernels/scan.rs` (no empty-input test)
**Issue:** The empty-slice "edge case" in the reduce oracle (lines 157-161) only asserts
`cb_core::sum_f64(&empty) == 0.0` — a pure host-side check that never touches the device or
the `launch_block_reduce_f64` empty short-circuit (gpu_runtime.rs:114-117). The scan oracle
has no empty-input test at all. The empty-input short-circuits in production
(`launch_block_reduce_f64`, `launch_block_scan_f64`) are therefore unverified.
**Fix:** Add assertions that `launch_block_reduce_f64(&[])` and
`launch_block_scan_f64(&[], _)` return `Ok(vec![])`, so the production empty short-circuits
are actually exercised rather than only the host baseline.
