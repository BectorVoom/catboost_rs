## Accomplishments
- Shipped `block_scan_kernel<F: Float>` (GPU-01 scan): a `#[comptime] inclusive`-flagged block prefix-scan that folds the within-plane segment via `plane_inclusive_sum` / `plane_exclusive_sum` (width = `PLANE_DIM`, never a literal) and carries across planes through a shared-memory Hillis-Steele stride-doubling loop (`s = 1,2,4,…`, `sync_cube()` between stages, bound from `CUBE_DIM_X`/`PLANE_DIM`) — structural parity with upstream `cuda_util/kernel/inplace_scan.cuh::InplaceInclusiveScan` (D-01). Wave-size-agnostic; no warp-size literal in any stride (D-09).
- `launch_block_scan_f64(input, inclusive)` in `gpu_runtime.rs`: per-element scan output (a scan is not a reduction — output length == input length), `inclusive` threaded through as the kernel's comptime arg, WR-05 typed-error read-back (`CbError::Degenerate`), no production `unwrap`/`expect`/`panic`/indexing.
- `kernels/scan.rs` rocm self-oracle: inclusive AND exclusive scan vs Rust CPU prefix-sums, f32 + f64, n=1 edge case; reports max abs/rel divergence; asserts `output[0] == 0` for the exclusive variant. **4/4 green on gfx1100, max divergence 0.000e0** (single plane on wave32 — the plane-op prefix is exact); wgpu builds+runs 4/4 (exercising the shared-mem Hillis-Steele cross-plane carry on the host); cuda compiles.
- Shipped `block_reduce_atomic_kernel<F: Float>` (D-03 / D-7.1-07): the CUDA in-kernel-atomic reduction structure — each cube's representative (`UNIT_POS == 0`) `fetch_add`s its partial into a length-1 global `Atomic<F>` accumulator, accepting the non-deterministic cross-cube summation order. Landed as a SIBLING kernel so the Plan-01 atomic-free `block_reduce_kernel` default is byte-for-byte unchanged.
- `launch_block_reduce_atomic_f64 -> (f64, AtomicFinalizePath)`: queries `client.properties().atomic_type_usage` for f64 `AtomicUsage::Add`; runs the in-kernel atomic when supported, else the portable Plan-01 host-sum fallback — and RETURNS which path ran (Pitfall 4). `kernels::reduce` extended with the atomic-variant oracle that runs the finalize 32× and reports the run-to-run spread + max divergence (D-03 non-determinism; GPU-06 epsilon left to 7.6). **rocm reduce 4/4 green**; wgpu/cuda/cpu builds unbroken.

## Decisions Made
- **Scan scope = single cube (N ≤ CUBE_DIM), cross-cube carry deferred (RESEARCH Open Q2):** the kernel performs the prefix-scan correctly within one cube; on wave32 gfx1100 at CUBE_DIM 32 this is exactly one plane and the cross-plane carry collapses to identity, which is why the rocm divergence is 0.000e0. The shared-mem Hillis-Steele cross-plane carry is fully implemented (and exercised on the wgpu host path), but the **cross-cube running carry** (adding each cube's running offset to the next cube's elements) is NOT performed and is recorded as the **first forward dependency for 7.2/7.3**. This is an explicit documented scope, not a silent cut — see "Forward Dependencies" below.
- **f64 atomic-add unsupported on HIP → host-sum fallback (Open Q1 RESOLVED empirically):** `launch_block_reduce_atomic_f64` reported `path=HostSumFallback` on gfx1100 — `client.properties().atomic_type_usage` does NOT contain `AtomicUsage::Add` for f64 at cubecl 0.10.0 on this HIP path (exactly the Pitfall-4 scenario the research predicted). The D-03 in-kernel-atomic kernel IS implemented and the host gates correctly to the portable fallback; the variance spread is therefore 0.000e0 (the deterministic host fold). This is REPORTED, not dropped — the GPU-06 epsilon / the real in-kernel-atomic non-determinism sign-off remains 7.6's job, and will require either an f64-atomic-capable device or the f32-atomic path.
- **Atomic variant as a sibling kernel, not a comptime branch:** keeps the Plan-01 default reduce path byte-for-byte unchanged (lower blast radius; the default has no atomic dependency at all).

## Forward Dependencies (explicit, NOT silent scope cuts)
- **Cross-cube scan carry (first 7.2/7.3 dependency):** `block_scan_kernel` is correct within a single cube (N ≤ CUBE_DIM). Multi-cube inputs need a second pass that adds each cube's running offset to the next cube's elements (the host-orchestrated or two-level device carry). The single-cube primitive + its oracle are the foundation; the cross-cube carry is owned by the first sub-phase that scans N > CUBE_DIM.
- **f64 in-kernel-atomic non-determinism sign-off (7.6):** because gfx1100/HIP lacks f64 atomic-add at cubecl 0.10.0, the true D-03 run-to-run variance was not observable here (the fallback is deterministic). 7.6 must either run on an f64-atomic-capable device, use the f32-atomic path, or sign off the epsilon against the host-fold baseline — the structure is in place behind the capability gate.
