## Decisions Made
- **Insertion site:** the monotone PAVA runs in the diagonal/pointwise leaf branch, AFTER `compute_leaf_deltas` returns the raw deltas and BEFORE `normalize_leaf_values` applies the learning-rate/centering — exactly where upstream `CalcMonotonicLeafDeltasSimple` (`approx_calcer.cpp:551`) inserts it (the deltas it adjusts are raw, pre-lr). `curr_leaf_values == 0` because `leaf_estimation_iterations == 1`.
- **Direction handling:** a `-1` (decreasing) split is encoded by the linear-order construction itself (the order starts at the larger leaf and descends), so the SAME ascending PAVA enforces non-increasing along it — no separate negation path, matching upstream `BuildLinearOrderOnLeafsOfMonotonicSubtree`.
- **Leaf-index bit parity:** verified cb-train's forward-bit `leaf_index` (split `i` -> bit `1<<i`) matches upstream's `currDepthBitMask`, so the transcribed bitmask order construction is directly compatible.

**1. [Rule 3 - Blocking] Test placed in cb-train/tests/ rather than cb-compute/tests/**
- **Found during:** Task 2 (test authoring)
- **Issue:** The plan listed `crates/cb-compute/tests/monotone_oracle_test.rs`, but the train->predict oracle harness (`train`, `compare_stage`, `predict_raw`, the `fixture()` resolver, `cb-oracle`/`cb-model` deps) lives in `cb-train` + `cb-oracle`; `cb-compute` has no training entry point. - **Files modified:** `crates/cb-train/tests/monotone_oracle_test.rs`
- **Committed in:** `fb40de4`

**2. [Rule 1 - Bug / non-vacuous oracle] Fixture pins model_shrink_rate=0 and uses binding constraints [-1,0,1,0]**
- **Found during:** Task 2 (initial oracle ran with `[1,0,-1,0]` and `model_shrink_rate` unset)
- **Issue:** Two distinct problems surfaced: (a) leaf values diverged ~1.2e-3 because CatBoost AUTO-enables model shrinkage (`model_shrink_rate ~= 0.01`, a per-tree multiplicative leaf decay) the moment monotone constraints are present — an unrelated mechanism that confounds the projection; and (b) `[1,0,-1,0]` was a NO-OP on this data (the unconstrained leaves already satisfied it), so the oracle would have passed even if the PAVA were a no-op (vacuous).
- **Fix:** Pinned `model_shrink_rate=0` in the generator (isolating the PAVA as the only difference vs unconstrained) and switched to `[-1,0,1,0]`, which GENUINELY binds (max leaf diff ~3.9e-1). Added a generator-time assertion that the constrained predictions differ from the unconstrained baseline (non-vacuous guard).
- **Files modified:** `crates/cb-oracle/generator/gen_monotone_fixtures.py`, regenerated `monotone/increasing_decreasing/*`
- **Verification:** `cargo test -p cb-train --test monotone_oracle_test` -> 2 passed (leaf values <=1e-5; end-to-end non-increasing-in-feature-0).
- **Committed in:** `fb40de4`
