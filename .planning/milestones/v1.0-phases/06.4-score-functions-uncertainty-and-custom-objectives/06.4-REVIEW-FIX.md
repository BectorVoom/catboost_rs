## Fixed Issues

### CR-01: Newton score functions silently degrade to first-order; der2-fill primitive is dead code

**Files modified:** `crates/cb-train/src/boosting.rs`, `crates/cb-train/src/tree.rs`, `crates/cb-compute/src/histogram.rs`, `crates/cb-compute/src/histogram_test.rs`, `crates/cb-compute/src/lib.rs`
**Commit:** 0eb036d
**Status:** fixed: requires human verification (behavior change — train-time rejection gate)
**Applied fix:** Chose review option (b). Added `validate_score_function` (called
right after `loss.validate()` in the train entry) that rejects `NewtonL2` /
`NewtonCosine` with a typed `CbError::OutOfRange` — these reuse the L2/Cosine
formula verbatim and rely on a der2-hessian histogram fill the CPU scoring path
never produces, so they would silently degrade to their first-order counterparts.
Deleted the dead `reduce_leaf_stats_newton` primitive, its `pub use` re-export, and
its two self-oracle tests (CLAUDE.md no-dead-code rule). Corrected the false
`split_score` doc-comment. The first-order GPU-only variants
(`SolarL2`/`LOOL2`/`SatL2`) compute correctly from first-order stats and are NOT
rejected. The `EScoreFunction` enum keeps all variants (the formula seam in
`multi_dim_split_score` is unchanged for a future GPU der2-fill path).

### WR-01: MultiQuantile backend swallows a device read-back failure as a zero gradient

**Files modified:** `crates/cb-backend/src/cpu_runtime.rs`
**Commit:** 044258c
**Applied fix:** Changed `compute_multiquantile_gradients` to return
`CbResult<(Vec<f64>, Vec<f64>)>` and propagate the `launch_quantile_f64` failure
with `?` (caller updated to `?`), matching the WR-05 discipline used by every other
launcher. A swallowed launch error no longer masquerades as an all-zero gradient.

### WR-02: Custom objective default leaf method is Gradient, contradicting documented Newton default

**Files modified:** `crates/cb-compute/src/custom.rs`, `crates/cb-compute/src/runtime.rs`
**Commit:** 8bca2bc
**Applied fix:** Chose review option (b) — corrected the docs rather than changing
training behavior. The `CustomObjective` trait doc and the `Loss::Custom` enum doc
now state the leaf method is caller-selected (Builder default `Gradient`, which uses
only der1) and that the user-supplied der2 is consumed only under the Newton leaf
method. (The Builder default of `LeafMethod::Gradient` is unchanged.)

### WR-03: RMSEWithUncertainty per-dim bias is not stored in the Model; scalar bias drops dim-1

**Files modified:** `crates/cb-model/src/apply.rs`
**Commit:** ae67e14
**Applied fix:** Applied the review's "at minimum" path. Documented
`predict_raw_multi` (and the shared `predict_raw_multi_cat` seed) as UNSAFE for a
genuine per-dim-bias model (RMSEWithUncertainty dim-1 log-scale), directing such
callers to `predict_raw_multi_biased`. The full `bias_per_dim` Model field reverses
the explicit 06.4-02 "no per-dim bias struct field" decision and touches the loader
+ serialization round-trip + scalar D-04 byte-identity surface, so it is deferred to
a planned change rather than forced in a review-fix pass.

### WR-04: apply_prediction_type(RmseWithUncertainty) returns empty on odd-length input

**Files modified:** `crates/cb-model/src/predict.rs`
**Commit:** 233b1d6
**Applied fix:** Applied the review's "at minimum" path. The function has no error
channel (`-> Vec<f64>`); documented on both the function and the
`RmseWithUncertainty` arm that an empty return on a NON-EMPTY input signals a shape
violation (not "no predictions") so consumers can detect it. A fallible signature
is a cross-arm/caller refactor and is deferred.

### WR-05: EvalMetric::eval and eval_grouped disagree on the missing-weight fallback

**Files modified:** `crates/cb-train/src/metrics.rs`
**Commit:** 8612f8a
**Status:** fixed: requires human verification (changes a fallback value)
**Applied fix:** Aligned `eval`'s `weight_at` closure from `unwrap_or(0.0)` to
`unwrap_or(1.0)`, matching `eval_grouped` and the upstream "missing weight == 1.0"
convention. After the up-front length check this branch is unreachable, so no
fixture metric value changes; the change makes the two paths consistent if the guard
is ever relaxed.

### WR-06: build_class_remap NaN labels collapse silently

**Files modified:** `crates/cb-train/src/boosting.rs`
**Commit:** 3587d71
**Applied fix:** Added a non-finite-label guard at the train entry's multiclass
branch (before `build_class_remap`): a NaN/Inf class label now returns a typed
`CbError::OutOfRange` rather than producing a non-total sort / surviving the
exact-difference dedup and corrupting `class_to_label`. Updated the
`build_class_remap` doc to record the upstream guard.

### WR-07: Score-function self-oracles are vacuous for the Newton variants

**Files modified:** `crates/cb-compute/tests/score_functions_test.rs`, `crates/cb-train/src/boosting_test.rs`
**Commit:** d47c0bf
**Applied fix:** Documented the true scope of the two `newton_*_reuses_*_formula`
self-oracles (formula-seam-only; cannot detect CR-01). Added three falsifiable
regression tests in `boosting_test.rs` asserting `validate_score_function` rejects
`NewtonL2`/`NewtonCosine` and accepts the first-order variants — the real guard
against silent first-order degradation. All three new tests pass.

### IN-01: multi_dim_split_score term vectors are not pre-sized

**Files modified:** `crates/cb-compute/src/score.rs`
**Commit:** 89d52cd
**Applied fix:** Pre-sized `num_terms` and the Solar/LOO/Sat `*_terms` Vecs with the
total per-(dim,leaf) count (`per_dim_leaves.iter().map(Vec::len).sum()`), matching
the Cosine denominator's `with_capacity`. The fold ORDER is unchanged
(dimension-then-leaf), so dim=1 byte-identity is preserved — verified by the
score-functions oracle suite (9/9 pass).

### IN-04: solar_l2_terms ignores scaled_l2 (correct per CUDA source, but easy to misread)
