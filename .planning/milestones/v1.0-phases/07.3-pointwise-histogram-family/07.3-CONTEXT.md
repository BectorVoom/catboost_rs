<domain>
## Phase Boundary

The third additive GPU sub-phase of the Phase 7 umbrella. It ports the
**pointwise (single-feature) histogram kernel family** —
`catboost/cuda/methods/kernel/pointwise_hist2*` — to run **device-resident** on
the GPU runtimes, generic over `R: Runtime`/`F: Float`, accumulating with
CUDA-style **in-kernel atomics** (umbrella D-03). `hist2` is the **2-channel**
histogram: for every (feature, bin) it sums the first-derivative ("target"
channel) and the weight channel — exactly upstream's "sums for 2 elements".
These histograms are what the 7.5 score/split path reduces to pick splits.

The umbrella `07-CONTEXT.md` (D-01…D-09), `07.1-CONTEXT.md` (D-7.1-01…10), and
`07.2-CONTEXT.md` (D-7.2-01…07) are the **binding shared context**: full
**structural-parity** fidelity bar (D-01), the **in-kernel atomic** determinism
contract with f64 finalization (D-03), the **empirical GPU-06 epsilon signed
off only in 7.6**, **`rocm`-only validation never in GitHub Actions** (D-06),
the **wave-size-agnostic** policy / no warp-size constants (D-09), and the 7.2
**device-resident der1/der2 hand-off seam** (D-7.2-04: der handles in, no host
fold; weight folded downstream HERE). Those are **already locked and NOT
re-decided here.** This phase resolves only the HOW-level choices specific to
the pointwise histogram fill and its self-oracle.

**In scope:** GPU-01 (the pointwise-histogram slice only) — the `pointwise_hist2`
family ported at structural parity: the **general one-byte non-binary path**
(`ComputeHist2NonBinary<Bits>`, 5/6/7/8-bit), the **half-byte** path
(`ComputeHist2HalfByte`), and the **binary** path (`ComputeHist2Binary`),
generic over `R`/`F`, device-resident, self-oracled against the Rust CPU
histogram baseline. Consumes the 7.2 der1/der2 device handles; produces a
device-resident `binSums`-equivalent histogram buffer.

**Out of scope (owned by later sub-phases):** the **pairwise** histogram family
(`pairwise_hist_*`, incl. 8-bit atomics + one-hot) → 7.4; the **score/split
calcers and the full device-resident tree-grow loop** (`pointwise_scores.cu`,
`ScanPointwiseHistograms`/`UpdatePointwiseHistograms` integration into the grow
loop, candidate selection) → 7.5; the **empirical GPU-06 epsilon sign-off +
`rocm` oracle-gate lock-in** → 7.6. The single-channel `pointwise_hist1*` and
the `greedy_subsets_searcher/kernel/hist*` variant are **not** ported here
(hist2 is the parity surface for the oblivious-tree path; hist1/subset-searcher
are additive follow-ons only if a downstream GPU consumer needs them).

## Implementation Decisions

All decisions below were auto-resolved to the recommended default in `--auto`
mode. They are HOW-level choices within the boundary the umbrella + 7.1 + 7.2
already locked; none reopen D-01…D-09, D-7.1-01…10, or D-7.2-01…07. Numbered
with a `7.3-` prefix to avoid collision.

### Variant coverage & in-phase sequencing (SC-1)
- **D-7.3-01:** The **full** `pointwise_hist2` family is in scope (SC-1 requires
  the general path plus 5/6/7/8-bit, half-byte, and binary specializations all
  run on `rocm`). Sequence them **narrowest-first across plans**, mirroring the
  7.1/7.2 "resist porting the whole set up front" pattern: land the general
  **one-byte non-binary** path + the device seam first (the `ComputeHist2NonBinary`
  template, parameterized to 8-bit), then add the 5/6/7-bit cases on the same
  template, then **half-byte**, then **binary** as additive plans on the same
  seam. Every variant must land in THIS phase — the sequencing is plan
  decomposition, not a scope cut.

### Bit-width specialization mechanism in CubeCL (research-flagged)
- **D-7.3-02:** Structurally mirror upstream's **three distinct kernel families**
  rather than one mega-kernel: (a) one-byte non-binary (`pointwise_hist2_one_byte_templ.cuh`),
  (b) half-byte (`pointwise_hist2_half_byte_template.cuh`), (c) binary
  (`pointwise_hist2_binary.cu`) — because upstream's shared-memory layout and
  load/unroll differ structurally between them (D-01). Within the non-binary
  family the **bit-count (5/6/7/8)** is carried as a **`#[comptime]` parameter**
  controlling histogram line size, shared-memory blocking, and unroll — NOT a
  runtime branch (mirrors the 7.1 `use_plane` comptime-specialization pattern,
  pruned at JIT time). **The exact comptime-vs-monomorphization expression in
  CubeCL is the research-flag item** (roadmap "Research flag" for 7.3) — leave
  the precise mechanism to RESEARCH.md, subject to D-09 (no warp constants) and
  AGENTS.md generics-float.

### Accumulation strategy — shared-memory blocking + in-kernel atomics (D-03)
- **D-7.3-03:** Structurally mirror `compute_point_hist2_loop.cuh`: per-block
  **shared-memory histogram** accumulation over the block's object slice, then
  an **atomic merge** of block-local histograms into the global histogram
  buffer, honoring D-03's **in-kernel-atomic determinism contract with f64
  finalization**. Reuse the 7.1 atomic-finalize building block
  (`block_reduce_atomic_kernel`, D-7.1) and the wave-agnostic plane/shared-mem
  fallback pattern (D-09 — shared-memory size is a `#[comptime]` const, stride
  derives from `CUBE_DIM_X`/`PLANE_DIM`, never a literal 32/64). This is the
  **real on-device fill** — distinct from the existing scatter-only
  `histogram_scatter_kernel` (D-02 per-object write + host-ordered sum), which
  remains the CPU/parity *baseline*, not the GPU fill path. D-03/D-09 are
  **locked by the umbrella — restated here, not reopened.**

### CPU self-oracle baseline & fixtures (SC-2)
- **D-7.3-04:** Self-oracle the device-resident histogram against a **host
  reference 2-channel histogram** built with the existing **ordered**
  `cb-core::sum_f64` reduction — generalize the `cb-compute/src/histogram.rs`
  `reduce_leaf_stats` gather-then-`sum_f64` pattern from leaves to **(feature,
  bin)** cells (per bin: Σ der1 = `sum_weighted_delta`, Σ weight = `sum_weight`).
  The harness **REPORTS** observed max absolute + relative divergence (run-to-run
  atomic variance from D-03 absorbed) as the current tolerance; it does **NOT**
  hard-code or sign off the GPU-06 epsilon — that is 7.6's job (D-7.2-06
  pattern). Fixtures: random f32 + f64, non-trivial weights, edge cases (empty,
  single object, size not a multiple of the block/cube, large N), exercised
  **across the covered bit-widths / border counts** (binary, half-byte, 5/6/7/8-bit).
  Tests live in dedicated `_test.rs` files (source/test separation), run on
  `rocm` in-env.

### Device-residency input/output seam (SC-1 / hand-off to 7.5)
- **D-7.3-05:** Histogram kernels consume the **7.2 der1/der2 device handles**
  directly (NO host fold inserted — the seam locked in D-7.2-04; **per-object
  weight is folded HERE downstream**, as 7.2 promised) plus a device-resident
  **quantized-bin input** (`cindex`) + **indices/partition** layout mirroring
  upstream's `(features, cindex, target, weight, indices, partition, binSums)`
  signature. Output is a **device-resident histogram buffer (handle)** —
  upstream's `binSums` — consumable by the 7.5 score/split + grow loop with **no
  host round-trip**. Define this seam (der handles + bins in → histogram handle
  out) now so 7.5 plugs in without a round-trip. The post-fill
  `ScanPointwiseHistograms`/`UpdatePointwiseHistograms` integration is 7.5's job;
  this phase delivers the FILL and its handle.

### Verification gating (carried from D-7.2-07, restated)
- **D-7.3-06:** Verification is **per-crate**, honoring the standing
  disk-pressure constraint (root ~100% full; full-workspace test links fail):
  `rocm` gets build + **run + self-oracle** in-env on gfx1100; `wgpu` gets build
  + run on host (GPU-04, no ROCm/CUDA needed); `cuda` gets **build/compile only**
  behind its feature gate (no NVIDIA hardware — D-07). `rocm` GPU tests run
  **locally/manually, never in GitHub Actions** (umbrella D-06).

### Boundary preservation (SC-4, carried)
- **D-7.3-07:** All cubecl coupling stays **inside `cb-backend`** (D-7.1-04 /
  D-7.2-05). Do **not** widen the frozen `cb-compute` `Runtime`/`Float` traits;
  prefer backend-internal launch helpers. `cb-compute` stays cubecl-free;
  `cb-core`/`cb-model` stay byte-unchanged from Phase 3–6 form (SC-4 / additive
  boundary). The CPU `cb-compute/src/histogram.rs` reduction is the parity
  baseline and is NOT modified to accommodate the GPU path.

### Claude's Discretion
- The exact CubeCL comptime-vs-monomorphization expression of the bit-width
  specialization (the research-flag item), the precise shared-memory histogram
  line-size / blocking geometry per bit-width, the launch geometry, the exact
  device-buffer handle type for the histogram output, the partition/`fullPass`
  handling depth needed for the single-tree fill (vs the multi-part grow-loop
  detail deferred to 7.5), and the precise representative fixture set are left to
  research + planning — subject to umbrella D-01 (structural parity), D-03
  (in-kernel atomics + f64 finalize), D-09 (no warp-size constants), AGENTS.md
  generics-float, and the 7.3 decisions above.

## Existing Code Insights

### Reusable Assets
- **`block_reduce_atomic_kernel`** (`cb-backend/src/kernels.rs`, 7.1) — the
  D-03 in-kernel atomic-finalize pattern (shared-mem tree-reduce → plane
  reduction, comptime shmem size, no warp literals) that the histogram block
  accumulation reuses as its merge primitive.
- **7.2 device-resident der seam** (`gpu_runtime.rs`: `launch_der_*_handle`,
  `const_der_handle`) — der1/der2 handles arrive on-device; the histogram fill
  plugs straight in, weight folded here (D-7.3-05).
- **`gradient_gpu.rs` self-oracle harness** — the all-backend
  (rocm/wgpu/cuda-build-only) launch + host-reference + report-divergence
  pattern to clone for the histogram self-oracle.
- **`cb-compute/src/histogram.rs::reduce_leaf_stats`** — the ordered
  `cb_core::sum_f64` 2-channel reduction; generalize leaf→bin for the host
  reference histogram (parity baseline).

### Established Patterns
- **Scatter (D-02) vs in-kernel fill (D-03):** the existing
  `histogram_scatter_kernel` deliberately does NOT accumulate on-device (host
  does the ordered sum). 7.3 introduces the genuine on-device accumulation with
  atomics — a NEW structural surface, self-oracled against the D-02/host path.
- **Comptime specialization (no warp literals):** the 7.1 `use_plane`
  `#[comptime]` flag pruned at JIT time is the template for expressing the
  5/6/7/8-bit / half-byte / binary specializations wave-agnostically (D-09).
- **Per-call client + `Bytes::from_elems` device memory** — the uniform
  host↔device transfer + launch geometry across CPU/wgpu/rocm/cuda arms.

### Integration Points
- **In:** 7.2 der1/der2 device handles + a device-resident quantized-bin
  (`cindex`) / indices / partition input.
- **Out:** a device-resident histogram (`binSums`) handle for the 7.5
  score/split + device-resident grow loop (no host round-trip).
- **Boundary:** all new code in `cb-backend`; `cb-compute`/`cb-core`/`cb-model`
  untouched (SC-4).

## Specific Ideas

- `hist2` = exactly 2 channels (Σ der1 "target", Σ weight) per (feature, bin) —
  the parity surface for the oblivious-tree split scorer. `hist1` (single
  channel) and the `greedy_subsets_searcher` hist variant are explicitly NOT in
  this phase.
- Upstream selects the variant by **feature bit-width** (binary → half-byte →
  5/6/7/8-bit non-binary); `histograms_helper.cpp` is the reference for that
  dispatch shape.
- The post-fill `ScanPointwiseHistograms`/`UpdatePointwiseHistograms` are part
  of the score/grow integration (7.5), NOT this phase — 7.3 delivers the fill +
  its device handle.
</specifics>
