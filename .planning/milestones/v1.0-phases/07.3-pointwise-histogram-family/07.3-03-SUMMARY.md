## FROZEN binSums seam (reused UNCHANGED across a DIFFERENT kernel family)

```text
index(feature, bin, channel) = (feature * n_bins + bin) * 2 + channel   channel 0 = Σ der1, channel 1 = Σ weight
buffer length = n_features * n_bins * 2 floats   (n_bins = HALF_BYTE_BINS = 16 for the half-byte family)
```

The half-byte kernel is a different `#[cube]` fn from the non-binary kernel, but it writes the SAME interleaved index with `n_bins=16`, so the 7.5 score/split consumer reads one layout regardless of feature bit-width. The seam, buffer-length formula, and der-handle-in -> binSums-handle-out hand-off are byte-identical to Plans A/B (D-7.3-01). The `fullPass=false` multi-part offset remains a 7.5 forward dependency (documented in Plan A, not filled here).

## Decisions Made
See `key-decisions` frontmatter. Headline: half-byte is a SEPARATE kernel family (fixed comptime 16-bin line + nibble decomposition) routed by an `n_bins==16` dispatch branch, not a comptime case of the non-binary kernel (D-7.3-02); it uses the direct global atomic merge (Plan A's MVP fill decision, D-03) with the shared-mem `TPointHistHalfByte` pre-reduce reserved as an additive perf follow-up; upstream's warp-tile Reduce is re-expressed wave-agnostically (D-09).

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] CubeCL `Atomic::add`/`store`/`load` are not the supported API**
- **Found during:** Task 2 (first rocm compile)
- **Issue:** The initial half-byte kernel draft used a per-cube `SharedMemory::<Atomic<F>>` working histogram with `Atomic::add(&local[cell], d)` / `Atomic::store` / `Atomic::load` to mirror upstream's `TPointHistHalfByte` shared Buffer. CubeCL rejected it: `no associated function 'add' found for Atomic<Inner>` / `Atomic: CubeAdd trait bounds not satisfied` (E0599) — the `Atomic::{add,store,load}` associated-function forms are not the supported API; only the `fetch_add` method form is.
- **Fix:** Per AGENTS.md, loaded `cubecl_error_solution_guide/mismatched types.md` (the E0599 guide) BEFORE fixing. It confirms the `__expand_*`/associated-function fragility pattern. Reverted to the PROVEN direct-global `bin_sums[cell].fetch_add(d)` merge already used by `pointwise_hist2_nonbinary_kernel` and `block_reduce_atomic_kernel` — which is also exactly Plan A's MVP fill decision (direct global atomic; shared-mem pre-reduce deferred as a perf follow-up). The half-byte structural distinctness is preserved via the FIXED comptime 16-bin line + the 4-bit nibble decomposition (NOT the shared-mem layout). No `SharedMemory<Atomic<F>>` is used.
- **Files modified:** `crates/cb-backend/src/kernels.rs`
