<objective>
Deliver GPUT-03 — the highest-engineering-content task of the phase. Refactor the per-tree
device grow so `approx` and `der1` stay **resident** on the session across boosting iterations,
eliminating the per-tree `der1` host read-back and the host approx-update that the Phase-7.5
`grow_boosting_pass_into` still performs (`gpu_runtime/mod.rs:~1996-2036`). Add one small
`#[cube]` kernel `apply_leaf_delta` that does `approx[i] += lr * leaf_values[leaf_of[i]]` on
device, then chain `der1 = der(approx, target)` via the Phase-7.2 `_into` seam WITHOUT read-back.
Add a handle-taking `grow_oblivious_tree_resident` variant so the matrix is NOT re-uploaded per
call (it was uploaded once in 10-03). Implement `GpuBackend::grow_tree_on_device` to grow a
depth-1 oblivious tree over the resident handles and return a host `DeviceGrownTree` — with only
the O(1) `BestSplit` descriptor + `2^depth` part-stats crossing host↔device per level (D-05);
`leaf_of` crosses ONLY for the oracle path, never in the hot loop.

Purpose: Make the device path actually device-resident (the real >20× root cause was the unwired
loop + host crossings), and grow the depth-1 tree (GPUT-04's device side; oracle fixtures + Kaggle
in 10-05).
Output: apply_leaf_delta kernel + resident grow variant + grow_tree_on_device override + in-env residency/structure smoke.
</objective>

| ComputeClient ↔ resident Handle | approx_h/der1_h live across iterations; cross-client read or read-back failure are the risks |
